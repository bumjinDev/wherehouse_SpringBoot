## 피드백의 핵심 전제: "읽는 사람"이 누구인가

이 피드백 전체를 관통하는 단 하나의 원칙은 **"읽는 사람은 기술 면접관이다"**라는 점이다.

면접관의 현실적 행동 패턴을 먼저 이해해야 한다. 현재 백엔드 신입 시장은 극도로 치열하다. 한 포지션에 수십~수백 개의 지원서가 들어오고, 면접관은 하루에 수십 개의 GitHub README를 스크롤한다. 이 상황에서 면접관의 README 읽기 방식은 다음과 같다:

1. **첫 1~2문단(3~5초)**: 이 프로젝트가 "읽을 가치가 있는가"를 판단한다. 여기서 임팩트가 없으면 **탭을 닫는다.**
2. **중반(관심이 생긴 경우에만)**: "이 사람이 실제로 어떤 문제를 풀었는가"를 확인한다.
3. **후반(면접 후보로 올린 경우에만)**: 실제 구현 코드, ERD, 설계 세부사항을 검증한다.

따라서 README의 정보 배치 순서는 면접관의 이 읽기 순서를 정확히 따라야 한다.

## 피드백 1: 기술 스택부터 나오면 읽기 싫다

### 문제 진단

현재 README의 구조를 보면, 프로젝트 개요 직후에 곧바로 아키텍처 다이어그램과 ASCII 흐름도가 등장하고(라인 24~59), 팀원/역할 분담 후 바로 "💻 주요 기술 구현 내용"(라인 90)으로 진입한다. 이 섹션은 Redis Sorted Set 자료구조, 코드 블록, 정규화 공식 등 **구현 세부사항**으로 가득 차 있다.

면접관 입장에서 이것이 왜 문제인가:

- README를 열었는데 첫 스크롤에 보이는 것이 ASCII 흐름도와 `ZRANGEBYSCORE` 명령어다. **"그래서 이걸로 뭘 했다는 건데?"**라는 의문만 남긴다.
- 기술 스택 나열은 **모든 신입 지원자의 README에 있다.** Java, Spring Boot, Redis, Docker... 이것만으로는 차별화가 전혀 되지 않는다.
- 면접관이 첫 화면에서 원하는 정보는 "이 사람이 **무슨 문제**를 풀었고, **얼마나 잘** 풀었는가"이지, "어떤 기술을 **사용했는가**"가 아니다.

### 구체적 개선 방향

**기술 스택 섹션(라인 514~524)의 배지 나열을 README 최하단 또는 완전히 제거한다.** 기술 스택은 이력서에 이미 있고, README에서 기술을 증명하는 방법은 배지가 아니라 **문제 해결 과정의 서술** 자체다.

**아키텍처 다이어그램(라인 24~59)도 앞단에서 제거한다.** 아키텍처는 "왜 이 구조인가"라는 맥락 없이 단독으로 제시되면 의미가 없다. 중후반 기술 구현 섹션에서 맥락과 함께 제시해야 한다.

## 피드백 2: 앞 1~2문단 내 확실한 임팩트가 없으면 안 읽힌다

### 문제 진단

현재 프로젝트 개요(라인 7~20)의 첫 문단:

> "서울 내 1인 가구의 주거지 선택 문제를 해결하기 위해, 사용자의 예산과 안전/편의 선호도를 기반으로 최적의 행정구를 추천하는 서비스입니다."

이 문장의 문제점은 세 가지다:

1. **비즈니스 문제의 규모감이 없다.** "750만 1인 가구 시대"라는 사회적 맥락이 빠져 있어, 왜 이 프로젝트가 의미 있는지 전달되지 않는다.
2. **정량적 성과가 첫 화면에 없다.** 면접관이 3초 안에 "이 사람은 N+1 쿼리를 93.9% 감소시켰다"라는 사실을 알 수 없다.
3. **차별화 포인트가 보이지 않는다.** "Spring Boot 기반으로 인증, 인가, 캐싱..." 이런 나열은 모든 부트캠프 프로젝트 README에 있는 문장이다.

### 구체적 개선 방향

README의 첫 화면(스크롤 없이 보이는 영역)에 반드시 포함되어야 하는 3가지:

**① 비즈니스 문제의 한 줄 정의**
- "MZ 1인가구 750만 시대, 주거 선택 시 범죄율/편의시설/가격을 통합 분석하는 맞춤형 추천 서비스"
- 면접관이 즉시 "아, 실제 사회 문제를 풀려고 했구나"를 파악할 수 있어야 한다.

**② 핵심 기술 챌린지와 성과 수치 요약 테이블**
- 단 3~4행으로 "어떤 문제 → 어떻게 해결 → 정량 결과"를 보여줘야 한다.
- 예시 형태:

| 기술 챌린지 | 해결 방식 | 정량 성과 |
|-----------|---------|---------|
| 실시간 다중 조건 검색 | Redis Sorted Set 인메모리 검색 엔진 | DB 조회 없이 즉시 응답 |
| 외부 API 15개 순차 호출 병목 | CompletableFuture 병렬 처리 | 응답 시간 72.3% 단축 |
| N+1 쿼리 패턴 (25회 반복 호출) | Bulk Fetch + IN절 Chunking | RDB 호출 93.9% 감소, 심각 경합 0% |
| 11만 건 일괄 로드 OOM 위험 | Slice 기반 청크 처리 | 힙 점유율 64.4% → 17.6% |

이 테이블 하나가 면접관에게 "이 사람은 읽어볼 가치가 있다"는 판단을 3초 안에 내리게 만든다.

**③ 본인 역할의 명확한 한 줄 요약**
- "3인 팀에서 백엔드 아키텍처 설계~구현 전담 (배치 파이프라인, 추천 엔진, 인증 시스템, DB 최적화)"

## 피드백 3: 앞단 구성 — 문제 → 기획 → 작업 프로세스 → 실제 작업

### 문제 진단

현재 README는 **"기술 구현 내용"이 곧 프로젝트 설명**이다. 즉, "배치 기반 데이터 파이프라인", "2단계 폴백 알고리즘", "JWT 인증 시스템" 등 구현 단위로 섹션이 구성되어 있다.

이 구조가 읽는 사람에게 전달하지 못하는 것:
- **왜 이 프로젝트를 시작했는가** (비즈니스 동기)
- **어떤 문제들이 있었는가** (도전 과제 정의)
- **어떤 순서로 접근했는가** (작업 프로세스/사고 과정)
- **그래서 실제로 무엇을 했는가** (작업 내역 요약)

면접관이 보고 싶은 것은 "이 사람의 **사고 과정**"이다. Redis를 썼다는 사실이 아니라, **왜** Redis를 쓸 수밖에 없었고, 다른 선택지는 뭐가 있었으며, 어떤 근거로 결정했는가.

### 구체적 개선 방향: 앞단 4단계 구조

**[1단계] 문제 정의 — "왜 이 프로젝트인가"**

현재 README에 완전히 빠져 있는 부분이다. 포트폴리오 PDF 2페이지에 있는 내용("MZ 1인가구 750만 시대, 주거 선택 시 정보 부재")을 README 앞단으로 끌어와야 한다.

포함해야 할 내용:
- 사회적 배경: 1인가구 증가 추세와 주거 정보 부재 문제
- 기존 서비스의 한계: 단순 가격 필터링만 제공, 안전성·편의성 통합 분석 부재
- 이 프로젝트의 목표: 범죄율, 편의시설, 가격을 통합 분석하여 개인 맞춤형 추천 제공

**[2단계] 기술 도전 과제 — "무엇이 어려웠는가"**

포트폴리오 PDF 2페이지의 3가지 Challenge를 README에 녹여야 한다. 단, "Challenge 1, 2, 3"이라는 포트폴리오 형식이 아니라, **서사적 흐름**으로 전달해야 한다:

- 25개 자치구 × 수만 건 매물 × 3개 우선순위 조건 → 실시간 다중 조건 필터링을 DB 조회 없이 처리해야 하는 문제
- Kakao Map API 15종 카테고리 순차 호출 → 외부 API 호출이 전체 응답 시간의 92.2%를 차지하는 문제
- N+1 패턴으로 인한 SQL 쿼리 폭발 → DBMS 파싱 비용과 Connection Pool 경합의 이중 병목(※ "병목"이 아니라 "이중 비효율" 또는 "성능 저하 원인"으로 표현 — 피드백 #6 참조)

**[3단계] 작업 프로세스 — "어떤 순서로 접근했는가"**

이 부분이 면접관에게 **사고력과 체계성**을 보여주는 핵심이다. 구현 결과만 나열하는 것이 아니라, "이런 순서로 분석하고 의사결정했다"는 프로세스를 보여줘야 한다.

예시 구조:
1. 데이터 수집 파이프라인 안정화 (배치/실시간 분리로 외부 API 의존성 격리)
2. Redis 기반 인메모리 검색 엔진 구축 (실시간 추천 응답 속도 확보)
3. 2단계 폴백 시스템으로 "결과 없음" 최소화 (사용자 경험 개선)
4. Oracle V$SQL 기반 정량 분석으로 쿼리 효율화 단계적 적용
5. OOM 위험 해소를 위한 청크 처리 전환

**[4단계] 핵심 성과 요약 — "실제로 무엇을 달성했는가"**

앞서 제안한 성과 테이블과 연결되지만, 여기서는 각 작업의 before/after를 좀 더 서사적으로 보여준다. 트러블슈팅 상세는 후반에 배치하되, 여기서는 "한 줄 요약 + 핵심 수치"만 제시한다.

## 피드백 5: 후반부 — 실제 구현사항의 구체적이고 심층적인 기술

### 문제 진단

현재 README의 기술 서술은 아이러니하게도 **"넓지만 얕다"**. 많은 기술을 나열하지만, 각각의 깊이가 부족하다.

예를 들어:
- Redis Sorted Set 선택에 대해 "O(log N) 범위 검색 특성을 활용하여"라고만 되어 있고, **왜 RDB B-Tree 인덱스가 아닌 Redis인지**, **HashMap이나 일반 Set 대비 이점이 무엇인지** 근거가 없다.
- N+1 해결에 대해 "Bulk Fetch 전환"이라고만 되어 있고, **JPA @EntityGraph나 Fetch Join을 왜 쓰지 않았는지**, **Hibernate @BatchSize는 왜 검토하지 않았는지** 대안 분석이 없다.
- IN절 Chunking에서 Chunk 61을 선택했다는 결과만 있고, **Oracle GLOBAL TEMPORARY TABLE이나 COLLECTION TYPE 바인딩 등 다른 우회 방법을 왜 선택하지 않았는지** 설명이 없다.

### 구체적 개선 방향

후반부 각 기술 구현 항목에는 반드시 다음 3가지가 포함되어야 한다:

**① 기술 선택 근거 (Why)**
- "왜 이 기술/패턴을 선택했는가"
- 선택 시점의 요구사항과 제약조건 명시

**② 대안 검토 (What Else)**
- 검토한 대안 최소 2개와 각각의 트레이드오프
- 왜 해당 대안을 선택하지 않았는지

**③ 정량적 근거 (How Much)**
- 측정 환경 명시 (로컬 Oracle 19c + Redis 7.0, 5만 건 데이터 기준 등)
- before/after 수치와 측정 방법

예시 — N+1 해결 부분:

> **검토한 대안:**
> - JPA @EntityGraph: Lazy → Eager 전환으로 N+1 해소 가능하나, 25개 자치구 전체 순회 시 불필요한 데이터까지 로드되어 메모리 효율 저하
> - Hibernate @BatchSize: IN절 자동 생성하나, Chunk 크기를 동적으로 제어할 수 없어 Oracle의 IN절 1000개 제한과 Library Cache 재사용률 최적화에 부적합
> - 선택: 수동 Bulk Fetch + IN절 Chunk 크기 고정 → V$SQL 분석으로 Soft Parse 비율 88.5% 달성 확인

## 피드백 6: 기존 피드백과의 통합 — 추가 적용 사항

과거 피드백에서 지적된 항목 중 현재 README에 여전히 미반영된 사항들:

### 6-1. "병목" 표현 전면 수정

"병목(bottleneck)"은 시스템 전체 처리량을 제한하는 구간을 의미한다. 로컬 환경에서 동시 사용자 없이 단일 요청을 테스트한 상황에서 이 표현은 기술적으로 부정확하다. "쿼리 효율화", "성능 개선", "구조적 비효율 해소" 등으로 대체해야 한다.

현재 README에서 수정 필요 지점:
- 라인 97: "응답 속도가 느렸습니다" → 환경과 측정 기준 명시 필요
- 포트폴리오 PDF와 일관성 유지 (포트폴리오도 동시 수정 대상)

### 6-2. 1차/2차/3차 프로젝트 분리 제거

라인 74~87의 "1차 프로젝트: UI 개발", "2차 프로젝트: Servlet 기반", "3차 프로젝트: Spring Boot 전환"은 **프로젝트 내부 이터레이션을 별도 프로젝트로 분리해 놓은 것**이다. 면접관 입장에서 이것은:
- "이게 프로젝트가 3개인 건가?" 혼란 유발
- 본인 역할이 3개 섹션에 분산되어 핵심 기여도 파악 어려움

**단일 프로젝트로 통합하고, 역할 분담을 한 테이블로 정리해야 한다.**

### 6-3. 테스트 환경 투명 명시

성능 수치를 제시하는 모든 곳에 테스트 환경을 명시해야 한다. "응답 시간 72.3% 단축"이라고만 쓰면 면접관이 "배포 환경에서도 같은 결과인가?"라고 물었을 때 답변이 불가능하다.

명시 형태 예시:
> 측정 환경: 로컬 Oracle 19c + Redis 7.0 standalone, 전세 매물 약 5만 건 기준, 단일 스레드 순차 요청

### 6-4. 배포 URL 표현 수정

현재: "배포 URL [ 임시 배포 중지 ] - 사유 : 정부의 공공데이터 API 서버 점검"

이 표현은 부정적 인상을 준다. 배포가 중지된 상태라면, 데모 GIF나 스크린샷으로 대체하고 배포 관련 언급은 최소화하는 것이 낫다.
