## 1. 개요

### 1.1. 문서 목적

본 문서는 'Wherehouse 사용자 매물 등록 시스템' 프로젝트에서 구현해야 할 모든 기능적 요구사항을 상세히 정의한다. 기존 배치 단일 소스(국토교통부 API, 월 1회)에 의존하는 매물 데이터 구조에 사용자 직접 등록 경로를 추가하여, 핵심 엔티티(매물)에 대한 사용자 쓰기가 기존 시스템의 전 계층을 관통하는 비즈니스 시나리오를 구축한다. 각 기능은 기능 개요, 입출력 데이터 모델 또는 사용자 시나리오, 기능 요구사항으로 구성되어 명확한 구현 방향을 제시한다.

### 1.2. 역할 분담 설계

본 프로젝트는 기존 시스템의 'CQRS(Command and Query Responsibility Segregation)' 원칙에 따라 영속 저장소(Write)와 캐시 저장소(Read)의 역할을 다음과 같이 정의한다.

| 구분 | 영속 저장소 (Oracle RDB) | 캐시 저장소 (Redis) |
|------|------------------------|-------------------|
| **핵심 책임** | 데이터의 원본 보관 및 트랜잭션 무결성 보장 (Source of Truth) | 추천 검색 및 실시간 조회를 위한 고속 데이터 제공 (Read Cache) |
| **쓰기 작업** | - 매물 등록/수정/상태변경의 원본 처리<br>- 배치 매물 UPSERT<br>- 리뷰 CUD + 통계 갱신 | - 영속 저장소 커밋 후 동기화 대상<br>- 직접 쓰기 없음 (파생 데이터) |
| **읽기 작업** | - 매물 전체 목록 조회 (페이지네이션)<br>- 매물 상세 조회<br>- 리뷰 목록 조회 | - 추천 검색 (Sorted Set 교집합)<br>- 매물 상세 캐시 (Hash)<br>- 정규화 범위 (bounds Hash) |
| **데이터 정합성** | - ACID 트랜잭션으로 원자성 보장<br>- FK 참조 무결성 강제 | - 영속 저장소 커밋 후 동기화로 최종 일관성 유지<br>- 부분 실패 시 보상 처리 필요 |

**설계 의의:** 사용자 매물 등록 경로가 추가되면서 쓰기 작업의 복잡도가 증가한다. 기존에는 리뷰 통계 Hash 1개만 동기화하면 됐으나, 매물 등록은 Hash + Sorted Set 복수 + bounds Hash를 동시에 갱신해야 한다. 이 변화는 기존 CQRS 아키텍처의 동기화 전략을 확장하는 계기가 되며, 부분 실패 처리라는 새로운 설계 과제를 생성한다.

## 2. 기능 요구사항 명세

### F001 - 매물 등록

#### 1. 기능 개요

사용자가 매물 정보를 직접 입력하여 시스템에 신규 매물을 등록하는 기능이다. 등록된 매물은 기존 배치 매물과 동일한 구조로 저장되며, 추천 검색 결과에 즉시 반영된다. 기존 시스템에서 배치(월 1회)에만 의존하던 매물 데이터 갱신을 사용자 행위로 확장하여, 실시간에 가까운 매물 정보 갱신을 가능하게 한다.

매물 식별자는 기존 배치 시스템과 동일한 MD5 해시 전략(불변 속성 5개: 시군구코드, 지번, 아파트명, 층, 전용면적 조합)을 사용하여, 사용자 등록 매물과 배치 매물 간의 동일 매물 판별이 가능하다.

#### 2. 기능 요구사항

- **요구사항 1:** 사용자는 임대 유형(전세/월세)을 선택하고, 아파트명, 주소(시군구+법정동+지번), 층수, 전용면적, 가격 정보를 입력하여 매물을 등록할 수 있어야 한다. 전세와 월세는 등록 경로가 분리되어 있으며, 임대 유형에 따라 요구되는 가격 필드가 다르다(전세: 전세금, 월세: 보증금+월세금).
- **요구사항 2:** 시스템은 입력된 불변 속성 5개(시군구코드, 지번, 아파트명, 층, 전용면적)로 MD5 해시를 생성하여 매물 식별자로 사용해야 한다.
- **요구사항 3:** 등록된 매물은 추천 검색 결과에 즉시 반영되어야 한다. 등록 직후 다른 사용자가 추천 검색을 수행했을 때 해당 매물이 결과에 포함되어야 한다.
- **요구사항 4:** 매물 데이터의 출처(사용자 등록)와 등록자 정보가 함께 저장되어야 한다.
- **요구사항 5:** 전용면적(㎡)은 평수로 자동 변환되어 저장 및 표시되어야 한다.
- **요구사항 6:** 인증된 사용자만 매물을 등록할 수 있어야 한다.
- **제약사항:** 동일한 MD5 해시를 가진 매물이 이미 존재하는 경우 중복 등록이 불가하다. 중복 감지 처리는 F006에서 정의한다.

### F002 - 매물 수정

#### 1. 기능 개요

매물 등록자가 본인이 등록한 매물의 가변 속성(가격, 계약일자 등)을 수정하는 기능이다. 단, 구현 시 요구사항이 변경되어 인증된 모든 사용자가 사용자 등록 매물의 가변 속성을 수정할 수 있도록 완화되었다. 불변 속성(시군구코드, 지번, 아파트명, 층, 전용면적)은 매물 식별자(MD5 해시) 생성의 기반이므로 수정할 수 없다. 수정된 가격이 해당 지역구의 기존 가격 범위(bounds)를 벗어나는 경우, 정규화 기준 변경에 의한 연쇄 점수 재계산이 발생할 수 있다(F009 참조).

#### 2. 사용자 시나리오

본 기능의 사용자 시나리오는 **수정 요청 → 상태 검증 → 데이터 갱신 → 검색 반영**의 4단계로 구성된다.

**2.1. 수정 요청**
사용자가 추천 결과 화면 또는 매물 전체 목록 페이지에서 매물의 "수정" 버튼을 클릭하고, 가변 속성(가격 등)을 변경하여 제출한다.

**2.2. 상태 검증**
시스템은 매물의 현재 상태가 ACTIVE인지 검증한다. 종료 상태(COMPLETED/DELETED) 매물은 수정을 거부한다. 등록자 본인 여부는 검증하지 않는다.

**2.3. 데이터 갱신**
상태 검증 통과 후, 영속 저장소의 매물 정보를 갱신하고 검색 인덱스를 동기화한다(F008 참조).

**2.4. 검색 반영**
수정된 가격이 검색 인덱스에 반영되어, 이후 추천 검색 시 변경된 가격 기준으로 매물이 노출된다.

#### 3. 기능 요구사항

- **요구사항 1:** 인증된 모든 사용자가 ACTIVE 상태의 매물을 수정할 수 있어야 한다. 종료 상태(COMPLETED/DELETED)의 매물은 수정할 수 없다.
- **요구사항 2:** 수정 가능한 항목은 가변 속성(전세금/보증금, 월세금, 건축연도, 계약일자)으로 한정한다. 전세와 월세 매물은 수정 경로가 분리되어 있으며, 임대 유형에 해당하지 않는 가격 필드는 수정 요청에 포함할 수 없다.
- **요구사항 3:** 불변 속성(시군구코드, 지번, 아파트명, 층, 전용면적)은 수정할 수 없다. 불변 속성을 변경하려면 기존 매물을 삭제하고 새로 등록해야 한다.
- **요구사항 4:** 수정된 내용은 추천 검색 결과에 즉시 반영되어야 한다.
- **요구사항 5:** 수정 이력(수정 일시)이 기록되어야 한다.
- **제약사항:** 가격 수정 시 해당 지역구의 정규화 경계값이 확장될 수 있다(F009 참조). 추천 점수는 조회 시점에 갱신된 경계값을 기준으로 실시간 계산되므로, 별도의 점수 재계산 작업은 발생하지 않는다.

### F003 - 매물 상태 변경

#### 1. 기능 개요

매물 등록자가 본인 매물의 상태를 "거래완료"로 변경하는 기능이다. 거래완료된 매물은 추천 검색 결과에서 제외되지만, 데이터 자체는 논리 삭제(Soft Delete) 방식으로 보존된다. 해당 매물에 작성된 리뷰도 유지된다.

사용자가 추천 결과를 조회한 시점에는 활성 상태였으나, 이후 등록자가 거래완료로 변경한 경우, 다른 사용자가 해당 매물을 클릭했을 때 이미 거래완료 상태일 수 있다. 이 시나리오에 대한 안내 처리가 필요하다.

#### 2. 사용자 시나리오

본 기능의 사용자 시나리오는 **상태 변경 요청 → 권한 검증 → 확인 다이얼로그 → 상태 변경 처리 → 검색 제외**의 5단계로 구성된다.

**2.1. 상태 변경 요청**
사용자가 추천 결과 화면 또는 매물 전체 목록 페이지에서 본인 등록 매물의 "거래완료" 또는 "삭제" 버튼을 클릭한다.

**2.2. 권한 검증**
시스템은 현재 로그인 사용자와 매물 등록자의 일치 여부를 검증한다.

**2.3. 확인 다이얼로그**
"거래완료로 변경하시겠습니까? 변경 후 추천 검색에서 제외됩니다." 확인 다이얼로그를 표시한다.

**2.4. 상태 변경 처리**
사용자 확인 후, 매물 상태를 활성에서 거래완료로 변경한다. 영속 저장소에서는 논리 삭제로 처리하여 데이터를 보존한다.

**2.5. 검색 제외**
해당 매물을 검색 인덱스에서 제거한다(F008 참조).

#### 3. 기능 요구사항

- **요구사항 1:** 등록자 본인만 매물 상태를 변경할 수 있어야 한다.
- **요구사항 2:** 허용되는 상태 변경은 활성 → 거래완료, 활성 → 삭제이다. 역방향 변경은 허용하지 않는다.
- **요구사항 3:** 거래완료/삭제 상태의 매물은 추천 검색 결과에서 제외되어야 한다.
- **요구사항 4:** 거래완료/삭제된 매물의 데이터는 논리 삭제로 보존되어야 한다. 해당 매물에 작성된 리뷰도 유지된다.
- **요구사항 5:** 다른 사용자가 이미 거래완료된 매물에 접근하는 경우 "거래완료된 매물입니다" 안내를 표시해야 한다.
- **요구사항 6:** 상태 변경 전 확인 다이얼로그를 통해 사용자의 의도를 재확인해야 한다.
- **제약사항:** 상태 변경 시 검색 인덱스 제거가 필요하다(F008 참조). 현재 경계값은 확장만 처리하며, 매물 제거에 의한 경계값 축소 재산출은 F009 범위 외이다.

### F004 - 매물 전체 목록 조회 및 검색

#### 1. 기능 개요

기존 주거지 추천 서비스와 별도로, 등록된 전체 매물을 게시판 형태로 조회하고 검색할 수 있는 신규 페이지를 제공하는 기능이다. 기존 리뷰 게시판과 동일한 UI 패턴(페이지네이션 + 검색)을 활용한다. 배치 매물과 사용자 등록 매물이 구분 없이 통합 노출되며, 데이터 출처 뱃지로 구분한다.

#### 2. 기능 요구사항

- **요구사항 1:** 배치 매물과 사용자 등록 매물을 통합하여 페이지네이션으로 제공해야 한다.
- **요구사항 2:** 아파트명 또는 지역구명으로 매물을 검색할 수 있어야 한다.
- **요구사항 3:** 각 매물의 데이터 출처(배치/사용자 등록/병합)를 식별할 수 있어야 한다.
- **요구사항 4:** 각 매물의 현재 상태(활성/거래완료)를 표시해야 한다.
- **요구사항 5:** "수정" 버튼은 인증된 모든 사용자에게 표시한다(F002 권한 정책 반영). "거래완료"/"삭제" 버튼은 본인 등록 매물에 한해 표시해야 한다.
- **요구사항 6:** 최신순, 가격 높은순, 가격 낮은순, 면적 넓은순, 면적 좁은순으로 정렬할 수 있어야 한다.
- **요구사항 7:** 임대 유형(전세/월세), 데이터 출처(배치/사용자/병합), 매물 상태(활성/거래완료)로 필터링할 수 있어야 한다.
- **제약사항:** 인증되지 않은 사용자도 목록 조회는 가능하나, 수정/거래완료/삭제 버튼은 노출되지 않는다.

### F005 - 추천 결과 화면 매물 정보 확장

#### 1. 기능 개요

기존 주거지 추천 서비스의 결과 화면에 사용자 매물 등록 시스템 도입에 따른 추가 정보를 표시하는 기능이다. 각 매물 카드에 데이터 출처 뱃지, 매물 상태, 본인 매물에 대한 수정/거래완료 버튼을 추가한다. 기존 추천 로직 자체의 변경 없이, 결과 표시 계층에서만 정보를 확장한다.

#### 2. 기능 요구사항

- **요구사항 1:** 추천 결과의 각 매물 카드에 데이터 출처를 뱃지로 표시해야 한다. 배치 매물은 "국토교통부 실거래 데이터", 사용자 등록 매물은 "사용자 등록", 병합 매물은 "국토교통부 + 사용자 등록"으로 구분한다.
- **요구사항 2:** 인증된 모든 사용자에게 ACTIVE 상태 매물에 대한 "수정" 버튼을 매물 카드 내에 표시해야 한다. "거래완료" 및 "삭제" 버튼은 본인이 등록한 매물에 한해 표시해야 한다.
- **요구사항 3:** 추천 결과 조회 시점에는 활성 상태였으나 이후 등록자가 거래완료로 변경한 경우, 해당 매물을 클릭했을 때 "거래완료된 매물입니다" 안내를 표시하고 다른 매물 탐색을 유도해야 한다.
- **요구사항 4:** 기존 추천 결과의 매물 정보(아파트명, 가격, 면적, 층수, 건축연도, 추천 점수, 리뷰 개수, 평균 별점)는 그대로 유지해야 한다.
- **제약사항:** 기존 추천 로직(RecommendationService)의 핵심 비즈니스 로직(검색 필터링, 점수 계산, 정렬)은 변경하지 않는다. 단, 요구사항 1~3을 충족하기 위해 요청 시 인증 정보를 통한 요청자 식별이 필요하므로 요청 정보 포맷에 대한 수정이 수반되며, 응답 정보에는 데이터 출처, 매물 상태, 등록자 본인 여부 등의 필드가 추가되어야 한다.

### F006 - 사용자 매물 등록 시 중복 감지

#### 1. 기능 개요

매물 등록 요청 시, 입력된 불변 속성으로 생성된 MD5 해시가 이미 존재하는 매물과 동일한지를 판단하는 시스템 내부 기능이다. 동일 MD5 해시가 이미 존재하는 경우, 그 기존 매물의 데이터 출처(배치/사용자)에 따라 처리가 분기된다. 또한 복수의 사용자가 같은 물리적 매물을 거의 동시에 등록하는 경우 Race Condition이 발생할 수 있으며, 이에 대한 충돌 처리가 필요하다.

#### 2. 사용자 시나리오

본 기능의 시나리오는 네 가지로 분류된다.

**2.1. 정상 등록 (중복 없음)**
사용자가 매물을 등록한다. 시스템은 MD5 해시를 생성하고, 동일 해시가 존재하지 않음을 확인한 뒤 등록을 완료한다.

**2.2. 기존 배치 매물과 중복**
사용자가 등록하려는 매물과 동일한 MD5 해시를 가진 매물이 배치에 의해 이미 저장되어 있는 경우. 사용자에게 "이 매물은 국토교통부 실거래 데이터에 이미 등록된 매물입니다"와 같이 기존 매물의 출처를 안내하고 중복 등록을 거부한다.

**2.3. 기존 사용자 등록 매물과 중복**
사용자 A가 등록하려는 매물과 동일한 MD5 해시를 가진 매물이 사용자 B에 의해 이미 등록되어 있는 경우. 사용자 A에게 "동일한 매물이 다른 사용자에 의해 이미 등록되어 있습니다"를 안내하고 중복 등록을 거부한다.

**2.4. 복수 사용자 동시 등록 (Race Condition)**
사용자 A와 B가 같은 물리적 매물을 거의 동시에 등록한다. 동일한 MD5 해시가 생성된다. 둘 다 존재 여부를 확인한 시점에는 "존재하지 않음"이었으나, 둘 다 INSERT를 시도하면서 충돌이 발생한다. 시스템은 중복을 감지하고, 선착순으로 하나만 등록을 허용한다. 후착 사용자에게는 중복 안내를 반환한다.

#### 3. 기능 요구사항

- **요구사항 1:** 불변 속성 5개(시군구코드, 지번, 아파트명, 층, 전용면적) 조합의 MD5 해시가 이미 존재하는 경우 중복으로 판단해야 한다.
- **요구사항 2:** 중복 감지 시, 기존 매물의 데이터 출처(배치/사용자)에 따라 사용자에게 구분된 안내 메시지를 반환해야 한다.
- **요구사항 3:** 복수 사용자가 동일 매물을 동시에 등록하는 경우, 정확히 하나의 등록만 성공해야 한다. 데이터 정합성이 보장되어야 한다.
- **요구사항 4:** 중복 감지 과정에서 선착 사용자의 등록 응답 시간이 과도하게 지연되어서는 안 된다.
- **제약사항:** 동시 등록 충돌 처리의 구체적 전략(락 방식, 수준, 폴백)은 구현 과정에서의 기술적 의사결정 사항이다.

### F007 - 배치-사용자 데이터 충돌 처리

#### 1. 기능 개요

배치 실행 시, 국토교통부 API에서 가져온 매물의 MD5 해시가 기존 사용자 등록 매물의 해시와 일치하는 경우 발생하는 데이터 충돌을 처리하는 시스템 내부 기능이다. 사용자가 "전세 1.8억"으로 등록한 매물에 대해 배치가 "전세 2억"을 가져오는 등, 동일 매물에 대해 서로 다른 가격 정보가 존재하는 상황을 해소한다.

이 충돌의 해소 정책으로 임계값 기반 하이브리드 머지 전략이 확정되었다. 가격 차이율을 기준으로 임계값 이내이면 최신성을, 초과이면 신뢰성(국토부 공식 데이터)을 우선하여 자동 확정한다.

#### 2. 사용자 시나리오

**2.1. 배치가 신규 매물을 추가 (충돌 없음)**
배치가 국토부 API에서 가져온 매물의 MD5 해시가 시스템에 존재하지 않는 경우이다. 기존 배치 동작과 동일하게 신규 UPSERT를 수행한다. 사용자 매물 등록 시스템 도입 전후로 동작 차이가 없는 정상 케이스이다.

**2.2. 충돌 해소 — 임계값 기반 하이브리드 머지 (Threshold-based Hybrid Merge)**

시스템은 사용자 등록 가격과 국토부 배치 가격 간의 차이율을 산출하고, 사전에 정의된 임계값을 기준으로 머지 전략을 분기한다.

- **가격 차이가 임계값 이내인 경우**: 두 데이터 소스 간 가격 차이가 정상 변동 범위에 해당하므로, 최신 데이터(타임스탬프 기준)를 메인 가격으로 채택한다. 사용자가 더 최근의 정확한 가격을 알고 있을 가능성을 허용하여 최신성을 우선한다.
- **가격 차이가 임계값 초과인 경우**: 가격 불일치가 정상 변동 범위를 벗어나므로, 국토부 데이터(공식 실거래 신고 기반)를 메인 가격으로 채택한다. 사용자 허위 등록 또는 데이터 오류 가능성을 고려하여 신뢰성을 우선한다.
- **모든 경우에 자동으로 결과가 확정**되므로 "보류" 상태가 발생하지 않는다. 관리자 해소 프로세스가 불필요하며, 1인 프로젝트 바운더리 안에서 완결된다.

머지 완료 후 데이터 출처를 "병합"으로 변경하고, 사용자가 등록한 원본 가격은 별도 필드에 보존한다.

임계값의 구체적 수치는 구현 단계에서 국토부 API 실데이터(동일 매물의 전월 대비 가격 변동률 분포)를 기반으로 정상 변동 범위를 실측하여 확정한다.

**2.3. 배치 매물 적재와 사용자 매물 등록이 동시 수행 (동시 쓰기)**
배치가 특정 매물을 적재하는 트랜잭션 진행 중에, 사용자가 같은 MD5의 매물을 등록하는 트랜잭션을 동시에 수행하는 경우이다. 양쪽 트랜잭션이 모두 진행 중이므로, 상대방 데이터가 아직 커밋되지 않은 상태에서 존재 여부를 확인하게 된다. 시나리오 2(시간차 충돌)와 달리 확정된 기존 데이터가 없는 상태에서 양쪽이 경쟁한다. 현실적 발생 빈도는 배치 월 1회(새벽 4시) + 사용자 수시이므로 극히 낮으나, 엣지 케이스로서 사전 식별이 필요하다.

#### 3. 기능 요구사항

- **요구사항 1:** 배치 실행 시 동일 MD5 해시를 가진 사용자 등록 매물이 존재하는 경우를 식별할 수 있어야 한다.
- **요구사항 2:** 충돌 발생 시 임계값 기반 하이브리드 머지 전략에 따라 가변 속성(가격 등)을 해소해야 한다. 가격 차이가 임계값 이내이면 최신 데이터를 채택하고, 임계값 초과이면 국토부 데이터를 채택한다. 모든 경우에 자동으로 결과가 확정되어야 하며, 별도의 수동 해소 프로세스 없이 완결되어야 한다.
- **요구사항 3:** 머지 후 데이터 출처를 "병합"으로 변경하여, 해당 매물이 양쪽 소스의 데이터를 포함하고 있음을 식별할 수 있어야 한다.
- **요구사항 4:** 사용자가 등록한 원본 가격 정보는 머지 후에도 별도로 보존되어야 한다.
- **요구사항 5:** 충돌 처리 과정에서 기존 매물에 연결된 리뷰 데이터의 참조 무결성이 유지되어야 한다.
- **요구사항 6:** 배치 매물 적재와 사용자 매물 등록이 동시에 수행되는 경우에도 데이터 정합성이 보장되어야 한다.
- **제약사항 1:** 머지 전략은 임계값 기반 하이브리드 방식으로 확정되었다. 임계값의 구체적 수치는 구현 단계에서 국토부 API 실데이터(동일 매물의 전월 대비 가격 변동률 분포)를 기반으로 정상 변동 범위를 실측하여 확정한다.
- **제약사항 2:** 시나리오 2.3(동시 쓰기)의 구체적 처리 방식은 구현 과정에서의 기술적 의사결정 사항이다.

### F008 - 검색 인덱스 동기화

#### 1. 기능 개요

매물 등록/수정/상태변경 시, 영속 저장소에 커밋된 데이터를 Redis 검색 인덱스에 반영하는 시스템 내부 기능이다. 기존 리뷰 시스템의 Write-Through 동기화는 통계 데이터 1건만 갱신하면 됐으나, 매물 등록은 매물 상세 + 가격/평수 Redis 검색 인덱스 + 정규화 범위를 동시에 갱신해야 한다. 복수의 대상을 갱신하는 과정에서 일부만 성공하는 부분 실패 상황이 발생할 수 있다.

#### 2. 기능 요구사항

- **요구사항 1:** 매물 등록 시 매물 상세, 가격 Redis 검색 인덱스, 평수 Redis 검색 인덱스, 정규화 범위를 갱신해야 한다.
- **요구사항 2:** 매물 수정 시 변경된 속성에 해당하는 Redis 검색 인덱스만 선택적으로 갱신해야 한다.
- **요구사항 3:** 매물 상태 변경(거래완료/삭제) 시 해당 매물을 Redis 검색 인덱스에서 제거해야 한다.
- **요구사항 4:** 영속 저장소 커밋과 Redis 검색 인덱스 갱신 사이의 데이터 정합성이 보장되어야 한다. 활성 상태의 매물이 영속 저장소에는 존재하지만 Redis 검색 인덱스에는 반영되지 않아 추천 검색 결과에서 누락되는 불일치 상태가 발생하면 안 된다. 단, 거래완료 또는 삭제 상태의 매물은 논리 삭제에 의해 영속 저장소에는 보존되지만 Redis 검색 인덱스에서는 제거된 상태가 정상이다.
- **요구사항 5:** 영속 저장소 트랜잭션이 실패하는 경우, 트랜잭션 롤백에 의해 Redis 동기화가 수행되지 않아야 한다. 영속 저장소에 커밋되지 않은 데이터가 Redis 검색 인덱스에 반영되는 상태가 발생하면 안 된다.
- **요구사항 6:** 영속 저장소 커밋 후 Redis 동기화가 전체 실패하는 경우, 활성 매물이 영속 저장소에만 존재하고 추천 검색에서 누락되는 불일치 상태를 해소할 수 있어야 한다.
- **요구사항 7:** 복수의 Redis 검색 인덱스 갱신 중 일부만 성공하는 부분 실패 상황에 대한 처리가 필요하다.
- **제약사항:** 요구사항 6(동기화 전체 실패) 및 요구사항 7(부분 실패)에 대한 보상 전략(롤백, 재시도, 보상 트랜잭션)의 구체적 방식은 구현 과정에서의 기술적 의사결정 사항이다.

### F009 - 정규화 경계값(bounds) 원자적 갱신

#### 1. 기능 개요

매물 등록 또는 수정으로 인해 해당 지역구의 정규화 경계값(가격/평수의 최솟값·최댓값)이 확장되는 경우, 해당 갱신이 동시 요청 환경에서도 정합성을 유지하도록 보장하는 시스템 내부 기능이다.

현재 경계값 갱신 로직은 기존 값 조회→비교→갱신을 별개의 연산으로 순차 수행하므로, 복수 사용자가 동시에 매물을 등록하면 경계값이 잘못된 값으로 덮어쓰이거나, 추천 조회 요청이 부분 갱신된 중간 상태의 경계값을 읽을 수 있다.

본 시스템의 추천 점수 계산은 조회 시점에 경계값을 읽어 실시간으로 수행되므로, 경계값 갱신 시 "전체 매물 점수 재계산"이라는 별도의 후속 작업은 존재하지 않는다. 따라서 F009의 범위는 경계값 갱신의 원자성 확보에 한정된다.

#### 2. 기능 요구사항

- **요구사항 1:** 매물 등록/수정 시 해당 매물의 가격 또는 면적이 기존 정규화 경계값을 벗어나는지 감지하고, 벗어나는 경우 해당 필드를 확장 갱신해야 한다.
- **요구사항 2:** 경계값 갱신 과정(조회→비교→갱신)이 원자적으로 수행되어야 한다. 복수 사용자가 동시에 매물을 등록할 때, 후착 갱신이 선착 갱신의 더 극단적인 값을 덮어쓰는 상황이 발생하면 안 된다.
- **요구사항 3:** 경계값의 복수 필드가 갱신되는 중간 상태에서 별도의 추천 검색 요청이 부분 갱신된 경계값을 읽는 상황이 방지되어야 한다.
- **제약사항 1:** 경계값 갱신의 원자성 확보 방식의 구체적 선택은 구현 과정에서의 기술적 의사결정 사항이다.
- **제약사항 2:** 현재 경계값은 확장만 처리하며, 매물 제거·가격 축소에 의한 경계값 재산출(축소)은 본 기능의 범위에 포함하지 않는다.
- **제약사항 3:** 추천 조회 경로의 지역구별 점수 계산 병렬화는 F009의 범위가 아니며, 별도 최적화 항목으로 정의한다.

### 기능별 상세 요구사항

**F001: 매물 등록**
- 입력 조건: 아파트명, 주소, 층수, 전용면적, 가격 정보 (필수) + 건축연도, 계약일자 (선택). 임대 유형은 전세/월세 분리된 등록 경로로 확정
- 처리 요구사항: MD5 해시 생성 → 중복 감지(F006) → 영속 저장 → 검색 인덱스 동기화(F008)
- 출력 요구사항: 매물 식별자, 등록 완료 정보 반환
- 에러 처리: 중복 매물 존재 시 안내 메시지, 필수 항목 누락 시 유효성 검증 에러

**F002: 매물 수정**
- 사용자 권한: 인증된 모든 사용자가 ACTIVE 상태 매물을 수정 가능
- 사용 시나리오: 추천 결과 또는 매물 목록에서 "수정" 클릭 → 가변 속성 변경 → 제출
- 성공 기준: 영속 저장소 갱신 + 검색 인덱스 반영 + 가격 변경 시 bounds 확장 갱신(F009)
- 실패 처리: 매물 미존재 시 404, 종료 상태 매물 수정 시 409

**F003: 매물 상태 변경**
- 사용자 권한: 등록자 본인만 상태 변경 가능
- 사용 시나리오: "거래완료" 또는 "삭제" 클릭 → 확인 다이얼로그 → 상태 변경
- 성공 기준: 매물 상태 변경 + 검색 인덱스 제거
- 실패 처리: 허용되지 않는 상태 전이 시 400, 매물 미존재 또는 삭제 상태 접근 시 404, 권한 불일치 시 403

**F004: 매물 전체 목록 조회 및 검색**
- 판단 기준: 인증 여부에 따라 수정/거래완료/삭제 버튼 노출 여부 결정. 수정 버튼은 인증 사용자 전체, 거래완료/삭제 버튼은 등록자 본인에게만 노출
- 분기 조건: 검색 키워드 유무에 따라 전체 조회 / 필터링 조회 분기. 임대 유형, 데이터 출처, 매물 상태 필터 적용 가능
- 타임아웃: 조회 응답은 일반적인 웹 페이지 응답 수준
- 재시도: 별도 재시도 정책 불필요

**F005: 추천 결과 화면 매물 정보 확장**
- 표시 형식: 데이터 출처 뱃지 (텍스트 레이블), 인증 사용자 대상 수정 버튼, 본인 매물 대상 거래완료/삭제 버튼
- 검증 방법: 기존 추천 결과의 정보가 누락 없이 유지되는지 확인
- 분기 조건: 로그인 여부에 따라 수정 버튼 노출, 로그인 사용자의 매물 여부에 따라 거래완료/삭제 버튼 노출 분기

**F006: 사용자 매물 등록 시 중복 감지**
- 판단 기준: MD5 해시(불변 속성 5개 조합) 일치 여부
- 분기 조건: 해시 미존재 → 등록 진행, 해시 존재 → 중복 안내 반환
- 처리 순서: 해시 생성 → 존재 여부 확인 → 등록 또는 거부
- 트랜잭션: 동시 등록 충돌 시 데이터 정합성 보장 필요

**F007: 배치-사용자 데이터 충돌 처리**
- 판단 기준: 배치 매물의 MD5 해시가 기존 사용자 등록 매물과 일치하는지 여부
- 처리 순서: 충돌 감지 → 가격 차이율 산출 → 임계값 기반 분기(이내: 최신 데이터 채택 / 초과: 국토부 데이터 채택) → 데이터 출처 "병합"으로 변경 → 사용자 원본 가격 별도 보존 → 검색 인덱스 갱신
- 트랜잭션: 머지 과정에서 기존 리뷰 참조 무결성 유지 필요
- 로깅: 충돌 발생 건수, 가격 차이율, 적용된 분기(최신성/신뢰성), 머지 결과 기록

**F008: 검색 인덱스 동기화**
- 처리 순서: 영속 저장소 커밋 → Redis 검색 인덱스 복수 키 갱신
- 트랜잭션: 영속 저장소와 Redis 검색 인덱스 간 정합성 보장
- 외부 연동: Redis 검색 인덱스에 대한 복수 키 갱신
- 로깅: 동기화 대상 키, 각 키별 성공/실패 여부 기록

**F009: 정규화 경계값(bounds) 원자적 갱신**
- 처리 순서: 매물 가격/면적과 현재 경계값 비교 → 확장 필요 시 원자적 갱신 → 변경 여부 반환
- 정합성: 경계값 갱신의 조회→비교→갱신 시퀀스가 원자적으로 수행되어야 함. 동시 갱신 시 값 소실 방지 필요
- 범위 제한: 경계값 확장만 처리. 축소(매물 제거에 의한 재산출)는 미포함. 추천 조회 경로 병렬화는 별도 항목
- 로깅: 경계값 변경 발생 여부, 변경 전/후 값 기록

## 4. 비기능 요구사항

### 4.1 성능 요구사항

- **응답 시간**: 매물 등록/수정/상태변경 API는 Redis 검색 인덱스 동기화 포함하여 사용자 체감 응답 시간 내에 완료되어야 한다.
- **동시 처리**: 동일 매물에 대한 동시 등록 요청 시 데이터 정합성이 보장되어야 한다(F006).
- **bounds 갱신 원자성**: 동시 매물 등록 시 bounds 갱신이 원자적으로 수행되어 값 소실이 발생하지 않아야 한다(F009).

### 4.2 보안 요구사항

- **인증**: 매물 등록/수정/상태변경은 인증된 사용자만 수행 가능하다.
- **권한 관리**: 매물 수정은 인증된 모든 사용자에게 허용된다. 매물 상태변경(거래완료/삭제)은 등록자 본인에게만 허용된다.
- **데이터 무결성**: 사용자 입력 데이터에 대한 유효성 검증(필수 항목, 값 범위, 형식)이 수행되어야 한다.

### 4.3 데이터 정합성 요구사항

- **영속 저장소 - Redis 검색 인덱스 일관성**: 활성 상태의 매물이 영속 저장소에 커밋된 경우 Redis 검색 인덱스에도 반영되어야 한다. 활성 매물이 영속 저장소에만 존재하고 Redis 검색 인덱스에서 누락되는 불일치 상태가 발생하면 안 된다. 단, 거래완료 또는 삭제 상태의 매물은 논리 삭제에 의해 영속 저장소에는 보존되지만 Redis 검색 인덱스에서는 제거된 상태가 정상이다.
- **배치 - 사용자 데이터 공존**: 동일 매물에 대한 두 데이터 소스의 충돌이 정의된 정책에 따라 해소되어야 한다.
- **리뷰 참조 무결성**: 매물 상태 변경(거래완료/삭제) 시 해당 매물에 연결된 리뷰 데이터가 유지되어야 한다.
