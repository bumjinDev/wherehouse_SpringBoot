# Wherehouse 복합 인덱스 설계 결정 기록

**문서 목적**: Wherehouse 고도화 설계 명세서 8.1.4에 명시된 복합 인덱스(`IDX_PROP_*_STATUS` on `STATUS, DATA_SOURCE`)의 설계 근거를 기록하고, 추후 실측·검증이 필요한 지점을 식별한다.

**문서 상태**: 설계 논리까지만 정리된 초안. 실측 검증과 벤치마크는 추후 공부 후 진행 예정.

**작성일**: 2026-04-22
**작성자**: 정범진

---

## 1. 논의 배경

설계 명세서 v2.1 섹션 8.1.4에서 매물 테이블에 추가할 신규 인덱스 4개가 정의되었다. 이 중 두 개는 단일 인덱스(`REGISTERED_USER_ID`), 나머지 두 개는 복합 인덱스(`STATUS, DATA_SOURCE`) 구조이다. DDL 작성 과정에서 "왜 복합 인덱스로 설계했는가" 라는 의문이 제기되었고, 이 결정이 단순한 관례가 아니라 **F004 매물 목록 조회 API의 쿼리 패턴과 Oracle B-Tree 인덱스의 동작 원리에 기반한 의사결정** 임을 확인했다.

본 기록은 이 결정을 문서화하여 추후 면접 대응 시 설계 근거로 활용하고, 실측을 통해 설계의 타당성을 정량적으로 검증하기 위한 앵커 역할을 한다.

## 2. 현재 설계안의 구조

### 2.1 대상 인덱스

| 인덱스명 | 대상 테이블 | 컬럼 구성 | 타입 | 용도 |
|---------|-----------|----------|------|------|
| `IDX_PROP_CHARTER_STATUS` | `PROPERTIES_CHARTER` | `(STATUS, DATA_SOURCE)` | B-Tree 복합 | F004 상태·출처 필터 |
| `IDX_PROP_MONTHLY_STATUS` | `PROPERTIES_MONTHLY` | `(STATUS, DATA_SOURCE)` | B-Tree 복합 | 동일 |

### 2.2 대상 쿼리 패턴 (F004 매물 목록 조회)

설계 명세서 섹션 7.5.1에 정의된 Query Parameter 기준, F004 API는 다음 조합의 필터링을 지원한다.

| Query Parameter | 기본값 | 발생 빈도 추정 |
|----------------|-------|--------------|
| `status` | `ACTIVE` (항상 필터에 포함) | 100% |
| `dataSource` | 미지정 (선택적) | 일부 요청 |
| `district` | 미지정 | 일부 요청 |
| `leaseType` | 미지정 | 일부 요청 |

즉 RDB로 내려가는 쿼리의 WHERE 절 패턴은 다음 세 가지로 요약된다.

```sql
-- 패턴 1: status만 필터링 (가장 빈번)
WHERE STATUS = 'ACTIVE' AND ...

-- 패턴 2: status + dataSource 모두 필터링
WHERE STATUS = 'ACTIVE' AND DATA_SOURCE = 'USER' AND ...

-- 패턴 3: dataSource만 필터링 (거의 발생하지 않음)
WHERE DATA_SOURCE = 'USER' AND ...
```

## 3. 설계 결정의 근거

> **⚠ 본 섹션의 지식 출처 및 검증 상태 안내**
>
> 본 섹션(3.1~3.3)의 내용은 설계 결정의 근거를 AI(Claude)에게 질의한 답변을 토대로 정리된 것이다. 본 문서 작성 시점에서 작성자 본인이 직접 학습·검증을 완료한 지식이 아니며, 다음 항목은 **추후 작성자 본인이 별도로 학습하고 실측을 통해 검증해야 하는 범주** 에 속한다.
>
> - B-Tree 인덱스의 카디널리티(Cardinality)와 선택도(Selectivity) 개념
> - Oracle 옵티마이저의 `INDEX BITMAP CONVERSION` 동작 원리
> - 복합 인덱스의 Leading Column 규칙 (맨 앞 컬럼부터 순차 사용)
> - Full Table Scan 과 Index Range Scan 중 옵티마이저가 선택하는 임계 기준
> - 각 쿼리 패턴별 인덱스 활용 가능성 판단 로직
>
> 따라서 본 섹션의 논리 체인은 "현재 작성자가 이해한 수준의 가설" 이며, 실제 Oracle 환경에서의 동작은 EXPLAIN PLAN 실측과 공식 문서·전문 서적(예: 『Oracle Performance Tuning』· 『SQL 튜닝의 시작』·『친절한 SQL 튜닝』 등) 학습을 통해 재검증되어야 한다. 면접 자리에서 본 섹션의 내용을 그대로 인용하는 것은 실측 전까지는 지양하며, 학습·실측 완료 이후 재작성된 버전을 기준으로 활용한다.

### 3.1 왜 단일 인덱스 2개가 아닌 복합 인덱스 1개인가

**단일 인덱스 대안** (가정): `IDX_STATUS` on `STATUS`, `IDX_DATA_SOURCE` on `DATA_SOURCE` 두 개를 분리 생성.

이 대안의 문제점은 두 컬럼 모두 **매우 낮은 카디널리티** 를 가진다는 데 있다.

- `STATUS`: `ACTIVE`·`COMPLETED`·`DELETED` 3개 값
- `DATA_SOURCE`: `BATCH`·`USER`·`MERGED` 3개 값

단일 인덱스로 만들면 각 인덱스 항목이 전체 테이블의 약 1/3을 가리킨다. Oracle 옵티마이저가 이런 낮은 선택도의 인덱스를 만났을 때의 선택지는 다음 두 가지이다.

1. 둘 중 한쪽만 사용 → 나머지 조건은 테이블 필터링 (부분 인덱스 스캔)
2. `INDEX BITMAP CONVERSION` 으로 두 인덱스를 합성 → 추가 CPU 비용 발생
3. 선택도가 여전히 부족하다고 판단 → Full Table Scan 선택

세 경로 모두 최적에 미치지 못한다.

**복합 인덱스의 이점**: `(STATUS='ACTIVE', DATA_SOURCE='USER')` 조합은 이론상 전체의 1/9 영역만 가리킨다(선택도 약 11%). 한 번의 인덱스 스캔으로 두 조건을 동시에 해소하며 합성 비용도 발생하지 않는다.

### 3.2 왜 컬럼 순서가 `(STATUS, DATA_SOURCE)` 인가 — Leading Column 규칙

B-Tree 복합 인덱스는 **맨 앞 컬럼부터 순차적으로만 사용 가능** 하다. 이를 Leading Column 규칙이라고 한다.

`(STATUS, DATA_SOURCE)` 인덱스의 활용 가능성은 다음과 같다.

| 쿼리 조건 | 인덱스 사용 |
|----------|-----------|
| `WHERE STATUS = ?` | 가능 (leading column 사용) |
| `WHERE STATUS = ? AND DATA_SOURCE = ?` | 가능 (두 컬럼 모두 사용) |
| `WHERE DATA_SOURCE = ?` 만 | **불가능** (leading column 건너뜀) |

F004 API의 실제 쿼리 패턴(섹션 2.2)을 보면 `STATUS`는 항상 필터에 포함되고 `DATA_SOURCE`는 선택적이다. 따라서 `STATUS` 를 leading column 으로 두면 패턴 1·패턴 2 두 경우 모두 인덱스를 활용할 수 있다.

만약 컬럼 순서를 역전해 `(DATA_SOURCE, STATUS)` 로 설계했다면, 패턴 1(빈도 100%)에서 인덱스가 전혀 사용되지 않는 치명적 결함이 발생한다.

### 3.3 설계 결정의 논리 체인 요약

```
F004 쿼리 패턴 분석
  └─ STATUS 100% 필터 포함, DATA_SOURCE 일부 요청만 포함
       └─ 두 컬럼이 함께 필터되는 경우가 지배적
            └─ 각각 카디널리티가 3으로 매우 낮음
                 └─ 단일 인덱스 2개는 선택도 부족으로 옵티마이저가 회피
                      └─ 복합 인덱스로 결합 → 선택도 1/9로 개선
                           └─ Leading Column 규칙상 STATUS를 앞에 배치
                                └─ (STATUS, DATA_SOURCE) 확정
```

## 4. 이 설계의 약점과 한계

설계 결정의 근거는 위와 같으나, 이 설계에는 **카디널리티 본질적 한계** 라는 약점이 있다. 이 지점이 본 문서를 기록 목적으로 남기는 핵심 이유이다.

### 4.1 카디널리티 한계

일반적으로 B-Tree 인덱스는 **선택도가 5% 이하일 때 유의미하게 작동** 한다고 알려져 있다. 본 인덱스의 선택도는 다음과 같이 추정된다.

- `STATUS` 단독: 약 33% (3개 값 균등 분포 가정 시)
- `(STATUS, DATA_SOURCE)` 조합: 약 11% (3×3=9 조합 균등 분포 가정 시)

11%는 여전히 5% 임계점을 초과한다. **실제 운영 환경에서 Oracle 옵티마이저가 이 복합 인덱스를 선택할지, 혹은 여전히 Full Table Scan 을 선호할지는 불확실하다.**

게다가 실제 값 분포는 균등하지 않을 가능성이 높다. 예를 들어:
- 초기 운영에서는 대다수가 `STATUS='ACTIVE'`, `DATA_SOURCE='BATCH'` 조합으로 쏠림 → 선택도가 악화
- `DELETED` 상태는 소수 → 선택도 개선 효과 있음
- `USER`·`MERGED` 출처 증가 추이에 따라 변동

### 4.2 설계서가 그럼에도 이 설계를 택한 추정 근거

설계 명세서 v2.1은 이 인덱스의 카디널리티 한계를 명시하지 않았다. 그럼에도 복합 B-Tree 인덱스를 선택한 이유는 다음으로 추정된다.

1. **Bitmap Index 의 회피**: 카디널리티가 극히 낮은 컬럼에는 Bitmap Index 가 이론상 더 적합하나, Bitmap Index 는 갱신 시 락 경합이 심해 OLTP(자주 CUD 가 발생하는 테이블)에는 부적합하다. 매물 테이블은 사용자 CUD 가 수시로 들어오는 OLTP 테이블이므로 B-Tree 가 안전한 선택이다.
2. **옵티마이저에게 선택지 제공**: 다른 필터(`district`·`keyword` 등) 와 결합된 쿼리에서 옵티마이저가 이 인덱스를 **보조적으로라도 사용할 가능성** 을 열어 두려는 목적. 인덱스가 있어도 안 쓰이는 경우는 있지만, 없으면 절대 못 쓴다.
3. **향후 데이터 증가에 대비**: 현재는 배치 데이터 위주이지만 사용자 매물이 쌓이면 `DATA_SOURCE` 값 분포가 다변화될 가능성. 장기적으로 선택도가 개선될 수 있다.

### 4.3 실효성에 대한 정직한 진단

이 인덱스는 **단독으로는 성능 개선 효과가 크지 않을 가능성이 높다**. 실제 효과를 내려면 다음 조건 중 하나가 필요하다.

- 선택도 높은 다른 컬럼(`district`, `REGISTERED_USER_ID` 등)과 결합된 쿼리에서 보조 인덱스로 활용
- 향후 데이터 분포가 균등해지면서 선택도가 자연스럽게 개선
- 옵티마이저 통계(`DBMS_STATS.GATHER_TABLE_STATS`) 가 최신으로 유지되어 정확한 Cost 계산이 이루어짐

## 5. 추후 실측 및 검증 계획

이 섹션은 추후 공부와 실측을 통해 채워 나갈 부분이다. 현재는 측정할 가치가 있는 지점과 대략적 방향만 기록한다.

### 5.1 측정할 가치가 있는 지점 (미정리, 추후 구체화 필요)

- 복합 인덱스 유/무에 따른 F004 쿼리의 EXPLAIN PLAN 차이
- 단일 인덱스 2개 vs 복합 인덱스 1개의 실제 실행 시간 비교
- 컬럼 순서 역전 (`DATA_SOURCE, STATUS`) 시의 패턴 1 쿼리 성능 저하 실측
- 데이터 규모별 (10만 건 / 100만 건) 성능 곡선
- Bitmap Index 대안과의 비교 (읽기 성능 vs 쓰기 성능 트레이드오프)
- `INDEX HINT` 로 인덱스 사용을 강제한 경우와 옵티마이저 자율 선택의 차이

### 5.2 실측 전 선행 학습이 필요한 영역

- Oracle 옵티마이저의 Cost 계산 방식 (CBO: Cost-Based Optimizer)
- `EXPLAIN PLAN` / `DBMS_XPLAN.DISPLAY` 출력 해석법
- `V$SQL_PLAN`, `V$SQL_PLAN_STATISTICS` 뷰를 통한 실행 계획 및 통계 조회
- `DBMS_STATS` 를 통한 통계 수집과 이것이 옵티마이저 결정에 미치는 영향
- Bitmap Index 의 락 메커니즘과 OLTP 부적합성의 구체적 근거
- Index Range Scan / Index Unique Scan / Index Fast Full Scan 의 차이

### 5.3 실측 환경 구성의 전제

현재 Wherehouse 의 실제 데이터는 국토부 배치 데이터만 존재하므로 `DATA_SOURCE` 가 거의 100% `BATCH` 단일 값이다. 복합 인덱스의 효과를 의미 있게 측정하려면 다음이 필요하다.

- 더미 사용자 매물 데이터 주입 (`DATA_SOURCE='USER'`)
- 일부 매물의 상태 변경 (`STATUS='COMPLETED'`·`'DELETED'`)
- 머지된 매물 시나리오 구성 (`DATA_SOURCE='MERGED'`) — F007 구현 이후
- 데이터 규모 확장을 위한 복제 (10만·100만 건 시뮬레이션)

## 6. 포트폴리오·면접 활용 시사점

### 6.1 이 주제가 중요한 이유

기존 Wherehouse 포트폴리오의 DB 관련 트러블슈팅은 "IN 절 Chunking 1건" 이 유일하다. 복합 인덱스 설계 결정은 **두 번째 DB 주제** 가 되며, 다음 면접 시 설명 자산이 된다.

- 카디널리티와 선택도 개념의 이해
- B-Tree 인덱스의 Leading Column 규칙
- Bitmap Index 와 B-Tree Index 의 트레이드오프
- 옵티마이저의 비용 기반 판단
- 설계의 약점을 스스로 인지하고 실측으로 검증하려는 태도

### 6.2 면접 답변 시 권장 구조

> **⚠ 본 절의 활용 제한**
>
> 아래 권장 답변 예시는 섹션 3 의 논리 체인을 전제로 작성되어 있다. 섹션 3 서두에 명시한 대로 해당 논리는 AI 답변을 통해 접한 지식을 정리한 것이므로, 본 예시 답변을 면접에서 실제로 사용하기 위해서는 작성자 본인이 먼저 관련 개념(카디널리티·Leading Column 규칙·옵티마이저 판단 기준)을 자기 언어로 학습하고 EXPLAIN PLAN 실측으로 주요 주장을 검증한 후 재작성해야 한다. 검증 전 단계에서 이 답변을 암기하여 사용하면 후속 질문(예: "그럼 실제로 옵티마이저가 이 인덱스를 선택했습니까?") 에 대응할 수 없다.

"설계 명세서 8.1.4 에서 매물 테이블에 `(STATUS, DATA_SOURCE)` 복합 인덱스를 추가했습니다. 이 결정의 근거는 F004 매물 목록 조회 API 의 쿼리 패턴 분석이었습니다. `STATUS` 는 항상 필터에 포함되고 `DATA_SOURCE` 는 선택적이기 때문에, 두 컬럼이 함께 필터되는 경우가 지배적이면서도 `STATUS` 단독 필터도 자주 발생합니다. 단일 인덱스 2 개로 분리하면 두 컬럼 모두 카디널리티가 3 으로 낮아 옵티마이저가 Full Table Scan 을 선호할 가능성이 있고, 복합 인덱스로 묶으면 선택도가 1/9 로 개선됩니다. `STATUS` 를 leading column 으로 둔 것은 leading column 규칙상 이 순서여야 `STATUS` 단독 필터와 `STATUS + DATA_SOURCE` 결합 필터 두 경우 모두 인덱스를 활용할 수 있기 때문입니다.

다만 이 설계에는 카디널리티 본질적 한계가 있습니다. 선택도 11% 는 여전히 일반적 기준인 5% 를 초과하므로 실제 운영에서 옵티마이저가 이 인덱스를 선택할지는 EXPLAIN PLAN 실측으로 검증해야 할 지점입니다. 현재는 설계 논리까지만 확정했고, 추후 더미 데이터 주입 후 단일 인덱스 / 복합 인덱스 / Bitmap Index 대안을 각각 벤치마크하여 옵티마이저의 실제 선택을 검증할 계획입니다."

### 6.3 주의해야 할 리스크

기존 포트폴리오에 IN 절 Chunking(DB 인덱스 관련) 트러블슈팅이 이미 있다. 이 복합 인덱스 건을 추가하면 **DB 인덱스 주제가 2 건** 이 된다. 면접관이 "인덱스 얘기만 반복하네" 로 인식할 수 있는 주제 편향 리스크를 경계해야 한다.

완화 방안:
- 본 주제를 별도 트러블슈팅 케이스로 독립시키지 않고, 매물 등록 시스템 고도화의 "설계 결정 부록" 으로 위치시킴
- 또는 F006~F009 후속 설계 트러블슈팅이 확정된 후, 전체 10 건(기존 6 건 + 고도화 4 건) 중 본 주제가 추가로 필요한지 검토 후 결정
