## 1. 탐색 방법론

### 1.1 기능 도출의 출발 원칙

고도화 기능 도출은 "기술에서 기능을 역산하는" 방향이 아니라 "비즈니스 요구사항이 기술을 끌어내는" 방향이다. 기술 적용의 출발점은 항상 비즈니스 요구사항이다. "상호배제 락을 적용하고 싶으니 기능을 만들자"가 아니라, "이 기능의 동시 쓰기 패턴에서 정합성을 보장하려면 상호배제 락이 필요하다"의 방향이어야 한다. 기술 선택의 근거가 비즈니스 시나리오에서 자연스럽게 도출되지 않으면 면접관이 "기술을 위한 기술"로 판단한다.

### 1.2 실제 서비스 리뷰 기반 기능 도출 근거 확보

자체 상상이 아닌 실제 서비스의 운영 맥락에서 기능 필요성의 근거를 확보한다. 직방·다방·네이버 부동산·피터팬·호갱노노·아실·부동산플래닛 등 7개 서비스의 Google Play Store/Apple App Store 사용자 리뷰, 블라인드/클리앙/DCInside 커뮤니티 불만, 앱 비교 기사를 종합 분석하여 피드백을 수집했다. 조사 결과 상세는 배경 및 근거자료 문서 7절 참조. 이 과정 자체가 포트폴리오에서 "기능 도출 근거"로 어필된다.

(기존 피드백에서 스스로가 고객이 되어 자체적인 지속적 피드백을 기반으로 프로젝트를 개선하라는 조언이 있었으나, 현재 서비스 구조에서는 그러한 기능적 맥락이 자연스럽게 요구되는 지점이 부재하여, 외부 실사용자 리뷰를 대체 근거로 활용한다.)

### 1.3 채용 시장 역량 매핑

한국 백엔드 신입·주니어 채용 공고 기술 역량 빈도 분석 결과를 기반으로 기능 후보가 "채용 시장에서 요구하는 기술 역량을 서류 통과에 지장이 없는 수준으로 포트폴리오에 온전하게 반영할 수 있는 구조"인지 여부를 필터링 기준으로 삼았다.

기능 후보는 다음 3가지 기준으로 필터링한다:

- 해당 기능이 누락 역량 중 어떤 것을 자연스럽게 요구하는가
- 개인 프로젝트 바운더리 내에서 구현 가능한가
- 테스트 유의성이 확보되는 데이터 규모인가 — 단순히 데이터를 "확보할 수 있는가"가 아니라, 확보된 데이터 규모에서 해당 기능이 요구하는 기술적 현상(락 경합·트랜잭션 충돌·스레드 풀 포화 등)이 실제로 관측 가능한가

### 1.4 개인 프로젝트 바운더리 준수

운영 환경(미니 PC, 다중 사용자 접속)에서 실제 동작하며, 테스트 시 유의미한 데이터 규모를 확보해야 한다. 인프라 과투자(Kafka·K8s·MSA 등), 프론트엔드 과투자, 도메인 지식 과투자 없이 백엔드 설계 역량을 증명하는 데 집중한다. 신입 포트폴리오로서 감당 가능한 범위를 초과하면 오히려 "실서비스 운영 수준이 아닌 학습용"이라는 인상을 줄 수 있다.

이 원칙이 양방향 상호작용(가격 네고·리뷰 반박·공동 관심 그룹), 구매 요청 시스템, 매물 방문 예약 같은 후보의 기각 판정에 직접 적용되었다.

## 2. 4차에 걸친 기능 도출 과정

### 2.1 제1차 제안 — 찜하기 + 가격 변동 알림 + 허위매물 신고

#### 2.1.1 제안 근거

현재 Wherehouse의 구조적 한계는 "조회 전용 단방향 파이프라인"이다. 사용자가 데이터를 쓰는 경로가 리뷰 CRUD 하나뿐이고, 그마저도 Write-Through 동기화 수준이라 동시 쓰기 충돌·트랜잭션 격리·락 전략 선택이 발생할 비즈니스 시나리오 자체가 없다.

필요한 것은 사용자가 능동적으로 데이터를 쓰는 경로를 2~3개 더 만들어서, 그 경로들이 서로 충돌하거나 기존 조회 경로와 간섭하는 상황을 만드는 것이다.

리서치에서 나온 수십 가지 기능 중 대부분은 프론트엔드 과투자(3D 단지투어·AI 챗봇)이거나 인프라 과투자(Kafka 기반 이벤트 버스)이거나 도메인 과투자(AVM 추정가 모델)라서 바운더리 밖이다. 바운더리 안에서 기존 Oracle + Redis 스택으로 구현 가능하고, 누락 역량을 자연스럽게 요구하는 기능은 3개로 수렴했다.

#### 2.1.2 후보 1 — 찜하기 (관심 매물)

사용자가 추천 결과에서 매물을 찜하고, 찜 수가 매물 카드에 표시되는 기능. 모든 부동산 앱이 공통으로 갖고 있는 기본 기능이고, 리서치에서 확인한 바와 같이 사용자 참여의 가장 기초적인 형태다.

이 기능이 자연스럽게 요구하는 기술적 의사결정: 인기 매물에 다수 사용자가 동시에 찜/취소를 누를 때 카운터 정합성 문제가 발생한다. Redis INCR 원자적 연산 vs Oracle row-level lock vs JPA @Version 낙관적 락 중 어떤 전략을 선택할 것인가? 찜 카운터를 Redis에만 둘 것인가, Oracle에 반정규화할 것인가? Redis 장애 시 폴백 전략은? 이것이 "대안 비교 → 트레이드오프 → 선택 근거" 구조의 트러블슈팅이 된다.

#### 2.1.3 후보 2 — 가격 변동 알림

배치가 새 데이터를 가져왔을 때 가격이 변경된 매물을 감지하고, 해당 매물을 찜한 사용자에게 알림을 보내는 기능. 리서치에서 MZ세대가 "아침마다 새 매물 확인"하는 패턴과, 가격 변동 추적에 대한 니즈가 반복적으로 나왔다.

이 기능이 자연스럽게 요구하는 기술적 의사결정: 배치가 Oracle을 업데이트하는 시점과 사용자가 매물을 조회하는 시점이 겹칠 때 어떤 격리 수준을 적용할 것인가? 알림 발송이 Oracle 커밋 전에 일어나면 순서 역전이 발생하는데, `@TransactionalEventListener(AFTER_COMMIT)` vs 별도 outbox 테이블 중 어떤 방식을 선택할 것인가? 알림 fan-out(1개 매물 변경 → N명 찜 사용자)을 메인 스레드에서 처리할 것인가, 전용 스레드풀을 분리할 것인가? 스레드풀 분리 설계·트랜잭션 격리 수준 선택·이벤트 순서 보장이 모두 이 기능 하나에서 나온다.

#### 2.1.4 후보 3 — 허위매물 신고

사용자가 매물을 신고하고, 신고 누적 시 자동으로 매물을 비공개 처리하는 기능. 리서치에서 허위매물이 부동산 앱 생태계의 #1 불만이고, 사용자 10명 중 3.4명이 경험하며, 기존 서비스의 신고 시스템 실효성에 대한 의문이 반복적으로 나왔다.

이 기능이 자연스럽게 요구하는 기술적 의사결정: 동시 신고 시 카운터 원자성(찜하기와 동일 패턴이지만 비즈니스 규칙이 다름 — 임계값 초과 시 상태 변경 트리거), 신고 임계값 도달과 매물 상태 변경의 트랜잭션 경계, 중복 신고 방지를 Redis Set으로 할 것인가 Oracle UNIQUE 제약조건으로 할 것인가.

#### 2.1.5 허위매물 신고 기각 사유

허위매물 신고는 기각했다. 찜하기와 기술 패턴이 거의 동일하다(Redis 원자적 카운터 + 임계값 판단). 같은 패턴을 두 번 구현하는 것은 포트폴리오에서 새로운 의사결정을 생성하지 않는다.

#### 2.1.6 1차 확정 — 찜하기 + 가격 변동 알림

찜하기와 가격 변동 알림은 기술 패턴이 완전히 다르다. 찜하기는 "동일 자원에 대한 동시 쓰기 → 락 전략 선택" 문제이고, 가격 변동 알림은 "배치 쓰기와 실시간 조회의 간섭 → 트랜잭션 격리 + 이벤트 순서 보장 + 스레드풀 분리" 문제다. 두 기능이 겹치는 기술 영역이 없으므로, 2개만으로 누락 역량 4개를 전부 커버한다고 판단했다.

### 2.2 제2차 수정 — 가격 변동 알림 기각, 사용자 행위 기반으로 전환

#### 2.2.1 기각 사유

가격 변동 알림의 이벤트 발생원이 배치(월 1회)이기 때문에 사용자 간 상호작용에 의한 비즈니스 맥락이 발생하지 않는다. 실 서비스가 아니기 때문에 실시간 매물 갱신이 되지 않고 한 달에 한 번밖에 없다. 배치가 아닌 사용자 행위에 의해 이벤트가 발생해야 수많은 사용자 간의 상호 작용에 의한 비즈니스 맥락이 주어진다.

#### 2.2.2 핵심 원칙 전환

**모든 이벤트의 발생원 = 사용자 행위**. 배치가 아닌, 사용자 A가 쓰기를 수행한 결과가 사용자 B의 데이터 상태에 영향을 주는 구조로 만든다. 이렇게 하면 사용자 수에 비례해서 이벤트 빈도가 증가하고, 월 1회 배치 한계에 종속되지 않는다.

#### 2.2.3 수정된 기능 구성 — 찜하기 + 인기 매물 알림 + 리뷰 투표

**기능 1: 찜하기 + 인기 매물 알림 (찜 급등 시 사용자 간 알림)**

기존에 제안한 찜하기에, 알림 트리거를 배치가 아닌 다른 사용자의 찜 행위로 바꾼다. 사용자 A가 매물을 찜한다 → 찜 카운터가 증가한다 → 카운터가 임계값(예: 10·30·50)을 넘는 순간 → 해당 매물을 이미 찜한 사용자 B·C·D 전원에게 알림이 생성된다("래미안그레이튼에 관심 급증, 현재 52명 찜").

이 구조에서 발생하는 기술적 의사결정:

사용자 A의 단일 찜 행위가 만드는 처리 체인 — 찜 INSERT → 카운터 증가 → 임계값 판단 → (초과 시) 찜 유저 목록 조회 → N명에게 알림 INSERT. 이 체인 전체를 하나의 트랜잭션으로 묶으면 N명 알림 INSERT 동안 해당 매물의 찜 테이블에 lock이 걸린다. 분리하면 알림 생성 중 다른 사용자의 찜이 추가되어 카운터와 알림 대상 목록이 불일치할 수 있다. 트랜잭션 경계를 어디서 끊을 것인가가 첫 번째 의사결정이다.

fan-out(1명의 찜 → N명에게 알림)을 메인 API 스레드에서 동기로 처리하면 사용자 A의 찜 API 응답이 N에 비례해서 느려진다. 알림 생성을 전용 스레드풀에서 비동기로 분리할 것인가, 메인 트랜잭션 커밋 후 이벤트로 위임할 것인가 — `@TransactionalEventListener(AFTER_COMMIT)` + `@Async("notificationExecutor")` 조합의 설계 근거가 여기서 나온다. 스레드풀 크기는 미니 PC 4코어 기준으로 I/O 대기 비율을 실측해서 정량적으로 결정한다.

기존 "가격 변동 알림"에서 필요했던 기술적 의사결정(트랜잭션 격리·이벤트 순서 보장·스레드풀 분리)이 전부 동일하게 발생하지만, 트리거가 사용자 행위다. 사용자가 많을수록 이벤트가 많아지고, 배치 주기에 종속되지 않는다.

**기능 2: 리뷰 도움돼요 투표**

사용자가 다른 사용자의 리뷰에 "도움돼요/별로에요" 투표를 한다. 투표 수는 리뷰 카드에 표시되고, 리뷰 목록의 정렬 기준이 된다. 도움돼요 비율이 높은 리뷰가 상단에 노출되고, 이 리뷰의 키워드가 해당 매물의 추천 점수에 더 큰 가중치로 반영된다.

이 기능이 만드는 기술적 의사결정:

사용자 A가 투표 한 번을 누르면 발생하는 처리 체인 — 투표 INSERT → 리뷰의 `helpful_count` 증가 → 리뷰 정렬 순위 변동 → (변동이 클 경우) 해당 매물의 키워드 가중치 재계산 → 매물 ReviewScore 변동 → Redis 캐시 갱신. 단일 사용자 행위가 4단계 cascade를 발생시킨다.

이 cascade의 각 단계를 어디까지 동기로 처리하고 어디서부터 비동기로 분리할 것인가. 투표 INSERT + helpful_count 증가까지는 동기 트랜잭션, 리뷰 순위 재계산과 매물 점수 갱신은 비동기 이벤트로 분리하는 것이 자연스럽다. 그런데 비동기 분리 시 "투표했는데 리뷰 순위가 바로 안 바뀐다"는 최종 일관성(Eventual Consistency) 문제가 발생한다. 즉시 일관성 vs 최종 일관성의 비즈니스 허용 범위를 결정하는 것이 설계 의사결정이 된다.

동시 투표 시 `helpful_count` 정합성은 찜하기 카운터와 동일한 패턴이지만, 비즈니스 규칙이 다르다. 찜하기는 카운터가 단순 표시용이라 1~2의 오차가 허용되지만, 투표는 정렬 순위에 직접 영향을 주므로 정합성 요구 수준이 다르다. 동일한 Redis INCR vs Oracle row-level lock 대안인데 비즈니스 요구사항의 차이로 선택이 달라지는 사례 — 이것이 포트폴리오에서 "같은 기술인데 맥락에 따라 다른 선택을 했다"를 보여주는 가장 강력한 근거가 된다.

### 2.3 제3차 확장 — 매물 등록 기능 추가

#### 2.3.1 추가 근거

지금까지 논의한 찜·투표·리뷰는 전부 매물이라는 핵심 엔티티에 대한 부수 데이터를 쓰는 행위다. 매물 자체는 배치가 월 1회 넣어주는 읽기 전용 데이터로 고정되어 있고, 사용자는 그 위에 메타데이터(리뷰·찜·투표)만 붙이는 구조다.

매물 등록을 사용자에게 열면 핵심 엔티티 자체가 사용자 쓰기 경로에 들어온다. 이것은 부수 데이터 쓰기와 질적으로 다른 시나리오를 대량으로 생성한다. 실 상용 서비스에서는 실제 판매자만 등록할 수 있지만, 현재는 상황 재연을 위한 포트폴리오 프로젝트이므로 누구나 매물 정보를 등록할 수 있는 구조로 설계한다.

#### 2.3.2 매물 등록이 생성하는 기술적 시나리오 5가지

**시나리오 1 — 동일 매물 동시 등록 (중복 감지 Race Condition)**: 사용자 A와 사용자 B가 같은 물리적 매물(같은 아파트·같은 층·같은 면적)을 거의 동시에 등록한다. 둘 다 동일한 MD5 해시가 생성된다. 둘 다 INSERT 전 확인 → 둘 다 "존재하지 않음" → 둘 다 INSERT 시도 → Oracle PK 제약조건에 의해 하나만 성공하고 하나는 DuplicateKeyException 발생.

**시나리오 2 — 사용자 등록 매물 vs 배치 매물 충돌 (데이터 소스 간 머지 전략)**: 사용자가 등록한 매물과 국토부 배치가 가져온 매물이 동일 MD5 해시를 가진다. 사용자가 "전세금 1.8억"으로 등록했는데 다음 배치에서 국토부 API가 같은 매물을 "전세금 2억"으로 가져온다. 누구의 데이터를 신뢰할 것인가.

**시나리오 3 — 매물 등록 → Redis 인덱스 즉시 반영 (쓰기 전파 정합성)**: 현재 시스템에서 Redis Sorted Set 인덱스는 배치가 일괄 갱신한다. 사용자가 매물을 등록하면 이 인덱스에 즉시 반영해야 해당 매물이 추천 결과에 나타난다. Oracle INSERT는 성공했는데 Redis Sorted Set ZADD 중간에 실패하면 매물은 DB에 존재하지만 추천 검색에서는 안 나오는 불일치 상태.

**시나리오 4 — 매물 등록자가 상태를 변경한다 (거래완료, 가격 수정)**: 등록자가 "거래완료"로 상태를 변경하면 해당 매물을 찜한 N명의 찜 목록에서 상태가 바뀌어야 하고, 해당 매물에 달린 리뷰의 노출 정책이 변경될 수 있고, 해당 매물이 Redis Sorted Set 인덱스에서 제거되어야 하고, 진행 중인 추천 요청이 해당 매물을 이미 조회한 상태일 수 있다. 등록자가 가격을 수정하면 bounds(정규화 범위)가 변경될 수 있어서 같은 지역구의 다른 매물 점수도 영향을 받는다. 이것이 앞서 기각한 "가격 변동 알림"이 사용자 행위 기반으로 부활하는 구조다.

**시나리오 5 — 리뷰와 매물 등록의 상호 의존성**: 기존 시스템에서 리뷰는 매물 FK를 참조한다. 배치 매물만 존재할 때는 FK 대상이 항상 존재했다. 사용자 등록 매물이 추가되면 FK 참조 무결성 문제가 발생한다. 리뷰가 달린 매물은 삭제 불가로 할 것인가, Soft Delete로 할 것인가.

#### 2.3.3 매물 등록 추가에 의한 시스템 구조 변환

| 항목 | 추가 전 | 매물 등록 추가 후 |
|------|------|-----------------|
| 매물 데이터 소스 | 배치(월 1회) 단일 | 배치 + 사용자 등록 이원 |
| 쓰기 경로 | 리뷰 1개 | 매물 등록/수정/삭제 + 리뷰 + 찜 + 투표 |
| 핵심 엔티티 쓰기 주체 | 시스템(배치) only | 시스템 + 사용자 |
| 동시성 충돌 지점 | 리뷰 통계 카운터 | 매물 중복 등록, 카운터 2종(찜/투표), 상태 변경 + 조회 |
| 트랜잭션 복잡도 | 리뷰 INSERT + 통계 UPDATE | 매물 INSERT + Redis 6개 인덱스 + bounds 재계산 |
| 데이터 정합성 이슈 | Write-Through 단일 | 배치-사용자 머지, 부분 실패 보상, FK 참조 보호 |

기존 "조회 전용 파이프라인" 구조가 완전히 해체되고, 다수의 쓰기 경로가 핵심 엔티티 수준에서 상호 간섭하는 시스템으로 전환된다.

### 2.4 제4차 심화 — 사용자 간 상호작용 중심축 전환

#### 2.4.1 전환 동기

개별 기능(찜·투표·매물 등록)을 먼저 정의하고 그 부산물로 사용자 간 상호작용이 "발생하기도 한다"는 구조가 아니라, 사용자 A의 행위가 사용자 B의 데이터 상태를 변경하는 것 자체를 설계의 중심축으로 놓고, 그 축을 구현하기 위해 어떤 기능이 필요한지를 역으로 도출한다.

현재 설계의 상호작용 구조를 냉정하게 보면, 전부 매물을 허브로 한 간접 상호작용이다. 사용자 A가 매물에 행위를 하면, 시스템이 같은 매물에 연결된 사용자 B에게 영향을 전파하는 구조다. A와 B가 직접 만나는 지점이 없다.

#### 2.4.2 상호작용 유형 3가지 식별

**유형 1 — 내 행위가 타인의 "목록"을 오염시키는 상호작용**: 사용자 A가 매물을 등록한다. 사용자 B가 그 매물을 찜한다. 사용자 C가 그 매물에 리뷰를 쓴다. 그런데 사용자 A가 매물을 "거래완료"로 변경하거나 삭제한다. 이 순간 사용자 B의 찜 목록에 있는 매물의 상태가 바뀌어야 한다. 사용자 C의 리뷰가 참조하는 매물의 노출 정책이 바뀌어야 한다. 추천 결과에서 해당 매물이 제거되어야 하므로, 현재 추천 요청을 처리 중인 사용자 D의 응답에도 영향을 줄 수 있다. 이것은 1명의 상태 변경이 N명의 저장 데이터를 cascade로 변경하는 구조다.

**유형 2 — 다수의 행위가 하나의 공유 상태를 경쟁적으로 변경하는 상호작용**: 사용자 A·B·C가 동시에 같은 매물에 리뷰를 작성한다. 세 명의 리뷰가 모두 성공하면 REVIEW_STATISTICS의 review_count·avg_rating·positive_keyword_count가 3번 갱신되어야 한다. 각각의 갱신이 SELECT(현재 통계) → 계산 → UPDATE 패턴이라면, A의 SELECT와 B의 SELECT가 같은 값을 읽고 각자 계산한 뒤 UPDATE하면 하나의 리뷰 반영분이 유실된다(Lost Update). 동시에 같은 매물을 여러 명이 찜하면 favorite_count에서 동일한 문제가 발생한다. 동시에 같은 리뷰에 여러 명이 투표하면 helpful_count에서 동일한 문제가 발생한다.

같은 패턴이 3개의 서로 다른 비즈니스 맥락에서 반복되지만, 요구되는 정합성 수준이 다르다. 찜 수는 표시용이라 일시적 오차가 허용된다. 리뷰 통계는 추천 점수에 직접 영향을 주므로 정합성이 중요하다. 투표 수는 리뷰 정렬 순위를 결정하므로 중간 수준이다. 이 차이가 동일 기술(Redis INCR vs Oracle row lock vs 낙관적 락)에 대해 맥락별로 다른 선택을 하는 근거가 된다.

**유형 3 — 내 행위가 타인에게 "알림"을 발생시키는 상호작용**: 사용자 A가 매물 가격을 수정한다 → 해당 매물을 찜한 B·C·D에게 가격 변동 알림이 생성된다. 사용자 A가 매물을 찜한다 → 찜 수가 임계값을 넘으면 이미 찜한 B·C에게 인기 급등 알림이 생성된다. 사용자 A가 리뷰를 작성한다 → 해당 매물을 찜한 B·C에게 새 리뷰 알림이 생성된다. 모든 알림의 트리거가 사용자 행위이고, 알림 대상이 "해당 매물을 찜한 사용자 집합"이라는 공통 fan-out 구조를 공유한다.

#### 2.4.3 상호작용 유형으로부터 기능 목록 역도출

위 상호작용 유형을 구현하려면 최소한 다음 기능이 필요하다:

**매물 등록/수정/상태변경** — 유형 1(cascade)과 유형 3(알림)의 트리거 역할. 사용자가 핵심 엔티티를 직접 쓰므로, 중복 감지·배치 충돌·Redis 다중 인덱스 동기화가 전부 여기서 발생한다.

**찜하기** — 유형 2(동시 카운터)와 유형 3(알림)의 매개 역할. 찜 자체가 쓰기이면서, 동시에 알림 대상 집합을 정의하는 데이터 구조이기도 하다. 즉 찜은 "이 매물에 관심 있는 사용자 목록"이라는 구독(subscription) 역할을 겸한다.

**리뷰 + 투표** — 유형 2(동시 카운터)의 두 번째·세 번째 인스턴스. 리뷰 작성은 유형 3(새 리뷰 알림)의 또 다른 트리거다.

### 2.5 양방향 상호작용 확장 검토 및 기각

#### 2.5.1 검토한 양방향 상호작용 후보

양방향 상호작용(A의 행위에 대해 B가 응답하고, 그 응답이 다시 A나 C의 상태를 변경하는 양방향 체인)을 만들려면 다음이 가능하다: 매물에 대한 가격 제안(네고) → 등록자가 수락/거절 → 수락 시 다른 제안자 전원에게 종료 알림, 리뷰에 대한 반박/이의 제기 → 원 작성자가 수정 또는 유지 → 다른 사용자의 투표로 최종 판정, 매물 공동 관심 그룹(같은 매물을 찜한 사용자끼리 정보 공유).

이것들은 기술적으로 구현 가능하고, 트랜잭션 체인도 길어지고, 상태 머신(제안→수락/거절→완료)이 들어오면서 동시성 문제도 더 복잡해진다.

#### 2.5.2 기각 사유

양방향 상호작용을 만들면 프론트엔드에서 상태 전이 UI(제안 목록·수락/거절 버튼·실시간 상태 반영)가 필수적으로 따라오고, 이것이 백엔드 1인 전담 프로젝트의 바운더리를 넘길 가능성이 높다. 또한 가격 네고나 리뷰 반박 같은 기능은 실제 부동산 서비스에서도 극히 일부만 존재하는 기능이라, 면접관이 "이걸 왜 만들었는가"에 대한 비즈니스 근거를 물었을 때 답이 약해진다.

현재 설계의 단방향 cascade 구조가 이미 커버하는 누락 역량은 다음과 같다: DB 레벨 락 전략 비교(매물 중복 등록), 동시성 제어 실측(카운터 3종), 트랜잭션 경계 설계(cascade 분리), 격리 수준 선택(배치 vs 사용자 충돌), 스레드풀 분리(알림 fan-out), 자료구조 비교(Set vs Sorted Set), 즉시/최종 일관성 비즈니스 결정. 채용 시장 분석에서 식별한 차별화 역량을 전부 커버한다.

양방향 상호작용의 핵심 기술 패턴(상태 머신 + 동시 상태 전이 제어)은 이미 ChatService의 2단계 입장 구조에서 증명한 바 있다.

### 2.6 4개 쓰기 경로 확정

상호작용 중심축 전환 결과 확정된 4개 쓰기 경로:

1. **리뷰 작성** (기존) — 사용자가 매물에 대해 별점과 텍스트 리뷰를 등록
2. **찜하기 + 인기 알림** (신규) — 사용자가 매물을 찜, 임계값 초과 시 찜 유저 전원에게 알림
3. **리뷰 투표** (신규) — 사용자가 다른 사용자의 리뷰에 도움돼요/별로에요 투표
4. **매물 등록/수정/상태변경** (신규) — 사용자가 매물 정보를 직접 등록·수정·거래완료 처리

이 4개 경로 확정이 이 단계의 최종 결론이었다. 이후 단일 기능 심화로의 전환은 3절에서 다룬다.

## 3. 4개 쓰기 경로에서 단일 기능 심화로의 전환

### 3.1 전환 판단의 3가지 근거

4개 쓰기 경로를 전부 구현할 경우 발생하는 문제는 다음과 같다.

#### 3.1.1 구현 분량 초과

매물 등록(Oracle INSERT + Redis 6개 인덱스 + bounds 재계산 + 배치 충돌 머지), 찜하기(카운터 동시성 + 알림 fan-out + 스레드풀), 리뷰 투표(cascade 4단계 + 즉시/최종 일관성), 기존 리뷰 시스템 — 4개를 전부 구현·테스트·문서화하는 것은 본 프로젝트에서 수행 가능한 작업 범위를 초과한다.

#### 3.1.2 포트폴리오 분량 비대화

4개 기능에서 각각 1~2건의 트러블슈팅이 나오면 기존 6건 + 신규 5~8건 = 11~14건. 포트폴리오가 비대해지고 면접에서 전부 다루는 것 자체가 불가능.

#### 3.1.3 기능 간 맥락 연결 부재

찜하기·투표·매물 등록이 각각 독립적 기능처럼 나열되어 "이 서비스가 왜 이 기능들을 전부 필요로 하는가"에 대한 통합 비즈니스 서사가 없다. 기능 하나하나는 기술적 의사결정을 만들지만, 전체를 관통하는 이야기가 없으면 면접관 입장에서 "기술 데모를 위해 기능을 끼워넣은 것"으로 보인다.

### 3.2 매물 등록/수정/상태변경이 선택된 이유

논의록에서 확정된 4개 중, 포트폴리오 가치가 가장 높은 것은 매물 등록/수정/상태변경이다.

**이유 1 — 기술적 의사결정의 다양성과 기존 시스템 관통성**: 매물 등록은 Oracle INSERT + Redis Hash + Sorted Set 5~6개 + bounds 재계산 + 배치 충돌 머지를 전부 건드린다. 기존 CQRS 아키텍처·Write-Through 동기화·Redis Pipeline·CompletableFuture 등 이미 포트폴리오에 있는 기술 스택 위에서 새로운 의사결정이 추가되는 구조다. 완전히 새로운 기능을 붙이는 것이 아니라, 기존 시스템의 확장으로서 자연스럽다.

**이유 2 — 비즈니스 근거의 강도**: 실제 서비스 조사 결론에서 "원룸·빌라 대상 실거주 후기 집중 플랫폼"이 개인 프로젝트가 노릴 수 있는 틈새로 식별됐다. 사용자가 직접 매물 정보를 등록하는 것은 피터팬(직거래 플랫폼)의 핵심 모델이고, 다방의 동네이야기 커뮤니티에서 매물 첨부 기능으로 이미 구현되어 있다.

나머지 3개(찜하기+인기알림·리뷰 투표·기존 리뷰 시스템 확장)는 본 범위에서 제외.

## 4. 기각된 기능별 기각 사유 종합

본 절은 고도화 과정에서 후보로 검토되었다가 최종적으로 배제된 모든 기능을 기각 시점과 기각 사유 순서로 정리한다. 면접관의 "왜 X는 안 하셨나요?" 질문에 대한 대응 자료.

### 4.1 허위매물 신고

**기각 시점**: 1차 제안 단계.

**기각 사유**: 찜하기와 기술 패턴이 거의 동일하다(Redis 원자적 카운터 + 임계값 판단). 같은 패턴을 두 번 구현하는 것은 포트폴리오에서 새로운 의사결정을 생성하지 않는다. 비즈니스적으로는 리서치에서 허위매물이 부동산 앱 #1 불만으로 확인되었으나, 기술적 중복성이 기각 근거가 되었다.

### 4.2 가격 변동 알림 (배치 트리거)

**기각 시점**: 2차 수정 단계.

**기각 사유**: 이벤트 발생원이 배치(월 1회)이므로 사용자 간 상호작용 맥락이 부재. 실 서비스가 아니기 때문에 실시간 매물 갱신이 되지 않고 한 달에 한 번밖에 없다. 배치가 아닌 사용자 행위에 의해 이벤트가 발생해야 수많은 사용자 간의 상호 작용에 의한 비즈니스 맥락이 주어진다.

**부활 조건**: 이 기각은 "배치 트리거" 형태에 한정된다. 사용자 행위(등록자의 가격 수정)를 트리거로 하는 가격 변동 알림은 3차 확장 단계에서 매물 등록 시나리오 4로 부활했다. 이후 단일 기능 심화로 전환하면서 알림 fan-out 부분은 현 구현 범위에서 제외되었으나, 가격 변동이 발생한다는 비즈니스 맥락 자체는 F009(bounds 변경 점수 재계산)에 흡수되었다.

### 4.3 찜하기 + 인기 알림

**기각 시점**: 단일 기능 심화 전환 단계.

**기각 사유**: 4개 쓰기 경로 전부 구현이 시간·분량·서사 통합의 3가지 문제를 일으키므로 단일 기능 심화로 전환. 4개 중 매물 등록이 포트폴리오 가치가 가장 높아 찜하기는 범위 외.

**포트폴리오 가치**: 찜하기 단독으로는 카운터 동시성 3종 비교(Redis INCR / Oracle row-level lock / JPA @Version)가 가능했고, 인기 알림 결합 시 fan-out 트랜잭션 경계 + 스레드풀 분리가 추가로 발생했다. 매물 등록의 F006(중복 감지 3종 비교)·F008(다중 키 부분 실패 보상)·F009(스레드풀 분리)가 유사한 기술 영역을 커버하므로, 찜하기 배제가 누락 역량 매핑에 큰 공백을 만들지 않는다.

### 4.4 리뷰 투표

**기각 시점**: 단일 기능 심화 전환 단계.

**기각 사유**: 찜하기와 동일. 4개 쓰기 경로 전부 구현의 비현실성으로 단일 기능 심화로 전환하면서 범위 외 처리.

**포트폴리오 가치**: 리뷰 투표는 cascade 4단계(투표 INSERT → helpful_count 증가 → 리뷰 순위 변동 → 매물 ReviewScore 변동 → Redis 캐시 갱신)로 즉시/최종 일관성 선택의 강한 근거를 가지는 기능이었다. 특히 "찜 카운터와 helpful_count가 같은 기술 대안(Redis INCR vs Oracle row-level lock)에 대해 비즈니스 맥락 차이로 다른 선택을 하는 사례"로서, 면접에서 "같은 기술인데 맥락에 따라 다른 선택을 했다"를 보여주는 강력한 자료였다. 배제 시 이 서사를 잃는다. 다만 매물 등록의 F006·F007·F008·F009에서 "동일 기술 대안에 대한 선택 근거 비교"는 이미 발생하므로, 리뷰 투표 배제가 치명적이지는 않다.

### 4.5 양방향 상호작용 3종 (가격 제안·리뷰 반박·공동 관심 그룹)

**기각 시점**: 4차 심화 단계에서 확장 검토 후 기각.

**기각 사유**:

- 프론트엔드 상태 전이 UI 필수, 백엔드 1인 프로젝트 바운더리 초과
- 비즈니스 근거 약함 — 실제 부동산 서비스에서도 극히 일부만 존재하는 기능이라, 면접관이 "이걸 왜 만들었는가"에 답이 약해짐
- 양방향 상호작용의 핵심 기술 패턴(상태 머신 + 동시 상태 전이 제어)은 이미 ChatService의 2단계 입장 구조에서 증명한 바 있음
- 단방향 cascade 구조가 이미 누락 역량을 커버 — DB 레벨 락·동시성 제어·트랜잭션 경계·격리 수준·스레드풀 분리·자료구조 비교·즉시/최종 일관성

### 4.6 구매 요청 시스템 (F003 확장 제안)

**기각 시점**: 기획서/요구사항 명세서 작성 단계에서 정범진이 F003(매물 상태 변경)의 비즈니스 복잡도 확장으로 제안 → 분석 후 기각.

**기각 사유**:

**기존 트러블슈팅 4건과의 중복 여부**: 구매 요청 INSERT는 각 사용자가 서로 다른 row를 생성하므로 동시성 충돌이 발생하지 않는다. F006(동일 매물 동시 등록)의 "같은 row에 대한 Race Condition"과 질적으로 다르다. 소유자 수락 시 매물 상태 변경은 기존 F003 → F008 → F009 체인을 트리거하므로 기존 맥락과 동일.

**새로운 기술적 의사결정 발생 여부**: "수락 1건 → 다른 모든 요청 일괄 거절"의 트랜잭션 경계가 후보이나, F008(다중 키 갱신의 원자성)과 패턴이 유사("1개 행위 → N개 대상 영향의 일괄 처리 트랜잭션 경계"). 새로운 유형의 의사결정이 아닌 기존 패턴의 반복.

**구현 비용**: 구매 요청 테이블·요청 API(생성·조회·취소)·수락/거절 API·소유자 UI 요청 목록 표시 등 프론트엔드 작업량이 상당히 추가. 4.5절에서 "가격 제안(네고)"이 동일 패턴으로 검토·기각된 이유와 동일.

**판단**: F003의 포트폴리오 가치는 F003 단독이 아니라, F003이 트리거하는 F008(검색 인덱스 제거) + F009(bounds 변경 시 점수 재계산) 체인에서 나온다. 구매 요청 시스템은 기존 4건과 질적으로 다른 새로운 트러블슈팅을 만들지 않으면서 구현 작업량을 상당히 소모한다.

## 5. 유보된 탐색 내용

본 절은 탐색 과정에서 후보로 검토되었으나, 현재 구현 범위(매물 등록 단일 기능)에서는 해당하지 않는 기술 지점을 기록한다. 추후 찜하기·투표·리뷰 확장을 구현할 경우 재활성화 가능한 내용이다.

### 5.1 자료구조/알고리즘 4개 지점

#### 5.1.1 Top-K 추출 — 매물 등록 구현 과정 포함 여부 결정

현재 추천 서비스의 최종 단계는 모든 매물의 finalScore를 기준으로 전체 정렬 → 상위 3개 지역구 선정 → 각 지역구 내 상위 3개 매물 추출이다. 설계 명세서 v2에서 이 부분은 "내림차순 정렬"이라고만 기술되어 있고, 구현이 `Collections.sort()` 또는 `Stream.sorted()`일 가능성이 높다.

문제: 강남구에 매물 1,000건, 서울 전체 25개 자치구에 총 11만 건이 있을 때, 최종 응답에 필요한 것은 상위 3개 지역구 × 3개 매물 = 9건뿐이다. 11만 건을 전체 정렬하는 것은 O(N log N)이고, 필요한 것은 상위 9건이다.

**대안 3개**:
- `Collections.sort()` — 전체 정렬 후 subList(0, K). O(N log N).
- `PriorityQueue`(min-heap, size K) — O(N log K).
- QuickSelect(Introselect) — O(N) 평균, 최악 O(N²).

**유보 상태**: 매물 등록 구현 과정에서 본 지점이 자연스럽게 발생하는지 여부에 따라 포함 결정.

#### 5.1.2 지역구별 매물 그룹핑 — HashMap vs TreeMap

bounds 변경 → 지역구 전체 매물 점수 재계산 시나리오(F009)에서, Redis에서 가져온 매물 리스트를 지역구별로 그룹핑해야 한다.

**대안 3개**: `HashMap<String, List<PropertyWithScore>>` — 그룹핑 O(N), 이후 지역구 대표 점수 기준 정렬 별도 필요. `TreeMap<Double, List<PropertyWithScore>>` — 키를 지역구 대표 점수로 설정하면 삽입 시 자동 정렬, 단 점수가 재계산되므로 삽입 시점에 정확한 키를 알 수 없어 두 번의 패스 필요. `LinkedHashMap` — 삽입 순서 유지, 점수 기반 정렬에는 도움 안 됨.

**유보 상태**: F009 구현 과정에서 자연스럽게 발생하는 지점이므로, 설계 명세 작성 시 포함 여부 결정.

#### 5.1.3 알림 병합 5종 비교 — 해당 없음

찜하기·투표 미구현으로 인해 알림 fan-out 시나리오가 발생하지 않으므로 현 시점에서 해당하지 않는다. `HashSet<NotificationKey>` / `HashMap<NotificationKey, NotificationPayload>` / `LinkedHashMap(accessOrder=true, removeEldestEntry)` / `PriorityQueue<TimestampedNotification>` / Redis SETEX — 5방향 비교는 찜하기 추가 구현 시 재활성화 가능.

#### 5.1.4 키워드 매칭 3종 비교 — 해당 없음

리뷰 확장 미구현으로 인해 해당하지 않는다. 키워드 사전 19개 → 100개+ 확장 시나리오에서 contains() / HashSet 토큰 매칭 / Aho-Corasick 3종 비교는 리뷰 기능 고도화 시 재활성화 가능.

### 5.2 CPU 바운드 맥락 중 F009 외 영역

매물 등록의 bounds 변경 → 지역구 전체 매물 점수 재계산이 F009로 흡수되면서 ForkJoinPool의 적용이 확보되었다. 이 외의 CPU 바운드 맥락(예: 리뷰 키워드 가중치 재계산, 찜 급등 임계값 계산 시의 배치 연산)은 해당 기능 미구현으로 현 범위에서 제외.

### 5.3 알림 인프라 공통화

4차 심화 단계에서 "모든 알림의 트리거가 사용자 행위이고, 알림 대상이 '해당 매물을 찜한 사용자 집합'이라는 공통 fan-out 구조를 공유한다"는 패턴이 식별되었다. 이 공통 구조를 하나의 알림 인프라로 설계하고, 알림 유형별로 트리거 조건과 페이로드만 달리하는 것이 설계 의사결정이 될 수 있었으나, 찜하기·투표 배제로 인해 이 맥락은 현재 구현 범위 외.

## 6. 혼란의 구조적 원인 진단

본 절은 기능 선정 과정에서 발생한 판단 지연의 구조적 원인을 기록한다. 이 진단 자체가 포트폴리오에서 "설계 과정의 자기 인식" 근거로 활용 가능하다.

### 6.1 논의록 자체의 논리적 견고성은 유효

4차에 걸친 기능 도출 과정(1차→2차→3차→4차)은 논리적으로 견고하게 진행되었다. 기각 사유도 명확하고, 상호작용 유형 3가지 식별도 정교하다.

### 6.2 논의록의 미완결 지점

최종 결론이 "4개 쓰기 경로 확정"으로 끝나면서, 정작 **"이것을 구현하면 포트폴리오에 정확히 어떤 트러블슈팅 케이스가 몇 건 생기는가"가 구체적으로 확정되지 않은 채 끝났다.** 기능 목록은 나왔는데, 그 기능이 포트폴리오 문서의 어떤 페이지에 어떤 구조로 들어가는지가 불명확한 상태였다.

추가로 식별된 근본 문제: 국토부 API가 월 1회만 갱신하는 데이터를 기반으로 하므로, 기반 데이터 자체가 매우 불충분하다. 결국 유저 간 상호작용 위주로 설계해야 하는데, 논의록의 기능 구성이 이 맥락을 구체적으로 반영하지 못한 상태에서 진행되었다.

### 6.3 위화감의 정체 — 시뮬레이션 기반 동시성 문제의 설득력

근본적인 문제가 존재한다. "찜하기"를 예로 들면 — 찜 카운터 동시성 3종 비교(Redis INCR / Oracle row lock / JPA @Version)라는 기술적 의사결정이 존재하긴 한다. 그런데 이 의사결정이 발생하려면 "인기 매물에 다수 사용자가 동시에 찜/취소를 누르는 상황"이 전제되어야 한다. 실제로 이 서비스에 동시 접속 사용자가 없다. 시드 데이터로 JMeter 부하를 걸어서 시뮬레이션하는 수밖에 없다. 그러면 면접관이 "이 동시성 문제가 실제로 발생한 건가요, 시뮬레이션한 건가요?"라고 물었을 때 — "시뮬레이션입니다"라고 답해야 한다. 이것이 포트폴리오로서 설득력이 있는가?

이 질문에 대한 답이 서지 않으니까 기능 선정 자체가 흔들리는 구조였다. 본 위화감에 대한 판단 결론은 현황 및 설계 의사결정 종합 기록 8절 참조.

### 6.4 도메인 경험 부재 인식과 판단

정범진은 기획자가 아니고 실제 부동산 매물을 사본 적도 없어서 비즈니스 맥락을 사고하기에 범위와 경험이 제한적임을 인식했다. 이에 대한 판단: 포트폴리오의 비즈니스 로직은 실제 서비스 수준의 도메인 완성도가 필요한 것이 아니라, 기술적 의사결정이 자연스럽게 발생하는 최소한의 비즈니스 시나리오만 있으면 충분하다. 이 판단이 매물 등록 단일 기능 선택의 근거 중 하나가 되었다.
