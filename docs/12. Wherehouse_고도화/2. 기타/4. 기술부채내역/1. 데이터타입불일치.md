## TD-001: DEAL_DATE 컬럼 타입 불일치

### 현재 상태
- DB 컬럼: PROPERTIES_CHARTER.DEAL_DATE, PROPERTIES_MONTHLY.DEAL_DATE — VARCHAR2(10 BYTE)
- 저장 포맷: "YYYY-MM-DD" 문자열 (ISO 8601)
- 발생 경로: 초기 BatchScheduler 구현 시 국토부 API 응답 파싱 결과를
             SimpleDateFormat → String으로 저장

### 문제점
1. RDB 네이티브 날짜 연산 기능 활용 불가
   - 날짜 범위 쿼리 시 문자열 비교에 의존 (사전식 정렬이 ISO 8601에서는 우연히 맞음)
   - BETWEEN 연산, 월별 집계 등에서 TO_DATE 변환 오버헤드
2. 애플리케이션 계층에서 DTO(LocalDate) ↔ Entity(String) 변환 로직 필요
3. DEAL_DATE 기반 인덱스의 카디널리티·효율이 DATE 타입보다 떨어짐

### 해결 방안 (향후)
Phase 1: ALTER TABLE ADD DEAL_DATE_NEW (DATE)
Phase 2: UPDATE로 기존 VARCHAR2 값을 TO_DATE 변환하여 이관
Phase 3: BatchScheduler·Entity·Repository 모두 DATE 타입 참조로 전환
Phase 4: 기존 DEAL_DATE 컬럼 DROP, DEAL_DATE_NEW → DEAL_DATE RENAME

### 우선순위
Low — 현재 비즈니스 로직에서 날짜 연산이 핵심 경로가 아니며,
      추천 검색이 dealDate를 필터 조건으로 사용하지 않음
