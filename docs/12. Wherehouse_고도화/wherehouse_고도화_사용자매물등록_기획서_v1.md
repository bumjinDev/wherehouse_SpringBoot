# Wherehouse 고도화 — 사용자 매물 등록 시스템 기획서

## 프로젝트 개요

**목표**: 현재 배치 단일 소스(국토교통부 API, 월 1회)에 의존하는 매물 데이터 구조에 사용자 직접 등록 경로를 추가하여, 핵심 엔티티(매물)에 대한 사용자 쓰기가 기존 CQRS 아키텍처(Oracle Write + Redis Read)의 전 계층을 관통하는 비즈니스 시나리오를 구축하고, 이 과정에서 발생하는 기술적 의사결정(동시 등록 중복 감지, 데이터 소스 간 머지 정책, Redis 다중 인덱스 부분 실패 전략, CPU 바운드 점수 재계산 병렬 처리)을 포트폴리오에 "비즈니스 요구사항 → 대안 분석 → 트레이드오프 → 선택 근거 → 정량 결과" 구조로 반영한다.

**핵심 질문**: "기존 조회 전용 단방향 파이프라인에 사용자 쓰기 경로를 추가했을 때, Oracle 커밋과 Redis 다중 인덱스 동기화 사이의 정합성, 배치 데이터와 사용자 데이터의 충돌 해소, 정규화 기준(bounds) 변경에 의한 연쇄 점수 재계산을 각각 어떤 전략으로 처리할 것인가?"

**접근 방식**: 실제 부동산 서비스(피터팬 직거래 모델, 다방 동네이야기 매물 첨부 기능)에서 검증된 "사용자 참여형 매물 등록" 모델을 Wherehouse에 도입한다. 구현 범위를 매물 등록/수정/상태변경 단일 기능으로 한정하되, 이 기능이 기존 시스템의 Oracle PROPERTIES 테이블, Redis Hash/Sorted Set/bounds 인덱스, CQRS Write-Through 동기화, 추천 점수 산출 파이프라인 전체를 관통하도록 설계한다. 각 관통 지점에서 발생하는 기술적 의사결정을 포트폴리오 트러블슈팅 케이스로 문서화한다.

## 현재 상태 분석

### 기존의 구체적 문제점

| 구현 부분 | 현재 로직 | 문제점 | 비즈니스 영향 |
|-----------|-----------|--------|---------------|
| **매물 데이터 소스** | 국토교통부 API를 BatchScheduler가 월 1회 수집하여 Oracle에 UPSERT하고 Redis에 동기화하는 단일 소스 구조. 사용자는 매물 데이터를 읽기만 가능하다. | 사용자가 매물 데이터에 쓰기를 수행하는 경로가 존재하지 않는다. 핵심 엔티티(매물) 자체가 시스템(배치) 전용 쓰기 대상이므로, 동시 쓰기 충돌·트랜잭션 격리·락 전략 선택이 발생할 비즈니스 시나리오 자체가 구조적으로 부재하다. | 포트폴리오에서 증명할 수 있는 설계 의사결정의 종류가 구조적으로 제한된다. 동시성 제어, 트랜잭션 설계, 데이터 정합성 관련 기술적 의사결정을 적용할 비즈니스 맥락이 없다. |
| **포트폴리오 트러블슈팅 패턴** | 기존 포트폴리오 트러블슈팅 6건 중 5건이 "느린 것을 측정 → 빠르게 만듦 → 수치 제시" 패턴으로 구성되어 있다. IN절 Chunking 1건만이 V$SQL 실행계획 분석 → Chunk 크기별 비교 → 트레이드오프 기반 선택 구조를 갖춘다. | 성능 측정/개선은 보여주지만, 비즈니스 요구사항 간 충돌에서 대안 간 트레이드오프를 분석하고 선택 근거를 명시하는 엔지니어링 의사결정이 아니다. | 면접관이 "설계 역량"이 아닌 "측정 역량"으로 평가할 위험이 있다. 취업박람회 멘토 2인이 공통으로 지적한 "설계 능력 = 신입 최우선 역량" 기준에 미달한다. |
| **데이터 갱신 빈도** | 국토교통부 API는 전월 기준 실거래 데이터를 제공하므로, BatchScheduler가 월 1회 실행된다. 두 배치 사이의 기간에는 매물 데이터가 정적이다. | 서비스 내 데이터가 동적으로 변경되는 상황이 월 1회 배치 시점에만 발생한다. 사용자 행위에 의한 데이터 변경이 없으므로, 사용자 간 데이터 상호 영향 시나리오가 존재하지 않는다. | 실서비스에서 요구되는 "지속적으로 동적으로 데이터가 변경되는 상황"을 포트폴리오에서 보여줄 수 없다. |

### 기존의 기술적 한계점

**동시성 제어 영역**

- synchronized와 ReentrantLock이 Wherehouse 프로젝트의 비즈니스 로직 맥락에서 기능별로 구분 적용된 적이 없다.
- DB 수준 락 기술(SELECT FOR UPDATE, 낙관적 락/@Version, gap lock)을 학습 및 비즈니스에 반영하지 않았다.
- ChatService의 Semaphore 적용은 입장 제어 단일 케이스에 한정되어, "왜 Semaphore인가"에 대한 4종 비교 실측은 존재하나 비즈니스 시나리오 다양성이 부족하다.

**트랜잭션 설계 영역**

- 트랜잭션 경계 설정, 전파 전략(Propagation), 격리 수준(Isolation Level) 선택이 비즈니스 정합성 요구사항에 따라 의도적으로 결정된 적이 없다.
- 설계 명세서 v2의 ReviewWriteService는 @Transactional 단일 적용 수준이며, 배치 트랜잭션과 실시간 서비스 트랜잭션 간 격리 전략이 구체적으로 설계되지 않았다.
- 서로 다른 데이터 소스(배치 vs 사용자)가 동일 엔티티를 수정할 때의 충돌 해소 정책이 부재하다.

**스레드 풀 설계 영역**

- CompletableFuture + ThreadPoolExecutor는 I/O 바운드 외부 API 호출(Kakao Map API 15개 카테고리 병렬)에만 적용되어 있다.
- CPU 바운드 연산에 ForkJoinPool을 적용하거나, 워크로드 특성에 따라 스레드 풀을 분리 설계한 적이 없다.
- 스레드 풀 크기 결정의 정량적 근거(코어 수, I/O 대기 비율, 코어 수 × (1 + W/S) 공식)가 부재하다.

**CQRS 동기화 복잡도 영역**

- 기존 Write-Through 동기화는 리뷰 통계 Hash 1개(`stats:{id}`)만 갱신하는 단순 구조다.
- Redis에 4개 이상의 키(Hash + Sorted Set 복수 + bounds Hash)를 동시 갱신해야 하는 시나리오가 존재하지 않는다.
- 다중 키 갱신 시 부분 실패(일부 키만 성공)에 대한 보상 전략이 설계된 적이 없다.

### 개선 목표

**1차 목표: 사용자 매물 등록 경로를 통한 비즈니스 복잡도 확보**

- 사용자가 매물 정보를 직접 등록·수정·상태변경(거래완료)할 수 있는 쓰기 경로를 추가한다.
- 이 쓰기 경로가 기존 CQRS 아키텍처의 Oracle(Write) → Redis(Read) 전 계층을 관통하도록 설계한다.
- 사용자 등록 매물이 추천 검색 결과에 즉시 반영되어, 등록 행위가 다른 사용자의 추천 결과에 영향을 주는 사용자 간 데이터 상호 영향 구조를 구축한다.

**2차 목표: 포트폴리오 트러블슈팅 구조 전환**

- 매물 등록 기능에서 자연스럽게 발생하는 4건의 기술적 의사결정을 "비즈니스 요구사항 → 대안 분석 → 트레이드오프 → 선택 근거 → 정량 결과" 구조로 문서화한다.
- 기존 포트폴리오 6건(측정→개선 5건 + 트레이드오프 1건)에 신규 4건(트레이드오프 기반 선택 4건)을 추가하여, 전체 10건 중 트레이드오프 기반 선택이 5건으로 과반을 넘기는 구조로 전환한다.
- 의사결정 유형 분포를 변환한다: 유형 2(설계 단계 사전 의사결정) 0건→1건, 유형 3(실측 기반 의사결정) 1건→3건.

## 비즈니스 맥락 및 기능 도출 근거

### 실제 서비스 조사 기반 근거

고도화 기능 설계 논의록(2026-04-06)에서 수행한 7개 부동산 서비스(직방, 다방, 네이버 부동산, 피터팬, 호갱노노, 아실, 부동산플래닛) 종합 분석 결과, 사용자 참여형 매물 등록이 실제 서비스에서 이미 검증된 모델임을 확인했다.

**피터팬의 좋은방 구하기**: 네이버 카페(회원 540만+)에서 출발한 국내 유일의 직거래 전문 플랫폼. 사용자가 직접 매물 정보를 등록하고 직거래하는 것이 핵심 모델이다. 연간 약 75만 건의 매물을 네이버 부동산에 공급하는 공식 CP 파트너이다.

**다방 동네이야기**: 2025년 4월 론칭한 동네 기반 커뮤니티로, 매물 첨부 기능을 지원한다. 사용자가 매물 정보를 직접 공유하는 UGC 구조를 채택했다.

**조사 핵심 결론**: 현재 어떤 단일 앱도 MZ세대 1인 가구의 주거 탐색 여정을 완결하지 못한다. 개인 프로젝트가 노릴 수 있는 가장 큰 틈새 중 하나가 "원룸·빌라 대상 실거주 후기 집중 플랫폼"이며, 사용자가 직접 매물 정보를 등록하는 것은 이 틈새를 채우는 핵심 기능이다.

### 비즈니스 서사

"기존 공공데이터(국토교통부) 기반 추천 서비스에 사용자 참여형 매물 데이터를 결합하는 플랫폼 확장." 국토부 API가 월 1회 전월 기준 실거래 데이터만 제공하는 한계를 사용자 직접 등록으로 보완하여, 실시간에 가까운 매물 정보 갱신을 가능하게 한다. 실 상용 서비스에서는 실제 판매자/임대인만 등록할 수 있지만, 현재는 비즈니스 시나리오 재연을 위한 포트폴리오 프로젝트이므로 누구나 매물 정보를 등록할 수 있는 구조로 설계한다.

### 기능 선정 과정 요약

고도화 기능 설계 논의록에서 4차에 걸친 반복 수정을 통해 4개 쓰기 경로(리뷰 작성, 찜하기+인기알림, 리뷰 투표, 매물 등록/수정/상태변경)가 도출되었다. 구현 시간 제약, 포트폴리오 분량 적정성, 기능 간 맥락 연결성을 종합 판단하여 **매물 등록/수정/상태변경 단일 기능에 집중**하는 방향으로 확정했다.

**매물 등록을 선택한 이유**:

1. 기술적 의사결정의 다양성과 기존 시스템 관통성이 가장 높다. Oracle INSERT + Redis Hash + Sorted Set 5~6개 + bounds 재계산 + 배치 충돌 머지를 전부 건드린다.
2. 기존 CQRS 아키텍처, Write-Through 동기화, Redis Pipeline, CompletableFuture 등 이미 포트폴리오에 있는 기술 스택 위에서 새로운 의사결정이 추가되는 구조다. 완전히 새로운 기능을 붙이는 것이 아니라, 기존 시스템의 확장으로서 자연스럽다.
3. 비즈니스 근거가 실제 서비스 조사에서 검증되었다(피터팬 직거래, 다방 매물 첨부).

**기각된 기능 및 기각 사유**:

| 기능 | 기각 사유 |
|------|---------|
| 찜하기 + 인기 알림 | 매물 등록 단일 심화 방향으로 전환. 추후 여유 시 추가 가능. |
| 리뷰 투표 | 동일하게 제외. |
| 허위매물 신고 | 찜하기와 기술 패턴 동일(Redis 원자적 카운터 + 임계값 판단). 새로운 의사결정 미생성. |
| 가격 변동 알림 (배치 트리거) | 이벤트 발생원이 배치(월 1회)이므로 사용자 간 상호작용 맥락 부재. |
| 양방향 상호작용 (가격 제안/리뷰 반박/공동 관심 그룹) | 프론트엔드 과투자, 백엔드 1인 프로젝트 바운더리 초과, 비즈니스 근거 약함. |

## 새로운 비즈니스 로직 설계

### 사용자 플로우 개요

사용자가 수행하는 핵심 행위는 3가지이다: 매물 등록, 매물 수정, 매물 상태 변경(거래완료).

**매물 등록**: 사용자가 매물 정보(아파트명, 주소, 층수, 면적, 전세금/보증금/월세금, 임대 유형)를 입력한다. 시스템은 불변 속성 5개(시군구코드, 지번, 아파트명, 층, 전용면적)로 MD5 해시를 생성하여 기존 매물과의 중복을 판단한다. 중복이 없으면 데이터를 영속 저장하고 검색 인덱스를 갱신한다. 이 매물은 즉시 추천 검색 결과에 반영된다.

**매물 수정**: 등록자가 가격 등 가변 속성을 수정한다. Oracle UPDATE 후 Redis Hash의 해당 필드를 갱신하고, Sorted Set의 score를 변경한다. 가격 변경이 해당 지역구의 bounds(minPrice/maxPrice)를 변경시키면, 같은 지역구 전체 매물의 정규화 점수가 재계산되어야 한다.

**매물 상태 변경(거래완료)**: 등록자가 "거래완료"를 선택하면, 해당 매물은 추천 검색에서 제외된다. Redis Sorted Set 인덱스에서 제거되고, bounds가 재계산될 수 있다. Oracle에서는 Soft Delete(상태 컬럼 변경)로 처리하여 데이터를 보존한다.

### 핵심 비즈니스 규칙

**규칙 1: 매물 식별자 일관성**

사용자 등록 매물과 배치 매물은 동일한 식별자 생성 전략(MD5 Hash)을 사용한다. 불변 속성 5개(시군구코드, 지번, 아파트명, 층, 전용면적) 조합의 MD5 해시(32자)를 매물 식별자로 사용한다. 이를 통해 사용자가 등록한 매물과 국토부 배치가 가져온 매물이 같은 물리적 매물인지를 판단할 수 있다. 동일 해시가 이미 존재하면 중복 등록으로 판단한다.

**규칙 2: 데이터 소스 구분**

매물 데이터의 출처를 명시적으로 구분한다. 배치에서 수집된 매물인지, 사용자가 직접 등록한 매물인지를 식별할 수 있어야 한다. 등록자 정보도 함께 관리하여, 배치 매물은 시스템이 등록한 것으로, 사용자 매물은 해당 사용자가 등록한 것으로 구분한다. 이 구분은 배치-사용자 데이터 충돌 시 머지 정책의 판단 기준이 된다.

**규칙 3: 사용자 매물의 수정/삭제 권한**

등록자 본인만 매물 정보를 수정하거나 상태를 변경할 수 있다. 인증된 사용자와 매물 등록자의 일치 여부를 검증하여 권한을 제어한다. 배치에서 수집된 매물은 사용자가 수정할 수 없다.

**규칙 4: 매물 상태 관리**

매물은 다음 상태를 가진다: 활성(검색 노출), 거래완료(검색 비노출), 삭제(검색 비노출). 상태 변경은 활성 → 거래완료, 활성 → 삭제만 허용된다. 거래완료 또는 삭제 상태의 매물은 추천 검색 인덱스에서 제거되지만, 데이터 자체는 보존된다(논리 삭제). 이 매물에 이미 작성된 리뷰는 유지되며, 리뷰 조회 시 매물 상태가 함께 표시된다.

**규칙 5: 배치-사용자 데이터 공존**

사용자가 등록한 매물과 배치 매물이 동일 해시를 가지는 경우가 발생할 수 있다. 사용자가 먼저 등록하고 이후 배치가 같은 매물을 다른 가격으로 가져오는 경우, 두 데이터 소스 간 충돌이 발생한다. 이 충돌을 어떤 정책으로 해소할 것인가(어느 소스를 우선할 것인가, 양쪽을 어떻게 보존할 것인가)는 비즈니스 정책 결정 사항이다.

**규칙 6: 추천 결과 즉시 반영**

사용자가 매물을 등록하면 해당 매물은 추천 검색 결과에 즉시 반영되어야 한다. 등록 직후 동일 사용자 또는 다른 사용자가 추천 검색을 수행했을 때 등록된 매물이 결과에 포함되어야 한다.

## 예상되는 기술적 도전 과제

매물 등록/수정/상태변경 기능의 구현 과정에서 다음의 기술적 도전 과제가 예상된다.

- **동일 매물 동시 등록**: 복수의 사용자가 같은 물리적 매물을 동시에 등록할 때 중복이 발생하며, 이를 어떤 수준에서 감지하고 처리할 것인가.
- **배치 매물과 사용자 매물의 데이터 충돌**: 사용자가 등록한 매물과 국토부 배치가 가져온 매물이 동일 매물인데 가격이 다를 때, 어느 소스를 기준으로 데이터를 관리할 것인가.
- **영속 저장소와 검색 인덱스 간 정합성**: 매물 등록 시 영속 저장과 검색 인덱스 갱신이 복수의 대상에 걸쳐 수행되며, 일부만 성공하는 부분 실패 상황에서의 데이터 정합성 보장.
- **정규화 기준 변경에 의한 연쇄 점수 재계산**: 신규 매물의 가격이 기존 지역구 가격 범위를 벗어나면 정규화 기준이 변경되어, 해당 지역구 전체 매물의 추천 점수가 재계산되어야 하며 이 연산의 처리 방식 결정.
- **조회 시점과 상태 변경 시점의 불일치**: 사용자가 추천 결과를 조회한 시점과 매물에 접근하는 시점 사이에 다른 사용자가 매물 상태를 변경할 수 있으며, 이 시간 창에서의 데이터 일관성 처리.

## UI/UX 연동 방식 및 정보 표시 전략

### 매물 등록 화면

**입력 폼 영역**: 매물 등록 페이지

사용자가 매물 정보를 입력하는 폼. 기존 추천 서비스의 검색 조건 입력 폼(CharterRecommendationRequestDto, MonthlyRecommendationRequestDto)과 유사한 구조이되, 입력 방향이 반대(검색 조건이 아닌 매물 속성 입력)이다.

필수 입력 항목:
- 임대 유형: 전세 / 월세 (라디오 버튼)
- 아파트명/건물명
- 주소 (시군구코드 + 법정동 + 지번) — 시군구 드롭다운(서울시 25개 자치구) + 법정동 + 지번 텍스트 입력
- 층수
- 전용면적 (㎡) — 평수 자동 변환 표시
- 전세금 (전세인 경우) 또는 보증금 + 월세금 (월세인 경우)

선택 입력 항목:
- 건축연도
- 계약일자

**등록 결과 표시**: 등록 성공 시 "매물이 등록되었습니다. 추천 검색에 즉시 반영됩니다." 메시지와 함께 등록된 매물의 상세 정보를 표시. 중복 매물 감지 시 "동일한 매물이 이미 등록되어 있습니다." 안내.

### 추천 결과 화면 변경사항

**매물 카드 내 데이터 소스 표시**: 추천 결과의 각 매물 카드에 데이터 출처를 표시한다.

- 배치 매물: "국토교통부 실거래 데이터" 뱃지
- 사용자 등록 매물: "사용자 등록" 뱃지
- 병합 매물: "국토교통부 + 사용자 등록" 뱃지

**매물 상태 표시**: 각 매물 카드에 현재 상태(활성/거래완료)를 표시한다. 거래완료 매물은 추천 결과에서 자동 제외되므로 추천 결과 화면에서는 활성 매물만 노출된다. 단, 사용자가 추천 결과를 조회한 시점에는 활성 상태였으나 이후 등록자가 거래완료로 변경한 경우, 사용자가 해당 매물을 클릭했을 때 이미 거래완료 상태일 수 있다. 이 경우 "거래완료된 매물입니다" 안내를 명확히 표시하고, 사용자가 혼란 없이 다른 매물을 탐색할 수 있도록 유도한다.

**수정/상태변경 버튼**: 추천 결과 화면 내 각 매물 카드에서, 본인이 등록한 매물에 한해 "수정" 버튼(가격 등 가변 속성 변경)과 "거래완료" 버튼을 표시한다. 본인 등록 매물이 아닌 경우 해당 버튼은 노출되지 않는다. "거래완료" 클릭 시 확인 다이얼로그를 표시하고, 확인 후 상태를 변경한다.

### 매물 전체 목록 페이지 (신규)

기존 주거지 추천 서비스와 별도로, 등록된 전체 매물을 조회할 수 있는 페이지를 신규 개발한다. 기존 리뷰 게시판(ReviewBoardViewController)과 동일한 게시판 형태로 구성한다.

**매물 목록**: 배치 매물과 사용자 등록 매물을 포함한 전체 매물을 페이지네이션으로 제공한다. 각 매물의 아파트명, 지역구, 가격, 면적, 데이터 출처, 현재 상태, 등록일을 목록에 표시한다.

**검색 기능**: 아파트명 또는 지역구명으로 매물을 검색할 수 있다. 기존 리뷰 게시판의 매물 검색 자동완성(PropertySearchController)과 동일한 패턴을 활용한다.

**수정/상태변경 버튼**: 목록 내 각 매물에 대해, 본인이 등록한 매물에 한해 "수정"/"거래완료" 버튼을 표시한다. 추천 결과 화면에서의 동작과 동일하다.

## 시스템 영향 범위

### 기존 시스템 대비 변경 범위 요약

| 계층 | 기존 | 변경 후 |
|------|------|--------|
| 매물 데이터 소스 | 배치(월 1회) 단일 | 배치 + 사용자 등록 이원 |
| Oracle 테이블 | 매물 테이블에 배치만 UPSERT | 사용자 등록/수정 경로 추가, 데이터 출처·등록자·상태·사용자 제시 가격 관련 컬럼 추가 |
| Redis Hash | 배치 동기화(RdbSyncListener)만 갱신 | 사용자 등록 시 서비스 레이어에서 직접 갱신 경로 추가 |
| Redis Sorted Set | 배치 동기화만 ZADD/ZREM | 사용자 등록/수정/상태변경 시 서비스 레이어에서 직접 ZADD/ZREM 경로 추가 |
| Redis bounds Hash | 배치 완료 후 1회 집계 | 사용자 등록/수정 시 bounds 변경 여부 판단 + 재계산 로직 추가 |
| 추천 점수 산출 | 배치 후 1회 고정 | bounds 변경 시 해당 지역구 전체 매물 점수 재계산 (비동기, CPU 바운드) |
| 동시성 제어 | 부재 | 동일 매물 동시 등록 중복 감지 락 전략 추가 |
| 트랜잭션 경계 | 리뷰 CRUD 단일 @Transactional | 매물 등록(Oracle + Redis 다중 키) 트랜잭션 경계 설계 추가 |
| API | 추천 조회 + 리뷰 CRUD | 매물 등록/수정/상태변경 API 3개 + 내 등록 매물 조회 API 1개 추가 |

### 기존 기능에 대한 영향

**추천 서비스 (RecommendationService)**: 변경 없음. Redis에서 Sorted Set 교집합 → 매물 상세 조회 → 점수 계산 → 정렬의 기존 파이프라인은 그대로 유지된다. 사용자 등록 매물은 Redis 인덱스에 동일한 구조로 적재되므로, 추천 서비스 입장에서는 배치 매물과 사용자 매물을 구분하지 않는다.

**리뷰 시스템 (ReviewService)**: 변경 없음. 리뷰는 매물 식별자(MD5 Hash)를 참조한다. 사용자 등록 매물도 동일한 해시 기반 식별자를 사용하므로, 리뷰 작성/조회 로직에 변경이 필요 없다.

**배치 시스템 (BatchScheduler)**: 변경 필요. 기존 UPSERT 로직에 데이터 출처 확인 로직을 추가한다. 배치가 가져온 매물이 이미 사용자 등록 매물로 존재하는 경우, 비즈니스 규칙 5(배치-사용자 데이터 공존)에 따른 머지 처리를 수행한다.

## 기대 효과

### 포트폴리오 구조 변환

- 기존 포트폴리오 트러블슈팅 6건(측정→개선 5건 + 트레이드오프 1건)에 신규 4건(트레이드오프 기반 선택 4건)이 추가되어 총 10건. 트레이드오프 기반 선택이 5건으로 과반을 넘기는 구조로 전환.
- 의사결정 유형 분포 변환: 유형 1(문제 발생 후 해결) 5건→6건, 유형 2(설계 단계 사전) 0건→1건, 유형 3(실측 기반) 1건→3건.
- 고도화 계획서가 목표한 "포트폴리오 설계 밀도 전환" 달성.

### 누락 역량 해소

- **동시성 제어**: 동일 매물 동시 등록 중복 감지 3종 비교 실측 (DB 레벨 락 신규 적용).
- **트랜잭션 설계**: 배치-사용자 데이터 머지 정책 (비즈니스 정책 기반 설계 단계 사전 의사결정).
- **CQRS 동기화 확장**: Redis 다중 인덱스 부분 실패 전략 (동기화 대상 1개→4개 확장 시 전략 변경).
- **스레드 풀 분리**: I/O 바운드(ThreadPoolExecutor) vs CPU 바운드(ForkJoinPool) 워크로드 특성별 분리 설계.

### 면접 대응력 강화

- 4건의 기술적 의사결정 각각에 대해 "왜 이 방식인가 / 대안은 무엇이었나 / 트레이드오프는 무엇인가"가 명시되어 있으므로, 3분 이내 구술로 즉시 제시 가능.
- 기존 포트폴리오의 트러블슈팅(외부 API 병렬, Redis Pipeline)과 신규 트러블슈팅(ForkJoinPool, 다중 인덱스 동기화)이 상호 참조 구조를 형성하여, "같은 기술인데 맥락에 따라 다른 선택을 한 사례"를 보여줄 수 있다.

## 부록

### 용어 정의

- **MD5 Hash**: 불변 속성 5개(시군구코드, 지번, 아파트명, 층, 전용면적) 조합을 MD5 알고리즘으로 해싱한 32자 문자열. 매물의 고유 식별자로 사용.
- **bounds**: 지역구별 매물 가격/평수의 최솟값·최댓값을 저장한 데이터. 추천 점수 정규화(0~100점)의 기준값.
- **데이터 출처 구분**: 매물 데이터의 출처를 구분하는 속성. 배치(국토부 API), 사용자 등록, 병합(양쪽 데이터 통합) 3가지로 분류.
- **논리 삭제 (Soft Delete)**: 물리적 삭제 대신 상태를 "삭제"로 변경하여 논리적으로 삭제 처리. 데이터는 보존.
