## 1. 대안 B 동시성 제어 메커니즘

### 핵심 구조

`@Transactional` AOP 프록시를 유지하고, findById(1차 필터) + PK 제약(2차 안전망)의 2단 구조로 중복을 감지한다.

```java
@Transactional
public PropertyCreateResponseDto createProperty(...) {

    // ── 1차 필터: 이미 커밋된 행이 있으면 빠르게 거부 ──
    Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
    if (existing.isPresent()) {
        throw new DuplicatePropertyException(...);  // dataSource별 분기 메시지
    }

    // ── 2차 안전망: Oracle PK 제약으로 동시 INSERT 차단 ──
    try {
        entityManager.persist(entity);
        entityManager.flush();    // 즉시 INSERT SQL 실행 강제
    } catch (PersistenceException e) {
        throw new DuplicatePropertyException(...);  // ORA-00001 → 409 변환
    }

    // afterCommit 콜백으로 Redis 동기화
}
```

### 설계 의도

| 구성 요소 | 역할 | 정합성 보장 |
|---------|------|---------|
| 1차 필터 (findById) | 이미 커밋된 행을 INSERT 없이 빠르게 거부 (성능 필터) | 보장하지 않음 — READ COMMITTED 스냅샷 읽기 |
| 2차 안전망 (PK 제약) | Oracle PK 인덱스의 배타적 락이 동일 PK INSERT를 원자적으로 직렬화 | **보장** — TOCTOU 구조 자체가 없음 |

### 직렬화 범위

PK 인덱스의 배타적 락은 **동일 PK에 대한 INSERT끼리만** 경합한다. 서로 다른 PK 요청은 독립적으로 병렬 처리된다.

```
강남구 역삼동 매물 (PK: aaa) ──┐
                               ├── 같은 PK → Oracle 인덱스 락 경합
강남구 역삼동 매물 (PK: aaa) ──┘

서초구 반포동 매물 (PK: bbb) ────── 다른 PK → 독립 병렬 처리
```

### flush 전략

`entityManager.persist()` + `entityManager.flush()`를 사용하여 INSERT SQL을 즉시 실행한다. `save()`만 사용하면 Hibernate가 flush를 `@Transactional` 커밋 시점까지 지연하여 `PersistenceException`이 try-catch 블록 바깥(AOP 프록시 레이어)에서 발생하므로 catch 불가.

**S2에서 실증된 병렬성:**

**S3에서 실증된 선택적 경합:**

**비즈니스 맥락에서의 의미:**
매물 등록의 일반적 부하 패턴은 서로 다른 매물을 서로 다른 사용자가 등록하는 것이다. 대안 B는 이 일반적 패턴에서 요청 간 간섭 없이 병렬 처리하며, 실제 충돌이 발생하는 동일 매물 동시 등록에서만 최소한의 경합이 발생한다.
