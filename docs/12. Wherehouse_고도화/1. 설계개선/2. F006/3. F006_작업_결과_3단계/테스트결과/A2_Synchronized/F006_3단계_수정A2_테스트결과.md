## 1. 수정 A-2 동시성 제어 메커니즘

### 핵심 구조

`@Transactional` AOP 프록시를 제거하고, `synchronized(this)` 블록 내부에서 `TransactionTemplate`으로 프로그래밍 방식 트랜잭션을 사용한다.

```java
// @Transactional 제거
public PropertyCreateResponseDto createProperty(...) {
    // Entity 생성 (synchronized 외부 — 임계 구간 최소화)
    PropertyCharterEntity entity = ...;

    synchronized (this) {
        TransactionTemplate txTemplate = new TransactionTemplate(transactionManager);
        txTemplate.execute(status -> {
            Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
            if (existing.isPresent()) {
                throw new DuplicatePropertyException(...);
            }
            charterRepository.save(entity);
            // afterCommit 콜백 등록
            return null;
        });
        // ← TransactionTemplate이 커밋 완료한 시점
    }
    // ← synchronized 해제. 커밋은 이미 완료됨.
}
```

### 설계 의도

기존 A-2(`@Transactional` + `synchronized`)의 구조적 결함을 해소한다:

| 문제 | 기존 A-2 | 수정 A-2 |
|------|---------|---------|
| 커밋 시점 | `synchronized` 해제 **이후** (AOP 프록시) | `synchronized` 해제 **이전** (TransactionTemplate) |
| 후속 스레드 가시성 | 미커밋 상태 → `findById`에서 "없음" 반환 가능 | 커밋 완료 → `findById`에서 "있음" 확인 보장 |

### 직렬화 범위

`synchronized(this)`가 Spring Bean 인스턴스의 모니터에 걸려 있으므로, **모든 propertyId의 모든 요청**이 이 모니터를 순차적으로 통과해야 한다.

```
강남구 역삼동 매물 (PK: aaa) ──┐
                               ├── 같은 synchronized 모니터 → 직렬 대기
서초구 반포동 매물 (PK: bbb) ──┘
```

### 6-2. 정합성 — 모든 시나리오에서 보장

- 동일 PK 동시 등록 시 "정확히 하나만 성공, 나머지는 409" 보장

- `DataIntegrityViolationException`(500 에러) 발생 건수: **0건**
- `synchronized` 내부에서 `TransactionTemplate`이 커밋을 완료한 후에야 다음 스레드가 진입하므로, 후속 스레드는 커밋된 행을 반드시 확인

### 6-3. 직렬화 범위 — 구조적 한계

수정 A-2의 `synchronized(this)`는 모든 propertyId의 모든 요청을 단일 모니터로 직렬화한다.

**S2에서 실증된 문제:**
- PK가 완전히 다른 5건의 요청이 21ms → 46ms로 계단식 증가
- per-request 직렬 대기: 약 6ms/건
- 마지막 스레드 추가 대기: 25ms

**S3에서 실증된 문제:**
- 충돌 가능성이 있는 2건(동일 PK)과 충돌 가능성이 없는 8건(각각 다른 PK)이 동일하게 직렬 처리
- 10건 전체가 23ms → 70ms로 계단식 증가
- per-request 직렬 대기: 약 5~7ms/건
- 마지막 스레드 추가 대기: 47ms

**비즈니스 맥락에서의 의미:**
매물 등록의 일반적 부하 패턴은 서로 다른 매물을 서로 다른 사용자가 등록하는 것이다. 수정 A-2는 이 일반적 패턴에서도 모든 요청을 직렬 처리하므로, 동시 등록 수가 증가할수록 대기 시간이 선형으로 증가한다.
