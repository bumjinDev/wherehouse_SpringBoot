## Test A — SELECT 무락 실증

### 목적

READ COMMITTED에서 일반 SELECT가 락을 걸지 않음을 확인한다.

### 실행 절차

1. 세션 1: 기존 PK(`7eca28e...`)로 SELECT 실행 (커밋 안 함)
2. 세션 2: 동일 PK로 SELECT 실행
3. V$LOCK 조회

### 결과

#### SQL Developer

| 세션 | 동작 | 결과 |
|------|------|------|
| 세션 1 | `SELECT * FROM PROPERTIES_CHARTER WHERE PROPERTY_ID = '7eca28e5901efdf74eeef00cc11007bf'` | 즉시 1건 반환 |
| 세션 2 | 동일 SELECT 실행 | 즉시 1건 반환, block 없음 |

#### V$LOCK 조회 결과

```
인출된 모든 행: 0행
```

**0행** — SELECT는 TX/TM 락을 생성하지 않는다.

#### JMeter (2 threads, SyncTimer 동시 실행)

| 항목 | 결과 |
|------|------|
| Thread 1 | 성공 (녹색) |
| Thread 2 | 성공 (녹색) |
| 응답 시간 차이 | 수 ms 이내 |

### 결론

READ COMMITTED에서 일반 SELECT는 락을 걸지 않는다. 두 세션이 동일 PK를 동시 조회해도 상호 간섭 없이 즉시 응답한다. 이것이 TOCTOU 갭의 원인이다 — `findById()`가 락을 걸지 않으므로 두 스레드가 동시에 "없음"을 받을 수 있다.

## Test B/C — INSERT 배타적 락 + PK 유니크 제약 위반 실증

### 목적

동일 PK로 두 세션이 INSERT할 때, 하나의 연속된 흐름 안에서 두 가지를 확인한다:
- **(B) 배타적 락**: 후착 세션이 선착 트랜잭션 종료까지 block되는가
- **(C) 유니크 제약 위반**: 선착 커밋 후 후착이 ORA-00001로 거부되는가

### 실행 절차

```
① 세션 1: INSERT 실행 (커밋 안 함)
② 세션 2: 동일 PK INSERT 실행 → block 발생  ← (B) 배타적 락 관찰 지점
③ V$LOCK 조회로 락 상태 확인
④ 세션 1: COMMIT
⑤ 세션 2: block 해제 → ORA-00001 수신       ← (C) 유니크 제약 위반 관찰 지점
```

### ① 세션 1 — INSERT (커밋 전)

```sql
INSERT INTO PROPERTIES_CHARTER (
    PROPERTY_ID, APT_NM, EXCLU_USE_AR, FLOOR, BUILD_YEAR,
    DEAL_DATE, DEPOSIT, LEASE_TYPE, UMD_NM, JIBUN,
    SGG_CD, ADDRESS, AREA_IN_PYEONG, DISTRICT_NAME, LAST_UPDATED,
    DATA_SOURCE, STATUS, REGISTERED_USER_ID, REGISTERED_AT
) VALUES (
    'F006_2_TESTBC_000000000000000001',
    'F006_락테스트아파트', 59.96, 10, 2020,
    '2026-04-30', 30000, '전세', '역삼동', '123-45',
    '11680', '강남구 역삼동 123-45', 18.14, '강남구', SYSDATE,
    'USER', 'ACTIVE', 'testUser_lock', SYSTIMESTAMP
);
-- 커밋하지 않음
```

결과: `1 행 이(가) 삽입되었습니다.`

### ② 세션 2 — 동일 PK INSERT (block 발생)

```sql
-- 세션 1과 동일한 INSERT 문 실행
-- → 커서가 멈추고 응답하지 않음 (block 상태)
```

### ③ V$LOCK 결과 (세션 2가 block된 상태에서 조회)

```
| 행 | SID  | SERIAL# | STATUS   | TYPE | LMODE | REQUEST | BLOCK | OBJECT_NAME        |
|----|------|---------|----------|------|-------|---------|-------|--------------------|
| 1  | 9    |         | INACTIVE | TM   | 3     | 0       | 0     | PROPERTIES_CHARTER |
| 2  | 9    |         | INACTIVE | TX   | 6     | 0       | 1     | (null)             |
| 3  | 1108 |         | ACTIVE   | TM   | 3     | 0       | 0     | PROPERTIES_CHARTER |
| 4  | 1108 |         | ACTIVE   | TX   | 0     | 4       | 0     | (null)             |
| 5  | 1108 |         | ACTIVE   | TX   | 6     | 0       | 0     | (null)             |
```

#### V$LOCK 해석

| 행 | 역할 |
|----|------|
| 1행 (SID=9, TM, LMODE=3) | 세션 1: PROPERTIES_CHARTER 테이블에 DML 작업 중 표시 |
| 2행 (SID=9, TX, LMODE=6, **BLOCK=1**) | 세션 1: PK 인덱스 엔트리를 독점 중이고, **누군가(세션 2)가 이 락 때문에 멈춰 있음** |
| 3행 (SID=1108, TM, LMODE=3) | 세션 2: 테이블에 DML 작업 중 표시 |
| 4행 (SID=1108, TX, LMODE=0, **REQUEST=4**) | 세션 2: 락을 획득하지 못함(LMODE=0). 세션 1의 TX 결과를 확인하기 위해 대기 중(REQUEST=4) |
| 5행 (SID=1108, TX, LMODE=6) | 세션 2: 자기 자신의 트랜잭션 존재 표시 (세션 1의 PK 락과 무관) |

- **SID=9 STATUS=INACTIVE**: SQL 실행은 끝났지만 커밋 안 한 채 대기. 트랜잭션은 열려 있으므로 **락은 유지**.
- **SID=1108 STATUS=ACTIVE**: INSERT가 완료되지 못한 채 block 중.
- **BLOCK=1 (2행)**: Oracle이 세션 2의 대기를 감지하고 세션 1의 BLOCK 값을 자동으로 1로 갱신.

### ④ 세션 1 — COMMIT 실행

```
커밋 완료.
```

### ④-1. COMMIT 후 V$LOCK 결과

```
인출된 모든 행: 0행
```

커밋과 동시에 세션 1의 락(1행, 2행)이 해제되고, 세션 2의 대기(4행)도 해소되어 모든 락이 사라졌다.

### ⑤ 세션 2 — block 해제 → ORA-00001 수신

```
작업이 완료되었습니다.(4098.592초)

오류 보고 -
ORA-00001: 무결성 제약 조건(SCOTT.PK_PROPERTIES_CHARTER)에 위배됩니다
```

- **4098.592초 (~68분)**: 세션 2가 INSERT를 실행한 시점부터 세션 1이 COMMIT할 때까지 block된 총 시간이다. 이 대기 시간 자체가 **PK 인덱스 배타적 락(B)의 직접적 증거**이다.
- 세션 1이 COMMIT하자 PK 값이 확정되었고, 세션 2의 INSERT는 **ORA-00001(C)**로 거부되었다.

### 결론

하나의 흐름에서 두 가지가 실증되었다:

| 관찰 | 증거 | 의미 |
|------|------|------|
| **(B) 배타적 락** | 후착 세션이 4098초 block + V$LOCK에서 BLOCK=1(선착)/REQUEST=4(후착) | 동일 PK INSERT 시 Oracle이 PK 인덱스에 TX 배타적 락을 걸어 후착을 자동 직렬화 |
| **(C) 유니크 제약 위반** | block 해제 후 `ORA-00001: PK_PROPERTIES_CHARTER 위배` | 선착 커밋으로 PK가 확정되면 후착 INSERT는 거부 → "정확히 하나만 성공" 보장 |

## Test D — PK 유니크 제약 없는 상태 동시 INSERT 실증

### 목적

PK 유니크 제약이 없는 테이블에서 동일 값 INSERT 시, block과 ORA-00001 없이 중복 행이 발생함을 확인한다. Test B/C와의 대조를 통해 PK 유니크 제약의 역할을 실증한다.

### 테스트 테이블

```sql
-- PK 없는 임시 테이블
CREATE TABLE TEST_NO_PK (
    PROPERTY_ID VARCHAR2(32),
    APT_NM VARCHAR2(100)
);
-- PK 제약 없음 → 유니크 인덱스 없음
```

### 실행 절차 및 결과

#### ① 세션 1 — INSERT (커밋 안 함)

```sql
INSERT INTO TEST_NO_PK VALUES ('abc...', '테스트아파트'); -- (커밋 안 함)
```

결과: `1 행 이(가) 삽입되었습니다.` (0.043초)

#### ①-1. V$LOCK (세션 1만 INSERT한 상태)

```
| 행 | SID | SERIAL# | STATUS   | TYPE | LMODE | REQUEST | BLOCK | OBJECT_NAME |
|----|-----|---------|----------|------|-------|---------|-------|-------------|
| 1  | 9   | 2824    | INACTIVE | TM   | 3     | 0       | 0     | TEST_NO_PK  |
| 2  | 9   | 2824    | INACTIVE | TX   | 6     | 0       | 0     | (null)      |
```

세션 1의 TM(3) + TX(6) — 여기까지는 PK 있는 테스트와 동일한 패턴이다.

#### ② 세션 2 — 동일 값 INSERT (즉시 성공)

```sql
INSERT INTO TEST_NO_PK VALUES ('abc...', '테스트아파트'); -- → block 없이 즉시 성공
```

결과: `1 행 이(가) 삽입되었습니다.` (0.024초)

**PK 있는 테스트(Test B/C)와의 결정적 차이**: block 없이 즉시 성공했다.

#### ③ V$LOCK (두 세션 모두 INSERT, 미커밋 상태)

```
| 행 | SID  | SERIAL# | STATUS   | TYPE | LMODE | REQUEST | BLOCK | OBJECT_NAME |
|----|------|---------|----------|------|-------|---------|-------|-------------|
| 1  | 9    | 2824    | INACTIVE | TM   | 3     | 0       | 0     | TEST_NO_PK  |
| 2  | 9    | 2824    | INACTIVE | TX   | 6     | 0       | 0     | (null)      |
| 3  | 1108 | 41087   | INACTIVE | TM   | 3     | 0       | 0     | TEST_NO_PK  |
| 4  | 1108 | 41087   | INACTIVE | TX   | 6     | 0       | 0     | (null)      |
```

핵심 관찰:
- **4행** (PK 있는 테스트는 5행) — `LMODE=0, REQUEST=4` 대기 행이 **없다**
- 두 세션 모두 **STATUS=INACTIVE** — 둘 다 INSERT가 즉시 완료됨
- **BLOCK=0 전부** — 아무도 다른 세션을 block하지 않음
- 각 세션이 독립적으로 TM(3) + TX(6)을 보유

PK 유니크 인덱스가 없으므로 Oracle이 "같은 PK 값이 들어오고 있다"는 것 자체를 감지할 방법이 없다. 각 INSERT는 완전히 독립적인 새 행 삽입으로 처리된다.

#### ④ 세션 1 — COMMIT

```
1 행 이(가) 삽입되었습니다.
커밋 완료.
```

V$LOCK: 세션 1의 락(1행, 2행) 해제, 세션 2의 락(3행, 4행)만 남음.

#### ⑤ 세션 2 — COMMIT

```
1 행 이(가) 삽입되었습니다.
커밋 완료.
```

**ORA-00001 없음** — 유니크 제약이 없으므로 위반 검사 자체가 발생하지 않는다.

#### ⑥ 최종 결과 확인

```sql
SELECT * FROM TEST_NO_PK;
```

```
| 행 | PROPERTY_ID | APT_NM     |
|----|-------------|------------|
| 1  | abc...      | 테스트아파트 |
| 2  | abc...      | 테스트아파트 |
```

**2건 반환 — 동일한 값의 중복 행이 발생했다.**

V$LOCK: 0행 (모든 트랜잭션 종료)

### 결론

PK 유니크 제약이 없으면 Oracle은 동일 값 INSERT를 충돌로 인식하지 못한다. 두 세션이 block 없이 즉시 INSERT를 완료하고, 커밋 후 중복 행이 그대로 저장된다.

## 전체 실증 요약

| 테스트 | 실증 대상 | 핵심 결과 | 의미 |
|--------|----------|----------|------|
| Test A | SELECT 무락 | V$LOCK 0행, 두 세션 즉시 응답 | `findById()`는 락 없이 통과 → TOCTOU 갭 발생 원인 |
| Test B/C | INSERT 배타적 락 + PK 유니크 제약 위반 (PK **있음**) | (B) LMODE=6/BLOCK=1, 4098초 block → (C) ORA-00001 | 동일 PK INSERT 시 DB가 자동 직렬화하고, 선착 커밋 후 후착은 거부 |
| Test D | PK 유니크 제약 없는 상태 동시 INSERT (PK **없음**) | block 없음, 두 세션 모두 즉시 성공, 중복 행 2건 | 유니크 제약 없으면 DB가 중복을 방지하지 못함 |

### PK 유니크 제약 유무 비교

| 항목 | PK 있음 (Test B/C) | PK 없음 (Test D) |
|------|-------------------|-----------------|
| V$LOCK 행 수 (미커밋 상태) | 5행 (REQUEST=4 대기 행 포함) | 4행 (대기 행 없음) |
| 후착 INSERT | 4098초 block → ORA-00001 | **즉시 성공 (0.024초)** |
| BLOCK 컬럼 | 선착 BLOCK=1 | 전부 BLOCK=0 |
| 최종 데이터 | 1건 (정확히 하나만 성공) | **2건 (중복 행 발생)** |

**종합 결론:**

현재 운영 테이블(`PROPERTIES_CHARTER`)은 PK 유니크 제약이 설정되어 있으므로, 동시 INSERT 시 "정확히 하나만 성공"이 DB 레벨에서 이미 보장된다. PK 유니크 제약이 없으면 이 보장이 사라지고 중복 행이 발생한다.

따라서 애플리케이션 코드에서는 PK 유니크 제약에 의해 발생하는 `DataIntegrityViolationException`을 catch하여 사용자 친화적 응답(409 Conflict)으로 변환하는 것이 올바른 전략이다.
