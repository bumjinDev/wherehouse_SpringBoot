- 트러블슈팅 나열이 아니라, **요구사항 분석 → 설계 판단 → 구현**의 흐름이 보여야 한다

이에 따라 F006은 "이론적 분석 → 대안 비교 → 선택"이 아니라, **"문제 재현 → 기술 학습 → 실증 비교 → 근거 기반 판단"** 순서로 진행하며, 이 과정 자체를 설계 의사결정 소재로 정리한다.

### 0-2. 현재 코드의 문제

`CharterPropertyWriteService.createProperty()` (MonthlyPropertyWriteService도 동일 구조):

```java
// ① SELECT — 중복 확인
Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
if (existing.isPresent()) {
    throw new DuplicatePropertyException("이미 등록된 전세 매물입니다. ...");
}

// ② INSERT — 실제 저장
charterRepository.save(entity);
```

`propertyId`는 불변 속성 5개(시군구코드·지번·아파트명·층·전용면적)의 MD5 해시로, 애플리케이션에서 사전 산출하여 PK로 주입한다. 동일 입력이면 동일 propertyId가 산출되므로, 두 사용자가 같은 매물을 등록하면 같은 PK로 충돌한다.

현재 코드는 ① 중복 확인과 ② 실제 저장 사이의 경쟁 구간에 대한 동시성 제어가 없다. 두 스레드가 동시에 ①을 통과하면 둘 다 ②로 진행하고, 후착의 `save()`에서 `DataIntegrityViolationException`이 발생하지만 catch되지 않아 500 에러로 빠진다.

---

### 0-3. 비즈니스 불변식과 동시성 제어 범위

F006의 비즈니스 불변식은 **동일 매물의 중복 등록 방지**이다.

본 시스템에서 동일 매물은 시군구코드, 지번, 아파트명, 층, 전용면적의 불변 속성으로부터 생성한 `propertyId`로 식별한다. 동일 `propertyId`를 갖는 매물은 하나만 등록되어야 하며, 이후 동일 `propertyId` 등록 시도는 `409 Conflict`로 거부되어야 한다.

단, 서로 다른 `propertyId`를 갖는 매물 등록 요청 사이에는 비즈니스적 순서 의존성이 없다. 따라서 전체 매물 등록 요청을 요청 도착 순서대로 직렬 처리할 필요는 없으며, 동시성 제어 범위는 동일 `propertyId` 충돌 범위로 제한되는 것이 바람직하다.

이 기준에서 본 작업의 비교 대상은 단순한 응답 시간 차이가 아니라, 각 대안이 **동일 매물 중복 등록 불변식**을 어떤 범위의 직렬화로 보장하는지이다. 전역 직렬화는 정합성을 보장할 수 있으나 서로 다른 매물 등록까지 불필요하게 대기시킬 수 있고, 반대로 동일 `propertyId` 충돌 범위에 한정된 직렬화는 비즈니스 충돌 범위와 기술적 동시성 제어 범위가 일치한다.

## 2단계 — Oracle 락 메커니즘 학습 및 현재 코드에 대한 영향 분석

### 2-1. 목적

Oracle이 SELECT와 INSERT 각각에서 어떤 락을 거는지, PK 유니크 제약이 설정된 상태에서 동작이 어떻게 달라지는지를 학습하고 정리한다. 이 이해가 3단계 대안 비교의 판단 근거가 된다.

### 2-2. 학습 및 정리 항목

#### A. Oracle의 기본 격리 수준과 SELECT 동작

- Oracle 기본 격리 수준: READ COMMITTED
- Spring `@Transactional`의 기본 격리 수준: READ COMMITTED (Oracle 기본과 동일)
- **READ COMMITTED에서 일반 SELECT는 락을 걸지 않는다** — 해당 시점의 커밋된 스냅샷만 읽음
- 결과: 두 트랜잭션이 동시에 `findById(동일PK)`를 실행하면, 상대방의 미커밋 INSERT를 볼 수 없으므로 둘 다 "없음"을 반환

#### B. Oracle의 INSERT 동작과 PK 유니크 제약

- INSERT 실행 시 PK 인덱스에 **배타적 락(exclusive lock)** 획득
- 같은 PK로 두 번째 INSERT 시도 시, 선착 트랜잭션의 커밋/롤백을 **대기(block)**
- 선착 커밋 → 후착은 `ORA-00001: unique constraint violated` 발생
- 선착 롤백 → 후착이 INSERT 성공
- **PK 유니크 제약이 없는 경우**: 두 INSERT 모두 성공하여 중복 행 발생 가능

#### C. PK 유니크 제약 유무에 따른 동작 비교

| 조건 | SELECT 동작 | INSERT 동작 | 동시 등록 결과 |
|------|------------|------------|--------------|
| PK 유니크 제약 있음 (현재) | 락 없음, 둘 다 "없음" 반환 | 후착 block → ORA-00001 | 하나만 INSERT 성공 |
| PK 유니크 제약 없음 (가정) | 락 없음, 둘 다 "없음" 반환 | 둘 다 INSERT 성공 | **중복 행 발생** |

#### D. 현재 코드에 대한 영향

- 현재 PK(`PROPERTY_ID CHAR(32)`)에 유니크 제약이 설정되어 있으므로, DB 레벨에서 "정확히 하나만 INSERT 성공"은 이미 보장되고 있다
- 문제는 이 보호를 **의도적으로 설계한 것이 아니라** 유일 키 목적으로 PK를 설정한 결과 우연히 맞아떨어진 것이며, 애플리케이션 코드에서 `DataIntegrityViolationException`을 처리하지 않아 500 에러로 노출되는 것이다

### 2-3. 실증 테스트 (Oracle SQL 레벨)

Oracle SQL*Plus 또는 DBeaver에서 두 세션을 열어 수동으로 확인한다.

#### PK 유니크 제약이 있는 상태 (현재 운영 테이블 기준)

- 테스트 A: 세션1 `SELECT ... WHERE PROPERTY_ID = 'xxx'` → 세션2 동일 SELECT → 둘 다 결과 없음 확인 (SELECT 무락 실증)
- 테스트 B: 세션1 `INSERT ... ('xxx')` 미커밋 → 세션2 동일 PK INSERT → 세션2 block 확인 (INSERT 시 PK 인덱스 배타적 락 실증)
- 테스트 C: 세션1 커밋 → 세션2 `ORA-00001` 발생 확인 (PK 유니크 제약 위반 실증)

#### PK 유니크 제약이 없는 상태 (임시 테스트 테이블)

대안 B("Oracle PK 제약이 후착을 block해주니까 catch하면 된다")의 전제가 PK 유니크 제약의 존재에 의존하므로, 이 전제가 없을 때의 동작을 대조 실험으로 확인한다.

- 테스트 D: PK 제약 없는 임시 테이블 생성 → 세션1 `INSERT ('xxx')` 미커밋 → 세션2 동일 데이터 INSERT → block 없이 둘 다 성공 → 중복 행 발생 확인

  ```sql
  -- 임시 테이블 (PK 제약 없음)
  CREATE TABLE TEST_NO_PK (
      PROPERTY_ID CHAR(32),
      APT_NM      VARCHAR2(100)
  );
  -- 세션1: INSERT INTO TEST_NO_PK VALUES ('abc...', '테스트아파트');  (커밋 안 함)
  -- 세션2: INSERT INTO TEST_NO_PK VALUES ('abc...', '테스트아파트');  → block 없이 즉시 성공
  -- 세션1: COMMIT;
  -- 세션2: COMMIT;
  -- SELECT COUNT(*) FROM TEST_NO_PK WHERE PROPERTY_ID = 'abc...';  → 2건 (중복 발생)
  ```

- **확인 포인트**: PK 유니크 제약이 없으면 Oracle은 동일 값 INSERT 간에 block을 걸지 않으며, 중복 행이 그대로 적재된다. 이로써 "PK 유니크 제약이 동시 등록 시 정확히 하나만 성공시키는 핵심 메커니즘"임을 실증한다.

## 3-0. 대안 재검토 배경

2단계에서 확인한 사실들이 3단계 대안 설계의 전제를 변경한다.

**확인된 사실 1 — SELECT 무락:** Oracle READ COMMITTED에서 일반 SELECT는 TX/TM 락을 생성하지 않는다(V$LOCK 0행 실증). `findById()`는 락 없이 통과하며, 두 스레드가 동시에 "없음"을 반환받을 수 있다.

**확인된 사실 2 — INSERT 배타적 락:** INSERT 실행 시 Oracle이 PK 인덱스 엔트리에 TX 배타적 락(LMODE=6)을 자동 생성한다. 동일 PK 후착 INSERT는 선착 트랜잭션 종료까지 block된다(REQUEST=4 대기, 4098초 block 실증).

**확인된 사실 3 — PK 유니크 제약의 역할:** 선착 커밋 후 후착에 ORA-00001이 발생하여 "정확히 하나만 INSERT 성공"이 DB 레벨에서 보장된다. PK 유니크 제약이 없으면 block 없이 두 INSERT 모두 성공하여 중복 행이 발생한다(Test D 실증).

이 사실들을 바탕으로 기존 대안 A-1, A-2, B를 재검토한 결과, A-1은 폐기하고 A-2는 구조적 결함을 수정한 변형안으로 교체하며, 비교 축 자체를 재정의한다.


### 3-0-1. 수정 A-2를 최종 채택 후보가 아닌 비교군으로 남기는 이유

2단계 학습 및 실증을 완료한 시점에서, 최종 채택 가능성이 가장 높은 방향은 대안 B이다. Oracle PK 유니크 제약이 동일 PK INSERT를 DB 레벨에서 이미 직렬화하고, 선착 커밋 후 후착을 ORA-00001로 거부한다는 점을 확인했기 때문이다. 따라서 최종 구현 관점에서는 `save()`에서 발생하는 `DataIntegrityViolationException`을 `DuplicatePropertyException`으로 변환하여 `409 Conflict`로 응답하는 방식이 가장 자연스럽다.

그럼에도 수정 A-2를 3단계 실증 비교군으로 남기는 이유는, A-2를 최종 채택하기 위해서가 아니라 **애플리케이션 레벨 상호배제가 중복 확인과 실제 저장 사이의 경쟁 구간을 제거할 수 있는지와, 그 대가로 어떤 직렬화 범위가 발생하는지 확인하기 위해서**이다.

즉, 수정 A-2의 목적은 다음 질문을 검증하는 데 있다.

1. `SELECT(findById)`와 `INSERT(save)` 사이의 간극을 애플리케이션 임계 구역으로 묶으면 동일 매물 중복 등록을 막을 수 있는가?
2. `@Transactional` AOP 방식의 기존 A-2는 왜 `synchronized` 해제 시점과 DB 커밋 시점이 어긋나는가?
3. `TransactionTemplate`으로 커밋 경계를 `synchronized` 내부로 이동하면 단일 JVM 내 정합성은 보장되는가?
4. 그 방식이 정합성을 보장하더라도, 모든 `propertyId` 요청을 하나의 Spring Bean 모니터로 통과시키는 전역 직렬화라는 비용을 발생시키는가?
5. 이 전역 직렬화 비용과 단일 JVM 제약을 고려할 때, 왜 대안 B가 더 적절한 최종 선택인가?

따라서 수정 A-2는 대안 B와 동등한 최종 후보가 아니라, **애플리케이션 전역 락 방식의 가능성과 한계를 드러내기 위한 대조군**이다. A-2 실증 결과는 "A-2도 사용할 수 있다"는 결론을 강화하기 위한 자료가 아니라, "A-2는 단일 JVM 내 정합성은 보장하지만 비즈니스 충돌 범위보다 넓은 요청 범위를 직렬화하므로 최종 채택하지 않는다"는 판단 근거로 사용한다.

이 구분을 명확히 하기 위해 3단계 테스트의 핵심 비교 축은 "수정 A-2와 대안 B 중 어느 쪽이 단순히 더 빠른가"가 아니라, **동일 매물 중복 등록이라는 비즈니스 불변식을 각 대안이 어느 범위의 직렬화로 보장하는가**로 둔다.

## 3-1. 대안 A-1 폐기 — synchronized(SELECT 범위만)

### 기존 설계 의도

`findById()`를 synchronized로 감싸서 SELECT 시점 자체를 순차적으로 실행하면, 후속 INSERT에 영향을 주지 않을까 하는 착안이었다.

```java
@Transactional
public PropertyCreateResponseDto createProperty(...) {
    synchronized (this) {
        Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
        if (existing.isPresent()) {
            throw new DuplicatePropertyException(...);
        }
    }  // ← synchronized 해제. 다른 스레드 진입 가능.

    charterRepository.save(entity);  // ← 비보호 구간에서 INSERT 실행
}
```

### 폐기 근거

A-1의 실패는 두 가지 원인이 겹친 결과다.

---

**실패 원인 ① — synchronized 보호 범위가 중복 확인과 실제 저장 사이의 경쟁 구간을 포함하지 않는다**

synchronized 블록이 CHECK(SELECT)만 감싸고 USE(INSERT)를 포함하지 않는다. Thread A가 synchronized를 해제한 시점부터 INSERT가 실행되는 시점까지의 구간이 비보호 상태로 노출된다.

```
Thread A: [sync 획득] → SELECT(없음) → [sync 해제] → ··········· → INSERT
                                           ↑
                                     이 시점에 Thread B 진입 가능
Thread B:             [sync 대기]  → [sync 획득] → SELECT(없음) → [sync 해제] → INSERT
```

Thread B는 Thread A의 synchronized 해제 직후 진입한다. Thread A의 INSERT가 아직 실행되지 않았거나, 실행되었더라도 미커밋 상태다.

---

**실패 원인 ② — READ COMMITTED에서 미커밋 INSERT는 보이지 않는다**

설령 Thread A가 synchronized 해제 후 INSERT를 먼저 실행했더라도, 그 INSERT는 아직 커밋되지 않은 상태다. Oracle READ COMMITTED에서 Thread B의 SELECT는 Multiversion Read Consistency(CR Clone)에 의해 미커밋 데이터를 볼 수 없다. Thread B도 "없음"을 반환받는다.

---

**결론:** 중복 확인과 실제 저장 사이의 경쟁 구간이 구조적으로 잔존한다. 기대 결과는 현재 코드(1단계)와 동일한 1건 201 + N-1건 500이므로, 실증 테스트 대상에서 제외한다.

---

## 3-2. 대안 A-2 수정 — synchronized(SELECT + INSERT) + TransactionTemplate

이 절은 두 부분으로 나뉜다. 전반부에서 기존 A-2의 구조적 결함을 분석하고, 후반부에서 그 결함을 제거한 수정 A-2를 제시한다.


A-2는 `propertyId` 또는 DB PK를 사용하지 않는 대안이 아니다. A-2에서도 애플리케이션은 동일한 방식으로 `propertyId`를 생성하고, 엔티티의 PK 값으로 주입한 뒤 INSERT를 수행한다.

다만 A-2의 설계 위치는 DB PK 유니크 제약을 중복 방지의 주된 수단으로 활용하는 것이 아니라, 애플리케이션 레벨 상호배제를 통해 `SELECT(findById)`와 `INSERT(save)`, 그리고 `COMMIT`까지의 구간을 하나의 임계 구역으로 묶는 것이다. 즉, A-2는 INSERT 시점의 DB PK 유니크 제약에 의한 충돌 판정보다, `synchronized + TransactionTemplate`을 통해 후속 요청이 선착 요청의 커밋 결과를 반드시 확인하도록 만드는 대안으로 정의한다.

따라서 A-2의 평가는 "PK를 사용하지 않은 설계"가 아니라, **PK 값은 사용하되 중복 방지 책임을 DB PK 제약보다 애플리케이션 전역 상호배제에 둔 설계**로 수행한다. 이 대안은 단일 JVM 내 정합성은 보장할 수 있으나, `synchronized(this)`가 propertyId 단위가 아니라 Spring Bean 인스턴스 단위의 전역 모니터이므로 서로 다른 매물 등록 요청까지 직렬화할 수 있다.

---

### 전반부 — 기존 A-2의 구조적 결함

기존 A-2는 메서드 전체에 `synchronized`를 적용하여 SELECT와 INSERT를 하나의 원자적 블록으로 묶으려 했다.

```java
@Transactional
public synchronized PropertyCreateResponseDto createProperty(...) {
    Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
    if (existing.isPresent()) {
        throw new DuplicatePropertyException(...);
    }
    charterRepository.save(entity);
}  // ← synchronized 해제. 이 이후에 @Transactional AOP 프록시가 커밋을 수행한다.
```

**결함의 본질: JVM 동기화 경계와 DB 트랜잭션 커밋 경계의 불일치**

Spring `@Transactional`의 호출 구조를 먼저 이해해야 한다.

```
Caller → TransactionalProxy.createProperty() → target.createProperty()
                                                      ↑
                                            synchronized는 여기에 걸려 있다
```

| 이벤트 | 수행 주체 | 시점 |
|--------|----------|------|
| synchronized 획득 | target method 진입 시 | 메서드 시작 |
| SELECT + INSERT 실행 | target method 내부 | 메서드 본문 |
| synchronized 해제 | target method return 시 | **메서드 종료** |
| 트랜잭션 커밋 | TransactionalProxy (AOP) | **메서드 종료 이후** |

synchronized는 target method가 return하면 해제된다. 그런데 커밋은 target method return **이후에** TransactionalProxy가 수행한다. 이 사이에 시간 갭이 존재한다.

```
Thread A: [sync 획득] → SELECT(없음) → INSERT(미커밋) → return → [sync 해제] → ··· → [AOP 커밋]
                                                                      ↑
                                                                이 시점에 Thread B 진입
Thread B:                                                       [sync 획득] → SELECT(없음!) → INSERT → ...
```

Thread B가 synchronized를 획득하고 SELECT를 실행하는 시점에, Thread A의 INSERT는 실행 완료되었지만 **아직 커밋되지 않았다.** READ COMMITTED에서 미커밋 INSERT는 보이지 않으므로, Thread B도 "없음"을 반환받고 INSERT로 진행한다.

**결론:** 기존 A-2의 문제는 성능 저하(이중 락)가 아니라, **정합성 보장 자체의 실패**다. JVM synchronized 모니터의 해제 시점과 DB 트랜잭션 커밋 시점이 구조적으로 일치하지 않기 때문이다.

---

### 후반부 — 수정 A-2: TransactionTemplate으로 커밋 경계 일치

기존 A-2의 결함을 제거하는 방법: `@Transactional` AOP 프록시 대신 `TransactionTemplate`을 사용하여, 커밋 시점을 synchronized 블록 **내부**로 끌어들인다.

```java
// @Transactional 제거
public PropertyCreateResponseDto createProperty(...) {
    String propertyId = generateMd5(request);

    synchronized (this) {
        return txTemplate.execute(status -> {
            Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
            if (existing.isPresent()) {
                throw new DuplicatePropertyException(...);
            }
            charterRepository.save(entity);
            return response;
            // ← txTemplate.execute()가 정상 반환하면 여기서 커밋 수행
        });
    }
    // ← synchronized 해제. 이 시점에서 커밋은 이미 완료됨.
}
```

**결함이 해소되는 구조:**

| 이벤트 | 수행 주체 | 시점 |
|--------|----------|------|
| synchronized 획득 | synchronized 블록 진입 시 | 블록 시작 |
| TX 시작 | TransactionTemplate.execute() 진입 시 | 블록 내부 |
| SELECT + INSERT 실행 | 콜백 내부 | 블록 내부 |
| **트랜잭션 커밋** | **TransactionTemplate (콜백 정상 반환 시)** | **블록 내부** |
| synchronized 해제 | synchronized 블록 종료 시 | 블록 종료 |

커밋이 synchronized 블록 내부에서 완료되므로, synchronized 해제 시점에 커밋은 이미 끝나 있다.

```
Thread A: [sync 획득] → [TX 시작] → SELECT(없음) → INSERT → [커밋 완료] → [sync 해제]
                                                                              ↑
                                                                     커밋 완료 후에 해제
Thread B:                                                              [sync 획득] → [TX 시작] → SELECT(있음!) → 409
```

Thread B가 synchronized를 획득하고 SELECT를 실행하면, Thread A의 커밋이 이미 완료되어 있으므로 INSERT된 행이 보인다. Thread B는 `DuplicatePropertyException`을 발생시키고 409를 반환한다.

**수정 A-2의 정합성:** 단일 JVM 내에서 동일 PK 동시 등록 시 "정확히 하나만 성공, 나머지는 409"이 보장된다.

---

## 3-3. 대안 B — Oracle PK 유니크 제약 활용 (catch + 변환)

기존 작업계획서의 대안 B와 동일하다. SELECT를 1차 필터로 사용하고, INSERT 시점의 PK 유니크 제약 위반을 catch하여 사용자 친화적 응답으로 변환한다.

```java
@Transactional
public PropertyCreateResponseDto createProperty(...) {
    // ── 1차 필터: 이미 커밋된 행이 존재하는 경우를 빠르게 거부 ──
    Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
    if (existing.isPresent()) {
        PropertyCharterEntity existingEntity = existing.get();
        if (existingEntity.getDataSource() == DataSource.BATCH) {
            throw new DuplicatePropertyException(
                "이 매물은 국토교통부 실거래 데이터에 이미 등록된 매물입니다.");
        }
        throw new DuplicatePropertyException(
            "동일한 매물이 다른 사용자에 의해 이미 등록되어 있습니다.");
    }

    // ── 2차 안전망: 동시 등록 레이스 대응 — Oracle PK 제약 활용 ──
    try {
        charterRepository.save(entity);
    } catch (DataIntegrityViolationException e) {
        throw new DuplicatePropertyException(
            "동시 등록 충돌: 동일 매물이 다른 사용자에 의해 먼저 등록되었습니다.", e);
    }
}
```

---

### 역할 분리 구조

대안 B에서 SELECT와 INSERT는 서로 다른 책임을 갖는다.

| 구간 | 역할 | 정합성 보장 여부 |
|------|------|----------------|
| SELECT (`findById`) | **1차 필터** — 이미 커밋된 행이 존재하는 경우를 INSERT 없이 빠르게 거부 | 보장하지 않음 (스냅샷 읽기) |
| INSERT (`save`) + PK 제약 | **2차 안전망** — PK 인덱스를 배타적으로 잠그고 동일 PK 존재 여부를 원자적으로 확인 | **보장함** (Current Block 기준 검사) |

SELECT의 역할은 정합성 보장이 아니다. 대부분의 중복 요청(최초 등록 이후의 모든 요청)을 INSERT에 도달하기 전에 차단하여, 불필요한 INSERT 시도 자체를 방지하는 필터다.

정합성 보장의 책임은 INSERT 시점의 PK 유니크 제약에 있다. PK 유니크 제약의 중복 검사는 READ COMMITTED의 스냅샷 읽기가 아니라, PK 인덱스 엔트리에 대한 현재 상태(Current Block) 기준의 검사다. PK 존재 확인과 행 삽입이 하나의 원자적 연산 안에서 수행되므로, 확인 시점과 저장 시점이 분리되는 구조 자체가 존재하지 않는다.

---

### 최초 시점 동시 N건 경합

특정 MD5 propertyId에 대해 DB에 행이 존재하지 않는 최초 시점에 N개 요청이 동시에 도착하는 경우의 동작이다.

**경합 발생 과정:**

```
N개 요청 동시 도착
  → N개 모두 SELECT에서 "없음" 반환 (SELECT 무락)
  → N개 모두 INSERT로 진행
  → 선착 1개가 PK 인덱스 엔트리 배타적 락 획득
  → 나머지 N-1개가 enqueue wait 상태로 대기 (CPU 비소비)
  → 선착 커밋 (수십 ms 이내)
  → N-1개 동시에 깨어남 → 각각 ORA-00001 수신 → catch → 409 변환
```

**경합 해소 이후:**

```
이후 같은 propertyId에 대한 모든 요청
  → SELECT에서 "있음" 확인
  → INSERT에 도달하지 않음
  → 경합 발생 없음
```

주목할 점은 두 가지다. 첫째, N-1개 세션이 하나씩 순차적으로 INSERT를 재시도하는 직렬 대기 체인이 아니라, 전부 같은 선착 TX 락을 대기하므로 선착 커밋 한 번에 N-1개가 동시에 해소된다. 둘째, 최초 시점의 일회성 경합 이후에는 1차 필터(SELECT)가 모든 중복 요청을 차단하므로, 동일 propertyId에 대한 경합은 재발하지 않는다.

---

## 3-4. 비교 축 — 직렬화 범위(Serialization Scope)

### 왜 "성능"이 아니라 "직렬화 범위"인가

수정 A-2와 대안 B는 동일 PK 동시 등록에서 정합성 결과가 동일하다 — 둘 다 1건 성공, 나머지 409. 동일 PK 경합만 측정하면 두 설계의 구조적 차이가 수치에 드러나지 않는다.

두 설계의 구조적 차이가 드러나는 지점은 **"서로 다른 매물 동시 등록"**이다.

---

### 직렬화 범위 비교

**수정 A-2 — 모든 요청이 하나의 모니터를 통과**

synchronized가 `this` 객체의 모니터에 걸려 있으므로, 모든 propertyId에 대한 모든 요청이 이 모니터를 순차적으로 통과해야 한다.

```
강남구 역삼동 매물 등록 (PK: aaa) ──┐
                                     ├── 같은 synchronized 모니터 경유 → 직렬 대기
서초구 반포동 매물 등록 (PK: bbb) ──┘
```

PK가 완전히 다르고 충돌 가능성이 없는 두 요청도 직렬 처리된다.

**대안 B — 동일 PK 요청에만 경합 발생**

경합이 발생하는 대상은 동일 PK 인덱스 엔트리에만 한정된다. propertyId가 다른 요청들은 서로 다른 PK 인덱스 엔트리에 접근하므로 경합이 발생하지 않는다.

```
강남구 역삼동 매물 등록 (PK: aaa) ── PK 인덱스 엔트리 aaa ── 독립 처리
서초구 반포동 매물 등록 (PK: bbb) ── PK 인덱스 엔트리 bbb ── 독립 처리 (병렬)
```

1단계 테스트 S3(서로 다른 매물 동시 3건)에서 3건 모두 201로 병렬 처리된 것이 이 동작의 실증이다.

---

### 비즈니스 맥락에서의 의미

매물 등록이라는 비즈니스 기능에서 일반적인 부하 패턴은 서로 다른 매물을 서로 다른 사용자가 등록하는 것이다. 동일 매물을 여러 사용자가 동시에 등록하는 빈도는 극히 낮다.

| 부하 패턴 | 발생 빈도 | 수정 A-2 | 대안 B |
|----------|----------|---------|--------|
| 서로 다른 매물 동시 등록 (일반적) | **높음** | 직렬 처리 (불필요한 대기) | **병렬 처리** |
| 동일 매물 동시 등록 (예외적) | 낮음 | 1건 성공 + 나머지 409 | 1건 성공 + 나머지 409 |

수정 A-2는 발생 빈도가 높은 일반적 패턴에서도 모든 요청을 직렬 처리하므로, 충돌이 없는 정상 요청의 처리량을 불필요하게 제한한다.

---

## 3-5. 구조적 한계 — 단일 JVM 제약 (테스트 불요, 설계 판단 근거)

synchronized는 단일 JVM 프로세스 내에서만 유효하다. Spring Boot 애플리케이션을 2개 이상의 인스턴스로 배포하면 각 인스턴스의 synchronized 모니터는 독립적이므로, 인스턴스 간 동시성 제어가 되지 않는다. 대안 B의 PK 유니크 제약은 Oracle DB 레벨에서 동작하므로, 몇 개의 애플리케이션 인스턴스가 동시에 접근하든 동일하게 보호된다.

이 한계는 실증 테스트로 확인할 필요 없이 구조적으로 확정되는 사항이며, 설계 결정에서 "수정 A-2가 단일 JVM에서 정합성을 보장하더라도, 배포 구조 변경 시 보호가 깨지므로 대안 B가 상위 선택지"라는 근거로 사용된다.

---


## 3-5-1. A-2 실증의 해석 범위

수정 A-2의 JMeter 테스트 결과는 운영 환경의 성능 수치를 일반화하기 위한 벤치마크가 아니다. 표본 수가 제한되어 있고, 로컬 단일 인스턴스 환경에서 수행되었으므로 실제 운영 트래픽의 평균 응답 시간, 처리량, p95/p99 지연 시간을 단정하는 용도로 사용하지 않는다.

수정 A-2 실증의 목적은 더 제한적이고 구조적이다. `synchronized(this)`가 적용된 경우, 동일 `propertyId` 요청뿐 아니라 서로 다른 `propertyId` 요청까지 동일한 JVM 모니터를 통과하는지를 확인한다. 이 동작은 통계적 추정이 아니라 코드 구조상 발생하는 동작이며, JMeter 결과는 그 구조적 특성을 관찰한 증거로 사용한다.

따라서 A-2 실증 결과를 해석할 때 다음과 같이 선을 긋는다.

- A-2가 동일 매물 중복 등록을 409로 처리할 수 있다는 점은 정합성 검증 결과로 사용한다.
- A-2가 서로 다른 매물 등록 요청까지 순차 통과시킨다는 점은 직렬화 범위 검증 결과로 사용한다.
- A-2가 운영 환경에서 대안 B보다 몇 ms 또는 몇 % 느리다는 식의 성능 일반화는 하지 않는다.
- A-2는 최종 채택 후보가 아니라, 애플리케이션 전역 상호배제 방식의 한계를 설명하기 위한 비교군으로 해석한다.

이 해석 범위를 전제로, 3-6의 JMeter 비교 테스트는 대안별 성능 우열을 확정하는 테스트가 아니라 **정합성 보장 방식과 직렬화 범위의 차이를 확인하는 테스트**로 설계한다.

## 3-6. JMeter 비교 테스트 설계

### 비교 대상

| 구분 | 대안 | 동시성 제어 메커니즘 | 직렬화 범위 |
|------|------|---------------------|-----------|
| 기준선 | 현재 코드 (1단계 결과) | 없음 (PK 제약만 존재, catch 없음) | — |
| 폐기 | A-1 (SELECT만 synchronized) | JVM synchronized (SELECT 범위) | 실증 불요 (구조적 실패 확정) |
| 비교 1 | 수정 A-2 (synchronized + TransactionTemplate) | JVM synchronized (SELECT + INSERT + 커밋) | 모든 propertyId의 모든 요청 |
| 비교 2 | 대안 B (PK 제약 catch + 변환) | Oracle PK 유니크 제약 | 동일 propertyId 요청에만 한정 |

### 테스트 시나리오

#### 시나리오 1 — 동일 매물 동시 등록 (정합성 검증)

| 항목 | 값 |
|------|-----|
| 스레드 수 | 5 |
| 매물 데이터 | 동일 (5건 모두 같은 불변 속성 5개 → 같은 MD5 propertyId) |
| 검증 대상 | 수정 A-2와 대안 B 모두 "1건 201 + 4건 409"을 보장하는가 |

이 시나리오에서 두 설계의 정합성 보장 여부가 확인된다. 차이가 나지 않아야 한다.

#### 시나리오 2 — 서로 다른 매물 동시 등록 (직렬화 범위 검증)

| 항목 | 값 |
|------|-----|
| 스레드 수 | 5 |
| 매물 데이터 | 각각 다른 (5건 모두 다른 불변 속성 → 5개의 서로 다른 propertyId) |
| 검증 대상 | 직렬화 범위의 차이가 응답 시간에 반영되는가 |

기대 결과:

| 대안 | 기대 응답 코드 | 기대 응답 시간 패턴 |
|------|--------------|-------------------|
| 수정 A-2 | 5건 모두 201 | 총 응답 시간이 단일 요청의 약 5배에 수렴 (순차 처리) |
| 대안 B | 5건 모두 201 | 총 응답 시간이 단일 요청과 유사 (병렬 처리) |

이 시나리오가 두 설계의 직렬화 범위 차이를 수치로 드러낸다. PK 충돌이 없는 정상 요청에서 수정 A-2는 불필요한 직렬 대기를 강제하고, 대안 B는 병렬 처리를 허용한다.

#### 시나리오 3 — 혼합 부하 (실제 운영 근사)

| 항목 | 값 |
|------|-----|
| 스레드 수 | 10 |
| 매물 데이터 | 2건 동일 propertyId + 8건 각각 다른 propertyId |
| 검증 대상 | 실제 운영 환경에 가까운 혼합 부하에서의 처리량 차이 |

기대 결과:

| 대안 | 기대 응답 코드 | 기대 처리 패턴 |
|------|--------------|-------------|
| 수정 A-2 | 동일 PK 2건 중 1건 201 + 1건 409, 나머지 8건 201 | 10건 전체가 순차 처리 |
| 대안 B | 동일 PK 2건 중 1건 201 + 1건 409, 나머지 8건 201 | 충돌 2건만 PK 락 경합, 나머지 8건 병렬 처리 |

응답 코드는 동일하지만, 처리 패턴(총 소요 시간)에서 차이가 드러난다.

### 기록 항목

각 시나리오별로 다음을 기록한다:
- 각 요청의 HTTP 응답 코드 및 응답 본문
- 각 요청의 응답 시간 (ms)
- 전체 테스트의 총 소요 시간
- 서버 로그에서 예외 발생 여부 및 처리 경로 (catch 정상 동작 확인)
- Oracle 테이블에 실제로 INSERT된 행 수

## 3-8. 비교 판단 기준 (4단계 선택에 사용할 기준)

3단계 실증 결과를 바탕으로, 4단계에서 다음 기준에 따라 최종 전략을 선택한다:

| 기준 | 수정 A-2 | 대안 B |
|------|---------|--------|
| 동일 PK 정합성 보장 | 보장 (단일 JVM 내) | 보장 (DB 레벨, 인스턴스 무관) |
| 서로 다른 PK 병렬 처리 | 불가 (모든 요청 직렬화) | 가능 (동일 PK만 경합) |
| 다중 인스턴스 배포 시 | 보호 깨짐 | 동일하게 보호 |
| 구현 복잡도 | TransactionTemplate 도입 + synchronized 관리 | try-catch 1개 추가 |
| 기존 보호 메커니즘 활용 | JVM 동기화를 추가로 도입 | Oracle이 이미 제공하는 PK 제약 활용 |

## 4단계 — 전략 선택 및 구현

### 4-1. 선택 근거 정리

3단계 실증 결과를 바탕으로, 다음 기준에 따라 최종 전략을 선택한다:

- 동시 등록 시 "정확히 하나만 성공, 나머지는 정상 에러 응답" 보장 여부
- Oracle 환경의 기존 보호 메커니즘 활용도
- 구현 복잡도 및 유지보수 부담
- 단일 인스턴스 제약 여부 (synchronized는 단일 JVM에서만 유효)

### 4-2. 구현 작업 (대안 B 선택 예상 기준)

1. `CharterPropertyWriteService.createProperty()` — save() try-catch 추가
2. `MonthlyPropertyWriteService.createProperty()` — 동일 적용
3. findById 사전 체크에서 기존 매물의 `dataSource`에 따라 분기 메시지:
   - BATCH → "이 매물은 국토교통부 실거래 데이터에 이미 등록된 매물입니다"
   - USER → "동일한 매물이 다른 사용자에 의해 이미 등록되어 있습니다"
4. 동시 등록 충돌(catch 경로)의 메시지: "동일 매물이 다른 사용자에 의해 먼저 등록되었습니다"

### 4-3. 구현 후 최종 검증

| 테스트 | 설명 | 기대 결과 |
|--------|------|----------|
| T1 | 정상 등록 (중복 없음) | 201 성공 |
| T2 | 이미 존재하는 배치 매물과 동일 PK 등록 | 409 + "국토교통부" 안내 |
| T3 | 이미 존재하는 사용자 매물과 동일 PK 등록 | 409 + "다른 사용자" 안내 |
| T4 | 동시 2건 동일 매물 (JMeter) | 1건 201, 1건 409 |
| T5 | 동시 5건 동일 매물 (JMeter) | 1건 201, 4건 409 |
