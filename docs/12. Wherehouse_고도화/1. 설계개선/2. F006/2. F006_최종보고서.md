# F006 — 사용자 매물 중복 등록 방지를 위한 동시성 제어 전략 결정

## 1. 설계 질문과 판단 기준

### 1-1. 핵심 설계 질문

사용자 매물 등록 기능에서 해결해야 할 핵심 질문은 다음이다.

> 두 사용자가 동일 매물을 동시에 등록할 때, 시스템은 어떻게 정확히 하나의 등록만 성공시키고 나머지 요청을 정상적인 `409 Conflict` 응답으로 처리할 것인가?

이 질문은 단순히 예외 하나를 catch하는 문제가 아니다. 등록 요청이 동시에 들어왔을 때 어느 계층이 중복 방지의 최종 책임을 가져야 하는지, 애플리케이션 락을 걸 것인지, DB 제약을 신뢰할 것인지, 그리고 그 선택이 서로 다른 매물 등록 요청의 병렬성에 어떤 영향을 주는지를 결정하는 문제다.

### 1-2. 비즈니스 불변식

F006에서 보장해야 하는 비즈니스 불변식은 **동일 매물의 중복 등록 방지**이다.

본 시스템에서 동일 매물은 다음 불변 속성으로부터 생성한 `propertyId`로 식별한다.

- 시군구코드
- 지번
- 아파트명
- 층
- 전용면적

동일한 불변 속성을 가진 매물은 동일한 `propertyId`를 갖는다. 따라서 동일 `propertyId`를 가진 매물은 테이블에 하나만 존재해야 하며, 이후 동일 `propertyId` 등록 시도는 신규 INSERT가 아니라 `409 Conflict`로 거부되어야 한다.

중요한 점은 반대 조건이다. 서로 다른 `propertyId`를 가진 매물 등록 요청 사이에는 비즈니스적 순서 의존성이 없다. 강남구 역삼동 매물과 서초구 반포동 매물은 서로 다른 매물이며, 두 요청을 요청 도착 순서대로 직렬 처리할 이유가 없다. 그러므로 올바른 동시성 제어 전략은 “모든 등록 요청을 순서대로 처리하는 것”이 아니라, **동일 `propertyId` 충돌 범위에서만 직렬화하고 서로 다른 `propertyId` 요청은 병렬 처리하는 것**이다.

### 1-3. 비교 기준

이번 의사결정의 비교 기준은 단순 응답 시간이 아니라 다음 네 가지다.

| 기준 | 의미 |
|---|---|
| 정합성 | 동일 `propertyId` 동시 등록 시 정확히 하나만 성공하는가 |
| 실패 응답 품질 | 후착 요청이 500이 아니라 도메인 의미가 있는 409로 변환되는가 |
| 직렬화 범위 | 충돌 가능성이 있는 동일 `propertyId`만 대기하는가, 아니면 모든 등록 요청이 대기하는가 |
| 운영 확장성 | DB 제약, 트랜잭션 경계, 직렬화 범위 관점에서 유지 가능한가 |

결론부터 말하면, 최종 선택은 **대안 B: Oracle PK 유니크 제약을 최종 안전망으로 사용하고, 애플리케이션은 해당 충돌을 `409 Conflict`로 변환하는 방식**이다. 수정 A-2도 테스트 환경에서는 정합성을 보장하지만, 모든 `propertyId` 요청을 하나의 JVM 모니터로 직렬화한다. 반면 대안 B는 동일 PK INSERT에 대해서만 Oracle PK 인덱스 경합이 발생하므로, 비즈니스 충돌 범위와 기술적 직렬화 범위가 일치한다.

## 2. Oracle 락 메커니즘 실증

### 2-1. 실증 목적

대안 B의 전제는 “DB가 동일 PK INSERT 충돌을 원자적으로 감지하고, 정확히 하나의 INSERT만 성공시킬 수 있다”는 것이다. 이 전제가 사실인지 확인하지 않고 단순히 `DataIntegrityViolationException`을 catch하는 방식으로 구현하면, 설계 판단이 아니라 우연한 예외 처리에 머물게 된다.

따라서 Oracle 환경에서 다음 세 가지를 실증했다.

1. 일반 SELECT, 즉 `findById()`에 해당하는 조회가 락을 거는가?
2. 동일 PK로 두 트랜잭션이 INSERT할 때 후착 INSERT는 어떻게 동작하는가?
3. PK 유니크 제약이 없으면 동일 값 INSERT가 어떻게 달라지는가?

### 2-2. 테스트 환경

| 항목 | 값 |
|---|---|
| DB | Oracle XE |
| 격리 수준 | READ COMMITTED |
| 테스트 도구 | SQL Developer 다중 세션, JMeter |
| 운영 테이블 | `PROPERTIES_CHARTER` |
| 운영 PK | `PROPERTY_ID CHAR(32)` |
| 비교 테이블 | `TEST_NO_PK` |
| 비교 조건 | PK 유니크 제약 없음 |
| 세션 1 SID | 9 |
| 세션 2 SID | 1108 |

독립적인 트랜잭션을 만들기 위해 SQL Developer를 별도 인스턴스 2개로 실행했다. 단순히 새 워크시트를 여는 방식은 같은 SID를 공유할 수 있으므로 락 테스트에 적합하지 않다.

### 2-3. Test A — 일반 SELECT는 락을 걸지 않는다

첫 번째 실험은 READ COMMITTED에서 일반 SELECT가 TX/TM 락을 생성하는지 확인하는 것이다. `CharterPropertyWriteService.createProperty()`의 사전 중복 체크는 결국 `findById(propertyId)`이고, 이는 SQL 관점에서 특정 PK에 대한 일반 SELECT다.

실행 흐름은 다음과 같다.

```sql
-- 세션 1
SELECT *
FROM PROPERTIES_CHARTER
WHERE PROPERTY_ID = '7eca28e5901efdf74eeef00cc11007bf';

-- 세션 2
SELECT *
FROM PROPERTIES_CHARTER
WHERE PROPERTY_ID = '7eca28e5901efdf74eeef00cc11007bf';
```

두 세션 모두 즉시 1건을 반환했고, 상호 block은 발생하지 않았다. 이후 V$LOCK을 조회했을 때 결과는 0행이었다.

```text
인출된 모든 행: 0행
```

이 결과가 의미하는 바는 명확하다. Oracle READ COMMITTED에서 일반 SELECT는 조회 대상 행에 대해 TX/TM 락을 생성하지 않는다. 따라서 `findById()`는 이미 커밋된 행을 빠르게 확인하는 **성능 필터**로는 의미가 있지만, 동시 INSERT 상황에서 정합성을 보장하는 락으로 사용할 수 없다.

즉, 두 트랜잭션이 동시에 같은 `propertyId`로 `findById()`를 실행하면 둘 다 “없음”을 받을 수 있다. 상대 트랜잭션의 미커밋 INSERT는 READ COMMITTED에서 보이지 않기 때문이다.

### 2-4. Test B/C — PK 유니크 제약이 있는 동일 PK INSERT

두 번째 실험은 현재 운영 테이블인 `PROPERTIES_CHARTER`에서 동일 PK로 두 세션이 INSERT할 때 Oracle이 어떻게 동작하는지 확인하는 것이다.

실행 흐름은 다음과 같다.

```text
① 세션 1: 동일 propertyId로 INSERT 실행, 커밋하지 않음
② 세션 2: 같은 propertyId로 INSERT 실행
③ 세션 2가 block되는지 확인
④ V$LOCK으로 락 상태 확인
⑤ 세션 1 COMMIT
⑥ 세션 2가 block 해제 후 어떤 결과를 받는지 확인
```

세션 1은 다음 PK로 INSERT를 수행하고 커밋하지 않았다.

```text
PROPERTY_ID = 'F006_2_TESTBC_000000000000000001'
```

세션 2가 동일 PK로 INSERT를 시도하자 즉시 완료되지 않고 block 상태가 되었다. 이 시점의 V$LOCK 핵심 결과는 다음과 같았다.

| SID | TYPE | LMODE | REQUEST | BLOCK | 의미 |
|---:|---|---:|---:|---:|---|
| 9 | TM | 3 | 0 | 0 | 세션 1이 `PROPERTIES_CHARTER`에 DML 수행 중 |
| 9 | TX | 6 | 0 | 1 | 세션 1이 동일 PK 충돌의 선착 트랜잭션이며 후착을 block 중 |
| 1108 | TM | 3 | 0 | 0 | 세션 2도 DML 수행 시도 중 |
| 1108 | TX | 0 | 4 | 0 | 세션 2가 세션 1의 트랜잭션 결과를 기다리는 중 |
| 1108 | TX | 6 | 0 | 0 | 세션 2 자신의 트랜잭션 표시 |

여기서 중요한 값은 세션 1의 `TX LMODE=6, BLOCK=1`과 세션 2의 `TX REQUEST=4`다. 이는 같은 PK 값에 대한 INSERT 충돌이 발생했고, 후착 세션이 선착 트랜잭션의 결과를 기다리고 있음을 보여준다.

이후 세션 1을 COMMIT하자 세션 2의 block이 해제되었고, 세션 2는 다음 오류를 받았다.

```text
ORA-00001: 무결성 제약 조건(SCOTT.PK_PROPERTIES_CHARTER)에 위배됩니다
```

세션 2의 총 대기 시간은 4098.592초였다. 이 긴 대기 시간은 실험자가 세션 1의 COMMIT을 늦게 수행했기 때문에 발생한 것이며, 핵심은 시간의 길이가 아니라 **후착 INSERT가 선착 트랜잭션 종료까지 대기했다는 사실**이다.

이 실험에서 확인한 메커니즘은 다음과 같다.

1. 동일 PK INSERT가 동시에 발생하면 Oracle은 후착 INSERT를 즉시 성공시키지 않는다.
2. 후착 INSERT는 선착 트랜잭션이 COMMIT 또는 ROLLBACK될 때까지 대기한다.
3. 선착이 COMMIT하면 PK 값이 확정되므로 후착은 `ORA-00001`로 거부된다.
4. 결과적으로 PK 유니크 제약이 있는 테이블에서는 동일 PK 동시 INSERT에서 **정확히 하나만 성공**한다.

여기서 주의할 점은 이 동작이 일반적인 “테이블 전체 락”이 아니라는 점이다. 실질적으로 중요한 것은 동일 유니크 키 값에 대한 충돌 감지다. 즉, 같은 `propertyId`에 대해서는 경합이 발생하지만, 서로 다른 `propertyId`는 같은 PK 엔트리를 두고 다투지 않는다.

### 2-5. Test D — PK 유니크 제약이 없으면 중복 행이 저장된다

세 번째 실험은 PK 유니크 제약이 없는 임시 테이블 `TEST_NO_PK`에서 동일 값 INSERT를 수행하는 것이다. 이 실험은 대안 B의 전제가 “Oracle이 INSERT를 무조건 하나만 성공시킨다”가 아니라, **PK 또는 유니크 제약이 존재할 때만 동일 키 충돌을 감지한다**는 점을 확인하기 위해 필요했다.

임시 테이블은 다음과 같다.

```sql
CREATE TABLE TEST_NO_PK (
    PROPERTY_ID VARCHAR2(32),
    APT_NM VARCHAR2(100)
);
```

세션 1이 다음 INSERT를 수행하고 커밋하지 않았다.

```sql
INSERT INTO TEST_NO_PK VALUES ('abc...', '테스트아파트');
```

세션 2가 동일 값을 INSERT했을 때 결과는 PK 있는 테이블과 달랐다. 세션 2는 block되지 않고 즉시 성공했다.

```text
1 행 이(가) 삽입되었습니다. (0.024초)
```

이후 세션 1과 세션 2를 모두 COMMIT하고 최종 조회를 수행하자 동일 값의 중복 행 2건이 반환되었다.

| 행 | PROPERTY_ID | APT_NM |
|---:|---|---|
| 1 | abc... | 테스트아파트 |
| 2 | abc... | 테스트아파트 |

이 실험은 대안 B의 핵심 전제를 분명하게 만든다. Oracle이 중복을 막아주는 것이 아니라, **PK 유니크 제약이 설정된 현재 운영 테이블이 중복을 막아주는 것**이다. 따라서 최종 전략은 반드시 `PROPERTY_ID`의 PK 유니크 제약이 유지된다는 전제 위에서 성립한다.

### 2-6. Oracle 실증 결론

2단계 실증을 통해 다음 결론을 얻었다.

| 실증 항목 | 결과 | 설계상 의미 |
|---|---|---|
| 일반 SELECT | V$LOCK 0행, 두 세션 즉시 응답 | `findById()`는 동시성 제어 수단이 아니라 1차 필터다 |
| PK 있는 동일 INSERT | 후착 block 후 선착 COMMIT 시 `ORA-00001` | DB가 동일 PK 충돌을 원자적으로 직렬화한다 |
| PK 없는 동일 INSERT | block 없이 양쪽 INSERT 성공, 중복 2건 | 유니크 제약이 없으면 DB는 중복을 방지하지 못한다 |

따라서 현재 운영 테이블의 PK 유니크 제약은 단순한 데이터 모델 제약이 아니라, 동시 등록 상황에서 동일 매물을 정확히 하나만 저장하게 만드는 핵심 동시성 제어 장치다. 애플리케이션이 해야 할 일은 DB의 원자성 보장을 다시 구현하는 것이 아니라, DB가 감지한 충돌 결과를 도메인 응답으로 변환하는 것이다.

## 3. 대안 A-2 — `synchronized + TransactionTemplate`

### 3-1. 왜 A-2를 테스트했는가

수정 A-2는 최종 채택 가능성이 높은 대안이어서 테스트한 것이 아니다. 2단계 Oracle 실증을 마친 시점에서 최종 채택 가능성이 가장 높은 방향은 이미 대안 B였다. Oracle PK 유니크 제약이 동일 PK INSERT를 DB 레벨에서 직렬화하고, 선착 커밋 후 후착을 `ORA-00001`로 거부한다는 점을 확인했기 때문이다.

그럼에도 A-2를 별도 구현하고 JMeter로 검증한 이유는, 애플리케이션 레벨 상호배제가 중복 확인과 실제 저장 사이의 경쟁 구간을 제거할 수 있는지, 그리고 그 대가로 어떤 직렬화 범위를 만들게 되는지 확인하기 위해서다. 즉, A-2는 “최종 후보”라기보다 **애플리케이션 전역 락 방식의 가능성과 한계를 드러내기 위한 대조군**이다.

A-2 테스트에서 확인하려 한 질문은 다음과 같다.

1. `SELECT(findById)`와 `INSERT(save)` 사이의 간극을 애플리케이션 임계 구역으로 묶으면 동일 매물 중복 등록을 막을 수 있는가?
2. 기존 `@Transactional + synchronized` 방식에서 왜 synchronized 해제 시점과 DB 커밋 시점이 어긋나는가?
3. `TransactionTemplate`으로 커밋 경계를 synchronized 내부로 이동하면 후속 스레드가 커밋된 행을 반드시 볼 수 있는가?
4. 그 방식이 정합성을 보장하더라도, 서로 다른 `propertyId` 요청까지 하나의 Spring Bean 모니터로 통과시키는가?
5. 이 직렬화 범위가 비즈니스 충돌 범위인 “동일 propertyId”보다 넓다면, 왜 대안 B가 더 적절한 최종 선택인가?

따라서 A-2 실증 결과는 “A-2도 사용할 수 있다”는 결론을 강화하기 위한 자료가 아니다. A-2는 동일 매물 중복 등록을 409로 처리할 수 있지만, 비즈니스적으로 독립적인 서로 다른 매물 등록까지 직렬화한다. 이 차이를 실측으로 확인해야만, 최종 선택 기준을 단순한 성공/실패가 아니라 **정합성을 어느 범위의 직렬화로 보장하는가**로 설명할 수 있다.

### 3-2. 대안 A-2의 설계 의도

수정 A-2는 애플리케이션 레벨에서 중복 확인과 저장을 하나의 임계 구역으로 묶는 방식이다.

단순히 `@Transactional` 메서드 내부에 `synchronized`를 사용하는 방식은 커밋 타이밍 문제가 있다. Spring AOP 기반 `@Transactional`은 프록시가 메서드 반환 이후 커밋을 수행한다. 따라서 메서드 내부의 `synchronized` 블록이 먼저 해제되고, DB 커밋은 그 이후에 발생할 수 있다. 이 경우 후속 스레드가 synchronized 블록에 진입했더라도 선착 INSERT가 아직 커밋되지 않았기 때문에 `findById()`에서 “없음”을 받을 수 있다.

수정 A-2는 이 문제를 피하기 위해 `@Transactional` AOP 프록시를 제거하고, `synchronized(this)` 내부에서 `TransactionTemplate`을 직접 실행한다. 이렇게 하면 트랜잭션 커밋이 synchronized 블록 내부에서 완료되고, 그 다음에 모니터가 해제된다.

핵심 구조는 다음과 같다.

```java
public PropertyCreateResponseDto createProperty(...) {
    PropertyCharterEntity entity = ...; // 임계 구역 밖에서 생성

    synchronized (this) {
        TransactionTemplate txTemplate = new TransactionTemplate(transactionManager);
        txTemplate.execute(status -> {
            Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
            if (existing.isPresent()) {
                throw new DuplicatePropertyException(...);
            }

            charterRepository.save(entity);
            // afterCommit 콜백 등록
            return null;
        });
        // 이 시점에는 TransactionTemplate 커밋 완료
    }

    // synchronized 해제 시점에는 DB 커밋 결과가 후속 스레드에 보임
}
```

이 설계의 핵심은 `findById()`와 `save()` 사이의 경쟁 구간을 애플리케이션 모니터로 제거하고, 후속 스레드가 진입하기 전에 선착 트랜잭션의 커밋을 완료시키는 것이다.

### 3-3. A-2의 정합성 검증 결과

수정 A-2는 테스트 환경에서 정합성을 보장했다.

#### S1 — 동일 매물 5건 동시 등록

### 3-4. A-2의 구조적 한계 — 전역 직렬화

수정 A-2의 한계는 정합성이 아니라 직렬화 범위다. `synchronized(this)`는 특정 `propertyId`에 대한 락이 아니라 Spring Bean 인스턴스 하나에 대한 모니터다. 따라서 동일 매물 요청뿐 아니라 서로 다른 매물 등록 요청도 모두 같은 모니터를 통과한다.

이것은 단순한 성능 수치 문제가 아니라 설계 범위 문제다. 매물 등록의 일반적인 부하 패턴은 서로 다른 사용자가 서로 다른 매물을 등록하는 것이다. 수정 A-2는 이 일반 패턴에서도 모든 요청을 직렬 처리하므로, 동시 등록 수가 증가할수록 대기 시간이 선형으로 증가한다.

### 3-5. A-2 평가

수정 A-2의 장단점은 다음과 같다.

| 항목 | 평가 |
|---|---|
| 정합성 | 단일 JVM에서는 동일 PK 동시 등록을 1건 201 + 나머지 409로 처리 가능 |
| 실패 경로 | 후속 요청이 `findById(found=true)`로 감지되어 도메인 예외 처리 가능 |
| 구현 명확성 | synchronized 블록 내부에서 TransactionTemplate 커밋을 완료하므로 커밋 타이밍이 명확함 |
| 직렬화 범위 | 모든 `propertyId` 요청을 하나의 Bean 모니터로 직렬화함 |
| 최종 판단 | 정합성 대조군으로는 유효하지만 최종 채택에는 부적합 |

수정 A-2는 “애플리케이션 레벨 상호배제로도 정합성을 만들 수 있다”는 점을 보여준다. 그러나 비즈니스 충돌 범위는 동일 `propertyId`인데, 기술적 직렬화 범위는 전체 등록 요청이다. 이 불일치 때문에 최종 전략으로는 적합하지 않다.

## 4. 대안 B — Oracle PK 유니크 제약 활용

### 4-1. 대안 B의 설계 의도

대안 B는 DB가 이미 제공하는 PK 유니크 제약의 원자성을 최종 안전망으로 사용한다. 애플리케이션은 `findById()`로 이미 커밋된 중복을 빠르게 거르고, 동시 INSERT 충돌은 `persist + flush` 시점에 발생하는 PK 제약 위반을 잡아 `DuplicatePropertyException`으로 변환한다.

핵심 구조는 다음과 같다.

```java
@Transactional
public PropertyCreateResponseDto createProperty(...) {
    Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
    if (existing.isPresent()) {
        throw new DuplicatePropertyException(...);
    }

    try {
        entityManager.persist(entity);
        entityManager.flush();
    } catch (PersistenceException e) {
        throw new DuplicatePropertyException(
            "동시 등록 충돌: 동일 매물이 다른 사용자에 의해 먼저 등록되었습니다. propertyId=" + propertyId
        );
    }

    // afterCommit 콜백으로 Redis 동기화
}
```

이 설계에서 `findById()`는 정합성 보장 장치가 아니다. 이미 커밋된 중복 행을 빠르게 확인하는 1차 필터다. 실제 동시성 정합성은 Oracle PK 유니크 제약이 보장한다.

## 5. A-2와 B의 핵심 차이

수정 A-2와 대안 B는 동일한 비즈니스 불변식을 목표로 한다. 동일 `propertyId`를 가진 매물은 정확히 하나만 등록되어야 하고, 후속 등록 시도는 서버 오류가 아니라 `409 Conflict`로 응답해야 한다.

두 대안은 동일 PK 동시 등록 시 표면적으로 같은 결과를 만든다. S1 동일 매물 5건 테스트에서 수정 A-2와 대안 B 모두 `1×201 + 4×409`를 반환했다. 따라서 두 대안을 비교할 때 핵심은 “둘 중 어느 쪽이 409를 만들 수 있는가”가 아니다. 두 대안의 차이는 **중복 등록 불변식을 어느 계층에서, 어느 범위의 직렬화로 보장하는가**에 있다.

---

### 5-1. A-2를 테스트한 이유: 정합성 가능성보다 직렬화 비용을 확인하기 위함

수정 A-2는 최종 채택을 전제로 한 대안이 아니라, 애플리케이션 레벨 상호배제가 `findById()`와 INSERT 사이의 경쟁 구간을 제거할 수 있는지 확인하기 위한 비교군이었다.

Oracle READ COMMITTED 환경에서 일반 SELECT는 미커밋 INSERT를 볼 수 없다. 따라서 단순히 `findById()`를 먼저 호출하는 방식만으로는 동시 등록 레이스를 차단할 수 없다. 이 문제를 애플리케이션에서 해결하려면, 중복 확인과 INSERT, 그리고 커밋 완료 시점까지 하나의 임계 구역 안에 넣어야 한다.

수정 A-2는 이 조건을 만족시키기 위해 다음 구조를 사용했다.

```java
synchronized (this) {
    transactionTemplate.execute(status -> {
        Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
        if (existing.isPresent()) {
            throw new DuplicatePropertyException(...);
        }
        charterRepository.save(entity);
        return null;
    });
    // 이 시점에는 TransactionTemplate에 의해 커밋이 완료되어 있음
}
```

이 구조의 핵심은 `synchronized` 자체가 아니라, **커밋 완료 시점이 모니터 해제보다 앞선다는 점**이다. 기존의 `@Transactional` AOP 방식에서는 메서드 반환 이후 프록시 계층에서 커밋이 수행되므로, `synchronized` 블록을 메서드 내부에 두더라도 모니터 해제와 DB 커밋 시점이 어긋날 수 있다. 반면 수정 A-2는 `TransactionTemplate`을 사용하여 커밋을 `synchronized` 블록 내부에서 완료시킨다.

그 결과 동일 PK 요청에서는 선착 요청이 INSERT와 커밋을 끝낸 뒤에야 후속 요청이 진입한다. 후속 요청은 `findById(found=true)`를 확인하고 `DuplicatePropertyException`으로 409를 반환한다. 즉, A-2 테스트는 “애플리케이션 임계 구역으로도 정합성을 만들 수 있는가”에 대해서는 성공을 확인했다.

하지만 이 테스트의 더 중요한 목적은 그 다음 질문이었다.

> 정합성을 만들기 위해 애플리케이션 전역 임계 구역을 사용하면, 실제 비즈니스 충돌 범위보다 더 넓은 요청까지 직렬화하지 않는가?

이 질문 때문에 S2와 S3 테스트가 중요하다. 동일 PK 요청만 보면 A-2와 B의 차이가 드러나지 않는다. 차이는 서로 다른 `propertyId` 요청, 즉 비즈니스적으로 충돌하지 않는 요청을 동시에 넣었을 때 드러난다.

---

### 5-2. A-2의 정합성 확보 방식: 커밋 가시성을 보장하는 전역 순차 처리

수정 A-2는 모든 등록 요청을 하나의 Spring Bean 모니터로 통과시킨다. 따라서 `propertyId`가 같든 다르든, 요청은 다음 순서로 처리된다.

```text
요청 1 진입 → findById → INSERT → 커밋 → 모니터 해제
요청 2 진입 → findById → INSERT 또는 409 → 커밋/종료 → 모니터 해제
요청 3 진입 → ...
```

이 구조에서는 후속 요청이 항상 선행 요청의 커밋 완료 이후에 실행된다. 동일 PK 요청이라면 후속 요청의 `findById()`는 이미 커밋된 행을 볼 수 있으므로 `found=true`가 된다. 이 때문에 A-2의 중복 감지 경로는 DB 제약 위반이 아니라 `findById(found=true)`다.

A-2 테스트 결과는 이 메커니즘을 보여준다.

| 시나리오 | 결과 | 해석 |
|---|---:|---|
| S1 동일 PK 5건 | `1×201 + 4×409`, 135~138ms | 선착 1건 커밋 후 나머지 4건이 `findById(found=true)`로 409 |
| S2 서로 다른 PK 5건 | 5건 모두 201, 21→46ms | 충돌 가능성이 없는 요청까지 모니터에서 순차 처리 |
| S3 혼합 10건 | `9×201 + 1×409`, 23→70ms | 동일 PK 2건뿐 아니라 다른 PK 8건까지 전체 계단식 처리 |

S2 결과가 특히 중요하다. 5건 모두 서로 다른 `propertyId`였으므로 DB 관점에서는 PK 충돌 가능성이 없다. 그런데도 응답 시간이 `21ms → 27ms → 33ms → 39ms → 46ms`로 증가했다. 이는 각 요청이 같은 모니터를 순차적으로 통과했기 때문이다.

S3에서도 같은 패턴이 반복됐다. 실제 충돌 가능성이 있는 요청은 동일 PK 2건뿐이었지만, 충돌 없는 8건까지 같은 순서 대기열에 들어갔다. 결과적으로 전체 응답 시간이 `23ms → 70ms`로 계단식 증가했다.

따라서 A-2의 결론은 다음과 같다.

> 수정 A-2는 정합성을 보장한다. 그러나 정합성을 보장하기 위해 모든 매물 등록 요청을 하나의 임계 구역으로 묶는다. 즉, 비즈니스 충돌 범위는 “동일 propertyId”인데, 기술적 직렬화 범위는 “모든 propertyId”로 확대된다.

---

### 5-3. B의 정합성 확보 방식: 동일 PK INSERT 충돌만 DB 제약으로 해결

대안 B는 `findById()`를 정합성 보장 수단으로 보지 않는다. `findById()`는 이미 커밋된 중복을 빠르게 거부하는 1차 필터일 뿐이다. 실제 동시 등록 충돌은 INSERT 시점의 Oracle PK 유니크 제약에 맡긴다.

대안 B의 처리 구조는 다음과 같다.

```text
1. findById(propertyId)
   - 이미 커밋된 행이 있으면 409
   - 없으면 INSERT 진행

2. INSERT 실행
   - 동일 PK 충돌이 없으면 성공
   - 동일 PK 충돌이 있으면 Oracle PK 제약에 의해 후착 INSERT가 대기 후 ORA-00001

3. ORA-00001을 도메인 예외로 변환
   - DuplicatePropertyException
   - API 응답은 409 Conflict
```

B 테스트에서 중요한 점은, 동일 PK 동시 요청이 모두 `findById(found=false)`를 통과했다는 사실이다. 이는 Oracle READ COMMITTED에서 일반 SELECT가 미커밋 행을 볼 수 없기 때문에 자연스러운 결과다. 즉, B는 `findById()`가 레이스를 막아준다고 가정하지 않는다.

정합성은 그 다음 단계에서 확보된다. 동일 PK로 INSERT가 동시에 시도되면 Oracle PK 유니크 제약이 동일 키 충돌을 원자적으로 판정한다. 선착 INSERT는 성공하고, 후착 INSERT는 선착 트랜잭션 결과를 기다린 뒤 PK 중복이면 `ORA-00001`을 받는다. 서비스 계층은 이 예외를 `DuplicatePropertyException`으로 변환하여 409로 응답한다.

B 테스트 결과는 이 메커니즘을 보여준다.

| 시나리오 | 결과 | 해석 |
|---|---:|---|
| S1 동일 PK 5건 | `1×201 + 4×409`, 149~153ms | 5건 모두 `found=false` 통과 후 PK 제약에서 4건 차단 |
| S2 서로 다른 PK 5건 | 5건 모두 201, 20~21ms | 서로 다른 PK는 경합 없이 병렬 처리 |
| S3 혼합 10건 | `9×201 + 1×409`, 24~32ms | 동일 PK 2건만 경합, 나머지 8건은 병렬 완료 |

S2에서 응답 시간은 `20ms, 20ms, 20ms, 21ms, 21ms`였다. A-2처럼 계단식 증가가 나타나지 않았다. 서로 다른 `propertyId`는 서로 다른 PK 인덱스 엔트리에 접근하므로 충돌하지 않았고, 애플리케이션 레벨에서 하나의 대기열로 묶이지도 않았다.

S3에서는 차이가 더 명확하다. 충돌 없는 8건은 `24~26ms`에 병렬 완료됐고, 동일 PK 2건만 경합했다. 후착 409 요청은 `32ms`로, 성공 요청보다 약 `6~8ms` 늦었다. 이 차이가 동일 PK INSERT 경합으로 인한 대기 시간이다.

따라서 B의 결론은 다음과 같다.

> 대안 B는 정합성을 DB의 PK 유니크 제약에 맡기고, 애플리케이션은 예외를 도메인 응답으로 변환한다. 이 방식은 실제로 충돌하는 동일 propertyId 요청만 경합시키며, 충돌하지 않는 서로 다른 propertyId 요청은 병렬 처리한다.

---

### 5-4. 같은 정합성, 다른 직렬화 범위

두 대안의 차이는 다음 항목으로 정리할 수 있다.

**정합성 보장 위치**  
수정 A-2는 애플리케이션 임계 구역과 `TransactionTemplate` 커밋 경계로 정합성을 만든다. 반면 대안 B는 Oracle PK 유니크 제약을 최종 정합성 보장 지점으로 사용한다.

**`findById()`의 역할**  
수정 A-2에서 `findById()`는 임계 구역 안에서 커밋 완료된 선행 INSERT를 확인한다. 대안 B에서 `findById()`는 이미 커밋된 중복을 빠르게 거부하는 1차 필터다. 대안 B는 `findById()`가 동시 등록 레이스를 차단한다고 보지 않는다.

**동시 등록 충돌 감지 경로**  
수정 A-2는 후속 요청이 `findById(found=true)`를 확인하면서 중복을 감지한다. 대안 B는 후착 INSERT가 Oracle PK 제약 위반(`ORA-00001`)을 받는 시점에 중복을 감지한다.

**동일 PK 5건 결과**  
두 대안 모두 `1×201 + 4×409`로 정합성 결과는 동일하다. 따라서 동일 PK 테스트만으로는 두 설계의 구조적 차이가 드러나지 않는다.

**서로 다른 PK 5건 결과**  
수정 A-2는 `21→46ms`로 계단식 증가가 발생했다. 서로 다른 `propertyId` 요청도 같은 모니터를 통과했기 때문이다. 대안 B는 `20~21ms`로 거의 동시에 완료됐다. 서로 다른 PK 요청은 DB에서도, 애플리케이션에서도 하나의 대기열로 묶이지 않았다.

**혼합 10건 결과**  
수정 A-2는 `23→70ms`로 전체 요청이 계단식 증가했다. 동일 PK 2건뿐 아니라 충돌 없는 8건도 함께 직렬화됐다. 대안 B는 `24~32ms` 범위에서 완료됐고, 동일 PK 2건만 경합했다.

**직렬화 범위**  
수정 A-2의 직렬화 범위는 모든 등록 요청이다. 대안 B의 직렬화 범위는 동일 PK INSERT 충돌 범위다.

**구현 부담**  
수정 A-2는 임계 구역 범위와 트랜잭션 커밋 경계를 함께 관리해야 한다. 대안 B는 PK 제약 위반 예외를 도메인 예외로 변환하는 책임에 집중한다.

**최종 판단**  
수정 A-2는 대조군으로 의미가 있다. 애플리케이션 임계 구역으로도 정합성을 만들 수 있음을 보여줬기 때문이다. 그러나 비즈니스 충돌 범위보다 기술적 직렬화 범위가 넓다. 대안 B는 비즈니스 충돌 범위와 기술적 경합 범위가 일치하므로 최종 채택에 더 적합하다.

---

### 5-5. 최종 판단: 문제의 본질은 락의 존재가 아니라 락의 범위다

A-2와 B의 비교에서 가장 중요한 결론은 “락을 걸어야 하는가”가 아니다. 이미 Oracle PK 유니크 제약은 동일 PK INSERT 충돌에서 필요한 직렬화를 제공한다. 문제는 그 직렬화를 어디까지 확대할 것인가다.

수정 A-2는 애플리케이션 임계 구역으로 문제를 해결한다. 이 방식은 이해하기 쉽고, 단일 실행 환경에서는 동일 PK 중복 등록을 안정적으로 막는다. 그러나 임계 구역이 `propertyId` 단위가 아니라 서비스 객체 단위이기 때문에, 서로 다른 매물 등록까지 하나의 순서 대기열에 넣는다.

대안 B는 DB의 기존 제약 조건을 동시성 제어 수단으로 명시적으로 활용한다. `findById()`는 레이스를 막는 수단이 아니라 이미 존재하는 중복을 빠르게 걸러내는 필터로 한정하고, 실제 동시 INSERT 충돌은 PK 제약 위반을 잡아 도메인 예외로 변환한다.

최종적으로 대안 B를 선택한 이유는 다음과 같다.

1. 동일 매물 중복 등록 방지라는 비즈니스 불변식을 만족한다.
2. 동일 PK 동시 등록에서 정확히 하나만 성공하고 나머지는 409로 변환된다.
3. 서로 다른 매물 등록은 불필요하게 대기하지 않는다.
4. 기술적 경합 범위가 비즈니스 충돌 범위와 일치한다.
5. 이미 존재하는 Oracle PK 유니크 제약의 원자성을 활용하므로 별도의 애플리케이션 전역 락을 도입하지 않아도 된다.

따라서 최종 설계는 다음과 같이 정리한다.

> 동일 매물 중복 등록 방지의 최종 정합성은 Oracle PK 유니크 제약에 맡긴다. 애플리케이션은 `findById()`를 1차 필터로 사용하고, 동시 INSERT 충돌에서 발생하는 PK 제약 위반을 `DuplicatePropertyException`으로 변환하여 `409 Conflict`로 응답한다. 이로써 정합성은 보장하면서도, 충돌하지 않는 서로 다른 매물 등록 요청의 병렬성은 유지한다.

---

## 6. 최종 결론

F006의 결론은 “락을 걸어서 해결했다”가 아니다. 더 정확한 결론은 다음과 같다.

> 동일 매물 중복 등록 방지는 애플리케이션 전역 락으로 해결할 문제가 아니라, 동일 매물을 식별하는 `PROPERTY_ID`의 PK 유니크 제약을 정합성의 최종 안전망으로 삼고, 애플리케이션은 그 충돌 결과를 사용자 친화적인 409 응답으로 변환해야 하는 문제다.

수정 A-2는 단일 JVM에서 정합성을 보장할 수 있었지만, 모든 등록 요청을 하나의 모니터로 통과시키기 때문에 서로 다른 매물 등록까지 직렬화했다. 대안 B는 Oracle PK 제약을 활용하여 동일 PK 요청끼리만 경합시키고, 서로 다른 PK 요청은 병렬 처리했다. 따라서 최종 선택은 대안 B가 타당하다.
