### 0-2. 현재 코드의 문제

`CharterPropertyWriteService.createProperty()` (MonthlyPropertyWriteService도 동일 구조):

```java
// ① SELECT — 중복 확인
Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
if (existing.isPresent()) {
    throw new DuplicatePropertyException("이미 등록된 전세 매물입니다. ...");
}

// ② INSERT — 실제 저장
charterRepository.save(entity);
```

`propertyId`는 불변 속성 5개(시군구코드·지번·아파트명·층·전용면적)의 MD5 해시로, 애플리케이션에서 사전 산출하여 PK로 주입한다. 동일 입력이면 동일 propertyId가 산출되므로, 두 사용자가 같은 매물을 등록하면 같은 PK로 충돌한다.

현재 코드는 ①과 ② 사이의 TOCTOU(Time-Of-Check to Time-Of-Use) 갭에 대한 동시성 제어가 없다. 두 스레드가 동시에 ①을 통과하면 둘 다 ②로 진행하고, 후착의 `save()`에서 `DataIntegrityViolationException`이 발생하지만 catch되지 않아 500 에러로 빠진다.

## 2단계 — Oracle 락 메커니즘 학습 및 현재 코드에 대한 영향 분석

### 2-1. 목적

Oracle이 SELECT와 INSERT 각각에서 어떤 락을 거는지, PK 유니크 제약이 설정된 상태에서 동작이 어떻게 달라지는지를 학습하고 정리한다. 이 이해가 3단계 대안 비교의 판단 근거가 된다.

### 2-2. 학습 및 정리 항목

#### A. Oracle의 기본 격리 수준과 SELECT 동작

- Oracle 기본 격리 수준: READ COMMITTED
- Spring `@Transactional`의 기본 격리 수준: READ COMMITTED (Oracle 기본과 동일)
- **READ COMMITTED에서 일반 SELECT는 락을 걸지 않는다** — 해당 시점의 커밋된 스냅샷만 읽음
- 결과: 두 트랜잭션이 동시에 `findById(동일PK)`를 실행하면, 상대방의 미커밋 INSERT를 볼 수 없으므로 둘 다 "없음"을 반환

#### B. Oracle의 INSERT 동작과 PK 유니크 제약

- INSERT 실행 시 PK 인덱스에 **배타적 락(exclusive lock)** 획득
- 같은 PK로 두 번째 INSERT 시도 시, 선착 트랜잭션의 커밋/롤백을 **대기(block)**
- 선착 커밋 → 후착은 `ORA-00001: unique constraint violated` 발생
- 선착 롤백 → 후착이 INSERT 성공
- **PK 유니크 제약이 없는 경우**: 두 INSERT 모두 성공하여 중복 행 발생 가능

#### C. PK 유니크 제약 유무에 따른 동작 비교

| 조건 | SELECT 동작 | INSERT 동작 | 동시 등록 결과 |
|------|------------|------------|--------------|
| PK 유니크 제약 있음 (현재) | 락 없음, 둘 다 "없음" 반환 | 후착 block → ORA-00001 | 하나만 INSERT 성공 |
| PK 유니크 제약 없음 (가정) | 락 없음, 둘 다 "없음" 반환 | 둘 다 INSERT 성공 | **중복 행 발생** |

#### D. 현재 코드에 대한 영향

- 현재 PK(`PROPERTY_ID CHAR(32)`)에 유니크 제약이 설정되어 있으므로, DB 레벨에서 "정확히 하나만 INSERT 성공"은 이미 보장되고 있다
- 문제는 이 보호를 **의도적으로 설계한 것이 아니라** 유일 키 목적으로 PK를 설정한 결과 우연히 맞아떨어진 것이며, 애플리케이션 코드에서 `DataIntegrityViolationException`을 처리하지 않아 500 에러로 노출되는 것이다

### 2-3. 실증 테스트 (Oracle SQL 레벨)

Oracle SQL*Plus 또는 DBeaver에서 두 세션을 열어 수동으로 확인한다.

#### PK 유니크 제약이 있는 상태 (현재 운영 테이블 기준)

- 테스트 A: 세션1 `SELECT ... WHERE PROPERTY_ID = 'xxx'` → 세션2 동일 SELECT → 둘 다 결과 없음 확인 (SELECT 무락 실증)
- 테스트 B: 세션1 `INSERT ... ('xxx')` 미커밋 → 세션2 동일 PK INSERT → 세션2 block 확인 (INSERT 시 PK 인덱스 배타적 락 실증)
- 테스트 C: 세션1 커밋 → 세션2 `ORA-00001` 발생 확인 (PK 유니크 제약 위반 실증)

#### PK 유니크 제약이 없는 상태 (임시 테스트 테이블)

대안 B("Oracle PK 제약이 후착을 block해주니까 catch하면 된다")의 전제가 PK 유니크 제약의 존재에 의존하므로, 이 전제가 없을 때의 동작을 대조 실험으로 확인한다.

- 테스트 D: PK 제약 없는 임시 테이블 생성 → 세션1 `INSERT ('xxx')` 미커밋 → 세션2 동일 데이터 INSERT → block 없이 둘 다 성공 → 중복 행 발생 확인

  ```sql
  -- 임시 테이블 (PK 제약 없음)
  CREATE TABLE TEST_NO_PK (
      PROPERTY_ID CHAR(32),
      APT_NM      VARCHAR2(100)
  );
  -- 세션1: INSERT INTO TEST_NO_PK VALUES ('abc...', '테스트아파트');  (커밋 안 함)
  -- 세션2: INSERT INTO TEST_NO_PK VALUES ('abc...', '테스트아파트');  → block 없이 즉시 성공
  -- 세션1: COMMIT;
  -- 세션2: COMMIT;
  -- SELECT COUNT(*) FROM TEST_NO_PK WHERE PROPERTY_ID = 'abc...';  → 2건 (중복 발생)
  ```

- **확인 포인트**: PK 유니크 제약이 없으면 Oracle은 동일 값 INSERT 간에 block을 걸지 않으며, 중복 행이 그대로 적재된다. 이로써 "PK 유니크 제약이 동시 등록 시 정확히 하나만 성공시키는 핵심 메커니즘"임을 실증한다.

## 3단계 — 대안 실증 비교

### 3-1. 목적

1단계에서 재현한 문제를 해결하기 위한 대안들을 **실제 코드로 각각 적용하고 JMeter로 동시 테스트**하여, 각 대안의 동작과 한계를 실증적으로 비교한다.

### 3-2. 대안 목록

#### 대안 A-1: 애플리케이션 레벨 `synchronized` — SELECT 범위만

```java
@Transactional
public PropertyCreateResponseDto createProperty(...) {
    ...
    synchronized (this) {
        Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
        if (existing.isPresent()) {
            throw new DuplicatePropertyException(...);
        }
    }  // lock 해제 → 다른 스레드 진입 가능

    charterRepository.save(entity);  // lock 없이 INSERT
    ...
}
```

- **검증 포인트**: SELECT만 직렬화하면, lock 해제 후 ~ INSERT 사이에 다른 스레드가 진입하여 동일하게 "없음"을 볼 수 있는지 확인
- **실패 예상 원인**: 락 범위가 SELECT에 한정되어 있으므로, Thread A가 lock 해제 후 INSERT를 실행하기 전에 Thread B가 lock을 획득하여 SELECT → "없음" 확인 → lock 해제 → INSERT 진행. 둘 다 INSERT로 도달한다.

  ```
  Thread A: [lock 획득] → SELECT(없음) → [lock 해제] → ......... → INSERT
  Thread B:              [lock 대기]  → [lock 획득] → SELECT(없음) → [lock 해제] → INSERT
  ```

- **기대 결과**: TOCTOU 갭 잔존 → 1건 201, 1건 500

#### 대안 A-2: 애플리케이션 레벨 `synchronized` — SELECT + INSERT 전체

```java
@Transactional
public synchronized PropertyCreateResponseDto createProperty(...) {
    ...
    Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
    if (existing.isPresent()) {
        throw new DuplicatePropertyException(...);
    }

    charterRepository.save(entity);
    ...
}  // synchronized 해제 → 이후 @Transactional AOP 프록시가 커밋
```

- **검증 포인트**: SELECT + INSERT를 모두 직렬화해도, `@Transactional` AOP 프록시가 **메서드 return 후에 커밋**하므로 synchronized 해제와 커밋 사이에 갭이 존재하는지 확인
- **실패 예상 원인**: synchronized는 메서드 종료 시 해제되지만, 트랜잭션 커밋은 AOP 프록시 레이어에서 메서드 return 후에 수행된다. 이 사이에 다른 스레드가 진입하면 선착의 미커밋 INSERT를 볼 수 없다 (READ COMMITTED).

  ```
  Thread A: lock 획득 → SELECT(없음) → INSERT(미커밋) → lock 해제 → [AOP 커밋 진행 중...]
  Thread B:                                                         lock 획득 → SELECT(없음!) → INSERT → ...
  ```

- **핵심 학습**: JVM 메모리 가시성(synchronized)과 DB 트랜잭션 가시성(커밋 시점)은 별개의 경계이며, 이 둘이 일치하지 않으면 동시성 갭이 잔존한다
- **기대 결과**: 타이밍에 따라 대부분 정상이나, 정밀한 동시 요청 시 여전히 500 발생 가능

#### 대안 B: Oracle PK 유니크 제약 활용 (catch + 변환)

```java
@Transactional
public PropertyCreateResponseDto createProperty(...) {
    ...
    // 1차 필터: 일반적 중복 시도를 DB 부하 없이 빠르게 거부
    Optional<PropertyCharterEntity> existing = charterRepository.findById(propertyId);
    if (existing.isPresent()) {
        PropertyCharterEntity existingEntity = existing.get();
        if (existingEntity.getDataSource() == DataSource.BATCH) {
            throw new DuplicatePropertyException(
                "이 매물은 국토교통부 실거래 데이터에 이미 등록된 매물입니다.");
        }
        throw new DuplicatePropertyException(
            "동일한 매물이 다른 사용자에 의해 이미 등록되어 있습니다.");
    }

    // 2차 안전망: 동시 등록 레이스 대응 — Oracle PK 제약 활용
    try {
        charterRepository.save(entity);
    } catch (DataIntegrityViolationException e) {
        throw new DuplicatePropertyException(
            "동시 등록 충돌: 동일 매물이 다른 사용자에 의해 먼저 등록되었습니다.", e);
    }
    ...
}
```

- **검증 포인트**: Oracle PK 제약이 INSERT 시점에 후착을 block → 선착 커밋 후 후착에 ORA-00001 → catch하여 409로 변환되는지 확인
- **기대 결과**: 1건 201, 나머지 409 (정상 에러 응답)

## 4단계 — 전략 선택 및 구현

### 4-1. 선택 근거 정리

3단계 실증 결과를 바탕으로, 다음 기준에 따라 최종 전략을 선택한다:

- 동시 등록 시 "정확히 하나만 성공, 나머지는 정상 에러 응답" 보장 여부
- Oracle 환경의 기존 보호 메커니즘 활용도
- 구현 복잡도 및 유지보수 부담
- 단일 인스턴스 제약 여부 (synchronized는 단일 JVM에서만 유효)

### 4-2. 구현 작업 (대안 B 선택 예상 기준)

1. `CharterPropertyWriteService.createProperty()` — save() try-catch 추가
2. `MonthlyPropertyWriteService.createProperty()` — 동일 적용
3. findById 사전 체크에서 기존 매물의 `dataSource`에 따라 분기 메시지:
   - BATCH → "이 매물은 국토교통부 실거래 데이터에 이미 등록된 매물입니다"
   - USER → "동일한 매물이 다른 사용자에 의해 이미 등록되어 있습니다"
4. 동시 등록 충돌(catch 경로)의 메시지: "동일 매물이 다른 사용자에 의해 먼저 등록되었습니다"

### 4-3. 구현 후 최종 검증

| 테스트 | 설명 | 기대 결과 |
|--------|------|----------|
| T1 | 정상 등록 (중복 없음) | 201 성공 |
| T2 | 이미 존재하는 배치 매물과 동일 PK 등록 | 409 + "국토교통부" 안내 |
| T3 | 이미 존재하는 사용자 매물과 동일 PK 등록 | 409 + "다른 사용자" 안내 |

## 5단계 — 설계 결정 문서화

### 5-1. 포트폴리오 페이지 구조 (피드백 반영)

```
[비즈니스 맥락]  사용자 매물 등록 경로 추가 시, 두 사용자가 같은 매물을
                동시에 등록하면 중복 데이터가 발생하는 문제
      ↓
[문제 재현]     JMeter 동시 등록 → 500 에러 발생 확인 (1단계)
      ↓
[기술 학습]     Oracle READ COMMITTED에서 SELECT 무락 / INSERT 시 PK 락 동작 (2단계)
      ↓
[대안 실증]     A-1: synchronized(SELECT만) → 락 범위 부족으로 TOCTOU 잔존
                A-2: synchronized(메서드 전체) → @Transactional 커밋 타이밍 갭으로 실패
                B:   PK 제약 catch → Oracle 기존 보호 활용으로 성공 (3단계)
      ↓
[선택 근거]     DB가 이미 보장하는 원자성을 활용하고,
                애플리케이션은 그 결과를 사용자 친화적 응답으로 변환하는 역할 분담 (4단계)
      ↓
[검증 결과]     동시 5건 → 1건 성공 4건 409, 출처별 분기 메시지 정상 (4단계)
```
