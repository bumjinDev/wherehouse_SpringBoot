# F007 — 배치-사용자 데이터 충돌 처리: 임계값 기반 머지 정책

**작성일:** 2026-05-08  
**핵심 질문:** 배치 매물과 사용자 매물이 동일 MD5로 충돌할 때, 어느 쪽 가격을 신뢰할 것인가?

---

## 1. 문제 정의

### 1-1. 충돌이 왜 발생하는가

매물 PK는 MD5(시군구코드|지번|아파트명|층수|전용면적)로 만든다. 물리적으로 같은 매물이면 누가 등록하든 같은 PK가 나온다.

| 경로 | 주체 | DATA_SOURCE | 시점 |
|------|------|-------------|------|
| 배치 적재 | BatchScheduler → RdbSyncListener | BATCH | 매월 새벽 4시 |
| 사용자 등록 | F001 API → CharterPropertyWriteService | USER | 수시 |

사용자가 먼저 등록한 매물이 다음 달 배치에서 국토부 API에 잡히면, 같은 PK로 충돌한다.

### 1-2. 기존 코드가 뭘 하고 있었나

```java
// RdbSyncListener.saveCharterPropertiesToRdb()
List<PropertyCharter> entities = properties.stream()
        .map(PropertyCharter::from)
        .collect(Collectors.toList());
propertyCharterRepository.saveAll(entities);
```

JPA `saveAll()` → PK 존재하면 `merge()` → UPDATE.

기존 레코드의 DATA_SOURCE를 보지 않고 그냥 덮어쓴다. 사용자가 등록한 매물의 가격, 등록자 정보, USER 상태가 배치에 의해 날아간다.

### 1-3. 시나리오별 정리

```
배치 매물 1건 적재 시:

시나리오 1: 기존 레코드 없음
  → INSERT (그대로)

시나리오 2: 기존 DATA_SOURCE = BATCH
  → UPSERT 덮어쓰기 (그대로 — 배치끼리는 최신 우선)

시나리오 3: 기존 DATA_SOURCE = USER ← 이게 문제
  → 머지 정책 필요

시나리오 4: 기존 DATA_SOURCE = MERGED
  → 이전 머지 위에 재머지 (시나리오 3과 같은 로직)
```

## 2. 머지 정책 대안 비교

### 대안 A: 국토부 무조건 우선

| 항목 | 내용 |
|------|------|
| 동작 | USER 레코드를 배치 데이터로 그냥 덮어씀 |
| 장점 | 가장 단순. 국토부 실거래 데이터 공신력에 기댈 수 있음 |
| 단점 | 사용자 가격이 무조건 날아감. 사용자 입장에서 "내가 등록한 건데 왜 바뀜?"에 대한 추적 수단이 없음 |

### 대안 B: 최신성 무조건 우선

| 항목 | 내용 |
|------|------|
| 동작 | dealDate 비교해서 최신 쪽 가격 채택 |
| 장점 | 직관적 |
| 단점 | 사용자가 가격을 잘못 입력해도 최신이면 그게 채택됨. 국토부 데이터의 신뢰성을 못 씀 |

### 대안 C: 임계값 기반 하이브리드 (선택)

| 항목 | 내용 |
|------|------|
| 동작 | 가격 차이율 산출 → 임계값 이내면 최신성 우선, 초과면 국토부 우선 |
| 장점 | 정상 변동은 최신 반영, 비정상 차이는 공신력 있는 쪽으로 보정. 사용자 원본 가격은 별도 컬럼에 남겨둠 |
| 단점 | 임계값을 잡아야 하고, 적정값은 실데이터 분석이 필요함 |

### 비교표

| 기준 | A: 국토부 무조건 | B: 최신성 무조건 | C: 하이브리드 |
|------|-----------------|-----------------|-------------|
| 정상 변동 대응 | 국토부 채택 (과잉) | 최신 채택 (적정) | 최신 채택 (적정) |
| 비정상 차이 대응 | 국토부 채택 (적정) | 최신 채택 (위험) | 국토부 채택 (적정) |
| 사용자 원본 보존 | 불가 | 불가 | USER_PROPOSED_* 보존 |
| DATA_SOURCE 추적 | BATCH로 덮어써짐 | 판단 불가 | MERGED로 전환 |
| 구현 복잡도 | 낮음 | 낮음 | 중간 |

## 3. 선택 근거

**선택: 대안 C (임계값 기반 하이브리드)**

### 3-1. 왜 이걸 골랐나

1. **정상 변동과 비정상 차이를 나눠야 한다.** 전세 시장에서 같은 매물의 가격이 한 달 사이에 20% 넘게 뛰는 경우는 드물다. 차이가 이 범위 안이면 시장 변동으로 보고, 크게 벗어나면 사용자 입력 오류거나 시점 차이에 의한 괴리로 본다.

2. **사용자 원본을 날리면 안 된다.** 머지가 일어나도 사용자가 처음 넣은 가격은 `USER_PROPOSED_DEPOSIT` / `USER_PROPOSED_MONTHLY_RENT`에 남긴다. 이 컬럼은 설계 명세서 8.1.1에서 미리 만들어 둔 것.

3. **머지됐다는 사실을 데이터로 알 수 있어야 한다.** USER → MERGED 전환으로 "이 매물은 머지를 거쳤다"가 DB 레벨에서 확인 가능.

### 3-2. 임계값: 20%

| 항목 | 값 |
|------|-----|
| 설정 위치 | `application.yml` → `batch.merge.price-diff-threshold-percent` |
| 설정값 | 20 (%) |
| 근거 | 전세 시장 전월 대비 변동률의 정상 범위를 포괄. 실데이터 분석 후 조정 가능 |

**차이율 공식:**
```
diffRate = |배치가격 - 사용자가격| / 사용자가격 × 100
```

**분기:**
```
diffRate ≤ 20% → dealDate 비교해서 최신 가격 채택
diffRate > 20%  → 국토부(배치) 가격 채택
```

## 4. 구현 상세

### 4-1. 변경 파일

| 파일 | 변경 | 내용 |
|------|------|------|
| `BatchMergeProcessor.java` | 신규 | 머지 로직 본체 |
| `RdbSyncListener.java` | 수정 | `saveAll()` 대신 `BatchMergeProcessor` 호출 |
| `PropertyCharter.java` (batch) | 수정 | `userProposedDeposit` 필드 추가 |
| `PropertyMonthly.java` (batch) | 수정 | `userProposedDeposit`, `userProposedMonthlyRent` 필드 추가 |
| `application.yml` | 수정 | `batch.merge.price-diff-threshold-percent: 20` 추가 |

### 4-2. 머지 흐름 (BatchMergeProcessor)

```
배치 매물 1건 처리:

1. findById(propertyId)로 기존 레코드 조회

2. 기존 레코드 없음
   → PropertyCharter.from(batchProp)으로 엔티티 만들어서 save() (INSERT)

3. 기존 DATA_SOURCE = BATCH
   → 마찬가지로 from()으로 새로 만들어서 save() (UPSERT — 기존 동작 그대로)

4. 기존 DATA_SOURCE = USER 또는 MERGED
   4-1. existing.setUserProposedDeposit(기존 deposit) — 사용자 원본 보존
   4-2. diffRate 계산
   4-3. 메인 가격 결정
        ≤ 20% → dealDate 비교, 최신 가격 채택
        > 20%  → 배치 가격 채택
   4-4. dealDate, lastUpdated, modifiedAt 갱신
   4-5. DATA_SOURCE = MERGED
   4-6. save() (UPDATE)
```

### 4-3. 전세/월세 차이

| 항목 | 전세 (Charter) | 월세 (Monthly) |
|------|---------------|---------------|
| 가격 결정 대상 | deposit | deposit + monthlyRent |
| 원본 보존 | USER_PROPOSED_DEPOSIT | USER_PROPOSED_DEPOSIT + USER_PROPOSED_MONTHLY_RENT |
| 차이율 기준 | deposit | deposit (monthlyRent는 동일 로직 독립 적용) |

---

## 5. 동시 쓰기 (배치 + 사용자 동시 등록)

### 5-1. 발생 빈도

| 조건 | 값 |
|------|-----|
| 배치 주기 | 월 1회, 새벽 4시 |
| 사용자 등록 | 주간 수시 |
| 동시 충돌 확률 | 극히 낮음 |

### 5-2. 대응

배치와 사용자가 같은 PK를 동시에 밀어넣는 경우:
- F006에서 구현한 PK 유니크 제약 catch가 그대로 적용됨
- 후착 쪽이 `DataIntegrityViolationException` → `DuplicatePropertyException`으로 변환
- 배치는 다음 건 진행, 사용자는 "이미 등록된 매물" 안내

별도 락은 필요 없다. 배치가 새벽 4시, 사용자가 주간이라 시간대가 겹칠 일이 거의 없고, 겹치더라도 PK 제약이 잡아준다.
