# F008 — RDB 커밋 이후 Redis 동기화 보상 전략 결정

## 1. 설계 질문과 판단 기준

### 1-1. 핵심 설계 질문

전세 매물 등록 기능에서 해결해야 할 핵심 질문은 다음이다.

> 사용자가 매물을 등록할 때 Oracle RDB 저장과 Redis 동기화가 함께 필요하다면, 시스템은 Redis 동기화를 언제 실행해야 하며, Redis 동기화 실패가 발생했을 때 어떤 방식으로 불일치 상태를 추적하고 종료해야 하는가?

이 질문은 단순히 Redis 예외를 `try-catch`로 감싸는 문제가 아니다. Oracle과 Redis는 같은 트랜잭션 경계에 있지 않다. Oracle 저장은 Spring의 `@Transactional` 경계 안에서 commit 또는 rollback될 수 있지만, Redis 명령은 Oracle 트랜잭션의 rollback 대상이 아니다. 따라서 Oracle 저장과 Redis 동기화를 같은 서비스 메서드 안에서 순차 호출하더라도, 두 저장소의 실패 원자성이 자동으로 보장되지는 않는다.

매물 등록 기능의 기존 흐름은 다음과 같았다.

```text
createProperty() 진입
  → Oracle에 매물 저장
  → Redis Hash + ZSet + bounds 동기화
  → 응답 반환
```

정상 상황에서는 이 흐름이 문제 없어 보인다. 사용자는 매물을 등록하고, Oracle에는 매물 본문 데이터가 저장되며, Redis에는 추천 검색과 상세 조회에 필요한 캐시/인덱스 데이터가 저장된다. 그러나 장애 상황에서는 다음과 같은 문제가 발생한다.

```text
Oracle 저장은 성공했지만 Redis 동기화가 실패할 수 있다.
Redis 일부 명령은 성공했지만 이후 명령이 실패할 수 있다.
Oracle 트랜잭션은 rollback되었지만 이미 실행된 Redis 명령은 rollback되지 않을 수 있다.
```

따라서 F008의 핵심은 “Redis 동기화 코드를 어디에 둘 것인가”가 아니라, **RDB 트랜잭션 경계 밖에 있는 외부 저장소 쓰기를 어떤 순서와 보상 전략으로 다룰 것인가**이다.

---

### 1-2. 비즈니스 불변식

F008에서 보장해야 하는 비즈니스 불변식은 **등록된 매물이 사용자의 주요 조회 경로에서 장기간 모순된 상태로 노출되지 않아야 한다**는 것이다.

현재 시스템에서 매물 등록 후 사용자가 매물을 확인하는 경로는 크게 두 가지다.

첫째, Oracle 직접 조회 기반의 매물 목록/관리 조회다. 이 경로는 `PROPERTIES_CHARTER` 테이블을 기준으로 매물 존재 여부를 판단한다.

둘째, Redis 기반 추천 검색이다. 이 경로는 Redis Sorted Set 인덱스를 기준으로 가격, 면적, 지역 조건에 맞는 매물을 찾는다. 따라서 Oracle에 매물이 존재하더라도 Redis ZSet 인덱스에 누락되어 있으면 추천 검색 결과에는 나타나지 않는다.

이 구조에서 다음 상태들은 모두 비즈니스적으로 문제가 된다.

| 상태 | 문제 |
|---|---|
| Oracle에는 ACTIVE 매물이 있지만 Redis에는 없음 | 목록 조회에는 보이지만 추천 검색에서는 누락됨 |
| Oracle에는 ACTIVE 매물이 있고 Redis Hash만 있음 | 상세 조회 경로 일부에서는 보일 수 있으나 추천 인덱스에는 없음 |
| Oracle에는 없는데 Redis Hash가 남아 있음 | DB 기준으로 존재하지 않는 유령 데이터가 Redis에 남음 |
| Redis 인덱스 일부만 반영됨 | 검색 조건에 따라 매물 노출 여부가 달라지는 반쪽짜리 상태가 됨 |

따라서 F008의 불변식은 다음과 같이 정리할 수 있다.

```text
1. Oracle에 commit되지 않은 매물은 Redis에 반영되면 안 된다.
2. Oracle에 commit된 매물의 Redis 동기화가 실패하면 그 실패는 추적 가능해야 한다.
3. Redis 동기화 실패는 무한히 방치되면 안 된다.
4. 일시적 장애라면 시스템이 자동 복구해야 한다.
5. 장기 장애라면 Oracle 매물까지 원복하여 불일치 상태를 종료해야 한다.
```

중요한 점은 1번과 2~5번이 서로 다른 문제라는 것이다. 1번은 **실행 순서**의 문제다. 2~5번은 **보상 전략**의 문제다. Redis 쓰기를 RDB commit 이후로 옮기더라도, commit 이후 Redis가 실패할 가능성은 여전히 남는다. 반대로 실패 기록과 재시도 구조가 있어도, Redis 쓰기가 RDB commit 전에 실행되면 Oracle rollback 후 Redis 유령 데이터가 남을 수 있다.

### 1-3. 판단 기준

이번 의사결정의 기준은 단순히 “정상 등록이 되는가”가 아니다. 정상 상황은 기존 구조에서도 동작했다. 판단 기준은 장애 상황에서의 정합성, 복구 가능성, 실패 응답 의미, 구현 범위의 현실성이다.

| 기준 | 의미 |
|---|---|
| 커밋 전 Redis 쓰기 차단 | Oracle commit 이전에 Redis 데이터가 생성되는 구조를 막는가 |
| rollback 안전성 | Oracle rollback 시 Redis에 유령 데이터가 남지 않는가 |
| 실패 추적성 | Oracle commit 이후 Redis 실패가 기록되어 후속 처리가 가능한가 |
| 자동 복구 가능성 | 일시적 Redis 장애가 해결되면 시스템이 재동기화할 수 있는가 |
| 장기 장애 종료성 | Redis 장애가 장기화될 때 Oracle-only 불일치를 무한히 방치하지 않는가 |
| 변경 범위 | 현재 프로젝트 구조에서 과도한 아키텍처 변경 없이 적용 가능한가 |
| 후속 확장성 | 월세, 수정, 상태 변경, Redis 부분 실패 세분화로 확장 가능한가 |

결론부터 말하면, 최종 선택은 **RDB commit 이후 `afterCommit`에서 Redis 동기화를 실행하고, commit 이후 Redis 실패는 실패 테이블에 기록한 뒤 스케줄러가 재시도 또는 원복하는 방식**이다.

다만 이 결론은 처음부터 바로 정한 것이 아니다. 먼저 기존 구조에서 Redis 전체 실패와 부분 실패를 실증했고, 그 결과 `@Transactional`만으로는 문제가 해결되지 않으며 오히려 Redis 부분 실패에서는 더 위험한 상태가 생긴다는 점을 확인했다. 그 다음 실행 순서 대안을 비교했고, 최종적으로 `afterCommit + 실패 기록 + 스케줄러 보상` 구조를 선택했다.

## 2. 기존 구조의 장애 시나리오 실증

### 2-1. 실증 목적

F008에서 가장 먼저 확인한 것은 기존 구조가 어떤 실패 상태를 만드는지였다. Redis 동기화 실패를 이론적으로만 가정하면 설계 판단이 추상적으로 흐를 수 있다. 따라서 실제 코드 흐름에 인위적으로 예외를 주입하여 Oracle과 Redis의 상태가 어떻게 갈라지는지 확인했다.

테스트는 두 가지로 나누었다.

| 시나리오 | 실패 위치 | 확인하려는 문제 |
|---|---|---|
| Redis 전체 실패 | `syncRedisAfterCreate()` 진입 직후 | Oracle 저장 후 Redis가 아예 반영되지 않는 상태 |
| Redis 부분 실패 | Redis Hash 저장 후 ZSet 추가 직전 | Redis 일부만 반영된 반쪽짜리 상태 |

Redis 동기화 메서드는 다음 작업들을 순차 수행한다.

```text
① Hash 저장      property:charter:{propertyId}
② 가격 ZSet 추가 idx:charterPrice:{districtName}
③ 면적 ZSet 추가 idx:area:{districtName}:전세
④ 가격 bounds 확장 bounds:{districtName}:전세
⑤ 면적 bounds 확장 bounds:{districtName}:전세
```

이 5개 작업은 각각 독립된 Redis 명령이다. 따라서 네트워크 장애, Redis 서버 중지, timeout, OOM 등이 발생하면 어느 지점에서든 이후 명령이 실행되지 않을 수 있다.

---

### 2-2. Redis 전체 실패 — `@Transactional` 미적용

첫 번째 테스트는 Redis 동기화가 아예 시작되지 못하는 상황이다. `syncRedisAfterCreate()` 진입 직후 RuntimeException을 발생시켜 Redis Hash, ZSet, bounds 중 어느 것도 실행되지 않도록 했다.

```java
private void syncRedisAfterCreate(PropertyCharterEntity entity) {
    if (true) {
        throw new RuntimeException("[F008-TEST] 테스트①: Redis 연결 실패 시뮬레이션");
    }

    String propertyId = entity.getPropertyId();
    // 이하 Redis 작업 미실행
}
```

`@Transactional`이 없는 상태에서 실행 흐름은 다음과 같다.

```text
createProperty() 진입
  → charterRepository.save(entity)
      → Oracle INSERT
      → repository 메서드 트랜잭션 commit
  → syncRedisAfterCreate(entity)
      → RuntimeException 발생
      → Redis 작업 미실행
  → API 500 응답
```

결과는 다음과 같았다.

| 확인 대상 | 결과 |
|---|---|
| API 응답 | 500 Internal Server Error |
| Oracle | ACTIVE 매물 1건 존재 |
| Redis Hash | 없음 |
| Redis ZSet | 없음 |

이 상태는 사용자 관점에서 심각하다. 요청은 500으로 실패했지만 Oracle에는 이미 매물이 저장되었다. 사용자가 다시 같은 매물을 등록하면 중복 매물로 거부될 수 있다. 그러나 Redis 인덱스가 없으므로 추천 검색에서는 해당 매물이 노출되지 않는다.

즉, `@Transactional`이 없는 구조에서는 Redis 전체 실패만으로도 다음 상태가 발생한다.

```text
Oracle에는 있음
Redis에는 없음
API는 실패 응답
재등록은 중복으로 차단될 수 있음
추천 검색에서는 누락됨
```

이 상태는 단순 실패가 아니라 자가 복구가 어려운 불일치다.

---

### 2-3. Redis 전체 실패 — `@Transactional` 적용

같은 Redis 전체 실패를 `@Transactional` 적용 상태에서 확인했다. 이 경우 `createProperty()` 진입 시 Spring이 트랜잭션을 시작하고, Oracle INSERT는 메서드가 정상 종료될 때까지 commit되지 않는다.

실행 흐름은 다음과 같다.

```text
createProperty() 진입
  → @Transactional 트랜잭션 시작
  → Oracle INSERT 전송, commit 보류
  → syncRedisAfterCreate(entity)
      → RuntimeException 발생
      → Redis 작업 미실행
  → @Transactional이 예외 감지
  → Oracle ROLLBACK
  → API 500 응답
```

결과는 다음과 같았다.

| 확인 대상 | 결과 |
|---|---|
| API 응답 | 500 Internal Server Error |
| Oracle | 없음 |
| Redis Hash | 없음 |
| Redis ZSet | 없음 |

Redis 전체 실패만 놓고 보면 `@Transactional` 적용은 깨끗한 실패를 만든다. Oracle도 rollback되고 Redis도 실행되지 않았기 때문이다. 그러나 이 결론은 Redis 작업이 시작되기 전에 실패했다는 제한된 조건에서만 성립한다.

실제 문제는 Redis 명령 일부가 이미 성공한 뒤 예외가 발생했을 때 드러난다.

---

### 2-4. Redis 부분 실패 — `@Transactional` 미적용

두 번째 테스트는 Redis Hash 저장이 성공한 뒤, 첫 번째 ZSet 추가 직전에 예외를 발생시키는 방식으로 진행했다.

```java
private void syncRedisAfterCreate(PropertyCharterEntity entity) {
    String propertyId = entity.getPropertyId();
    String districtName = entity.getDistrictName();

    String hashKey = "property:charter:" + propertyId;
    Map<String, Object> hashFields = propertyHashBuilder.buildCharterHash(entity);
    redisHandler.redisTemplate.opsForHash().putAll(hashKey, hashFields);

    if (true) {
        throw new RuntimeException("[F008-TEST] 테스트②: Hash 저장 후 ZSet 추가 시점 실패 시뮬레이션");
    }

    redisHandler.redisTemplate.opsForZSet().add(
            "idx:charterPrice:" + districtName,
            propertyId,
            entity.getDeposit().doubleValue());
}
```

이 위치를 선택한 이유는 명확하다. Hash는 매물 상세 데이터이고, ZSet은 추천 검색 인덱스다. Hash만 있고 ZSet이 없으면 일부 조회 경로에서는 매물이 있는 것처럼 보이지만 추천 검색에서는 누락된다.

`@Transactional`이 없는 상태의 실행 흐름은 다음과 같다.

```text
createProperty() 진입
  → Oracle INSERT + commit
  → Redis Hash 저장 성공
  → ZSet 추가 직전 RuntimeException
  → Redis ZSet 미반영
  → API 500 응답
```

결과는 다음과 같았다.

| 확인 대상 | 결과 |
|---|---|
| Oracle | ACTIVE 매물 존재 |
| Redis Hash | 존재 |
| Redis ZSet | 없음 |

이 상태는 Redis 전체 실패보다 더 복잡하다. Oracle에도 있고 Redis Hash에도 있으므로 일부 경로에서는 매물이 정상 등록된 것처럼 보일 수 있다. 그러나 추천 검색은 ZSet 인덱스를 사용하므로 해당 매물이 검색 결과에서 누락된다.

즉, Redis 부분 실패는 “있거나 없음”의 문제가 아니라, **어디까지 성공했고 어디서부터 실패했는지에 따라 불일치 형태가 달라지는 문제**다.

---

### 2-5. Redis 부분 실패 — `@Transactional` 적용

가장 중요한 결과는 `@Transactional` 적용 상태에서 발생했다.

`@Transactional`이 적용되면 Oracle INSERT는 commit 보류 상태다. 그런데 Redis Hash 저장은 Oracle commit 전에 이미 실행된다. 이후 ZSet 추가 직전에 예외가 발생하면 Spring은 Oracle 트랜잭션을 rollback한다. 하지만 이미 실행된 Redis Hash 저장은 Oracle rollback 대상이 아니다.

실행 흐름은 다음과 같다.

```text
createProperty() 진입
  → @Transactional 트랜잭션 시작
  → Oracle INSERT 전송, commit 보류
  → Redis Hash 저장 성공
  → ZSet 추가 직전 RuntimeException
  → @Transactional이 예외 감지
  → Oracle ROLLBACK
  → Redis Hash는 그대로 남음
  → API 500 응답
```

결과는 다음과 같았다.

| 확인 대상 | 결과 |
|---|---|
| Oracle | 없음 |
| Redis Hash | 존재 |
| Redis ZSet | 없음 |

이 상태는 최악이다. Oracle 기준으로는 등록되지 않은 매물인데, Redis Hash에는 매물 데이터가 남아 있다. 즉, DB에는 없는 데이터가 Redis에만 남는 유령 Hash가 발생한다.

이 결과가 F008 설계의 전환점이었다. `@Transactional`을 적용하면 항상 정합성이 좋아지는 것이 아니다. Redis 쓰기가 RDB commit보다 먼저 실행되는 구조에서는 `@Transactional`이 Oracle만 rollback하고 Redis는 되돌리지 못한다. 결과적으로 `@Transactional` 적용이 오히려 불일치를 악화시킬 수 있다.

### 2-6. 문제 실증 결론

Phase 1 테스트 결과는 다음과 같이 정리된다.

| 테스트 | 조건 | Oracle | Redis Hash | Redis ZSet | 상태 |
|---|---|---|---|---|---|
| 시나리오1-A | `@Transactional` 없음 + Redis 전체 실패 | 있음 | 없음 | 없음 | 불일치 — 검색 누락 + 재등록 곤란 |
| 시나리오1-B | `@Transactional` 있음 + Redis 전체 실패 | 없음 | 없음 | 없음 | 정합 — 깨끗한 실패 |
| 시나리오2-A | `@Transactional` 없음 + Redis 부분 실패 | 있음 | 있음 | 없음 | 반쪽짜리 — 상세 가능, 추천 누락 |
| 시나리오2-B | `@Transactional` 있음 + Redis 부분 실패 | 없음 | 있음 | 없음 | 최악 — Oracle에 없는 유령 Hash |

핵심 결론은 다음이다.

> RDB commit이 확정되기 전에 Redis에 쓰기가 발생하면, `@Transactional`이 오히려 Redis/RDB 불일치를 악화시킬 수 있다.

따라서 Redis 쓰기는 반드시 RDB commit 확정 이후에 실행되어야 한다.

---

## 3. 설계 판단 1 — Redis 동기화 실행 시점

### 3-1. 판단의 목표

첫 번째 설계 판단은 Redis 동기화의 실행 시점이다. 목표는 다음 구조를 만드는 것이다.

```text
변경 전:
  Oracle INSERT
  → Redis 동기화
  → 메서드 종료
  → Oracle commit

변경 후:
  Oracle INSERT
  → 메서드 종료
  → Oracle commit 확정
  → Redis 동기화
```

이 구조가 필요한 이유는 Redis가 Oracle rollback 대상이 아니기 때문이다. Oracle commit 전에 Redis 쓰기를 실행하면, 이후 Oracle rollback 시 Redis 데이터만 남을 수 있다. 반대로 Redis 쓰기를 commit 이후로 이동하면 Oracle rollback 시 Redis 동기화가 아예 실행되지 않는다.

즉, 이 판단의 목적은 Redis 실패를 모두 해결하는 것이 아니다. 목적은 더 좁고 명확하다.

> Oracle에 commit되지 않은 데이터가 Redis에 반영되는 구조를 제거한다.

---

### 3-2. 대안 A — `TransactionSynchronizationManager.afterCommit`

첫 번째 대안은 Spring 트랜잭션 동기화 콜백을 사용하는 것이다.

```java
TransactionSynchronizationManager.registerSynchronization(
        new TransactionSynchronization() {
            @Override
            public void afterCommit() {
                syncRedisAfterCreate(entity);
            }
        }
);
```

이 방식은 현재 트랜잭션이 정상적으로 commit된 뒤 `afterCommit()` 콜백을 실행한다. rollback이 발생하면 `afterCommit()`은 실행되지 않는다.

장점은 변경 범위가 작다는 점이다. 기존 서비스 클래스 내부에서 Redis 동기화 책임을 유지할 수 있고, 컨트롤러나 별도 파사드 계층으로 호출 순서를 옮길 필요가 없다. 또한 `syncRedisAfterCreate()` 내부 구조를 크게 바꾸지 않고 호출 시점만 이동할 수 있다.

단점은 `afterCommit()` 내부 예외가 사용자 응답으로 직접 전파되지 않는다는 점이다. 메인 트랜잭션은 이미 commit되었고, 응답 경계도 사실상 지나간 상태다. 따라서 Redis 동기화가 실패해도 API는 201을 반환할 수 있다. 이 단점은 별도의 실패 기록과 재시도 보상 구조가 있어야만 받아들일 수 있다.

---

### 3-3. 대안 B — `@TransactionalEventListener(phase = AFTER_COMMIT)`

두 번째 대안은 이벤트 기반 구조다.

```java
@Transactional
public PropertyCreateResponseDto createProperty(...) {
    PropertyCharterEntity entity = saveEntity(...);
    applicationEventPublisher.publishEvent(new PropertyCreatedEvent(entity));
    return response;
}

@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void onPropertyCreated(PropertyCreatedEvent event) {
    syncRedisAfterCreate(event.getEntity());
}
```

이 방식도 commit 이후 실행을 보장한다. rollback 시 리스너는 실행되지 않는다.

장점은 서비스 계층과 Redis 동기화 로직을 분리할 수 있다는 점이다. Redis 동기화가 명확히 후속 처리로 분리되므로, 이벤트 기반 아키텍처가 이미 있는 프로젝트라면 자연스럽다.

단점은 현재 프로젝트에 이벤트 발행/리스너 패턴이 없다는 점이다. F008 하나를 위해 새로운 이벤트 구조를 도입하면 코드가 분산되고, 기존 서비스 흐름을 이해하기 위해 이벤트 클래스와 리스너까지 함께 추적해야 한다. 또한 afterCommit 콜백과 마찬가지로 리스너 내부 예외는 사용자 응답으로 직접 전파되지 않는다.

---

### 3-4. 대안 C — 서비스 메서드 분리

세 번째 대안은 RDB 저장 메서드와 Redis 동기화 메서드를 분리하고, 호출자가 순차적으로 실행하는 방식이다.

```java
PropertyCharterEntity entity = writeService.saveProperty(dto);
writeService.syncRedis(entity);
return response;
```

이 방식은 가장 명시적이다. `saveProperty()`의 트랜잭션이 종료된 뒤 `syncRedis()`가 호출되므로 코드 흐름상 commit 이후 Redis 동기화가 드러난다. Redis 실패도 호출자에게 전파할 수 있다.

그러나 단점도 분명하다. Redis 동기화 호출 책임이 컨트롤러나 파사드 계층으로 이동한다. create, update, changeStatus와 전세/월세 조합이 늘어날수록 호출 누락 위험이 생긴다. 또한 기존 서비스 내부에 모여 있던 쓰기 책임이 호출자 쪽으로 흩어진다.

---

### 3-5. 최종 선택 — afterCommit

1차 구현에서는 대안 A인 `afterCommit`을 선택했다.

선택 이유는 다음과 같다.

1. Oracle commit 이전 Redis 쓰기를 구조적으로 차단한다.
2. Oracle rollback 시 Redis 동기화가 실행되지 않는다.
3. 기존 서비스 구조를 크게 바꾸지 않는다.
4. F008 1차 구현 범위인 `CharterPropertyWriteService.createProperty()`에 한정하여 적용하기 적합하다.
5. Redis 실패 예외가 사용자 응답으로 전파되지 않는 단점은 실패 기록과 스케줄러 보상으로 보완할 수 있다.

중요한 점은 `afterCommit`이 전체 해결책은 아니라는 것이다. `afterCommit`은 “commit 전 Redis 쓰기” 문제를 해결한다. 그러나 “commit 이후 Redis 실패” 문제는 여전히 남는다. 따라서 `afterCommit` 선택은 반드시 두 번째 설계 판단, 즉 보상 전략과 결합되어야 한다.

---

## 4. 설계 판단 2 — commit 이후 Redis 실패 보상 전략

### 4-1. 남는 문제

Redis 동기화를 `afterCommit`으로 옮기면 Oracle rollback 후 Redis 유령 데이터가 생기는 문제는 막을 수 있다. 하지만 새로운 문제가 남는다.

```text
Oracle commit은 성공했다.
afterCommit에서 Redis 동기화를 실행했다.
Redis 서버가 내려가 있거나 timeout이 발생했다.
그러면 Oracle에는 매물이 있지만 Redis에는 반영되지 않는다.
```

즉, `afterCommit`은 rollback 안전성을 확보하지만, commit 이후 Redis 장애를 제거하지는 않는다.

이 문제에 대해 가능한 대응은 세 가지다.

```text
1. Redis 실패 시 사용자에게 실패 응답을 반환한다.
2. Redis 실패를 기록하고 자동 재시도한다.
3. 재시도 상한 초과 시 Oracle 매물까지 원복한다.
```

1번은 afterCommit 구조와 잘 맞지 않는다. 이미 Oracle commit이 끝난 뒤이므로 사용자 응답을 실패로 바꾸더라도 Oracle 저장은 되돌릴 수 없다. 또한 사용자에게 실패를 반환했지만 DB에는 매물이 남는 상태는 시나리오1-A와 유사한 혼란을 만든다.

따라서 F008에서는 2번과 3번을 결합했다.

> 일시적 장애는 실패 기록 후 스케줄러가 자동 재시도하고, 장기 장애는 재시도 상한 초과 시 Oracle 매물을 원복한다.

---

### 4-2. 실패 기록의 필요성

Redis 동기화 실패는 로그만으로 처리하면 안 된다. 로그는 운영자가 확인할 수는 있지만, 시스템이 자동 복구할 기준 데이터로 사용하기 어렵다.

자동 복구를 위해서는 다음 정보가 구조화되어 남아야 한다.

| 정보 | 이유 |
|---|---|
| 어떤 매물의 동기화가 실패했는가 | Redis 재동기화 대상 식별 |
| 전세/월세 중 무엇인가 | 후속 확장 시 동기화 메서드 분기 |
| 어떤 작업 중 실패했는가 | CREATE/UPDATE/STATUS_CHANGE 분기 |
| 몇 번 재시도했는가 | 무한 재시도 방지 |
| 해결되었는가 | 중복 재처리 방지 |
| 해결 방식은 무엇인가 | 자동 복구 성공인지 원복인지 추적 |

이를 위해 `PROPERTY_SYNC_FAILURES` 테이블을 도입했다. 이 테이블은 단순 에러 로그가 아니라, 스케줄러가 복구 작업을 수행하기 위한 상태 저장소다.

1차 구현에서는 범위를 `CHARTER + CREATE`로 제한했으므로 `LEASE_TYPE = CHARTER`, `OPERATION_TYPE = CREATE`, `FAIL_STEP = FULL`로 기록한다. Redis 내부 명령별 실패 지점 세분화는 후속 과제로 남겼다.

---

### 4-3. 즉시 재시도와 스케줄러 재시도의 구분

Redis 장애에는 일시적 장애와 장기 장애가 있다. 예를 들어 짧은 timeout이나 순간적인 연결 실패는 즉시 재시도만으로 해결될 수 있다. 반면 Redis 서버 중지, 네트워크 단절, OOM 등은 사용자 요청 스레드 안에서 반복 재시도해도 해결되지 않을 가능성이 높다.

따라서 1차 구현은 다음 정책을 사용했다.

```text
1. afterCommit에서 Redis 동기화를 1차 시도한다.
2. 연결 실패 또는 timeout 성격의 예외는 200ms 대기 후 1회 재시도한다.
3. 즉시 재시도에 성공하면 실패 기록 없이 종료한다.
4. 즉시 재시도까지 실패하면 PROPERTY_SYNC_FAILURES에 기록한다.
5. 스케줄러가 주기적으로 미해결 실패를 조회하여 재시도한다.
6. 재시도 상한을 초과하면 Oracle 매물을 삭제하여 불일치 상태를 종료한다.
```

이 정책은 사용자 요청 스레드에서 무한히 대기하지 않으면서도, 짧은 장애는 빠르게 흡수하고, 긴 장애는 스케줄러로 넘긴다.

---

### 4-4. 장기 장애 시 Oracle 원복을 선택한 이유

Redis 동기화가 계속 실패하면 Oracle에는 매물이 있지만 Redis 추천 인덱스에는 없는 상태가 지속된다. 이 상태를 그대로 두면 사용자는 목록이나 관리 화면에서는 매물을 볼 수 있지만 추천 검색에서는 보지 못한다.

이때 선택지는 두 가지다.

```text
1. Oracle 매물을 유지하고 Redis 복구를 계속 시도한다.
2. 일정 상한 이후 Oracle 매물을 삭제하여 최종 실패로 정리한다.
```

실서비스라면 1번을 선택하고 관리자 알림, 사용자 상태 표시, 재처리 큐, outbox pattern 등을 붙일 수 있다. 그러나 현재 F008 1차 구현의 목표는 포트폴리오 범위에서 “불일치 상태를 무한히 방치하지 않는 구조”를 만드는 것이다.

따라서 재시도 상한을 초과하면 Oracle 매물을 삭제하고, 실패 레코드는 `RESOLVED = Y`, `RESOLVED_METHOD = ROLLBACK`으로 처리한다. 이 선택은 다음 의미를 갖는다.

```text
Redis 장애가 일시적이면 자동 복구한다.
Redis 장애가 장기화되면 Oracle-only 상태를 남기지 않는다.
최종적으로 양쪽 모두 반영되거나, 양쪽 모두 없는 상태로 정리한다.
```

즉, 이 구조의 목표는 “반드시 등록을 성공시키는 것”이 아니라, **불일치 상태의 무한 지속을 막는 것**이다.

---

## 5. 설계 판단 3 — afterCommit 내부 실패 기록 저장의 트랜잭션 경계

### 5-1. 문제 상황 — 실패 기록은 스케줄러 보상의 기준 데이터다

F008 1차 구현에서 Redis 동기화 실패를 단순 로그로만 남기지 않고 `PROPERTY_SYNC_FAILURES` 테이블에 저장한 이유는, 이후 스케줄러가 이 테이블을 기준으로 자동 복구 또는 원복을 수행해야 하기 때문이다.

즉, `recordSyncFailure()`는 부가적인 로그 저장 메서드가 아니다. F008 보상 아키텍처에서 다음 단계를 가능하게 만드는 핵심 연결 지점이다.

```text
Oracle 매물 저장 commit 성공
  → afterCommit에서 Redis 동기화 시도
  → Redis 동기화 실패
  → PROPERTY_SYNC_FAILURES에 실패 기록 저장
  → 스케줄러가 미해결 실패 레코드 조회
  → Redis 재동기화 재시도
  → 성공 시 RETRY로 해결 처리
  → 상한 초과 시 Oracle 매물 원복 후 ROLLBACK으로 해결 처리
```

따라서 Redis 실패 기록이 DB에 남지 않으면 단순히 로그 하나가 누락되는 것이 아니다. 스케줄러가 복구 대상을 찾을 수 없게 된다. 그러면 Oracle에는 매물이 commit되어 있는데 Redis에는 반영되지 않은 상태가 추적되지 않고 방치된다. 이는 F008에서 해결하려던 Oracle-only 불일치 문제를 다시 만드는 것이다.

초기 구현에서는 Redis 동기화 실패 시 다음 메서드에서 실패 정보를 저장했다.

```java
/**
 * Redis 동기화 실패를 PROPERTY_SYNC_FAILURES 테이블에 기록.
 */
private void recordSyncFailure(String propertyId, String failReason) {
    PropertySyncFailure failure = PropertySyncFailure.builder()
            .propertyId(propertyId)
            .leaseType("CHARTER")
            .operationType("CREATE")
            .failStep("FULL")
            .failReason(failReason)
            .retryCount(0)
            .maxRetries(5)
            .resolved("N")
            .build();

    syncFailureRepository.save(failure);

    log.warn("[SYNC_FAILURE_RECORDED] propertyId={}", propertyId);
}
```

의도한 의미는 다음과 같았다.

```text
Redis 동기화 실패
  → 실패 기록을 Oracle에 남김
  → 스케줄러가 나중에 이 레코드를 보고 복구
```

하지만 실제 검증에서는 `syncFailureRepository.save(failure)`가 예외 없이 실행되고 `[SYNC_FAILURE_RECORDED]` 로그도 출력되었는데, Oracle에서 `PROPERTY_SYNC_FAILURES` 테이블을 조회하면 데이터가 없었다.

증상은 다음과 같이 정리된다.

```text
recordSyncFailure() 호출됨
syncFailureRepository.save(failure) 예외 없음
[SYNC_FAILURE_RECORDED] 로그 출력됨
Oracle에서 PROPERTY_SYNC_FAILURES 조회 시 0건
```

이 문제는 스케줄러 보상 구조 전체를 깨뜨리는 문제였다. 실패 기록이 남지 않으면 스케줄러는 해당 매물을 재시도할 수도 없고, 재시도 상한 초과 후 원복할 수도 없다. 따라서 원인은 단순 repository 호출 문제가 아니라, **afterCommit 내부에서 다시 RDB에 쓰는 작업의 트랜잭션 경계 문제**로 봐야 했다.

---

### 5-2. 원인 — afterCommit은 commit 이후이며, save는 새로운 commit을 만들지 않는다

이 문제의 핵심은 `save()`와 `commit`을 구분하는 것이다.

JPA에서 `repository.save(entity)`는 DB 변경을 영속성 컨텍스트에 반영하거나 INSERT 대상 엔티티로 등록하는 작업이다. 그러나 `save()` 자체가 DB commit을 수행하는 것은 아니다. DB에 최종 반영되는 시점은 트랜잭션 commit 시점이다.

일반적인 `@Transactional` 메서드에서는 다음 흐름이 자연스럽게 이어진다.

```text
@Transactional 메서드 진입
  → repository.save(entity)
  → 메서드 정상 종료
  → Spring 트랜잭션 AOP가 commit
  → DB 변경 최종 확정
```

그래서 일반적인 서비스 코드에서는 `save()`를 호출하면 저장된 것처럼 보인다. 하지만 실제로 DB 반영을 확정한 것은 `save()`가 아니라 메서드 종료 후 실행된 commit이다.

F008에서는 상황이 다르다. 실패 기록 저장은 일반적인 `@Transactional` 메서드 본문에서 실행된 것이 아니라, 메인 매물 등록 트랜잭션이 commit된 뒤 실행되는 `afterCommit()` 콜백 내부에서 실행되었다.

전체 흐름은 다음과 같다.

```text
createProperty() 진입
  → @Transactional 트랜잭션 시작
  → Oracle에 매물 INSERT
  → afterCommit 콜백 등록
  → createProperty() 정상 종료

Spring 트랜잭션 AOP
  → 메인 트랜잭션 commit
  → PROPERTIES_CHARTER 매물 저장 확정

afterCommit() 콜백 실행
  → Redis 동기화 시도
  → Redis 실패
  → recordSyncFailure()
  → PROPERTY_SYNC_FAILURES에 실패 기록 저장 시도
```

여기서 중요한 점은 `afterCommit()`이 이미 메인 트랜잭션 commit 이후에 실행된다는 것이다. 즉, `recordSyncFailure()`가 호출되는 시점에는 `createProperty()`의 commit 기회가 이미 지나갔다.

또한 `afterCommit()`은 commit 이후 실행되지만, Spring의 트랜잭션 리소스 cleanup이 완전히 끝나기 전 구간에서 호출될 수 있다. 따라서 다음 두 조건이 동시에 성립할 수 있다.

```text
1. 메인 트랜잭션 commit은 이미 끝났다.
2. 현재 스레드에는 아직 기존 트랜잭션 리소스가 바인딩되어 있을 수 있다.
```

이 상태에서 repository의 기본 전파 속성인 `PROPAGATION_REQUIRED`로 `save()`가 실행되면 문제가 생길 수 있다. `REQUIRED`는 현재 트랜잭션이 없으면 새 트랜잭션을 열지만, 현재 트랜잭션 리소스가 있다고 판단하면 새 트랜잭션을 열지 않고 기존 리소스에 참여하려 한다.

문제는 그 기존 트랜잭션이 이미 commit을 완료했다는 점이다.

```text
메인 트랜잭션 owner = createProperty()
메인 트랜잭션 commit = 이미 완료됨
afterCommit 내부 save = 기존 리소스에 참여한 것처럼 처리될 수 있음
추가 commit = 없음
```

결과적으로 다음 상황이 발생한다.

```text
syncFailureRepository.save(failure)는 호출된다.
예외도 발생하지 않는다.
[SYNC_FAILURE_RECORDED] 로그도 출력된다.
하지만 해당 save를 확정해 줄 commit이 없다.
따라서 PROPERTY_SYNC_FAILURES 테이블에는 데이터가 남지 않는다.
```

이 현상은 “repository save가 실패했다”가 아니다. `save()` 호출 자체는 성공했다. 문제는 `save()` 이후 변경을 DB에 확정할 트랜잭션 commit 경계가 없었다는 것이다.

F008 보상 구조의 관점에서 이 문제를 다시 표현하면 다음과 같다.

```text
Redis 동기화는 Oracle commit 이후에 실행되어야 한다.
그래서 afterCommit으로 옮겼다.

그런데 afterCommit 내부에서 Redis 실패를 스케줄러용 테이블에 저장하려면,
그 저장 작업은 이미 끝난 메인 트랜잭션에 기대면 안 된다.

실패 기록 저장은 메인 매물 등록 트랜잭션과 별개의 새 트랜잭션으로 commit되어야 한다.
```

따라서 이 섹션의 결론은 단순히 “save는 commit이 아니다”가 아니다. 더 정확한 결론은 다음이다.

> afterCommit 내부에서 스케줄러 보상을 위한 RDB 실패 기록을 저장하려면, 이미 commit이 끝난 메인 트랜잭션이 아니라 독립적인 새 트랜잭션 경계가 필요하다.

---

### 5-3. 해결 — 실패 기록 저장은 REQUIRES_NEW 독립 트랜잭션으로 분리

해결 방향은 명확했다. Redis 동기화 실패 기록은 스케줄러가 반드시 조회할 수 있어야 하므로, `recordSyncFailure()`는 자체 commit 경계를 가져야 한다.

이를 위해 `TransactionTemplate`에 `PROPAGATION_REQUIRES_NEW`를 지정했다.

```java
private void recordSyncFailure(String propertyId, String failReason) {
    TransactionTemplate txTemplate = new TransactionTemplate(transactionManager);
    txTemplate.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRES_NEW);

    txTemplate.execute(status -> {
        PropertySyncFailure failure = PropertySyncFailure.builder()
                .propertyId(propertyId)
                .leaseType("CHARTER")
                .operationType("CREATE")
                .failStep("FULL")
                .failReason(failReason)
                .retryCount(0)
                .maxRetries(5)
                .resolved("N")
                .build();

        syncFailureRepository.save(failure);
        return null;
    });

    log.warn("[SYNC_FAILURE_RECORDED] propertyId={}", propertyId);
}
```

수정 후 흐름은 다음과 같다.

```text
createProperty() 메인 트랜잭션 commit 완료
  → afterCommit 실행
  → Redis 동기화 실패
  → recordSyncFailure() 호출
  → TransactionTemplate(REQUIRES_NEW) 실행
      → 실패 기록 저장용 새 트랜잭션 시작
      → syncFailureRepository.save(failure)
      → 새 트랜잭션 commit
  → PROPERTY_SYNC_FAILURES에 실패 기록 확정
  → 스케줄러가 이후 해당 레코드를 조회 가능
```

`REQUIRES_NEW`를 사용한 이유는 기존 트랜잭션 리소스가 남아 있더라도 그것에 참여하지 않고, 실패 기록 저장용 트랜잭션을 별도로 열기 위해서다. 이 새 트랜잭션은 `createProperty()`의 메인 트랜잭션과 별개의 commit 경계를 가진다. 따라서 afterCommit 내부에서 실행되더라도 실패 기록이 DB에 확정된다.

`@Transactional(propagation = REQUIRES_NEW)`를 별도 메서드에 붙이는 방식도 가능하다. 그러나 같은 클래스 내부에서 메서드를 직접 호출하면 Spring AOP 프록시를 거치지 않는 self-invocation 문제가 발생할 수 있다. 이 경우 `@Transactional`이 기대대로 적용되지 않을 수 있다. 1차 구현에서는 별도 서비스 클래스를 추가하기보다, 현재 서비스 내부에서 트랜잭션 경계를 명시적으로 제어하기 위해 `TransactionTemplate`을 사용했다.

최종 판단은 다음과 같다.

```text
Redis 동기화 실행 시점:
  Oracle commit 이후여야 하므로 afterCommit 사용

Redis 실패 기록 저장:
  스케줄러 보상의 기준 데이터이므로 반드시 DB에 commit되어야 함
  afterCommit 내부에서 실행되므로 메인 트랜잭션에 기대면 안 됨
  REQUIRES_NEW 독립 트랜잭션으로 저장
```

이 보강으로 F008 보상 구조의 인과관계가 명확해졌다.

```text
afterCommit
  → commit 전 Redis 쓰기 방지

PROPERTY_SYNC_FAILURES
  → commit 이후 Redis 실패 추적

REQUIRES_NEW
  → afterCommit 내부 실패 기록의 독립 commit 보장

스케줄러
  → 실패 기록을 기준으로 재시도 또는 원복 수행
```

따라서 `REQUIRES_NEW`는 단순한 트랜잭션 옵션이 아니다. F008에서 스케줄러 기반 보상 구조가 실제로 동작하기 위해 필요한 저장 보장 장치다.



## 6. 최종 구현 범위와 검증 결과

### 6-1. 1차 구현 범위

F008 1차 구현 범위는 명확히 제한했다.

```text
대상: CharterPropertyWriteService.createProperty()
범위: CHARTER + CREATE
목표: 전세 매물 생성 시 RDB commit 이후 Redis 동기화 및 실패 보상
```

의도적으로 제외한 범위는 다음과 같다.

```text
1. 월세 매물 생성
2. 전세/월세 매물 수정
3. 전세/월세 상태 변경
4. Redis 명령어별 실패 지점 세분화
5. 부분 성공분 즉시 롤백
6. Pipeline, MULTI/EXEC, Lua Script 기반 Redis 원자성 보장
7. bounds 축소 또는 재계산
8. 사용자에게 동기화 대기/실패 상태를 노출하는 상태 모델
```

이 범위 제한은 구현 누락이 아니라 단계적 설계 판단이다. 먼저 RDB commit 전 Redis 쓰기를 차단하고, commit 이후 Redis 장애에 대한 보상 루프를 만드는 것이 1차 목표였다. Redis 내부 명령 단위 원자성은 별도의 판단 영역으로 남겨두었다.

---

### 6-2. 최종 흐름

1차 구현 후 전세 매물 생성 흐름은 다음과 같다.

```text
createProperty() 진입
  → @Transactional 트랜잭션 시작
  → 입력값 및 중복 검증
  → Oracle INSERT
  → afterCommit 콜백 등록
  → 메서드 정상 종료
  → Oracle commit 확정
  → afterCommit 실행
      → Redis 동기화 1차 시도
      → 일시적 장애면 200ms 후 1회 재시도
      → 최종 실패 시 실패 테이블 기록
          → TransactionTemplate REQUIRES_NEW로 독립 commit
  → 스케줄러가 미해결 실패 조회
      → Redis 재동기화 성공 시 RETRY 해결
      → 상한 초과 시 Oracle DELETE 후 ROLLBACK 해결
```

현재 코드에서는 중복 등록 대응을 위해 `persist + flush`를 사용한다. 이 부분은 F006의 대안 B와 연결된다. F008에서 중요한 지점은 `flush()` 이후 Redis를 바로 호출하지 않고, `afterCommit` 콜백으로 Redis 동기화를 예약한다는 점이다.

```java
entityManager.persist(entity);
entityManager.flush();

TransactionSynchronizationManager.registerSynchronization(
        new TransactionSynchronization() {
            @Override
            public void afterCommit() {
                executeRedisSyncWithCompensation(entity);
            }
        }
);
```

`flush()`는 INSERT SQL을 DB에 전송하지만 commit은 아니다. 따라서 이 시점에서 Redis를 직접 호출하면 여전히 Oracle rollback 후 Redis 유령 데이터가 생길 수 있다. Redis 동기화는 commit 이후 실행되는 `afterCommit()`에 있어야 한다.

---

### 6-3. 검증 시나리오 요약

1차 구현 후 다음 시나리오를 검증했다.

| # | 시나리오 | 기대 결과 | 확인한 내용 |
|---|---|---|---|
| 1 | 정상 흐름 | Oracle, Redis 모두 반영 | afterCommit 기본 동작 |
| 2 | Redis 중지 상태에서 등록 | Oracle 저장, 실패 테이블 기록 | commit 이후 Redis 실패 추적 |
| 3 | Redis 중지 지속 | retryCount 증가 | 스케줄러 재시도 |
| 4 | Redis 복구 | Redis 재동기화, RETRY 해결 | 자동 복구 |
| 5 | 재시도 상한 초과 | Oracle DELETE, ROLLBACK 해결 | 장기 불일치 종료 |
| 6 | RDB 커밋 실패 | Oracle/Redis/실패테이블 모두 없음 | afterCommit 미실행 보장 |

---

### 6-4. 정상 흐름 검증

Redis가 정상인 상태에서 매물 등록 요청을 보냈다.

결과는 다음과 같았다.

| 확인 대상 | 결과 |
|---|---|
| API 응답 | 201 Created |
| Oracle | ACTIVE 매물 존재 |
| Redis Hash | 존재 |
| Redis 가격 ZSet | score 반영 |
| 실패 테이블 | 0건 |

이 결과는 `afterCommit` 구조가 정상 등록 흐름을 깨지 않는다는 것을 확인한다. Oracle commit 이후 Redis 동기화가 실행되었고, 실패 기록은 생성되지 않았다.

---

### 6-5. Redis 중지 상태 등록 검증

Redis 서버를 중지한 상태에서 매물 등록 요청을 보냈다.

결과는 다음과 같았다.

| 확인 대상 | 결과 |
|---|---|
| API 응답 | 201 Created |
| Oracle | ACTIVE 매물 존재 |
| Redis | 반영 없음 |
| 실패 테이블 | `RESOLVED = N`, `RETRY_COUNT = 0` 레코드 생성 |

이 결과의 의미는 다음이다.

```text
Oracle commit은 성공했다.
afterCommit에서 Redis 동기화가 실패했다.
실패는 사용자 응답으로 직접 전파되지 않았다.
대신 실패 테이블에 기록되었다.
스케줄러가 후속 복구할 수 있는 상태가 되었다.
```

여기서 API가 201을 반환하는 것은 afterCommit 방식의 특성이다. 이미 Oracle commit이 완료된 뒤 Redis 실패가 발생하기 때문이다. 이 구조를 사용하려면 반드시 실패 기록과 후속 보상이 필요하다.

---

### 6-6. 스케줄러 재시도 검증

Redis가 계속 중지된 상태에서 스케줄러가 실행되면, 실패 테이블의 미해결 레코드를 조회하고 Redis 재동기화를 시도한다.

Redis가 여전히 중지되어 있으므로 재시도는 실패하고 `RETRY_COUNT`가 증가한다.

```text
RESOLVED = N
RETRY_COUNT 증가
```

이 검증은 실패 테이블이 단순 로그가 아니라 스케줄러 복구 루프의 상태 저장소로 동작한다는 것을 확인한다.

---

### 6-7. Redis 복구 후 자동 복구 검증

Redis를 다시 실행한 뒤 스케줄러가 실행되면, 미해결 실패 레코드의 매물을 Oracle에서 조회하고 Redis 동기화를 다시 수행한다.

성공 시 결과는 다음과 같다.

| 확인 대상 | 결과 |
|---|---|
| Redis Hash | 생성 |
| Redis ZSet | member/score 반영 |
| 실패 테이블 | `RESOLVED = Y` |
| 해결 방식 | `RESOLVED_METHOD = RETRY` |

이 결과는 일시적 Redis 장애가 시스템 내부 재시도로 복구될 수 있음을 보여준다. 사용자는 최초 요청에서 201을 받았고, Redis 장애가 해소되면 추천 검색 인덱스도 최종적으로 회복된다.

---

### 6-8. 재시도 상한 초과 원복 검증

Redis 장애가 계속되어 재시도 상한에 도달하면, 스케줄러는 장기 장애로 판단하고 Oracle 매물을 삭제한다.

결과는 다음과 같았다.

| 확인 대상 | 결과 |
|---|---|
| Oracle | 대상 매물 DELETE |
| Redis 잔존 정리 | 시도 |
| 실패 테이블 | `RESOLVED = Y` |
| 해결 방식 | `RESOLVED_METHOD = ROLLBACK` |

이 검증은 F008 보상 구조의 중요한 특성을 보여준다. Redis 장애가 장기화되어 Redis 추천 인덱스에 반영되지 않는 매물이 Oracle에만 계속 남는 상황을 방치하지 않는다. 재시도 상한을 넘으면 Oracle 매물도 제거하여 최종적으로 불일치 상태를 종료한다.

---

### 6-9. RDB 커밋 실패 시 afterCommit 미실행 검증

마지막으로 Oracle 트랜잭션이 rollback될 때 afterCommit 콜백이 실행되지 않는지 확인했다. `createProperty()` 내부에서 afterCommit 등록 전 RuntimeException을 발생시켰다.

실행 흐름은 다음과 같다.

```text
createProperty() 진입
  → @Transactional 트랜잭션 시작
  → Oracle INSERT 전송, commit 보류
  → RuntimeException 발생
  → @Transactional이 rollback
  → afterCommit 콜백 미등록 또는 미실행
  → Redis 동기화 미실행
```

결과는 다음과 같았다.

| 확인 대상 | 결과 |
|---|---|
| API 응답 | 500 Internal Server Error |
| Oracle | 없음 |
| Redis | 없음 |
| 실패 테이블 | 없음 |
| afterCommit 관련 로그 | 없음 |

이 검증은 afterCommit 구조의 근본 보장을 확인한다.

> Oracle commit이 실패하면 Redis는 아예 건드리지 않는다.

따라서 기존 시나리오2-B에서 발생했던 “Oracle rollback + Redis Hash 잔존” 상태는 afterCommit 구조에서는 구조적으로 발생하지 않는다.

---

## 7. 최종 판단

### 7-1. afterCommit을 선택한 이유

F008의 핵심 판단은 다음이다.

> Redis 쓰기는 Oracle commit 이후에만 실행되어야 한다.

이를 위해 `afterCommit`을 선택했다. 이유는 다음과 같다.

1. Oracle rollback 시 Redis 동기화가 실행되지 않는다.
2. 기존 서비스 구조를 크게 변경하지 않는다.
3. 전세 매물 생성 한정 1차 구현에 적합하다.
4. Redis 동기화 호출 누락 위험이 서비스 내부에서 통제된다.
5. commit 이후 Redis 실패는 실패 테이블과 스케줄러 보상으로 처리할 수 있다.

`afterCommit`의 단점은 Redis 실패가 사용자 응답으로 직접 전파되지 않는다는 점이다. 그러나 이 단점은 F008에서 회피한 것이 아니라 명시적으로 받아들이고 보상 구조를 붙였다. 실패 테이블과 스케줄러가 없다면 afterCommit만으로는 불완전하다.

---

### 7-2. 실패 테이블과 스케줄러를 선택한 이유

Redis 장애는 사용자 입력 오류가 아니다. 사용자가 같은 요청을 다시 보낸다고 해서 해결되는 문제가 아닐 수 있다. 특히 Redis 서버 중지나 네트워크 장애는 시스템이 내부적으로 재시도하거나 운영자가 복구해야 한다.

따라서 Redis 동기화 실패를 사용자에게 즉시 전가하는 대신 다음 구조를 선택했다.

```text
일시적 장애
  → 즉시 재시도 또는 스케줄러 재시도로 복구

장기 장애
  → 재시도 상한 초과 후 Oracle 원복
```

이 구조의 목표는 “모든 등록을 무조건 성공시키는 것”이 아니다. 목표는 **최종적으로 일관된 상태로 수렴하는 것**이다.

```text
성공 경로:
  Oracle 있음 + Redis 있음

최종 실패 경로:
  Oracle 없음 + Redis 없음
```

반대로 피해야 하는 상태는 다음이다.

```text
Oracle 있음 + Redis 없음
Oracle 없음 + Redis 있음
Oracle 있음 + Redis 일부만 있음
```

F008 1차 구현은 이 상태들을 무한히 방치하지 않도록 실패 추적과 후속 처리를 추가했다.

---

### 7-3. REQUIRES_NEW를 선택한 이유

`afterCommit` 내부에서 실패 기록을 저장할 때 단순 repository `save()`만으로는 DB 반영이 보장되지 않았다. 원인은 `save()`가 commit이 아니며, afterCommit 시점에는 메인 트랜잭션 commit이 이미 끝났기 때문이다.

따라서 실패 기록 저장에는 독립 트랜잭션이 필요했다.

최종 판단은 다음이다.

```text
Redis 동기화 실행 시점:
  afterCommit

Redis 실패 기록 저장:
  TransactionTemplate + PROPAGATION_REQUIRES_NEW
```

이 조합을 통해 다음을 보장한다.

```text
1. Oracle commit 전 Redis 쓰기 금지
2. Oracle commit 이후 Redis 실패 추적
3. afterCommit 내부 실패 기록의 독립 commit
```

---

## 8. 현재 한계와 후속 과제

### 8-1. Redis 부분 실패 세분화 미완성

F008 1차 구현은 Redis 내부 명령 단위의 부분 실패를 세분화하지 않았다. 현재는 실패 지점을 `FULL`로 기록하고 전체 Redis 동기화를 다시 수행한다.

후속 과제는 다음과 같다.

```text
1. FAIL_STEP 세분화
   - HASH
   - ZSET_PRICE
   - ZSET_AREA
   - BOUNDS_PRICE
   - BOUNDS_AREA

2. Redis 명령 묶음의 원자성 검토
   - Pipeline
   - MULTI/EXEC
   - Lua Script

3. 부분 성공분 정리 전략
   - Hash만 성공한 경우
   - ZSet 일부만 성공한 경우
   - bounds만 확장된 경우
```

특히 bounds는 단순 삭제나 축소가 어렵다. bounds는 여러 매물의 최솟값/최댓값을 나타내므로, 한 매물의 실패나 삭제만 보고 즉시 축소하면 다른 매물의 경계값을 깨뜨릴 수 있다. 따라서 bounds 재계산은 별도 배치 또는 재구축 전략이 필요하다.

---

### 8-2. 적용 범위 제한

현재 구현은 `CHARTER + CREATE`에 한정된다.

아직 다음 기능에는 동일한 보상 구조가 적용되지 않았다.

```text
1. 월세 매물 생성
2. 전세 매물 수정
3. 월세 매물 수정
4. 전세 매물 상태 변경
5. 월세 매물 상태 변경
```

이 기능들은 Redis 동기화 방식이 서로 다르다. 예를 들어 수정은 기존 인덱스 제거 후 새 인덱스 추가가 필요할 수 있고, 상태 변경은 COMPLETED와 DELETED에 따라 Hash 유지 또는 삭제 정책이 달라진다. 따라서 create와 동일한 방식으로 단순 복사하면 안 되고, operation type별 보상 전략을 별도로 정의해야 한다.

---

### 8-3. 사용자 응답 모델의 한계

현재 afterCommit 구조에서는 Redis 동기화 실패가 사용자 응답으로 직접 전파되지 않는다. 사용자는 201 Created를 받지만, 이후 스케줄러가 장기 장애로 판단하여 Oracle 매물을 원복할 수 있다.

포트폴리오 1차 구현에서는 불일치 상태를 무한히 방치하지 않는 것에 초점을 두었다. 그러나 실서비스라면 다음 상태 모델을 추가로 고려해야 한다.

```text
REGISTERED_SYNC_PENDING
REGISTERED_SYNC_FAILED
ACTIVE
ROLLBACKED
```

또는 사용자에게 “등록 처리 중”, “검색 반영 대기 중”, “등록 실패로 원복됨”과 같은 상태를 노출할 필요가 있다.

현재 구현은 이 상태 모델까지 확장하지 않았다. 대신 내부 실패 테이블과 스케줄러를 통해 시스템 내부 정합성을 관리하는 1차 보상 구조에 집중했다.

---

## 9. 최종 결론

F008의 결론은 “Redis 예외를 잡아서 처리했다”가 아니다. 더 정확한 결론은 다음과 같다.

> Oracle RDB와 Redis를 함께 갱신하는 매물 등록 기능에서, Redis는 Oracle 트랜잭션의 rollback 대상이 아니므로 Redis 쓰기는 반드시 Oracle commit 이후에 실행되어야 한다. commit 이후 Redis 실패는 별도의 실패 기록과 재시도/원복 보상 구조로 관리해야 한다.

기존 구조에서는 Redis 동기화가 `@Transactional` 경계 내부에서 실행되었다. 이 구조는 Redis 전체 실패에서는 Oracle-only 상태를 만들 수 있고, Redis 부분 실패에서는 `@Transactional` 적용 상태에서도 Oracle rollback + Redis Hash 잔존이라는 더 위험한 유령 데이터 상태를 만들 수 있었다.

이를 해결하기 위해 Redis 동기화를 `afterCommit`으로 이동했다. 그 결과 Oracle rollback 시 Redis 동기화가 실행되지 않으므로, commit되지 않은 매물이 Redis에 반영되는 문제가 구조적으로 차단된다.

그러나 `afterCommit`만으로는 충분하지 않았다. Oracle commit 이후 Redis가 실패하면 Oracle에는 매물이 있고 Redis에는 없는 상태가 된다. 이를 추적하기 위해 `PROPERTY_SYNC_FAILURES` 실패 기록을 두고, 스케줄러가 미해결 실패를 재시도하도록 했다. Redis 장애가 일시적이면 자동 복구하고, 장기화되면 재시도 상한 초과 시 Oracle 매물을 삭제하여 불일치 상태를 종료한다.

또한 구현 과정에서 `afterCommit` 내부의 repository `save()`가 DB에 남지 않는 문제를 확인했다. 원인은 `save()`가 commit이 아니며, afterCommit 시점에는 메인 트랜잭션 commit이 이미 끝났다는 점이었다. 이 문제는 `TransactionTemplate + PROPAGATION_REQUIRES_NEW`로 실패 기록 저장을 독립 트랜잭션으로 분리하여 해결했다.

최종적으로 F008 1차 구현은 다음을 보장한다.

```text
1. Oracle commit 전 Redis 쓰기 차단
2. Oracle rollback 시 Redis 미실행
3. Oracle commit 이후 Redis 실패 기록
4. 일시적 장애 자동 재시도
5. 장기 장애 시 Oracle 원복
6. afterCommit 내부 RDB 쓰기의 독립 commit 보장
```

현재 구현은 전세 매물 생성 한정의 1차 구현이다. 후속 단계에서는 Redis 명령 단위 부분 실패 세분화, 월세/수정/상태 변경 확장, bounds 재계산 전략, 사용자 노출 상태 모델을 추가로 검토해야 한다.

따라서 F008은 단순한 캐시 동기화 실패 처리가 아니라, **RDB 트랜잭션과 외부 저장소 Redis 사이에서 commit 순서, 실패 추적, 자동 복구, 장기 장애 원복을 어떻게 설계할 것인지 판단한 작업**이다.
