## 1. 검증 대상 아키텍처 개요

### 1.1. 변경 전 구조

```
createProperty() 진입
  → charterRepository.save(entity)     // Oracle INSERT
  → syncRedisAfterCreate(entity)       // Redis Hash + ZSet + bounds 반영
  → 응답 반환
```

이 구조에서는 Redis 동기화가 `@Transactional` 경계 내부에서 RDB 커밋 확정 이전에 실행된다. F008 시나리오1/2 테스트에서 두 가지 구조적 결함이 실증되었다.

- **시나리오1(@Transactional 미적용):** Redis 전체 실패 시 Oracle에만 데이터 잔존, 재등록 시 DuplicatePropertyException으로 자가 복구 불가.
- **시나리오2(@Transactional 적용):** Redis 부분 실패 시 Oracle은 ROLLBACK되지만 Redis Hash가 잔존하는 유령 데이터 발생. `@Transactional` 적용이 오히려 불일치를 악화시키는 결과.

### 1.2. 변경 후 구조 (F008 1차 구현)

```
createProperty() 진입 → @Transactional 트랜잭션 시작
  → charterRepository.save(entity)            // Oracle INSERT (COMMIT 보류)
  → afterCommit 콜백 등록                       // 콜백 "예약"만 됨, 실행 아님
  → 메서드 정상 종료 → @Transactional이 COMMIT 실행
  → afterCommit 콜백 실행
    → executeRedisSyncWithCompensation(entity)
      → 1차 시도: syncRedisAfterCreate(entity)
      → 실패 시 예외 분류 → 일시적 장애면 200ms 대기 후 재시도 1회
      → 최종 실패 시 recordSyncFailure() → PROPERTY_SYNC_FAILURES INSERT
  → 201 응답 반환
```

보상 구조 요소:

- **afterCommit 콜백:** Oracle 커밋 확정 후에만 Redis 동기화를 실행하여 유령 데이터를 구조적으로 차단.
- **예외 분류 + 즉시 재시도:** `RedisConnectionFailureException`/`QueryTimeoutException`(일시적 장애)은 200ms 대기 후 재시도 1회. `RedisSystemException`(OOM 등 인프라 장애)은 재시도 무의미하므로 즉시 실패 기록.
- **실패 테이블 기록:** `PROPERTY_SYNC_FAILURES` 테이블에 실패 정보 INSERT.
- **복구 스케줄러:** `PropertySyncRetryScheduler`가 1분 주기로 미해결 레코드를 조회하여 Redis 재동기화 시도. 재시도 상한(MAX_RETRIES) 초과 시 Oracle DELETE + Redis 잔존 정리(ROLLBACK).

### 1.3. 신규 파일 3개

| 파일 | 패키지 | 역할 |

### 4.5. 개발 과정에서 발견된 문제 — afterCommit 내부의 트랜잭션 컨텍스트

**현상:** 최초 구현에서 `recordSyncFailure()` 내부의 `syncFailureRepository.save(failure)`는 예외 없이 실행되었고, `[SYNC_FAILURE_RECORDED]` 로그도 정상 출력되었으나, Oracle `PROPERTY_SYNC_FAILURES` 테이블을 조회하면 **0건**이었다.

**원인:** afterCommit 콜백은 기존 `@Transactional` 트랜잭션이 커밋된 직후에 실행된다. 이 시점에서 트랜잭션은 "완료" 상태이지만 Spring 내부에 여전히 바인딩되어 있다. `syncFailureRepository.save()` 내부의 `@Transactional`이 이미 완료된 기존 트랜잭션에 참여하려고 시도하면서, INSERT SQL은 전송되지만 별도의 COMMIT이 발생하지 않는다. 결과적으로 INSERT는 실행되었으나 커밋되지 않아 데이터가 테이블에 반영되지 않는다.

**검토한 대안:** `recordSyncFailure()` 메서드에 `@Transactional(propagation = REQUIRES_NEW)`를 직접 선언하는 방법은 유효하지 않다. `createProperty()` 내부에서 같은 클래스의 `recordSyncFailure()`를 호출하는 것은 Spring AOP 프록시를 거치지 않는 자기 자신 내부 호출(self-invocation)이므로, 메서드에 선언된 `@Transactional` 어노테이션을 Spring이 인식하지 못한다.

**적용한 해결:** `recordSyncFailure()` 내부에서 프로그래밍 방식으로 기존 트랜잭션과 완전히 독립적인 새 트랜잭션을 열어 INSERT + COMMIT하도록 변경했다. 이 방식은 프록시에 의존하지 않으므로 같은 클래스 내부 호출에서도 정상 동작한다.

```java
private void recordSyncFailure(String propertyId, String failReason) {
    try {
        TransactionTemplate txTemplate = new TransactionTemplate(transactionManager);
        txTemplate.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRES_NEW);

        txTemplate.execute(status -> {
            PropertySyncFailure failure = PropertySyncFailure.builder()
                    .propertyId(propertyId)
                    .leaseType("CHARTER")
                    .operationType("CREATE")
                    .failStep("FULL")
                    .failReason(failReason != null && failReason.length() > 500
                            ? failReason.substring(0, 500) : failReason)
                    .failTime(LocalDateTime.now())
                    .retryCount(0)
                    .maxRetries(5)
                    .resolved("N")
                    .build();
            syncFailureRepository.save(failure);
            return null;
        });
        log.warn("[SYNC_FAILURE_RECORDED] propertyId={}", propertyId);
    } catch (Exception e) {
        log.error("[SYNC_FAILURE_RECORD_FAILED] propertyId={}, error={}",
                propertyId, e.getMessage());
    }
}
```

## 7. 시나리오 5 — 재시도 상한 초과 → RDB 원복 (ROLLBACK)

### 7.1. 목적

MAX_RETRIES 초과 시 스케줄러가 Oracle DELETE + Redis 잔존 데이터 정리를 수행하는지 확인한다.

### 7.2. 사전 조건

1. 이전 테스트 데이터 정리:

```sql
DELETE FROM PROPERTIES_CHARTER WHERE PROPERTY_ID = 'aa2114b966c7d4aec522192ac905dc59';
DELETE FROM PROPERTY_SYNC_FAILURES;
COMMIT;
```

2. Redis 중지:

```bash
redis-cli SHUTDOWN NOSAVE
```

### 7.3. 요청

`jibun`: `"200-2"` (정리 후 재사용)

**Postman 응답:** `201 Created`

### 7.4. MAX_RETRIES 수동 조정 (테스트 시간 단축)

```sql
UPDATE PROPERTY_SYNC_FAILURES
SET MAX_RETRIES = 2
WHERE FAILURE_ID = 4 AND RESOLVED = 'N';
COMMIT;
```

MAX_RETRIES=5에서 2로 변경하여 스케줄러 3회 실행(약 3분)으로 ROLLBACK 분기 진입을 앞당겼다. retryCount가 0→1→2로 증가한 후, 3회차에서 retryCount(2) >= maxRetries(2) 조건을 충족하여 분기 1(상한 초과 → 원복)에 진입한다.

### 7.5. 결과

약 3분 대기 후 확인.

**실패 테이블:**

```sql
SELECT FAILURE_ID, RETRY_COUNT, RESOLVED, RESOLVED_METHOD
FROM PROPERTY_SYNC_FAILURES
WHERE FAILURE_ID = 4;
```

| FAILURE_ID | RETRY_COUNT | RESOLVED | RESOLVED_METHOD |
|------------|-------------|----------|-----------------|
| 4 | 5 | N | (null) |

RETRY_COUNT가 5까지 증가하여 MAX_RETRIES(테스트 조정값 2)를 초과했다.

**Oracle — 매물 DELETE 확인:**

```sql
SELECT COUNT(*) FROM PROPERTIES_CHARTER
WHERE PROPERTY_ID = 'aa2114b966c7d4aec522192ac905dc59';
```

→ 0건. 스케줄러가 `charterRepository.deleteById()`를 실행하여 매물을 삭제했다.

**Spring Boot 콘솔 로그:**

```

20:03:36.065 ERROR PropertySyncRetryScheduler - [SYNC_ROLLBACK] 재시도 상한 초과, RDB+Redis 원복 완료: propertyId=aa2114b966c7d4aec522192ac905dc59
```

Redis가 중지 상태이므로 `cleanupRedisResidual()`에서 Redis 잔존 정리는 실패했으나, 이 실패는 `try-catch`로 감싸져 있어 전체 ROLLBACK 흐름을 중단하지 않았다. Oracle DELETE와 실패 레코드 갱신은 정상 수행되었다. Redis 잔존 데이터는 Redis가 NOSAVE로 종료되어 존재하지 않으므로, 실질적 불일치는 없다.

### 7.6. 판정

**통과.** 재시도 상한 초과 시 Oracle DELETE 수행 및 ROLLBACK 해결 확인.

## 8. 시나리오 6 — RDB 커밋 실패 시 afterCommit 미실행 확인

### 8.1. 목적

Oracle ROLLBACK이 발생하면 afterCommit 콜백이 실행되지 않아, Redis에 유령 데이터가 절대 생성되지 않음을 확인한다. 이것은 afterCommit 기반 보상 아키텍처의 근본 보장이다.

### 8.2. 사전 조건

Redis 가동 상태. `createProperty()` 내부에서 `charterRepository.save(entity)` 직후, afterCommit 등록 직전에 임시로 RuntimeException 삽입:

```java
charterRepository.save(entity);

// ── F008 테스트: RDB 커밋 실패 시뮬레이션 ──
if (true) { throw new RuntimeException("[F008-TEST] RDB 커밋 실패 시뮬레이션"); }

TransactionSynchronizationManager.registerSynchronization(
```

이 예외가 발생하면 `@Transactional`이 감지하여 Oracle INSERT를 ROLLBACK한다. afterCommit 콜백 등록 코드에 도달하지 못하므로 콜백 자체가 예약되지 않는다.

### 8.3. 요청

`jibun`: `"200-2"` (이전 테스트 데이터 정리 후 재사용)

### 8.4. 결과

**Postman 응답:** `500 Internal Server Error`

**Spring Boot 콘솔 로그:**

```
20:23:27.721 INFO  PropertyWriteController - 전세 매물 등록 요청: userId=yhjj815, aptNm=F008검증아파트2, sggCd=11680
20:23:27.731 INFO  CharterPropertyWriteService - 매물 ID=aa2114b966c7d4aec522192ac905dc59
20:23:27.752 ERROR o.a.c.c.C.[.[.[/wherehouse] - RuntimeException: [F008-TEST] RDB 커밋 실패 시뮬레이션
```

afterCommit 관련 로그(`[REDIS_SYNC_RETRY]`, `[REDIS_SYNC_RETRY_OK]`, `[REDIS_SYNC_RETRY_FAIL]`, `[SYNC_FAILURE_RECORDED]`, `전세 매물 Redis 동기화(등록) 완료` 등)가 **일절 출력되지 않았다.**

**Oracle 확인:**

```sql
SELECT PROPERTY_ID, STATUS FROM PROPERTIES_CHARTER
WHERE APT_NM = 'F008검증아파트' AND JIBUN = '200-2';
```

→ 0건. ROLLBACK으로 INSERT 취소됨.

**실패 테이블 확인:**

```sql
SELECT COUNT(*) FROM PROPERTY_SYNC_FAILURES WHERE RESOLVED = 'N';
```

→ 0건. afterCommit이 실행되지 않았으므로 `recordSyncFailure()`도 호출되지 않았다.

**Redis 확인:** afterCommit이 실행되지 않았으므로 `syncRedisAfterCreate()`가 호출될 경로 자체가 없다. Redis에 신규 데이터 생성 없음.

### 8.5. 동작 원리 분석

```
createProperty() 진입 → @Transactional 트랜잭션 시작 (autoCommit=false)
  → save(entity)                              // Oracle INSERT 전송 (COMMIT 보류)
  → RuntimeException 발생                       // afterCommit 등록 코드에 미도달
  → @Transactional이 예외 감지 → ROLLBACK 실행
  → afterCommit 콜백이 예약되지 않았으므로 실행할 것 없음
  → 예외 전파 → 500 에러 응답
```

Oracle에 데이터 없음(ROLLBACK). Redis에 데이터 없음(afterCommit 미실행). 실패 테이블에 기록 없음(afterCommit 내부의 `recordSyncFailure()`가 호출되지 않음). 세 군데 모두 깨끗한 상태.

이것은 F008 시나리오2-B 테스트에서 발생한 최악의 불일치(Oracle ROLLBACK + Redis Hash 잔존 = 유령 데이터)가 afterCommit 구조에서는 **구조적으로 발생할 수 없음**을 실증한다.

### 8.6. 테스트 후 조치

삽입한 `if (true) { throw new RuntimeException("[F008-TEST] RDB 커밋 실패 시뮬레이션"); }` 코드를 제거하고 정상 코드로 복원했다.

### 8.7. 판정

**통과.** "Oracle 커밋이 실패하면 Redis는 아예 건드리지 않는다" — afterCommit 구조의 근본 보장 실증 완료.

## 10. 개발 과정에서 발견된 문제 및 개선 이력

### 10.1. afterCommit 내부 트랜잭션 컨텍스트 문제

| 항목 | 내용 |
|------|------|
| 발견 시점 | 시나리오 2 최초 실행 |
| 현상 | `syncFailureRepository.save()` 실행 및 `[SYNC_FAILURE_RECORDED]` 로그 출력 정상, 그러나 테이블 0건 |
| 원인 | afterCommit 시점에서 기존 트랜잭션이 완료되었으나 Spring 내부에 바인딩 잔존. `syncFailureRepository.save()` 내부의 `@Transactional`이 완료된 트랜잭션에 참여 시도하여 INSERT SQL은 전송되지만 별도 COMMIT 미발생 |
| 검토한 대안 | `recordSyncFailure()` 메서드에 `@Transactional(propagation = REQUIRES_NEW)` 선언 — **불가**. self-invocation(같은 클래스 내부 호출)에서는 Spring AOP 프록시를 거치지 않으므로 어노테이션이 무력화됨 |
| 적용한 해결 | 프록시에 의존하지 않는 프로그래밍 방식으로 독립 트랜잭션을 열어 INSERT + COMMIT |

## 11. 후속 과제 (1차 범위 밖)

### 11.1. 사용자 상태 통지

현재 구조에서 Redis 동기화 실패 시 사용자는 201 + STATUS=ACTIVE 응답을 받지만, 추천 검색에는 노출되지 않는다. ROLLBACK 시에도 사용자 통지 없이 매물이 삭제된다. `PENDING_SYNC` 중간 상태 도입, 또는 다음 조회 시 실패 건 존재 여부 체크를 통한 통지 채널이 필요하다.

### 11.2. 월세(MONTHLY) 확장

1차 구현은 CHARTER + CREATE 한정이다. `MonthlyPropertyWriteService`에 동일한 afterCommit + 보상 구조를 적용해야 한다. 이때 `recordSyncFailure()`를 별도 `@Service` 클래스로 추출하여 양쪽 서비스에서 공유하는 구조가 적절하다.

### 11.3. UPDATE / STATUS_CHANGE 확장

`updateProperty()`, `changeStatus()`에도 afterCommit + 보상 구조를 적용해야 한다. `OPERATION_TYPE` 컬럼이 이를 위해 설계되어 있으며, 스케줄러의 재시도 분기에서 `operationType`에 따라 호출 메서드를 분기하는 구조로 확장한다.

### 11.4. Redis 부분 실패 세분화 대응

현재 `FAIL_STEP='FULL'`로 sync 전체를 하나의 단위로 취급한다. Hash는 성공했지만 ZSet이 실패한 경우, 재시도 시 Hash를 중복 저장하는 비효율이 있다. `FAIL_STEP`을 `HASH`, `ZSET_PRICE`, `ZSET_AREA`, `BOUNDS_PRICE`, `BOUNDS_AREA`로 세분화하고, 실패 지점 이후부터만 재실행하는 구조가 2차 대상이다.
