## 1. 설계 범위 및 전제

### 1.1. 범위

`CharterPropertyWriteService.createProperty()` 단 하나.  
전세 매물 생성 시 RDB 커밋 후 Redis 동기화 실패에 대한 보상 아키텍처.

### 1.2. 전제

**판단 ①(실행 순서):** 대안 A(afterCommit) 적용.  
논의기록 1.2에서 유력으로 기록되었고, 판단 ②의 A+B 혼합과 자연스럽게 결합됨이 확인됨(논의기록 7.2). 판단 ③이 미확정이나, afterCommit 구조 자체는 판단 ③ 결과와 무관하므로 선행 적용 가능.

**판단 ②(전체 실패 보상):** A+B 혼합 확정(논의기록 5.7).  
일시적 장애 → 스케줄러 자동 복구. 장기 장애 → 상한 초과 시 RDB 원복.

**판단 ③(부분 실패 대응):** 미확정.  
1차에서는 syncRedis 내부의 5개 Redis 명령을 현재 코드 그대로(독립 순차 실행) 유지. 예외 발생 시 FAIL_STEP = "FULL"로 기록하고 스케줄러가 sync 전체를 재실행. 2차에서 Pipeline/MULTI-EXEC 선택 후 세분화.

### 1.3. 1차에서 의도적으로 하지 않는 것

- Redis 명령어별 개별 예외 처리 / 개별 재시도
- FAIL_STEP 세분화 (HASH / ZSET_PRICE / ZSET_AREA / BOUNDS_PRICE / BOUNDS_AREA)
- 부분 성공분 롤백 (Hash만 삭제 등)
- 명령어별 멱등성 / 동시성 보장
- MonthlyPropertyWriteService 변경
- updateProperty / changeStatus 변경

## 2. 현재 코드 흐름과 변경 목표

### 2.1. 현재 코드

```
@Transactional
createProperty() {
    validateDistrictCode()
    generatePropertyId()
    findById → 중복 체크
    charterRepository.save(entity)       // Oracle INSERT (커밋 보류)
    syncRedisAfterCreate(entity)         // Redis Hash + ZSet×2 + bounds×2
    return response
}  // 메서드 종료 시 COMMIT
```

### 2.2. 변경 후

```
@Transactional
createProperty() {
    validateDistrictCode()
    generatePropertyId()
    findById → 중복 체크
    charterRepository.save(entity)       // Oracle INSERT (커밋 보류)
    afterCommit 콜백 등록 {              // 콜백만 등록, Redis 작업은 아직 미실행
        syncRedisAfterCreate(entity)
    }
    return response
}  // 메서드 종료 → COMMIT 확정 → afterCommit 콜백 실행
```

### 2.3. 변경 근거

논의기록 0.1 핵심 결론:

> "RDB 커밋이 확정되기 전에 Redis에 쓰기가 발생하면, `@Transactional`이 오히려 불일치를 악화시킨다."

시나리오2-B 테스트에서 실증: `@Transactional` 적용 + Redis 부분 실패 → Oracle ROLLBACK + Redis Hash 잔존 = 유령 Hash(최악의 불일치).

afterCommit으로 이동하면 Oracle ROLLBACK 시 콜백 미실행 → 유령 데이터 구조적 차단.

## 3. afterCommit 콜백 내부 처리 흐름

논의기록 3.2 단계4 예외 맵 + 3.3 전체 흐름도 기반.

```
afterCommit() 실행
│
├─ try: syncRedisAfterCreate(entity)
│   └─ 성공 → 종료
│
├─ catch(RedisCommandTimeoutException | QueryTimeoutException)
│   │  일시적 장애. 논의기록 3.2: "재시도 1~2회"
│   └─ 200ms 대기 → 재시도 1회
│       ├─ 성공 → 종료
│       └─ 실패 → recordSyncFailure() → 스케줄러 위임
│
├─ catch(RedisConnectionFailureException)
│   │  연결 실패. 일시적일 수 있으므로 재시도 1회
│   └─ 200ms 대기 → 재시도 1회
│       ├─ 성공 → 종료
│       └─ 실패 → recordSyncFailure()
│
├─ catch(RedisCommandExecutionException)
│   │  OOM 또는 코드 버그. 논의기록 3.2: "재시도 의미 없음"
│   └─ 재시도 없이 즉시 recordSyncFailure() + log.error
│
└─ catch(Exception)
    └─ 예상 외 예외. 안전망. recordSyncFailure() + log.error
```

### 3.1. 사용자 응답과의 관계

afterCommit 내부 예외는 Controller에 전파되지 않는다(논의기록 1.2 대안A 단점). 사용자는 이미 201을 받은 상태. Redis 실패는 스케줄러가 후속 처리.

이것이 논의기록 4.3 수정된 비즈니스 원칙("RDB + Redis 모두 완료 = 성공")과 어떻게 양립하는가:

A+B 혼합 보상 전략(논의기록 5.7)이 이 gap을 메운다.
- 스케줄러가 복구하면 → 최종 성공.
- 상한 초과 시 RDB DELETE → 양쪽 모두 없는 상태 = 최종 실패.
- 어느 경우든 불일치가 무한히 지속되지 않는다.

## 4. 실패 테이블

### 4.1. DDL

논의기록 5.6 DDL 기반. 스케줄러 동작에 필수적인 컬럼을 추가.

```sql
CREATE TABLE PROPERTY_SYNC_FAILURES (
    FAILURE_ID      NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    PROPERTY_ID     VARCHAR2(32)  NOT NULL,
    LEASE_TYPE      VARCHAR2(10)  NOT NULL,
    OPERATION_TYPE  VARCHAR2(20)  NOT NULL,
    FAIL_STEP       VARCHAR2(50)  NOT NULL,
    FAIL_REASON     VARCHAR2(500),
    FAIL_TIME       TIMESTAMP DEFAULT SYSTIMESTAMP,
    RETRY_COUNT     NUMBER DEFAULT 0,
    MAX_RETRIES     NUMBER DEFAULT 5,
    RESOLVED        CHAR(1) DEFAULT 'N',
    RESOLVED_TIME   TIMESTAMP,
    RESOLVED_METHOD VARCHAR2(20)
);
```

### 4.2. 논의기록 5.6 DDL 대비 추가 컬럼 2개와 근거

**`OPERATION_TYPE`** — 현재는 항상 'CREATE'가 들어간다. 향후 UPDATE/STATUS_CHANGE 확장 시 스케줄러 재시도 분기 키로 사용. 논의기록 5.6의 스케줄러 개념 코드는 `syncRedisAfterCreate(entity)` 하드코딩이지만, 이것은 createProperty() 시나리오만 다룬 개념 코드이다. 지금 컬럼을 만들어두면 확장 시 DDL 변경 불필요.

**`RESOLVED_METHOD`** — 논의기록 5.4의 A+B 혼합에 RETRY(스케줄러 복구 성공) / ROLLBACK(상한 초과 원복) 두 분기가 존재. 어느 경로로 해결되었는지 기록.

### 4.3. 현재 범위에서 각 컬럼에 들어가는 값

| 컬럼 | 값 |
|------|-----|
| LEASE_TYPE | 'CHARTER' 고정 |
| OPERATION_TYPE | 'CREATE' 고정 |
| FAIL_STEP | 'FULL' 고정 (1차: Redis 명령 전체를 하나의 단위로 취급) |
| RESOLVED_METHOD | 'RETRY' 또는 'ROLLBACK' |

### 7.2. executeRedisSyncWithCompensation()

```java
private void executeRedisSyncWithCompensation(PropertyCharterEntity entity) {

    // 1차 시도
    try {
        syncRedisAfterCreate(entity);
        return;  // 성공 → 종료
    } catch (RedisCommandTimeoutException | QueryTimeoutException e) {
        // 일시적 장애 — 논의기록 3.2: "재시도 1~2회"
        log.warn("[REDIS_SYNC_RETRY] 일시적 장애, 재시도 진입: propertyId={}, cause={}",
                 entity.getPropertyId(), e.getClass().getSimpleName());
    } catch (RedisConnectionFailureException e) {
        // 연결 실패 — 일시적일 수 있으므로 재시도 1회
        log.warn("[REDIS_SYNC_RETRY] 연결 실패, 재시도 진입: propertyId={}",
                 entity.getPropertyId());
    } catch (RedisCommandExecutionException e) {
        // OOM 또는 코드 버그 — 논의기록 3.2: "재시도 의미 없음"
        log.error("[REDIS_SYNC_INFRA] 인프라 장애(OOM 등), 재시도 생략: propertyId={}, msg={}",
                  entity.getPropertyId(), e.getMessage());
        recordSyncFailure(entity.getPropertyId(), e.getMessage());
        return;
    } catch (Exception e) {
        // 예상 외 예외 — 안전망
        log.error("[REDIS_SYNC_UNEXPECTED] 예상 외 예외: propertyId={}, msg={}",
                  entity.getPropertyId(), e.getMessage());
        recordSyncFailure(entity.getPropertyId(), e.getMessage());
        return;
    }

    // 2차 시도 (재시도 1회)
    try {
        Thread.sleep(200);
        syncRedisAfterCreate(entity);
        log.info("[REDIS_SYNC_RETRY_OK] 재시도 성공: propertyId={}", entity.getPropertyId());
        return;  // 재시도 성공 → 종료
    } catch (InterruptedException ie) {
        Thread.currentThread().interrupt();
    } catch (Exception retryEx) {
        log.warn("[REDIS_SYNC_RETRY_FAIL] 재시도 실패: propertyId={}", entity.getPropertyId());
    }

    // 재시도도 실패 → 실패 테이블 기록
    recordSyncFailure(entity.getPropertyId(), "즉시 재시도 1회 실패 후 스케줄러 위임");
}
```

### 7.3. recordSyncFailure()

afterCommit 시점에서는 기존 `@Transactional` 트랜잭션이 이미 완료된 상태.  
`syncFailureRepository.save()`가 내부적으로 자체 트랜잭션을 열어 INSERT + COMMIT.

```java
private void recordSyncFailure(String propertyId, String failReason) {

    try {
        PropertySyncFailure failure = PropertySyncFailure.builder()
                .propertyId(propertyId)
                .leaseType("CHARTER")
                .operationType("CREATE")
                .failStep("FULL")
                .failReason(failReason != null && failReason.length() > 500
                           ? failReason.substring(0, 500) : failReason)
                .failTime(LocalDateTime.now())
                .retryCount(0)
                .maxRetries(5)
                .resolved("N")
                .build();

        syncFailureRepository.save(failure);

        log.warn("[SYNC_FAILURE_RECORDED] propertyId={}", propertyId);

    } catch (Exception e) {
        // 실패 기록 자체가 실패 — 최후 방어선
        log.error("[SYNC_FAILURE_RECORD_FAILED] propertyId={}, error={}",
                  propertyId, e.getMessage());
    }
}
```

### 7.4. retrySyncForCreate() — 스케줄러 호출용 public 메서드

현재 `syncRedisAfterCreate()`는 `private`. 스케줄러가 직접 호출할 수 없으므로 public 래퍼를 제공한다.

```java
public void retrySyncForCreate(String propertyId) {
    PropertyCharterEntity entity = charterRepository.findById(propertyId)
            .orElseThrow(() -> new PropertyNotFoundException(
                    "재시도 대상 매물이 존재하지 않습니다. propertyId=" + propertyId));

    syncRedisAfterCreate(entity);
}
```

예외를 catch하지 않는다. 호출자(스케줄러)가 try-catch로 감싸서 성공/실패를 판단한다.

## 8. syncRedisAfterCreate() — 변경 없음

현재 코드 그대로 유지. 5개 Redis 명령(Hash putAll + ZSet add × 2 + bounds tryExtend × 2)을 순차 실행.  
1차에서는 이 내부를 건드리지 않는다.

```java
private void syncRedisAfterCreate(PropertyCharterEntity entity) {
    String propertyId = entity.getPropertyId();
    String districtName = entity.getDistrictName();

    // ① Hash
    String hashKey = "property:charter:" + propertyId;
    Map<String, Object> hashFields = propertyHashBuilder.buildCharterHash(entity);
    redisHandler.redisTemplate.opsForHash().putAll(hashKey, hashFields);

    // ② ZSet — 가격 인덱스
    redisHandler.redisTemplate.opsForZSet().add(
            "idx:charterPrice:" + districtName, propertyId,
            entity.getDeposit().doubleValue());

    // ③ ZSet — 평수 인덱스
    redisHandler.redisTemplate.opsForZSet().add(
            "idx:area:" + districtName + ":전세", propertyId,
            entity.getAreaInPyeong().doubleValue());

    // ④⑤ bounds 확장
    String boundsKey = "bounds:" + districtName + ":전세";
    boundsUpdater.tryExtend(boundsKey, "minPrice", "maxPrice",
            entity.getDeposit().doubleValue(), PRICE_ZERO_DELTA);
    boundsUpdater.tryExtend(boundsKey, "minArea", "maxArea",
            entity.getAreaInPyeong().doubleValue(), AREA_ZERO_DELTA);

    log.info("전세 매물 Redis 동기화(등록) 완료: propertyId={}", propertyId);
}
```

## 9. 복구 스케줄러 — PropertySyncRetryScheduler

CREATE + CHARTER만 처리하므로 분기 로직이 단순하다.

### 9.1. 의존성

```
@Component @RequiredArgsConstructor
PropertySyncRetryScheduler

의존성:
  - PropertySyncFailureRepository
  - PropertyCharterRegistrationRepository
  - CharterPropertyWriteService
  - RedisHandler (상한 초과 시 Redis 잔존 정리용)
```

### 9.2. 스케줄러 로직

```
@Scheduled(fixedDelay = 60000)
retryFailedSync():

    미해결 목록 = failureRepository.findByResolvedOrderByFailTimeAsc("N")

    for each 실패:

        ────────────────────────────
        분기 1: retryCount >= maxRetries (상한 초과)
        ────────────────────────────

        // 1. Oracle DELETE — 논의기록 5.4 원복 전략
        charterRepository.deleteById(실패.propertyId)

        // 2. Redis 잔존 데이터 정리
        //    시나리오2 테스트에서 부분 실패 시 Hash가 남을 수 있음을 실증.
        cleanupRedisResidual(실패.propertyId)

        실패.resolved = "Y"
        실패.resolvedMethod = "ROLLBACK"
        실패.resolvedTime = now
        failureRepository.save(실패)
        log.error("[SYNC_ROLLBACK] 상한 초과, RDB+Redis 원복: {}", 실패.propertyId)
        continue

        ────────────────────────────
        분기 2: retryCount < maxRetries (재시도)
        ────────────────────────────

        try:
            charterWriteService.retrySyncForCreate(실패.propertyId)

            실패.resolved = "Y"
            실패.resolvedMethod = "RETRY"
            실패.resolvedTime = now
            failureRepository.save(실패)
            log.info("[SYNC_RETRY_OK] 재시도 성공 ({}회차): {}",
                     실패.retryCount + 1, 실패.propertyId)

        catch (PropertyNotFoundException e):
            // 매물이 다른 경로로 이미 삭제됨 (F003 등)
            cleanupRedisResidual(실패.propertyId)
            실패.resolved = "Y"
            실패.resolvedMethod = "ROLLBACK"
            실패.resolvedTime = now
            failureRepository.save(실패)
            log.warn("[SYNC_ENTITY_GONE] 매물 부재, Redis 정리 후 종료: {}",
                     실패.propertyId)

        catch (Exception e):
            실패.retryCount = 실패.retryCount + 1
            failureRepository.save(실패)
            log.warn("[SYNC_RETRY_FAIL] 재시도 실패 ({}회): {}",
                     실패.retryCount, 실패.propertyId)
```

### 9.3. cleanupRedisResidual

```
cleanupRedisResidual(propertyId):

    hashKey = "property:charter:" + propertyId

    // Hash에서 districtName 조회 — 부분 실패로 Hash가 남아있을 수 있음
    districtName = redisTemplate.opsForHash().get(hashKey, "districtName")

    if (districtName != null):
        // ZSet에서 member 제거 — 존재하지 않아도 Redis는 에러 없이 0 반환
        redisTemplate.opsForZSet().remove("idx:charterPrice:" + districtName, propertyId)
        redisTemplate.opsForZSet().remove("idx:area:" + districtName + ":전세", propertyId)

    // Hash 삭제 — 존재하지 않아도 Redis는 에러 없이 0 반환
    redisTemplate.delete(hashKey)

    // bounds는 정리하지 않는다.
    // bounds는 "경계 확장"이므로 매물 하나를 제거한다고 축소해야 하는 것이 아니다.
    // BoundsUpdater Javadoc: "경계 확장만 처리, 축소 없음. F009 승격 조건."
```

## 11. 검증 시나리오

논의기록 5.6 로컬 테스트 시나리오 7단계 기반. 대상이 전세 매물 생성 한정.

| # | 행동 | 검증 |
|---|------|------|
| 1 | Redis 정상 + 전세 매물 등록 | 201, Oracle 존재, Redis Hash+ZSet+bounds 존재, 실패 테이블 0건 |
| 2 | Redis 중지(`redis-cli SHUTDOWN`) | — |
| 3 | 전세 매물 등록 | 201, Oracle 존재, Redis 없음, 실패 테이블 1건 (CHARTER/CREATE/FULL/resolved=N) |
| 4 | 1분 대기 (스케줄러 1회 실행) | retry_count=1, resolved=N |
| 5 | Redis 재시작 | — |
| 6 | 1분 대기 | resolved=Y, resolvedMethod=RETRY, Redis Hash+ZSet 존재 |
| 7 | MAX_RETRIES=2로 설정 + Redis 중지 유지 + 전세 매물 등록 → 스케줄러 3회 | Oracle DELETE 확인, Redis 잔존 Hash DEL 확인, resolvedMethod=ROLLBACK |
| 8 | RDB 커밋 실패 시뮬레이션 (save 후 강제 예외) | afterCommit 콜백 미실행. Oracle 없음, Redis 없음, 실패 테이블 0건 |
