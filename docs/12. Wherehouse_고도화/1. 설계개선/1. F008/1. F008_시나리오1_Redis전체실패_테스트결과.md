# F008 문제 시나리오 1 — Redis 전체 실패 테스트 결과

**작성일:** 2026-04-28  
**대상 코드:** `CharterPropertyWriteService.createProperty()`  
**테스트 방식:** Postman 단발 요청 + 인위적 실패 주입 (syncRedisAfterCreate 진입부에 RuntimeException)

---

## 1. 문제 정의

매물 등록 요청 시, RDB(Oracle) 저장은 성공하지만 Redis 동기화가 전체 실패하는 상황이다. Redis 연결 장애, 타임아웃, Redis 서버 다운 등이 원인이 될 수 있다.

현재 코드 흐름:

```
createProperty() 진입
  → charterRepository.save(entity)     // Oracle INSERT
  → syncRedisAfterCreate(entity)       // Redis Hash + ZSet + bounds 반영
  → 응답 반환
```

`syncRedisAfterCreate()`가 전체 실패하면, Oracle에는 ACTIVE 매물이 존재하지만 Redis에는 아무것도 없다. 추천 검색(Redis Sorted Set 교집합 기반)에서 이 매물이 누락된다. 매물 목록 조회(F004, Oracle 직접 조회)에서는 표시되지만, 사용자가 추천 검색을 통해 이 매물을 발견할 수 없는 불일치 상태가 발생한다.

---

## 2. 인위적 실패 주입 방식

`syncRedisAfterCreate()` 메서드 진입 직후, 모든 Redis 작업 시작 전에 `RuntimeException`을 삽입하여 Redis 연결 실패를 시뮬레이션한다.

```java
private void syncRedisAfterCreate(PropertyCharterEntity entity) {
    // ── F008 테스트①: @Transactional 없이 Redis 전체 실패 ──
    // Oracle에만 데이터가 존재하고 추천 검색에서 누락되는 불일치를 실증한다.
    if(true) { throw new RuntimeException("[F008-TEST] 테스트①: Redis 연결 실패 시뮬레이션"); }

    String propertyId = entity.getPropertyId();
    // ... 이하 Redis 작업 (미실행)
}
```

## 4. 테스트 A — @Transactional 없이 Redis 전체 실패

### 4.1. 조건

- `createProperty()` 메서드에 `@Transactional` **미적용** (주석 상태)
- `syncRedisAfterCreate()` 진입부에 RuntimeException 삽입

### 4.2. 동작 원리

`@Transactional`이 없으면 `charterRepository.save(entity)` 내부에서 Spring Data JPA가 자체 트랜잭션을 열고, INSERT 실행 후 즉시 COMMIT한다. `save()` 호출이 끝나는 시점에 Oracle 데이터가 확정된다. 이후 `syncRedisAfterCreate()`에서 예외가 발생해도 이미 커밋된 Oracle 데이터를 되돌릴 수 없다.

```
createProperty() 진입
  → save(entity)                    // Oracle INSERT + 즉시 COMMIT (확정)
  → syncRedisAfterCreate(entity)    // RuntimeException 발생 → Redis 작업 미실행
  → 예외 전파 → 500 에러 응답
```

### 4.3. 결과

| 확인 대상 | 확인 방법 | 결과 |
|-----------|----------|------|
| API 응답 | Postman | 500 Internal Server Error, message: "[F008-TEST] 테스트①: Redis 연결 실패 시뮬레이션" |
| Oracle | `SELECT property_id, apt_nm, status, data_source FROM properties_charter WHERE apt_nm = 'F008테스트아파트';` | 1건 존재. PROPERTY_ID=5c627892ec0c5bdd218bdd1a70d9f931, STATUS=ACTIVE, DATA_SOURCE=USER |
| Redis Hash | `EXISTS property:charter:5c627892ec0c5bdd218bdd1a70d9f931` | 0 (없음) |
| Redis ZSet | `ZSCORE idx:charterPrice:강남구 5c627892ec0c5bdd218bdd1a70d9f931` | (nil) (없음) |

### 4.4. 문제 상태

Oracle에 ACTIVE 상태의 매물이 확정 저장되었으나, Redis에는 Hash도 인덱스도 존재하지 않는다. 이 매물은 매물 목록 조회(F004, Oracle 직접 조회)에서는 정상 표시되지만, 추천 검색(Redis Sorted Set 교집합 기반)에서는 누락된다. 사용자 관점에서 "등록했는데 추천에 안 나오는" 불일치가 발생한다.

## 5. 테스트 B — @Transactional 적용 후 Redis 전체 실패

### 5.1. 사전 작업

테스트 A에서 Oracle에 남은 레코드를 삭제한다.

```sql
DELETE FROM PROPERTIES_CHARTER WHERE PROPERTY_ID = '5c627892ec0c5bdd218bdd1a70d9f931';
COMMIT;
```

### 5.2. 조건

- `createProperty()` 메서드에 `@Transactional` **적용** (주석 해제)
- `syncRedisAfterCreate()` 진입부의 RuntimeException은 동일하게 유지

### 5.3. 동작 원리

`@Transactional`이 적용되면, 메서드 진입 시 Spring이 트랜잭션을 열고 `autoCommit=false`로 설정한다. `charterRepository.save(entity)` 호출 시 Oracle에 INSERT SQL이 전송되지만, COMMIT은 메서드가 정상 종료할 때까지 보류된다. 메서드 내부에서 RuntimeException이 발생하면 Spring이 ROLLBACK을 실행하여 INSERT를 취소한다.

`@Transactional`의 롤백 대상은 **RDB(JDBC Connection)에 전송된 SQL만**이다. Redis 작업, Java 변수, 기타 외부 시스템 호출은 롤백 대상이 아니다. 이 테스트에서는 Redis 작업 진입 전에 예외가 발생하므로 Redis에는 아무것도 반영되지 않는다.

```
createProperty() 진입 → @Transactional이 트랜잭션 시작 (autoCommit=false)
  → save(entity)                    // Oracle INSERT 전송 (COMMIT 보류)
  → syncRedisAfterCreate(entity)    // RuntimeException 발생 → Redis 작업 미실행
  → @Transactional이 예외 감지 → ROLLBACK 실행 → Oracle INSERT 취소
  → 예외 전파 → 500 에러 응답
```

### 5.4. 결과

| 확인 대상 | 확인 방법 | 결과 |
|-----------|----------|------|
| API 응답 | Postman | 500 Internal Server Error |
| Oracle | `SELECT property_id, apt_nm, status, data_source FROM properties_charter WHERE apt_nm = 'F008테스트아파트';` | 0건 (ROLLBACK으로 INSERT 취소됨) |
| Redis Hash | `EXISTS property:charter:5c627892ec0c5bdd218bdd1a70d9f931` | 0 (없음) |
| Redis ZSet | `ZSCORE idx:charterPrice:강남구 5c627892ec0c5bdd218bdd1a70d9f931` | (nil) (없음) |

### 5.5. 상태 분석

Oracle과 Redis 양쪽 모두 데이터가 없다. **데이터 정합성은 유지된다.** 그러나 사용자 관점에서는 매물 등록이 실패한 것이며, 500 에러 응답을 받는다. Redis 장애가 복구되면 사용자가 다시 등록을 시도해야 한다.

---

## 6. 테스트 A vs B 비교

| 구분 | @Transactional 없음 (테스트 A) | @Transactional 있음 (테스트 B) |
|------|-------------------------------|-------------------------------|
| Oracle | 데이터 있음 (즉시 커밋) | 데이터 없음 (롤백) |
| Redis | 데이터 없음 (실패) | 데이터 없음 (실패) |
| 정합성 | **불일치** — Oracle에만 존재 | 일치 — 양쪽 모두 없음 |
| 사용자 경험 | 500 에러, 재등록 시 DuplicatePropertyException | 500 에러, 재등록 가능 |
| 후속 문제 | 추천 검색 누락 + 재등록 불가 (PK 충돌) | 없음 (깨끗한 실패) |

## 7. 시사점

### 7.1. @Transactional 미적용의 치명적 결함

테스트 A에서 확인된 가장 심각한 문제는 단순 불일치가 아니라, **불일치 상태에서 자가 복구가 불가능하다**는 점이다. Oracle에 해당 PROPERTY_ID로 레코드가 이미 존재하기 때문에, 사용자가 동일 매물을 다시 등록하면 MD5 해시 중복으로 `DuplicatePropertyException`이 발생한다. Oracle에 데이터가 있으니 "이미 등록된 매물"이라는 응답을 받지만, 추천 검색에서는 찾을 수 없는 교착 상태에 빠진다.

### 7.2. @Transactional 적용의 한계

테스트 B에서 정합성은 유지되었지만, 이것은 **Redis 동기화가 RDB 트랜잭션 내부에서 실행**되기 때문이다. Redis 실패가 RDB 롤백을 유발하는 구조이므로, Redis 장애가 매물 등록 자체를 차단하는 결과를 낳는다. RDB 저장은 성공할 수 있는 상황인데도 Redis 때문에 전체가 실패하는 것은 과도한 결합이다.

### 7.3. 후속 설계 판단 필요 사항

이 테스트 결과는 다음의 후속 설계 판단이 필요함을 시사한다:

1. **RDB 커밋과 Redis 동기화의 실행 순서 설계** — RDB 커밋 확정 후 Redis 동기화를 실행하는 구조로 변경하면, Redis 실패가 RDB 롤백을 유발하지 않으면서도 시나리오 1(유령 데이터)을 방지할 수 있다.

2. **Redis 전체 실패 시 보상 전략** — RDB 커밋 성공 후 Redis 동기화가 전체 실패하면 어떻게 처리할 것인가 (수용+복구 / 재시도 / RDB 수동 롤백).

3. **Redis 부분 실패 시 보상 전략** — Redis 동기화 내부 작업(Hash, ZSet, bounds) 중 일부만 성공하면 어떻게 처리할 것인가.

이 세 판단은 순차적이며, 본 문서의 테스트 결과가 판단의 출발 근거가 된다.
