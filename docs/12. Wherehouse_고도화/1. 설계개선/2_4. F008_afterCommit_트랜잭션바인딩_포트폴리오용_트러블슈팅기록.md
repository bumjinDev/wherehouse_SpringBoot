# F008 Redis 동기화 실패 기록 누락 문제 — 포트폴리오용 트러블슈팅 기록

**작성일:** 2026-04-29  
**대상 기능:** F008 보상 아키텍처 1차 구현  
**대상 코드:** `CharterPropertyWriteService.recordSyncFailure()`  
**핵심 키워드:** `afterCommit()`, `@Transactional`, `PROPAGATION_REQUIRED`, `PROPAGATION_REQUIRES_NEW`, `TransactionTemplate`, JPA `save`, `commit`

---

## 1. 트러블슈팅 요약

Redis 캐시 동기화를 Oracle DB commit 이후 수행하기 위해 `afterCommit()` 콜백을 사용했다.  
Redis 장애 상황에서 동기화 실패 기록을 Oracle의 `PROPERTY_SYNC_FAILURES` 테이블에 저장하도록 구현했지만, `save()` 호출과 성공 로그에도 불구하고 실패 기록이 DB에 남지 않는 문제가 발생했다.

핵심 원인은 다음이었다.

```text
Repository save는 commit이 아니다.
afterCommit은 메인 트랜잭션 commit 이후 실행된다.
하지만 cleanup 이전이라 기존 트랜잭션 리소스가 남아 있을 수 있다.
이 상태에서 REQUIRED save는 새 트랜잭션을 열지 않고 기존 리소스에 참여할 수 있다.
기존 트랜잭션은 이미 commit이 끝났으므로 추가 commit이 없다.
```

해결은 `TransactionTemplate + PROPAGATION_REQUIRES_NEW`로 실패 기록 저장을 독립 트랜잭션으로 분리하는 것이었다.

## 2. 문제 상황

### 2.1. 의도한 흐름

F008 구현에서는 매물 등록 후 Redis 캐시 동기화를 수행했다. 중요한 점은 Redis 동기화가 Oracle DB commit 이후에 실행되어야 한다는 점이다.

의도한 흐름은 다음과 같았다.

```text
1. 사용자가 매물 등록 요청을 보낸다.
2. Oracle에 매물 본문 데이터를 저장한다.
3. Oracle 트랜잭션이 성공적으로 commit된 뒤 afterCommit()에서 Redis 동기화를 수행한다.
4. Redis가 중지되어 있으면 Redis 동기화는 실패한다.
5. Redis 동기화 실패 정보를 Oracle의 PROPERTY_SYNC_FAILURES 테이블에 기록한다.
6. 이후 별도의 재시도 로직 또는 운영자 확인 로직이 실패 기록을 기준으로 보상 처리한다.
```

이 설계에서 `afterCommit()`을 사용한 이유는 명확하다.

```text
Oracle에 매물 저장이 commit되지 않았는데 Redis에 먼저 반영되면 안 된다.
따라서 Redis 동기화는 DB commit이 성공한 뒤 실행되어야 한다.
```

즉 Redis 동기화 자체를 `afterCommit()`에 둔 결정은 적절했다. 문제가 된 것은 Redis 동기화 실패 이후, 다시 Oracle에 실패 기록을 저장하는 보조 작업이었다.

### 2.2. 실제 증상

Redis 서버를 중지한 상태에서 매물 등록을 검증했다.

Spring Boot 콘솔에는 다음과 같은 로그가 출력되었다.

```text
[SYNC_FAILURE_RECORDED]
```

이 로그는 코드 흐름상 다음을 의미한다.

```text
recordSyncFailure()가 호출되었다.
syncFailureRepository.save(failure)가 예외를 던지지 않았다.
try 블록이 정상적으로 끝났다.
로그 출력 라인까지 도달했다.
```

그런데 Oracle에서 직접 조회하면 실패 기록이 없었다.

```sql
SELECT *
FROM PROPERTY_SYNC_FAILURES
WHERE PROPERTY_ID = 'aa2114b966c7d4aec522192ac905dc59';

-- 0건 반환
```

증상은 다음과 같이 정리된다.

```text
save() 호출 성공
예외 없음
로그 출력 성공
하지만 Oracle 테이블에는 데이터 없음
```

따라서 원인은 `save()` 호출 여부가 아니라, `save()`가 실행된 트랜잭션 경계와 commit 타이밍에 있었다.

## 3. 원인 분석

### 3.1. 핵심 전제: save는 commit이 아니다

JPA에서 `repository.save(entity)`는 DB 변경을 현재 트랜잭션 또는 영속성 컨텍스트에 반영하는 작업이다.  
하지만 DB 변경을 최종 확정하는 것은 `commit`이다.

일반적인 `@Transactional` 메서드는 다음처럼 동작한다.

```text
@Transactional 메서드 진입
→ Spring이 트랜잭션 시작
→ repository.save(entity)
→ 엔티티 저장 작업 등록 또는 SQL flush
→ 메서드 정상 종료
→ Spring이 commit
→ DB 변경 최종 확정
```

일반적인 상황에서는 `save()` 이후에 commit 경계가 따라오기 때문에 데이터가 DB에 남는다.  
하지만 이번 문제에서는 `save()`가 이미 메인 commit이 끝난 뒤인 `afterCommit()` 내부에서 호출되었다.

### 3.2. 실제 실행 흐름

```text
createProperty()          ← @Transactional 적용
  → charterRepository.save(entity)
  → TransactionSynchronizationManager.registerSynchronization(afterCommit 콜백 등록)
  → 메서드 정상 종료

→ Spring @Transactional AOP가 메인 트랜잭션 commit 실행

→ afterCommit() 콜백 실행
  → executeRedisSyncWithCompensation(entity)
    → syncRedisAfterCreate(entity)    ← Redis 중지 상태이므로 실패
    → recordSyncFailure(propertyId, failReason)
      → syncFailureRepository.save(failure)
         ← REQUIRED 전파 속성으로 기존 트랜잭션 리소스에 참여한 것으로 처리될 수 있음
         ← 하지만 메인 commit은 이미 끝났고 이후 추가 commit은 없음

→ afterCompletion() 콜백 실행
→ cleanupAfterCompletion() 실행
  → 트랜잭션 리소스 정리
```

가장 중요한 구간은 다음이다.

```text
메인 트랜잭션 commit 완료
  ↓
afterCommit() 실행
  ↓
트랜잭션 리소스 cleanup 수행 전
```

`afterCommit()`은 commit 이후 실행되지만, Spring의 트랜잭션 리소스 정리보다 먼저 실행된다. 따라서 `afterCommit()` 시점에는 다음 두 조건이 동시에 성립할 수 있다.

```text
1. 메인 DB 트랜잭션 commit은 이미 끝났다.
2. 현재 스레드에는 아직 기존 트랜잭션 리소스가 남아 있을 수 있다.
```

### 3.3. REQUIRED 전파 속성과 추가 commit 부재

`syncFailureRepository.save(failure)`는 기본적으로 `@Transactional(REQUIRED)` 성격을 가진다.

`REQUIRED`의 규칙은 다음이다.

```text
현재 트랜잭션이 없으면 새 트랜잭션을 시작한다.
현재 트랜잭션이 있으면 새 트랜잭션을 시작하지 않고 기존 트랜잭션에 참여한다.
```

이번 경우에는 `afterCommit()` 시점에 기존 리소스가 아직 남아 있을 수 있으므로, `REQUIRED`가 새 트랜잭션을 열지 않고 기존 리소스에 참여한 것으로 처리될 수 있다.

그러나 기존 트랜잭션은 이미 commit을 완료했다.

```text
기존 트랜잭션 owner = createProperty()
기존 트랜잭션 commit = 이미 완료
syncFailureRepository.save(failure) = 기존 리소스에 참여한 것으로 처리될 수 있음
participant는 독립 commit을 수행하지 않음
```

결과적으로 다음 상황이 된다.

```text
save()는 호출되었다.
예외도 발생하지 않았다.
로그도 출력되었다.
하지만 해당 save를 확정해 줄 추가 commit이 없다.
따라서 실패 기록은 DB에 최종 반영되지 않는다.
```

이번 문제의 핵심 인과관계는 다음이다.

> `save()`는 commit이 아니며, `afterCommit()`은 이미 메인 commit 이후 실행되므로, `afterCommit()` 내부 RDB 쓰기는 별도의 commit 경계 없이는 DB 반영을 보장할 수 없다.

## 4. 해결 방법

해결 방향은 명확하다.

```text
afterCommit 내부에서 RDB에 실패 기록을 저장해야 한다면,
기존 트랜잭션 리소스에 기대지 말고 독립 트랜잭션을 새로 열어야 한다.
```

이를 위해 `TransactionTemplate`에 `PROPAGATION_REQUIRES_NEW`를 적용했다.

### 4.1. 수정 코드

`CharterPropertyWriteService`에 다음 필드를 추가한다.

```java
private final PlatformTransactionManager transactionManager;
```

필요한 import는 다음과 같다.

```java
import org.springframework.transaction.PlatformTransactionManager;
import org.springframework.transaction.TransactionDefinition;
import org.springframework.transaction.support.TransactionTemplate;
```

`recordSyncFailure()`를 다음과 같이 수정했다.

```java
private void recordSyncFailure(String propertyId, String failReason) {

    try {
        TransactionTemplate txTemplate = new TransactionTemplate(transactionManager);
        txTemplate.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRES_NEW);

        txTemplate.execute(status -> {
            PropertySyncFailure failure = PropertySyncFailure.builder()
                    .propertyId(propertyId)
                    .leaseType("CHARTER")
                    .operationType("CREATE")
                    .failStep("FULL")
                    .failReason(failReason != null && failReason.length() > 500
                            ? failReason.substring(0, 500)
                            : failReason)
                    .failTime(LocalDateTime.now())
                    .retryCount(0)
                    .maxRetries(5)
                    .resolved("N")
                    .build();

            syncFailureRepository.save(failure);
            return null;
        });

        log.warn("[SYNC_FAILURE_RECORDED] propertyId={}", propertyId);

    } catch (Exception e) {
        log.error("[SYNC_FAILURE_RECORD_FAILED] propertyId={}, error={}",
                propertyId, e.getMessage());
    }
}
```

### 4.2. 수정 후 흐름

```text
afterCommit()
→ recordSyncFailure()
→ TransactionTemplate.execute()
→ PROPAGATION_REQUIRES_NEW
→ 기존 트랜잭션 리소스가 남아 있어도 독립 트랜잭션 시작
→ syncFailureRepository.save(failure)
→ execute 블록 정상 종료
→ 새 트랜잭션 commit
→ 실패 기록 DB 확정
```

핵심은 `save()` 호출 자체가 아니라, `save()` 이후에 해당 변경을 확정할 독립적인 commit 경계를 만들었다는 점이다.

## 5. TransactionTemplate을 선택한 이유

`REQUIRES_NEW`를 적용하는 방법은 두 가지가 있다.

### 5.1. 별도 Service Bean 분리

```java
@Service
public class PropertySyncFailureService {

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void recordFailure(...) {
        syncFailureRepository.save(failure);
    }
}
```

이 방식은 구조적으로 깔끔하다. 트랜잭션 경계가 메서드 선언에 명시되기 때문이다.

### 5.2. TransactionTemplate 사용

```java
TransactionTemplate txTemplate = new TransactionTemplate(transactionManager);
txTemplate.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRES_NEW);

txTemplate.execute(status -> {
    syncFailureRepository.save(failure);
    return null;
});
```

이번 구현에서는 `recordSyncFailure()`가 같은 서비스 클래스 내부의 private 보조 메서드였다. 이 경우 같은 클래스 내부 메서드에 `@Transactional(REQUIRES_NEW)`를 붙이는 방식은 Spring AOP 프록시를 거치지 않아 기대대로 동작하지 않을 수 있다.

따라서 코드 블록 단위로 명시적인 트랜잭션 경계를 만들 수 있는 `TransactionTemplate`을 사용했다.

정리하면 선택 이유는 다음과 같다.

```text
1. afterCommit 내부에서 독립 commit 경계가 필요했다.
2. recordSyncFailure()는 private 보조 메서드였다.
3. 같은 클래스 내부 @Transactional은 self-invocation 문제로 안전하지 않다.
4. TransactionTemplate은 코드 블록 단위로 REQUIRES_NEW 트랜잭션을 명시할 수 있다.
```

## 7. 설계 판단

이번 문제에서 중요한 점은 `afterCommit()` 사용 자체가 잘못된 것이 아니라는 점이다.

Redis 동기화는 Oracle DB commit 이후 실행되어야 하므로 `afterCommit()`에 두는 것이 적절하다. 문제는 Redis 동기화 실패 후 다시 Oracle DB에 실패 기록을 남기는 RDB 쓰기 작업까지 같은 `afterCommit()` 내부에서 기본 `REQUIRED` 트랜잭션에 맡긴 것이다.

정리하면 다음과 같다.

```text
Redis 동기화 작업
→ DB commit 이후 실행되어야 하므로 afterCommit 사용이 적절함

Redis 실패 기록 저장 작업
→ afterCommit 내부에서 다시 RDB에 쓰는 작업이므로 별도 트랜잭션 필요
```

따라서 최종 설계 판단은 다음이다.

```text
afterCommit은 유지한다.
단, afterCommit 내부의 RDB 실패 기록 저장은 REQUIRES_NEW로 분리한다.
```

## 8. 주의점과 후속 고려 사항

`REQUIRES_NEW`는 이번 문제에 적합한 해결책이지만, 남용하면 안 된다.

첫째, `REQUIRES_NEW`는 독립 트랜잭션을 열기 때문에 추가 DB Connection이 필요할 수 있다.

```text
외부 트랜잭션 리소스 유지
+ REQUIRES_NEW 내부 트랜잭션 리소스 필요
```

동시 요청이 많은 상황에서 커넥션 풀 크기가 작으면 커넥션 고갈 위험이 생길 수 있다.

둘째, 실패 기록 저장 자체가 실패할 수 있다. 현재 코드는 실패 기록 저장 예외를 catch하고 로그를 남긴다. 이는 다음 의미를 가진다.

```text
매물 본문 저장은 이미 commit되었다.
Redis 동기화는 실패했다.
실패 기록 저장도 실패할 수 있다.
하지만 이미 응답 경계와 메인 commit 경계는 지나갔으므로 원본 등록을 되돌릴 수 없다.
```

더 강한 보장이 필요하다면 다음 구조도 고려할 수 있다.

```text
1. Outbox Pattern 도입
2. 실패 기록 저장을 메인 트랜잭션 안에서 먼저 남기고 이후 상태 갱신
3. Redis 동기화 전용 재시도 큐 도입
4. 실패 기록 저장 실패에 대한 별도 알림/모니터링 추가
```

현재 F008 단계에서는 `afterCommit + REQUIRES_NEW 실패 기록 저장`이 현실적인 해결책이다.

## 9. 최종 결론

이번 트러블슈팅의 결론은 다음이다.

```text
1. Repository save 호출 성공은 DB commit 성공을 의미하지 않는다.
2. JPA에서 save는 현재 트랜잭션에 변경을 반영하는 작업이고, 최종 확정은 commit이 한다.
3. afterCommit은 메인 트랜잭션 commit 이후 실행된다.
4. 하지만 cleanup 이전에 실행되므로 기존 트랜잭션 리소스가 남아 있을 수 있다.
5. 이 상태에서 REQUIRED save는 새 트랜잭션을 열지 않고 기존 리소스에 참여한 것으로 처리될 수 있다.
6. 기존 트랜잭션은 이미 commit이 끝났으므로 추가 commit은 발생하지 않는다.
7. 따라서 afterCommit 내부에서 RDB 쓰기를 수행해야 한다면 REQUIRES_NEW로 독립 트랜잭션을 열어야 한다.
```

한 문장으로 정리하면 다음과 같다.

> `afterCommit()` 내부의 Repository `save()`는 호출 성공만으로 DB 반영을 보장하지 않는다. `save()`는 commit이 아니며, `afterCommit()`은 이미 메인 commit 이후 실행되므로, 이 시점에서 RDB 쓰기를 해야 한다면 `REQUIRES_NEW`로 독립 트랜잭션을 열어 별도의 commit 경계를 만들어야 한다.
