## 0. 배경 — 보류했던 설계 판단의 경위

Phase 1 테스트 과정에서, 시나리오 테스트 외에 다음의 설계 판단이 필요하다는 것이 식별되었으나, "테스트 실증을 먼저 완료한 뒤 설계 판단에 진입하는 것이 순서"라는 판단 하에 보류하였다.

### 0.1. Phase 1 테스트에서 확정된 사실

| 테스트 | 조건 | Oracle | Redis Hash | Redis ZSet | 상태 |
|--------|------|--------|-----------|-----------|------|
| 시나리오1-A | `@Transactional` 없음 + Redis 전체 실패 | 있음 | 없음 | 없음 | 불일치 — 검색 누락 + 재등록 불가 |
| 시나리오1-B | `@Transactional` 있음 + Redis 전체 실패 | 없음 | 없음 | 없음 | 정합 — 깨끗한 실패 |
| 시나리오2-A | `@Transactional` 없음 + Redis 부분 실패 | 있음 | 있음 | 없음 | 반쪽짜리 — 상세 조회 가능, 추천 누락 |
| 시나리오2-B | `@Transactional` 있음 + Redis 부분 실패 | 없음 | 있음 | 없음 | **최악** — 유령 Hash, Oracle에 없는 데이터 |

**핵심 결론:** RDB 커밋이 확정되기 전에 Redis에 쓰기가 발생하면, `@Transactional`이 오히려 불일치를 악화시킨다. Redis 쓰기는 반드시 RDB 커밋 확정 이후에 실행되어야 한다.

### 0.2. 보류했던 설계 판단 3개의 구조

테스트 결과에서 도출된 문제를 해결하기 위해 3개의 설계 판단이 필요하며, 이 판단들은 순차적이다. ①을 먼저 정해야 ②의 전제가 확정되고, ②를 정해야 ③의 범위가 결정된다.

```
설계 판단 ①: RDB 커밋과 Redis 동기화의 실행 순서
  "RDB 커밋 확정 후 Redis 동기화" — 어떤 기술로 구현하는가
       ↓
설계 판단 ②: Redis 전체 실패 대응 (RDB 커밋 성공 + Redis 동기화 전체 실패)
  "재시도 / 수동 롤백 / 최종 일관성" — 어떤 보상 전략을 선택하는가
       ↓
설계 판단 ③: Redis 부분 실패 대응 (Redis 내부 일부만 성공)
  "성공분 롤백 / Pipeline / MULTI-EXEC" — 어떤 원자성 보장 방식을 선택하는가
```

이 3개의 판단은 Phase 1 테스트 중 다음의 과정에서 식별되었다:

시나리오 1 테스트 후, `@Transactional` 적용 여부와 Redis 실패의 조합에서 단일 대안으로 모든 문제를 해결할 수 없다는 것이 드러났다. 작업항목 문서의 "1-2. 대안 비교 분석"에 나열된 대안 A/B/C는 **판단 ①(실행 순서)만 다루고 있으며**, 판단 ②(전체 실패 시 보상)와 판단 ③(부분 실패 시 보상)은 별도의 설계 영역이라는 것이 확인되었다. 이 인식을 정리한 뒤, Phase 1 테스트를 먼저 완료하고 설계 판단 논의는 보류하였다.

## 1. 설계 판단 ① — RDB 커밋과 Redis 동기화의 실행 순서

### 1.1. 판단의 목표

"RDB 커밋이 확정된 후에만 Redis 동기화를 실행한다"를 구현한다.

```
[변경 전] save() → syncRedis() → 메서드 종료 시 커밋
[변경 후] save() → 메서드 종료 시 커밋 → (커밋 확정 후) syncRedis()
```

시나리오 2-B에서 확인한 "Redis에 부분 쓰기가 발생한 후 Oracle이 롤백되어 유령 데이터가 남는 문제"가 구조적으로 불가능해진다.

### 1.2. 대안 3개

#### 대안 A: TransactionSynchronizationManager.registerSynchronization(afterCommit)

```java
@Transactional
public PropertyCreateResponseDto createProperty(...) {
    PropertyCharterEntity entity = charterRepository.save(buildEntity(dto));

    TransactionSynchronizationManager.registerSynchronization(
        new TransactionSynchronization() {
            @Override
            public void afterCommit() {
                syncRedisAfterCreate(entity);
            }
        }
    );

    return buildResponse(entity);
}
```

- 동작: Spring 트랜잭션 인프라에 콜백을 등록한다. 메서드가 정상 종료하여 `@Transactional`이 COMMIT을 실행한 후, 등록된 afterCommit() 콜백이 호출된다. ROLLBACK 시에는 콜백이 실행되지 않는다.
- 장점: 기존 서비스 클래스 구조를 유지한다. `syncRedisAfterCreate()` 메서드 자체는 변경 없이, 호출 시점만 이동한다. 별도 클래스나 이벤트 인프라 추가 없이 서비스 메서드 내에서 완결된다.
- 단점: afterCommit() 내부에서 발생한 예외가 호출자(Controller)에게 전파되지 않는다. 즉 Redis 동기화가 실패해도 API 응답은 200이 반환된다. 또한 콜백 등록 코드가 비즈니스 로직 사이에 섞여 가독성이 떨어진다.

#### 대안 B: @TransactionalEventListener(phase = AFTER_COMMIT)

```java
// 서비스
@Transactional
public PropertyCreateResponseDto createProperty(...) {
    PropertyCharterEntity entity = charterRepository.save(buildEntity(dto));
    applicationEventPublisher.publishEvent(new PropertyCreatedEvent(entity));
    return buildResponse(entity);
}

// 별도 리스너 클래스
@Component
public class PropertyRedisSync {
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void onPropertyCreated(PropertyCreatedEvent event) {
        syncRedisAfterCreate(event.getEntity());
    }
}
```

- 동작: 서비스에서 이벤트를 발행하면, Spring이 트랜잭션 커밋 후에 리스너를 호출한다. ROLLBACK 시 리스너는 호출되지 않는다.
- 장점: 서비스 계층과 Redis 동기화 로직이 물리적으로 분리된다. 테스트 시 리스너를 교체하거나 비활성화하기 쉽다.
- 단점: 이벤트 클래스, 리스너 클래스 등 인프라 코드가 추가된다. 현재 프로젝트에 `ApplicationEventPublisher` 사용 패턴이 없으므로 새로운 아키텍처 패턴 도입이다. 대안 A와 마찬가지로 리스너 내 예외는 호출자에게 전파되지 않는다.

#### 대안 C: 서비스 메서드 분리

```java
// 서비스
@Transactional
public PropertyCharterEntity saveProperty(...) {
    return charterRepository.save(buildEntity(dto));
}

public void syncRedis(PropertyCharterEntity entity) {
    syncRedisAfterCreate(entity);
}

// 컨트롤러 또는 파사드
public PropertyCreateResponseDto createProperty(...) {
    PropertyCharterEntity entity = writeService.saveProperty(dto);
    writeService.syncRedis(entity);
    return buildResponse(entity);
}
```

- 동작: `@Transactional` 메서드는 RDB 저장만 수행하고 반환. 메서드 반환 시점에 커밋 완료. 호출자가 이후에 Redis 동기화를 별도 호출.
- 장점: 가장 명시적. 트랜잭션 경계가 코드에서 시각적으로 보인다. Redis 실패 시 예외가 호출자에게 전파된다.
- 단점: Controller 또는 Facade 계층에 동기화 호출 책임이 넘어간다. 새로운 write 메서드 추가 시 호출 누락 위험. 현재 Controller 3개(create, update, changeStatus) × 전세/월세 2개 = 6곳 모두 변경 필요.

### 1.3. 대안 비교표

| 기준 | 대안 A (afterCommit) | 대안 B (EventListener) | 대안 C (메서드 분리) |
|------|---------------------|----------------------|-------------------|
| 커밋 전 Redis 반영 차단 | ✓ | ✓ | ✓ |
| 기존 코드 변경 범위 | 서비스 메서드 내부만 | 이벤트+리스너 클래스 신규 | 서비스+컨트롤러 모두 |
| 동작 원리의 명시성 | 중 (콜백 등록 패턴) | 하 (이벤트 위상 이해 필요) | **상** (순차 호출) |
| Redis 실패 시 예외 전파 | 안 됨 | 안 됨 | **됨** (호출자가 try-catch 가능) |
| 호출 누락 위험 | 없음 (메서드 내부 완결) | 없음 (이벤트 발행만) | **있음** (호출자 책임) |
| 1인 프로젝트 현실성 | 높음 | 중간 | 높음 |

## 2. 판단 ①에 대한 문제 제기 — 기술 선택 전에 예외 처리 설계가 선행되어야 한다

### 2.1. 제기된 문제

설계 판단 ①의 대안 A/B/C를 비교하는 과정에서, 두 가지 근본적인 문제가 제기되었다.

**첫째: 대안 자체는 실행 순서만 다루고, 예외 발생 시 처리 설계가 없다.**

대안 A/B/C 어느 것을 선택하든, RDB 커밋 후 Redis 동기화 단계에서 예외가 발생하면 현재 구조에서는 사용자에게 무조건 실패 응답(500)을 반환하거나, 아무 응답 없이 내부에서 실패가 묻히게 된다. 그러나 예외에는 성격이 있다:

- 시스템이 내부적으로 재시도하면 해결 가능한 예외가 있고
- 사용자가 입력을 수정해야 하는 예외가 있고
- 아무도 즉시 해결할 수 없는 예외가 있다

이 세 유형에 대해 동일한 처리를 하는 것은 설계 부재다. 대안 A/B/C 모두 예외 처리 설계 없이 적용하면, 예외 유형과 무관하게 사용자에게 실패 응답만 반환하는 구조가 된다. 시스템이 재시도하면 해결 가능한 일시적 장애까지 사용자에게 그대로 전가되는 것이다.

**따라서 기술 선택 이전에 예외 분류와 처리 전략이 먼저 설계되어야 하고, 그 설계를 지원하는 기술을 선택하는 것이 올바른 순서다.**

**둘째: 운영 경험이 없어 어떤 예외가 실제로 발생하는지 모른다.**

포트폴리오 프로젝트이므로 실제 운영에서 발생하는 예외 유형을 경험적으로 알 수 없다. 이를 보완하기 위해 createProperty() 전체 흐름에서 발생 가능한 예외를 구체적으로 식별하고, 각각에 대해 적절한 처리 방식을 설계하는 작업이 필요하다.

### 2.2. 이 문제 제기가 설계 판단 흐름에 미친 영향

판단 ①의 대안을 먼저 선택하고 구현하려던 흐름을 중단하고, 예외 처리 설계를 먼저 수행한 뒤 그 설계가 대안 선택에 미치는 영향을 역으로 반영하는 순서로 전환하였다.

## 3. 예외 처리 설계 — 예외 3분류 원칙

### 3.1. 분류 체계

모든 예외를 3개 카테고리로 분류하고, 카테고리별 처리 방식을 다르게 한다.

| 카테고리 | 처리 방식 | 사용자 응답 |
|---------|----------|-----------|
| **사용자 오류** (입력 검증 실패, 중복 매물) | 재시도 안 함. 오류 원인을 구체적으로 안내 | 400/409 + 구체적 메시지 |
| **일시적 장애** (타임아웃, 잠금 경합, Redis 일시 불능) | 내부 재시도 1~2회. 재시도 성공 시 사용자는 실패를 모름 | 재시도 성공: 200 / 재시도 실패: 503 + "잠시 후 다시 시도해주세요" |
| **인프라 장애** (서버 다운, OOM, 네트워크 단절) | 재시도 불가. 로그 기록 + 운영 알림 | 503 + "일시적 오류" |

### 3.2. createProperty() 전체 흐름 예외 맵

#### 단계 1: 입력 검증 (Controller → DTO)

| 예외 | 원인 | 재시도 가능 | 적절한 응답 |
|------|------|-----------|-----------|
| `HttpMessageNotReadableException` | JSON 파싱 실패 (잘못된 형식) | 아니오 — 사용자 입력 오류 | 400 + "요청 형식이 올바르지 않습니다" |
| `MethodArgumentNotValidException` | 필수 필드 누락, 범위 초과 등 | 아니오 — 사용자 입력 오류 | 400 + 어떤 필드가 잘못됐는지 구체적 안내 |

사용자가 입력을 수정해야 해결된다. 시스템 재시도 의미 없음. 현재 `GlobalExceptionHandlerProperty`에서 이미 처리 중.

#### 단계 2: 비즈니스 검증 (Service 진입부)

| 예외 | 원인 | 재시도 가능 | 적절한 응답 |
|------|------|-----------|-----------|
| `PropertyValidationException` | 존재하지 않는 자치구 코드 | 아니오 — 사용자 입력 오류 | 400 + "유효하지 않은 자치구 코드입니다" |
| `DuplicatePropertyException` | 동일 매물 이미 등록됨 | 아니오 — 비즈니스 규칙 위반 | 409 + "이미 등록된 매물입니다" |

사용자 오류. 현재 코드에서 이미 커스텀 예외로 처리 중.

#### 단계 3: RDB 저장 (charterRepository.save)

| 예외 | 원인 | 재시도 가능 | 적절한 응답 |
|------|------|-----------|-----------|
| `DataIntegrityViolationException` | Oracle 제약 조건 위반 (PK 중복, NOT NULL 등) | 아니오 — 코드 버그 또는 데이터 문제 | 500 + 내부 오류 로그 기록 |
| `CannotAcquireLockException` | Oracle 행 잠금 대기 타임아웃 (다른 트랜잭션이 동일 행을 잠금 중) | **예** — 일시적 경합 | 내부 재시도 1~2회 후 실패 시 503 + "일시적 오류, 잠시 후 다시 시도해주세요" |
| `QueryTimeoutException` | Oracle 쿼리 실행 시간 초과 | **예** — 일시적 부하 | 내부 재시도 1회 후 실패 시 503 |
| `CannotCreateTransactionException` | HikariCP 커넥션 풀 소진 | **예** — 일시적 부하 | 내부 재시도 불가 (풀이 이미 소진), 503 + "서버가 일시적으로 바쁩니다" |
| `JDBCConnectionException` | Oracle 서버 다운, 네트워크 단절 | 아니오 — 인프라 장애 | 503 + "일시적 오류" + 운영 알림 로그 |

RDB 단계에서 재시도 가능한 예외(`CannotAcquireLockException`, `QueryTimeoutException`)가 존재한다. 일시적 경합이므로 수백 밀리초 후 재시도하면 성공 가능성이 높다. 반면 `JDBCConnectionException`은 서버 자체가 죽은 것이므로 재시도 의미 없음.

#### 단계 4: Redis 동기화 (syncRedisAfterCreate)

| 예외 | 원인 | 재시도 가능 | 적절한 응답 |
|------|------|-----------|-----------|
| `RedisConnectionFailureException` | Redis 서버 다운, 네트워크 단절 | 상황에 따라 — 서버 다운이면 불가, 일시적 네트워크면 가능 | 재시도 1~2회 후 실패 시 보상 처리로 이동 |
| `RedisCommandTimeoutException` | Redis 명령 실행 시간 초과 (느린 응답) | **예** — 일시적 부하 | 재시도 1~2회 |
| `RedisCommandExecutionException` | Redis OOM (`maxmemory` 초과), 잘못된 데이터 타입 | 아니오 (OOM) / 아니오 (코드 버그) | OOM: 운영 알림 + 보상 처리 / 버그: 로그 기록 |
| `QueryTimeoutException` (Lettuce) | Lettuce 클라이언트 타임아웃 | **예** — 일시적 | 재시도 1~2회 |

### 3.3. 설계 기반 전체 흐름도

```
[1단계] 입력 검증 (Controller)
  → 실패 시: 400 + 구체적 오류 안내. 재시도 의미 없음.

[2단계] 비즈니스 검증 (Service)
  → 실패 시: 400/409. 재시도 의미 없음.

[3단계] RDB 저장 (@Transactional)
  → 실패 유형별 분기:
    ├─ 사용자 오류 (제약 조건 위반): 400 + 오류 안내
    ├─ 일시적 장애 (잠금 경합, 타임아웃): 내부 재시도 1~2회 → 성공 시 4단계 진행 / 실패 시 503
    └─ 인프라 장애 (서버 다운): 503 + "일시적 오류"

[4단계] Redis 동기화 (RDB 커밋 확정 후)
  → 실패 유형별 분기:
    ├─ 일시적 장애 (타임아웃, 일시적 연결 실패): 즉시 재시도 1~2회
    │   ├─ 재시도 성공: 200 응답 (사용자는 실패를 모름)
    │   └─ 재시도 실패: [보상 처리]로 이동
    │
    └─ 인프라 장애 (서버 다운, OOM): 즉시 재시도 의미 없음 → [보상 처리]로 이동

[보상 처리] Redis 동기화 최종 실패 시
  → RDB에 이미 커밋된 매물을 어떻게 할 것인가? → 설계 판단 ②에서 결정
```

## 4. 비즈니스 요구사항 수정 — Redis 동기화의 필수성

### 4.1. 초기 원칙 (폐기)

예외 처리 설계 초기에 "RDB 저장 성공 = 사용자에게 200 응답, Redis 실패는 내부 처리"라는 원칙을 설정하였다. 근거는 매물 목록 조회(F004)가 Oracle에서 직접 조회하므로 Redis 실패 시에도 매물이 보인다는 것이었다.

### 4.2. 수정 근거

이 원칙은 Wherehouse의 비즈니스 맥락에 부합하지 않는다. Wherehouse의 핵심 기능은 **주거지 추천 검색**이며, 추천 검색은 Redis Sorted Set 교집합 기반이다. 매물이 Oracle에만 저장되고 Redis에 반영되지 않으면, 그 매물은 서비스의 핵심 기능에서 존재하지 않는 것과 같다. 매물 목록 조회(F004)에서 보인다는 것은 부차적이다. Redis 실패 시 단순히 "매물 조회 서비스만 수행된다"고 해서 사용자에게 부분적 성공 메시지를 보내서는 안 된다.

### 4.3. 수정된 원칙

**RDB 저장 + Redis 동기화 모두 완료되어야 "매물 등록 성공"이다. 어느 한쪽이라도 최종적으로 실패하면 정합성이 깨진 상태를 허용하지 않는다.**

이 원칙 수정이 설계에 미치는 영향: "Redis 실패를 수용한다"가 아니라 "Redis 실패 시 반드시 복구하거나, 복구 불가능하면 RDB도 원복한다"가 요구된다.

### 4.4. 수정된 원칙이 대안 선택에 미치는 영향

- 대안 A/B (afterCommit/EventListener): afterCommit 콜백/리스너 내 예외가 호출자에게 전파되지 않으므로, "RDB 커밋 성공 = 사용자에게 200 응답 + Redis는 내부에서 보상 처리"라는 구조와 자연스럽게 결합된다. Redis 실패 시 보상 전략(재시도, 실패 기록, 스케줄러 복구)을 콜백/리스너 내부에서 처리하면 된다.
- 대안 C (메서드 분리): 호출자가 Redis 예외를 받을 수 있으므로 "Redis 실패 시 사용자에게 다른 응답을 보낸다"는 전략도 가능하다. 하지만 6곳의 호출 지점(create/update/changeStatus × 전세/월세)마다 try-catch 패턴을 반복해야 하고, 한 곳이라도 빠뜨리면 Redis 실패가 사용자에게 500으로 노출된다.

## 5. 설계 판단 ② — Redis 전체 실패 시 보상 전략

### 5.1. 보상 전략 A: RDB 수동 롤백 (강한 정합성)

```
Redis 재시도 전부 실패
  → RDB에서 해당 매물 DELETE + COMMIT
  → 사용자에게 503 + "일시적 오류, 잠시 후 다시 시도해주세요"
```

- 장점: 어떤 시점에서든 Oracle과 Redis가 불일치하는 상태가 존재하지 않는다. 구현이 단순하다. 포트폴리오 규모에서 충분히 구현 가능하다.
- 단점: RDB 저장은 성공했는데 Redis 때문에 전체가 실패한다. Redis 장애가 지속되면 매물 등록 자체가 차단된다. DELETE 자체가 실패하면(Oracle 연결도 끊긴 경우) 불일치가 발생할 수 있다.
- 트레이드오프: **가용성을 희생하고 정합성을 선택한다.** Redis 장애 시 매물 등록 불가.

### 5.2. 보상 전략 B: 실패 기록 + 복구 스케줄러 (최종 일관성)

```
Redis 재시도 전부 실패
  → Oracle에 동기화 실패 기록 INSERT (실패 테이블: property_id, 실패 시각, 실패 단계, 재시도 횟수)
  → 사용자에게 200 + "등록 완료" (RDB 기준 성공)
  → 복구 스케줄러가 주기적으로 실패 테이블을 조회하여 Redis 재동기화 시도
```

- 장점: RDB 저장이 성공했으면 사용자에게 성공을 반환한다. Redis 장애가 매물 등록을 차단하지 않는다. 복구 스케줄러가 Redis 장애 복구 후 자동으로 정합성을 맞춘다. 실무에서 표준적인 패턴.
- 단점: 복구 스케줄러가 실행되기 전까지 일시적으로 추천 검색에서 누락된다. 실패 테이블, 스케줄러, 재동기화 메서드 등 구현 범위가 넓다. "일시적 불일치"를 허용하는 것이므로 순간적으로는 정합성이 깨진다.
- 트레이드오프: **정합성을 일시적으로 희생하고 가용성을 선택한다.** 최종적으로는 정합성이 보장되지만, 복구 완료까지 시간차가 존재한다.

### 5.3. 보상 전략 B의 한계 — Redis 장기 장애 시 사용자 경험 문제

보상 전략 B에 대해 다음의 의문이 제기되었다: "Redis 서버 자체가 장시간 다운되면, 스케줄러가 계속 실패하면서 추천 검색 누락이 지속된다. 사용자는 등록 완료 응답을 받았지만 추천에서 자기 매물을 찾을 수 없다. 이 사용자 경험은 어떻게 처리해야 하는가? 오류 유형에 따라 분리 처리할 수 있는가?"

이에 대한 분석:

**일시적 장애 (수초~수분)의 경우:** Redis 타임아웃, 일시적 네트워크 단절 등. 즉시 재시도 1~2회로 대부분 해결되고, 스케줄러까지 갈 일이 거의 없다. 사용자는 실패를 인지하지 못한다. 이 범위에서는 보상 전략 B가 잘 동작한다.

**장기 장애 (수십 분~수 시간)의 경우:** Redis 서버 다운, OOM 등. 스케줄러가 1분마다 재시도해도 계속 실패한다. 사용자는 "등록 완료"라는 응답을 받았지만 추천 검색에서 자기 매물을 찾을 수 없다. 이 시간 동안 사용자 경험이 깨진다.

장기 장애 완화 방법들과 한계:

| 완화 방법 | 동작 | 한계 |
|----------|------|------|
| 재시도 횟수 상한 설정 | N회 실패 후 사용자에게 알림(이메일, 앱 내 알림 등) | 알림 인프라 필요. 포트폴리오 규모 초과. |
| 매물 상태에 "동기화 대기" 표시 | 매물 목록에서 "추천 반영 중" 상태로 표시 | 구현 가능하지만 사용자가 이 상태를 이해할지 불명확 |
| Redis 장애 감지 시 서비스 공지 | 추천 검색 페이지에 "현재 일시적 장애" 배너 | 모니터링 인프라 필요 |

이것이 보상 전략 B의 본질적 트레이드오프다. "가용성을 선택한다"는 곧 "장기 장애 시 일시적으로 불완전한 상태를 허용한다"는 뜻이며, 그 불완전한 기간의 사용자 경험 관리는 별도의 설계 영역이다.

### 5.4. 보상 전략 A+B 혼합 — 재시도 + 상한 초과 시 원복

위 한계를 포트폴리오 범위 내에서 해결하기 위한 혼합 방식:

```
Redis 동기화 실패
  → 실패 테이블에 기록 (PROPERTY_SYNC_FAILURES)
  → 사용자에게 200 응답
  → 스케줄러가 재시도 (1분 간격)
    ├─ 재시도 성공: 실패 레코드 resolved 처리, 정합성 회복
    └─ 재시도 N회 초과: RDB에서 해당 매물 DELETE → 양쪽 모두 없는 상태로 원복
                        → 사용자에게 "등록 처리 중 오류 발생, 재등록 필요" 알림
```

동작:
- 일시적 장애: 스케줄러가 자동 복구 → 사용자는 실패를 모름
- 장기 장애: N회 초과 시 깨끗하게 원복 → 불일치 상태가 무한히 지속되지 않음
- N의 값 자체가 설계 판단이 됨 (예: 5회 = 5분 이내 복구 기대)

### 5.5. 보상 전략 비교표

| 기준 | 보상 전략 A (수동 롤백) | 보상 전략 B (최종 일관성) | A+B 혼합 |
|------|---------------------|----------------------|---------|
| 정합성 보장 수준 | **즉시 정합** | 최종 정합 (복구 전까지 불일치) | 최종 정합 + 상한 초과 시 즉시 정합 |
| Redis 장애 시 매물 등록 | **불가** | 가능 | 가능 (상한 초과 시 원복) |
| 구현 범위 | 작음 (DELETE + 예외 응답) | 큼 (실패 테이블 + 스케줄러 + 재동기화) | 큼 (B와 동일 + 상한 초과 로직) |
| 포트폴리오 설계 깊이 | 중간 | **높음** | **높음** |
| 실서비스 현실성 | 낮음 (과도한 결합) | **높음** | **높음** |

### 5.6. 로컬 구현 가능성

혼합 방식은 현재 환경(Oracle + Redis + Spring Boot)에서 전부 구현하고 테스트할 수 있다. 특별한 인프라 불필요.

#### 실패 테이블 DDL

```sql
CREATE TABLE PROPERTY_SYNC_FAILURES (
    FAILURE_ID      NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    PROPERTY_ID     VARCHAR2(32) NOT NULL,
    LEASE_TYPE      VARCHAR2(10) NOT NULL,
    FAIL_STEP       VARCHAR2(50) NOT NULL,
    FAIL_REASON     VARCHAR2(500),
    FAIL_TIME       TIMESTAMP DEFAULT SYSTIMESTAMP,
    RETRY_COUNT     NUMBER DEFAULT 0,
    MAX_RETRIES     NUMBER DEFAULT 5,
    RESOLVED        CHAR(1) DEFAULT 'N',
    RESOLVED_TIME   TIMESTAMP
);
```

#### 스케줄러 개념 코드

```java
@Scheduled(fixedDelay = 60000)
public void retryFailedSync() {
    List<PropertySyncFailure> failures = failureRepository.findUnresolved();

    for (PropertySyncFailure failure : failures) {
        if (failure.getRetryCount() >= failure.getMaxRetries()) {
            charterRepository.deleteById(failure.getPropertyId());
            failure.setResolved('Y');
            log.error("Redis 동기화 재시도 상한 초과, RDB 원복: {}", failure.getPropertyId());
            continue;
        }

        try {
            PropertyCharterEntity entity = charterRepository
                .findById(failure.getPropertyId()).orElseThrow();
            syncRedisAfterCreate(entity);
            failure.setResolved('Y');
            failure.setResolvedTime(LocalDateTime.now());
        } catch (Exception e) {
            failure.setRetryCount(failure.getRetryCount() + 1);
            log.warn("Redis 동기화 재시도 실패 ({}회): {}", failure.getRetryCount(), failure.getPropertyId());
        }
    }
}
```

#### 로컬 테스트 시나리오

| 단계 | 행동 | 확인 |
|------|------|------|
| 1 | Redis 서버 정상 상태에서 매물 등록 | 200 응답, Oracle + Redis 모두 존재 |
| 2 | Redis 서버 중지 (`redis-cli SHUTDOWN`) | — |
| 3 | 매물 등록 요청 | 200 응답, Oracle 존재, Redis 없음, 실패 테이블에 1건 기록 |
| 4 | 1분 대기 (스케줄러 1회 실행) | 실패 테이블 retry_count = 1, 여전히 미해결 |
| 5 | Redis 서버 재시작 | — |
| 6 | 1분 대기 (스케줄러 실행) | 실패 테이블 resolved = 'Y', Redis에 데이터 존재 확인 |
| 7 | 장기 장애 테스트: MAX_RETRIES를 2로 설정, Redis 중지 상태 유지 | 스케줄러 3회 실행 후 RDB DELETE 확인 |

### 5.7. 판단 ② 확정: A+B 혼합 선택

보상 전략 **A+B 혼합(재시도 + 상한 초과 시 원복)**을 확정한다. 선택 근거: 일시적 장애에서는 스케줄러가 자동 복구하여 사용자에게 실패를 노출하지 않고, 장기 장애에서는 상한 초과 시 RDB를 원복하여 불일치 상태가 무한히 지속되는 것을 방지한다. 실서비스 현실성과 포트폴리오 설계 깊이를 모두 충족하며, 로컬 환경에서 전체 흐름을 구현하고 테스트할 수 있다.

## 6. 설계 판단 ③ — Redis 부분 실패 대응

### 6.1. 판단의 배경

설계 판단 ②에서 **A+B 혼합(재시도 + 상한 초과 시 원복)**을 확정하였다. 이 보상 전략은 Redis 동기화가 **전체 실패**한 경우의 복구를 다룬다. 그러나 Phase 1 시나리오 2에서 실증한 바와 같이, Redis 동기화 내부 5개 작업(Hash, ZSet×2, bounds×2) 중 **일부만 성공하는 부분 실패**도 발생 가능하다.

부분 실패 시 어떤 작업까지 성공했고 어디서부터 실패했는지에 따라 불일치 형태가 달라지며, 이를 어떻게 대응할 것인가가 판단 ③의 영역이다.

### 6.2. 선택지 두 가지

**선택지 A: Redis Pipeline으로 묶기**
- 5개 명령을 파이프라인으로 일괄 전송하여 네트워크 라운드트립을 줄인다.
- 부분 실패 가능성이 감소하지만 완전히 제거되지는 않는다 (Pipeline은 트랜잭션이 아님).
- 코드 변경 범위가 작다.

**선택지 B: Redis MULTI/EXEC로 묶기**
- 5개 명령을 Redis 트랜잭션으로 묶어 전체 성공 or 전체 실패로 만든다.
- 부분 실패를 구조적으로 제거할 수 있다.
- 단, bounds 확장은 HGET → 비교 → HSET 흐름이므로 단순 MULTI로 묶기 어렵다 (읽기와 쓰기가 분리되어 있어 WATCH/MULTI/EXEC 또는 Lua 스크립트가 필요할 수 있다).
- 코드 변경 범위가 중간 이상이다.

### 6.3. 미확정 — 후속 세션에서 논의

Pipeline과 MULTI/EXEC 각각의 동작 원리를 학습하고, 현재 `syncRedisAfterCreate()` 코드에 적용했을 때의 구체적 변경 범위와 제약사항을 파악한 뒤 선택한다. 이 선택에 따라 보상 전략(판단 ②)의 스케줄러 재시도 방식과 실패 테이블의 FAIL_STEP 활용 방식도 함께 확정한다.

## 부록: 비즈니스 맥락 설계 한계 — 매물 등록 API의 단순성

### 한계 인식

F008 설계 과정에서 "현재 매물 등록 요청 자체가 비즈니스적으로 매우 단순하여, 비즈니스 맥락을 구체적으로 세분화하거나 다양한 상황을 가정하는 데 한계가 있다"는 문제가 제기되었다.

### 매물 등록의 비즈니스 흐름

```
사용자가 매물 정보를 입력한다 → 시스템이 저장한다 → 추천 검색에 반영된다
```

이 흐름에 비즈니스 분기가 거의 없다. 중개사 인증이 필요하지 않고, 결제가 붙지 않고, 다른 사용자의 승인이 필요하지 않고, 외부 API 연동(예: 국토부 실거래가 검증)이 등록 시점에 필요하지 않다. 입력값이 유효하고 중복이 아니면 저장하면 끝이다.

### F008에서 다루는 복잡성의 성격

현재 고민하고 있는 복잡성은 비즈니스 복잡성이 아니라 **인프라 정합성 복잡성**이다. "RDB와 Redis라는 두 저장소를 동시에 갱신해야 하는 구조에서 부분 실패를 어떻게 처리할 것인가"는 매물 등록이라는 비즈니스가 단순하든 복잡하든 동일하게 발생하는 기술적 설계 문제이며, 이 문제 자체가 이미 충분히 깊다.

### 인위적 비즈니스 복잡성을 추가하지 않는 근거

포트폴리오에서 존재하지 않는 비즈니스 요구사항을 억지로 추가하면 면접에서 "이 기능이 왜 필요합니까?"라는 질문에 답할 수 없다. 기존 면접 피드백의 핵심이 "기술 나열이 아니라 비즈니스 맥락에서 출발하라"였다. 비즈니스 맥락이 실재하지 않는 기능을 추가하면 그 피드백의 정반대를 하는 것이다. "단순한 기능인데 이런 설계 깊이를 보였다"가 오히려 강점이 된다.

### Wherehouse에서 비즈니스 복잡성이 실제로 존재하는 지점

매물 등록이 아닌 다른 F006~F009 항목들에서 비즈니스 맥락이 풍부한 설계 판단이 가능하다:

- **F006 (중복 매물 탐지):** 배치 데이터와 사용자 등록 데이터가 동시에 들어올 때 동일 매물인지 판별하는 문제. "같은 아파트, 같은 층, 같은 면적이지만 보증금이 다르면 동일 매물인가 다른 매물인가?" 이건 비즈니스 판단이다.
- **F007 (배치-사용자 데이터 머지):** 국토부 배치 데이터와 사용자 직접 등록 데이터가 충돌할 때 어느 쪽을 우선하는가. "사용자가 등록한 매물의 가격과 국토부 실거래가가 다르면?" 이것도 비즈니스 판단이다.
- **F009 (추천 점수 재계산):** 매물 변경 시 추천 점수를 언제, 어떻게 다시 계산하는가. "가격이 바뀌면 즉시 재계산하는가, 배치로 모아서 하는가?" 이건 사용자 경험과 시스템 부하의 트레이드오프다.

F008은 기술적 설계 깊이를 보여주는 항목이고, F006/F007/F009에서 비즈니스 맥락 깊이를 보완하는 것이 포트폴리오 전체의 균형에 적합하다.
