## 1. 문서 작성 배경

F009는 전세 추천 조회 API와 전세 매물 수정 API 사이의 Write-Read 경합 문제를 다룬다.

기존 전세 추천 조회 구조는 Redis ZSet 인덱스를 이용해 후보 매물 ID를 빠르게 줄이고, 이후 Redis Hash에서 매물 상세 정보를 읽어 점수 계산과 응답 생성을 수행한다. 쓰기 쪽에서는 매물 수정 이후 Redis Hash와 ZSet 인덱스를 갱신한다.

문제는 매물 수정과 추천 조회가 동시에 실행될 때, Reader가 어느 시점의 데이터를 관찰하느냐에 따라 응답 내부에서 자기모순이 발생할 수 있다는 점이다. 특히 사용자의 조회 조건이 `budgetMax=60,000`인데 응답 매물의 `price=80,000`으로 내려가는 상태가 실제 테스트에서 재현되었다.

초기에는 이 문제가 Writer의 Redis 갱신이 `Hash putAll`과 `ZSet add`를 개별 커맨드로 순차 실행하기 때문에 발생한다고 판단하였다. 그래서 Writer에 Redis `MULTI/EXEC`를 적용하였다. 그러나 이후 재테스트에서 Writer의 Redis 쓰기 원자성은 확보되었음에도, Reader의 2단계 읽기 구조 때문에 금지 상태가 여전히 재현되었다.

따라서 F009의 핵심은 단순히 Writer의 Redis 갱신을 원자화하는 수준이 아니라, 추천 조회 API가 비즈니스적으로 어떤 정합성 계약을 보장해야 하는지 정의하고, 그 계약에 맞게 Reader 파이프라인의 책임 경계를 재설계하는 것이다.

## 3. F009-TEST-REPORT-001 — 개별 커맨드 방식에서의 경합 재현

### 3.1 테스트 목적

F009-TEST-REPORT-001의 목적은 Redis Hash와 ZSet 인덱스를 개별 커맨드로 순차 갱신하는 구조에서, Writer와 Reader가 동시에 실행될 때 Reader가 Writer의 중간 상태를 관찰하는지 검증하는 것이었다.

테스트에서 사용한 핵심 시나리오는 다음과 같다.

```text
전제:
- 매물 A의 기존 deposit = 50,000
- 매물 A의 가격 ZSet score = 50,000
- 사용자 조회 조건 budgetMax = 60,000

수정 요청:
- 매물 A의 deposit을 80,000으로 변경

위험 시퀀스:
1. Writer가 Redis Hash를 먼저 갱신한다.
   - Hash deposit = 80,000
2. Writer가 아직 ZSet을 갱신하지 못한다.
   - ZSet score = 50,000
3. Reader가 ZSet으로 후보를 선정한다.
   - A는 score=50,000이므로 후보에 포함된다.
4. Reader가 Hash에서 상세를 조회한다.
   - deposit=80,000을 읽는다.

결과:
- budgetMax=60,000 조건의 추천 응답에 price=80,000 매물이 포함된다.
```

이 상태는 비즈니스적으로 금지 상태다. 사용자는 예산 상한을 60,000으로 요청했는데, 응답은 80,000 매물을 “조건에 맞는 전세 매물”로 보여주기 때문이다.

### 3.2 테스트 판정 기준

F009-TEST-REPORT-001의 판정 기준은 다음과 같다.

| Reader 응답 상태 | 판정 |
|---|---|
| TARGET_A 포함 + price=80,000 | 경합 재현 성공, 금지 상태 |
| TARGET_A 포함 + price=50,000 | 수정 전 일관 상태이므로 정상 |
| TARGET_A 미포함 | 수정 후 일관 상태이므로 정상 |

핵심은 `TARGET_A`가 포함되었는지 자체가 아니라, 포함되었을 때 응답의 표시 가격이 사용자의 hard condition을 위반하는지 여부다.

### 3.3 F009RaceLatch의 역할

F009RaceLatch는 우연한 타이밍에 의존하지 않고 Writer와 Reader의 실행 순서를 강제로 고정하기 위한 테스트 장치다.

기존 개별 커맨드 방식에서는 다음 순서를 만든다.

```text
Writer:
1. Hash putAll 실행
2. syncAwait()에서 대기

Reader:
3. ZSet 가격/면적 인덱스 조회
4. 교집합 후보 ID 확정
5. releaseWriterIfTargetIncluded() 호출

Writer:
6. 대기 해제
7. ZSet add 실행
```

이 구조를 통해 Reader는 의도적으로 `Hash=신값`, `ZSet=구값` 상태를 관찰한다.

### 3.4 1차 결론

F009-TEST-REPORT-001 기준으로는 Writer의 Redis 동기화가 다음처럼 분리되어 있었기 때문에 문제가 발생하였다.

```text
Hash putAll
→ 중간 간극
→ ZSet add
```

따라서 1차 해결 방향은 Writer의 Redis 갱신을 `MULTI/EXEC`로 묶어 다음 상태만 허용하는 것이었다.

```text
허용 상태 A:
- Reader가 수정 전 전체 상태를 관찰한다.
- Hash deposit = 50,000
- ZSet score = 50,000
- TARGET_A 포함 + price=50,000

허용 상태 B:
- Reader가 수정 후 전체 상태를 관찰한다.
- Hash deposit = 80,000
- ZSet score = 80,000
- TARGET_A는 budgetMax=60,000 가격 범위에서 탈락한다.
```

즉, Writer의 `Hash=신값`, `ZSet=구값` 중간 상태를 없애는 것이 첫 번째 조치였다.

## 4. F009-TEST-REPORT-002 — Writer MULTI/EXEC 적용 후에도 남은 문제

### 4.1 Writer MULTI/EXEC 적용 내용

F009-TEST-REPORT-002에서는 `CharterPropertyWriteService.syncRedisAfterUpdate()`에 Redis `MULTI/EXEC`를 적용하였다.

변경 후 구조는 다음과 같다.

```java
redisHandler.redisTemplate.execute(new SessionCallback<List<Object>>() {
    @Override
    public List<Object> execute(RedisOperations operations) throws DataAccessException {
        operations.multi();

        operations.opsForHash().putAll(hashKey, hashFields);

        if (f009RaceLatch != null) {
            f009RaceLatch.syncAwait(propertyId);
        }

        if (changedFields.contains("deposit")) {
            operations.opsForZSet().add(
                    "idx:charterPrice:" + districtName,
                    propertyId,
                    entity.getDeposit().doubleValue());

            String boundsKey = "bounds:" + districtName + ":전세";
            boundsUpdater.tryExtend(
                    boundsKey,
                    "minPrice",
                    "maxPrice",
                    entity.getDeposit().doubleValue(),
                    PRICE_ZERO_DELTA);
        }

        return operations.exec();
    }
});
```

이 구조에서 `Hash putAll`과 `ZSet add`는 `EXEC` 전까지 Redis 실제 데이터에 반영되지 않고 QUEUED 상태가 된다.

### 4.2 Writer 쪽에서 해결된 것

Writer MULTI/EXEC 적용 이후, 개별 커맨드 방식에서 발생했던 다음 중간 상태는 제거되었다.

```text
Hash deposit = 80,000
ZSet score = 50,000
```

즉, Writer 내부에서 Hash와 ZSet의 물리적 갱신 순서가 외부 Reader에게 노출되는 문제는 해결되었다.

F009-TEST-REPORT-002의 최종 Redis 상태도 다음처럼 정합하게 끝났다.

```text
Hash deposit = 80,000
ZSet score = 80,000
```

따라서 Writer MULTI/EXEC는 실패한 조치가 아니다. 이것은 여전히 유지해야 하는 1계층 방어다.

### 4.3 그런데도 금지 상태가 재현된 이유

Writer MULTI/EXEC 적용 후에도 다음 순서가 가능했다.

```text
1. Reader가 강남구 가격 ZSet을 조회한다.
   - 아직 Writer EXEC 전이므로 TARGET_A score=50,000이 보인다.

2. Reader가 강남구 면적 ZSet을 조회한다.
   - TARGET_A area=30이 보인다.

3. Reader가 가격/면적 교집합을 계산한다.
   - TARGET_A가 후보 ID에 포함된다.

4. Reader가 latch를 해제한다.

5. Writer가 EXEC를 수행한다.
   - Hash deposit=80,000
   - ZSet score=80,000

6. Reader가 이후 단계에서 Hash 상세를 읽는다.
   - TARGET_A deposit=80,000을 읽는다.

7. 응답에 TARGET_A price=80,000이 포함된다.
   - 사용자 조건 budgetMax=60,000과 충돌한다.
```

이 문제는 더 이상 Writer의 `Hash putAll`과 `ZSet add` 사이의 개별 커맨드 간극이 아니다. 문제의 초점은 Reader의 2단계 읽기 구조로 이동한다.

## 5. 현재 Reader 구조의 핵심 문제

### 5.1 현재 흐름

현재 `CharterRecommendationService`의 전세 추천 조회 흐름은 다음과 같다.

```text
getCharterDistrictRecommendations()
  → S-01 performCharterStrictSearch()
       → findValidCharterPropertiesInDistrict()
          - 가격 ZSet 조회
          - 면적 ZSet 조회
          - Java retainAll()로 교집합
          - List<String> propertyIds 반환
  → S-02 checkAndPerformCharterFallback()
       - propertyIds 개수 기준으로 fallback 판단
  → S-04 calculateCharterPropertyScores()
       - 전체 propertyId 추출
       - ReviewStatistics RDB chunk 조회
       - getMultipleCharterPropertiesFromRedis(propertyIds)로 Hash 상세 조회
       - 점수 계산
  → S-05 지역구 점수 계산 및 정렬
  → S-06 최종 응답 생성
```

문제는 S-01이 “실제 유효 매물”을 반환하는 것이 아니라 “인덱스 기준 후보 ID”만 반환한다는 점이다.

### 5.2 `findValidCharterPropertiesInDistrict()`의 이름과 실제 책임 불일치

현재 메서드명은 `findValidCharterPropertiesInDistrict()`다. 이름만 보면 “유효한 전세 매물”을 찾는 메서드처럼 보인다.

그러나 실제 책임은 다음 수준에 머물러 있다.

```text
- 가격 ZSet 범위 조회
- 면적 ZSet 범위 조회
- 두 결과의 ID 교집합
- ID 목록 반환
```

즉, 현재 메서드가 반환하는 것은 “valid property”가 아니라 “valid할 가능성이 있는 propertyId 후보”다.

이 상태에서 S-02는 이 ID 개수만 보고 fallback 여부를 판단한다. 그러나 ZSet 후보 ID 개수와 Hash 상세 기준으로 검증된 실제 유효 매물 개수는 다를 수 있다.

### 5.3 S-04 상세 조회 지연의 문제

`getMultipleCharterPropertiesFromRedis(propertyIds)`가 S-04에서 호출되면 다음 문제가 생긴다.

```text
S-01:
- ZSet 기준 후보 ID 3개 확보

S-02:
- 후보 ID가 3개이므로 fallback 불필요로 판단

S-04:
- 그제야 Hash 상세 조회
- 실제로는 1개가 budgetMax를 초과함

결과:
- fallback 판단은 이미 끝났다.
- 조건 밖 매물이 점수 계산과 응답 생성으로 흘러갈 수 있다.
```

따라서 S-04의 상세 조회를 `findValidCharterPropertiesInDistrict()` 내부로 앞당겨야 한다.

이 변경의 목적은 “전체 조회 요청의 완전한 스냅샷을 만들기 위해서”가 아니다. 목적은 검색 단계에서 실제 유효 매물을 확정한 뒤, S-02 fallback 판단과 S-04 점수 계산이 검증된 입력을 받도록 만드는 것이다.

## 6. 비즈니스 정합성 모델 재정의

### 6.1 모든 시간 간극 제거는 목표가 아니다

추천 조회 요청은 다음과 같은 시간 단계를 가진다.

```text
1. 사용자가 UI에서 조회 버튼을 누른다.
2. 브라우저/클라이언트가 HTTP 요청을 전송한다.
3. 서버가 요청을 수신한다.
4. Reader가 Redis 가격 인덱스를 조회한다.
5. Reader가 Redis 면적 인덱스를 조회한다.
6. Java에서 교집합을 계산한다.
7. 교집합 ID로 Redis Hash 상세를 조회한다.
8. 상세값 기준으로 조건을 검증한다.
9. ReviewStatistics를 RDB에서 chunk 조회한다.
10. 점수를 계산한다.
11. 지역구 점수를 계산하고 정렬한다.
12. 최종 응답을 생성한다.
13. HTTP 응답이 클라이언트에 도착한다.
```

이 과정 중 어느 시점에도 매물 수정 요청이 들어올 수 있다.

따라서 조회와 수정을 전역 직렬화하지 않는 이상, “사용자가 버튼을 누른 순간의 완전한 스냅샷”을 보장할 수 없다. 또한 추천 조회 서비스에서 그런 강한 정합성을 요구하는 것도 일반적으로 적절하지 않다.

전세 추천 조회는 결제, 계좌 잔액, 재고 차감과 같은 강한 직렬화 업무가 아니다. Redis read model을 기반으로 빠르게 후보를 탐색하고, 사용자 조건에 맞는 추천 결과를 제공하는 서비스다.

따라서 목표는 모든 간극을 제거하는 것이 아니라, 비즈니스적으로 허용 가능한 간극과 금지해야 하는 불일치를 구분하는 것이다.

### 6.2 허용 가능한 간극

다음은 추천 조회 서비스에서 허용 가능한 간극이다.

| 상황 | 허용 이유 |
|---|---|
| 조회 직후 새로 등록된 매물이 이번 응답에 포함되지 않는다. | 추천 조회는 요청 시점 전체 스냅샷을 보장하지 않는다. 신규 매물은 다음 조회에서 반영될 수 있다. |
| 조회 중 수정된 매물이 이번 응답에서 제외된다. | 조건에 맞는지 불확실하거나 조건을 벗어났다면 제외하는 것이 안전하다. |
| 조회 중 수정된 매물이 응답에 포함되지만, 응답에 표시되는 값 기준으로 사용자의 hard condition을 만족한다. | 응답 내부 자기모순이 없으므로 허용 가능하다. |
| 경합으로 인해 추천 매물 개수가 줄어든다. | 추천 개수 확보는 hard condition이 아니라 서비스 품질 목표다. |
| 추천 점수와 정렬이 응답 생성 과정에서 읽은 read model 기준으로 산출된다. | 추천 점수는 실시간 트랜잭션 정합성보다 read model 기반 탐색 성격이 강하다. |

### 6.3 금지해야 하는 불일치

다음은 반드시 금지해야 하는 불일치다.

| 상황 | 금지 이유 |
|---|---|
| `budgetMax`를 초과하는 매물이 예산 조건 내 추천으로 응답된다. | 응답 본문이 사용자 요청 조건과 직접 충돌한다. |
| `areaMin`/`areaMax` 범위를 벗어난 매물이 응답된다. | 사용자의 명시 조건을 위반한다. |
| `ACTIVE`가 아닌 매물이 일반 추천 결과에 노출된다. | 거래완료/삭제 등 상태 정책 위반 가능성이 있다. |
| 전세 추천 API에 전세가 아닌 매물이 포함된다. | API 의미와 응답 데이터가 충돌한다. |
| “조건 내 추천 매물”이라고 설명하면서 응답 본문에는 조건 밖 매물이 포함된다. | 사용자 경험과 데이터 신뢰성을 동시에 훼손한다. |

### 6.4 F009의 최종 정합성 계약

F009의 목표는 다음 문장으로 고정한다.

```text
추천 조회 API는 요청 시작 시점의 완전한 스냅샷을 보장하지 않는다.
대신 응답에 포함되는 모든 매물은 응답 생성 과정에서 Redis Hash로 확인한 실측값 기준으로 사용자의 hard condition을 만족해야 한다.
```

이 계약에 따르면, F009의 문제는 “조회 중 매물이 바뀌었다”가 아니다. 문제는 “응답에 표시된 매물 값이 사용자 조건을 위반했다”는 것이다.

## 7. Hard Condition과 Soft Condition 분리

### 7.1 Hard Condition

Hard condition은 응답에 포함되는 매물이 반드시 만족해야 하는 조건이다. 이 조건을 위반한 매물은 응답에서 제거되어야 하며, 점수 계산이나 fallback 판단의 입력으로 사용되면 안 된다.

전세 추천 API 기준 hard condition은 다음과 같다.

```text
- leaseType = 전세
- status = ACTIVE
- deposit >= budgetMin
- deposit <= budgetMax
- areaInPyeong >= areaMin
- areaInPyeong <= areaMax
- 필요 시 districtName 일치
```

이 조건은 Redis ZSet 인덱스 결과만으로 최종 판정하면 안 된다. ZSet은 후보 축소용 자료구조일 뿐이며, 최종 판정 기준은 Hash 상세의 실측값이어야 한다.

### 7.2 Soft Condition

Soft condition은 추천 품질을 높이기 위한 조건이다. 부족하면 fallback 또는 완화 정책의 대상이 될 수 있다.

예시는 다음과 같다.

```text
- 가격/안전/공간 우선순위
- 추천 점수 산정
- 지역구 대표 점수
- 추천 매물 3개 확보
- fallback 단계에서 완화 가능한 조건
```

추천 매물 3개 확보는 서비스 품질 목표이지, hard condition이 아니다. 따라서 3개를 맞추기 위해 조건 밖 매물을 포함하거나 무한 재시도하는 것은 잘못된 설계다.

## 8. Reader 인덱스 조회에 MULTI/EXEC를 적용하는 의미

### 8.1 적용 목적

Reader의 `findValidCharterPropertiesInDistrict()` 내부에는 최소 두 개의 Redis ZSet 조회가 있다.

```text
1. idx:charterPrice:{district}
2. idx:area:{district}:전세
```

현재 이 두 조회는 분리된 Redis 명령이다. 따라서 이론적으로는 다음 문제가 발생할 수 있다.

```text
- 가격 인덱스는 수정 전 상태를 읽는다.
- 그 직후 Writer가 Redis 갱신을 완료한다.
- 면적 인덱스는 수정 후 상태를 읽는다.
```

가격 인덱스와 면적 인덱스가 서로 다른 시점의 상태로 섞일 수 있다.

이를 줄이기 위해 Reader의 가격 ZSet 조회와 면적 ZSet 조회를 `MULTI/EXEC`로 묶는 것은 타당하다.

### 8.2 보장 범위

Reader 인덱스 MULTI/EXEC가 보장하는 것은 다음이다.

```text
- 가격 ZSet 조회와 면적 ZSet 조회가 Redis 서버에서 하나의 isolated operation으로 실행된다.
- 두 인덱스 조회 사이에 다른 클라이언트의 Redis 명령이 끼어들지 않는다.
```

이것은 Redis 공식 문서에서 설명하는 transaction의 보장과 일치한다. Redis transaction의 명령들은 serialize되어 순차 실행되며, 다른 client request가 transaction 실행 중간에 처리되지 않는다.

### 8.3 보장하지 않는 것

Reader 인덱스 MULTI/EXEC가 보장하지 않는 것은 다음이다.

```text
- 사용자 요청 전체의 스냅샷
- ZSet 후보 조회와 Hash 상세 조회 사이의 동일 시점
- Hash 상세 조회 이후 점수 계산과 응답 생성까지의 전체 직렬화
```

따라서 Reader 인덱스 MULTI/EXEC는 “인덱스 간 시점 불일치 축소”를 위한 보조 장치다. 이것만으로 F009 금지 상태를 완전히 막을 수는 없다.

## 9. Pipeline의 의미와 한계

### 9.1 Pipeline은 성능 최적화다

`getMultipleCharterPropertiesFromRedis(propertyIds)`가 내부적으로 pipeline을 사용한다면, 이것은 여러 Redis Hash 조회 명령을 한 번에 보내 네트워크 round-trip 비용을 줄이기 위한 방식이다.

예를 들어 다음 명령들을 개별 왕복하지 않고 한 번에 보내는 것이다.

```text
HGETALL property:charter:A
HGETALL property:charter:B
HGETALL property:charter:C
```

Redis 공식 문서 기준으로 pipelining은 여러 명령을 각 응답을 기다리지 않고 발행하여 round-trip cost를 줄이는 기술이다.

### 9.2 Pipeline은 MULTI/EXEC가 아니다

Pipeline은 다음을 보장하지 않는다.

```text
HGETALL A, HGETALL B, HGETALL C 전체가 하나의 원자적 블록으로 실행된다.
```

Redis의 개별 명령 하나는 원자적으로 실행되지만, pipeline에 들어간 여러 명령 전체가 다른 client 명령과 interleaving되지 않는다고 일반화하면 안 된다.

따라서 다음 상태는 가능하다.

```text
1. Reader가 후보 ID A, B, C를 확보한다.
2. Reader가 pipeline으로 Hash 상세 조회를 보낸다.
3. 그 전후 또는 중간에 Writer의 EXEC가 처리된다.
4. Reader가 일부 매물은 수정 전 Hash, 일부 매물은 수정 후 Hash 기준으로 읽을 수 있다.
```

Pipeline은 정합성 장치가 아니라 성능 장치다.

### 9.3 Hash 상세 조회 묶음에 MULTI/EXEC를 적용할 수 있는가

가능하다.

후보 ID가 이미 확정된 뒤라면, 그 ID들에 대한 `HGETALL` 또는 `HMGET`을 `MULTI/EXEC` 안에 넣어 Hash 상세 조회 묶음 자체의 interleaving을 줄일 수 있다.

그러나 이것도 전체 조회 스냅샷을 보장하지는 않는다. 이유는 다음과 같다.

```text
1. ZSet 조회 묶음
2. Java에서 교집합 계산
3. Hash 조회 묶음
```

2번 Java 교집합 계산 때문에 1~3 전체를 단일 Redis `MULTI/EXEC`로 묶기 어렵다. MULTI 안의 명령 결과는 EXEC 이후에야 애플리케이션으로 돌아오므로, 앞선 ZSet 결과를 이용해 동적으로 Hash key 목록을 구성할 수 없다.

따라서 기본 설계에서 Hash 조회 묶음의 원자성보다 더 중요한 것은 Hash 실측값 기준 hard condition 검증이다.

## 10. S-04의 상세 조회를 S-01 검색 단계로 앞당기는 이유

### 10.1 변경 전 책임 분리의 문제

기존 구조에서는 다음과 같이 책임이 나뉘어 있다.

```text
S-01:
- Redis 인덱스로 후보 ID 조회

S-02:
- 후보 ID 개수로 fallback 판단

S-04:
- Hash 상세 조회
- 점수 계산
```

이 구조는 fallback 판단을 실제 유효 매물 기준이 아니라 인덱스 후보 기준으로 수행한다는 문제가 있다.

### 10.2 변경 후 책임 재배치

변경 후에는 `findValidCharterPropertiesInDistrict()`가 다음 책임을 가져야 한다.

```text
1. 가격 ZSet 조회
2. 면적 ZSet 조회
3. 인덱스 교집합 계산
4. 교집합 ID에 대한 Hash 상세 조회
5. Hash 실측값 기준 hard condition 검증
6. 검증 통과 PropertyDetail 목록 반환
```

즉, 이 메서드는 더 이상 `List<String>` ID 후보를 반환하지 않고, `List<PropertyDetail>` 유효 매물 상세를 반환하는 방향이 맞다.

### 10.3 변경 후 전체 흐름

변경 후 Reader 흐름은 다음과 같아야 한다.

```text
getCharterDistrictRecommendations()
  → S-01 performCharterStrictSearch()
       → findValidCharterPropertiesInDistrict()
          - 가격/면적 ZSet 조회
          - 교집합
          - Hash 상세 조회
          - hard condition 검증
          - List<PropertyDetail> 반환
  → S-02 checkAndPerformCharterFallback()
       - 검증 통과 PropertyDetail 개수 기준으로 fallback 판단
  → S-04 calculateCharterPropertyScores()
       - 이미 확보된 PropertyDetail 사용
       - Redis Hash를 다시 읽지 않음
       - ReviewStatistics 조회용 propertyId만 추출
       - 점수 계산
  → S-05 지역구 점수 계산 및 정렬
  → S-06 최종 응답 생성
```

### 10.4 타입 의미 변경

기존 타입 구조는 다음과 같다.

```java
Map<String, List<String>> districtProperties
```

이 타입은 “지역구별 매물 ID 목록”을 의미한다.

변경 후에는 다음 구조가 되어야 한다.

```java
Map<String, List<PropertyDetail>> districtProperties
```

이 타입은 “지역구별 hard condition 검증을 통과한 실제 추천 가능 매물 목록”을 의미한다.

타입 변경은 단순 리팩터링이 아니다. 이것은 `districtProperties`라는 데이터가 코드 안에서 가지는 의미를 바꾸는 작업이다.

## 11. Fallback 정책 재정의

### 11.1 기존 fallback의 문제

기존 fallback은 다음 기준으로 실행된다.

```text
각 지역구의 propertyId 목록 size가 MIN_PROPERTIES_THRESHOLD보다 작은가?
```

그러나 이 목록은 Redis 인덱스 기준 후보 ID일 뿐이다. Hash 상세 검증 전이므로 실제 유효 매물 개수와 다를 수 있다.

따라서 기존 fallback은 다음 오판을 할 수 있다.

```text
- ZSet 후보 ID는 3개다.
- S-02는 fallback이 필요 없다고 판단한다.
- S-04에서 Hash 상세를 읽어보니 1개는 budgetMax를 초과한다.
- 실제 유효 매물은 2개뿐이다.
```

### 11.2 변경 후 fallback 기준

변경 후 fallback은 다음 기준으로 판단해야 한다.

```text
Hash 상세 조회와 hard condition 검증까지 통과한 PropertyDetail 개수가 MIN_PROPERTIES_THRESHOLD보다 작은가?
```

즉, fallback의 입력은 `List<String>` 후보 ID가 아니라 `List<PropertyDetail>` 유효 매물이어야 한다.

### 11.3 Fallback은 무한 재시도가 아니다

본 논의에서 중요한 수정점은 “불일치가 발생하면 인덱스 조회부터 재시도한다”를 기본 설계로 두지 않는다는 것이다.

무한 재시도는 다음 문제를 만든다.

```text
- 조회 트래픽이 많을 때 재시도 자체가 추가 부하가 된다.
- 수정 요청이 계속 발생하면 조회 재시도도 계속 실패할 수 있다.
- 사용자 응답시간이 무한히 늘어날 수 있다.
- 시스템이 경합을 해결하는 것이 아니라 경합 비용을 증폭시킬 수 있다.
```

따라서 fallback은 “정합성 재시도”가 아니라 “추천 결과 부족 시 조건 완화”로 이해해야 한다.

### 11.4 Fallback의 올바른 위치

Hard condition 검증에서 탈락한 매물은 fallback 판단 전부터 제거되어야 한다.

그 후 실제 유효 매물 수가 부족하면 기존의 조건 완화 흐름을 수행한다.

```text
1. strict search 실행
2. Hash 상세 기준 hard condition 검증
3. 검증 통과 매물 수 판단
4. 부족하면 fallback 수행
5. fallback 결과도 다시 Hash 상세 기준 hard condition 검증
6. 그래도 부족하면 SUCCESS_EXPANDED 또는 NO_RESULTS 정책에 따라 응답
```

## 12. 재시도, 락, 큐, 쓰기 중단 설계에 대한 판단

### 12.1 재시도 중심 설계는 기본안으로 부적절하다

조회 중 데이터 불일치가 감지되었을 때, 인덱스 조회부터 다시 수행하는 재시도는 기술적으로 가능하다. 그러나 이것을 기본 정합성 해결책으로 삼는 것은 적절하지 않다.

이유는 다음과 같다.

```text
- 재시도는 최신 스냅샷을 보장하지 않는다.
- 재시도하는 동안 또 다른 수정이 발생할 수 있다.
- 무한 재시도는 응답시간 지연을 만든다.
- 조회 요청 수가 많으면 재시도 자체가 Redis/RDB 부하를 키운다.
- 사용자는 내부 정합성 경합보다 결과 품질과 응답시간에 더 민감하다.
```

따라서 재시도는 Redis 연결 실패, 일시적 timeout 등 장애성 예외에 제한적으로 적용할 수는 있지만, 조회 정합성 보장 수단으로 사용하지 않는다.

### 12.2 조회 때문에 수정 API를 대기시키는 설계는 기본안으로 부적절하다

기술적으로는 조회와 수정을 서버측에서 직렬화할 수 있다. 예를 들어 지역구 단위 read/write lock, Redis 분산락, queue 기반 순차 처리 등을 설계할 수 있다.

그러나 추천 조회는 다수 사용자가 빈번하게 호출하는 read-heavy API다. 조회가 수정 API를 막거나, 수정이 조회 API 전체를 막는 구조는 응답 지연과 서비스 병목을 만들 수 있다.

특히 현재 추천 조회는 서울 25개 자치구를 순회하고, 인덱스 조회, 교집합, Hash 상세 조회, 리뷰 통계 조회, 점수 계산, 지역구 정렬까지 수행한다. 이 전체 구간을 쓰기와 직렬화하면 lock 범위가 과도하게 커진다.

따라서 현재 단계에서는 조회/수정 전역 직렬화나 수정 API 일시 중단을 기본 설계로 채택하지 않는다.

### 12.3 Queue 기반 순차 처리도 현재 기본안은 아니다

조회와 수정을 모두 queue로 받아 순차 처리하면 강한 순서 보장은 가능하다. 그러나 그 대가로 다음 문제가 생긴다.

```text
- 조회 처리량이 급격히 낮아질 수 있다.
- 하나의 느린 조회가 뒤따르는 수정/조회 요청을 지연시킬 수 있다.
- 추천 조회의 실시간 응답성이 떨어진다.
- 구현 복잡도가 현재 문제 범위보다 커진다.
```

따라서 queue 기반 직렬화는 이 서비스의 기본 모델과 맞지 않는다.

### 12.4 현재 설계의 기본 원칙

현재 단계의 기본 원칙은 다음이다.

```text
- Writer는 Redis 쓰기 중간 상태를 노출하지 않는다.
- Reader는 Redis 인덱스를 후보 축소용으로만 사용한다.
- Reader는 Hash 실측값 기준으로 hard condition을 검증한다.
- 검증 통과 매물만 fallback 판단과 점수 계산에 넘긴다.
- 무한 재시도, 전역 락, queue 직렬화는 기본 설계로 채택하지 않는다.
```

## 13. Redis Lua Script / Redis Function 확장안

### 13.1 왜 확장안으로 검토했는가

Java 애플리케이션에서 다음 단계를 분리해 실행하면 간극이 남는다.

```text
1. Redis 가격 인덱스 조회
2. Redis 면적 인덱스 조회
3. Java 교집합 계산
4. Redis Hash 상세 조회
5. Java hard condition 검증
```

Reader 인덱스 조회를 MULTI/EXEC로 묶어도, ZSet 후보 조회와 Hash 상세 조회 사이의 간극은 남는다. Pipeline은 성능 최적화일 뿐 원자성 보장 수단이 아니다.

따라서 “인덱스 조회, 교집합, Hash 상세 조회, hard condition 검증을 Redis 서버 내부에서 하나의 실행 단위로 강제할 방법이 있는가”를 검토하였다.

그 방법이 Redis Lua Script 또는 Redis Function이다.

### 13.2 Redis Lua/Function이 제공하는 것

Redis Lua Script 또는 Redis Function을 사용하면 다음 작업을 Redis 서버 내부의 단일 atomic execution으로 묶을 수 있다.

```text
- 가격 ZSet 조회
- 면적 ZSet 조회
- 인덱스 교집합 계산
- 교집합 ID에 대한 Hash 상세 조회
- Hash 실측값 기준 hard condition 검증
- 검증 통과 ID 또는 상세 데이터 반환
```

Redis 공식 문서 기준으로 Lua script/function은 실행 동안 원자적으로 수행되고, 실행 중 Redis 서버 활동을 block한다. 따라서 이 방식은 인덱스 후보 조회와 Hash 상세 조회 사이에 Writer Redis 명령이 끼어드는 문제를 Redis 서버 내부에서는 제거할 수 있다.

### 13.3 개념 예시

개념적으로는 다음과 같은 script 흐름이 가능하다.

```lua
-- KEYS[1] = idx:charterPrice:{district}
-- KEYS[2] = idx:area:{district}:전세
-- ARGV[1] = budgetMin
-- ARGV[2] = budgetMax
-- ARGV[3] = areaMin
-- ARGV[4] = areaMax

local priceIds = redis.call('ZRANGEBYSCORE', KEYS[1], ARGV[1], ARGV[2])
local areaIds = redis.call('ZRANGEBYSCORE', KEYS[2], ARGV[3], ARGV[4])

local areaSet = {}
for _, id in ipairs(areaIds) do
  areaSet[id] = true
end

local result = {}

for _, id in ipairs(priceIds) do
  if areaSet[id] then
    local hashKey = 'property:charter:' .. id

    local deposit = tonumber(redis.call('HGET', hashKey, 'deposit'))
    local area = tonumber(redis.call('HGET', hashKey, 'areaInPyeong'))
    local leaseType = redis.call('HGET', hashKey, 'leaseType')
    local status = redis.call('HGET', hashKey, 'status')

    if deposit ~= nil
       and area ~= nil
       and deposit >= tonumber(ARGV[1])
       and deposit <= tonumber(ARGV[2])
       and area >= tonumber(ARGV[3])
       and area <= tonumber(ARGV[4])
       and leaseType == '전세'
       and status == 'ACTIVE'
    then
      table.insert(result, id)
    end
  end
end

return result
```

이 script는 예시일 뿐이며, 실제 적용 시에는 반환 형태, field 직렬화 방식, 후보 수 제한, timeout 위험, script 배포 방식을 별도로 설계해야 한다.

### 13.4 장점

Redis Lua/Function 확장안의 장점은 다음과 같다.

```text
- 인덱스 조회와 Hash 상세 조회 사이의 Redis-level interleaving을 제거할 수 있다.
- Java 애플리케이션과 Redis 사이의 왕복 횟수를 줄일 수 있다.
- hard condition 검증을 Redis 데이터가 존재하는 위치에서 수행할 수 있다.
- Writer MULTI/EXEC와 Reader Lua/Function이 Redis 서버에서 서로 중간에 섞이지 않는 실행 단위가 된다.
```

### 13.5 단점과 주의점

Lua/Function은 실행 동안 Redis 서버를 점유한다. 따라서 script가 무거우면 다른 Redis 요청이 지연될 수 있다.

현재 추천 조회는 25개 자치구를 순회할 수 있고, 각 자치구별 후보 수가 많아질 수 있다. 전체 추천 API를 하나의 Lua script로 묶으면 Redis 응답 지연이 커질 수 있다.

따라서 이 확장안을 적용한다면 다음처럼 범위를 제한해야 한다.

```text
권장 적용 범위:
- 자치구 단위
- 후보 수 제한 단위
- hard condition 검증 중심

비권장 적용 범위:
- 25개 자치구 전체를 하나의 script로 처리
- 리뷰 통계 조회 또는 점수 계산까지 Redis script에 포함
- 오래 걸리는 정렬/계산 로직을 Redis script에 포함
```

### 13.6 현재 단계에서의 채택 여부

현재 단계에서는 Redis Lua Script / Redis Function을 기본 구현으로 채택하지 않는다.

기본 개선안은 Java 애플리케이션 레벨에서 다음 구조를 갖는 것이다.

```text
1. Writer MULTI/EXEC 유지
2. Reader 인덱스 조회 MULTI/EXEC 적용
3. 교집합 직후 Hash 상세 조회
4. Hash 실측값 기준 hard condition 검증
5. 검증 통과 PropertyDetail 기준 fallback 및 점수 계산
```

Lua/Function은 더 강한 Redis-level 정합성이 필요할 때 적용할 수 있는 확장안으로 문서화한다.

## 14. 최종 개선 설계 방향

### 14.1 Writer

Writer 쪽 기본 방향은 다음과 같다.

```text
- RDB 수정은 기존 트랜잭션 정책에 맞게 처리한다.
- Redis 동기화는 RDB 커밋 이후 실행되어야 한다.
- Redis Hash와 관련 ZSet 인덱스 갱신은 MULTI/EXEC로 묶는다.
- Hash와 ZSet 사이의 중간 상태를 외부 Reader에게 노출하지 않는다.
```

F009에서 확인된 Writer MULTI/EXEC는 유지한다. 이것은 1차 경합 원인인 개별 Redis 커맨드 간 중간 상태 노출을 제거한다.

### 14.2 Reader — 검색 단계

Reader의 검색 단계는 다음 책임을 가져야 한다.

```text
- Redis 가격 인덱스 조회
- Redis 면적 인덱스 조회
- 두 인덱스 결과의 교집합
- 교집합 ID에 대한 Hash 상세 조회
- hard condition 검증
- 검증 통과 PropertyDetail 반환
```

가격 인덱스와 면적 인덱스 조회는 MULTI/EXEC로 묶어 인덱스 간 시점 불일치를 줄인다.

### 14.3 Reader — fallback 단계

Fallback 판단 기준은 ID 후보 개수가 아니라 hard condition 검증 통과 매물 개수다.

```text
변경 전:
- List<String> propertyIds.size() 기준 fallback 판단

변경 후:
- List<PropertyDetail> validatedProperties.size() 기준 fallback 판단
```

### 14.4 Reader — 점수 계산 단계

S-04는 더 이상 Redis Hash 상세를 다시 읽지 않는다.

변경 후 S-04는 다음만 수행한다.

```text
- 검증 통과 PropertyDetail 목록을 입력으로 받는다.
- ReviewStatistics 조회를 위해 propertyId만 추출한다.
- RDB chunk 조회를 수행한다.
- 기존 가격/공간/안전/리뷰 점수 계산 로직을 적용한다.
```

### 14.5 응답 생성

응답에 포함되는 모든 매물은 Hash 실측값 기준 hard condition을 통과한 매물이어야 한다.

응답 메시지는 내부 동시성 경합을 사용자에게 노출하지 않는다. 조건에 맞는 매물이 부족하면 기존 fallback 또는 조건 완화/NO_RESULTS 메시지로 처리한다.

### 15.1 핵심 타입 변경

private Map<String, List<String>> performCharterStrictSearch(...)

### 15.2 `SearchResult` 의미 변경

기존 `SearchResult.districtProperties`가 `Map<String, List<String>>`라면, 변경 후에는 `Map<String, List<PropertyDetail>>`로 바뀌어야 한다.

이 변경은 단순히 타입만 바꾸는 것이 아니다. `SearchResult`가 의미하는 검색 결과가 “후보 ID”에서 “검증된 실제 매물”로 바뀐다.

### 15.3 `findValidCharterPropertiesInDistrict()`의 변경 후 책임

변경 후 이 메서드의 책임은 다음과 같다.

```text
입력:
- district
- request

처리:
- 가격/면적 Redis 인덱스 조회
- 교집합 계산
- 교집합 ID에 대한 Redis Hash 상세 조회
- hard condition 검증

출력:
- List<PropertyDetail>
- 이 목록은 응답 또는 점수 계산으로 넘겨도 되는 실제 유효 매물 목록이다.
```

### 15.4 hard condition 검증 메서드

다음과 같은 검증 메서드가 필요하다.

```java
private boolean matchesHardCondition(
        PropertyDetail detail,
        CharterRecommendationRequestDto request
) {
    if (detail == null) return false;
    if (detail.getDeposit() == null) return false;
    if (detail.getAreaInPyeong() == null) return false;

    if (!"전세".equals(detail.getLeaseType())) return false;
    if (!"ACTIVE".equals(detail.getStatus())) return false;

    return detail.getDeposit() >= request.getBudgetMin()
            && detail.getDeposit() <= request.getBudgetMax()
            && detail.getAreaInPyeong() >= request.getAreaMin()
            && detail.getAreaInPyeong() <= request.getAreaMax();
}
```

실제 타입이 `Long`, `Integer`, `BigDecimal`, `Double` 중 무엇인지에 따라 비교 로직은 구현 시 조정해야 한다.

## 16. 최종 설계 원칙

F009 개선의 최종 설계 원칙은 다음과 같다.

```text
1. Redis ZSet 인덱스는 최종 정합성 판정 기준이 아니라 후보 축소용 자료구조다.

2. Redis Hash 상세값은 응답 직전 hard condition 검증의 기준이다.

3. Writer는 Redis Hash와 ZSet 인덱스의 쓰기 중간 상태를 노출하지 않기 위해 MULTI/EXEC를 사용한다.

4. Reader는 가격/면적 인덱스 조회를 MULTI/EXEC로 묶어 인덱스 간 시점 불일치를 줄인다.

5. Reader는 인덱스 교집합 직후 Hash 상세를 조회하고, hard condition을 검증한다.

6. Fallback 판단은 검증 통과 PropertyDetail 개수를 기준으로 수행한다.

7. S-04 점수 계산은 검증된 PropertyDetail을 입력으로 받아야 하며, Redis Hash를 다시 읽어서는 안 된다.

8. 무한 재시도, 조회/수정 전역 락, queue 기반 직렬화, 수정 API 일시 중단은 현재 기본 설계로 채택하지 않는다.

9. 더 강한 Redis-level 정합성이 필요할 경우 Redis Lua Script 또는 Redis Function을 확장안으로 검토한다.

10. 추천 조회 API는 요청 시작 시점의 완전한 스냅샷을 보장하지 않는다. 대신 응답에 포함되는 모든 매물이 응답 생성 과정에서 확인한 실측값 기준으로 사용자의 hard condition을 만족하도록 보장한다.
```

## 17. 포트폴리오 관점에서의 설명 포인트

면접 또는 포트폴리오 설명에서는 다음 흐름으로 설명하는 것이 좋다.

### 17.1 단순 Redis 동시성 버그가 아니었다

처음에는 Writer가 Redis Hash와 ZSet을 개별 커맨드로 갱신했기 때문에 중간 상태가 노출되었다. 이를 재현하기 위해 CountDownLatch 기반 테스트 훅을 삽입하고 JMeter로 Writer/Reader 경합을 재현하였다.

그 결과 `budgetMax=60,000` 조건에서 `price=80,000` 매물이 응답되는 금지 상태를 확인하였다.

### 17.2 Writer MULTI/EXEC는 필요하지만 충분하지 않았다

Writer에 MULTI/EXEC를 적용하여 Hash와 ZSet의 쓰기 중간 상태는 제거하였다.

그러나 재테스트 결과, Reader가 ZSet으로 후보 ID를 확정한 뒤 S-04에서 뒤늦게 Hash 상세를 읽는 2단계 구조 때문에 Writer EXEC 이후의 Hash 신값을 읽어 금지 상태가 다시 발생하였다.

따라서 문제는 Writer 원자성만이 아니라 Reader의 검색 단계와 상세 조회 단계 간 책임 분리 문제로 확장되었다.

### 17.3 비즈니스 정합성 기준을 먼저 정의했다

모든 조회 요청에 대해 완전한 스냅샷을 보장하려면 조회와 수정을 직렬화해야 한다. 하지만 추천 조회 서비스에서 그 수준의 정합성은 과도하며, 응답시간과 처리량을 해칠 수 있다.

따라서 다음 기준을 세웠다.

```text
허용:
- 조회 중 수정된 매물이 이번 응답에서 빠지는 것
- 새로 등록된 매물이 다음 조회부터 반영되는 것
- 경합으로 추천 매물 수가 줄어드는 것

금지:
- 응답에 포함된 매물이 사용자의 예산/면적/상태/전세 조건을 위반하는 것
```

### 17.4 인덱스는 후보 축소, Hash는 최종 검증 기준으로 재정의했다

Redis ZSet 인덱스는 빠른 후보 축소용이다. 최종 정합성 기준으로 사용하면 안 된다.

따라서 인덱스 교집합 직후 Hash 상세를 조회하고, Hash 실측값 기준으로 hard condition을 검증하도록 Reader 책임을 재배치하였다.

### 17.5 재시도보다 검증과 책임 재배치를 선택했다

불일치가 발생할 때마다 인덱스 조회부터 재시도하는 방식은 무한 재시도와 응답 지연을 만들 수 있다.

따라서 정합성 해결책은 재시도가 아니라 다음 구조로 설계하였다.

```text
인덱스 후보 조회
→ Hash 상세 조회
→ hard condition 검증
→ 검증 통과 매물만 fallback/점수 계산에 사용
```

### 17.6 확장안으로 Redis Lua/Function을 검토했다

더 강한 Redis-level 정합성이 필요하면 Redis Lua Script 또는 Redis Function으로 인덱스 조회, 교집합, Hash 상세 조회, hard condition 검증을 Redis 서버 내부의 단일 atomic execution으로 묶을 수 있다.

하지만 현재 단계에서는 Redis 서버 block 위험과 구현 복잡도를 고려하여 기본안으로 채택하지 않고, 확장안으로 문서화한다.

## 19. 최종 결론

F009의 본질은 Redis 명령 몇 개를 원자화하는 문제가 아니라, 추천 조회 API가 어떤 정합성 계약을 보장해야 하는지 정의하는 문제다.

초기 문제는 Writer가 Redis Hash와 ZSet을 개별 커맨드로 갱신하여 중간 상태를 노출한 것이었다. Writer MULTI/EXEC 적용으로 이 문제는 해결되었다.

그러나 Writer의 쓰기 원자성 확보 후에도 Reader의 2단계 읽기 구조 때문에 금지 상태가 남았다. Reader는 ZSet 인덱스로 후보 ID를 확정한 뒤, 시간이 지난 S-04 단계에서 Hash 상세를 읽는다. 이 사이에 Writer EXEC가 완료되면, Reader는 ZSet 기준으로는 조건 내 후보였던 매물을 Hash 기준으로는 조건 밖 상세값으로 읽을 수 있다.

따라서 개선의 핵심은 다음이다.

```text
S-04에서 뒤늦게 수행하던 Hash 상세 조회를 S-01/S-03 검색 단계의 findValidCharterPropertiesInDistrict 내부로 이동한다.

findValidCharterPropertiesInDistrict는 더 이상 후보 ID를 반환하지 않고, Hash 실측값 기준 hard condition 검증을 통과한 PropertyDetail 목록을 반환한다.

S-02 fallback 판단과 S-04 점수 계산은 이 검증 통과 PropertyDetail 목록을 기준으로 수행한다.
```

이 설계는 모든 시간 간극을 제거하지 않는다. 대신 추천 조회 서비스에서 비즈니스적으로 필요한 정합성 계약을 보장한다.

```text
추천 조회 API는 요청 시작 시점의 완전한 스냅샷을 보장하지 않는다.
대신 응답에 포함되는 모든 매물은 응답 생성 과정에서 확인한 Redis Hash 실측값 기준으로 사용자의 hard condition을 만족해야 한다.
```

이것이 F009의 최종 설계 기준이다.

### 20.2 외부 기술 근거

```text
- Redis Transactions 공식 문서
  - MULTI/EXEC는 transaction 내부 명령을 serialize하여 sequential하게 실행한다.
  - 다른 client request는 transaction 실행 중간에 처리되지 않는다.
  - 명령은 EXEC 호출 전까지 QUEUED 상태가 된다.

- Redis Pipelining 공식 문서
  - pipelining은 여러 명령을 각 응답을 기다리지 않고 보내 round-trip cost를 줄이는 기술이다.
  - 목적은 성능 최적화다.
  - pipeline 자체는 MULTI/EXEC와 같은 원자성 보장 장치가 아니다.

- Redis Lua Scripting / Redis Programmability 공식 문서
  - Lua script/function은 Redis 서버 내부에서 atomic execution을 보장한다.
  - 실행 동안 Redis 서버 활동을 block한다.
  - 따라서 무거운 script는 전체 Redis latency에 영향을 줄 수 있다.
```
