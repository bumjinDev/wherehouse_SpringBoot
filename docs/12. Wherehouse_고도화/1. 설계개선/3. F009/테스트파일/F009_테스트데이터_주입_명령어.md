# F009 테스트 데이터 주입 명령어 모음

## 주의 사항

**CHAR(32) 패딩 이슈**

`PROPERTIES_CHARTER.PROPERTY_ID`는 엔티티 정의상 `CHAR(32)`이다.
Oracle CHAR는 고정 길이이므로 `F009_TARGET_A0000000000000000000`(13자)를 삽입하면 19자의 공백이 뒤에 패딩된다.

Hibernate가 이 값을 읽을 때 패딩된 상태로 반환되면,
Writer가 `f009RaceLatch.syncAwait("F009_TARGET_A0000000000000000000                   ")`를 호출하고
latch의 `TARGET_PROPERTY_ID = "F009_TARGET_A0000000000000000000"`와 불일치하여 훅이 동작하지 않을 수 있다.

이 경우 다음 중 하나로 대응한다.

1. `F009RaceLatch.syncAwait()`에서 `propertyId.trim().equals(targetPropertyId)`로 비교
2. 테스트 propertyId를 정확히 32자로 맞춤 (예: `F009TARGETA000000000000000000000`)
3. DDL이 실제로 `VARCHAR2`인 경우 이 이슈는 발생하지 않음

테스트 실행 전 다음 쿼리로 실제 컬럼 타입을 확인한다.

```sql
SELECT COLUMN_NAME, DATA_TYPE, DATA_LENGTH
FROM USER_TAB_COLUMNS
WHERE TABLE_NAME = 'PROPERTIES_CHARTER' AND COLUMN_NAME = 'PROPERTY_ID';
```

`DATA_TYPE = CHAR`이면 위 대응이 필요하고, `VARCHAR2`이면 불필요하다.

## 6. Redis — 데이터 주입 방식

**중요: Spring RedisTemplate이 JSON serializer를 사용하여 모든 값에 따옴표(`"`)를 포함하여 저장한다.**

따라서 Redis 데이터 주입은 두 가지 방식 중 하나를 선택한다.

### 방식 A: 애플리케이션 자동 동기화 (권장)

`RdbSyncListener`의 `@Scheduled` 주석 처리를 해제하면, 애플리케이션 부팅 시 RDB → Redis 자동 동기화가 실행된다.
이 경우 **SQL 데이터만 삽입하면 Redis는 자동으로 채워진다.** 수동 Redis 주입이 불필요하다.

1. 섹션 1~3의 SQL 삽입 실행
2. 애플리케이션을 `f009-test` 프로파일로 기동
3. 부팅 후 1초 뒤 `RdbSyncListener`가 RDB → Redis 동기화 수행

### 방식 B: 수동 Redis 주입 (자동 동기화 비활성 시)

`RdbSyncListener`의 `@Scheduled`을 주석 처리한 상태에서는 Redis Insight CLI에서 수동 주입한다.
**모든 값에 JSON 직렬화 따옴표를 포함해야 한다.**

## 13. Redis — 테스트 중 상태 관찰 명령어

### 13-1. 수정 API 호출 전 (초기 상태)

```
HGET property:charter:F009_TARGET_A0000000000000000000 deposit
ZSCORE idx:charterPrice:강남구 "\"F009_TARGET_A0000000000000000000\""
```

기대: `deposit="50000"`, `ZSCORE=50000`

### 13-2. Writer가 syncAwait에서 멈춘 상태 (핵심 중간 상태)

수정 API 호출 후, 추천 조회 API 호출 전에 확인한다.

```
HGET property:charter:F009_TARGET_A0000000000000000000 deposit
ZSCORE idx:charterPrice:강남구 "\"F009_TARGET_A0000000000000000000\""
```

기대: `deposit="80000"` (Hash 신값), `ZSCORE=50000` (ZSet 구값)

이 불일치가 F009 경합의 핵심이다.

### 13-3. Writer 해제 후 (최종 상태)

추천 조회 API가 Writer를 해제한 후 확인한다.

```
HGET property:charter:F009_TARGET_A0000000000000000000 deposit
ZSCORE idx:charterPrice:강남구 "\"F009_TARGET_A0000000000000000000\""
```

기대: `deposit="80000"`, `ZSCORE=80000` (둘 다 신값)

### 13-4. 최종 정상 상태 — 가격 ZSet 범위 조회

```
ZRANGEBYSCORE idx:charterPrice:강남구 40000 60000 WITHSCORES
```

기대: `F009_TARGET_A0000000000000000000`는 더 이상 포함되지 않음 (score=80000 > budgetMax=60000).
`F009_DUMMY_B00000000000000000000`, `F009_DUMMY_C00000000000000000000`만 포함.

## 14. Redis — 테스트 반복 시 초기화

테스트를 다시 수행하려면 RDB와 Redis를 모두 초기 상태로 되돌려야 한다.

### RDB 초기화

```sql
UPDATE PROPERTIES_CHARTER SET DEPOSIT = 50000, MODIFIED_AT = NULL
WHERE PROPERTY_ID = 'F009_TARGET_A0000000000000000000';
COMMIT;
```

### Redis 초기화 — 방식 A (애플리케이션 재시작)

가장 간단한 방법. `f009-test` 프로파일로 애플리케이션 재시작 시 RDB → Redis 자동 동기화 + F009RaceLatch 재초기화.

### Redis 초기화 — 방식 B (수동)

```
HMSET property:charter:F009_TARGET_A0000000000000000000 deposit "\"50000\"" modifiedAt "\"\""
ZADD idx:charterPrice:강남구 50000 "\"F009_TARGET_A0000000000000000000\""
```

F009RaceLatch는 1회성(CountDownLatch)이므로 애플리케이션 재시작이 필요하다.
