## 1. Claude에게 전달하는 작업 목적

이번 작업의 목적은 F009 Write-Read 경합에서 확인된 전세 추천 조회 API의 응답 자기모순을 제거하는 것이다.

기존 F009 테스트에서 사용자는 `budgetMax=60000`으로 전세 추천 조회를 요청했으나, 응답에는 `price=80000`인 매물이 포함되었다. 이 상태는 비즈니스적으로 금지 상태다.

Writer 쪽 Redis 갱신에 `MULTI/EXEC`를 적용하여 Hash와 ZSet 갱신 사이의 중간 상태 노출은 줄였으나, Reader 쪽에서 다음 구조가 남아 있어 문제가 계속 발생한다.

```text
S-01: Redis ZSet 인덱스로 후보 propertyId 조회
S-02: 후보 propertyId 개수 기준 fallback 판단
S-04: 후보 propertyId로 Redis Hash 상세 조회
S-04 이후: 점수 계산 및 응답 생성
```

이 구조에서는 S-01에서 뽑은 `propertyId`가 S-04에서 상세 조회될 때 이미 수정되어 있을 수 있다. 즉, `idx:charterPrice` 기준으로는 조건에 맞는 후보였지만, Hash 상세 기준으로는 `budgetMax`를 초과하는 매물이 될 수 있다.

이번 수정의 목표는 “조회 요청 시작 시점의 완전한 snapshot”을 보장하는 것이 아니다. 추천 조회 서비스의 비즈니스 정합성 계약을 다음처럼 정의하고, 그 계약을 코드에 반영하는 것이다.

```text
응답에 포함되는 모든 전세 매물은 응답 생성 과정에서 Redis Hash로 확인한 실측값 기준으로 사용자의 hard condition을 만족해야 한다.
```

## 2. 반드시 지켜야 할 설계 원칙

### 2.1 Redis ZSet 인덱스는 후보 축소용이다

`idx:charterPrice:{district}`와 `idx:area:{district}:전세`는 최종 정합성 판정 기준이 아니다.

이 인덱스들은 후보 ID를 빠르게 줄이기 위한 read model이다. 최종적으로 응답에 포함해도 되는지는 Redis Hash 상세값 기준으로 다시 판단해야 한다.

### 2.2 Redis Hash 상세값이 최종 응답 검증 기준이다

응답에 포함되는 `price`, `area`, `leaseType`, `status`는 Redis Hash에서 읽은 `PropertyDetail` 기준이다.

따라서 Redis Hash 상세 조회 후 다음 조건을 반드시 검증해야 한다.

```text
- leaseType = 전세
- status = ACTIVE
- deposit >= request.budgetMin
- deposit <= request.budgetMax
- areaInPyeong >= request.areaMin
- areaInPyeong <= request.areaMax
```

위 조건을 만족하지 않는 매물은 응답, fallback 판단, 점수 계산 대상에서 제외한다.

### 2.3 Reader 인덱스 조회는 MULTI/EXEC로 묶는다

가격 ZSet 조회와 면적 ZSet 조회는 서로 다른 Redis 명령이다.

따라서 두 인덱스 조회 사이에 Writer의 Redis 갱신이 끼어들지 않도록, Reader의 가격/면적 인덱스 조회를 `MULTI/EXEC`로 묶는다.

단, 이것은 전체 조회 snapshot을 보장하는 것이 아니다. 보장 범위는 다음으로 제한된다.

```text
보장:
- 가격 ZSet 조회와 면적 ZSet 조회 사이에 다른 Redis 명령이 끼어들지 않는다.

보장하지 않음:
- 사용자 HTTP 요청 시작 시점의 snapshot
- 인덱스 조회와 Hash 상세 조회 사이의 완전한 동일 시점
- 응답 생성 완료까지의 전체 직렬화
```

### 2.4 Pipeline은 원자성 보장 수단이 아니다

`getMultipleCharterPropertiesFromRedis(propertyIds)`는 현재 pipeline으로 여러 Hash를 조회한다.

Pipeline은 네트워크 왕복 비용을 줄이는 성능 최적화다. `MULTI/EXEC`처럼 여러 명령 전체를 하나의 isolated operation으로 보장하지 않는다.

따라서 pipeline을 유지하더라도 Hash 상세 조회 후 hard condition 검증은 반드시 필요하다.

### 2.5 무한 재시도는 금지한다

정합성 불일치를 발견했을 때 인덱스 조회부터 무한 반복하는 구조를 만들지 않는다.

재시도는 최신 snapshot을 보장하지 못하고, 조회 트래픽이 많을 때 Redis/RDB 부하와 사용자 응답 지연을 증폭시킬 수 있다.

이번 F009 개선에서 fallback은 “정합성 재시도”가 아니라 “유효 매물 부족 시 조건 완화”로 취급한다.

### 2.6 조회 때문에 수정 API를 대기시키지 않는다

조회와 수정을 전역 lock, read/write lock, queue로 직렬화하지 않는다.

추천 조회는 read-heavy API다. 조회 트래픽 때문에 수정 API가 대기하거나, 수정 때문에 조회 API 전체가 대기하는 구조는 현재 문제 범위보다 과한 설계다.

## 3. 현재 코드에서 문제되는 지점

### 3.1 현재 메인 흐름

현재 `CharterRecommendationService.getCharterDistrictRecommendations()`는 다음 흐름이다.

```java
Map<String, List<String>> districtProperties = performCharterStrictSearch(request, SEOUL_DISTRICTS);

SearchResult searchResult = checkAndPerformCharterFallback(districtProperties, request);

Map<String, List<PropertyWithScore>> districtPropertiesWithScores =
        calculateCharterPropertyScores(searchResult.getDistrictProperties(), request);

List<DistrictWithScore> sortedDistricts = calculateDistrictScoresAndSort(districtPropertiesWithScores);

CharterRecommendationResponseDto finalResponse =
        generateCharterFinalResponse(sortedDistricts, searchResult, request, currentUserId);
```

문제는 `districtProperties`의 의미가 “검증된 매물”이 아니라 “인덱스 기준 후보 ID 목록”이라는 점이다.

### 3.2 현재 S-01의 문제

현재 `performCharterStrictSearch()`는 다음 타입을 반환한다.

```java
private Map<String, List<String>> performCharterStrictSearch(
        CharterRecommendationRequestDto request,
        List<String> targetDistricts)
```

그리고 내부에서 다음 메서드를 호출한다.

```java
List<String> validProperties = findValidCharterPropertiesInDistrict(district, request);
```

하지만 현재 `findValidCharterPropertiesInDistrict()`의 반환값은 실제 valid property가 아니라 ZSet 교집합을 통과한 propertyId 목록이다.

즉, 메서드명은 `valid`지만 실제 의미는 `candidatePropertyIds`에 가깝다.

### 3.3 현재 S-02 fallback의 문제

현재 fallback 판단은 다음과 같은 기준이다.

```java
boolean hasInsufficientDistricts = districtProperties.values().stream()
        .anyMatch(propertyList -> propertyList.size() < MIN_PROPERTIES_THRESHOLD);
```

이때 `propertyList.size()`는 실제 유효 매물 수가 아니라 ZSet 후보 ID 수다.

따라서 다음 오판이 가능하다.

```text
1. ZSet 기준 후보 ID는 3개다.
2. S-02는 fallback 불필요로 판단한다.
3. S-04에서 Hash 상세를 읽어보니 1개는 budgetMax를 초과한다.
4. 실제 유효 매물은 2개뿐이다.
5. 그럼에도 기존 fallback 판단은 이미 지나갔다.
```

### 3.4 현재 S-04의 문제

현재 `calculateCharterPropertyScores()`는 `Map<String, List<String>>`을 입력받고, 여기서 Redis Hash 상세를 조회한다.

```java
List<PropertyDetail> propertyDetails = getMultipleCharterPropertiesFromRedis(propertyIds);
```

이 시점은 fallback 판단 이후다. 따라서 Hash 상세 기준으로 실제 조건 위반 매물이 발견되어도 fallback 판단에는 반영되지 않는다.

또한 F009 경합에서는 S-01에서 후보 ID를 선정한 뒤 S-04에서 Hash를 읽기 전까지 Writer Redis 갱신이 완료될 수 있다. 그 결과 `budgetMax=60000` 요청에서 `deposit=80000` Hash를 읽어 응답에 포함할 수 있다.

## 4. 변경 후 목표 구조

### 4.1 변경 후 전체 흐름

변경 후 Reader 흐름은 다음이 되어야 한다.

```text
S-01 performCharterStrictSearch()
  → 각 지역구별 findValidCharterPropertiesInDistrict()
      → 가격 ZSet 조회 + 면적 ZSet 조회를 MULTI/EXEC로 실행
      → Java에서 교집합 계산
      → 교집합 ID에 대해 Redis Hash 상세 조회
      → hard condition 검증
      → List<PropertyDetail> 반환

S-02 checkAndPerformCharterFallback()
  → 검증 통과 PropertyDetail 개수 기준으로 fallback 판단

S-03 performCharterExpandedSearch()
  → 완화 조건으로 다시 S-01 수행
  → fallback 결과도 동일하게 Hash 상세 검증 통과 PropertyDetail 기준

S-04 calculateCharterPropertyScores()
  → 이미 검증된 PropertyDetail을 입력으로 받음
  → Redis Hash를 다시 읽지 않음
  → ReviewStatistics 조회용 propertyId만 추출
  → 기존 점수 계산 유지

S-05 calculateDistrictScoresAndSort()
  → 기존 흐름 유지

S-06 generateCharterFinalResponse()
  → 기존 흐름 유지
```

### 4.2 핵심 타입 변경

기존 타입:

```java
Map<String, List<String>> districtProperties
```

변경 타입:

```java
Map<String, List<PropertyDetail>> districtProperties
```

변경 후 이 타입의 의미는 다음이다.

```text
지역구별 hard condition 검증을 통과한 실제 추천 가능 매물 목록
```

이 타입 변경은 단순 리팩터링이 아니라 검색 결과의 의미 변경이다.

## 5. 필수 수정 항목

## 5.1 `SearchResult` 타입 변경

현재 내부 클래스는 다음 구조다.

```java
private static class SearchResult {
    private String searchStatus;
    private String message;
    private Map<String, List<String>> districtProperties;
    ...
}
```

다음처럼 변경한다.

```java
private static class SearchResult {
    private String searchStatus;
    private String message;
    private Map<String, List<PropertyDetail>> districtProperties;

    public static SearchResultBuilder builder() {
        return new SearchResultBuilder();
    }

    public static class SearchResultBuilder {
        private String searchStatus;
        private String message;
        private Map<String, List<PropertyDetail>> districtProperties;

        public SearchResultBuilder searchStatus(String searchStatus) {
            this.searchStatus = searchStatus;
            return this;
        }

        public SearchResultBuilder message(String message) {
            this.message = message;
            return this;
        }

        public SearchResultBuilder districtProperties(
                Map<String, List<PropertyDetail>> districtProperties) {
            this.districtProperties = districtProperties;
            return this;
        }

        public SearchResult build() {
            SearchResult result = new SearchResult();
            result.searchStatus = this.searchStatus;
            result.message = this.message;
            result.districtProperties = this.districtProperties;
            return result;
        }
    }

    public String getSearchStatus() {
        return searchStatus;
    }

    public String getMessage() {
        return message;
    }

    public Map<String, List<PropertyDetail>> getDistrictProperties() {
        return districtProperties;
    }
}
```

주의:
- 기존 `SearchResult`를 Lombok builder로 바꾸는 것은 선택 사항이다.
- 현재 코드 스타일을 유지하려면 수동 builder를 그대로 유지해도 된다.
- 중요한 것은 `districtProperties` 타입과 의미 변경이다.

## 5.7 Reader 인덱스 조회 MULTI/EXEC 적용

`findValidCharterPropertiesInDistrict()` 내부의 가격 ZSet 조회와 면적 ZSet 조회를 `SessionCallback` 기반 `MULTI/EXEC`로 묶는다.

필요 import가 없다면 추가한다.

```java
import org.springframework.dao.DataAccessException;
import org.springframework.data.redis.core.RedisOperations;
import org.springframework.data.redis.core.SessionCallback;
```

구현 예시는 다음과 같다.

```java
@SuppressWarnings("unchecked")
private List<PropertyDetail> findValidCharterPropertiesInDistrict(
        String district,
        CharterRecommendationRequestDto request) {
    try {
        String charterPriceIndexKey = "idx:charterPrice:" + district;
        String areaIndexKey = "idx:area:" + district + ":전세";

        List<Object> txResults = redisHandler.redisTemplate.execute(
                new SessionCallback<List<Object>>() {
                    @Override
                    public List<Object> execute(RedisOperations operations) throws DataAccessException {
                        operations.multi();

                        operations.opsForZSet().rangeByScore(
                                charterPriceIndexKey,
                                request.getBudgetMin(),
                                request.getBudgetMax());

                        operations.opsForZSet().rangeByScore(
                                areaIndexKey,
                                request.getAreaMin(),
                                request.getAreaMax());

                        return operations.exec();
                    }
                }
        );

        if (txResults == null || txResults.size() < 2) {
            return Collections.emptyList();
        }

        Set<Object> priceValidObjects = (Set<Object>) txResults.get(0);
        Set<Object> areaValidObjects = (Set<Object>) txResults.get(1);

        if (priceValidObjects == null || priceValidObjects.isEmpty()) {
            return Collections.emptyList();
        }

        if (areaValidObjects == null || areaValidObjects.isEmpty()) {
            return Collections.emptyList();
        }

        Set<String> priceValidIds = priceValidObjects.stream()
                .map(Object::toString)
                .collect(Collectors.toSet());

        Set<String> areaValidIds = areaValidObjects.stream()
                .map(Object::toString)
                .collect(Collectors.toSet());

        priceValidIds.retainAll(areaValidIds);

        if (priceValidIds.isEmpty()) {
            return Collections.emptyList();
        }

        if (f009RaceLatch != null) {
            log.info("Latch 해제");
            f009RaceLatch.releaseWriterIfTargetIncluded(priceValidIds);
        }

        List<PropertyDetail> propertyDetails =
                getMultipleCharterPropertiesFromRedis(new ArrayList<>(priceValidIds));

        return propertyDetails.stream()
                .filter(Objects::nonNull)
                .filter(detail -> matchesHardCondition(detail, request))
                .collect(Collectors.toList());

    } catch (Exception e) {
        log.info("전세 매물 검색 중 오류 - 지역구: {}", district, e);
        return Collections.emptyList();
    }
}
```

주의:
- 위 코드는 설계 기준 예시다.
- 실제 프로젝트의 Redis serializer 설정에 따라 `txResults.get(0)`의 타입이 `Set<Object>`가 아닐 수 있으므로, 컴파일 및 런타임 확인이 필요하다.
- 기존 코드에서 `areaIndexKey`가 중복 선언되어 있다면 정리한다.
- `f009RaceLatch`는 테스트 훅이다. production 코드 정리 시 `@Profile("f009-test")` 복구 여부를 별도 판단한다.

## 5.8 hard condition 검증 메서드 추가

`CharterRecommendationService` 내부에 다음 메서드를 추가한다.

```java
private boolean matchesHardCondition(
        PropertyDetail detail,
        CharterRecommendationRequestDto request) {

    if (detail == null) {
        return false;
    }

    if (detail.getDeposit() == null) {
        return false;
    }

    if (detail.getAreaInPyeong() == null) {
        return false;
    }

    if (!"전세".equals(detail.getLeaseType())) {
        return false;
    }

    if (!"ACTIVE".equals(detail.getStatus())) {
        return false;
    }

    return detail.getDeposit() >= request.getBudgetMin()
            && detail.getDeposit() <= request.getBudgetMax()
            && detail.getAreaInPyeong() >= request.getAreaMin()
            && detail.getAreaInPyeong() <= request.getAreaMax();
}
```

중요:
- 위 비교식은 현재 DTO/도메인 타입에 맞춰 조정해야 한다.
- `deposit`이 `Long`, `Integer`, `BigDecimal` 중 무엇인지 확인하고 컴파일 가능한 비교식으로 수정한다.
- `status`가 `String`이 아니라 enum 또는 enum name 문자열이라면 실제 타입에 맞춰 비교한다.
- `leaseType`이 코드값 `CHARTER`인지 한글값 `전세`인지 현재 Redis Hash 저장값을 확인하고 맞춘다.
- 기존 Hash builder가 `leaseType`을 어떤 값으로 저장하는지 반드시 확인한다.

권장 보강:

```java
private boolean isActiveStatus(PropertyDetail detail) {
    return "ACTIVE".equals(String.valueOf(detail.getStatus()));
}
```

이처럼 `String.valueOf()`를 사용하면 status가 enum이어도 비교가 가능하다. 다만 무분별한 문자열 비교가 싫다면 실제 타입을 확인한 뒤 enum 비교로 구현한다.

## 6. Writer 쪽 처리 기준

### 6.1 Writer MULTI/EXEC는 유지한다

`CharterPropertyWriteService.syncRedisAfterUpdate()`에 적용된 Redis `MULTI/EXEC` 구조는 유지한다.

목적:

```text

- Writer Redis 갱신 단위의 원자성 확보
```

### 6.2 이번 Claude 작업의 기본 수정 범위는 Reader다

`CharterPropertyWriteService`는 다음만 확인한다.

```text

- F009 테스트 목적 주석/프로파일 처리 상태가 운영 코드에 남아도 되는지 별도 확인 필요
```

## 7. Lua Script / Redis Function 확장안

### 7.1 이번 기본 구현에는 포함하지 않는다

이번 Claude 구현 작업의 기본안은 Java 애플리케이션 레벨에서 다음을 수행하는 것이다.

```text
- Reader 인덱스 조회 MULTI/EXEC
- 교집합 직후 Hash 상세 조회
- hard condition 검증
- 검증 통과 PropertyDetail 기준 fallback 및 점수 계산
```

Redis Lua Script 또는 Redis Function은 이번 기본 구현에 포함하지 않는다.

### 7.2 확장안으로만 문서화한다

더 강한 Redis-level 정합성이 필요하다면 Lua Script 또는 Redis Function으로 다음 작업을 Redis 서버 내부의 단일 atomic execution으로 묶을 수 있다.

```text
- 가격 ZSet 조회
- 면적 ZSet 조회
- 교집합 계산
- 교집합 ID에 대한 Hash 상세 조회
- hard condition 검증
- 검증 통과 ID 또는 상세 데이터 반환
```

이 방식은 인덱스 후보 조회와 Hash 상세 조회 사이에 Writer Redis 명령이 끼어드는 문제를 Redis 서버 내부에서는 제거할 수 있다.

단점:

```text
- script/function 실행 동안 Redis 서버를 점유한다.
- 후보 수가 많으면 Redis 응답 지연이 증가할 수 있다.
- 25개 자치구 전체 추천 흐름을 하나의 script로 묶으면 과도하다.
```

적용한다면 다음 범위로 제한한다.

```text
권장:
- 자치구 단위
- 후보 수 제한 단위
- hard condition 검증 중심

비권장:
- 25개 자치구 전체를 하나의 script로 처리
- 리뷰 통계 조회 포함
- 점수 계산 포함
- 정렬/추천 전체 로직 포함
```

## 10. 수정 금지 및 주의 사항

### 10.1 질의-수정 전역 직렬화 금지

다음 구조를 구현하지 않는다.

```text
- 추천 조회 전체에 전역 lock 적용
- 수정 API 전체에 전역 lock 적용
- 조회/수정 요청을 queue로 순차 처리
- 조회 중이면 수정 API를 대기시키는 구조
- 수정 중이면 조회 API 전체를 대기시키는 구조
```

### 10.2 무한 재시도 금지

다음 구조를 구현하지 않는다.

```text
while (정합성 불일치) {
    인덱스 조회부터 다시 수행
}
```

정합성 해결은 재시도가 아니라 hard condition 검증과 책임 재배치로 수행한다.

### 10.3 S-04에서 Redis Hash 재조회 금지

변경 후 S-04는 Redis Hash 상세를 다시 읽지 않는다.

다음 호출이 S-04에 남아 있으면 설계 위반이다.

```java
getMultipleCharterPropertiesFromRedis(propertyIds)
```

이 호출은 `findValidCharterPropertiesInDistrict()` 내부로 이동해야 한다.

### 10.4 조건 밖 매물을 응답에 포함하는 보정 금지

추천 매물 3개를 맞추기 위해 budget/area/status/leaseType 조건을 위반한 매물을 포함하면 안 된다.

추천 개수 확보는 soft 목표다. hard condition은 절대 깨면 안 된다.

## 13. 참고 공식 문서

Redis 동작 근거는 다음 공식 문서를 기준으로 한다.

```text
Redis Transactions / MULTI / EXEC
https://redis.io/docs/latest/develop/using-commands/transactions/

Redis Pipelining
https://redis.io/docs/latest/develop/using-commands/pipelining/

Redis Lua Scripting
https://redis.io/docs/latest/develop/programmability/eval-intro/
```

요약:

```text
- MULTI/EXEC는 트랜잭션 안의 명령들이 순차 실행되고 다른 클라이언트 요청이 중간에 끼어들지 않는 isolated operation을 제공한다.
- Pipeline은 여러 명령을 한 번에 보내 왕복 비용을 줄이는 성능 최적화이며, 원자성 보장 수단이 아니다.
- Lua Script / Redis Function은 Redis 서버 내부에서 atomic execution을 제공하지만, 실행 동안 Redis 서버를 점유하므로 기본 구현에는 포함하지 않는다.
```

## 14. 최종 요약

이번 수정의 본질은 다음이다.

```text
기존 구조:
Redis 인덱스 후보 ID 조회
→ 후보 ID 개수로 fallback 판단
→ S-04에서 Hash 상세 조회
→ 조건 밖 매물이 응답에 포함될 수 있음

변경 구조:
Redis 인덱스 후보 ID 조회
→ 교집합
→ 즉시 Hash 상세 조회
→ hard condition 검증
→ 검증 통과 PropertyDetail 개수로 fallback 판단
→ S-04 점수 계산
→ 조건 밖 매물 응답 차단
```

F009의 목표는 조회 요청 전체를 완전 직렬화하거나 완전한 snapshot을 제공하는 것이 아니다.

F009의 목표는 다음 불변조건을 보장하는 것이다.

```text
응답에 포함되는 모든 매물은 응답 생성 과정에서 Redis Hash로 확인한 실측값 기준으로 사용자의 hard condition을 만족해야 한다.
```
