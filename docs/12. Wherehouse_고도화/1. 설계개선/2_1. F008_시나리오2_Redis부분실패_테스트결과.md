# F008 문제 시나리오 2 — Redis 부분 실패 테스트 결과

**작성일:** 2026-04-28  
**대상 코드:** `CharterPropertyWriteService.createProperty()` → `syncRedisAfterCreate()`  
**테스트 방식:** Postman 단발 요청 + 인위적 실패 주입 (Hash 저장 완료 후, ZSet 추가 직전에 RuntimeException)

---

## 1. 문제 정의

매물 등록 요청 시, RDB 저장은 성공하고 Redis 동기화도 시작되었으나 **일부 Redis 작업만 성공**하는 상황이다.

현재 `syncRedisAfterCreate()`의 Redis 작업 순서:

```
① Hash 저장      (property:charter:{id})              — 매물 상세 데이터
② ZSet 추가      (idx:charterPrice:{district})        — 가격 기반 추천 검색 인덱스
③ ZSet 추가      (idx:area:{district}:전세)           — 평수 기반 추천 검색 인덱스
④ bounds 확장    (bounds:{district}:전세 — minPrice/maxPrice)
⑤ bounds 확장    (bounds:{district}:전세 — minArea/maxArea)
```

5개 작업은 각각 독립된 Redis 명령으로, 어느 지점에서든 네트워크 에러, 타임아웃, Redis OOM 등이 발생하면 해당 시점 이후의 작업은 미실행된다.

시나리오 1(Redis 전체 실패)에서는 "있거나 없거나" 두 상태만 존재했으나, 시나리오 2에서는 **"어디까지 성공했는지에 따라 불일치 형태가 달라진다"**는 점에서 복구 난이도가 높다.

---

## 2. 테스트 범위 및 한계

Redis 부분 실패는 5개 작업의 어느 지점에서든 발생 가능하며, 장애 원인(네트워크 단절, 타임아웃, OOM 등)과 무관하게 결과는 동일하다: **해당 시점 이후의 Redis 작업이 미실행된다.**

모든 지점별 장애를 개별 재현하는 것은 비현실적이며, 목적에도 부합하지 않는다. 본 테스트는 비즈니스 영향이 가장 큰 지점 — **① Hash 저장 성공 + ② ZSet 추가 실패** — 에 인위적 예외를 삽입하여 부분 실패 상태를 실증한다.

이 지점을 선택한 근거:
- Hash가 존재하면 매물 상세 조회(Redis Hash 기반)에서 이 매물의 데이터가 반환된다.
- 하지만 ZSet(가격 인덱스, 평수 인덱스)에는 없으므로 추천 검색(Sorted Set 교집합 기반)에서 누락된다.
- 사용자 관점에서 "매물 정보는 조회되는데 추천에는 안 나오는" 반쪽짜리 상태가 발생한다.
- 이것은 시나리오 1(전체 없음)과 질적으로 다른 불일치이며, 시나리오 1은 "없으니까 전부 다시 넣으면 됨"이지만 시나리오 2는 "어디까지 성공했고 어디서부터 실패했는지 파악해야 함"이라는 점에서 복구 난이도가 높다.

각 지점별 세분화된 예외 처리 및 보상 전략은 후속 설계 판단에서 다룬다.

## 3. 인위적 실패 주입 방식

`syncRedisAfterCreate()` 내부에서 Hash 저장(`opsForHash().putAll()`) 완료 직후, 첫 번째 ZSet 추가(`opsForZSet().add()`) 직전에 `RuntimeException`을 삽입한다.

```java
private void syncRedisAfterCreate(PropertyCharterEntity entity) {
    String propertyId = entity.getPropertyId();
    String districtName = entity.getDistrictName();

    String hashKey = "property:charter:" + propertyId;
    Map<String, Object> hashFields = propertyHashBuilder.buildCharterHash(entity);
    redisHandler.redisTemplate.opsForHash().putAll(hashKey, hashFields);   // ① Hash 저장 성공

    // ── F008 테스트②: Redis 부분 실패 (Hash 성공 + ZSet 실패) ──
    // Hash는 저장되었으나 인덱스(ZSet)와 bounds는 미반영되어
    // 매물 상세 조회는 가능하지만 추천 검색에서 누락되는 반쪽짜리 불일치를 실증한다.
    if(true) { throw new RuntimeException("[F008-TEST] 테스트②: Hash 저장 후 ZSet 추가 시점 실패 시뮬레이션"); }

    redisHandler.redisTemplate.opsForZSet().add(                           // ② ZSet 추가 (미실행)
            "idx:charterPrice:" + districtName, propertyId,
            entity.getDeposit().doubleValue());
    // ③④⑤ 이하 미실행
}
```

## 5. 테스트 A — @Transactional 없이 Redis 부분 실패

### 5.1. 조건

- `createProperty()` 메서드에 `@Transactional` **미적용** (주석 상태)
- `syncRedisAfterCreate()` 내 Hash 저장 후, ZSet 추가 전에 RuntimeException 삽입

### 5.2. 동작 원리

`@Transactional`이 없으므로 `save()`가 즉시 COMMIT한다. 이후 `syncRedisAfterCreate()`에서 Hash 저장까지는 성공하고, ZSet 추가 전에 예외가 발생한다. Oracle 데이터는 이미 확정되었으므로 되돌릴 수 없다.

```
createProperty() 진입
  → save(entity)                               // Oracle INSERT + 즉시 COMMIT (확정)
  → syncRedisAfterCreate(entity) 진입
    → opsForHash().putAll(...)                  // ① Redis Hash 저장 (확정)
    → RuntimeException 발생                     // ② ZSet 추가 미실행
  → 예외 전파 → 500 에러 응답
```

### 5.3. 결과

| 확인 대상 | 확인 방법 | 결과 |
|-----------|----------|------|
| API 응답 | Postman | 500 Internal Server Error |
| Oracle | `SELECT ... WHERE apt_nm = 'F008테스트아파트';` | 1건 존재. PROPERTY_ID=5c627892ec0c5bdd218bdd1a70d9f931, STATUS=ACTIVE |
| Redis Hash | `EXISTS property:charter:5c627892ec0c5bdd218bdd1a70d9f931` | **1 (있음)** |
| Redis ZSet | `ZSCORE idx:charterPrice:강남구 5c627892ec0c5bdd218bdd1a70d9f931` | **(nil) (없음)** |

### 5.4. 문제 상태

**반쪽짜리 불일치** — Oracle에 ACTIVE 매물이 존재하고, Redis Hash에도 매물 상세 데이터가 있다. 그러나 가격 인덱스(ZSet)에는 없으므로 추천 검색에서 이 매물이 누락된다. 매물 상세 조회(Hash 기반)는 정상 동작하지만, 사용자가 추천 검색으로 이 매물을 발견할 수 없다.

시나리오 1(Redis 전체 실패, @Transactional 미적용)에서는 Oracle에만 데이터가 있었으나, 시나리오 2에서는 Redis에도 **부분적으로** 데이터가 존재하여 불일치 형태가 더 복잡하다.

## 6. 테스트 B — @Transactional 적용 후 Redis 부분 실패

### 6.1. 사전 작업

테스트 A에서 Oracle과 Redis에 남은 데이터를 정리한다.

```sql
DELETE FROM PROPERTIES_CHARTER WHERE PROPERTY_ID = '5c627892ec0c5bdd218bdd1a70d9f931';
COMMIT;
```

```bash
redis-cli DEL property:charter:5c627892ec0c5bdd218bdd1a70d9f931
```

### 6.2. 조건

- `createProperty()` 메서드에 `@Transactional` **적용** (주석 해제)
- 예외 삽입 위치는 테스트 A와 동일

### 6.3. 동작 원리

`@Transactional`이 적용되므로 `save()`는 COMMIT을 보류한다. 이후 `syncRedisAfterCreate()`에서 Hash 저장은 Redis에 즉시 확정되고, ZSet 추가 전에 예외가 발생한다. `@Transactional`이 예외를 감지하여 Oracle INSERT를 ROLLBACK하지만, 이미 확정된 Redis Hash는 롤백 대상이 아니므로 그대로 남는다.

```
createProperty() 진입 → @Transactional이 트랜잭션 시작
  → save(entity)                               // Oracle INSERT 전송 (COMMIT 보류)
  → syncRedisAfterCreate(entity) 진입
    → opsForHash().putAll(...)                  // ① Redis Hash 저장 (즉시 확정)
    → RuntimeException 발생                     // ② ZSet 추가 미실행
  → @Transactional이 예외 감지 → Oracle ROLLBACK
  → 예외 전파 → 500 에러 응답
```

### 6.4. 결과

| 확인 대상 | 확인 방법 | 결과 |
|-----------|----------|------|
| API 응답 | Postman | 500 Internal Server Error |
| Oracle | `SELECT ... WHERE apt_nm = 'F008테스트아파트';` | **0건 (ROLLBACK)** |
| Redis Hash | `EXISTS property:charter:5c627892ec0c5bdd218bdd1a70d9f931` | **1 (있음)** |
| Redis ZSet | `ZSCORE idx:charterPrice:강남구 5c627892ec0c5bdd218bdd1a70d9f931` | **(nil) (없음)** |

### 6.5. 문제 상태

**유령 Hash** — Oracle에는 매물이 존재하지 않지만(ROLLBACK), Redis에는 Hash 데이터가 남아있다. 이것은 시나리오 1의 `@Transactional` 적용 결과(양쪽 모두 없음 — 정합성 유지)보다 **나쁜 상태**다.

시나리오 1에서는 `@Transactional` 적용이 정합성을 보호했으나, 시나리오 2에서는 `@Transactional`이 **오히려 불일치를 악화**시킨다. Redis 작업이 트랜잭션 경계 안에서 부분 실행된 후 Oracle만 롤백되기 때문이다.

## 7. 전체 결과 비교

### 7.1. 시나리오 2 — @Transactional 유무 비교

| 조건 | Oracle | Redis Hash | Redis ZSet | 불일치 유형 |
|------|--------|-----------|-----------|------------|
| `@Transactional` 없음 | 있음 (커밋) | 있음 | 없음 | 반쪽짜리 — 상세 조회 가능, 추천 검색 누락 |
| `@Transactional` 있음 | **없음 (롤백)** | 있음 | 없음 | **유령 Hash** — Oracle에 없는 매물의 Hash가 Redis에 잔존 |

### 7.2. 시나리오 1 vs 시나리오 2 — @Transactional 적용 시 비교

| 시나리오 | 예외 발생 시점 | Oracle | Redis Hash | Redis ZSet | @Transactional 효과 |
|---------|-------------|--------|-----------|-----------|-------------------|
| 시나리오 1 (Redis 전체 실패) | Redis 작업 진입 전 | 없음 (롤백) | 없음 | 없음 | **정합성 유지** — 양쪽 모두 없음 |
| 시나리오 2 (Redis 부분 실패) | Redis 작업 도중 | 없음 (롤백) | **있음** | 없음 | **불일치 악화** — 유령 Hash 발생 |

## 8. 시사점

### 8.1. @Transactional만으로는 Redis 부분 실패를 보호할 수 없다

`@Transactional`이 롤백하는 것은 JDBC Connection을 통해 전송된 SQL뿐이다. Redis 작업은 별도의 네트워크 연결이므로 롤백 대상이 아니다. Redis 작업이 `@Transactional` 메서드 내부에서 부분 실행된 후 예외가 발생하면, Oracle은 롤백되지만 Redis에 이미 확정된 데이터는 그대로 남아 유령 데이터가 된다.

### 8.2. RDB 커밋 확정 전에 Redis를 건드리면 안 된다

시나리오 1에서는 `@Transactional`이 정합성을 보호했지만, 시나리오 2에서는 오히려 악화시켰다. 두 시나리오의 차이는 **예외 발생 시점에 Redis에 이미 쓰기가 발생했는가** 여부다. Redis 쓰기가 RDB 커밋 확정 이전에 실행되는 한, `@Transactional`의 ROLLBACK이 Redis-Oracle 불일치를 만드는 구조적 결함은 해소되지 않는다.

이 결과는 후속 설계 판단 — RDB 커밋 확정 후에 Redis 동기화를 실행하는 구조(afterCommit / 이벤트 리스너 / 메서드 분리) — 가 **필수적**임을 실증한다.

### 8.3. Redis 부분 실패는 별도의 보상 전략이 필요하다

RDB-Redis 실행 순서를 변경하더라도(커밋 후 Redis 동기화), Redis 동기화 내부의 부분 실패는 여전히 발생 가능하다. Hash는 성공했지만 ZSet이 실패한 상태에서 어떻게 복구할 것인가 — 성공분 롤백(삭제), Pipeline 일괄 실행, Redis MULTI/EXEC 원자 실행 등의 선택지가 존재하며, 이는 별도의 설계 판단으로 다룬다.
