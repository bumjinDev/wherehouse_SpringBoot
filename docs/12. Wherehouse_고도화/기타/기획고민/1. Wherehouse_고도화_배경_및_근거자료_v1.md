## 1. 고도화의 필요성 진단

Wherehouse 서비스와 그 포트폴리오의 현재 상태를 세 가지 관점에서 구조적 문제로 식별한다.

### 1.1 포트폴리오 트러블슈팅 구조의 편중

기존 Wherehouse 포트폴리오의 트러블슈팅 6건 중 5건이 **"느린 것을 측정 → 빠르게 만듦 → 수치 제시"** 패턴으로 구성되어 있다. 이는 성능 측정/개선이지, 비즈니스 요구사항 간 충돌에서 대안 간 트레이드오프를 분석하고 선택 근거를 명시하는 엔지니어링 의사결정이 아니다. IN절 Chunking 1건만이 V$SQL 실행계획 분석 → Chunk 크기별 비교 → 트레이드오프 기반 선택 구조를 갖추고 있다.

구체적 기존 6건의 구성(포트폴리오 PDF 기준):

- **#1 외부 API 병목**: CompletableFuture + ThreadPoolExecutor 병렬화로 72.3% 응답 시간 감소(1,324ms → 367ms). ForkJoinPool(commonPool) 기각 사유 포함. 대안 검토표(ForkJoinPool vs CompletableFuture+ThreadPoolExecutor) 존재.
- **#3 Oracle IN절 Chunking**: N+1 → Bulk Fetch → Chunk 61, V$SQL 기반 3단계 실행계획 분석. Chunk 크기별 실측 비교표(Bulk/Chk1000/Chk22/Chk61) 7개 지표 비교. Soft Parse 88.5%, 심각 경합 0%, 총 비용 Chk22 대비 -11%. **트레이드오프 기반 선택 구조를 갖춘 유일한 건**.
- **#4 LIKE 검색 Index**: '%keyword%' Full Table Scan → LIKE 'keyword%' + apt_nm 인덱스 전환. Oracle Text(Full-Text Index) 대안 기각(별도 역인덱스 구조 생성·동기화 관리 필요, 1인 프로젝트 운영 복잡도 대비 과다). 실행계획 비교(TABLE ACCESS FULL Cost:137 → INDEX RANGE SCAN Cost:5). 쿼리 소요시간 77.2% 감소, 슬로우 쿼리 103건→0건.
- **#5 OOM Slice 기반 청크**: findAll() 일괄 로드 → Slice 페이징. Page<T> vs Slice<T> 대안 비교(Page는 COUNT(*) 쿼리 자동 실행으로 불필요한 오버헤드). 힙 피크 64.4%→17.6%, 엔티티 메모리 99.2MB→32.5MB. OOM으로 인한 배치 전체 실패 vs 처리 시간 63% 증가의 트레이드오프, 메모리 안정성 우선 판단.
- **#6 Redis Pipeline RTT**: 순차 Redis 호출 → Pipeline 일괄 전송. 지역구당 3회 RTT × 25 자치구 = 75회 네트워크 왕복을 Pipeline으로 1/3 절감. Wireshark TCP 패킷 분석으로 Layer 2 RTT 검증(순차 38.5ms 평균 vs Pipeline 39.3ms 평균, 1회 왕복 비용 증가 없음 확인). 전체 검색 시간 34.0% 감소, 지역구별 평균 응답 54.4% 감소, 네트워크 왕복 횟수 66.7% 감소.

위 5건(#1·#4·#5·#6 + 기타)은 "측정 → 개선 → 수치" 패턴이고, #3 하나만 트레이드오프 기반 선택 구조를 갖는다.

**비즈니스 영향**: 면접관이 "설계 역량"이 아닌 "측정 역량"으로 평가할 위험. 취업박람회 멘토 2인이 공통으로 지적한 "설계 능력 = 신입 최우선 역량" 기준에 미달한다.

### 1.2 서비스 비즈니스 복잡도의 부재

현재 서비스는 Redis 읽기 → 점수 계산 → 정렬 → 반환의 단방향 조회 파이프라인에 고착되어 있다. 설계 명세서 v2의 리뷰 시스템은 단순 CRUD + Write-Through 동기화 수준이다. 동시 쓰기 충돌·트랜잭션 간 격리 문제·락 전략 선택의 필요가 발생하지 않는 구조이며, 동시성 제어·트랜잭션 설계·스레드 풀 분리 등 핵심 백엔드 역량을 적용할 비즈니스 시나리오가 부재하다.

**비즈니스 영향**: 포트폴리오에서 증명할 수 있는 설계 의사결정의 종류 자체가 구조적으로 제한된다.

### 1.3 기술 적용의 깊이 부족

CompletableFuture + ThreadPoolExecutor는 I/O 바운드 외부 API 호출에만 적용되어 있다. Redis Sorted Set·HashMap 등은 단순 사용 수준이다. Semaphore는 ChatService 입장 제어 단일 케이스에 한정되어 있다. 워크로드 특성별 스레드 풀 분리·비즈니스 요구사항의 시간/공간복잡도 제약에 따른 자료구조 비교 선택·낙관적/비관적 락의 시나리오별 구분 적용 등 의사결정 과정이 없다.

**비즈니스 영향**: "기술을 사용했다"는 보여주지만, "왜 이 기술이고 대안은 무엇이었으며 무엇을 포기했는가"를 보여주지 못한다.

## 2. 누락 역량 4개 상세

1절에서 진단한 3가지 구조적 문제를 기술 역량 관점에서 재분류하면 4개 누락 역량으로 수렴한다.

### 2.1 동시성 제어 영역

- synchronized와 ReentrantLock이 비즈니스 로직 맥락에서 기능별로 구분 적용된 적 없음
- **DB 수준 락 기술을 학습 및 비즈니스에 반영하지 않았음**
- ChatService의 Semaphore 적용은 입장 제어 단일 케이스에 한정되어, "왜 Semaphore인가"에 대한 4종 비교 실측은 존재하나 비즈니스 시나리오 다양성이 부족

### 2.2 트랜잭션 설계 영역

- 트랜잭션 경계 설정·전파 전략(Propagation)·격리 수준(Isolation Level) 선택이 비즈니스 정합성 요구사항에 따라 의도적으로 결정된 적 없음
- 설계 명세서 v2의 ReviewWriteService는 @Transactional 단일 적용 수준이며, 배치 트랜잭션과 실시간 서비스 트랜잭션 간 격리 전략이 "Overwrite 방식" 서술에 그침

### 2.3 스레드 풀 설계 영역

- CompletableFuture + ThreadPoolExecutor는 I/O 바운드 외부 API 호출에만 적용되어 있음
- **CPU 바운드 연산에 ForkJoinPool을 적용하거나, 워크로드 특성에 따라 스레드 풀을 분리 설계한 적 없음**
- 스레드 풀 크기 결정의 정량적 근거(코어 수·I/O 대기 비율 등)가 부재

### 2.4 자료구조/알고리즘 영역

- Redis Sorted Set·HashMap 등을 "사용"했으나, 비즈니스 요구사항의 시간복잡도/공간복잡도 제약에 따라 자료구조를 비교 선택한 의사결정 과정이 없음
- 알고리즘이 비즈니스 로직에 반영된 적 없음 (추천 점수 산출은 가중 합산 수준)

## 3. 고도화 목표

### 3.1 1차 목표 — 채용 시장 요구 역량을 비즈니스 기능에 내재화

현재 백엔드 채용 공고에서 필수/우대로 반복 등장하는 기술 역량(DB 락·동시성 제어·트랜잭션 설계·스레드 풀 분리·자료구조/알고리즘 비교 선택 등)을 프로젝트 진행 과정에서 조사하고, 조사 결과를 기반으로 포트폴리오에 자연스럽게 녹여낼 수 있는 비즈니스 기능을 도출한다.

기술 역량 목록은 사전에 고정하지 않고, 프로젝트 초기에 실제 채용 공고 조사를 통해 확정한다. 이를 통해 "학습을 위해 기능을 만든" 구조가 아니라, **"채용 시장이 요구하는 역량이 비즈니스 기능에 의해 자연스럽게 요구되는" 구조**를 확보한다.

### 3.2 2차 목표 — 포트폴리오 구조 전환

기존 "측정 → 개선 → 수치" 패턴에서 **"비즈니스 요구사항 충돌 → 대안 분석 → 트레이드오프 → 선택 근거 → 정량 결과"** 패턴으로 트러블슈팅 서술 구조를 전환한다.

- 각 설계 결정마다 "왜 이 방식인가 / 대안은 무엇이었나 / 트레이드오프는 무엇인가"를 명시한다
- JUnit/Mockito 기반 테스트 코드로 설계 의도의 검증 가능성을 확보한다

### 3.3 기대 효과

- **포트폴리오 설계 밀도 전환**: 기존 "측정 → 개선 → 수치" 5건 + "트레이드오프 기반 선택" 1건 구조에서, 모든 트러블슈팅이 "비즈니스 요구사항 충돌 → 대안 분석 → 트레이드오프 → 선택 근거 → 정량 결과" 구조를 갖추는 방향으로 전환
- **누락 역량 내재화**: 동시성 제어(낙관적/비관적 락 시나리오별 선택)·트랜잭션 설계(Propagation/Isolation Level 의도적 결정)·스레드 풀 분리 설계·자료구조/알고리즘 비교 선택이 비즈니스 시나리오에 의해 자연스럽게 요구되고 적용된 포트폴리오 확보
- **면접 대응력 강화**: 각 설계 결정마다 "왜 이 방식인가 / 대안은 무엇이었나 / 트레이드오프는 무엇인가"가 명시되어 있으므로, 취업박람회 멘토가 강조한 "기술 선택 근거의 논리적 설명" 역량을 3분 이내 구술로 즉시 제시 가능
- **테스트 코드 부재 해소**: JUnit/Mockito 기반 단위 테스트가 포트폴리오에 포함되어, "테스트 수행 흔적이 존재하고 기능별 설계 방법이 간결하게 기술된" 서류 통과 포인트 충족
- **비즈니스 마인드 어필**: 기능 도출 근거가 실제 부동산 서비스의 사용자 리뷰에서 출발하므로, "기술 오타쿠"가 아닌 "서비스의 목적과 사용자 관점에서 기술을 설명하는" 페르소나 전환

## 4. 핵심 규칙 4가지

### 4.1 규칙 1 — 비즈니스가 기술을 끌어내는 구조

기술 적용의 출발점은 항상 비즈니스 요구사항이다. "상호배제 락을 적용하고 싶으니 기능을 만들자"가 아니라, "이 기능의 동시 쓰기 패턴에서 정합성을 보장하려면 상호배제 락이 필요하다"의 방향이어야 한다. 기술 선택의 근거가 비즈니스 시나리오에서 자연스럽게 도출되지 않으면 면접관이 "기술을 위한 기술"로 판단한다.

### 4.2 규칙 2 — 실제 서비스 리뷰 기반 기능 도출 근거 확보

자체 상상이 아닌 실제 서비스의 운영 맥락에서 기능 필요성의 근거를 확보한다. 직방·다방·네이버 부동산의 앱스토어 리뷰에서 피드백을 수집하여, 해당 피드백이 기능 도출의 비즈니스 근거로 사용된다. 이 과정 자체가 포트폴리오에서 "기능 도출 근거"로 어필된다.

### 4.3 규칙 3 — 개인 프로젝트 바운더리 준수

운영 환경(미니 PC, 다중 사용자 접속)에서 실제 동작하며, 테스트 시 유의미한 데이터 규모를 확보해야 한다. 인프라 과투자(Kafka·K8s·MSA 등)·프론트엔드 과투자·도메인 지식 과투자 없이 백엔드 설계 역량을 증명하는 데 집중한다. 신입 포트폴리오로서 감당 가능한 범위를 초과하면 오히려 "실서비스 운영 수준이 아닌 학습용"이라는 인상을 줄 수 있다.

### 4.4 규칙 4 — 포트폴리오 문서화 구조 일관성

모든 트러블슈팅/설계 결정은 "문제 정의 → 대안 검토 → 트레이드오프 → 선택 근거 → 정량 결과" 구조로 문서화한다. 기존 포트폴리오 3페이지(기획의도·아키텍처·트러블슈팅) 구조와 일관성을 유지하며, 취업박람회 피드백에서 강조된 "문제 해결 프로세스 강조"·"데이터의 선별적 노출" 원칙을 적용한다.

## 5. 의사결정 문서화 유형 3분류

각 설계 결정을 문서화할 때, 해당 결정이 어떤 맥락에서 발생했는지를 명시하여 면접에서 "이 결정은 사전에 한 건가, 문제가 발생해서 한 건가, 실측해서 한 건가"라는 질문에 맥락이 즉시 전달되도록 한다.

### 5.1 유형 1 — 실제 문제 발생 후 해결

운영 또는 테스트 중 측정된 문제가 출발점이다. 서술 순서: "문제 발견 → 원인 분석 → 대안 검토 → 해결 → 정량 결과".

기존 포트폴리오의 #1·#4·#5·#6이 이 유형에 속한다. 매물 등록 고도화에서는 F008(Redis 다중 인덱스 부분 실패 전략)이 이 유형이다.

### 5.2 유형 2 — 설계 단계 사전 의사결정

문제가 발생하기 전, 아키텍처 또는 기술 선택 수준에서 사전에 판단한다. 서술 순서: "요구사항 분석 → 대안 비교 → 선택 근거".

기존 포트폴리오에 이 유형이 부재했다. 매물 등록 고도화의 F007(배치-사용자 데이터 머지 정책)이 이 유형의 첫 신규 건.

### 5.3 유형 3 — 실측 기반 의사결정

이론적으로 판단이 불가능하여, 실제 환경에서 측정한 데이터로만 결정이 가능한 경우다. 서술 순서: "가설 수립 → 실측 설계 → 측정 결과 → 데이터 기반 선택".

기존 포트폴리오의 #3(IN절 Chunking)이 이 유형의 유일한 건. 매물 등록 고도화에서는 F006(중복 감지 락 전략)과 F009(bounds 변경 점수 재계산)가 이 유형이다.

## 6. 채용 시장 기술 역량 빈도 분석

한국 백엔드 신입·주니어 채용 공고 기술 역량 빈도 분석(사람인·점핏 10만 건 공고 리포트, 캔디드 88,492건 JD 분석, 실제 공고 15건+ 직접 분석, 채용 담당자 71명 설문 종합) 결과.

### 6.1 1층 — 절대 필수 (없으면 서류 탈락)

| 기술 | 공고 등장 빈도 |
|------|--------------|
| API 설계 | ~95% |
| DB/쿼리 최적화 | ~90% |
| 자료구조/알고리즘 | ~80% |
| 설계 역량 | ~70% |

공고의 70~95%에서 필수로 등장하며, 이것 없이는 서류 심사 자체를 통과할 수 없다.

### 6.2 2층 — 핵심 우대 (있으면 서류 통과율 급상승)

| 기술 | 공고 등장 빈도 |
|------|--------------|
| CI/CD·Docker | ~60% |
| Redis 캐싱 | ~50% |
| 테스트 코드 | ~40% |
| 메시지 큐 | ~25% |

인프런 로드맵에서 "이것까지 갖추면 취업에 문제없다"고 평가한 구간에 해당한다.

### 6.3 3층 — 강력 차별화 (면접에서 압도적 인상)

| 기술 | 공고 등장 빈도 |
|------|--------------|
| 스레드풀/비동기 | ~20% |
| 모니터링 | ~20% |
| 트랜잭션 설계 | ~15% |
| 동시성 제어 | ~10% |

공고에서 직접 요구하는 빈도는 10~25%로 낮지만, 코멘토 합격자 분석에서 대기업·중견기업 합격자들이 공통적으로 보유한 차별화 역량으로 확인되었다.

### 6.4 핵심 발견

공고 빈도와 합격 기여도는 다르며, 동시성 제어·트랜잭션 설계는 공고 언급률 10~15%이지만 **면접 출제 빈도와 포트폴리오 차별화 효과는 최상급**이다. 트러블슈팅 문서화가 포트폴리오 #1 차별화 요소이다. "왜 이 기술을 선택했는가"를 설명하는 능력이 기술 자체만큼 중요하다.

이 발견이 매물 등록 고도화가 F006(동시성 제어)·F007(트랜잭션 설계 성격의 머지 정책)·F008(트랜잭션 경계)·F009(스레드풀 분리)로 수렴한 근거다. 공고 빈도 10~20%대의 3층 역량을 전부 커버하는 구성.

## 7. 실제 부동산 서비스 7개 분석

직방·다방·네이버 부동산·피터팬의 좋은방 구하기·호갱노노·아실·부동산플래닛 등 7개 주요 한국 부동산 서비스를 대상으로, Google Play Store/Apple App Store 사용자 리뷰, 블라인드·클리앙·DCInside 커뮤니티 사용자 불만, 앱 비교 기사, 기술 블로그를 종합 분석한 결과.

### 7.1 3강 플랫폼 핵심 기능 분석

#### 7.1.1 직방

632만 설치·MAU 229만(설치 수 1위). 핵심 차별화 기능은 전국 30세대 이상 모든 아파트를 구현한 3D 단지투어(동호수별 일조량·창밖 뷰 시뮬레이션)와, 중개 이력이 검증된 중개사의 1:1 케어와 계약서 정밀 진단을 제공하는 지킴중개 서비스다. 허위매물 대응으로는 헛걸음 시 현금 3만 원을 보상하는 헛걸음 보상제·삼진아웃(신고 3회 누적 시 영구 퇴출)제를 운영하며, 한국부동산원 허위매물 모니터링에 참여해 거래 완료 매물을 당일 삭제하고 있다. 2018년 호갱노노를 인수해 아파트 시세 분석 역량을 강화했고, 삼성SDS 홈IoT 인수로 스마트홈 사업까지 확장했다.

#### 7.1.2 다방

설치 342만·MAU 103만(2위). UGC와 AI에서 가장 앞서 있다. 동네이야기 커뮤니티는 동네 기반으로 이웃과 일상·알짜 정보를 공유하고 투표·매물 첨부 기능까지 지원한다. 부동산 리뷰 서비스는 방문 이용자가 직접 남기는 중개사 리뷰를 공개한다. AI 방찾기는 생성형 AI 챗봇으로 자연어 매물 검색을 지원한다. 필터링에서도 층수·창문 방향·엘리베이터·전세대출 가능 여부·반려동물 허용까지 세분화되어 있어, 원룸·자취생 시장에서 가장 높은 필터 만족도를 보인다. 중개사에게 카카오톡 메시지로 매물 가용 여부를 확인하는 매물확인 메신저(48시간 내 답변 의무, 미답변 시 매물 자동 비노출)도 다방만의 신뢰성 장치다.

#### 7.1.3 네이버 부동산

MAU 134만·매물 수 압도적 1위(중개사 광고 수수료가 저렴해 가장 많은 매물 등록). 네이버페이 부동산으로 개편하면서 매물지도 중심의 모바일 임장 환경을 구축했다. 최대 강점은 한국부동산원 시세·KB부동산 시세·매물 호가를 동시에 보여주는 3종 시세 비교이며, 방향 10가지 이상·현관 구조(계단식/복도식/복합식)·엘리베이터 유무까지 지원하는 업계 최정밀 필터링을 제공한다. 국토부 조사에서 낚시성 매물이 1.37%로 가장 낮은 허위매물 비율을 기록했다.

### 7.2 니치 플랫폼 4인방 분석

**피터팬**: 네이버 카페(회원 540만+)에서 출발한 국내 유일의 직거래 전문 플랫폼. 연간 약 75만 건의 매물을 네이버 부동산에 공급하는 공식 CP 파트너. First American과 제휴한 안심직거래 서비스로 전·월세 권리보험이 보증금 10억 원까지 커버하고 보증금 반환 소송 시 무료 법률 대리를 제공한다. 동네생활백서 거주후기에서는 실거주자가 층간소음·난방·주차·치안 등에 대한 리뷰를 남긴다.

**호갱노노**: 직방이 2018년 인수했지만 독립 노선 유지. MAU 179만·이탈률 21.8%(업계 최저)·평균 사용 기간 519일(최장). 3D 단지투어에서 계절별·시간대별 일조량을 시뮬레이션하고, 입주민이 직접 쓰는 층간소음·주차·학군 리뷰가 최대 자산이다. 매달 가장 많이 조회된 단지에 왕관 아이콘을 부여해 "대장 아파트"를 식별해준다.

**아실**: 2006년부터 현재까지의 국토부 실거래 데이터를 여러 단지를 하나의 차트에 겹쳐서 비교하는 기능이 핵심. 경쟁사에서 유료인 이 기능을 무료로 제공해 "투자자 필수 앱" 평가. 외지인 매수 비율·학군 비교·대지지분 계산·역전세 물건 식별 등 투자 분석 특화.

**부동산플래닛**: 국내 유일의 상업용 부동산 전문 데이터 플랫폼. 상가·사무실 AI 추정가(AVM)는 서울 57.9만 개·전국 240만 개 상업용 건물 대상으로 중위절대오차 5.80% 정확도 달성.

### 7.3 공통 기능 vs 차별화 매트릭스

**모든 서비스가 공통으로 제공하는 기능**: 지도 기반 매물 검색·찜/관심 매물·최근 본 매물·실거래가 조회·중개사 연결·신규 매물 알림.

**차별화가 발생하는 세 영역**:

- **UGC/커뮤니티**: 다방(동네이야기)과 피터팬(거주후기·카페)이 가장 활발. 호갱노노(입주민 리뷰)가 아파트 한정. 직방·네이버는 앱 내 직접적 UGC가 상대적으로 제한적.
- **AI 기능**: 다방(AI 방찾기 챗봇)·호갱노노(AI 중개사 추천)·부동산플래닛(AI 추정가 AVM)·아실(AI 미래가격 예측)이 각자 다른 방향으로 AI를 적용.
- **데이터 분석 깊이**: 아실(단지 간 가격 비교 차트)·호갱노노(3D 일조량·학원가·경사도)·부동산플래닛(AVM·입주기업·실효임대료)이 각 영역에서 가장 깊은 분석 제공.

### 7.4 사용자 리뷰 및 커뮤니티 불만 분석

#### 7.4.1 허위매물

KISO 부동산매물클린관리센터 공식 데이터에 따르면 허위매물 신고는 2024년 29,812건·2025년 34,528건(+15.8% YoY)으로 증가세 지속. 한국소비자원 조사에서 앱 상 매물과 실제 내용이 모두 일치한 경우는 100건 중 41건에 불과. 가장 흔한 수법은 이미 거래 완료된 매물 방치·시세보다 현저히 낮은 가격 게시 후 다른 매물 유도·대출금 차감 가격 표시·광각렌즈 활용 과장 사진이다. 블라인드에서 "네이버 빼고는 다 허위인 듯"이라는 댓글이 공감을 얻고, 클리앙에서는 "직방·다방은 '이 동네에 이런 중개사가 있구나'가 정보의 끝"이라는 평가까지 나온다.

#### 7.4.2 필터링 격차

직방에 대한 가장 빈번한 UI/UX 불만은 필터 부족. App Store 리뷰에서 "방 크기·베란다·엘리베이터·전세대출 가능 여부 같은 꼭 필터링해야 하는 중요한 조건이 없다"가 반복적으로 등장. 반면 다방은 클리앙에서 "필터 설정 유지하고 아침마다 지역 옮겨다니면서 새 매물 확인하면 됨"이라는 호평.

#### 7.4.3 MZ세대와 1인 가구가 갈증을 느끼는 정보 6가지

1. **층간소음·벽간소음** — "원룸은 옆방 소음이 더 심각한 문제인데 벽이 가벽인지 확인할 방법이 없다"(DCInside 자취갤)
2. **실제 관리비** — 전월세 신고제 회피를 위해 "월세는 낮추고 관리비를 올리는 꼼수"가 늘면서, 전기·수도·가스 포함 여부가 앱에 명확하지 않음
3. **야간 치안·동네 분위기** — "낮에만 방 보러 가니 밤 분위기를 알 수 없다"
4. **벌레(바퀴벌레 등)** — 호갱노노 후기에서 "바퀴벌레가 많다는 리뷰 보고 계약 포기" 사례
5. **집주인 평판** — 직접 공유하는 플랫폼 부재(개인정보 이슈)
6. **건물 연식별 방음 수준** — "계단식이고 한 층에 방 5개 이상이면 방음 안 되는 방일 확률 80%"(DCInside)

#### 7.4.4 호갱노노 리뷰 생태계의 신뢰성 문제

App Store 리뷰에서 "부정적 평가 남기면 타 거주민들 인신공격 댓글 10개씩 달린다", "집값 올리려고 좋은 얘기만 도배" 비판이 반복. 집값 상승을 원하는 현 거주자가 솔직한 부정적 리뷰를 억압하는 구조적 문제 존재.

### 7.5 조사의 핵심 결론

**현재 어떤 단일 앱도 MZ세대 1인 가구의 주거 탐색 여정을 완결하지 못한다.** 매물 탐색은 다방·직방, 시세 확인은 호갱노노·아실, 동네 정보는 당근마켓, 안전 확인은 안심전세앱으로 분산되어 있다.

개인 프로젝트가 노릴 수 있는 가장 큰 틈새는 세 가지다:

1. 원룸·빌라 대상 실거주 후기 집중 플랫폼
2. 커뮤니티 리뷰의 조작 방지 메커니즘
3. 여러 앱의 핵심 정보를 한 곳에 모아 비교·분석하는 메타 서비스

이 결론이 매물 등록 기능 선택의 비즈니스 근거가 되었다. "기존 공공데이터 기반 추천에 사용자 참여형 매물 데이터를 결합하는 플랫폼 확장" 서사는 틈새 1(실거주 후기 집중 플랫폼)의 확장 형태로 성립한다.

## 8. 실행 계획 단계

본 프로젝트의 실행 계획은 5단계로 구성된다. 각 단계의 투입 순서와 산출물은 아래와 같다.

### 8.1 Phase 0 — 채용 시장 조사 + 실제 서비스 리뷰 조사

**채용 시장 필수 기술 조사**: 2절에서 식별한 누락 역량(동시성 제어·트랜잭션 설계·스레드 풀 분리·자료구조/알고리즘) 외에, 실제 백엔드 채용 공고에서 필수·우대로 반복 등장하는 기술 목록을 조사한다. 이를 통해 기능 도출 시 채용 시장의 실질적 수요와 일치시키는 방향성을 확보한다. 산출물: 6절의 3층 빈도 분석 결과.

**실제 서비스 리뷰 조사**: 직방·다방·네이버 부동산의 앱스토어/플레이스토어 리뷰를 수집한다. 피드백을 추출하여 기능 후보의 비즈니스 근거를 확보한다. 산출물: 7절의 7개 서비스 분석 결과.

### 8.2 Phase 1 — 기능 확정 및 설계 명세

Phase 0에서 확보한 비즈니스 근거와 누락 역량 매핑 결과를 기반으로 구현 대상 기능을 확정한다. 확정된 기능에 대해 설계 명세서 v2와 동일한 수준의 설계 문서를 작성한다. 이때 각 설계 결정마다 다음을 명시한다:

- **대안 목록**: 해당 설계 지점에서 고려 가능한 기술적 대안 전체
- **대안별 장단점**: 각 대안의 정량적/정성적 장단점
- **트레이드오프**: 선택 시 포기하는 것과 얻는 것
- **선택 근거**: 현재 비즈니스 맥락에서 해당 대안을 선택한 이유

각 설계 결정을 문서화할 때 5절의 3가지 유형(유형 1·2·3) 중 어느 것에 속하는지를 명시한다.

산출물: 기획서·요구사항 명세서·설계 명세서(매물 등록 단일 기능).

### 8.3 Phase 2 — 구현 + 성능 측정/실측

구현 → JMeter/VisualVM 기반 성능 측정 및 실측. 구현된 기능의 성능 특성을 먼저 측정하고, 측정 결과를 기반으로 병목 지점 식별 및 대안 간 비교를 수행한다. 동시성 테스트는 멀티스레드 통합 테스트로 정합성을 검증한다.

산출물: F006·F007·F008·F009 각각의 대안 실측 데이터.

### 8.4 Phase 3 — 포트폴리오 문서화

기존 포트폴리오 3페이지 구조(기획의도·아키텍처·트러블슈팅)에 맞춰 문서화한다. "문제 정의 → 대안 검토 → 트레이드오프 → 선택 → 정량 결과" 구조를 엄격히 적용한다.

산출물: 트러블슈팅 신규 4건(F006·F007·F008·F009) + 기존 6건의 재구성 버전.

### 8.5 Phase 4 — JUnit/Mockito 기반 단위 테스트

Phase 2에서 구현 및 측정이 완료된 기능에 대해 JUnit/Mockito 기반 단위 테스트를 작성한다. 테스트 코드는 설계 의도를 검증하는 구조로 작성하며, 포트폴리오에서 "테스트 수행 흔적"으로 어필하는 기반이 된다.

산출물: Service layer 단위 테스트 코드 + 테스트 수행 결과 문서.

## 9. 취업박람회 멘토 피드백 정리

취업박람회에서 멘토 2인(세션 1 이름 미상, 세션 2 최성진/카페24)으로부터 받은 피드백의 핵심. 이 피드백이 본 고도화 프로젝트 전체의 방향성 설정 근거다.

### 9.1 공통 핵심 — 설계 능력이 신입 최우선 역량

멘토 2인 공통으로 강조: **설계 능력이 신입 최우선 역량**이다. 트러블슈팅 깊이보다 논리적 설명 능력이 중요하고, 기술 선택 근거(왜 Memcached가 아닌 Redis인가, 왜 모놀리식이 아닌 MSA인가)에 대한 질문이 면접의 주류다.

### 9.2 포트폴리오 구성 원칙

- 포트폴리오 2~3개 프로젝트 적정
- 이력서 3~5페이지 최대
- 테스트 수행 흔적이 서류 통과 긍정 포인트
- 문제 해결 프로세스 강조
- 데이터의 선별적 노출 (모든 수치를 보이는 것이 아니라, 의사결정에 영향을 준 수치만)

### 9.3 AI 활용에 대한 경고

AI 활용은 기본 지식이 전제되어야 하며, 코드에 대한 이해 없이 AI만 사용하면 면접에서 탈락.

### 9.4 자격증에 대한 평가

자격증은 채용 시 실질적으로 검토하지 않는다. 자격증 취득에 투자할 시간을 프로젝트 품질 향상에 투입하는 것이 합리적이다.

### 9.5 본 고도화 프로젝트와의 연결

본 프로젝트 전체의 설계 철학인 **"비즈니스 요구사항 충돌 → 대안 분석 → 트레이드오프 → 선택 근거 → 정량 결과"** 구조는 9.1~9.2의 피드백을 직접 반영한 것이다. F007 배치-사용자 머지 정책에서 대안 4(임계값 기반 하이브리드)를 도출하여 "3개 중 1개 선택"이 아닌 "3개 한계 분석 후 4번째 합성"의 서사 구조를 만든 것도 9.1의 "논리적 설명 능력"을 극대화하기 위한 설계다.
