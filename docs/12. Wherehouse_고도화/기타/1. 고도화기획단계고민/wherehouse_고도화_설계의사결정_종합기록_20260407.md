## 1. 고도화 방향 전환

### 1.1 전환 전 상태

고도화 기능 설계 논의록(2026-04-06)에서 4차에 걸친 반복 수정을 통해 4개 쓰기 경로가 도출되었다: 리뷰 작성(기존), 찜하기+인기알림(신규), 리뷰 투표(신규), 매물 등록/수정/상태변경(신규). 그러나 다음 개발 방향이 명확히 서지 않는 상태였다.

### 1.2 혼란의 근본 원인 진단

논의록의 기능 도출 과정 자체는 논리적으로 견고했으나, 최종 결론이 "4개 쓰기 경로 확정"으로 끝나면서 "이것을 구현하면 포트폴리오에 정확히 어떤 트러블슈팅 케이스가 몇 건 생기는가"가 구체적으로 확정되지 않은 채 끝났다. 기능 목록은 나왔는데, 그 기능이 포트폴리오 문서의 어떤 페이지에 어떤 구조로 들어가는지가 불명확했다.

추가로 식별된 근본 문제: 국토부 API가 월 1회만 갱신하는 데이터를 기반으로 하므로, 기반 데이터 자체가 매우 불충분하다. 결국 유저 간 상호작용 위주로 설계해야 하는데, 논의록의 기능 구성이 이 맥락을 구체적으로 반영하지 못한 상태에서 진행되었다.

### 1.3 전환 판단 — 4개 기능 전부 구현의 비현실성

4개 쓰기 경로를 전부 구현할 경우 발생하는 3가지 문제:

**구현 시간 초과**: 매물 등록(Oracle INSERT + Redis 6개 인덱스 + bounds 재계산 + 배치 충돌 머지), 찜하기(카운터 동시성 + 알림 fan-out + 스레드풀), 리뷰 투표(cascade 4단계 + 즉시/최종 일관성), 기존 리뷰 시스템 — 전부 구현·테스트·문서화하면 수개월 소요. 현실 일정과 양립 불가.

**포트폴리오 분량 비대화**: 4개 기능 × 1~2건 트러블슈팅 = 기존 6건 + 신규 5~8건 = 11~14건. 면접에서 전부 다루는 것 자체가 불가능.

**기능 간 맥락 연결 부재**: 찜하기, 투표, 매물 등록이 각각 독립적 기능처럼 나열되어 "이 서비스가 왜 이 기능들을 전부 필요로 하는가"에 대한 통합 비즈니스 서사가 없다. 면접관 입장에서 "기술 데모를 위해 기능을 끼워넣은 것"으로 보일 위험.

### 1.4 확정된 방향

**1개의 핵심 기능을 선택하고 그 기능 하나에서 발생하는 기술적 의사결정 3~4건을 깊게 파는 것**이 시간 대비 포트폴리오 가치가 훨씬 높다.

## 2. 매물 등록/수정/상태변경 선택 근거

### 2.1 선택 이유 — 기술적 관점

논의록에서 확정된 4개 쓰기 경로 중 매물 등록/수정/상태변경이 포트폴리오 가치가 가장 높다고 판단한 이유:

**기술적 의사결정의 다양성과 기존 시스템 관통성이 가장 높다.** 매물 등록은 Oracle INSERT + Redis Hash + Sorted Set 5~6개 + bounds 재계산 + 배치 충돌 머지를 전부 건드린다. 기존 CQRS 아키텍처, Write-Through 동기화, Redis Pipeline, CompletableFuture 등 이미 포트폴리오에 있는 기술 스택 위에서 새로운 의사결정이 추가되는 구조. 완전히 새로운 기능을 붙이는 것이 아니라, 기존 시스템의 확장으로서 자연스럽다.

### 2.2 선택 이유 — 비즈니스 관점

논의록 2.6절 조사 결론에서 "원룸·빌라 대상 실거주 후기 집중 플랫폼"이 개인 프로젝트가 노릴 수 있는 틈새로 식별됐다. 사용자가 직접 매물 정보를 등록하는 것은 피터팬(직거래 플랫폼)의 핵심 모델이고, 다방의 동네이야기 커뮤니티에서 매물 첨부 기능으로 이미 구현되어 있다. "기존 공공데이터 기반 추천에 사용자 참여형 매물 데이터를 결합하는 플랫폼 확장"이라는 비즈니스 서사가 성립한다.

### 2.3 선택 이유 — 정범진 개인 판단

정범진은 매물 등록이 주거지 추천 서비스에서 가장 직관적으로 인식되는 기능이며, 부동산 관련 비즈니스 로직에서 직관적으로 이해되어 기획이 용이하다고 판단했다. 도메인 경험이 부재한 상황에서, 본인이 가장 자연스럽게 사고할 수 있는 기능을 선택하는 것이 설계 품질에 유리하다는 판단.

### 2.4 기각된 기능 및 기각 사유

| 기능 | 기각 사유 |
|------|---------|
| 찜하기 + 인기 알림 | 매물 등록 단일 심화 방향으로 전환. 추후 여유 시 추가 가능. |
| 리뷰 투표 | 동일하게 제외. |
| 허위매물 신고 | 찜하기와 기술 패턴 동일(Redis 원자적 카운터 + 임계값 판단). 새로운 의사결정 미생성. (논의록에서 기각) |
| 가격 변동 알림 (배치 트리거) | 이벤트 발생원이 배치(월 1회)이므로 사용자 간 상호작용 맥락 부재. (논의록에서 기각) |
| 양방향 상호작용 (가격 제안/리뷰 반박/공동 관심 그룹) | 프론트엔드 과투자, 백엔드 1인 프로젝트 바운더리 초과, 비즈니스 근거 약함. (논의록에서 기각) |

### 2.5 도메인 경험 부재에 대한 판단

정범진은 기획자가 아니고 실제 부동산 매물을 사본 적도 없어서, 비즈니스 맥락을 사고하기에 범위와 경험이 제한적임을 인식했다. 이에 대한 판단: 포트폴리오의 비즈니스 로직은 실제 서비스 수준의 도메인 완성도가 필요한 것이 아니라, 기술적 의사결정이 자연스럽게 발생하는 최소한의 비즈니스 시나리오만 있으면 충분하다.

## 3. 기술적 의사결정 4건 식별 및 포트폴리오 구조 설계

### 3.1 매물 등록 단일 기능에서 도출되는 4건

| # | 의사결정 | 의사결정 유형 | 대안 수 | 기존 포트폴리오 연결 |
|---|---------|------------|--------|-------------------|
| 1 | 동시 등록 중복 감지 락 전략 | 유형 3 (실측 기반) | 3개 (Oracle PK 예외 / SELECT FOR UPDATE / Redis SETNX) | ChatService Semaphore 4종 비교의 DB 레벨 확장 |
| 2 | 배치-사용자 데이터 머지 정책 | 유형 2 (설계 단계 사전) | 3개 → 4번째 합성 전략 도출 | 기존에 없는 유형 — 신규 |
| 3 | Redis 다중 인덱스 부분 실패 전략 | 유형 1 (문제 발생 후 해결) | 3개 (전체 동기 / 비동기 재시도 / Pipeline+보상) | Redis Pipeline #6의 확장 |
| 4 | bounds 변경 점수 재계산 병렬 처리 | 유형 3 (실측 기반) | 2개 (ThreadPoolExecutor vs ForkJoinPool) | 외부 API 병렬 #1과 비교 서술 |

### 3.2 포트폴리오 구조 변환 효과

기존 6건(측정→개선 5건 + 트레이드오프 1건) + 신규 4건(트레이드오프 기반 선택 4건) = 총 10건. 트레이드오프 기반 선택이 5건으로 과반을 넘기는 구조로 전환.

의사결정 유형 분포 변환:
- 유형 1(문제 발생 후 해결): 5건 → 6건
- 유형 2(설계 단계 사전 의사결정): 0건 → 1건
- 유형 3(실측 기반 의사결정): 1건 → 3건

### 3.3 찜하기/투표 없이 충분한 이유

4건은 전부 매물 등록이라는 하나의 비즈니스 행위에서 자연스럽게 파생된다. 면접에서 각각 3분씩 = 12분. 기존 6건 중 주요 2~3건 + 신규 4건으로 30분 기술 면접 커버. 찜하기는 추후 여유 시 추가 가능하나, 매물 등록만으로 포트폴리오 기술 요건 충족.

### 3.4 기획서에서의 기술적 의사결정 표현 수위에 대한 판단

기획서 초안에 "기술적 의사결정 발생 지점" 섹션을 상세하게(대안 나열, 유형 분류, 포트폴리오 연결) 기술했으나, 기획서의 역할이 "무엇을 왜 만드는가"이지 "어떤 기술적 선택이 발생하는가"가 아니라는 판단에 따라 "예상되는 기술적 도전 과제" 수준의 간략한 목록(1~2줄씩)으로 축소했다. 상세 내용은 세션 기록 문서에 원본 보존.

## 4. 비즈니스 규칙 설계 과정

### 4.1 규칙 5(배치-사용자 데이터 공존) — 기획서에서 전략을 확정하면 안 되는 이유

기획서 초안에서 규칙 5를 "국토부 공공데이터를 권위 있는 소스로 취급하여 가격은 국토부 데이터로 갱신한다"로 기술했다. 이것은 의사결정 2(데이터 소스 간 머지 전략)의 3개 대안 중 Authority Override를 기획서 단계에서 확정해버린 것이다.

**판단**: 기획서에서는 "충돌이 발생한다"는 문제와 "해소 정책이 필요하다"는 요구사항만 기술하고, 구체적 전략 선택은 구현 과정에서 대안 비교 후 결정해야 한다. 이 머지 정책 선택 자체가 포트폴리오 트러블슈팅(유형 2 — 설계 단계 사전 의사결정)으로 문서화될 건이므로, 기획 단계에서 확정하면 의사결정 과정이 소멸한다.

**수정 결과**: "이 충돌을 어떤 정책으로 해소할 것인가는 비즈니스 정책 결정 사항이다."로 변경.

### 4.2 기획서에 "구현 과정에서 확정한다"를 쓰면 안 되는 이유

규칙 5 수정 과정에서 "구현 과정에서 대안 비교를 통해 확정한다"라는 문구가 포함되었다. 정범진이 지적: 기획서는 요구사항과 제약조건을 기술하는 문서이지, 내부 작업 계획을 기술하는 문서가 아니다. "구현 과정에서 확정한다"는 작업 프로세스에 대한 메모이고, 기획서에 들어갈 내용이 아니다.

**수정 결과**: 해당 문구 제거, 요구사항 수준("비즈니스 정책 결정 사항이다")으로만 마무리.

### 4.3 규칙 6(추천 결과 즉시 반영) — 구현 키워드 제거

기획서 초안에서 "Oracle 커밋 후 Redis 인덱스 갱신이 동기적으로 또는 최소 지연으로 수행되어야 한다"로 기술했다. "Oracle", "Redis", "인덱스 갱신"은 구현 레벨 용어이므로 기획서에 부적절하다.

**수정 결과**: "등록 직후 동일 사용자 또는 다른 사용자가 추천 검색을 수행했을 때 등록된 매물이 결과에 포함되어야 한다"로 변경. 비즈니스 요구사항만 서술.

### 4.4 기획서 전반의 구현 키워드 정리 원칙

기획서 전체에서 다음 유형의 구현 키워드를 제거했다: 영어 상태 코드(ACTIVE/COMPLETED/DELETED → 활성/거래완료/삭제), 컬럼명(DATA_SOURCE, REGISTERED_BY, USER_PRICE), 어노테이션명(@AuthenticationPrincipal), 테이블 컬럼 참조(SGG_CD, JIBUN, APT_NM, FLOOR, EXCLU_USE_AR → 시군구코드, 지번, 아파트명, 층, 전용면적), Redis 키 패턴, Oracle/Redis 직접 명시.

**판단 근거**: 기획서는 GitHub 리포지토리 상단에 노출되는 문서이므로, 비즈니스 요구사항 수준의 서술이 적절하다. 구현 세부사항은 설계 명세서에서 다룬다.

## 5. UI/UX 설계 판단

### 5.1 "내 등록 매물 관리 화면" 폐기 및 대체

기획서 초안에 개인별 매물 관리 페이지(마이페이지 개념)를 포함했다. 정범진이 대안을 제시: 별도의 개인 관리 페이지가 아니라, 페이지네이션 형태의 매물 전체 목록 게시판 페이지를 신규 개발하고, 본인 매물에만 수정/거래완료 버튼을 표시하는 구조.

**채택 이유**:
- 기존 리뷰 게시판(ReviewBoardViewController)과 동일한 UI 패턴으로 일관성 유지
- 마이페이지 개념 도입 시 프론트엔드 작업량 증가, 백엔드 1인 프로젝트 바운더리 초과 위험
- 게시판 형태에서 본인 매물에만 버튼이 보이는 구조는 기존 리뷰 게시판에서 본인 리뷰에만 수정/삭제 버튼이 보이는 것과 동일한 패턴

**결과**: "내 등록 매물 관리 화면" 섹션 삭제, "매물 전체 목록 페이지 (신규)" 섹션 추가.

### 5.2 수정/상태변경 버튼의 위치 구체화

초안에서 "각 매물에 대해 수정 버튼을 제공한다"로만 기술되어 위치가 불명확했다. 정범진이 지시: "기존 주거지 추천 서비스 결과 화면 내 각 매물에 대해"로 위치를 명시.

### 5.3 매물 상태 표시에서 동시성 시나리오 강조

기획서 초안에서 거래완료 매물 접근 시 안내 표시를 단순 기술했다. 정범진이 지적: "이것 자체가 동시성 영역"이다. 사용자 A가 추천 결과를 조회한 시점에는 활성이었으나, 클릭 시점에는 등록자 B가 이미 거래완료로 변경한 상태일 수 있다. 조회 시점과 접근 시점 사이의 상태 불일치.

**처리 방식**: 기획서에서는 사용자 경험 관점으로 강조("사용자가 혼란 없이 다른 매물을 탐색할 수 있도록 유도"), 기술적 도전 과제에 "조회 시점과 상태 변경 시점의 불일치" 항목 추가. 동시성 해결 전략 자체는 설계/구현 단계로 위임.

## 6. 권한 모델 설계 판단

### 6.1 검토한 대안

**대안 A: 모든 로그인 사용자가 모든 매물을 수정 가능.**
정범진이 제안한 근거: 포트폴리오 맥락에서 다양한 비즈니스 문제(동일 매물에 대한 동시 수정 Lost Update 등) 발생에 유리할 것.

**대안 B: 등록자 본인만 수정 가능 (현행).**

### 6.2 대안 A 기각 사유

**테스트 제약 관점**: 실질적 제약이 없다. 등록자 본인 제한이더라도 동일 계정의 동시 요청 N건으로 동시성 시나리오 재현 가능. 모든 사용자 수정을 열어야만 얻을 수 있는 고유한 동시성 시나리오가 사실상 없다.

**심사자 관점 (결정적)**: "누구나 아무 매물이나 수정 가능"은 부동산 서비스에서 가장 기본적인 비즈니스 상식에 반한다. 면접관이 "왜 권한 제어가 없는가"를 반드시 묻고, "동시성 테스트를 위해 열었다"고 답하면 취업박람회 멘토가 경고한 "기술을 위한 기술" 구조가 정확히 드러난다.

### 6.3 확정된 판단

등록자 본인만 수정/상태변경 가능을 유지. 동시성 시나리오는 동시 등록(F006, 중복 감지)에서 이미 충분히 확보되어 있고, 권한 제어 자체가 비즈니스 설계 역량을 보여주는 항목이 된다.

## 7. 구매 요청 시스템 검토 및 기각

### 7.1 제안 내용

정범진이 F003(매물 상태 변경)의 비즈니스 복잡도를 높이기 위해 제안: 여러 사용자가 구매 요청 버튼으로 금액을 설정하여 요청하고, 등록자 UI에서 요청 목록을 확인하여 수락하면 거래완료로 반영되는 구조.

### 7.2 분석 결과

**기존 트러블슈팅 4건과의 중복 여부**: 구매 요청 INSERT는 각 사용자가 서로 다른 row를 생성하므로 동시성 충돌이 발생하지 않는다. F006(동일 매물 동시 등록)의 "같은 row에 대한 Race Condition"과 질적으로 다르다. 소유자 수락 시 매물 상태 변경은 기존 F003 → F008 → F009 체인을 트리거하므로 기존 맥락과 동일.

**새로운 기술적 의사결정 발생 여부**: "수락 1건 → 다른 모든 요청 일괄 거절"의 트랜잭션 경계가 후보이나, F008(다중 키 갱신의 원자성)과 패턴이 유사("1개 행위 → N개 대상 영향의 일괄 처리 트랜잭션 경계"). 새로운 유형의 의사결정이 아닌 기존 패턴의 반복.

**구현 비용**: 구매 요청 테이블, 요청 API(생성/조회/취소), 수락/거절 API, 소유자 UI 요청 목록 표시 등 프론트엔드 작업량이 상당히 추가. 논의록 3.5절에서 "가격 제안(네고)"이 동일 패턴으로 검토·기각된 이유와 동일: 프론트엔드 상태 전이 UI 필수, 백엔드 1인 프로젝트 바운더리 초과.

### 7.3 기각 판단

F003의 포트폴리오 가치는 F003 단독이 아니라, F003이 트리거하는 F008(검색 인덱스 제거) + F009(bounds 변경 시 점수 재계산) 체인에서 나온다. 구매 요청 시스템은 기존 4건과 질적으로 다른 새로운 트러블슈팅을 만들지 않으면서 구현 시간을 상당히 소모한다.

## 8. F005 추천 결과 화면 — 제약사항 정밀화

### 8.1 초안의 문제

초안 제약사항: "기존 추천 로직(RecommendationService)의 내부 동작은 변경하지 않는다. 결과 표시 계층에서만 정보를 추가한다."

정범진이 지적: 요구사항 1~3(데이터 출처 뱃지, 본인 매물 수정 버튼, 거래완료 안내)을 구현하려면 요청자가 누구인지 인증 정보로 구분해야 하며, 반환 값 유형도 추가되어야 한다. "내부 동작은 변경하지 않는다"가 너무 포괄적이다.

### 8.2 확정된 제약사항

기존 추천 로직의 핵심 비즈니스 로직(검색 필터링, 점수 계산, 정렬)은 변경하지 않는다. 단, 요청 시 인증 정보를 통한 요청자 식별이 필요하므로 요청 정보 포맷에 대한 수정이 수반되며, 응답 정보에는 데이터 출처, 매물 상태, 등록자 본인 여부 등의 필드가 추가되어야 한다.

**구분의 핵심**: "변경하지 않는 것"(검색/점수/정렬의 핵심 로직)과 "변경해야 하는 것"(요청 포맷, 응답 필드)을 명시적으로 분리.

## 9. F006 사용자 매물 등록 시 중복 감지 — 시나리오 확장

### 9.1 초안 상태

초안에서는 2가지 시나리오만 상정: 정상 등록(중복 없음), 동시 등록(Race Condition).

### 9.2 누락 식별

정범진이 지적: 이미 존재하는 배치 매물과 동일한 MD5로 사용자가 등록하는 케이스, 이미 존재하는 다른 사용자 매물과 동일한 MD5로 등록하는 케이스가 누락되어 있다.

### 9.3 확정된 4가지 시나리오

**2.1 정상 등록 (중복 없음)**: 동일 해시 미존재, 등록 완료.

**2.2 기존 배치 매물과 중복**: 동일 MD5가 배치 매물로 이미 존재. "이 매물은 국토교통부 실거래 데이터에 이미 등록된 매물입니다" 안내, 중복 등록 거부.

**2.3 기존 사용자 등록 매물과 중복**: 동일 MD5가 다른 사용자 매물로 이미 존재. "동일한 매물이 다른 사용자에 의해 이미 등록되어 있습니다" 안내, 중복 등록 거부.

**2.4 복수 사용자 동시 등록 (Race Condition)**: 동일 MD5로 동시 INSERT 시도. 선착순 1건만 허용, 후착 거부.

### 9.4 설계 판단 — 케이스 2.2와 2.3에서 안내 메시지를 구분하는 이유

기존 매물의 데이터 출처(배치/사용자)에 따라 사용자에게 다른 안내를 제공한다. "이미 등록된 매물입니다"라는 단일 메시지보다, 출처를 알려주는 것이 사용자 경험상 자연스럽다. 사용자는 "국토부 데이터에 있는 매물"과 "다른 사용자가 등록한 매물"을 다르게 인식한다.

## 10. F007 배치-사용자 데이터 충돌 처리 — 시나리오 분류 및 머지 전략 도출

### 10.1 시나리오 분류 체계 (3건 확정 + 1건 미확정)

**시나리오 1: 배치가 신규 매물을 추가 (충돌 없음)**
배치가 가져온 매물의 MD5 해시가 시스템에 존재하지 않는 경우. 기존 배치 동작과 동일. 정상 케이스.
- 상태: 요구사항 명세서 반영 완료 (시나리오 2.1)

**시나리오 2: 배치 도착 시 동일 MD5의 사용자 매물이 이미 존재 (시간차 충돌)**
사용자가 먼저 등록(커밋 완료) → 이후 배치가 동일 MD5 매물을 다른 가격으로 가져옴. 한쪽 데이터가 확정되어 있으므로 순차 처리 가능.
- 상태: 요구사항 명세서 반영 완료 (시나리오 2.2). 머지 전략 구체 내용은 "구현 과정에서의 비즈니스 정책 결정 사항"으로 위임

**시나리오 3: 배치 UPSERT와 사용자 INSERT가 동시 수행 (동시 쓰기)**
배치 트랜잭션 진행 중에 사용자가 같은 MD5의 매물을 INSERT. 양쪽 트랜잭션이 모두 진행 중이므로 상대방 데이터가 커밋 전 상태에서 존재 여부 확인됨.
- 현실적 발생 빈도: 배치 월 1회(새벽 4시) + 사용자 수시이므로 극히 낮음
- 포트폴리오 관점: 엣지 케이스 사전 식별 가치는 있음
- 상태: **미확정. 요구사항 명세서 미반영. 다음 세션에서 포함 여부 결정 필요.**

### 10.2 시나리오 2의 머지 전략 — 대안 4 도출 전체 과정

#### 10.2.1 출발점 — 고민 자체가 어려운 이유에 대한 자기 진단

정범진은 머지 정책을 결정하기 어려운 근본 원인을 두 가지로 진단했다.

첫째, 실제 비즈니스 상황에 노출된 것이 아니므로 사용자 요구사항이 전혀 없다. 실서비스라면 "사용자가 등록한 가격이 배치로 덮어씌워졌다"는 CS 불만이 들어오고, 그 불만이 정책 결정의 근거가 된다. 포트폴리오 프로젝트에서는 이 피드백 루프가 존재하지 않는다.

둘째, 그러한 이유로 어떤 비즈니스 정책을 결정해야 하는지 큰 방향성이 없고, 트러블슈팅 관점에서 고민해야 하는 핵심 맥락 자체가 명료하게 파악되지 않는 상태였다.

#### 10.2.2 핵심 인식 전환 — 포트폴리오의 가치는 정책 자체가 아닌 과정에 있다

면접관이 보는 것은 어떤 정책을 골랐는가가 아니라 세 가지이다:
1. 이 문제를 설계 단계에서 사전에 인식했는가
2. 가능한 대안이 복수 존재한다는 것을 인지하고 비교했는가
3. 선택 근거가 비즈니스 맥락에서 나왔는가

실사용자가 없는 상황에서 프로젝트의 물리적 제약(1인 개발, 관리자 페이지 부재, 배치 월 1회)이 선택 근거가 된다. 프로젝트 제약 자체가 비즈니스 맥락이다. 이것이 핵심 전환점.

#### 10.2.3 대안 3개 정밀 검토

동일 전제 상황(사용자 1.8억 등록 → 배치 2억 도착)에서 비교.

**대안 1 — 국토부 데이터 우선 (Authority Override)**
배치 가격(2억)으로 메인 가격 덮어씀. 사용자 가격(1.8억)은 별도 필드 보존.
- 근거: 국토부 데이터는 공식 실거래 신고 기반, 신뢰도 높음
- 장점: 구현 단순(조건문 하나), 데이터 신뢰도 일정
- 단점: 사용자가 더 최근의 정확한 가격을 알고 있어도 무조건 교체. 사용자 혼란("내 가격이 왜 바뀌었는가")
- **핵심 한계: 신뢰성 확보, 최신성 유실**

**대안 2 — 최신 데이터 우선 (Last Write Wins)**
타임스탬프 비교로 더 최근 데이터 반영.
- 근거: 가장 최신 정보가 현재 상황을 가장 잘 반영
- 장점: 직관적 규칙, 소스 편향 없음
- 단점: 사용자 허위 가격이 공식 데이터를 밀어낼 수 있음. 소스별 신뢰도 차이 미반영
- **핵심 한계: 최신성 확보, 신뢰성 유실**

**대안 3 — 충돌 감지 + 플래깅 (Conflict Detection)**
가격 차이 임계값 초과 시 "충돌" 상태로 보류, 양쪽 보존, 별도 해소 프로세스.
- 근거: 자동 판단 불가능한 불일치는 보류
- 장점: 정보 유실 없음, 정확한 시점에 결정 가능
- 단점: 구현 복잡도 최고. 충돌 상태 매물의 추천 포함/제외 결정, 해소 전 표시 가격 결정, 해소 프로세스 구성 등 후속 결정 연쇄 발생. 1인 프로젝트에서 관리자 해소 프로세스 구축은 바운더리 초과 위험
- **핵심 한계: 신뢰성+최신성 보존, 구현 복잡도 과다**

#### 10.2.4 결정적 사고 — 면접 대응 관점

정범진의 판단: "신중해야 되는 것이 실제 면접에서 의사결정을 어떻게 하냐는 질의를 받을 수 있는 중요한 지점일 수도 있을 것 같다."

핵심 인식: 단순히 3개 중 1개를 고르는 것이 아니라, 면접에서 "왜 이렇게 결정했는가"를 설명할 때 설득력 있는 구조를 만들어야 한다. 대안 1~3 각각을 단독 선택하면 "신뢰성 또는 최신성 중 하나를 포기했다"는 한계가 답변에 남는다.

#### 10.2.5 대안 4 도출 — 임계값 기반 하이브리드 (Threshold-based Hybrid)

정범진이 제안한 합성 전략: "신뢰성과 최신성을 모두 반영할 수 있도록 하는 설계 지점은 어떠한가? 구체적으로 특정 임계치(가격 10% 이하 변동 발생) 내에서는 최신 데이터를 사용하되 그 이상이라면 국토부 데이터를 따르는 것으로."

사고 구조:
- 대안 1의 강점(신뢰성) + 대안 2의 강점(최신성)을 임계값으로 분기하여 동시 확보
- 가격 차이가 작으면(정상 변동 범위) → 최신 데이터가 더 정확할 가능성 높음 → 최신성 우선
- 가격 차이가 크면(비정상 변동) → 사용자 허위 등록 또는 데이터 오류 가능성 높음 → 공식 데이터(신뢰성) 우선
- 모든 경우에 자동 확정 → 대안 3의 한계(보류 상태, 관리자 프로세스) 회피
- 1인 프로젝트 바운더리 안에서 완결

#### 10.2.6 임계값 10%의 근거

현 시점에서는 예시 수치. 구현 단계에서 국토부 API 실데이터(동일 매물의 전월 대비 가격 변동률 분포)를 기반으로 정상 변동 범위를 실측하여 확정 예정. 이 실측 과정 자체가 포트폴리오에서 "데이터 기반 임계값 결정"으로 서술 가능.

#### 10.2.7 포트폴리오 서술 구조

"3개 대안을 검토하고, 각각의 한계(신뢰성 단독/최신성 단독/구현 복잡도)를 분석한 후, 신뢰성과 최신성을 임계값으로 분기하는 4번째 합성 전략을 도출했다." 트러블슈팅 의사결정 2(유형 2 — 설계 단계 사전 의사결정)로 문서화.

면접 서사: "대안 1만 쓰면 최신성을 잃고, 대안 2만 쓰면 신뢰성을 잃는다. 대안 3은 1인 프로젝트에서 관리자 해소 프로세스가 불가능하다. 그래서 임계값을 기준으로 신뢰성과 최신성을 분기하는 합성 전략을 도출했고, 임계값은 실데이터 변동률 분포에서 실측했다."

## 11. 문서 계층 간 역할 분리 원칙

본 세션에서 확립된, 기획서 / 요구사항 명세서 / 설계 명세서 간의 역할 분리 원칙.

### 11.1 기획서의 역할

"무엇을 왜 만드는가"를 기술한다. 비즈니스 요구사항, 비즈니스 규칙, 기능 도출 근거, 기대 효과를 포함한다. 구현 키워드(컬럼명, 어노테이션, Redis 키 패턴 등)를 포함하지 않는다. 기술적 도전 과제는 간략한 목록(1~2줄) 수준으로만 언급한다. 내부 작업 계획("구현 과정에서 확정한다" 등)을 기술하지 않는다. GitHub 리포지토리 상단에 노출되는 공개 문서이므로 개인 사정(경제적 제약 등)을 포함하지 않는다.

### 11.2 요구사항 명세서의 역할

"무엇이 어떻게 동작해야 하는가"를 기술한다. 기능별 요구사항, 사용자 시나리오, 제약사항을 포함한다. 구체적 기술 선택(락 전략, 머지 정책, 스레드풀 유형 등)은 "구현 과정에서의 기술적 의사결정 사항"으로 위임한다. 입출력 데이터 모델(JSON 필드명, API 스키마)은 설계 명세서의 영역이므로 포함하지 않는다.

### 11.3 설계 명세서의 역할

"구체적으로 어떻게 구현하는가"를 기술한다. API 엔드포인트, 테이블 스키마, Redis 키 구조, DTO 정의, 처리 흐름(동기/비동기 경계), 각 의사결정 지점의 대안 비교 및 선택 근거를 포함한다. 기존 설계 명세서 v2와 동일한 수준.

## 12. 시뮬레이션 기반 동시성 문제의 설득력에 대한 판단

### 12.1 위화감의 정체

정범진이 기능 선정 과정에서 느낀 위화감의 근본 원인: 동시성 문제가 실제로 발생한 것이 아니라 JMeter 시뮬레이션으로 재현해야 하는 상황에서, "이것이 포트폴리오로서 설득력이 있는가?"라는 질문에 답이 서지 않았다.

### 12.2 판단

시뮬레이션이라도 설득력은 충분하다. 신입 포트폴리오에서 실트래픽 기반 동시성 문제를 경험한 사람은 인턴 경력자 외에는 사실상 없다. 면접관도 이것을 안다. 면접관이 보는 것은 "실제로 발생했느냐"가 아니라 "이 상황이 발생했을 때 어떤 선택지가 있고, 왜 이걸 골랐는지를 논리적으로 설명할 수 있느냐"이다. 취업박람회 멘토 2인이 공통으로 강조한 것도 정확히 이것.

핵심은 기능의 수가 아니라, 선택한 기능이 만드는 기술적 의사결정의 "진정성"이다. "기술을 위해 기능을 만들었다"가 보이면 안 되고, "이 기능의 비즈니스 맥락에서 이 문제가 자연스럽게 발생한다"가 보여야 한다.
