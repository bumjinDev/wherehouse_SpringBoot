# Wherehouse 고도화 계획서 — 비즈니스 복잡도 확장을 통한 백엔드 설계 역량 증명

## 프로젝트 개요

**목표**: 현재 "조회 전용" 구조에 고착된 Wherehouse 서비스에 동시 쓰기 충돌·트랜잭션 격리·락 전략 선택이 자연스럽게 요구되는 비즈니스 기능을 도입하여, 포트폴리오의 엔지니어링 의사결정 밀도를 구조적으로 전환한다.  
**핵심 질문**: "성능 측정/개선을 넘어, 비즈니스 요구사항 간 충돌 상황에서 대안 간 트레이드오프를 분석하고 선택 근거를 명시하는 엔지니어링 의사결정을 포트폴리오에 어떻게 구조적으로 내재시킬 것인가?"  
**접근 방식**: 실제 부동산 서비스를 참고하여 기능 맥락을 도출하고, 도출된 각 기능이 누락 역량(동시성 제어, 트랜잭션 설계, 스레드 풀 분리, 자료구조 비교 선택)을 자연스럽게 요구하는 구조로 설계한 후, 구현·테스트·포트폴리오 문서화까지 수행한다.

## 현재 상태 분석

### 기존의 구체적 문제점

| 구현 부분 | 현재 로직 | 문제점 | 비즈니스 영향 |
|-----------|-----------|--------|---------------|
| **포트폴리오 트러블슈팅 구조** | 6건 중 5건이 "느린 것을 측정 → 빠르게 만듦 → 수치 제시" 패턴으로 구성 | 성능 측정/개선이지, 비즈니스 요구사항 간 충돌에서 대안 간 트레이드오프를 분석하고 선택 근거를 명시하는 엔지니어링 의사결정이 아님. IN절 Chunking 1건만이 V$SQL 실행계획 분석 → Chunk 크기별 비교 → 트레이드오프 기반 선택 구조를 갖춤 | 면접관이 "설계 역량"이 아닌 "측정 역량"으로 평가할 위험. 취업박람회 멘토 2인이 공통으로 지적한 "설계 능력 = 신입 최우선 역량" 기준에 미달 |
| **서비스 비즈니스 복잡도** | Redis 읽기 → 점수 계산 → 정렬 → 반환의 단방향 조회 파이프라인. 설계 명세서 v2의 리뷰 시스템은 단순 CRUD + Write-Through 동기화 수준 | 동시 쓰기 충돌, 트랜잭션 간 격리 문제, 락 전략 선택의 필요가 발생하지 않는 구조. 동시성 제어·트랜잭션 설계·스레드 풀 분리 등 핵심 백엔드 역량을 적용할 비즈니스 시나리오가 부재 | 포트폴리오에서 증명할 수 있는 설계 의사결정의 종류 자체가 구조적으로 제한됨 |
| **기술 적용의 깊이** | CompletableFuture + ThreadPoolExecutor는 I/O 바운드 외부 API 호출에만 적용. Redis Sorted Set, HashMap 등은 단순 사용 수준. Semaphore는 ChatService 입장 제어 단일 케이스에 한정 | 워크로드 특성별 스레드 풀 분리, 비즈니스 요구사항의 시간/공간복잡도 제약에 따른 자료구조 비교 선택, 낙관적/비관적 락의 시나리오별 구분 적용 등 의사결정 과정이 없음 | "기술을 사용했다"는 보여주지만, "왜 이 기술이고 대안은 무엇이었으며 무엇을 포기했는가"를 보여주지 못함 |

### 기존의 기술적 한계점

**동시성 제어 영역**
- synchronized와 ReentrantLock이 비즈니스 로직 맥락에서 기능별로 구분 적용된 적 없음
- DB 수준 락 기술을 학습 및 비즈니스에 반영하지 않았음
- ChatService의 Semaphore 적용은 입장 제어 단일 케이스에 한정되어, "왜 Semaphore인가"에 대한 4종 비교 실측은 존재하나 비즈니스 시나리오 다양성이 부족

**트랜잭션 설계 영역**
- 트랜잭션 경계 설정, 전파 전략(Propagation), 격리 수준(Isolation Level) 선택이 비즈니스 정합성 요구사항에 따라 의도적으로 결정된 적 없음
- 설계 명세서 v2의 ReviewWriteService는 @Transactional 단일 적용 수준이며, 배치 트랜잭션과 실시간 서비스 트랜잭션 간 격리 전략이 "Overwrite 방식" 서술에 그침

**스레드 풀 설계 영역**
- CompletableFuture + ThreadPoolExecutor는 I/O 바운드 외부 API 호출에만 적용
- CPU 바운드 연산에 ForkJoinPool을 적용하거나, 워크로드 특성에 따라 스레드 풀을 분리 설계한 적 없음
- 스레드 풀 크기 결정의 정량적 근거(코어 수, I/O 대기 비율 등)가 부재

**자료구조/알고리즘 영역**
- Redis Sorted Set, HashMap 등을 "사용"했으나, 비즈니스 요구사항의 시간복잡도/공간복잡도 제약에 따라 자료구조를 비교 선택한 의사결정 과정이 없음
- 알고리즘이 비즈니스 로직에 반영된 적 없음 (추천 점수 산출은 가중 합산 수준)

### 개선 목표

**1차 목표: 채용 시장 요구 역량을 비즈니스 기능에 내재화**
- 현재 백엔드 채용 공고에서 필수/우대로 반복 등장하는 기술 역량(DB 락, 동시성 제어, 트랜잭션 설계, 스레드 풀 분리, 자료구조/알고리즘 비교 선택 등)을 프로젝트 진행 과정에서 조사하고, 조사 결과를 기반으로 포트폴리오에 자연스럽게 녹여낼 수 있는 비즈니스 기능을 도출한다
- 기술 역량 목록은 사전에 고정하지 않고, 프로젝트 초기(Phase 0)에서 실제 채용 공고 조사를 통해 확정한다. 이를 통해 "학습을 위해 기능을 만든" 구조가 아니라, "채용 시장이 요구하는 역량이 비즈니스 기능에 의해 자연스럽게 요구되는" 구조를 확보한다

**2차 목표: 포트폴리오 구조 전환**
- 기존 "측정 → 개선 → 수치" 패턴에서 "비즈니스 요구사항 충돌 → 대안 분석 → 트레이드오프 → 선택 근거 → 정량 결과" 패턴으로 트러블슈팅 서술 구조를 전환한다
- 각 설계 결정마다 "왜 이 방식인가 / 대안은 무엇이었나 / 트레이드오프는 무엇인가"를 명시한다
- JUnit/Mockito 기반 테스트 코드로 설계 의도의 검증 가능성을 확보한다

## 새로운 비즈니스 로직 설계

### 기능 도출 방법론: 실제 서비스 리뷰 기반 비즈니스 근거 확보

**Step 1: 실제 서비스 조사**

직방, 다방, 네이버 부동산, 카카오맵 부동산, 피터팬의 좋은방 구하기 등에서 사용자 리뷰(앱스토어/플레이스토어 후기, 커뮤니티 불만글)를 수집한다. "이런 기능이 없어서 불편하다" 류의 피드백을 도출한다.

**결과 판단 기준**: 추출된 피드백으로부터 도출한 기능이, 채용 시장에서 요구하는 기술 역량을 서류 통과에 지장이 없는 수준으로 포트폴리오에 온전하게 반영할 수 있는 구조인지 여부

**적합 케이스 처리**

기능 후보를 다음 기준으로 필터링한다:
- 해당 기능이 누락 역량 중 어떤 것을 자연스럽게 요구하는가?
- 개인 프로젝트 바운더리 내에서 구현 가능한가?
- **테스트 유의성이 확보되는 데이터 규모인가?** 단순히 데이터를 "확보할 수 있는가"가 아니라, 확보된 데이터 규모에서 해당 기능이 요구하는 기술적 현상(락 경합, 트랜잭션 충돌, 스레드 풀 포화 등)이 실제로 관측 가능한가를 판단한다. 기존 피드백에서 현재 데이터 규모가 유의미한 테스트 결과를 도출하기에 충분한지 자체가 검증되지 않았다는 지적이 있었으므로, Phase 0에서 현재 시스템의 실제 데이터 규모(테이블별 건수, 커넥션 풀 크기, 자치구당 매물 분포 등)를 정확히 측정하고, 도입 대상 기능별로 요구되는 최소 데이터 규모와 대조하여 시드 데이터 보충 여부를 확정한다.

**부적합 케이스 처리**

기술을 위한 기술이 되는 기능, 인프라 과투자(Kafka, K8s 등)가 필요한 기능, 프론트엔드 과투자가 필요한 기능은 기각한다.

**Step 2: 기능 확정 및 요구사항 명세**

필터링을 통과한 기능 후보에 대해 요구사항을 명세한다. 이때 각 요구사항이 어떤 기술적 의사결정을 요구하는지를 명시하여, 기능과 역량 간의 매핑 관계를 확보한다.

**Step 3: 설계 명세 작성**

구현 대상 기능에 대해 기존 설계 명세서 v2와 동일한 수준의 설계 문서를 작성한다.

**Step 4: 설계 의사결정 문서화**

각 기능의 설계 과정에서 발생한 기술 선택 근거, 대안 검토, 트레이드오프 분석 등 설계적 맥락은 설계 명세서와 별도의 문서로 작성한다. 이 문서가 포트폴리오 서술의 원본이 된다.

### 핵심 비즈니스 규칙

**규칙 1: 비즈니스가 기술을 끌어내는 구조**

기술 적용의 출발점은 항상 비즈니스 요구사항이다. "상호배제 락을 적용하고 싶으니 기능을 만들자"가 아니라, "이 기능의 동시 쓰기 패턴에서 정합성을 보장하려면 상호배제 락이 필요하다"의 방향이어야 한다. 기술 선택의 근거가 비즈니스 시나리오에서 자연스럽게 도출되지 않으면 면접관이 "기술을 위한 기술"로 판단한다.

**규칙 2: 실제 서비스 리뷰 기반 기능 도출 근거 확보**

자체 상상이 아닌 실제 서비스의 운영 맥락에서 기능 필요성의 근거를 확보한다. 직방/다방/네이버 부동산의 앱스토어 리뷰에서 피드백을 수집하여, 해당 피드백이 기능 도출의 비즈니스 근거로 사용된다. 이 과정 자체가 포트폴리오에서 "기능 도출 근거"로 어필된다. (기존 피드백에서 스스로가 고객이 되어 자체적인 지속적 피드백을 기반으로 프로젝트를 개선하라는 조언이 있었으나, 현재 서비스 구조에서는 그러한 기능적 맥락이 자연스럽게 요구되는 지점이 부재하여, 외부 실사용자 리뷰를 대체 근거로 활용한다.)

**규칙 3: 개인 프로젝트 바운더리 준수**

운영 환경(미니 PC, 다중 사용자 접속)에서 실제 동작하며, 테스트 시 유의미한 데이터 규모를 확보해야 한다. 인프라 과투자(Kafka, K8s, MSA 등), 프론트엔드 과투자, 도메인 지식 과투자 없이 백엔드 설계 역량을 증명하는 데 집중한다. 신입 포트폴리오로서 감당 가능한 범위를 초과하면 오히려 "실서비스 운영 수준이 아닌 학습용"이라는 인상을 줄 수 있다.

**규칙 4: 포트폴리오 문서화 구조 일관성**

모든 트러블슈팅/설계 결정은 "문제 정의 → 대안 검토 → 트레이드오프 → 선택 근거 → 정량 결과" 구조로 문서화한다. 기존 포트폴리오 3페이지(기획의도, 아키텍처, 트러블슈팅) 구조와 일관성을 유지하며, 취업박람회 피드백에서 강조된 "문제 해결 프로세스 강조", "데이터의 선별적 노출" 원칙을 적용한다.

## 실행 계획

### Phase 0 — 백엔드 채용 시장 필수 기술 조사 + 실제 서비스 리뷰 조사

**백엔드 채용 시장 필수 기술 목록 조사**

1.2에서 식별한 누락 역량(동시성 제어, 트랜잭션 설계, 스레드 풀 분리, 자료구조/알고리즘) 외에, 실제 백엔드 채용 공고에서 필수/우대로 반복 등장하는 기술 목록을 조사한다. 이를 통해 기능 도출 시 채용 시장의 실질적 수요와 일치시키는 방향성을 확보한다.

**실제 서비스 리뷰 조사**

직방, 다방, 네이버 부동산의 앱스토어/플레이스토어 리뷰를 수집한다. 피드백을 추출하여 기능 후보의 비즈니스 근거를 확보한다. 이 과정 자체가 포트폴리오에서 "기능 도출 근거"로 사용된다.

### Phase 1 — 기능 확정 및 설계 명세

Phase 0에서 확보한 비즈니스 근거와 누락 역량 매핑 결과를 기반으로 구현 대상 기능을 확정한다. 확정된 기능에 대해 설계 명세서 v2와 동일한 수준의 설계 문서를 작성한다. 이때 각 설계 결정마다 다음을 명시한다:

- **대안 목록**: 해당 설계 지점에서 고려 가능한 기술적 대안 전체
- **대안별 장단점**: 각 대안의 정량적/정성적 장단점
- **트레이드오프**: 선택 시 포기하는 것과 얻는 것
- **선택 근거**: 현재 비즈니스 맥락에서 해당 대안을 선택한 이유

단, 위 4개 항목의 서술 방식은 의사결정이 발생한 맥락에 따라 구분한다. 의사결정 맥락은 다음 세 가지로 분류되며, 각 유형별로 문서화 구조가 다르다:

- **유형 1 — 실제 문제 발생 후 해결**: 운영 또는 테스트 중 측정된 문제가 출발점이다. "문제 발견 → 원인 분석 → 대안 검토 → 해결 → 정량 결과" 순서로 서술한다.
- **유형 2 — 설계 단계 사전 의사결정**: 문제가 발생하기 전, 아키텍처 또는 기술 선택 수준에서 사전에 판단한다. "요구사항 분석 → 대안 비교 → 선택 근거" 순서로 서술한다.
- **유형 3 — 실측 기반 의사결정**: 이론적으로 판단이 불가능하여, 실제 환경에서 측정한 데이터로만 결정이 가능한 경우다. "가설 수립 → 실측 설계 → 측정 결과 → 데이터 기반 선택" 순서로 서술한다.

각 설계 결정을 문서화할 때 해당 결정이 어떤 유형에 속하는지를 명시하여, 면접에서 "이 결정은 사전에 한 건가, 문제가 발생해서 한 건가, 실측해서 한 건가"라는 질문에 맥락이 즉시 전달되도록 한다.

### Phase 2 — 구현 + 성능 측정/실측

구현 → JMeter/VisualVM 기반 성능 측정 및 실측. 구현된 기능의 성능 특성을 먼저 측정하고, 측정 결과를 기반으로 병목 지점 식별 및 대안 간 비교를 수행한다. 동시성 테스트는 멀티스레드 통합 테스트로 정합성을 검증한다.

### Phase 3 — 포트폴리오 문서화

기존 포트폴리오 3페이지 구조(기획의도, 아키텍처, 트러블슈팅)에 맞춰 문서화한다. "문제 정의 → 대안 검토 → 트레이드오프 → 선택 → 정량 결과" 구조를 엄격히 적용한다.

### Phase 4 — JUnit/Mockito 기반 단위 테스트

Phase 2에서 구현 및 측정이 완료된 기능에 대해 JUnit/Mockito 기반 단위 테스트를 작성한다. 테스트 코드는 설계 의도를 검증하는 구조로 작성하며, 포트폴리오에서 "테스트 수행 흔적"으로 어필하는 기반이 된다.

## 기대 효과

- **포트폴리오 설계 밀도 전환**: 기존 "측정 → 개선 → 수치" 5건 + "트레이드오프 기반 선택" 1건 구조에서, 모든 트러블슈팅이 "비즈니스 요구사항 충돌 → 대안 분석 → 트레이드오프 → 선택 근거 → 정량 결과" 구조를 갖추는 방향으로 전환
- **누락 역량 내재화**: 동시성 제어(낙관적/비관적 락 시나리오별 선택), 트랜잭션 설계(Propagation/Isolation Level 의도적 결정), 스레드 풀 분리 설계, 자료구조/알고리즘 비교 선택이 비즈니스 시나리오에 의해 자연스럽게 요구되고 적용된 포트폴리오 확보
- **면접 대응력 강화**: 각 설계 결정마다 "왜 이 방식인가 / 대안은 무엇이었나 / 트레이드오프는 무엇인가"가 명시되어 있으므로, 취업박람회 멘토가 강조한 "기술 선택 근거의 논리적 설명" 역량을 3분 이내 구술로 즉시 제시 가능
- **테스트 코드 부재 해소**: JUnit/Mockito 기반 단위 테스트가 포트폴리오에 포함되어, "테스트 수행 흔적이 존재하고 기능별 설계 방법이 간결하게 기술된" 서류 통과 포인트 충족
- **비즈니스 마인드 어필**: 기능 도출 근거가 실제 부동산 서비스의 사용자 리뷰에서 출발하므로, "기술 오타쿠"가 아닌 "서비스의 목적과 사용자 관점에서 기술을 설명하는" 페르소나 전환
