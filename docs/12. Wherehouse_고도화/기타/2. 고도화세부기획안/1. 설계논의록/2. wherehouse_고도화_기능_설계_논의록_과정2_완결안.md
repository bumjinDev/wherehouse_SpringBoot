## 1. 세션 목적 및 투입 문서 검토

### 1.1 세션 목적

전일(2026-04-06) 작성한 "고도화 계획서"와 "기능 설계 논의록"을 기반으로, 다음 개발 방향이 명확히 서지 않는 상태에서 기능 선정 최종 방향을 확정하는 것이 본 세션의 목적이다.

### 1.2 투입 문서 5건 검토 요약

세션 시작 시 5건의 문서를 전량 검토하였으며, 각 문서의 핵심 내용은 다음과 같다.

#### 문서 1: 취업박람회 상담 녹음본 (2026.03.31)

멘토 2인(세션 1 이름 미상, 세션 2 최성진/카페24) 공통 핵심: **설계 능력이 신입 최우선 역량**. 트러블슈팅 깊이보다 논리적 설명 능력이 중요하고, 기술 선택 근거(왜 Memcached가 아닌 Redis인가, 왜 모놀리식이 아닌 MSA인가)에 대한 질문이 면접의 주류. 포트폴리오 2~3개 프로젝트 적정, 이력서 3~5페이지 최대, 테스트 수행 흔적이 서류 통과 긍정 포인트. AI 활용은 기본 지식이 전제되어야 하며, 코드에 대한 이해 없이 AI만 사용하면 면접에서 탈락. 자격증은 채용 시 실질적으로 검토하지 않는다.

#### 문서 2: Wherehouse 포트폴리오 PDF (9페이지, 현행)

기획의도(비즈니스 문제 + 3 Challenge) → 시스템 아키텍처(CQRS + Hybrid Storage, 설계 결정 Why 3건) → 트러블슈팅 5건(PDF에서 확인된 것 기준):

- **#1**: 외부 API 병목 — CompletableFuture + ThreadPoolExecutor 병렬화로 72.3% 응답 시간 감소 (1,324ms → 367ms). ForkJoinPool(commonPool) 기각 사유 포함. 대안 검토표(ForkJoinPool vs CompletableFuture+ThreadPoolExecutor) 존재.
- **#3**: Oracle IN절 Chunking — N+1 → Bulk Fetch → Chunk 61, V$SQL 기반 3단계 실행계획 분석. Chunk 크기별 실측 비교표(Bulk/Chk1000/Chk22/Chk61) 7개 지표 비교. Soft Parse 88.5%, 심각 경합 0%, 총 비용 Chk22 대비 -11%. 트레이드오프 기반 선택 구조를 갖춘 유일한 건.
- **#4**: LIKE 검색 Index — '%keyword%' Full Table Scan → LIKE 'keyword%' + apt_nm 인덱스 전환. Oracle Text(Full-Text Index) 대안 기각(별도 역인덱스 구조 생성·동기화 관리 필요, 1인 프로젝트 운영 복잡도 대비 과다). 실행계획 비교(TABLE ACCESS FULL Cost:137 → INDEX RANGE SCAN Cost:5). 쿼리 소요시간 77.2% 감소, 슬로우 쿼리 103건→0건.
- **#5**: OOM Slice 기반 청크 — findAll() 일괄 로드 → Slice 페이징. Page<T> vs Slice<T> 대안 비교(Page는 COUNT(*) 쿼리 자동 실행으로 불필요한 오버헤드). 힙 피크 64.4%→17.6%, 엔티티 메모리 99.2MB→32.5MB. OOM으로 인한 배치 전체 실패 vs 처리 시간 63% 증가의 트레이드오프, 메모리 안정성 우선 판단.
- **#6**: Redis Pipeline RTT — 순차 Redis 호출 → Pipeline 일괄 전송. 지역구당 3회 RTT × 25 자치구 = 75회 네트워크 왕복을 Pipeline으로 1/3 절감. Wireshark TCP 패킷 분석으로 Layer 2 RTT 검증(순차 38.5ms 평균 vs Pipeline 39.3ms 평균, 1회 왕복 비용 증가 없음 확인). 전체 검색 시간 34.0% 감소, 지역구별 평균 응답 54.4% 감소, 네트워크 왕복 횟수 66.7% 감소.

고도화 계획서가 지적한 대로, 6건(포트폴리오 전체 기준) 중 5건이 "측정→개선→수치" 패턴이고, IN절 Chunking 1건만 트레이드오프 기반 선택 구조를 갖추고 있다.

#### 문서 3: 설계 명세서 v2 (3,292줄, 리뷰 시스템 + CQRS 전체 명세)

- **아키텍처**: CQRS 패턴 — Oracle RDB(Write Master/SSOT) + Redis(Read Cache) + Sync Layer(Event-Driven)
- **식별자 전략**: 불변 속성 5개(SGG_CD, JIBUN, APT_NM, FLOOR, EXCLU_USE_AR) 조합 MD5 해시 → 배치 실행 후에도 리뷰 FK 유지
- **테이블 설계**: PROPERTIES_CHARTER/MONTHLY(매물), REVIEWS(CLOB, Redis 미적재), REVIEW_STATISTICS(Redis 동기화), REVIEW_KEYWORDS(감성 점수 ±1)
- **Redis 구조**: Hash(매물 상세 1:1 미러링, 리뷰 통계), Sorted Set(가격/평수 범위 검색 인덱스), Hash(정규화 범위 bounds, 안전성 점수)
- **추천 로직**: Phase 1(기존 정량 점수 파이프라인 100% 재사용) → Phase 2(리뷰 통계 Pipeline 일괄 조회 → 하이브리드 점수 산출, reviewCount≥5이면 50:50, <5이면 LegacyScore 100%)
- **API 명세**: 리뷰 CRUD 4개 + 매물 검색 자동완성 1개 + 추천 결과 변경사항, 컨트롤러 5개 분리
- **DTO 구조**: Request/Response 전체 정의 완료, propertyId 타입 Long→String(MD5) 변경, reviewCount/avgRating 필드 추가
- **배치**: BatchScheduler → ID Generator(MD5) → Event Publisher → RdbSyncListener(UPSERT + Redis 동기화)
- **핵심 서비스 로직**: 추천 조회(User → RecommendationService → Redis, RDB 무접근), 리뷰 작성(User → ReviewService → RDB → Redis 동기화), 배치 갱신(BatchScheduler → 국토부 API → RDB → Redis)

#### 문서 4: 고도화 계획서 (2026-04-06, v1.0)

**핵심 문제 3가지**:
1. 포트폴리오 트러블슈팅이 "측정→개선→수치" 패턴에 편중 (6건 중 5건)
2. 비즈니스 복잡도 부재 — 조회 전용 단방향 파이프라인에 고착
3. 기술 적용 깊이 부족 — "사용했다"는 있지만 "왜 이 기술인가"가 없음

**누락 역량 4개**: 동시성 제어(DB 락/낙관적·비관적), 트랜잭션 설계(Propagation/Isolation Level 의도적 결정), 스레드풀 분리(워크로드 특성별), 자료구조/알고리즘 비교 선택.

**핵심 규칙 4가지**: 규칙 1(비즈니스가 기술을 끌어냄) → 규칙 2(실제 서비스 리뷰 기반 기능 도출) → 규칙 3(개인 프로젝트 바운더리 준수) → 규칙 4(문서화 구조 일관성: "문제 정의 → 대안 검토 → 트레이드오프 → 선택 근거 → 정량 결과").

**의사결정 문서화 유형 분류**:
- 유형 1 — 실제 문제 발생 후 해결: "문제 발견 → 원인 분석 → 대안 검토 → 해결 → 정량 결과"
- 유형 2 — 설계 단계 사전 의사결정: "요구사항 분석 → 대안 비교 → 선택 근거"
- 유형 3 — 실측 기반 의사결정: "가설 수립 → 실측 설계 → 측정 결과 → 데이터 기반 선택"

**실행 계획**: Phase 0(채용 시장 조사 + 서비스 리뷰 조사) → Phase 1(기능 확정 + 설계 명세) → Phase 2(구현 + 실측) → Phase 3(포트폴리오 문서화) → Phase 4(JUnit/Mockito 테스트).

#### 문서 5: 고도화 기능 설계 논의록 (2026-04-06, v1.0, 575줄)

**채용 시장 분석**: 1층(절대 필수 — 없으면 서류 탈락) API 설계(~95%)/DB·쿼리 최적화(~90%)/자료구조·알고리즘(~80%)/설계 역량(~70%), 2층(핵심 우대 — 있으면 서류 통과율 급상승) CI/CD·Docker(~60%)/Redis 캐싱(~50%)/테스트 코드(~40%)/메시지 큐(~25%), 3층(강력 차별화 — 면접에서 압도적 인상) 스레드풀·비동기(~20%)/모니터링(~20%)/트랜잭션 설계(~15%)/동시성 제어(~10%). 핵심 발견: 동시성 제어·트랜잭션 설계는 공고 언급률 10~15%이지만 면접 출제 빈도와 포트폴리오 차별화 효과는 최상급.

**실제 서비스 조사**: 직방/다방/네이버 부동산/피터팬/호갱노노/아실/부동산플래닛 7개 분석. 3강 플랫폼(직방 632만 설치 MAU 229만, 다방 342만 설치 MAU 103만, 네이버 부동산 MAU 134만) 핵심 기능 비교. 니치 플랫폼 4인방(피터팬 직거래 전문, 호갱노노 입주민 리뷰, 아실 투자 분석, 부동산플래닛 상업용 AVM). MZ 1인 가구 갈증 정보 6가지(층간소음, 관리비, 야간 치안, 벌레, 집주인 평판, 방음). 조사 핵심 결론: 어떤 단일 앱도 MZ세대 1인 가구의 주거 탐색 여정을 완결하지 못하며, 개인 프로젝트가 노릴 수 있는 틈새는 원룸·빌라 대상 실거주 후기 집중 플랫폼 / 커뮤니티 리뷰 조작 방지 메커니즘 / 핵심 정보 통합 비교 메타 서비스.

**기능 도출 과정 (4차에 걸친 반복 수정)**:

- **1차 제안**: 찜하기 + 가격 변동 알림 + 허위매물 신고 (3개). 허위매물 신고는 찜하기와 기술 패턴 동일(Redis 원자적 카운터 + 임계값 판단)하여 기각. 1차 확정: 찜하기 + 가격 변동 알림.
- **2차 수정**: 가격 변동 알림 기각 — 이벤트 발생원이 배치(월 1회)이므로 사용자 간 상호작용 맥락 부재. 핵심 원칙 전환: 모든 이벤트의 발생원 = 사용자 행위. 수정된 구성: 찜하기 + 인기 매물 알림(찜 기반) + 리뷰 투표.
- **3차 확장**: 매물 등록 기능 추가. 핵심 엔티티 자체가 사용자 쓰기 경로에 진입. 5개 시나리오 생성(동일 매물 동시 등록, 사용자 vs 배치 충돌, Redis 인덱스 즉시 반영, 상태 변경 cascade, 리뷰와 매물의 상호 의존성).
- **4차 심화**: 사용자 간 상호작용 중심축 전환. 상호작용 유형 3가지 식별(cascade 상호작용 / 공유 상태 경쟁적 변경 / 알림 발생). 양방향 상호작용(가격 제안/리뷰 반박/공동 관심 그룹) 검토 후 기각(프론트엔드 바운더리 초과 + 비즈니스 근거 약함).

**확정된 4개 쓰기 경로**: 리뷰 작성(기존) + 찜하기+인기알림(신규) + 리뷰 투표(신규) + 매물 등록/수정/상태변경(신규).

**CPU 바운드 맥락**: 매물 등록 → bounds 변경 → 지역구 전체 매물 점수 재계산이 순수 CPU 바운드. ForkJoinPool의 work-stealing이 지역구별 매물 수 불균일(강남구 1,000건 vs 도봉구 50건)에 적합. 기존 I/O 바운드 ThreadPoolExecutor(20스레드)와 대비되는 CPU 바운드 ForkJoinPool(코어 수=4)의 두 병렬 처리 비교 가능.

**자료구조/알고리즘 4개 지점**: Top-K 추출(sort/heap/quickselect 3종), 지역구 그룹핑(HashMap/TreeMap/one-pass 3종), 알림 병합(HashSet/HashMap/LinkedHashMap/PriorityQueue/Redis SETEX 5종), 키워드 매칭(contains/HashSet/Aho-Corasick 3종).

**누락 역량 매핑**: 동시성 제어(카운터 3종 + 중복 감지 3종), 트랜잭션 설계(cascade 경계, 배치-사용자 머지, 즉시/최종 일관성), 스레드풀 분리(알림 fan-out + 점수 재계산), CPU 바운드 병렬(ForkJoinPool), 자료구조(4지점 14종 대안), 알고리즘(Top-K + 키워드 매칭 교차점 실측). 포트폴리오 추가 트러블슈팅 예상 4~5건.

## 2. 정범진 현황 진술 — 혼란의 구조

정범진의 현황 진술 원문을 항목별로 정리한다.

### 2.1 목적 복기

포트폴리오로써 기술사항을 포트폴리오에 녹여내야 한다. 이를 위해 기존의 단순 조회 수준에 머무르는 것이 아닌, 지속해서 동적으로 데이터가 변경되는 상황과 사용자가 지속적으로 참여함으로써 사용자 간 데이터에 영향을 받는 상황, 즉 "비즈니스 로직" 관점에서 포트폴리오에 문제 해결을 반영해야 한다.

### 2.2 기능 선정이 어려운 이유 분석 (정범진 자체 분석)

어제(2026-04-06) 고민한 내용을 복기한 결과, 기능 선정이 어려운 이유는 다음과 같다:

- **찜하기 수준의 기능으로는 비즈니스 로직을 구현하기에 맥락적, 규모적, 복잡성이 부족**하여 포트폴리오 내 반영할 기술적인 맥락 자체가 존재하지 않았기 때문에 철회함.
- 나머지 기능들은 이러한 비즈니스 맥락을 충족하는지에 대한 **명확한 판단이 서지 않았음**.

### 2.3 핵심 고민 — 기능 방향의 이분법

두 가지 방향 중 선택이 필요하다:

1. **현재 상용 서비스와 동일 혹은 비슷한 기능을 구현하는 방향** — 충분한 복잡성과 운영 환경을 가정한 유저 간 데이터 영향을 줄 수 있는 기능.
2. **기존 사용자 리뷰 등을 반영해서 차별화되고 더욱 복잡하면서 동시에 유의미한 맥락의 기능을 만드는 방향** — 후자라면 포트폴리오로서 비즈니스 로직을 더욱 충족할 것.

### 2.4 근본적 데이터 한계 인식

근간이 되는 지속적인 데이터가 국토부 API 한 달에 한 번 갱신하는 것을 끌고 오는 것밖에 없어서 기반 데이터 자체가 매우 불충분하다. 결국 유저 간 상호작용 위주로 설계해야 하는데, 현재 논의된 내용 자체도 이러한 맥락을 구체적으로 덜 잡은 상태에서 진행된 것 같아서 기준점 없이 혼돈인 상태.

### 2.5 도메인 경험 부재 인식

본인이 기획자가 아니고 실제 부동산 매물을 사본 적도 없어서 이러한 맥락 자체를 사고하기에는 범위도 경험도 없다. 기능 선정을 처음부터 다시 해야 할지도 의문인 상태.

## 3. 혼란의 구조적 원인 진단

### 3.1 논의록 자체의 논리적 견고성은 유효

논의록의 기능 도출 과정 자체는 논리적으로 견고하게 진행됐다. 1차→2차→3차→4차까지의 수정 과정에서 기각 사유도 명확하고, 상호작용 유형 3가지 식별도 정교하다.

### 3.2 논의록의 미완결 지점

최종 결론이 "4개 쓰기 경로 확정"으로 끝나면서, 정작 **"이것을 구현하면 포트폴리오에 정확히 어떤 트러블슈팅 케이스가 몇 건 생기는가"가 구체적으로 확정되지 않은 채 끝났다.** 기능 목록은 나왔는데, 그 기능이 포트폴리오 문서의 어떤 페이지에 어떤 구조로 들어가는지가 불명확한 상태.

### 3.3 위화감의 정체 — 시뮬레이션 기반 동시성 문제의 설득력 문제

근본적인 문제가 존재한다. 정범진이 느낀 위화감의 정체는 다음과 같다:

"찜하기"를 예로 들면 — 찜 카운터 동시성 3종 비교(Redis INCR / Oracle row lock / JPA @Version)라는 기술적 의사결정이 존재하긴 한다. 그런데 이 의사결정이 발생하려면 "인기 매물에 다수 사용자가 동시에 찜/취소를 누르는 상황"이 전제되어야 한다. 실제로 이 서비스에 동시 접속 사용자가 없다. 시드 데이터로 JMeter 부하를 걸어서 시뮬레이션하는 수밖에 없다. 그러면 면접관이 "이 동시성 문제가 실제로 발생한 건가요, 시뮬레이션한 건가요?"라고 물었을 때 — "시뮬레이션입니다"라고 답해야 한다. 이것이 포트폴리오로서 설득력이 있는가?

이 질문에 대한 답이 서지 않으니까 기능 선정 자체가 흔들리는 것이다.

### 3.4 시뮬레이션 설득력에 대한 판단

**결론: 시뮬레이션이라도 설득력은 충분하다. 단, 조건이 있다.**

신입 포트폴리오에서 실트래픽 기반 동시성 문제를 경험한 사람은 인턴 경력자 외에는 사실상 없다. 면접관도 이걸 안다. 면접관이 보는 것은 "실제로 발생했느냐"가 아니라 **"이 상황이 발생했을 때 어떤 선택지가 있고, 왜 이걸 골랐는지를 논리적으로 설명할 수 있느냐"**이다. 취업박람회 멘토 2인이 공통으로 말한 것도 정확히 이것이다.

핵심은 기능의 수가 아니라, **선택한 기능이 만드는 기술적 의사결정의 "진정성"**이다. "기술을 위해 기능을 만들었다"가 보이면 안 되고, "이 기능의 비즈니스 맥락에서 이 문제가 자연스럽게 발생한다"가 보여야 한다.

## 4. 논의록의 진짜 문제점 — 기능 4개 전부 구현의 비현실성

논의록이 4개 쓰기 경로를 확정했는데, 4개를 전부 구현하면 다음 3가지 문제가 발생한다.

### 4.1 구현 시간 초과

매물 등록(Oracle INSERT + Redis 6개 인덱스 + bounds 재계산 + 배치 충돌 머지), 찜하기(카운터 동시성 + 알림 fan-out + 스레드풀), 리뷰 투표(cascade 4단계 + 즉시/최종 일관성), 기존 리뷰 시스템 — 4개를 전부 구현하고 테스트하고 문서화하는 것은 현실적으로 수개월이 소요된다. 부친 지원 2개월(2026년 5월 말까지), 4/28 채용박람회 일정과 양립 불가.

### 4.2 포트폴리오 분량 비대화

4개 기능에서 각각 1~2건의 트러블슈팅이 나오면 기존 6건 + 신규 5~8건 = 11~14건. 포트폴리오가 비대해지고 면접에서 전부 다루는 것 자체가 불가능.

### 4.3 기능 간 맥락 연결 부재

찜하기, 투표, 매물 등록이 각각 독립적인 기능처럼 나열되어 있고, "이 서비스가 왜 이 기능들을 전부 필요로 하는가"에 대한 하나의 통합된 비즈니스 서사가 없다. 기능 하나하나는 기술적 의사결정을 만들지만, 전체를 관통하는 이야기가 없으면 면접관 입장에서 "기술 데모를 위해 기능을 끼워넣은 것"으로 보인다.

## 5. 방향 전환 — "기능 수 확대"에서 "단일 기능 심화"로

### 5.1 방향 전환의 핵심

4개 쓰기 경로를 전부 만드는 대신, **1개의 핵심 기능을 선택하고 그 기능 하나에서 발생하는 기술적 의사결정 3~4건을 깊게 파는 것**이 시간 대비 포트폴리오 가치가 훨씬 높다.

### 5.2 매물 등록/수정/상태변경이 최적인 이유

논의록에서 확정된 4개 중, 포트폴리오 가치가 가장 높은 것은 **매물 등록/수정/상태변경**이다.

**이유 1: 기술적 의사결정의 다양성과 기존 시스템 관통성.** 매물 등록은 Oracle INSERT + Redis Hash + Sorted Set 5~6개 + bounds 재계산 + 배치 충돌 머지를 전부 건드린다. 기존 CQRS 아키텍처, Write-Through 동기화, Redis Pipeline, CompletableFuture 등 이미 포트폴리오에 있는 기술 스택 위에서 새로운 의사결정이 추가되는 구조다. 완전히 새로운 기능을 붙이는 것이 아니라, 기존 시스템의 확장으로서 자연스럽다.

**이유 2: 비즈니스 근거의 강도.** 논의록 2.6절 조사 결론에서 "원룸·빌라 대상 실거주 후기 집중 플랫폼"이 개인 프로젝트가 노릴 수 있는 틈새로 식별됐다. 사용자가 직접 매물 정보를 등록하는 것은 피터팬(직거래 플랫폼)의 핵심 모델이고, 다방의 동네이야기 커뮤니티에서 매물 첨부 기능으로 이미 구현되어 있다. "기존 공공데이터 기반 추천에 사용자 참여형 매물 데이터를 결합하는 플랫폼 확장"이라는 서사가 성립한다.

## 6. 정범진 판단 — 매물 등록/수정/상태변경 선택 근거 및 제약

### 6.1 선택 근거 (정범진 진술)

매물 등록/수정/상태변경을 선택하고 싶다. 이유는 다음과 같다:

1. **서비스 직관성**: 주거지 추천 서비스에서 매물 자체에 직접 접근하는 것이 가장 직관적으로 인식된다.
2. **도메인 이해 용이성**: 개인적으로 부동산 관련 비즈니스 로직에서 직관적으로 이해되어 기획이 용이하다.
3. **동적 비즈니스 로직 맥락**: 다른 사람들 입장에서 리뷰 등보다도 매물 자체에 직접 접근하는 것이 가장 관심도 있게, 동적으로 비즈니스 로직 맥락에서도 구현이 용이할 것으로 판단된다.

### 6.2 미해소 불확실성 (정범진 진술)

1. **비즈니스 맥락 충분성**: 매물 등록 내부에 비즈니스 맥락을 충분히 그리고 풍부하게 구현 가능해서 포트폴리오 기술적 요건을 충족시키는지에 대해 아직 선명하게 논의되지 않았다.
2. **도메인 경험 부재**: 본인이 기획자가 아니고 실제 부동산 매물을 사본 적도 없어서, 확인하고 이해할 수 있는 상용 부동산 시스템 기능의 범위가 제한적이다. 이러한 맥락 자체를 사고하기에는 범위도 경험도 없다.

### 6.3 도메인 경험 부재에 대한 판단

도메인 경험이 없는 것은 문제가 아니다. 포트폴리오의 비즈니스 로직은 실제 서비스 수준의 도메인 완성도가 필요한 것이 아니라, **기술적 의사결정이 자연스럽게 발생하는 최소한의 비즈니스 시나리오**만 있으면 된다.

## 7. 매물 등록 단일 기능에서 도출되는 4건의 기술적 의사결정 상세 전개

사용자 플로우 한 줄 압축: **사용자가 "서울시 마포구 래미안 10층 59㎡ 전세 1.8억"을 등록한다 → 이 매물이 추천 결과에 즉시 반영된다 → 등록자가 가격을 2억으로 수정한다 → 추천 결과의 점수가 바뀐다 → 등록자가 "거래완료"를 누른다 → 추천 결과에서 사라진다.**

### 7.1 의사결정 1: 동일 매물 동시 등록 — 중복 감지 락 전략

**시나리오**: 사용자 A와 B가 같은 물리적 매물(같은 아파트, 같은 층, 같은 면적)을 거의 동시에 등록한다. MD5 해시가 동일하게 생성된다. 둘 다 SELECT로 "이 해시 존재하는가?" 확인 → 둘 다 "없음" → 둘 다 INSERT 시도 → Race Condition.

**대안 3개**:

1. **Oracle PK 예외 catch**: Oracle UNIQUE 제약조건에 의존하고 DuplicateKeyException을 catch해서 "이미 등록된 매물입니다"를 반환. 구현이 가장 단순하지만, 예외 기반 흐름 제어는 비용이 크고 사용자 B가 입력한 데이터(가격 등)가 유실된다.
2. **SELECT FOR UPDATE 비관적 락**: 동일 해시에 대한 두 번째 트랜잭션이 첫 번째 커밋까지 대기. 정합성은 완벽하지만 락 대기 시간이 발생하고, 신규 INSERT에 SELECT FOR UPDATE를 거는 것은 해당 row가 아직 존재하지 않으므로 gap lock이 필요하다(Oracle에서는 다르게 동작).
3. **Redis SETNX 분산 락**: `SETNX lock:property:{md5_hash} {user_id} EX 30`으로 30초간 해당 해시의 등록 권한을 선점. 선점 실패 시 즉시 "다른 사용자가 등록 중입니다"를 반환. Oracle 레벨 락보다 가볍고 사용자 경험이 좋지만, Redis 장애 시 폴백이 필요하다.

**실측 구조**: JMeter로 동시 요청 N건을 쏘고, 각 전략별 성공률/응답 시간/락 대기 시간을 실측해서 비교.

**기존 포트폴리오 연결**: ChatService의 Semaphore 4종 비교와 동일한 패턴(동일 문제에 대한 N개 대안 실측 비교 → 트레이드오프 기반 선택)이되, **DB 레벨 락이라는 새로운 기술 영역**으로 확장. 고도화 계획서에서 "DB 수준 락 기술을 학습 및 비즈니스에 반영하지 않았음"이라고 지적한 누락 역량이 여기서 채워진다.

**의사결정 문서화 유형**: 유형 3 (실측 기반 의사결정)

**포트폴리오 1건 확정.**

### 7.2 의사결정 2: 사용자 매물 vs 배치 매물 충돌 — 데이터 소스 간 머지 전략

**시나리오**: 사용자가 "전세 1.8억"으로 등록. 다음 달 배치가 국토부 API에서 같은 매물을 "전세 2억"으로 가져온다. 동일 MD5 해시. 누구의 데이터를 신뢰할 것인가.

**대안 3개**:

1. **국토부 데이터 우선 (Authority Override)**: 공공 데이터가 공식 실거래 신고 기반이므로 항상 우선한다. 사용자 등록 가격은 "사용자 제시 가격"으로 별도 컬럼에 보존하고, 메인 가격은 국토부 데이터로 덮어쓴다.
2. **최신 데이터 우선 (Last Write Wins)**: LAST_UPDATED 타임스탬프가 더 최근인 데이터를 반영. 단, 사용자가 허위 가격을 등록할 경우 공식 데이터가 밀린다.
3. **수동 병합 (Conflict Detection + 관리자 결정)**: 가격 차이가 임계값(예: 10%) 이상이면 CONFLICT 상태로 플래깅하고, 양쪽 데이터를 모두 보존한 뒤 별도 해소 프로세스를 둔다.

**핵심 특성**: 이것은 코드 성능 문제가 아니라 **비즈니스 정책 결정**이다. 대안별 장단점을 분석하고 비즈니스 맥락에 따라 선택 근거를 명시하는 구조.

**기존 포트폴리오 연결**: 기존 포트폴리오에 이 유형(설계 단계 사전 의사결정)이 없다. 신규 유형 추가.

**의사결정 문서화 유형**: 유형 2 (설계 단계 사전 의사결정)

**포트폴리오 1건 확정.**

### 7.3 의사결정 3: Oracle 커밋 후 Redis 다중 인덱스 동기화 — 부분 실패 전략

**시나리오**: 매물 등록 시 Redis에 갱신해야 하는 대상: Hash 1개(`property:charter:{id}`) + Sorted Set 최소 2개(`idx:charterPrice:{district}`, `idx:area:{district}:전세`) + bounds Hash 1개(`bounds:{district}:전세`). 기존 리뷰 Write-Through는 Hash 1개(`stats:{id}`)만 갱신하면 됐다. 매물 등록은 **4개 이상의 Redis 키를 동시에 갱신**해야 한다.

Oracle INSERT는 성공했는데 Redis Sorted Set ZADD 중간에 실패하면 — 매물은 DB에 존재하지만 추천 검색에서는 안 나오는 불일치 상태.

**대안 3개**:

1. **전체 동기 처리**: Oracle INSERT + Redis 6개 연산을 하나의 서비스 메서드 안에서 순차 실행. Redis 실패 시 Oracle 트랜잭션을 롤백. 정합성은 보장되지만, Redis 장애가 Oracle 쓰기 가용성을 낮춘다.
2. **Oracle 우선 커밋 + Redis 비동기 재시도**: Oracle INSERT를 먼저 커밋하고, 커밋 후 이벤트로 Redis 갱신을 비동기 실행. Redis 실패 시 재시도 큐에 넣고, 다음 폴링 주기에 재시도. 최종 일관성이지만, 사용자가 등록 직후 추천 검색을 하면 자기가 등록한 매물이 안 보일 수 있다.
3. **Redis Pipeline 일괄 + 실패 시 보상 트랜잭션**: Redis Pipeline으로 6개 연산을 일괄 전송하고, 부분 실패 시 성공한 키만 롤백(DEL/ZREM).

**기존 포트폴리오 연결**: 기존 CQRS Write-Through의 자연스러운 확장. **"동기화 대상이 1개에서 4개로 늘어났을 때 전략이 어떻게 바뀌는가"**라는 질문에 답하는 구조. 기존 포트폴리오의 Redis Pipeline 트러블슈팅(#6)과도 연결.

**의사결정 문서화 유형**: 유형 1 (실제 문제 발생 후 해결)

**포트폴리오 1건 확정.**

### 7.4 의사결정 4: bounds 변경에 의한 점수 재계산 — CPU 바운드 병렬 처리

**시나리오**: 사용자가 등록한 매물의 전세금이 해당 지역구의 기존 minPrice보다 낮다. bounds가 변경된다. 정규화 기준이 바뀌었으므로 해당 지역구 전체 매물의 priceScore가 무효화된다. 강남구 1,000건이면 1,000건 전부 재계산.

이것은 순수 CPU 바운드 연산이다 — 네트워크 I/O가 아니다(Redis에서 매물 데이터를 읽는 것은 초반 1회이고 이후는 순수 산술 연산), 디스크 I/O가 아니다(Oracle 조회 없이 Redis에서 가져온 데이터로 메모리 내에서 계산), 1,000건 × (정규화 3개 항목 + 가중치 적용 + 리뷰 점수 합산 + 최종 점수 산출) = 수천 회의 부동소수점 연산.

**대안 2개**:

1. **CompletableFuture + ThreadPoolExecutor (기존 I/O 바운드 방식)**: 기존 외부 API 병렬 호출에 사용한 구조. I/O 대기 시간 >> 연산 시간이므로 코어 수 대비 5배(20스레드). 하지만 CPU 바운드에서는 컨텍스트 스위칭 오버헤드만 증가.
2. **ForkJoinPool (CPU 바운드 최적)**: 코어 수와 동일한 스레드(4). work-stealing이 지역구별 매물 수 불균일(강남구 1,000건 vs 도봉구 50건)에서 부하 자동 균형. RecursiveTask로 매물 리스트 분할 → 서브태스크 fork → 점수 계산 → join.

**포트폴리오 서술 구조**: 기존 I/O 바운드 병렬 처리(트러블슈팅 #1)와 나란히 놓고 **"같은 프로젝트 내에서 워크로드 특성에 따라 다른 스레드풀을 선택한 사례"**로 비교 서술. "왜 여기서는 ForkJoinPool이고 저기서는 ThreadPoolExecutor인가?" — 워크로드 특성 분석(I/O 비율, CPU 점유율), 스레드 수 결정의 정량적 근거(코어 수 × (1 + W/S) 공식), 분할 전략의 차이(균일 vs 불균일 데이터)를 실측 데이터로 설명.

**기존 포트폴리오 연결**: 고도화 계획서에서 지적한 "워크로드 특성별 스레드 풀 분리 설계" 누락 역량을 정확히 채움.

**의사결정 문서화 유형**: 유형 3 (실측 기반 의사결정)

**포트폴리오 1건 확정.**

## 9. 찜하기/투표 없이 충분한 이유

위 4건은 전부 매물 등록이라는 **하나의 비즈니스 행위**에서 자연스럽게 파생된다. 찜이나 투표를 추가하면 cascade가 더 깊어지고 알림 fan-out이 생기지만, 그것은 구현 시간 대비 추가되는 포트폴리오 가치가 낮다.

4건이면 면접에서 각각 3분씩 = 12분. 기존 6건 중 주요 2~3건 + 신규 4건을 면접에서 다루면 30분 기술 면접을 커버하고도 남는다.

찜하기는 나중에 여유가 생기면 추가할 수 있지만, 매물 등록만으로도 포트폴리오 기술 요건은 충족된다.
