## 1. 논의 배경 및 출발점

### 1.1 현재 시스템의 구조적 한계 (고도화 계획서 기반)

고도화 계획서에서 식별한 Wherehouse의 구조적 문제는 세 가지로 요약된다.

**첫째, 포트폴리오 트러블슈팅 구조의 편중.** 기존 포트폴리오의 트러블슈팅 6건 중 5건이 "느린 것을 측정 → 빠르게 만듦 → 수치 제시" 패턴으로 구성되어 있다. 이는 성능 측정/개선이지, 비즈니스 요구사항 간 충돌에서 대안 간 트레이드오프를 분석하고 선택 근거를 명시하는 엔지니어링 의사결정이 아니다. IN절 Chunking 1건만이 V$SQL 실행계획 분석 → Chunk 크기별 비교 → 트레이드오프 기반 선택 구조를 갖추고 있다. 면접관이 "설계 역량"이 아닌 "측정 역량"으로 평가할 위험이 있으며, 취업박람회 멘토 2인이 공통으로 지적한 "설계 능력 = 신입 최우선 역량" 기준에 미달한다.

**둘째, 서비스 비즈니스 복잡도의 부재.** 현재 서비스는 Redis 읽기 → 점수 계산 → 정렬 → 반환의 단방향 조회 파이프라인에 고착되어 있다. 설계 명세서 v2의 리뷰 시스템은 단순 CRUD + Write-Through 동기화 수준이다. 동시 쓰기 충돌, 트랜잭션 간 격리 문제, 락 전략 선택의 필요가 발생하지 않는 구조이며, 동시성 제어·트랜잭션 설계·스레드 풀 분리 등 핵심 백엔드 역량을 적용할 비즈니스 시나리오가 부재하다.

**셋째, 기술 적용의 깊이 부족.** CompletableFuture + ThreadPoolExecutor는 I/O 바운드 외부 API 호출에만 적용되어 있다. Redis Sorted Set, HashMap 등은 단순 사용 수준이다. Semaphore는 ChatService 입장 제어 단일 케이스에 한정되어 있다. 워크로드 특성별 스레드 풀 분리, 비즈니스 요구사항의 시간/공간복잡도 제약에 따른 자료구조 비교 선택, 낙관적/비관적 락의 시나리오별 구분 적용 등 의사결정 과정이 없다. "기술을 사용했다"는 보여주지만, "왜 이 기술이고 대안은 무엇이었으며 무엇을 포기했는가"를 보여주지 못한다.

### 1.2 기존의 기술적 한계점 (누락 역량 상세)

**동시성 제어 영역**: synchronized와 ReentrantLock이 비즈니스 로직 맥락에서 기능별로 구분 적용된 적 없다. DB 수준 락 기술을 학습 및 비즈니스에 반영하지 않았다. ChatService의 Semaphore 적용은 입장 제어 단일 케이스에 한정되어, "왜 Semaphore인가"에 대한 4종 비교 실측은 존재하나 비즈니스 시나리오 다양성이 부족하다.

**트랜잭션 설계 영역**: 트랜잭션 경계 설정, 전파 전략(Propagation), 격리 수준(Isolation Level) 선택이 비즈니스 정합성 요구사항에 따라 의도적으로 결정된 적 없다. 설계 명세서 v2의 ReviewWriteService는 @Transactional 단일 적용 수준이며, 배치 트랜잭션과 실시간 서비스 트랜잭션 간 격리 전략이 "Overwrite 방식" 서술에 그친다.

**스레드 풀 설계 영역**: CompletableFuture + ThreadPoolExecutor는 I/O 바운드 외부 API 호출에만 적용되어 있다. CPU 바운드 연산에 ForkJoinPool을 적용하거나, 워크로드 특성에 따라 스레드 풀을 분리 설계한 적 없다. 스레드 풀 크기 결정의 정량적 근거(코어 수, I/O 대기 비율 등)가 부재하다.

**자료구조/알고리즘 영역**: Redis Sorted Set, HashMap 등을 "사용"했으나, 비즈니스 요구사항의 시간복잡도/공간복잡도 제약에 따라 자료구조를 비교 선택한 의사결정 과정이 없다. 알고리즘이 비즈니스 로직에 반영된 적 없다(추천 점수 산출은 가중 합산 수준).

### 1.3 고도화의 핵심 목표

기존 "조회 전용" 구조에 고착된 서비스에 동시 쓰기 충돌·트랜잭션 격리·락 전략 선택이 자연스럽게 요구되는 비즈니스 기능을 도입하여, 포트폴리오의 엔지니어링 의사결정 밀도를 구조적으로 전환한다.

**핵심 질문**: "성능 측정/개선을 넘어, 비즈니스 요구사항 간 충돌 상황에서 대안 간 트레이드오프를 분석하고 선택 근거를 명시하는 엔지니어링 의사결정을 포트폴리오에 어떻게 구조적으로 내재시킬 것인가?"

**핵심 규칙 4가지**:
- **규칙 1**: 비즈니스가 기술을 끌어내는 구조. 기술 적용의 출발점은 항상 비즈니스 요구사항이다.
- **규칙 2**: 실제 서비스 리뷰 기반 기능 도출 근거 확보. 자체 상상이 아닌 실제 서비스의 운영 맥락에서 기능 필요성의 근거를 확보한다.
- **규칙 3**: 개인 프로젝트 바운더리 준수. 인프라 과투자(Kafka, K8s 등), 프론트엔드 과투자, 도메인 지식 과투자 없이 백엔드 설계 역량 증명에 집중한다.
- **규칙 4**: 포트폴리오 문서화 구조 일관성. 모든 트러블슈팅/설계 결정은 "문제 정의 → 대안 검토 → 트레이드오프 → 선택 근거 → 정량 결과" 구조로 문서화한다.

### 1.4 채용 시장 기술 역량 빈도 분석 요약

한국 백엔드 신입/주니어 채용 공고 기술 역량 빈도 분석(사람인·점핏 10만 건 공고 리포트, 캔디드 88,492건 JD 분석, 실제 공고 15건+ 직접 분석, 채용 담당자 71명 설문 종합) 결과, 12개 기술 카테고리의 빈도는 다음과 같다.

**1층(절대 필수 — 없으면 서류 탈락)**: API 설계(~95%), DB/쿼리 최적화(~90%), 자료구조/알고리즘(~80%), 설계 역량(~70%). 공고의 70~95%에서 필수로 등장하며, 이것 없이는 서류 심사 자체를 통과할 수 없다.

**2층(핵심 우대 — 있으면 서류 통과율 급상승)**: CI/CD·Docker(~60%), Redis 캐싱(~50%), 테스트 코드(~40%), 메시지 큐(~25%). 인프런 로드맵에서 "이것까지 갖추면 취업에 문제없다"고 평가한 구간에 해당한다.

**3층(강력 차별화 — 면접에서 압도적 인상)**: 스레드풀/비동기(~20%), 모니터링(~20%), 트랜잭션 설계(~15%), 동시성 제어(~10%). 공고에서 직접 요구하는 빈도는 10~25%로 낮지만, 코멘토 합격자 분석에서 대기업·중견기업 합격자들이 공통적으로 보유한 차별화 역량으로 확인되었다.

**핵심 발견**: 공고 빈도와 합격 기여도는 다르며, 동시성 제어·트랜잭션 설계는 공고 언급률 10~15%이지만 면접 출제 빈도와 포트폴리오 차별화 효과는 최상급이다. 트러블슈팅 문서화가 포트폴리오 #1 차별화 요소이다. "왜 이 기술을 선택했는가"를 설명하는 능력이 기술 자체만큼 중요하다.

## 2. 실제 부동산 서비스 조사 결과

### 2.1 조사 범위 및 방법론

직방, 다방, 네이버 부동산, 피터팬의 좋은방 구하기, 호갱노노, 아실, 부동산플래닛 등 7개 주요 한국 부동산 서비스를 대상으로, Google Play Store/Apple App Store 사용자 리뷰, 블라인드·클리앙·DCInside 커뮤니티 사용자 불만, 앱 비교 기사, 기술 블로그를 종합 분석했다.

### 2.2 3강 플랫폼 핵심 기능 분석

**직방**: 632만 설치, MAU 229만(설치 수 1위). 핵심 차별화 기능은 전국 30세대 이상 모든 아파트를 구현한 3D 단지투어(동호수별 일조량·창밖 뷰 시뮬레이션)와, 중개 이력이 검증된 중개사의 1:1 케어와 계약서 정밀 진단을 제공하는 지킴중개 서비스다. 허위매물 대응으로는 헛걸음 시 현금 3만 원을 보상하는 헛걸음 보상제, 삼진아웃(신고 3회 누적 시 영구 퇴출)제를 운영하며, 2025년 1월부터 한국부동산원 허위매물 모니터링에 참여해 거래 완료 매물을 당일 삭제하고 있다. 2018년 호갱노노를 인수해 아파트 시세 분석 역량을 강화했고, 삼성SDS 홈IoT 인수(2022)로 스마트홈 사업까지 확장했다.

**다방**: 설치 342만, MAU 103만(2위). UGC와 AI에서 가장 앞서 있다. 2025년 4월 론칭한 동네이야기 커뮤니티는 동네 기반으로 이웃과 일상·알짜 정보를 공유하고 투표·매물 첨부 기능까지 지원한다. 2019년부터 운영 중인 부동산 리뷰 서비스는 방문 이용자가 직접 남기는 중개사 리뷰를 공개한다. 2025년 10월 출시된 AI 방찾기는 생성형 AI 챗봇으로 자연어 매물 검색을 지원한다. 필터링에서도 층수·창문 방향·엘리베이터·전세대출 가능 여부·반려동물 허용까지 세분화되어 있어, 원룸·자취생 시장에서 가장 높은 필터 만족도를 보인다. 중개사에게 카카오톡 메시지로 매물 가용 여부를 확인하는 매물확인 메신저(48시간 내 답변 의무, 미답변 시 매물 자동 비노출)도 다방만의 신뢰성 장치다.

**네이버 부동산**: MAU 134만, 매물 수 압도적 1위(중개사 광고 수수료가 저렴해 가장 많은 매물 등록). 2026년 3월 네이버페이 부동산으로 대규모 개편하면서 매물지도 중심의 모바일 임장 환경을 구축했다. 최대 강점은 한국부동산원 시세, KB부동산 시세, 매물 호가를 동시에 보여주는 3종 시세 비교이며, 방향 10가지 이상·현관 구조(계단식/복도식/복합식)·엘리베이터 유무까지 지원하는 업계 최정밀 필터링을 제공한다. 국토부 조사에서 낚시성 매물이 1.37%로 가장 낮은 허위매물 비율을 기록했다.

### 2.3 니치 플랫폼 4인방 분석

**피터팬**: 네이버 카페(회원 540만+)에서 출발한 국내 유일의 직거래 전문 플랫폼. 연간 약 75만 건의 매물을 네이버 부동산에 공급하는 공식 CP 파트너. First American과 제휴한 안심직거래 서비스로 전·월세 권리보험이 보증금 10억 원까지 커버하고 보증금 반환 소송 시 무료 법률 대리를 제공한다. 동네생활백서 거주후기에서는 실거주자가 층간소음·난방·주차·치안 등에 대한 리뷰를 남긴다.

**호갱노노**: 직방이 2018년 인수했지만 독립 노선 유지. MAU 179만·이탈률 21.8%(업계 최저)·평균 사용 기간 519일(최장). 3D 단지투어에서 계절별·시간대별 일조량을 시뮬레이션하고, 입주민이 직접 쓰는 층간소음·주차·학군 리뷰가 최대 자산이다. 매달 가장 많이 조회된 단지에 왕관 아이콘을 부여해 "대장 아파트"를 식별해준다.

**아실**: 2006년부터 현재까지의 국토부 실거래 데이터를 여러 단지를 하나의 차트에 겹쳐서 비교하는 기능이 핵심. 경쟁사에서 유료인 이 기능을 무료로 제공해 "투자자 필수 앱" 평가. 외지인 매수 비율, 학군 비교, 대지지분 계산, 역전세 물건 식별 등 투자 분석 특화.

**부동산플래닛**: 국내 유일의 상업용 부동산 전문 데이터 플랫폼. 2025년 출시한 상가·사무실 AI 추정가(AVM)는 서울 57.9만 개, 전국 240만 개 상업용 건물 대상으로 중위절대오차 5.80% 정확도 달성.

### 2.4 공통 기능 vs 차별화 매트릭스

모든 서비스가 공통으로 제공하는 기능: 지도 기반 매물 검색, 찜/관심 매물, 최근 본 매물, 실거래가 조회, 중개사 연결, 신규 매물 알림.

차별화가 발생하는 세 영역:
- **UGC/커뮤니티**: 다방(동네이야기)과 피터팬(거주후기·카페)이 가장 활발. 호갱노노(입주민 리뷰)가 아파트 한정. 직방·네이버는 앱 내 직접적 UGC가 상대적으로 제한적.
- **AI 기능**: 다방(AI 방찾기 챗봇), 호갱노노(AI 중개사 추천), 부동산플래닛(AI 추정가 AVM), 아실(AI 미래가격 예측)이 각자 다른 방향으로 AI를 적용.
- **데이터 분석 깊이**: 아실(단지 간 가격 비교 차트), 호갱노노(3D 일조량·학원가·경사도), 부동산플래닛(AVM·입주기업·실효임대료)이 각 영역에서 가장 깊은 분석 제공.

### 2.5 사용자 리뷰 및 커뮤니티 불만 분석

**허위매물**: KISO 부동산매물클린관리센터 공식 데이터에 따르면 허위매물 신고는 2024년 29,812건, 2025년 34,528건(+15.8% YoY)으로 증가세 지속. 한국소비자원 조사에서 앱 상 매물과 실제 내용이 모두 일치한 경우는 100건 중 41건에 불과. 가장 흔한 수법은 이미 거래 완료된 매물 방치, 시세보다 현저히 낮은 가격 게시 후 다른 매물 유도, 대출금 차감 가격 표시, 광각렌즈 활용 과장 사진이다. 블라인드에서 "네이버 빼고는 다 허위인 듯"이라는 댓글이 공감을 얻고, 클리앙에서는 "직방, 다방은 '이 동네에 이런 중개사가 있구나'가 정보의 끝"이라는 평가까지 나온다.

**필터링 격차**: 직방에 대한 가장 빈번한 UI/UX 불만은 필터 부족. App Store 리뷰에서 "방 크기, 베란다, 엘리베이터, 전세대출 가능 여부 같은 꼭 필터링해야 하는 중요한 조건이 없다"가 반복적으로 등장. 반면 다방은 클리앙에서 "필터 설정 유지하고 아침마다 지역 옮겨다니면서 새 매물 확인하면 됨"이라는 호평.

**MZ세대와 1인 가구가 갈증을 느끼는 정보 6가지**:
1. 층간소음·벽간소음 — "원룸은 옆방 소음이 더 심각한 문제인데 벽이 가벽인지 확인할 방법이 없다"(DCInside 자취갤)
2. 실제 관리비 — 전월세 신고제 회피를 위해 "월세는 낮추고 관리비를 올리는 꼼수"가 늘면서, 전기·수도·가스 포함 여부가 앱에 명확하지 않음
3. 야간 치안·동네 분위기 — "낮에만 방 보러 가니 밤 분위기를 알 수 없다"
4. 벌레(바퀴벌레 등) — 호갱노노 후기에서 "바퀴벌레가 많다는 리뷰 보고 계약 포기" 사례
5. 집주인 평판 — 직접 공유하는 플랫폼 부재(개인정보 이슈)
6. 건물 연식별 방음 수준 — "계단식이고 한 층에 방 5개 이상이면 방음 안 되는 방일 확률 80%"(DCInside)

**호갱노노 리뷰 생태계의 신뢰성 문제**: App Store 리뷰에서 "부정적 평가 남기면 타 거주민들 인신공격 댓글 10개씩 달린다", "집값 올리려고 좋은 얘기만 도배" 비판이 반복. 집값 상승을 원하는 현 거주자가 솔직한 부정적 리뷰를 억압하는 구조적 문제 존재.

### 2.6 조사의 핵심 결론

현재 어떤 단일 앱도 MZ세대 1인 가구의 주거 탐색 여정을 완결하지 못한다. 매물 탐색은 다방·직방, 시세 확인은 호갱노노·아실, 동네 정보는 당근마켓, 안전 확인은 안심전세앱으로 분산되어 있다. 개인 프로젝트가 노릴 수 있는 가장 큰 틈새는 세 가지다: 원룸·빌라 대상 실거주 후기 집중 플랫폼, 커뮤니티 리뷰의 조작 방지 메커니즘, 여러 앱의 핵심 정보를 한 곳에 모아 비교·분석하는 메타 서비스.

## 3. 기능 설계 논의 과정

### 3.1 제1차 제안 — 찜하기 + 가격 변동 알림 + 허위매물 신고 (3개 기능)

#### 3.1.1 제안 근거

현재 Wherehouse의 구조적 한계는 "조회 전용 단방향 파이프라인"이다. 사용자가 데이터를 쓰는 경로가 리뷰 CRUD 하나뿐이고, 그마저도 Write-Through 동기화 수준이라 동시 쓰기 충돌, 트랜잭션 격리, 락 전략 선택이 발생할 비즈니스 시나리오 자체가 없다.

필요한 것은 사용자가 능동적으로 데이터를 쓰는 경로를 2~3개 더 만들어서, 그 경로들이 서로 충돌하거나 기존 조회 경로와 간섭하는 상황을 만드는 것이다.

리서치에서 나온 수십 가지 기능 중 대부분은 프론트엔드 과투자(3D 단지투어, AI 챗봇)이거나 인프라 과투자(Kafka 기반 이벤트 버스)이거나 도메인 과투자(AVM 추정가 모델)라서 바운더리 밖이다. 바운더리 안에서 기존 Oracle + Redis 스택으로 구현 가능하고, 누락 역량을 자연스럽게 요구하는 기능은 3개로 수렴했다.

#### 3.1.2 후보 1: 찜하기 (관심 매물)

사용자가 추천 결과에서 매물을 찜하고, 찜 수가 매물 카드에 표시되는 기능. 모든 부동산 앱이 공통으로 갖고 있는 기본 기능이고, 리서치에서 확인한 바와 같이 사용자 참여의 가장 기초적인 형태다.

이 기능이 자연스럽게 요구하는 기술적 의사결정: 인기 매물에 다수 사용자가 동시에 찜/취소를 누를 때 카운터 정합성 문제가 발생한다. Redis INCR 원자적 연산 vs Oracle row-level lock vs JPA @Version 낙관적 락 중 어떤 전략을 선택할 것인가? 찜 카운터를 Redis에만 둘 것인가, Oracle에 반정규화할 것인가? Redis 장애 시 폴백 전략은? 이것이 "대안 비교 → 트레이드오프 → 선택 근거" 구조의 트러블슈팅이 된다.

#### 3.1.3 후보 2: 가격 변동 알림

배치가 새 데이터를 가져왔을 때 가격이 변경된 매물을 감지하고, 해당 매물을 찜한 사용자에게 알림을 보내는 기능. 리서치에서 MZ세대가 "아침마다 새 매물 확인"하는 패턴과, 가격 변동 추적에 대한 니즈가 반복적으로 나왔다.

이 기능이 자연스럽게 요구하는 기술적 의사결정: 배치가 Oracle을 업데이트하는 시점과 사용자가 매물을 조회하는 시점이 겹칠 때 어떤 격리 수준을 적용할 것인가? 알림 발송이 Oracle 커밋 전에 일어나면 순서 역전이 발생하는데, `@TransactionalEventListener(AFTER_COMMIT)` vs 별도 outbox 테이블 중 어떤 방식을 선택할 것인가? 알림 fan-out(1개 매물 변경 → N명 찜 사용자)을 메인 스레드에서 처리할 것인가, 전용 스레드풀을 분리할 것인가? 스레드풀 분리 설계, 트랜잭션 격리 수준 선택, 이벤트 순서 보장이 모두 이 기능 하나에서 나온다.

#### 3.1.4 후보 3: 허위매물 신고

사용자가 매물을 신고하고, 신고 누적 시 자동으로 매물을 비공개 처리하는 기능. 리서치에서 허위매물이 부동산 앱 생태계의 #1 불만이고, 사용자 10명 중 3.4명이 경험하며, 기존 서비스의 신고 시스템 실효성에 대한 의문이 반복적으로 나왔다.

이 기능이 자연스럽게 요구하는 기술적 의사결정: 동시 신고 시 카운터 원자성(찜하기와 동일 패턴이지만 비즈니스 규칙이 다름 — 임계값 초과 시 상태 변경 트리거), 신고 임계값 도달과 매물 상태 변경의 트랜잭션 경계, 중복 신고 방지를 Redis Set으로 할 것인가 Oracle UNIQUE 제약조건으로 할 것인가.

#### 3.1.5 허위매물 신고 기각 사유

허위매물 신고는 기각했다. 찜하기와 기술 패턴이 거의 동일하다(Redis 원자적 카운터 + 임계값 판단). 같은 패턴을 두 번 구현하는 것은 포트폴리오에서 새로운 의사결정을 생성하지 않는다.

#### 3.1.6 1차 확정: 찜하기 + 가격 변동 알림

찜하기와 가격 변동 알림은 기술 패턴이 완전히 다르다. 찜하기는 "동일 자원에 대한 동시 쓰기 → 락 전략 선택" 문제이고, 가격 변동 알림은 "배치 쓰기와 실시간 조회의 간섭 → 트랜잭션 격리 + 이벤트 순서 보장 + 스레드풀 분리" 문제다. 두 기능이 겹치는 기술 영역이 없으므로, 2개만으로 누락 역량 4개를 전부 커버한다.

### 3.2 제2차 수정 — 가격 변동 알림 기각, 사용자 행위 기반으로 전환

#### 3.2.1 기각 사유

가격 변동 알림의 이벤트 발생원이 배치(월 1회)이기 때문에 사용자 간 상호작용에 의한 비즈니스 맥락이 발생하지 않는다. 실 서비스가 아니기 때문에 실시간 매물 갱신이 되지 않고 한 달에 한 번밖에 없다. 배치가 아닌 사용자 행위에 의해 이벤트가 발생해야 수많은 사용자 간의 상호 작용에 의한 비즈니스 맥락이 주어진다.

#### 3.2.2 핵심 원칙 전환

**모든 이벤트의 발생원 = 사용자 행위**. 배치가 아닌, 사용자 A가 쓰기를 수행한 결과가 사용자 B의 데이터 상태에 영향을 주는 구조로 만든다. 이렇게 하면 사용자 수에 비례해서 이벤트 빈도가 증가하고, 월 1회 배치 한계에 종속되지 않는다.

#### 3.2.3 수정된 기능 구성: 찜하기 + 인기 매물 알림(찜 기반) + 리뷰 투표

**기능 1: 찜하기 + 인기 매물 알림 (찜 급등 시 사용자 간 알림)**

기존에 제안한 찜하기에, 알림 트리거를 배치가 아닌 다른 사용자의 찜 행위로 바꾼다.

사용자 A가 매물을 찜한다 → 찜 카운터가 증가한다 → 카운터가 임계값(예: 10, 30, 50)을 넘는 순간 → 해당 매물을 이미 찜한 사용자 B, C, D 전원에게 알림이 생성된다("래미안그레이튼에 관심 급증, 현재 52명 찜").

이 구조에서 발생하는 기술적 의사결정:

사용자 A의 단일 찜 행위가 만드는 처리 체인: 찜 INSERT → 카운터 증가 → 임계값 판단 → (초과 시) 찜 유저 목록 조회 → N명에게 알림 INSERT. 이 체인 전체를 하나의 트랜잭션으로 묶으면 N명 알림 INSERT 동안 해당 매물의 찜 테이블에 lock이 걸린다. 분리하면 알림 생성 중 다른 사용자의 찜이 추가되어 카운터와 알림 대상 목록이 불일치할 수 있다. 트랜잭션 경계를 어디서 끊을 것인가가 첫 번째 의사결정이다.

fan-out(1명의 찜 → N명에게 알림)을 메인 API 스레드에서 동기로 처리하면 사용자 A의 찜 API 응답이 N에 비례해서 느려진다. 알림 생성을 전용 스레드풀에서 비동기로 분리할 것인가, 메인 트랜잭션 커밋 후 이벤트로 위임할 것인가 — `@TransactionalEventListener(AFTER_COMMIT)` + `@Async("notificationExecutor")` 조합의 설계 근거가 여기서 나온다. 스레드풀 크기는 미니 PC 4코어 기준으로 I/O 대기 비율을 실측해서 정량적으로 결정한다.

기존 "가격 변동 알림"에서 필요했던 기술적 의사결정(트랜잭션 격리, 이벤트 순서 보장, 스레드풀 분리)이 전부 동일하게 발생하지만, 트리거가 사용자 행위다. 사용자가 많을수록 이벤트가 많아지고, 배치 주기에 종속되지 않는다.

**기능 2: 리뷰 도움돼요 투표**

사용자가 다른 사용자의 리뷰에 "도움돼요"/"별로에요" 투표를 한다. 투표 수는 리뷰 카드에 표시되고, 리뷰 목록의 정렬 기준이 된다. 도움돼요 비율이 높은 리뷰가 상단에 노출되고, 이 리뷰의 키워드가 해당 매물의 추천 점수에 더 큰 가중치로 반영된다.

이 기능이 만드는 기술적 의사결정:

사용자 A가 투표 한 번을 누르면 발생하는 처리 체인: 투표 INSERT → 리뷰의 `helpful_count` 증가 → 리뷰 정렬 순위 변동 → (변동이 클 경우) 해당 매물의 키워드 가중치 재계산 → 매물 ReviewScore 변동 → Redis 캐시 갱신. 단일 사용자 행위가 4단계 cascade를 발생시킨다.

이 cascade의 각 단계를 어디까지 동기로 처리하고 어디서부터 비동기로 분리할 것인가. 투표 INSERT + helpful_count 증가까지는 동기 트랜잭션, 리뷰 순위 재계산과 매물 점수 갱신은 비동기 이벤트로 분리하는 것이 자연스럽다. 그런데 비동기 분리 시 "투표했는데 리뷰 순위가 바로 안 바뀐다"는 최종 일관성(Eventual Consistency) 문제가 발생한다. 즉시 일관성 vs 최종 일관성의 비즈니스 허용 범위를 결정하는 것이 설계 의사결정이 된다.

동시 투표 시 `helpful_count` 정합성은 찜하기 카운터와 동일한 패턴이지만, 비즈니스 규칙이 다르다. 찜하기는 카운터가 단순 표시용이라 1~2의 오차가 허용되지만, 투표는 정렬 순위에 직접 영향을 주므로 정합성 요구 수준이 다르다. 동일한 Redis INCR vs Oracle row-level lock 대안인데 비즈니스 요구사항의 차이로 선택이 달라지는 사례 — 이것이 포트폴리오에서 "같은 기술인데 맥락에 따라 다른 선택을 했다"를 보여주는 가장 강력한 근거가 된다.

### 3.3 제3차 확장 — 매물 등록 기능 추가

#### 3.3.1 추가 근거

지금까지 논의한 찜, 투표, 리뷰는 전부 매물이라는 핵심 엔티티에 대한 부수 데이터를 쓰는 행위다. 매물 자체는 배치가 월 1회 넣어주는 읽기 전용 데이터로 고정되어 있고, 사용자는 그 위에 메타데이터(리뷰, 찜, 투표)만 붙이는 구조다.

매물 등록을 사용자에게 열면 핵심 엔티티 자체가 사용자 쓰기 경로에 들어온다. 이것은 부수 데이터 쓰기와 질적으로 다른 시나리오를 대량으로 생성한다. 실 상용 서비스에서는 실제 판매자만 등록할 수 있지만, 현재는 상황 재연을 위한 포트폴리오 프로젝트이므로 누구나 매물 정보를 등록할 수 있는 구조로 설계한다.

#### 3.3.2 매물 등록이 생성하는 기술적 시나리오 5가지

**시나리오 1: 동일 매물 동시 등록 (중복 감지 Race Condition)**

현재 시스템에서 매물 식별자는 불변 속성 5개(SGG_CD, JIBUN, APT_NM, FLOOR, EXCLU_USE_AR) 조합의 MD5 해시다. 배치에서만 이 해시를 생성하므로 중복 문제가 발생하지 않았다.

사용자 등록을 열면: 사용자 A와 사용자 B가 같은 물리적 매물(같은 아파트, 같은 층, 같은 면적)을 거의 동시에 등록한다. 둘 다 동일한 MD5 해시가 생성된다. 둘 다 INSERT 전에 "이 해시가 이미 존재하는가?" 확인 쿼리를 날린다. 둘 다 "존재하지 않음"을 확인한다. 둘 다 INSERT를 시도한다. Oracle PK 제약조건에 의해 하나만 성공하고 하나는 DuplicateKeyException이 발생한다.

여기서 의사결정 — 이 Race Condition을 어느 레벨에서 어떻게 처리할 것인가:

대안 1: Oracle UNIQUE 제약조건에 의존하고 예외를 catch해서 "이미 등록된 매물입니다"를 반환한다. 구현이 가장 단순하지만, 예외 기반 흐름 제어는 비용이 크고 사용자 B가 입력한 데이터(가격 등)가 유실된다.

대안 2: SELECT FOR UPDATE로 비관적 락을 건다. 동일 해시에 대한 두 번째 트랜잭션이 첫 번째 커밋까지 대기한다. 정합성은 완벽하지만 락 대기 시간이 발생하고, 신규 INSERT에 SELECT FOR UPDATE를 거는 것은 해당 row가 아직 존재하지 않으므로 gap lock이 필요하다(Oracle에서는 다르게 동작).

대안 3: Redis SETNX (SET if Not eXists)로 해시 기반 분산 락을 선점한다. `SETNX lock:property:{md5_hash} {user_id} EX 30`으로 30초간 해당 해시의 등록 권한을 선점하고, 선점 실패 시 즉시 "다른 사용자가 등록 중입니다"를 반환한다. Oracle 레벨 락보다 가볍고 사용자 경험이 좋지만, Redis 장애 시 폴백이 필요하다.

이것이 ChatService의 Semaphore 4종 비교와 동일한 패턴(동일 문제에 대한 N개 대안 실측 비교 → 트레이드오프 기반 선택)이지만, DB 레벨 락이라는 새로운 기술 영역으로 확장된다. 고도화 계획서에서 "DB 수준 락 기술을 학습 및 비즈니스에 반영하지 않았음"이라고 정확히 지적한 누락 역량이 여기서 채워진다.

**시나리오 2: 사용자 등록 매물 vs 배치 매물 충돌 (데이터 소스 간 머지 전략)**

사용자가 등록한 매물과 국토부 배치가 가져온 매물이 동일 MD5 해시를 가진다. 사용자가 "전세금 1.8억"으로 등록했는데 다음 배치에서 국토부 API가 같은 매물을 "전세금 2억"으로 가져온다.

누구의 데이터를 신뢰할 것인가? 이 결정은 단순 기술이 아니라 비즈니스 정책이다.

대안 1 — 국토부 데이터 우선(Authority Override): 공공 데이터가 공식 실거래 신고 기반이므로 항상 우선한다. 사용자 등록 가격은 "사용자 제시 가격"으로 별도 컬럼에 보존하고, 메인 가격은 국토부 데이터로 덮어쓴다.

대안 2 — 최신 데이터 우선(Last Write Wins): LAST_UPDATED 타임스탬프가 더 최근인 데이터를 반영한다. 단, 사용자가 허위 가격을 등록할 경우 공식 데이터가 밀린다.

대안 3 — 수동 병합(Conflict Detection + 관리자 결정): 가격 차이가 임계값(예: 10%) 이상이면 CONFLICT 상태로 플래깅하고, 양쪽 데이터를 모두 보존한 뒤 별도 해소 프로세스를 둔다.

이것이 고도화 계획서에서 요구한 "유형 2 — 설계 단계 사전 의사결정" 패턴이다. 문제가 발생하기 전에 아키텍처 수준에서 판단하고, 대안별 장단점을 분석하고, 비즈니스 맥락에 따라 선택 근거를 명시하는 구조.

**시나리오 3: 매물 등록 → Redis 인덱스 즉시 반영 (쓰기 전파 정합성)**

현재 시스템에서 Redis Sorted Set 인덱스(`idx:charterPrice:{district}` 등)는 배치가 일괄 갱신한다. 사용자가 매물을 등록하면 이 인덱스에 즉시 반영해야 해당 매물이 추천 결과에 나타난다.

처리 체인: 사용자가 매물 등록 → Oracle INSERT → Redis Hash에 매물 상세 저장 → Redis Sorted Set 5~6개에 ZADD → Redis 정규화 범위(bounds) 재계산 → Redis 안전성 점수 매핑.

이 체인에서 Oracle INSERT는 성공했는데 Redis Sorted Set ZADD 중간에 실패하면 어떻게 되는가? 매물은 DB에 존재하지만 추천 검색에서는 나오지 않는 불일치 상태가 된다.

대안 1 — 전체 동기 처리: Oracle INSERT + Redis 6개 연산을 하나의 서비스 메서드 안에서 순차 실행한다. Redis 실패 시 Oracle 트랜잭션을 롤백한다. 정합성은 보장되지만, Redis 장애가 Oracle 쓰기 가용성을 낮춘다.

대안 2 — Oracle 우선 + Redis 비동기 동기화: Oracle INSERT를 먼저 커밋하고, 커밋 후 이벤트로 Redis 갱신을 비동기 실행한다. Redis 실패 시 재시도 큐에 넣고, 다음 폴링 주기에 재시도한다. 최종 일관성이지만, 사용자가 등록 직후 추천 검색을 하면 자기가 등록한 매물이 안 보일 수 있다.

대안 3 — Redis Pipeline 일괄 + 실패 시 보상 트랜잭션: Redis Pipeline으로 6개 연산을 일괄 전송하고, 부분 실패 시 성공한 키만 롤백(DEL/ZREM)한다.

이것이 기존 CQRS Write-Through 동기화의 확장이다. 기존에는 리뷰 통계 1개 Hash만 동기화했지만, 매물 등록은 Hash 1개 + Sorted Set 5~6개 + bounds Hash 1개를 동시에 갱신해야 한다. 동기화 대상이 많아지면서 부분 실패 처리 전략이 새로운 의사결정이 된다.

**시나리오 4: 매물 등록자가 상태를 변경한다 (거래완료, 가격 수정)**

등록자가 "거래완료"로 상태를 변경하면: 해당 매물을 찜한 N명의 찜 목록에서 상태가 바뀌어야 하고, 해당 매물에 달린 리뷰의 노출 정책이 변경될 수 있고, 해당 매물이 Redis Sorted Set 인덱스에서 제거되어야 하고, 진행 중인 추천 요청이 해당 매물을 이미 조회한 상태일 수 있다.

등록자가 가격을 수정하면: Redis Sorted Set의 score가 변경되어야 하고, 해당 매물을 찜한 사용자에게 알림이 발생하고(이것이 앞서 기각한 "가격 변동 알림"인데, 이번에는 사용자 행위가 트리거다), bounds(정규화 범위)가 변경될 수 있어서 같은 지역구의 다른 매물 점수도 영향을 받는다.

이것이 앞서 기각한 "가격 변동 알림"이 사용자 행위 기반으로 부활하는 구조다. 배치가 아닌 등록자의 가격 수정 행위가 트리거이므로, 빈도가 사용자 활동에 비례하고 월 1회 배치에 종속되지 않는다.

**시나리오 5: 리뷰와 매물 등록의 상호 의존성**

기존 시스템에서 리뷰는 매물 FK를 참조한다. 배치 매물만 존재할 때는 FK 대상이 항상 존재했다. 사용자 등록 매물이 추가되면: 사용자 A가 매물을 등록하고, 사용자 B가 해당 매물에 리뷰를 쓰고, 사용자 A가 매물을 삭제하면 — 리뷰의 FK 참조가 깨진다.

매물 삭제를 허용할 것인가? 리뷰가 달린 매물은 삭제 불가로 할 것인가, Soft Delete(논리 삭제)로 할 것인가? Soft Delete 시 삭제된 매물의 리뷰와 찜 데이터를 어떻게 처리할 것인가?

#### 3.3.3 매물 등록 추가에 의한 시스템 구조 변환

| 항목 | 현재 | 매물 등록 추가 후 |
|------|------|-----------------|
| 매물 데이터 소스 | 배치(월 1회) 단일 | 배치 + 사용자 등록 이원 |
| 쓰기 경로 | 리뷰 1개 | 매물 등록/수정/삭제 + 리뷰 + 찜 + 투표 |
| 핵심 엔티티 쓰기 주체 | 시스템(배치) only | 시스템 + 사용자 |
| 동시성 충돌 지점 | 리뷰 통계 카운터 | 매물 중복 등록, 카운터 2종(찜/투표), 상태 변경 + 조회 |
| 트랜잭션 복잡도 | 리뷰 INSERT + 통계 UPDATE | 매물 INSERT + Redis 6개 인덱스 + bounds 재계산 |
| 데이터 정합성 이슈 | Write-Through 단일 | 배치-사용자 머지, 부분 실패 보상, FK 참조 보호 |

기존 "조회 전용 파이프라인" 구조가 완전히 해체되고, 다수의 쓰기 경로가 핵심 엔티티 수준에서 상호 간섭하는 시스템으로 전환된다.

### 3.4 제4차 심화 — 사용자 간 상호작용 중심축 전환

#### 3.4.1 전환 동기

개별 기능(찜, 투표, 매물 등록)을 먼저 정의하고 그 부산물로 사용자 간 상호작용이 "발생하기도 한다"는 구조가 아니라, 사용자 A의 행위가 사용자 B의 데이터 상태를 변경하는 것 자체를 설계의 중심축으로 놓고, 그 축을 구현하기 위해 어떤 기능이 필요한지를 역으로 도출한다.

현재 설계의 상호작용 구조를 냉정하게 보면, 전부 매물을 허브로 한 간접 상호작용이다. 사용자 A가 매물에 행위를 하면, 시스템이 같은 매물에 연결된 사용자 B에게 영향을 전파하는 구조다. A와 B가 직접 만나는 지점이 없다. A가 뭘 하든 B는 수동적으로 알림을 받거나, 목록 상태가 바뀌는 것을 나중에 확인할 뿐이다.

#### 3.4.2 상호작용 유형 3가지 식별

**유형 1: 내 행위가 타인의 "목록"을 오염시키는 상호작용**

사용자 A가 매물을 등록한다. 사용자 B가 그 매물을 찜한다. 사용자 C가 그 매물에 리뷰를 쓴다. 그런데 사용자 A가 매물을 "거래완료"로 변경하거나 삭제한다.

이 순간 사용자 B의 찜 목록에 있는 매물의 상태가 바뀌어야 한다. 사용자 C의 리뷰가 참조하는 매물의 노출 정책이 바뀌어야 한다. 추천 결과에서 해당 매물이 제거되어야 하므로, 현재 추천 요청을 처리 중인 사용자 D의 응답에도 영향을 줄 수 있다.

이것은 1명의 상태 변경이 N명의 저장 데이터를 cascade로 변경하는 구조다.

기술적으로 이 cascade에서 발생하는 의사결정: 사용자 A의 "거래완료" 변경 트랜잭션 안에서 B의 찜 목록 상태 갱신, C의 리뷰 노출 상태 변경, Redis 인덱스 제거를 전부 처리할 것인가? 아니면 A의 트랜잭션은 매물 상태 변경만 커밋하고, 나머지는 이벤트로 비동기 전파할 것인가? 동기로 묶으면 A의 API 응답이 N에 비례해서 느려지고 트랜잭션 범위가 비대해진다. 비동기로 분리하면 B가 찜 목록을 조회했을 때 이미 거래완료된 매물이 아직 "활성"으로 보이는 시간 창(stale window)이 존재한다.

**유형 2: 다수의 행위가 하나의 공유 상태를 경쟁적으로 변경하는 상호작용**

사용자 A, B, C가 동시에 같은 매물에 리뷰를 작성한다. 세 명의 리뷰가 모두 성공하면 REVIEW_STATISTICS의 review_count, avg_rating, positive_keyword_count가 3번 갱신되어야 한다. 각각의 갱신이 SELECT(현재 통계) → 계산 → UPDATE 패턴이라면, A의 SELECT와 B의 SELECT가 같은 값을 읽고 각자 계산한 뒤 UPDATE하면 하나의 리뷰 반영분이 유실된다(Lost Update).

동시에 같은 매물을 여러 명이 찜하면 favorite_count에서 동일한 문제가 발생한다. 동시에 같은 리뷰에 여러 명이 투표하면 helpful_count에서 동일한 문제가 발생한다.

같은 패턴이 3개의 서로 다른 비즈니스 맥락에서 반복되지만, 요구되는 정합성 수준이 다르다. 찜 수는 표시용이라 일시적 오차가 허용된다. 리뷰 통계는 추천 점수에 직접 영향을 주므로 정합성이 중요하다. 투표 수는 리뷰 정렬 순위를 결정하므로 중간 수준이다. 이 차이가 동일 기술(Redis INCR vs Oracle row lock vs 낙관적 락)에 대해 맥락별로 다른 선택을 하는 근거가 된다.

**유형 3: 내 행위가 타인에게 "알림"을 발생시키는 상호작용**

사용자 A가 매물 가격을 수정한다 → 해당 매물을 찜한 B, C, D에게 가격 변동 알림이 생성된다.

사용자 A가 매물을 찜한다 → 찜 수가 임계값을 넘으면, 이미 찜한 B, C에게 인기 급등 알림이 생성된다.

사용자 A가 리뷰를 작성한다 → 해당 매물을 찜한 B, C에게 새 리뷰 알림이 생성된다.

모든 알림의 트리거가 사용자 행위이고, 알림 대상이 "해당 매물을 찜한 사용자 집합"이라는 공통 fan-out 구조를 공유한다. 알림 종류는 다르지만(가격 변동, 인기 급등, 새 리뷰), 기술적 처리 구조는 동일하다: 트리거 행위 커밋 → 찜 유저 목록 조회 → N건 알림 INSERT → (선택적) 실시간 푸시. 이 공통 구조를 하나의 알림 인프라로 설계하고, 알림 유형별로 트리거 조건과 페이로드만 달리하는 것이 설계 의사결정이 된다.

#### 3.4.3 상호작용 유형으로부터 기능 목록 역도출

위 상호작용 유형을 구현하려면 최소한 다음 기능이 필요하다:

**매물 등록/수정/상태변경** — 유형 1(cascade)과 유형 3(알림)의 트리거 역할. 사용자가 핵심 엔티티를 직접 쓰므로, 중복 감지·배치 충돌·Redis 다중 인덱스 동기화가 전부 여기서 발생한다.

**찜하기** — 유형 2(동시 카운터)와 유형 3(알림)의 매개 역할. 찜 자체가 쓰기이면서, 동시에 알림 대상 집합을 정의하는 데이터 구조이기도 하다. 즉 찜은 "이 매물에 관심 있는 사용자 목록"이라는 구독(subscription) 역할을 겸한다.

**리뷰 + 투표** — 유형 2(동시 카운터)의 두 번째, 세 번째 인스턴스. 리뷰 작성은 유형 3(새 리뷰 알림)의 또 다른 트리거다.

### 3.5 양방향 상호작용 확장 검토 및 기각

#### 3.5.1 검토한 양방향 상호작용 후보

양방향 상호작용(A의 행위에 대해 B가 응답하고, 그 응답이 다시 A나 C의 상태를 변경하는 양방향 체인)을 만들려면 다음이 가능하다: 매물에 대한 가격 제안(네고) → 등록자가 수락/거절 → 수락 시 다른 제안자 전원에게 종료 알림, 리뷰에 대한 반박/이의 제기 → 원 작성자가 수정 또는 유지 → 다른 사용자의 투표로 최종 판정, 매물 공동 관심 그룹(같은 매물을 찜한 사용자끼리 정보 공유).

이것들은 기술적으로 구현 가능하고, 트랜잭션 체인도 길어지고, 상태 머신(제안→수락/거절→완료)이 들어오면서 동시성 문제도 더 복잡해진다.

#### 3.5.2 기각 사유

양방향 상호작용을 만들면 프론트엔드에서 상태 전이 UI(제안 목록, 수락/거절 버튼, 실시간 상태 반영)가 필수적으로 따라오고, 이것이 백엔드 1인 전담 프로젝트의 바운더리를 넘길 가능성이 높다. 또한 가격 네고나 리뷰 반박 같은 기능은 실제 부동산 서비스에서도 극히 일부만 존재하는 기능이라, 면접관이 "이걸 왜 만들었는가"에 대한 비즈니스 근거를 물었을 때 답이 약해진다.

현재 설계의 단방향 cascade 구조가 이미 커버하는 누락 역량은: DB 레벨 락 전략 비교(매물 중복 등록), 동시성 제어 실측(카운터 3종), 트랜잭션 경계 설계(cascade 분리), 격리 수준 선택(배치 vs 사용자 충돌), 스레드풀 분리(알림 fan-out), 자료구조 비교(Set vs Sorted Set), 즉시/최종 일관성 비즈니스 결정. 채용 시장 분석 문서에서 식별한 차별화 역량을 전부 커버한다.

양방향 상호작용의 핵심 기술 패턴(상태 머신 + 동시 상태 전이 제어)은 이미 ChatService의 2단계 입장 구조에서 증명한 바 있다.

경제적 제약(부친 지원 2개월 이내), 4/28 채용박람회, 취업 일정을 고려하면 설계 단계에서 더 시간을 쓰는 것보다 구현 + 트러블슈팅 문서화에 시간을 확보하는 것이 합리적이다.

## 4. 확정된 기능 및 사용자 간 상호작용 체인

### 4.1 확정된 4개 쓰기 경로

1. **리뷰 작성** (기존) — 사용자가 매물에 대해 별점과 텍스트 리뷰를 등록
2. **찜하기 + 인기 알림** (신규) — 사용자가 매물을 찜, 임계값 초과 시 찜 유저 전원에게 알림
3. **리뷰 투표** (신규) — 사용자가 다른 사용자의 리뷰에 도움돼요/별로에요 투표
4. **매물 등록/수정/상태변경** (신규) — 사용자가 매물 정보를 직접 등록·수정·거래완료 처리

### 4.2 상호작용 체인 종합표

| 사용자 A의 행위 | 영향 받는 사용자 | 변경되는 B의 데이터 | 발생하는 기술적 의사결정 |
|---------------|---------------|------------------|---------------------|
| 매물 등록 | 추천 조회 중인 전체 사용자 | 추천 결과에 신규 매물 포함 | Redis 6개 인덱스 동시 갱신, 부분 실패 보상 |
| 매물 가격 수정 | 해당 매물 찜한 N명 | 찜 목록의 가격 정보 변경 + 알림 생성 | 이벤트 순서 보장, fan-out 스레드풀 분리 |
| 매물 거래완료 | 찜한 N명 + 리뷰 작성자 M명 | 찜 목록 상태 변경 + 리뷰 노출 정책 변경 + 인덱스 제거 | cascade 트랜잭션 경계, Soft Delete 정책 |
| 찜하기 | 같은 매물 찜한 N명 (임계값 초과 시) | 인기 급등 알림 생성 | 동시 카운터 정합성, 임계값 판단 원자성 |
| 리뷰 작성 | 같은 매물 찜한 N명 | 새 리뷰 알림 + 추천 점수 변동 | 리뷰 통계 동시 갱신, 즉시/최종 일관성 선택 |
| 리뷰 투표 | 리뷰 작성자 + 같은 매물 조회자 | 리뷰 정렬 순위 변동 + 매물 점수 변동 | cascade 깊이 결정(투표 → 정렬 → 점수 재계산) |
| 매물 동시 등록(중복) | 같은 매물 등록 시도한 다른 사용자 | 등록 실패 + 안내 | 중복 감지 락 전략 3종 비교 |

### 4.3 4개 쓰기 경로 간 간섭 시나리오

**간섭 1: 리뷰 작성 × 리뷰 투표 — 같은 매물의 ReviewScore 동시 변경**

사용자 A가 리뷰를 작성하는 동시에 사용자 B가 같은 매물의 다른 리뷰에 투표한다. 둘 다 해당 매물의 ReviewScore를 변경하려고 한다. 리뷰 작성은 REVIEW_STATISTICS 테이블을 UPDATE하고, 투표는 리뷰 가중치를 변경해서 같은 REVIEW_STATISTICS를 UPDATE한다. 같은 row에 대한 두 가지 다른 비즈니스 경로의 동시 쓰기가 발생한다.

**간섭 2: 찜하기 × 배치 갱신 — 알림과 매물 데이터의 일관성**

사용자 A가 매물을 찜하여 인기 알림이 발생하는 시점에, 배치가 해당 매물의 가격을 갱신 중이다. 알림에 포함되는 매물 정보가 갱신 전인지 후인지에 따라 사용자가 보는 데이터가 달라진다. 두 개의 독립적인 쓰기 트랜잭션이 같은 매물 데이터를 참조하는 격리 문제가 발생한다.

## 5. CPU 바운드 맥락 — ForkJoinPool 적용 지점

### 5.1 bounds 변경에 의한 점수 재계산 연쇄

현재 추천 점수 산출 구조에서, 매물의 priceScore는 해당 지역구의 bounds:{district}:전세에 저장된 minPrice, maxPrice로 정규화된다. 공식은 `priceScore = 100 - ((현재전세금 - minPrice) / (maxPrice - minPrice) × 100)`이다.

사용자가 매물을 등록한다. 이 매물의 전세금이 해당 지역구의 기존 minPrice보다 낮다. 이 순간 bounds:강남구:전세의 minPrice가 변경된다. minPrice가 변경되면 강남구의 모든 기존 매물의 priceScore가 전부 무효화된다. 정규화 기준 자체가 바뀌었기 때문이다.

강남구에 매물이 1,000건 있다고 하면, 1,000건 전부에 대해 priceScore, spaceScore를 재계산하고, 가중치를 적용해서 finalScore를 재산출하고, 지역구 대표 점수를 재산출해야 한다.

### 5.2 이 연산이 CPU 바운드인 이유

네트워크 I/O가 아니다 — Redis에서 매물 데이터를 읽는 것은 초반 1회이고, 이후는 순수 산술 연산이다. 디스크 I/O가 아니다 — Oracle 조회 없이 Redis에서 가져온 데이터로 메모리 내에서 계산한다. 순수 CPU 바운드 연산이다 — 1,000건 × (정규화 3개 항목 + 가중치 적용 + 리뷰 점수 합산 + 최종 점수 산출) = 수천 회의 부동소수점 연산.

### 5.3 ForkJoinPool이 요구되는 이유

기존 CompletableFuture + ThreadPoolExecutor는 Kakao Map API 15개 카테고리 호출에 사용했다. 이것은 I/O 바운드였다. 각 스레드가 대부분의 시간을 네트워크 응답 대기에 소비하므로, 코어 수보다 훨씬 많은 스레드(20개)를 생성하는 것이 정당했고, I/O 전용 풀 분리가 설계 근거였다.

점수 재계산은 성격이 다르다. 네트워크 대기가 0이고, CPU가 100% 활용된다. 이 경우 스레드 수를 코어 수 이상으로 늘리면 컨텍스트 스위칭 오버헤드만 증가하고 처리량은 오히려 감소한다. ForkJoinPool의 work-stealing이 이 워크로드에 적합한 이유는, 지역구별 매물 수가 균일하지 않기 때문이다(강남구 1,000건 vs 도봉구 50건). 정적 분할(ThreadPoolExecutor에 지역구별 태스크 할당)을 하면 강남구 처리 스레드가 병목이 되고 나머지 스레드는 유휴 상태가 된다. ForkJoinPool은 작업이 빨리 끝난 스레드가 느린 스레드의 큐에서 작업을 훔쳐가므로 부하가 자동으로 균형된다.

### 5.4 동일 프로젝트 내 두 가지 병렬 처리 비교

| 항목 | 외부 API 병렬 호출 (기존) | 점수 재계산 병렬 처리 (신규) |
|------|------------------------|------------------------|
| 워크로드 성격 | I/O 바운드 | CPU 바운드 |
| 병목 자원 | 네트워크 RTT | CPU 코어 |
| 스레드풀 | ThreadPoolExecutor(20스레드) | ForkJoinPool(코어 수 = 4) |
| 스레드 수 근거 | I/O 대기 시간 >> 연산 시간이므로 코어 수 대비 5배 | 컨텍스트 스위칭 최소화를 위해 코어 수와 동일 |
| 분할 전략 | 카테고리별 1:1 매핑 (균일 분할) | 지역구별 매물 수 불균일 → work-stealing |
| 장애 격리 | exceptionally() 개별 실패 → 빈 리스트 | RecursiveTask 예외 → 해당 지역구만 기존 점수 유지 |

"왜 여기서는 ForkJoinPool이고 저기서는 ThreadPoolExecutor인가?" — 이 질문에 대해 워크로드 특성 분석(I/O 비율, CPU 점유율), 스레드 수 결정의 정량적 근거(코어 수 × (1 + W/S) 공식), 분할 전략의 차이(균일 vs 불균일 데이터)를 실측 데이터로 설명할 수 있다. 이것이 고도화 계획서에서 지적한 "워크로드 특성별 스레드 풀 분리 설계" 누락 역량을 정확히 채운다.

### 5.5 구현 형태

매물 등록 API의 처리 흐름 끝에서, bounds 변경 여부를 판단한다. 변경이 발생하면 해당 지역구의 전체 매물 점수 재계산을 `@Async("scoreRecalcExecutor")`로 비동기 위임한다. 이 executor의 내부가 ForkJoinPool이다.

```
사용자 매물 등록
  → Oracle INSERT + Redis 인덱스 갱신 (동기, 메인 스레드)
  → bounds 변경 감지 (동기, 메인 스레드)
  → [bounds 변경 시] 점수 재계산 이벤트 발행 (@TransactionalEventListener AFTER_COMMIT)
  → scoreRecalcExecutor (ForkJoinPool, CPU 코어 수)
    → RecursiveTask: 매물 리스트 분할 → 서브태스크 fork → 점수 계산 → join
    → Redis Sorted Set score 일괄 갱신 (Pipeline)
```

사용자 A의 매물 등록 1건이, 같은 지역구 1,000건 매물의 점수를 재계산하는 CPU 바운드 연산을 트리거한다. 이것이 앞서 논의한 "매물 등록 → cascade" 체인의 마지막 단계에 자연스럽게 들어간다. 별도 기능을 추가한 것이 아니라, 매물 등록이라는 기존 기능의 후속 처리에서 CPU 바운드 맥락이 필연적으로 발생한 것이다.

## 6. 자료구조 및 알고리즘 명시적 선택 지점

새 기능을 추가하는 것이 아니라, 이미 확정된 기능의 내부 처리 과정에서 자료구조/알고리즘 선택이 필연적으로 요구되는 지점을 명시적으로 식별한다.

### 6.1 지점 1: 추천 결과 조립 — Top-K 추출 알고리즘 선택

현재 추천 서비스의 최종 단계(C-08/M-08)는 모든 매물의 finalScore를 기준으로 전체 정렬 → 상위 3개 지역구 선정 → 각 지역구 내 상위 3개 매물 추출이다. 설계 명세서 v2에서 이 부분은 "내림차순 정렬"이라고만 기술되어 있고, 구현이 `Collections.sort()` 또는 `Stream.sorted()`일 가능성이 높다.

문제: 강남구에 매물 1,000건, 서울 전체 25개 자치구에 총 11만 건이 있을 때, 최종 응답에 필요한 것은 상위 3개 지역구 × 3개 매물 = 9건뿐이다. 11만 건을 전체 정렬하는 것은 O(N log N)이고, 필요한 것은 상위 9건이다.

**대안 1: `Collections.sort()` — 전체 정렬 후 subList(0, K)**. 시간복잡도 O(N log N), 공간복잡도 O(1)(in-place sort일 경우). 구현이 가장 단순하지만, N이 크고 K가 작을 때 불필요한 연산이 많다.

**대안 2: `PriorityQueue`(min-heap, size K) — 크기 K의 최소 힙을 유지하면서 N개를 순회**. 현재 원소가 힙의 최솟값보다 크면 poll() + offer(). 시간복잡도 O(N log K), 공간복잡도 O(K). K=3이면 log K ≈ 1.58이고 log N(N=1000) ≈ 10이므로, 이론적으로 약 6배 차이가 난다.

**대안 3: QuickSelect(Introselect) — K번째 원소를 O(N) 평균으로 찾고, 상위 K개를 추출**. 시간복잡도 O(N) 평균이지만, 최악 O(N²)이고 정렬 순서가 보장되지 않아 추가 정렬이 필요하다.

선택 기준 — 비즈니스 맥락에 의한 결정: 추천 API는 사용자가 조건을 입력하고 응답을 기다리는 동기 요청이다. N=1,000(지역구 단위)에서 K=3이면 PriorityQueue의 이점이 명확하다. 하지만 Fallback 확장 검색에서 조건이 완화되어 N이 작아지면(예: 50건), 힙 유지 오버헤드가 단순 정렬보다 오히려 클 수 있다. N의 분포에 따라 전략을 분기하는 것 자체가 설계 의사결정이 된다. JMH 벤치마크로 N의 임계값(crossover point)을 실측해서 결정한다.

### 6.2 지점 2: 점수 재계산 시 지역구별 매물 그룹핑 — HashMap vs TreeMap

bounds 변경 → 지역구 전체 매물 점수 재계산 시나리오에서, Redis에서 가져온 매물 리스트를 지역구별로 그룹핑해야 한다.

**`HashMap<String, List<PropertyWithScore>>`** — 그룹핑 O(N), 이후 지역구 대표 점수 기준 정렬이 별도 필요. 전체 O(N + M log M) (M = 지역구 수 = 25).

**`TreeMap<Double, List<PropertyWithScore>>`** — 키를 지역구 대표 점수로 설정하면 삽입 시 자동 정렬. 하지만 점수가 재계산되므로 삽입 시점에 정확한 키를 알 수 없다. 즉 두 번의 패스가 필요하다(1차: 그룹핑 + 점수 계산, 2차: 정렬).

**`LinkedHashMap`** — 삽입 순서 유지. 지역구를 고정된 순서로 처리하고 싶을 때 유리하지만 점수 기반 정렬에는 도움이 되지 않는다.

이 선택은 단독으로는 사소해 보이지만, Top-K 추출과 결합되면 구조가 달라진다: 25개 지역구를 전부 점수 계산한 뒤 상위 3개만 뽑을 것인가? 아니면 계산하면서 PriorityQueue(size 3)로 즉시 필터링할 것인가? 후자는 메모리 사용량을 줄이지만, 각 지역구 내부의 매물 정렬은 별도로 해야 한다. 데이터 처리 파이프라인의 전체 구조(one-pass vs two-pass)가 자료구조 선택에 의해 결정된다.

### 6.3 지점 3: 알림 중복 제거와 병합 — 자료구조가 비즈니스 규칙을 구현하는 사례

사용자 A가 매물을 등록하고, 바로 이어서 가격을 수정한다. 해당 매물을 찜한 사용자 B에게 알림이 2건 발생한다: "새 매물 등록" + "가격 변동". 짧은 시간 내에 동일 사용자에게 동일 매물에 대한 알림 2건을 보내는 것은 UX상 바람직하지 않다. 시간 윈도우(예: 5분) 내의 알림을 병합해야 한다.

**`HashSet<NotificationKey>`** — 단순 중복 제거. 같은 (userId, propertyId) 조합이면 후속 알림을 무시. 하지만 "새 매물" 알림과 "가격 변동" 알림은 종류가 다르므로, 무시가 아니라 병합이 필요하다.

**`HashMap<NotificationKey, NotificationPayload>`** — 기존 알림 페이로드에 새 이벤트 정보를 머지. "래미안그레이튼: 새 매물 등록 + 가격 1.8억으로 변경"처럼 병합된 알림 생성. 시간 윈도우 만료 시 flush.

**`LinkedHashMap(accessOrder=true, removeEldestEntry)`** — LRU 방식으로 시간 윈도우를 구현. 가장 오래된 엔트리부터 자동 제거되므로, TTL 기반 윈도우를 Java 레벨에서 구현할 수 있다. 별도 스케줄러 없이 put() 호출 시 자동으로 만료된 알림을 flush.

**`PriorityQueue<TimestampedNotification>`** — 타임스탬프 기반 최소 힙으로 만료 순서를 관리. 가장 오래된 알림이 항상 힙 루트에 위치하므로, 폴링 시 O(1)으로 만료 여부를 확인할 수 있다.

**Redis SETEX** — `SETEX notification:{userId}:{propertyId} 300 {payload}` 로 TTL 기반 병합을 Redis에 위임. 기존 키가 있으면 페이로드를 머지하고 TTL을 리셋.

이것이 자료구조 비교 선택의 가장 좋은 사례인 이유: 알림 병합이라는 비즈니스 요구사항 하나에 대해 5개 대안이 존재하고, 각 대안은 시간복잡도뿐 아니라 메모리 사용 패턴, 구현 복잡도, 스레드 안전성(ConcurrentHashMap vs synchronized)에서 차이가 난다. "왜 Redis가 아닌 Java 자료구조를 선택했는가" 또는 "왜 Java가 아닌 Redis를 선택했는가"라는 저장소 선택까지 포함한 5방향 비교가 가능하다.

### 6.4 지점 4: 리뷰 키워드 매칭 — 알고리즘 선택

현재 키워드 추출은 Enum 사전(19개 키워드) 기반 단순 String.contains() 매칭이다. 사용자 매물 등록으로 데이터가 증가하고 리뷰가 많아지면, 키워드 사전 확장이 자연스럽다(19개 → 100개+).

키워드가 100개가 되면, 리뷰 텍스트 1건에 대해 contains()를 100번 호출하는 것은 O(T × K × L) (T: 텍스트 길이, K: 키워드 수, L: 평균 키워드 길이)이다.

**대안 1 — 현행 유지(contains() 루프)**: K가 작을 때 단순하고 빠르다. K=19에서는 최적이지만 K=100+에서 성능 저하.

**대안 2 — HashSet 기반 토큰 매칭**: 텍스트를 토큰(공백 분리)으로 분할하고, 각 토큰을 HashSet에서 O(1) 조회. 시간복잡도 O(T + K). 하지만 "층간소음"처럼 띄어쓰기 없는 복합 키워드를 놓친다.

**대안 3 — Aho-Corasick 알고리즘**: 키워드 사전 전체로 Trie + failure link를 구축하고, 텍스트를 1회 스캔으로 모든 키워드를 동시 매칭. 시간복잡도 O(T + K + M) (M: 매칭 수). 복합 키워드도 처리 가능. 사전 구축 비용 O(K × L)이 있지만 사전이 정적이므로 애플리케이션 시작 시 1회만 구축.

K의 크기에 따라 최적 알고리즘이 달라지는 전형적인 교차점(crossover point) 문제이고, 이것을 JMH 벤치마크로 K=10/50/100/200에서 실측하면 알고리즘 선택의 정량적 근거가 된다.

### 6.5 자료구조/알고리즘 지점 종합

| 지점 | 대안 수 | 핵심 선택 기준 | 연결되는 기존 기능 |
|------|--------|------------|---------------|
| Top-K 추출 | 3개 (sort/heap/quickselect) | N 대비 K의 비율, 동기 응답 레이턴시 | 추천 서비스 최종 정렬 |
| 지역구 그룹핑 | 3개 (HashMap/TreeMap/one-pass) | 파이프라인 구조(단일/이중 패스) | 점수 재계산 (ForkJoinPool) |
| 알림 병합 | 5개 (HashSet/HashMap/LHM/PQ/Redis) | 시간 윈도우, 스레드 안전성, 저장소 선택 | 찜 기반 알림 fan-out |
| 키워드 매칭 | 3개 (contains/HashSet/Aho-Corasick) | 사전 크기 K의 증가에 따른 교차점 | 리뷰 키워드 추출 |

이 4개 지점은 새로운 기능을 추가한 것이 아니라, 이미 확정된 기능의 내부 구현에서 자료구조/알고리즘 선택이 비즈니스 맥락에 의해 요구되는 지점을 명시적으로 식별한 것이다. 포트폴리오에서 "이 알고리즘을 사용했다"가 아니라 "이 비즈니스 요구사항의 시간/공간 제약 하에서 N개 대안을 실측 비교하여 선택했다"는 구조가 된다.

### 7.2 포트폴리오 트러블슈팅 예상 건수

기존 포트폴리오 6건(성능 측정/개선 5건 + 트레이드오프 기반 선택 1건)에, "비즈니스 요구사항 충돌 → 대안 분석 → 트레이드오프 → 선택 근거 → 정량 결과" 구조의 트러블슈팅이 최소 4~5건 추가된다.

추가 예상 트러블슈팅:
1. 매물 동시 등록 — 중복 감지 락 전략 3종 비교 실측
2. 찜 카운터 vs 투표 카운터 — 동일 기술 대안에서 비즈니스 맥락에 따라 다른 선택을 한 사례
3. 알림 fan-out — 트랜잭션 경계 분리 + 스레드풀 설계 정량 근거
4. bounds 변경 점수 재계산 — I/O 바운드 vs CPU 바운드 워크로드에 따른 스레드풀 선택
5. Top-K 추출 or 키워드 매칭 — 자료구조/알고리즘 교차점 실측
