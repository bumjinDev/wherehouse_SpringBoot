# F004 슬롯 예약 동시성 제어 — 학습 정리

본 문서는 F004 시나리오 1(같은 슬롯에 대한 동시 예약 경합)의 동시성 제어 후보를 학습·비교·검토한 내용을 정리한다. 측정 수치는 별도 파일(`결과/F004_측정결과.xlsx`)에 있고, 본 문서는 후보의 메커니즘과 선택 근거에 대한 고민을 담는다. 측정 맥락은 후보 비교에 필요한 배경 수준으로만 다룬다.

---

## 1. 시나리오 1의 본질

### 1.1 Check-then-Act race

`createReservation`에 같은 슬롯 식별자로 N개 호출이 거의 동시에 도착한다. 각 호출은 독립 트랜잭션이고 모두 같은 흐름을 따른다.

1. 슬롯 행 조회 (`SELECT`)
2. 슬롯이 `AVAILABLE`인지 검증
3. 슬롯 상태 전이 (`UPDATE` status → `RESERVED`)
4. 예약 행 등록 (`INSERT`)

조회(1)와 변경(3) 사이에 시간 간격이 있고, 그 사이에 추가 검증(자기 매물·중복·시간 겹침)이 들어간다. 이 간격에 다른 트랜잭션이 도착하면 그 트랜잭션도 같은 옛 값(`AVAILABLE`)을 보고 검증을 통과한 뒤 변경으로 진입한다. N개 트랜잭션이 모두 같은 옛 상태를 본 채 변경을 시도하는 race다.

검사(Check, "AVAILABLE인가")와 실행(Act, "RESERVED로 바꾼다")의 시점이 분리되면 검사 결과가 실행 시점에도 유효하다는 가정이 깨진다. 단일 스레드에서는 항상 참이지만 동시성 환경에서는 보장되지 않는다.

### 1.2 유형 1과 유형 2

시나리오 문서 2.3절의 구분이다.

**유형 1 — 부재 확인 + 저장.** 행이 아직 없는 상태에서 두 트랜잭션이 같은 식별자로 INSERT를 시도한다. 경합 단위가 식별자이므로 DB의 식별자 기반 메커니즘(유일 인덱스, PK)이 두 번째 INSERT를 거부한다. 예: 매물 등록 시 같은 `property_id`로 INSERT 두 건.

**유형 2 — 상태 읽기 + 판단 + 상태 쓰기.** 행은 이미 존재하고 어떤 상태(`AVAILABLE`)다. 두 트랜잭션이 그 상태를 읽고, 그 판단을 근거로 다른 상태(`RESERVED`)로 UPDATE한다. 경합 단위가 식별자가 아니라 상태값이다. UPDATE 시 자동 행 락은 두 UPDATE를 순차 실행시킬 뿐 두 번째 UPDATE를 거부하지 않는다. 결국 두 트랜잭션 모두 UPDATE를 완료하고 INSERT 단계로 진입하며, 별도의 DB 차원 메커니즘이 없으면 한 슬롯에 두 개의 `CONFIRMED` 예약이 남는다.

F004는 유형 2다. 유형 1의 식별자 기반 메커니즘만으로는 race 자체를 막지 못하고 결과적 정합성만 보장한다.

### 1.3 비즈니스 규칙

- **규칙 1 — 슬롯당 정원 1.** 한 슬롯의 확정 예약은 어느 시점에나 최대 1건. N개 동시 요청에도 두 명이 확정되는 사건은 0건이어야 한다.
- **규칙 2 — 거부의 즉시 통보.** 거부된 탐색자는 거부 사실·사유·대체 행동 진입점을 즉시 받아야 한다.

규칙 1은 정합성(잔여 `CONFIRMED` 건수)으로, 규칙 2는 거부 응답의 사유 명확성으로 판정된다. 이 두 규칙이 후보 평가의 trade-off 축이다.

## 2. 코드 흐름과 race 구간 (배경)

`VisitReservationWriteService.createReservation`의 단계:

| 단계 | 작업 |
|---|---|
| 조회 | `findById`로 슬롯·윈도우 |
| AVAILABLE 검증 | 슬롯 상태가 `AVAILABLE`인가 |
| 추가 검증 | startTime / 자기 매물 / 동일 매물 중복 / 시간 겹침 |
| 슬롯 UPDATE | status: `AVAILABLE` → `RESERVED` |
| 예약 INSERT | `VISIT_RESERVATION`에 `CONFIRMED` 행 추가 |

조회부터 UPDATE 직전까지가 race 구간이다. 조회 결과에 모든 검증이 의존하고, UPDATE의 변경이 INSERT의 전제가 된다.

측정용 로그 지점은 세 곳이다. CP1(AVAILABLE 검증 통과 직후), CP2(슬롯 UPDATE 호출 직후), CP3(예약 INSERT 호출 직후). 후보별로 race가 어느 단계에서 막히는지를 CP 통과 인원으로 관측한다.

측정에는 두 가지 한계가 있다. 첫째, CP1이 검증 통과 직후라서 검증에서 throw된 트랜잭션은 CP1 이전에 끝나 로그에 안 남는다. 둘째, JPA/Hibernate의 dirty checking 때문에 `slotRepository.save(slot)`는 영속성 컨텍스트에 dirty 표시만 하고 실제 UPDATE SQL은 트랜잭션 flush(통상 commit) 시점에 발송될 수 있다. 따라서 CP2 시각은 "JPA가 UPDATE를 예약한 시점"이지 "Oracle이 행 락을 잡고 UPDATE를 실행한 시점"이 아니다. 실제 행 락 보유 시간을 정확히 보려면 `V$LOCK`/`V$SESSION_WAIT` 폴링이 필요하나 본 시리즈는 미측정이다. 이 한계는 SERIALIZABLE 후보의 예상 동작 해석에 영향을 준다(5.5 참조).

## 3. 측정 baseline (배경)

후보 비교의 양 극단을 데이터로 확보했다.

- **Step 1 — 부분 유일 인덱스만 적용.** `@Transactional`만(옵션 없음), DB에 `UQ_VISIT_RESERVATION_CONFIRMED_SLOT` 적용. N=5에서 CP1·CP2는 5건 통과(race 발생), CP3에서 1건만 키 등재·4건 ORA-00001 거부, 잔여 `CONFIRMED` 1건. race 자체는 발생하나 INSERT 단계의 DB 게이트가 결과적 정합성을 보장한다.
- **Step 2 — 제어 부재 baseline.** 동일 코드, 부분 유일 인덱스 제거. N=5에서 CP1·CP2·CP3 모두 5건 통과, DB 거부 0건, 잔여 `CONFIRMED` 5건 → 규칙 1 위반을 직접 관측. 1.2절의 명제가 실측으로 확인됐다.

거부 트랜잭션이 ORA-00001 처리·핸들러 통과 비용을 추가로 지불하므로, Step 1의 응답 시간은 Step 2보다 분산이 크다.

## 4. 동시성 제어 후보 — 6 카테고리

| 카테고리 | 핵심 메커니즘 | race 차단 위치 |
|---|---|---|
| A. 부분 유일 인덱스 | unique 위반 예외가 INSERT를 거부 | INSERT (마지막) |
| B. 비관적 락 (`SELECT FOR UPDATE`) | 조회 시점에 행 X-락 획득, 후속 트랜잭션의 같은 SELECT가 대기 | SELECT (맨 앞) |
| C. 낙관적 락 (`@Version`) | 락 추가 없이 version 비교, UPDATE 시 mismatch면 예외 | UPDATE (중간) |
| D. 상태 조건부 UPDATE | `UPDATE ... WHERE status='AVAILABLE'`, affected=0이면 거부 | UPDATE (중간) |
| E. SERIALIZABLE 격리 수준 | snapshot 기반 충돌 검출, 충돌 시 ORA-08177 | UPDATE (중간) |

## 5. 후보별 메커니즘

### 5.A 부분 유일 인덱스

```sql
CREATE UNIQUE INDEX UQ_VISIT_RESERVATION_CONFIRMED_SLOT
    ON VISIT_RESERVATION (
        CASE WHEN STATUS = 'CONFIRMED' THEN SLOT_ID END
    );
```

`STATUS='CONFIRMED'`인 행만 인덱스 키(`SLOT_ID`)를 가지며, 그 외 상태의 행은 NULL 키가 되어 제약에서 빠진다. Oracle은 인덱스 키 전체가 NULL인 항목을 저장하지 않으므로 비-`CONFIRMED` 행은 유일성 판정에 참여하지 않는다. 한 슬롯에 예약 행이 누적돼도 `CONFIRMED`는 슬롯당 최대 1건임이 DB 차원에서 보장된다.

A는 race 자체를 막지 않는다. N 트랜잭션이 CP1·CP2·CP3까지 다 진행하고, 마지막 INSERT 단계에서 DB 게이트가 후착을 `DataIntegrityViolationException`으로 거부한다(409, E7002로 변환). 추가 코드가 없는 대신 거부 트랜잭션이 검증·UPDATE 비용을 다 지불한 뒤 INSERT에서 ROLLBACK하므로 거부 비용이 큰 편이다.

### 5.B 비관적 락 (`SELECT FOR UPDATE`)

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT s FROM VisitSlotEntity s WHERE s.slotId = :slotId")
Optional<VisitSlotEntity> findByIdForUpdate(@Param("slotId") Long slotId);
```

실행 SQL은 `SELECT * FROM VISIT_SLOT WHERE SLOT_ID = ? FOR UPDATE`. SELECT 시점에 행 X-락을 잡고 트랜잭션 종료까지 유지한다. 후속 트랜잭션의 같은 행에 대한 `SELECT FOR UPDATE`/`UPDATE`/`DELETE`는 대기한다.

차단이 흐름의 맨 앞(SELECT)에서 일어난다. T1이 락을 쥔 동안 T2의 `SELECT FOR UPDATE`는 대기하고, T1 commit으로 락이 풀리면 T2가 행을 받는다. 그 시점의 상태는 이미 `RESERVED`이므로 T2는 AVAILABLE 검증에서 throw한다. 거부 트랜잭션은 락 대기 시간 + 검증 비용을 지불한다.

거부가 비즈니스 검증에서 발생하므로 SELECT 결과를 보고 명확한 사유 메시지를 만들 수 있다(규칙 2에 유리). 단일 행 락만 잡으므로 본 시나리오에서 데드락 위험은 낮으나, 여러 행을 다른 순서로 잡는 일반 상황에서는 데드락 위험이 있다.

### 5.C 낙관적 락 (`@Version`)

```java
@Entity
public class VisitSlotEntity {
    @Id private Long slotId;
    private VisitSlotStatus status;
    @Version private Long version;   // 추가
}
```

JPA가 UPDATE에 version 비교를 자동 삽입한다.

```sql
UPDATE VISIT_SLOT
   SET STATUS = 'RESERVED', VERSION = VERSION + 1
 WHERE SLOT_ID = ? AND VERSION = ?   -- SELECT 시점의 version
```

전략 차원의 추가 락 키워드는 없다. 다만 이 UPDATE 자체는 Oracle의 자동 행 X-락을 거는 일반 DML이다(6.1 참조). 같은 슬롯 경합에서 후착의 UPDATE는 선착의 행 락이 풀릴 때까지 대기하고, 락 해제 후 재평가에서 `VERSION` 불일치로 affected=0가 되어 `OptimisticLockException`이 발생한다. 무경합이면 락 대기가 없고 비용도 낮다.

version 값은 세 곳에 함께 존재한다. 본체는 DB의 `VERSION` 컬럼이고, Java 엔티티 인스턴스에 사본이 있으며, EntityManager의 1차 캐시(영속성 컨텍스트)에 SELECT 시점 스냅샷이 보관된다. UPDATE의 `WHERE VERSION=v0`의 `v0`는 1차 캐시 스냅샷의 값이다.

차단이 UPDATE 단계에서 일어나며, 거부 사유가 `OptimisticLockException`이라 비즈니스 사유로 직접 매핑되지 않는다. 엔티티 스키마 변경(`VERSION` 컬럼)과 충돌 시 retry 처리가 필요하고, 그 한 행의 version만 보호한다.

### 5.D 상태 조건부 UPDATE

조회·검증·변경을 분리된 코드 단계가 아니라 단일 `UPDATE ... WHERE` 한 문장으로 묶는다.

```java
@Modifying
@Query("UPDATE VisitSlotEntity s SET s.status = ...RESERVED " +
       " WHERE s.slotId = :slotId AND s.status = ...AVAILABLE")
int transitionAvailableToReserved(@Param("slotId") Long slotId);
```

```java
int affected = slotRepository.transitionAvailableToReserved(slotId);
if (affected == 0) throw SlotUnavailableException.alreadyReserved(...);
```

작동 근거는 Oracle UPDATE의 처리 방식이다(6.2 참조). 같은 슬롯에 두 UPDATE가 동시 도착하면 T1이 행 X-락을 잡고 변경·commit, T2는 락 대기 후 락을 얻는다. T2가 락을 얻은 뒤 Oracle이 WHERE를 다시 평가하는데, 이때 status가 이미 `RESERVED`라 `AND STATUS='AVAILABLE'`이 거짓이 되어 affected=0가 된다.

현재 코드(Step 2)의 UPDATE는 `WHERE SLOT_ID=?`로 PK 조건만 있어, 재평가가 항상 참이라 후속 트랜잭션이 모두 affected=1로 같은 행을 덮어쓴다. D는 거기에 `AND STATUS='AVAILABLE'` 한 줄을 더해, 락 후 재평가가 거짓이 되도록 만든다. 즉 현재 코드의 UPDATE 단계엔 보장 지점이 없었고(보장 지점은 INSERT의 unique뿐이었음), D는 그 보장 지점을 UPDATE의 WHERE 절에 추가하는 방식이다.

추가 락 키워드·`@Version` 컬럼·retry가 없고, 행 락 보유 시간이 UPDATE 실행 시점으로 짧다. `WHERE status='AVAILABLE'`이 비즈니스 규칙을 SQL에 직접 표현한다. 단점은 거부 사유 구분이 어렵다는 점이다. affected=0만으로는 "방금 점유됨", "이미 CLOSED", "WITHDRAWN", "잘못된 slot_id"를 구분하지 못한다. 사유를 안내하려면 affected=0 직후 별도 SELECT가 필요하다(규칙 2와 충돌). 또 `@Modifying @Query`는 영속성 컨텍스트의 엔티티 상태와 어긋날 수 있어 같은 트랜잭션에서 슬롯을 다시 다루면 `refresh` 또는 `clearAutomatically=true`가 필요하다.

### 5.E SERIALIZABLE 격리 수준

```java
@Transactional(isolation = Isolation.SERIALIZABLE)
public ReservationCreateResponseDto createReservation(...) { ... }
```

Oracle의 SERIALIZABLE은 ANSI 표준의 직렬화 가능 실행 정의와 다른 snapshot isolation 기반 구현이다. 트랜잭션 시작 시점의 SCN 기준으로 데이터 스냅샷을 보고, UPDATE 시 "내가 본 행이 내 트랜잭션 시작 이후 commit된 다른 트랜잭션에 의해 바뀌었는가"를 검출한다. 검출되면 `ORA-08177`로 트랜잭션을 ROLLBACK한다. 충돌 검출 모델 자체는 C(`@Version`)와 유사하나, SERIALIZABLE은 트랜잭션 전체의 모든 읽기에 일관성을 적용한다는 점에서 더 넓다.

여기서 중요한 점은, SERIALIZABLE이 읽기 락을 없앨 뿐 쓰기 락을 없애지 않는다는 것이다(6.1 참조). 같은 슬롯 경합에서 E의 실제 동작은:

1. T1·T2가 각자 스냅샷에서 SELECT(읽기 락 없음), 둘 다 옛 상태 `AVAILABLE`을 보고 검증 통과.
2. T1이 UPDATE에서 행 X-락을 잡고 변경·commit.
3. T2의 UPDATE는 T1의 락이 풀릴 때까지 **대기**한다. T1 commit으로 락이 풀리면 T2가 깨어나고, 자기 스냅샷 이후 commit된 변경이 검출되어 `ORA-08177`이 발생한다.

즉 E는 "모두 락 없이 병렬로 진행하다가 UPDATE에서 무효화"가 아니다. 후착의 UPDATE는 선착의 행 락에서 대기한 뒤 ORA-08177을 받는다. 거부 트랜잭션의 응답 시간에는 락 대기 시간이 포함된다. 읽기 단계는 락 없이 동시에 진행하지만, 쓰기 단계는 같은 행에서 직렬화된다.

예상 동작에는 단서가 붙는다. 4건의 후착이 ORA-08177을 받는 시점은, JPA가 UPDATE를 commit-flush까지 지연하느냐에 따라 달라진다. 지연되면(2절 한계) UPDATE SQL이 commit 시점에 나가므로 CP2·CP3는 save() 호출 직후 로그라 통과로 찍히고 ORA-08177은 commit-flush에서 발생한다. 지연되지 않으면 UPDATE 문장(CP2 이전)에서 락 대기 후 ORA-08177이 발생하고 commit까지 가지 않는다. 본 시리즈의 코드 경로(JPA 지연 flush)에서는 전자에 가깝다.

코드 변경이 어노테이션 한 줄로 작고, 트랜잭션의 모든 SQL에 일관성이 적용돼 슬롯 외 검증의 race도 함께 보호된다. 단점은 모든 읽기에 SCN 검증 비용이 들어 처리량 영향이 크고, `ORA-08177`이 비즈니스 예외가 아니라 `CannotSerializeTransactionException`이라 핸들러 변환과 retry가 필요할 수 있다는 점이다.

## 6. Oracle 공통 토대

후보 비교의 전제가 되는 Oracle의 동작이다. 후보별 차이는 모두 이 토대 위에서 갈린다.

### 6.1 모든 쓰기는 행 X-락을 건다 — 격리 수준·전략과 무관

Oracle에서 UPDATE / DELETE / `SELECT FOR UPDATE`는 변경 대상 행에 자동으로 행 X-락을 걸고, 트랜잭션 commit/rollback까지 유지한다. 이는 격리 수준(READ COMMITTED든 SERIALIZABLE이든)과 무관하고, 낙관적/비관적 전략 선택과도 무관하다. 같은 행에 두 트랜잭션이 쓰기 락을 동시에 보유하는 격리 수준은 Oracle에 없다.

따라서 같은 슬롯 경합에서 B·C·D·E 중 어느 것도 후착의 쓰기가 락 대기를 면하지 못한다. 후착의 쓰기는 선착의 락이 풀릴 때까지 대기한다. 후보들이 갈리는 지점은 두 가지다. 첫째, 락을 어느 단계에서 잡는가(B는 SELECT, C·D·E는 UPDATE). 둘째, 락 해제 후 후착에게 무엇이 일어나는가(B는 재조회 후 비즈니스 검증 throw, C·D는 affected=0, E는 ORA-08177). "락 대기가 있는가 없는가"가 아니라 "락을 언제 잡고 그 뒤 어떻게 거부되는가"가 차이다.

### 6.2 Oracle UPDATE의 내부 동작

`UPDATE VISIT_SLOT SET status='RESERVED' WHERE slot_id=? AND status='AVAILABLE'`의 처리:

1. 인덱스로 `slot_id=?` 후보 행을 찾는다.
2. 후보 행에 X-락을 시도한다. 다른 트랜잭션이 X-락을 쥐고 있으면 여기서 대기한다.
3. 경합으로 대기했다가 락을 얻은 경우, Oracle은 statement를 restart해 행을 다시 읽고 WHERE를 재평가한다. 선행 트랜잭션이 commit한 뒤라면 status가 `RESERVED`라 조건이 거짓이 된다.
4. 조건에 맞는 행만 UPDATE하고, 불일치 행은 affected에서 제외한다.

WHERE 재평가(restart)는 별도의 SELECT 문으로 변환되는 게 아니라 UPDATE 처리 흐름의 일부다. restart는 경합으로 락 대기·변경이 발생한 경우에 트리거되는 동작이며, 무경합 경로에서는 일어나지 않는다. D가 작동하는 근거가 바로 이 재평가다. WHERE에 `AND status='AVAILABLE'`이 박혀 있으면 재평가에서 그 조건이 확인되어 후착이 affected=0가 된다.

### 6.3 readers don't block writers

Oracle의 원칙은 "읽는 쪽이 쓰는 쪽을 막지 않고, 쓰는 쪽이 읽는 쪽을 막지 않는다"이다. 락 없는 일반 SELECT는 다른 트랜잭션의 작업을 차단하지 않고, 자기 스냅샷의 데이터를 본다. `SELECT FOR UPDATE`/`UPDATE`/`DELETE`는 행 X-락을 잡고, 같은 X-락을 잡으려는 트랜잭션만 대기시킨다. 그래서 E에서 후착의 읽기는 차단되지 않지만(스냅샷에서 옛 `AVAILABLE`을 봄), 후착의 쓰기는 같은 행 X-락에서 대기한다.

### 6.4 격리 수준의 적용 단위와 Oracle의 지원 범위

격리 수준은 세션이 가진 속성이다. dedicated server mode에서 클라이언트 스레드·커넥션·서버 프로세스·세션이 1:1:1:1로 매칭되고, 각 세션은 자기 격리 수준을 가지며 다른 세션에 영향을 주지 않는다. Oracle에는 instance-level의 기본 격리 수준 설정이 없다. 모든 세션의 default가 READ COMMITTED이고, 애플리케이션이 트랜잭션마다 명시적으로 끌어올리거나 그대로 둔다.

Spring `@Transactional(isolation=...)`은 그 트랜잭션이 빌린 커넥션의 세션에만 영향을 주고, 커넥션 반환 시 Spring의 트랜잭션 인프라가 이전 격리 수준으로 복원한다. 동시 N 트랜잭션은 N개의 독립 세션에서 각자 시작한다.

Oracle이 지원하는 격리 수준은 READ COMMITTED와 SERIALIZABLE 둘뿐이다. READ UNCOMMITTED와 REPEATABLE READ는 지원하지 않으며, 이를 요청해도 다른 수준으로 조용히 승격/하향되지 않고 JDBC 드라이버가 예외를 던진다. 즉 `@Transactional(isolation=REPEATABLE_READ)`를 Oracle 위에 붙이면 SERIALIZABLE로 동작하는 게 아니라, 커넥션의 `setTransactionIsolation` 호출에서 예외가 발생해 트랜잭션이 시작되지 못한다. Spring `Isolation` enum에는 5개 값이 있지만, Oracle에서 실제로 동작하는 것은 `DEFAULT`(=READ COMMITTED), `READ_COMMITTED`, `SERIALIZABLE`이다.

각 격리 수준이 막는 anomaly(ANSI 기준):

| 격리 수준 | Dirty Read | Non-Repeatable Read | Phantom Read |
|---|:---:|:---:|:---:|
| READ UNCOMMITTED | 허용 | 허용 | 허용 |
| READ COMMITTED | 방지 | 허용 | 허용 |
| REPEATABLE READ | 방지 | 방지 | 허용 |
| SERIALIZABLE | 방지 | 방지 | 방지 |

### 6.5 측정 시점의 격리 수준 (배경)

Step 1·2의 `createReservation`은 옵션 없는 `@Transactional`이고, `application.yml`에 `transactionIsolation` 명시가 없으므로 `Isolation.DEFAULT` → Oracle default인 READ COMMITTED로 동작했다. Step 2에서 5건 모두 commit되는 결과는 READ COMMITTED에서 예상되는 lost update 패턴과 일치한다. 측정 결과 자체는 메커니즘과 정합하지만, 어느 격리 수준에서 측정했는지를 보고서에 명시하지 않으면 해석이 흐려진다. 특히 E(SERIALIZABLE) 회차와 비교할 때 두 회차의 격리 수준 차이가 명시되어야 비교가 성립한다. 이를 위해 회차별 메타(격리 수준 / propagation / UQ 제약 / 동시성 제어 기법)를 보고서에 명시했다.

## 7. 후보 비교

### 7.1 5축 비교

| 축 | A 부분 유일 인덱스 | B FOR UPDATE | C @Version | D 상태 조건부 UPDATE | E SERIALIZABLE |
|---|---|---|---|---|---|
| race 차단 위치 | INSERT (마지막) | SELECT (맨 앞) | UPDATE (중간) | UPDATE (중간) | UPDATE (중간) |
| 락 동작 | unique 위반 예외 (행 락 아님) | SELECT 시점 명시적 행 X-락 | UPDATE의 자동 행 X-락 + version 비교 | UPDATE의 자동 행 X-락 + WHERE 재평가 | UPDATE의 자동 행 X-락 + SCN 충돌 검출 |
| 적용 범위 | DB 인덱스 (스키마 영구) | 그 SELECT의 그 행 | 그 행의 version 컬럼 | 그 UPDATE 한 번 | 트랜잭션 전체 |
| 거부 형태 | `DataIntegrityViolationException` → 409 | 재조회 후 비즈니스 검증 throw | affected=0 → `OptimisticLockException` | affected=0 → 자바에서 throw | `ORA-08177` → ROLLBACK |
| 코드 변경 | 0 (DB 인덱스만) | repo에 `@Lock` | 엔티티에 `@Version` + retry | repo의 `@Modifying @Query` + 호출부 | 메서드에 `@Transactional(isolation=SERIALIZABLE)` |

C·D·E의 UPDATE는 모두 6.1의 자동 행 X-락을 거는 동일한 DML이다. 표의 "락 동작"은 전략이 그 위에 무엇을 더하는가의 차이다. A만 행 락이 아니라 INSERT 단계의 인덱스 유일성 위반으로 거부한다.

### 7.2 B와 E의 차이 — 같은 결과, 다른 직렬화 지점

B와 E는 같은 결과(1건 commit, 4건 거부)를 만들고, 둘 다 같은 슬롯 행의 쓰기에서 직렬화된다. 차이는 직렬화 지점과 거부 형태다.

- **B는 직렬화가 SELECT 단계에서 일어난다.** 후착의 `SELECT FOR UPDATE`가 락 대기에 들어가고, 락을 얻은 뒤 행을 재조회해 `RESERVED`를 보고 비즈니스 검증에서 throw한다. 거부 사유가 명확하다.
- **E는 직렬화가 UPDATE 단계에서 일어난다.** 후착의 읽기는 스냅샷에서 옛 `AVAILABLE`을 보고 검증을 통과하지만, UPDATE가 선착의 행 락에서 대기하고 락 해제 후 `ORA-08177`을 받는다. 거부 사유가 시스템 예외다.

두 후보 모두 거부 트랜잭션이 락 대기 비용을 지불한다. B는 흐름 앞쪽(SELECT)에서 대기하고 검증으로 거부되며, E는 흐름 중간(UPDATE)까지 진행한 뒤 대기하고 충돌로 거부된다. "B는 직렬·E는 병렬"이 아니라 "둘 다 쓰기에서 직렬화하되 직렬화 지점과 거부 형태가 다르다"가 정확한 대비다.

### 7.3 C와 E의 닮음

| 단계 | E (Oracle SCN) | C (@Version) |
|---|---|---|
| SELECT | 행과 함께 그 행의 SCN을 가져옴 | 행을 읽어 1차 캐시에 저장 (version 포함) |
| UPDATE 시 판정 | 내가 본 SCN 이후 commit이 있었는가 → ORA-08177 | WHERE의 version 비교 → affected=0 시 OptimisticLockException |
| 비교 기준 보관 | Oracle 내부 트랜잭션 메타데이터 | EntityManager의 1차 캐시 |
| 비교 키 | 행마다 자동 부여된 SCN (DB 관리) | 컬럼 값 (애플리케이션 정의) |
| 적용 범위 | 트랜잭션 안의 모든 행·검증 (넓고 자동) | 그 한 행의 version (좁고 정밀) |

둘 다 "SELECT 시점의 키를 보관했다가 UPDATE 시점에 DB 현재 값과 비교"하는 충돌 검출 모델이다. 키를 어디에 두느냐(DB가 자동 / 애플리케이션이 명시)와 적용 범위가 다르다.

### 7.4 격리 수준과 B/C는 다른 레이어

격리 수준(E)은 트랜잭션 시작 시 "다른 트랜잭션 변경을 어느 정도 보일지"의 읽기 일관성 정책이고, 그중 SERIALIZABLE만 쓰기 충돌까지 검출한다. B(명시적 락)와 C(version 비교)는 그 위에 얹는 별도 메커니즘이다. 격리 수준이 무엇이든 B·C는 그 위에서 작동한다. B/C가 E의 일종이 아니라 다른 트랙이다.

### 7.5 unique 제약과의 직교성

트랜잭션은 `@Transactional`로 항상 동작하며 unique 제약과 무관하다. unique 제약은 INSERT가 commit 직전 통과해야 할 DB 단 검증 게이트다(없어도 트랜잭션은 진행됨 — Step 2가 증거). B·C·D·E는 INSERT 단계에 도달하기 전에 race를 잡으므로 INSERT엔 1건만 들어온다. 따라서 unique가 있어도 B·C·D·E의 작동에 관여하지 않는다. unique와 B·C·D·E는 다른 단계의 다른 도구로 직교한다.

후보의 단독 효과를 측정하려면 unique를 끈 상태에서 후보만 적용해야 한다(모드 A). unique를 켠 상태로는 어느 메커니즘이 막았는지 결과 해석이 흐려진다. 본 시리즈는 모드 A를 채택했고, 운영 환경에서는 보호망 중첩이 합리적이다.

## 8. 의사결정

### 8.1 측정 진행 회차

| 회차 | 환경 | 상태 |
|---|---|---|
| Step 1 — A 부분 유일 인덱스 | DB UQ 적용, 동시성 제어 코드 없음 | 측정 완료 |
| Step 2 — 제어 부재 baseline | DB UQ 제거, 동시성 제어 코드 없음 | 측정 완료 |
| Step 4-1 — E SERIALIZABLE | DB UQ 제거, `@Transactional(isolation=SERIALIZABLE)` | 진행 예정 |
| Step 4-2 — B FOR UPDATE | DB UQ 제거, `@Lock(PESSIMISTIC_WRITE)` | 진행 예정 |

차단 위치가 결정적으로 다른 두 후보(E — 격리 정책 변경, B — 명시적 락)를 측정 비교 대상으로 잡았다.

예상 시그니처:

- **B**: 후착 4건은 SELECT 단계에서 락 대기 → 락 획득 후 `RESERVED` 인식 → AVAILABLE 검증에서 throw. CP1을 통과하는 건 1건뿐. 잔여 `CONFIRMED` 1건.
- **E**: 후착 4건은 읽기 단계 통과(스냅샷 `AVAILABLE`) → UPDATE에서 선착 락 대기 → 충돌 검출 → ORA-08177. CP 통과 인원은 JPA flush 지연 여부에 좌우된다(5.5). 잔여 `CONFIRMED` 1건.

응답 시간 분포와 CP 단계별 통과 인원의 차이가 후보 비교의 본체다. 두 후보 모두 거부 트랜잭션이 락 대기 비용을 지불하지만, 대기가 일어나는 단계(B는 SELECT, E는 UPDATE)와 거부 형태(검증 throw / ORA-08177)가 다르다.

### 8.2 측정 제외 후보

| 후보 | 제외 근거 |
|---|---|
| C 낙관적 락 | 충돌 검출 모델이 E와 유사(SCN ↔ version). 같은 race에서 유사한 결과가 예상되어 측정의 부가 가치가 적음 |
| D 상태 조건부 UPDATE | affected=0만으로 거부 사유 구분이 어려워 규칙 2(거부 사유 명확성)와 충돌. trade-off 평가로 사전 제외 |
| F 분산 락 | 단일 인스턴스 환경 — 본래 목적(다중 인스턴스 자원 보호)과 부정합 |

측정하지 않는 것 자체가 의사결정의 한 형태다. 코드 검토와 메커니즘 분석만으로 trade-off가 결정 가능한 후보는 측정 회차를 추가하지 않았다.

## 9. 고민의 흐름

후보 비교 과정에서 짚은 핵심 판단들이다.

**unique를 켠 상태로는 비교가 불공정하다.** Step 1에서 정합성이 충족돼도, unique 자체가 동시성 제어 메커니즘 중 하나이므로 다른 후보의 효과가 가려진다. unique를 제거한 진짜 baseline(Step 2)을 측정해야 정합성 위반이 실제 발생함을 보이고 후속 비교의 출발점을 잡을 수 있다.

**현재 코드엔 UPDATE 단계의 보장 지점이 없었다.** 보장 지점은 INSERT의 unique뿐이었다. Step 2에서 5명이 모두 UPDATE에 성공한 이유는 WHERE가 PK 조건만이라 락 후 재평가가 항상 참이기 때문이다. D는 WHERE에 status 조건을 더해 그 재평가를 보장 지점으로 만든다.

**B·C·E는 결과가 같아 보이나 동작이 다르다.** 셋 다 1건 commit·4건 거부지만 직렬화 지점과 거부 형태가 다르다. 셋 다 같은 행 쓰기에서 직렬화된다는 공통점과, 직렬화가 어느 단계에서 일어나며 거부가 무엇으로 나타나는가의 차이를 분리해 봐야 한다.

**격리 수준과 B/C는 다른 레이어다.** "SERIALIZABLE이 강하다"가 아니라, 격리 수준은 읽기 일관성 정책이고 B/C는 그 위에 얹는 쓰기 race 차단·검출 도구다.

**E와 C는 충돌 검출 모델이 같다.** Oracle SERIALIZABLE이 SCN 기반 자동 충돌 검출이고, C가 version 기반 명시 충돌 검출이다. 키의 보관 위치와 적용 범위만 다르다.

**측정 환경(격리 수준) 명시는 보고서 신뢰도의 문제다.** 결과 자체가 메커니즘과 정합하더라도, 어느 격리 수준에서 측정했는지가 명시되지 않으면 비교 근거가 약해진다.

## 부록. 핵심 SQL·코드 스니펫

**현재 `createReservation`의 핵심 단계 (Step 1·2 동일)**

```java
@Transactional
public ReservationCreateResponseDto createReservation(ReservationCreateRequestDto dto, String userId) {
    VisitSlotEntity slot = slotRepository.findById(dto.getSlotId()).orElseThrow(...);
    if (slot.getStatus() != VisitSlotStatus.AVAILABLE) throw SlotUnavailableException.alreadyReserved(...);
    // startTime / 자기 매물 / 중복 / 시간 겹침 검증
    slot.setStatus(VisitSlotStatus.RESERVED);
    slotRepository.save(slot);                       // dirty 표시, 실제 UPDATE는 flush 시점
    VisitReservationEntity r = VisitReservationEntity.builder()
            .slotId(slot.getSlotId()).searcherUserId(userId)
            .status(VisitReservationStatus.CONFIRMED).confirmedAt(now).build();
    reservationRepository.save(r);
    // 후속 처리·응답
}
```

`slotRepository.save(slot)`는 dirty checking 환경에서 redundant하다(setter 호출만으로 commit 시 자동 UPDATE). 명시적 의도 표현으로 둘 수 있다.
