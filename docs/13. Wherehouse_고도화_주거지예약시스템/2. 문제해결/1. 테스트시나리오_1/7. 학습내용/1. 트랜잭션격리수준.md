# 트랜잭션 격리 수준 정리 (Oracle 중심)

복습용 문서. ANSI 표준 정의와 각 등급이 무엇을 막고 무엇을 허용하는지, Oracle이 어떤 등급을 지원하는지를 정리한다. 동작 결과를 중심으로 하되, **Oracle이 왜 그렇게 동작하는지**(SCN·언두·스냅샷, UPDATE의 두 단계)를 필요한 만큼 함께 본다 — "Oracle은 읽기가 쓰기를 막지 않는다" 같은 서술을 결과로만 외우지 않고 그 메커니즘까지 내려가기 위해서다. 처음 참고한 자료(레벨 0~3 요약 이미지)는 ANSI 표준 분류이고, Oracle은 이 4단계를 그대로 지원하지 않는다는 점이 출발점이다.


## 1. 격리 수준이 왜 필요한가

여러 트랜잭션이 동시에 같은 데이터를 읽고 쓴다. 통제가 없으면 한 트랜잭션이 보는 데이터가 다른 트랜잭션 때문에 일관성을 잃는다.

격리 수준은 "다른 트랜잭션의 영향을 내 트랜잭션이 어디까지 차단하는가"를 정하는 등급이다. 등급이 높을수록 일관성은 올라가고 동시성은 떨어진다. 둘은 반대로 움직이고, 격리 수준 선택은 이 trade-off를 어디에 둘지 정하는 일이다.


## 2. 세 가지 이상 현상 (Read Phenomena)

ANSI/ISO SQL 표준은 격리 수준을 세 가지 이상 현상으로 정의한다. 각 등급은 이 셋 중 무엇을 허용하고 무엇을 막느냐로 구분된다.

### Dirty Read
아직 커밋되지 않은 다른 트랜잭션의 변경 데이터를 읽는 것. 그 트랜잭션이 롤백되면 존재한 적 없는 값을 읽은 셈이 된다.

### Non-Repeatable Read
시점 T1에 한 행을 읽고 T2에 그 **같은 행**을 다시 읽었을 때 값이 바뀌어 있는 것. 다른 트랜잭션이 그 행을 수정·삭제하고 커밋했기 때문이다. 핵심은 "이미 읽은 그 행"이 달라진다는 것.

### Phantom Read
T1에 쿼리를 실행하고 T2에 같은 쿼리를 다시 실행했을 때 조건을 만족하는 **행이 추가되어** 결과가 달라지는 것.

### Non-Repeatable Read vs Phantom Read (혼동 포인트)
가르는 기준은 "기존 행이 바뀌느냐" vs "새 행이 끼어드느냐"다.

- Non-Repeatable Read: 이미 읽은 행의 값이 변한다.
- Phantom Read: 이미 읽은 행은 그대로지만, 같은 조건을 만족하는 행이 많아진다.

이 구분이 REPEATABLE READ와 SERIALIZABLE을 가르는 기준이다.


## 3. ANSI 표준 4단계 등급

| 격리 수준 | Dirty Read | Non-Repeatable Read | Phantom Read |
|---|---|---|---|
| READ UNCOMMITTED | 허용 | 허용 | 허용 |
| READ COMMITTED | 차단 | 허용 | 허용 |
| REPEATABLE READ | 차단 | 차단 | 허용 |
| SERIALIZABLE | 차단 | 차단 | 차단 |

이 표가 각 등급의 정의 그 자체다. 인접 등급 간 차이는 한 칸씩이다.

- READ COMMITTED → REPEATABLE READ: Non-Repeatable Read를 막느냐의 차이.
- REPEATABLE READ → SERIALIZABLE: Phantom Read를 막느냐의 차이.

ANSI 표준은 이상 현상으로 등급을 정의할 뿐 구체적 구현을 강제하지 않는다. 그래서 같은 등급 이름이라도 DBMS마다 동작이 다를 수 있다.


## 4. 같은 등급, 다른 동작

같은 격리 수준이라도 DB에 따라 동작이 다르다. 핵심 차이는 다음 두 가지다.

- 전통적 비-Oracle DB: 정확성을 위해 다른 트랜잭션을 락으로 블로킹한다. 읽기와 쓰기가 서로를 막을 수 있다.
- Oracle: 읽기가 쓰기를 막지 않고, 쓰기가 읽기를 막지 않는다. 같은 READ COMMITTED라도 비-Oracle처럼 읽기가 락에 걸려 대기하는 일이 없다.

또 하나, "어느 시점의 데이터를 보느냐"가 등급을 가른다.

- 쿼리(statement) 시작 시점 기준으로 본다 → READ COMMITTED
- 트랜잭션 시작 시점 기준으로 본다 → SERIALIZABLE

이 "기준 시점"이 등급 이해의 중심축이다.

여기까지는 "Oracle은 읽기가 락에 안 막힌다"와 "기준 시점"을 **결과로만** 적었다. 그 둘이 실제로 어떻게 구현되는지가 다음 두 절이다. 5절은 "읽기가 왜 안 막히나"(SCN·언두·스냅샷), 6절은 그 위에서 "UPDATE 한 문장이 내부에서 어떻게 두 단계로 도나"를 본다. 이 둘을 깔아야 RC와 SERIALIZABLE의 쓰기 충돌 처리(8·10절)가 설명된다.


## 5. Oracle은 읽기를 어떻게 락 없이 처리하나 — SCN·언두·스냅샷

4절에서 "Oracle은 읽기가 쓰기를 막지 않는다"고 했다. 어떻게 가능한가. Oracle은 읽기를 락으로 처리하지 않고, 옛 값을 보관해 두었다가 읽는 쪽에게 "특정 시점의 옛 모습"을 재구성해 보여준다(다중 버전). 부품 셋이다.

```
SCN (System Change Number)
   = 커밋이 일어날 때마다 1씩 올라가는, 데이터베이스 전체의 버전 번호(논리 시계).
     "SCN 105" = 105번째 변경 묶음까지 커밋된 시점의 상태.

언두(undo, 롤백 세그먼트)
   = 어떤 행을 바꿀 때, 새 값을 테이블에 쓰기 '전에' 그 행의 옛 값을 따로 복사해 두는 영역.

스냅샷
   = "특정 SCN 시점의 데이터 모습". 테이블을 통째로 복사한 사본이 아니다.
     실체는 ★숫자 하나(read SCN)★이고, 읽을 때 블록마다 언두로 그 시점 값을 즉석 재구성한다.
```

동작을 행 R(값 V1)로 본다.

```
출발: 행 R = V1, 이 값이 커밋된 버전은 SCN 100 이하.
T1: R을 V2로 UPDATE 후 COMMIT (커밋 SCN=105).
    → 테이블 블록의 현재 값은 V2(버전 도장 105), 언두에는 옛값 V1이 보관됨.

read SCN=100 으로 고정된 트랜잭션이 R을 SELECT:
   (1) 블록을 본다 → 현재 V2(SCN 105)
   (2) "105 > 내 기준 100" → 내 시점 이후 변경, 나에겐 너무 최신
   (3) 언두에서 옛값 V1을 꺼내 'SCN 100 시점' 블록을 임시 사본으로 재구성
   (4) V1을 반환한다 → 쓰는 쪽(T1)을 막지도 기다리지도 않는다
```

범위와 실체를 정확히 한다.

```
(가) 논리적 범위는 데이터베이스 전체다. read SCN 하나가 그 트랜잭션의 모든 읽기에 적용된다.
     어느 테이블을 읽든 같은 SCN 시점으로 보므로 테이블들 사이가 서로 모순되지 않는다.
     "그 행이 든 테이블 하나만"의 스냅샷이 아니다.
(나) 물리적으로는 미리 복사하지 않는다. 실제로 읽은 블록만, 그 블록이 내 read SCN 이후에
     바뀌었을 때만 언두로 재구성한다.
(다) 증거 — ORA-01555(snapshot too old): 재구성에 필요한 옛 값을 그 사이 언두가 덮어써
     버리면 재구성 불가 → 이 에러. '복사본'이었다면 날 이유가 없는 에러다.
```

**핵심**: 격리 수준의 차이는 결국 **"이 read SCN을 언제 잡고 언제까지 쓰느냐"** 하나로 환원된다. 4절의 "기준 시점"이 바로 이것이다 — READ COMMITTED는 문장마다 read SCN을 새로, SERIALIZABLE은 트랜잭션 시작에 한 번 고정.


## 6. UPDATE 한 문장의 두 단계 — 찾기와 바꾸기

RC와 SERIALIZABLE이 쓰기 충돌을 처리하는 방식은 모두 이 한 가지에서 갈린다: `UPDATE ... WHERE ...` 한 문장은 내부적으로 두 단계로 나뉜다.

```
UPDATE T SET S='B' WHERE key=K AND S='A'

 ① 찾기 (consistent read) : WHERE 조건에 맞는 행을 '고른다'.
                            읽는 값은 내 스냅샷(read SCN) 시점 값. 락을 걸지 않는다.
 ② 바꾸기 (current read + 락): 고른 행을 '실제로 고친다'.
                            행을 고치려면 스냅샷이 아니라 ★그 행의 현재 커밋된 최신 버전★을 봐야 한다.
                            그 행에 X-락을 걸고 값을 바꾼다.
                            이 단계에서 WHERE를 ★현재값 기준으로 다시 확인★한다.
```

- ①찾기는 스냅샷을 읽으므로 **격리 수준이 무엇을 보느냐를 지배한다.**
- ②바꾸기는 격리 수준과 무관하게 **언제나 "현재 최신 버전 + X-락"으로 동작한다.**

즉 한 UPDATE 안에서 "내가 읽은 시점 값"과 "내가 덮어쓰는 순간의 값"이 다를 수 있고, 그 사이 다른 트랜잭션이 끼면 ②바꾸기에서 드러난다. 이 ②의 처리가 RC(8절)와 SERIALIZABLE(10절)에서 다르게 나타난다.


## 7. READ UNCOMMITTED

가장 낮은 등급. Dirty Read를 허용한다. 다른 DB가 이 등급을 두는 이유는 락 없이 읽기를 얻기 위해서다. Oracle은 이 등급을 지원하지 않고 Dirty Read도 허용하지 않는다. Oracle은 어차피 읽기가 락에 막히지 않으므로(5절) Dirty Read 없이도 "읽기가 안 막히는" 목적을 달성한다. 이미지의 "Oracle은 Read Uncommitted를 지원하지 않는다"가 이 의미다.


## 8. READ COMMITTED

### 정의
커밋된 데이터만 읽는다(Dirty Read 차단). Non-Repeatable Read와 Phantom Read는 허용한다. 대부분 DBMS의 기본 모드이며 Oracle의 기본값이다.

### 기준 시점: 쿼리 단위
READ COMMITTED의 기준은 트랜잭션 시작이 아니라 **개별 쿼리 시작 시점**이다. 각 쿼리는 그 쿼리가 시작되기 전에 커밋된 데이터를 본다. (5절 용어로는, 문장이 시작될 때마다 read SCN을 그 순간의 최신값으로 새로 잡는다.)

그래서 같은 트랜잭션 안에서 동일한 SELECT를 두 번 실행하면, 그 사이 다른 트랜잭션이 커밋했을 경우 두 번째 SELECT는 새 값을 본다. 이것이 READ COMMITTED에서 Non-Repeatable Read가 생기는 이유다.

### Oracle vs 비-Oracle (계좌 합산 예시, 표 4)
같은 READ COMMITTED라도 동작이 갈린다.

상황: accounts 테이블 전체 balance를 SUM하는 긴 쿼리(읽기 세션)가 돈다. 중간에 다른 트랜잭션이 account 123에서 987로 $400을 옮긴다(123 차감, 987 증가). 이체는 두 행을 수정한 뒤 아직 커밋하지 않았다.

- Oracle: SUM 쿼리는 account 987의 원래 값 $100을 읽고 완료한다. 블로킹 없음. 합계는 일관된 값.
- 비-Oracle(표 4): SUM 쿼리가 account 987을 읽으려 할 때 그 행이 수정 중이라 블로킹된다. 이체가 커밋할 때까지 대기하다가 증가된 값 $500을 읽는다. 차감 전 123을 이미 합산해 둔 상태에서 증가 후 987을 합산하므로 $400이 중복된, 실재한 적 없는 합계가 나온다.

표 4의 타임라인:

| 시점 | 읽기 세션(SUM) | 이체 트랜잭션 |
|---|---|---|
| T1 | row 1 읽음. Sum=$500 | — |
| T2 | row 2 읽음. Sum=$740.25 | — |
| T3 | — | row 1 수정, 락. row 1 = $100 |
| T4 | row N 읽음 | — |
| T5 | — | row 342,023 수정, 락. = $500 |
| T6 | row 342,023 읽으려다 락 발견 → 블로킹, 대기 | — |
| T7 | — | 커밋 |
| T8 | row 342,023에서 $500 읽고, $400 중복된 최종 답 | — |

핵심 교정: **표 4는 Oracle 동작이 아니라 비-Oracle DB 동작을 보여주는 표다.** 제목에 "비 Oracle"이 명시돼 있다. Oracle과 대조하려고 일부러 다른 DBMS를 가져온 것이고, 여기서 드러나는 비-Oracle의 특성은 "쓰는 쪽이 읽는 쪽을 블로킹한다"는 것이다.

### 쓰기 일관성 재시작 (조건부 UPDATE가 경합을 막는 원리)
RC는 읽기 차원에선 Non-Repeatable Read를 허용하지만, **쓰기 차원에선 같은 행을 다투는 두 UPDATE를 정확히 직렬화한다.** 6절의 두 단계로 본다. 두 트랜잭션이 같은 문장 `UPDATE T SET S='B' WHERE key=K AND S='A'`를 동시에 실행:

```
출발: 행 K의 S='A' (SCN 100)

s1  T1: ①찾기(스냅샷 100): S='A' → 대상.  ②바꾸기: K에 X-락, S='B'로 기록 (미커밋)
s2  T2: ①찾기(스냅샷 100): S='A' → 대상.  ②바꾸기: 같은 K에 X-락 시도 → T1 보유 → 대기
s3  T1: COMMIT (커밋 SCN=105), X-락 해제
s4  T2: 대기 풀림. ②바꾸기 재개 — 여기서 WHERE를 ★두 번째로★ 평가한다:
        · 기준: 스냅샷(100)이 아니라 K의 현재 커밋 버전(SCN105, S='B')
        · WHERE의 S='A'가 거짓 → 이 행 탈락
        · 오라클은 새 read SCN(106)으로 문장을 재시작 → 대상 없음 → 0 rows updated
```

- **시점**: ②바꾸기에서 막혔던 락을 다시 잡는 순간(선착 커밋 직후). ①찾기 때가 아니다.
- **기준**: 내 스냅샷이 아니라 그 행의 현재 커밋된 최신 버전.
- **조건의 중요성**: WHERE에 조건 컬럼(S='A')이 있어야 재평가에서 후착이 걸러진다. WHERE가 `key=K`뿐이면 현재 버전도 key는 그대로라 재평가를 통과해 후착이 안 걸러진다. 즉 "조건부 UPDATE"가 경합을 막는 힘은 WHERE의 상태 조건에서 나온다.


## 9. REPEATABLE READ

### 정의
Dirty Read와 Non-Repeatable Read를 차단한다. 한 트랜잭션 안에서 이미 읽은 행은 끝날 때까지 같은 값으로 보인다. 단, Phantom Read는 막지 못한다. 조건에 맞는 새 행이 끼어드는 것까지는 보장하지 않아서 이름이 "Serializable"이 아니라 "Repeatable Read"다.

### Lock 기반 동작
비-Oracle DB는 행 단위 shared read lock으로 구현한다. READ COMMITTED와의 차이는 락 유지 기간이다.

- READ COMMITTED: 행을 읽는 순간만 락을 잡고 읽고 나면 즉시 푼다.
- REPEATABLE READ: 읽은 행의 락을 트랜잭션이 끝날 때까지 유지한다.

이미 읽은 행에 락이 계속 걸려 있으니 다른 트랜잭션이 그 행을 수정하지 못한다. 그래서 다시 읽어도 값이 안 변한다(Non-Repeatable Read 차단).

### 락 호환성 (혼동 포인트)
락 충돌은 "누가 먼저 요청했나"(시간 순서)가 아니라 "그 행에 이미 어떤 락이 있고 새 요청이 어떤 락을 원하는가"로 판단한다. 읽기는 shared lock(S), 수정은 exclusive lock(X).

| | 요청: S(읽기) | 요청: X(수정) |
|---|---|---|
| 기존: S(읽기) | 호환 | 충돌 |
| 기존: X(수정) | 충돌 | 충돌 |

읽기끼리(S-S)는 공존한다. 여러 트랜잭션이 같은 행을 동시에 읽을 수 있다. 수정(X)은 다른 어떤 락과도 공존하지 못한다.

### 케이스별 동작 (계좌 합산, 표 5)
트랜잭션 A가 먼저 행에 작업하고 B가 나중에 같은 행에 접근하는 상황.

1. A가 그 행을 **읽기만** 한 경우(A가 S 보유):
   - B의 **읽기**는 가능(S-S 호환). 횟수 제한 없음. 오히려 여러 번 읽어도 같은 값이 나온다.
   - B의 **수정**은 막힘(S-X 충돌). A가 끝날 때까지 블로킹.
2. A가 그 행을 **수정 중**인 경우(A가 X 보유):
   - B의 **읽기조차** 막힘(X-S 충돌).

표 5의 타임라인:

| 시점 | 읽기 세션(SUM) | 이체 트랜잭션 |
|---|---|---|
| T1 | row 1 읽음. Sum=$500. row 1에 S lock 유지 | — |
| T2 | row 2 읽음. Sum=$740.25. row 2에 S lock 유지 | — |
| T3 | — | row 1 수정 시도 → S lock과 충돌, 블로킹 |
| T4 | row N 읽음 | — |
| T5 | row 342,023에서 $100 읽고 최종 답(정확) | — |
| T6 | 커밋 → 모든 S lock 해제 | — |
| T7 | — | 비로소 row 1 수정 = $100 |
| T8 | — | row 342,023 수정 = $500, 커밋 |

결과: 정확한 답을 얻지만, 이체 트랜잭션을 블로킹해 두 트랜잭션을 순차 실행시킨 대가다. 비-Oracle Lock 방식에서는 "읽는 쪽이 쓰는 쪽을 블로킹한다." 표 4의 "쓰는 쪽이 읽는 쪽을 블로킹"에 더해 반대 방향까지 생긴다(양방향 블로킹). 이체 방향만 바꾼 표 6에서는 서로의 락을 기다리다 데드락에 빠진다.

### 왜 이 방식으로는 Phantom을 못 막나
이 S-락은 "이미 존재해서 실제로 읽은 행"에만 걸린다. 아직 없는 행, 즉 앞으로 INSERT될 행에는 걸 수 없다(존재하지 않는 행에는 잠글 대상이 없다). 그래서 범위 조건으로 여러 행을 읽고 다시 읽는 사이, 조건에 맞는 새 행이 INSERT되면 그 새 행은 기존 S-락과 겹치는 자원이 없어 막히지 않고 들어온다 = Phantom. 이를 막으려면 개별 행이 아니라 "조건에 맞는 행이 들어올 범위" 자체를 잠그는 범위 락(키 범위 락)이 필요하고, 그게 곧 락 기반 SERIALIZABLE이다. 즉 락 방식에서 REPEATABLE READ → SERIALIZABLE의 차이는 "읽은 개별 행만 잠그느냐 vs 범위까지 잠그느냐"다.

### 사용자 명제 교정
"선행 트랜잭션 진행 중이면 후행은 1회 읽을 수 있으나 수정은 불가능"은 두 군데가 틀린다.
- "1회 읽기"가 아니다. 읽기 횟수 제한은 없다.
- 막히는 주체는 읽은 트랜잭션 자신이 아니라, 읽힌 행을 수정하려는 **다른** 트랜잭션이다.
- 정확한 문장: "REPEATABLE READ에서 한 트랜잭션이 읽은 행은, 그 트랜잭션이 끝날 때까지 다른 트랜잭션이 수정하지 못한다."
- 트랜잭션이 단순히 "진행 중"인 것만으로는 제약이 안 생긴다. 그 트랜잭션이 해당 행에 실제로 락을 걸어 둔 상태여야 한다.

### Oracle에서의 위치
Oracle은 REPEATABLE READ를 격리 수준으로 제공하지 않는다(지원 등급은 READ COMMITTED와 SERIALIZABLE뿐). "이미 읽은 행이 안 바뀜"이 필요하면 SERIALIZABLE로 올리거나, 특정 행에만 필요하면 SELECT ... FOR UPDATE로 명시적 락을 건다. 이미지의 "Oracle은 레벨 2를 for update 절로 구현 가능"이 이 의미다. 등급을 제공하는 게 아니라 효과를 다른 수단으로 대체하는 것이다.

왜 Oracle에 이 등급이 없는지는 5절과 연결된다. 위 REPEATABLE READ는 **읽기 S-락**이 있어야 성립한다(읽은 행의 락을 끝까지 보유). 그런데 Oracle은 읽기에 락을 걸지 않고 read SCN 스냅샷만 쓴다. 스냅샷을 트랜잭션 단위로 고정하면 "내 read SCN 이후 커밋된 변경"이 전부 안 보이는데, 여기엔 기존 행 수정과 새 행 삽입이 똑같이 들어간다 — 즉 Non-Repeatable Read와 Phantom이 **동시에** 막힌다. "Non-Repeatable은 막고 Phantom은 허용"이라는 중간 상태를 스냅샷으로는 만들 수 없으므로, Oracle은 트랜잭션 단위 스냅샷 수준을 SERIALIZABLE로 부르고 REPEATABLE READ를 따로 두지 않는다. (참고: MySQL InnoDB·PostgreSQL도 스냅샷을 쓰면서 같은 수준에 "REPEATABLE READ"라는 이름을 붙였을 뿐 실제로는 Phantom까지 대체로 막혀 표준보다 강하다. 명명 차이다.)


## 10. SERIALIZABLE

### 정의
세 가지 이상 현상을 전부 차단한다(Dirty / Non-Repeatable / Phantom 모두). 다른 사용자가 데이터를 수정하지 않는 것처럼 보이는 환경에서 동작한다. REPEATABLE READ와의 차이는 Phantom Read까지 막는다는 점 하나다.

### 기준 시점: 트랜잭션 시작
READ COMMITTED의 기준이 쿼리 시작 시점이라면, SERIALIZABLE은 **트랜잭션 시작 시점**으로 끌어올린다. 트랜잭션 시작 후 다른 트랜잭션이 만든 변경은, 커밋된 변경이라도 내 트랜잭션에 보이지 않는다. (5절 용어로는, read SCN을 트랜잭션 시작에 한 번 잡고 끝까지 고정한다.)

예시:
```sql
select * from T;
begin dbms_lock.sleep( 60*60*24 ); end;
select * from T;
```
중간에 24시간을 자도 두 SELECT 결과는 동일하다. 그 사이 다른 트랜잭션이 T를 수정·커밋해도 내 트랜잭션은 자기 시작 시점의 T를 계속 본다.

### Lock 기반 동작
REPEATABLE READ가 읽은 행의 수정만 막았다면, SERIALIZABLE은 새 행 삽입까지 막아야 Phantom을 차단한다. 그래서 행 단위로는 부족하고 범위 전체나 테이블 단위로 락을 건다. 이미지의 "테이블 레벨 Lock 사용, 트랜잭션 완료될 때까지 유지"가 이 동작이다. 대가는 동시성의 큰 하락이다.

### Oracle 동작
Oracle은 읽기를 락으로 막지 않는다. 읽기가 쓰기를 막지 않고 데드락도 없다. 이것이 Lock 기반 SERIALIZABLE과의 결정적 차이다.

대신 쓰기 충돌은 낙관적(optimistic) 방식으로 처리한다. 내가 수정하려는 데이터를 다른 트랜잭션이 수정하지 않을 것이라 가정하고 진행하다가, 내 트랜잭션 시작 이후 다른 트랜잭션이 변경·커밋한 행을 내가 수정하려 하면 다음 에러가 난다.

```
ORA-08177: can't serialize access for this transaction
```

미리 막는 게 아니라 충돌이 실제로 일어난 시점에 에러로 거부한다. 따라서 SERIALIZABLE을 쓰는 애플리케이션은 이 에러를 받으면 트랜잭션을 재시도하는 로직이 필요하다.

### ORA-08177이 나는 메커니즘
"낙관적 방식"이 실제로 무엇인지를 6절의 두 단계로 본다. 같은 행을 두 SERIALIZABLE 트랜잭션이 동시에 수정하는 흐름이다.

```
출발: 행 K의 S='A' (SCN 100). 두 트랜잭션 모두 T1 커밋 전에 시작.

s1  T1: read SCN=100 고정.  ①찾기: S='A' → 대상.  ②바꾸기: K에 X-락, S='B' (미커밋)
s2  T2: read SCN=100 고정.  ①찾기: S='A' → 대상.  ②바꾸기: 같은 K에 X-락 시도 → 대기
s3  T1: COMMIT (커밋 SCN=105), X-락 해제
s4  T2: 대기 풀림. ②바꾸기 재개:
        · 수정 대상 K의 현재 커밋 버전 SCN(105) > 내 read SCN(100)
          = 내가 트랜잭션을 시작한 뒤에 그 행이 바뀌어 커밋됨
        · RC라면 새 read SCN으로 재시작 가능 → 0행. 그러나 SERIALIZABLE은 read SCN을
          고정해 두므로 재시작 불가 → ORA-08177
```

- **시점**: ②바꾸기에서 막혔던 락을 다시 잡는 순간(같은 행 경합이면 선착 커밋 직후). 문장 끝도, 커밋 시점도 아니다.
- **기준**: "수정 대상 행의 현재 커밋 SCN"과 "내 read SCN"을 비교 — 전자가 크면 충돌.
- **어긋남**: s4 시점에 T2가 그 행을 SELECT하면 read SCN=100 기준으로 여전히 S='A'를 본다. 그런데 UPDATE는 거부된다. 충돌 판정이 읽기(①찾기)가 아니라 쓰기(②바꾸기) 시점에 이뤄지기 때문이다.
- **RC와 비교**: 같은 경합에서 정합성 결과는 같다(한 트랜잭션만 수정 성공). 다른 것은 후착이 받는 신호뿐 — RC는 0행, SERIALIZABLE은 ORA-08177.

### 적합한 조건
- 같은 데이터를 다른 트랜잭션이 동시에 수정할 확률이 낮음
- 트랜잭션 레벨 read consistency가 필요함
- 짧은 트랜잭션

충돌 확률이 낮아야 낙관적 가정이 자주 맞고, 트랜잭션이 짧아야 충돌 가능성이 줄기 때문이다.

### 이름과 실제 보장의 괴리 (핵심 혼동 포인트, 표 7)
SERIALIZABLE이라는 이름은 "트랜잭션들이 순차로 하나씩 실행된 것과 동일한 결과"를 떠올리게 하지만, Oracle SERIALIZABLE은 그것을 보장하지 않는다.

빈 테이블 A, B에서 두 세션이 모두 SERIALIZABLE로 실행:

| 시점 | Session 1 | Session 2 |
|---|---|---|
| T1 | isolation = serializable | — |
| T2 | — | isolation = serializable |
| T3 | insert into a select count(*) from b; | — |
| T4 | — | insert into b select count(*) from a; |
| T5 | commit | — |
| T6 | — | commit |

- T3: Session 1은 B의 행 수를 세서 A에 넣는다. B가 비었으니 0을 넣는다.
- T4: Session 2는 A의 행 수를 세서 B에 넣는다. Session 2는 자기 시작 시점(T2)의 A를 보므로 Session 1이 넣은 행이 안 보인다. A도 비었다고 보고 0을 넣는다.

결과: A에 0 한 행, B에 0 한 행.

이 결과는 어떤 직렬 순서로도 나올 수 없다. Session 1이 먼저였다면 B에 1이, Session 2가 먼저였다면 A에 1이 들어가야 한다. 둘 다 0이라는 건 각 세션이 자기 트랜잭션만 존재하는 것처럼 실행됐다는 뜻이다.

결론: Oracle SERIALIZABLE이 보장하는 것은 "각 트랜잭션이 자기 시작 시점의 일관된 결과를 본다"는 것이지, "전체 트랜잭션 집합이 어떤 직렬 순서와 동일한 결과를 낸다"는 것이 아니다. 이름과 실제 보장 범위가 어긋나 있다는 점이 이 등급에서 가장 헷갈리는 지점이다. (엄밀히는 snapshot isolation에 해당하며 진짜 serializability와 구분된다.)

### 표 7 현상의 이름과 메커니즘 — write-skew
표 7의 이 현상에는 이름이 있다. **write-skew**다. 왜 SERIALIZABLE이 이걸 못 막는지는 한 가지 사실에서 나온다: **SERIALIZABLE의 충돌 검출(ORA-08177)은 '실제로 쓴 행'에만 적용되고, '읽기만 한 행'은 검출 대상이 아니다.**

```
write-skew의 성립 조건:
   1) 두 트랜잭션이 '같은 데이터'를 읽는다   (표 7: Session1·2가 서로 상대 테이블 count를 읽음)
   2) 각자 그 읽은 값을 근거로 판단한다       (둘 다 "상대가 0이니 0을 넣는다")
   3) 그런데 각자 '서로 다른 행'을 쓴다        (Session1은 A에, Session2는 B에 INSERT)
   → 쓴 행이 겹치지 않으므로 쓰기-쓰기 충돌이 없다 → ORA-08177이 안 난다 → 둘 다 커밋
   → 합쳐 보면 어떤 직렬 실행과도 다른 결과 = 불변식 깨짐
```

대비점: **같은 행**을 다투는 경합(앞 "ORA-08177이 나는 메커니즘"의 K)은 쓰기-쓰기 충돌이 반드시 생겨 SERIALIZABLE이 정확히 잡는다. write-skew는 **서로 다른 행**을 쓸 때만 생긴다.

막는 법: 읽기만 하던 행을 `SELECT ... FOR UPDATE`로 X-락을 걸어 읽으면, 그 행이 쓰기 대상처럼 잠겨 읽기 겹침이 쓰기-쓰기 충돌로 바뀌고 검출된다. (PostgreSQL의 SERIALIZABLE은 읽은 행의 의존성까지 추적해 write-skew를 자동 차단하지만 — SSI — Oracle에는 그 방식이 없다.)


## 11. 락 메커니즘 — SELECT … FOR UPDATE와 배타 락

9·10절에서 락 호환성(S/X)과 write-skew 차단에 `SELECT ... FOR UPDATE`가 등장했다. 이 절은 그 FOR UPDATE와 그것이 거는 **배타 락**이 정확히 무엇인지, Oracle의 행/테이블 락(TX·TM)이 격리 수준과 어떻게 다른 층인지를 정리한다. 격리 수준(5절: read SCN 스냅샷)이 **"읽기 일관성 — 어느 시점을 보느냐"** 를 정한다면, 락은 **"쓰기 충돌 직렬화 — 같은 자원을 동시에 못 바꾸게"** 를 정하는 별개의 층이다.

### 락은 데이터에 붙는 물건이 아니다 / 모드와 호환성
락 = 어떤 자원(행·테이블)에 대해 트랜잭션이 등록한 **점유 표시**다. 데이터 위에 물리적으로 얹히는 게 아니라, 락 관리자(Oracle은 *enqueue* + 데이터 블록 헤더의 *ITL 슬롯*)에 "이 트랜잭션이 이 자원을 이 모드로 점유 중"이 기록되고, 다른 트랜잭션이 같은 자원을 건드릴 때 그 기록으로 **허용/대기**를 판정한다.

락의 정체는 **모드**와, 한 자원에 여러 락이 공존 가능한지를 정한 **호환성 매트릭스**다(9절 표의 일반화).
- **공유(S, Shared)** = 읽을 권리. 여러 트랜잭션이 동시에 가질 수 있다(S–S 양립).
- **배타(X, eXclusive)** = 독점 권리. 한 트랜잭션만.

```
            다른 TX가 S 요청    다른 TX가 X 요청
내가 S 보유      양립               대기
내가 X 보유      대기               대기
```

### 배타(Exclusive)의 정확한 정의
배타 락 = 그 자원에 대해 **다른 어떤 락(S든 X든)과도 공존할 수 없는 락.** 두 가지를 보장한다.
1. **단일 보유자** — 한 시점에 그 자원의 X는 최대 1개(한 트랜잭션).
2. **완전 배제** — 내가 X를 쥔 동안 다른 트랜잭션은 그 자원에 S도 X도 못 잡는다 → 충돌 작업은 내가 풀(COMMIT/ROLLBACK) 때까지 대기.

"배타적"이라는 말 그대로 남을 전부 배제한다. 이름이 곧 정의다.

### ★ Oracle 특화 — 행 락은 항상 배타, 그리고 reader를 막지 않는다
위 매트릭스는 **잠금 기반(2PL)** DB의 일반론이고(9·10절의 비-Oracle 표가 그 동작), Oracle은 **행 단위에서 이걸 그대로 쓰지 않는다.**

- **Oracle의 행 락은 항상 배타(X) 하나뿐이다. "공유 행 락"을 쓰지 않는다.** UPDATE·DELETE·INSERT·SELECT FOR UPDATE — 행을 잠그는 모든 것이 X 행 락을 잡는다.
- **왜 공유 행 락이 없나**: 평문 읽기(SELECT)가 락을 아예 안 잡기 때문이다(5절 — read SCN 스냅샷 + 언두). reader가 S를 요청하지 않으니 "S 행 락"이 존재할 이유가 없다.
- 그래서 **Oracle에서 배타 행 락이 막는 대상 = 다른 writer / FOR UPDATE locker 이지, reader가 아니다.**
```
일반 2PL:      X 행 락 → 다른 writer(X) 차단 + 다른 reader(S) 도 차단
Oracle(MVCC):  X 행 락 → writer/locker(X) 만 차단, reader 는 안 막음(언두로 옛 버전 읽음)
```
9절의 "X-S 충돌(읽기조차 막힘)"은 **비-Oracle** 동작이다. Oracle에선 그 reader가 애초에 S를 요청하지 않으므로 그 충돌 자체가 생기지 않는다.

### 두 종류의 락 — TX(행)와 TM(테이블)
Oracle이 거는 락은 두 층이다. 용어를 정확히 한다.

| 락 | 정체 | 누가 잡나 | 목적 |
|---|---|---|---|
| **TX** (트랜잭션 enqueue) | **행 락** | 행을 바꾸거나 FOR UPDATE 한 트랜잭션 | 같은 행 동시 변경 직렬화 (`enq: TX - row lock contention`) |
| **TM** (테이블 enqueue) | **테이블 락** | DML · SELECT FOR UPDATE | 그 테이블에 **DDL 못 걸게** 막기 |

- INSERT/UPDATE/DELETE: **TX(행, 배타) + TM(테이블, Row Exclusive)** 둘 다.
- **SELECT … FOR UPDATE: TX(행, 배타) + TM(테이블, 행수준 모드) 둘 다 잡는다.**
- 행 단위는 배타(X) 하나뿐이지만, **테이블 단위(TM)는 모드 스펙트럼**(Row Share·Row Exclusive·Share·Exclusive)이 다 있다. DML·FOR UPDATE가 잡는 행수준 TM은 서로 양립하지만, **DDL이 요구하는 배타 TM(Exclusive)과는 충돌**한다(아래).

### SELECT … FOR UPDATE — 정의·절차·현재모드 읽기
**행을 읽으면서 그 행에 배타 행 락(TX)을 걸고, 트랜잭션 끝까지 유지하는 SELECT.** = 비관적 락의 SQL 표준 구현. "지금 읽지만 곧 쓸 거니 그때까지 잠가둬라"는 쓰기 의도 선언이다.

6절에서 본 `UPDATE`의 두 단계 중 **②바꾸기(현재값 + X락)** 를, FOR UPDATE는 **읽기 시점으로 당겨** 수행하는 것으로 보면 정확하다.
```
SELECT … WHERE key=K FOR UPDATE 실행:
 ① WHERE로 대상 행을 '현재 모드'로 읽는다          ← 스냅샷 아님 (6절 ②바꾸기와 동일)
 ② 그 행에 TX(배타) + TM 획득
       └ 이미 다른 TX가 그 행을 잠갔으면 → ③
 ③ 대기 (enq: TX - row lock contention)
       └ 선행 COMMIT/ROLLBACK 까지 (기본 무한 대기)
       └ 풀리면 그 행을 '현재 커밋값으로 다시 읽고' 잠근다 (write consistency 재시작)
 ④ 잠근 행 반환,  락은 COMMIT/ROLLBACK 까지 유지(문장 끝이 아니라 트랜잭션 끝까지)
```

**현재모드 읽기(핵심)**: 락을 걸려면 스냅샷(과거 버전)이 아니라 그 행의 **현재 커밋된 최신 버전**을 봐야 한다. 그래서 대기가 풀린 직후, 선행이 방금 값을 바꾸고 커밋했다면 FOR UPDATE는 그 **새 값**을 읽는다.
- WHERE가 PK뿐(`key=K`)이면 재평가에서 행이 안 빠지고 **현재값**(예: status=RESERVED)을 반환한다 → 거부 판정은 애플리케이션의 `if`가 한다. (= F004 비관적 락 B.)
- WHERE에 상태 조건(`AND status='AVAILABLE'`)이 있으면 8절처럼 재평가에서 행이 빠져 `affected=0`이 된다. (= 조건부 UPDATE D.)

**옵션**(잠김을 만났을 때의 행동을 바꾼다):
```sql
… FOR UPDATE;             -- 기본: 무한 대기
… FOR UPDATE NOWAIT;      -- 잠겨 있으면 즉시 ORA-00054
… FOR UPDATE WAIT 3;      -- 최대 3초 대기 후 ORA-30006
… FOR UPDATE SKIP LOCKED; -- 잠긴 행은 건너뛰고 안 잠긴 것만 (큐 분배)
… FOR UPDATE OF t.col;    -- 조인 중 t의 행만 잠금 (OF 없으면 양쪽 다)
```
제약: `DISTINCT`·`GROUP BY`·집계·`UNION`과는 못 쓴다(어느 실제 행을 잠글지 모호해서).

### TM 락과 DDL — 트랜잭션 도중 스키마 변경이 막히는 이유
FOR UPDATE·DML이 잡는 TM(테이블 락)의 목적은 **그 테이블의 DDL을 막는 것**이다. DDL(`ALTER TABLE … MODIFY`, `DROP` 등)은 **배타 테이블 락(Exclusive)** 을 요구하는데, 그것은 행수준 TM과 충돌한다.
```
선행 TX: SELECT … FOR UPDATE → TX(행) + TM(테이블) 보유, 미커밋
외부:    ALTER TABLE … MODIFY → 배타 테이블 락 필요 → 선행 TM과 충돌
                              → 기본(DDL_LOCK_TIMEOUT=0)이면 즉시 ORA-00054 실패
                                (양수면 그 초만큼 대기 후 실패)
선행 TX: … COMMIT/ROLLBACK    → 이때 TM 해제 → 그제서야 DDL 가능
```
그래서 **"트랜잭션이 행을 쥔 채 트랜잭션 도중 컬럼 타입이 바뀌어 정합성이 깨진다"는 일어날 수 없다.** 그 위험을 TM 락이 원천 차단한다 — 행을 잠그는(TX) 문장은 그 행이 속한 테이블이 DDL로 바뀌지 않도록 TM도 반드시 함께 잡기 때문이다.

### JPA 대응과 F004 연결
- `@Lock(LockModeType.PESSIMISTIC_WRITE)` 단 쿼리 → Hibernate가 `… FOR UPDATE` 생성. (repo `findByIdForUpdate`.)
- `entityManager.refresh(entity, LockModeType.PESSIMISTIC_WRITE)` → 그 엔티티를 `… FOR UPDATE` 로 다시 읽어 현재값+락.
- F004 비관적 락(B) 측정에서 후행 `SELECT … FOR UPDATE`의 `took`(73~128ms)이 위 ③의 대기였고, 후행이 확인 시점에 읽은 `RESERVED`가 현재모드 읽기였다. SQL Developer 2세션 재현으로 `enq: TX - row lock contention`(③의 대기 이벤트)을 직접 관측했다.

> 배타(X)는 **잠그는** 방식의 동시성 제어다. 반대편 — **잠그지 않고** 커밋 시점에 version 비교로 충돌을 걸러내는 낙관적 락(@Version) — 은 10절 Oracle SERIALIZABLE의 "낙관적(ORA-08177)"과 닮았으나, 그것은 DB가 read SCN으로 판정하는 것이고 @Version은 애플리케이션이 version 컬럼으로 판정한다는 점에서 층이 다르다.

## 12. Oracle 지원 등급 정리

### 지원 격리 수준
- READ COMMITTED (기본값)
- SERIALIZABLE
- READ ONLY (ANSI 표준 외 Oracle 고유)

READ UNCOMMITTED와 REPEATABLE READ는 등급으로 제공하지 않는다.

### READ ONLY
SERIALIZABLE과 같은 성질을 갖되 수정(INSERT/UPDATE/DELETE)을 허용하지 않는 등급. 트랜잭션 시작 시점에 커밋돼 있던 변경만 본다. 수정 없는 보고성 조회에 적합하다. (SYS / SYSDBA 접속은 READ ONLY나 SERIALIZABLE 트랜잭션을 가질 수 없다.)

### 공통 동작
Oracle은 어느 등급에서도 읽기가 쓰기를 막지 않고 쓰기가 읽기를 막지 않는다. 등급 차이는 "어느 시점 기준의 데이터를 보느냐"로 환원된다: READ COMMITTED는 쿼리 시작 시점, SERIALIZABLE은 트랜잭션 시작 시점. (그 "기준 시점"의 실제 구현이 read SCN을 잡는 단위다 — 5절.)


## 13. 자주 혼동하는 지점 모음

- 처음 본 레벨 0~3 요약은 ANSI 표준 분류다. Oracle 실제 지원 등급(RC, SERIALIZABLE, READ ONLY)과 일대일로 맞지 않는다.
- Tom Kyte 글의 표 4(READ COMMITTED)와 표 5·6(REPEATABLE READ)은 모두 **비-Oracle** DB 동작을 보여주는 표다. Oracle과 대조용이지 Oracle 동작이 아니다.
- READ COMMITTED의 기준은 트랜잭션이 아니라 **쿼리** 시작 시점이다. 트랜잭션 시작 시점으로 고정하려면 SERIALIZABLE이어야 한다.
- 락 충돌은 시간 순서가 아니라 락 호환성(S-S 호환, S-X·X-X 충돌)으로 판단한다.
- REPEATABLE READ에서 읽기 횟수 제한은 없다. 막히는 것은 읽힌 행을 수정하려는 다른 트랜잭션이고, 막히는 주체는 읽은 트랜잭션 자신이 아니다.
- 트랜잭션이 "진행 중"이라는 사실만으로 제약이 생기지 않는다. 해당 행에 실제로 건 락이 제약을 만든다.
- SERIALIZABLE이라는 이름은 직렬 실행과의 결과 동등성을 떠올리게 하지만, Oracle SERIALIZABLE은 각 트랜잭션의 시작 시점 결과를 보장할 뿐 그것을 보장하지 않는다.
- Oracle은 어느 등급에서도 읽기와 쓰기가 서로를 막지 않는다. 차이는 기준 시점뿐이다.
- Oracle "읽기 안 막힘"의 실제 구현은 락이 아니라 SCN+언두 스냅샷이다(5절). "기준 시점" 차이는 그 read SCN을 문장마다 잡느냐 트랜잭션 시작에 고정하느냐로 환원된다.
- 한 UPDATE 안에는 읽기가 둘이다 — ①찾기(스냅샷)와 ②바꾸기(현재값+락). 쓰기 충돌은 ②에서 갈리고, 이게 RC의 재시작(0행)과 SERIALIZABLE의 ORA-08177을 모두 설명한다(6·8·10절).
- 표 7의 현상 이름은 write-skew다. SERIALIZABLE이 못 막는 이유는 충돌 검출이 '쓴 행'에만 적용되기 때문이고, 같은 행을 다투는 경합이라면 ORA-08177로 잡힌다.
