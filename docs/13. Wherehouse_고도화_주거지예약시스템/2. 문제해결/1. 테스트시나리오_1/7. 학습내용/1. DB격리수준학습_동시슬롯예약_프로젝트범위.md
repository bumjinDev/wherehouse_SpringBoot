# 동시 슬롯 예약의 정합성 — 처음부터

슬롯 예약 경합(`createReservation`에서 여러 명이 같은 슬롯을 동시에 예약하는 상황)을 출발점으로, 그 경합을 막는 일이 왜 격리 수준이 아니라 쓰기 구조의 문제인지를, 오라클 내부 동작(SCN·언두·락·UPDATE의 두 단계)까지 내려가 ⓪~⑧로 정리한 문서다. ANSI 표준 등급 자체의 정리는 `3. 트랜잭션격리수준.md`에 따로 있고, 이 문서는 그 메커니즘과 슬롯 예약 적용을 다룬다.

## 목차

- ⓪ 우리가 푸는 문제 — 불변식, check-then-act, 두 겹 방어, 부분 유일 인덱스의 INSERT 통과/거부, 측정 결과
- ① 오라클의 락 — 쓰기는 잠그고, 읽기는 안 잠근다
- ② 오라클의 다중 버전 읽기 — SCN, 언두, 스냅샷
- ③ 하나의 UPDATE 안에는 읽기가 두 종류 있다
- ④ READ COMMITTED — read SCN을 '문장마다' 새로 잡는다
- ⑤ SERIALIZABLE — read SCN을 '트랜잭션 시작에 한 번' 고정한다 (+ 제어하는 범위와 방법)
- ⑥ READ COMMITTED와 SERIALIZABLE을 같은 경합에 나란히 놓고 비교
- ⑦ REPEATABLE READ와 오라클이 그것을 두지 않는 이유
- ⑧ write-skew — SERIALIZABLE 스냅샷만으로는 못 막는 정합성 문제

# ⓪ 우리가 푸는 문제

## 대상과 규칙

대상 코드는 슬롯 예약 처리 `createReservation`이다. 여기 등장하는 것은 셋이다.

- `VISIT_SLOT` 테이블에 들어 있는 **슬롯 한 행**. 예를 들어 슬롯 식별자가 410인 행이고, 이 행에는 status라는 칸이 있어서 AVAILABLE(예약 가능), RESERVED(점유됨) 같은 값을 가진다.
- `VISIT_RESERVATION` 테이블에 들어 있는 **예약 한 행**. 이 행에는 예약 식별자, 어느 슬롯인지 가리키는 slot_id, status 칸이 있다. status가 CONFIRMED이면 그 예약은 확정된 상태다.
- 같은 슬롯을 같은 순간에 잡으려고 들어오는 **여러 개의 트랜잭션**. 탐색자 여러 명이 동시에 같은 슬롯을 잡으려는 상황이다.

지켜야 할 규칙: **한 슬롯에 대해 status=CONFIRMED인 예약 행은 어느 순간에도 하나를 넘으면 안 된다.** 한 슬롯은 한 사람만 확정 예약할 수 있다는 뜻이다.

## check-then-act 틈과 race

지금 코드는 "먼저 확인하고, 통과하면 바꾼다"는 순서다.

```
한 예약 트랜잭션의 진행:

  (1) SELECT slot410             슬롯 410의 현재 status를 읽는다
  (2) if (status != AVAILABLE)   AVAILABLE이 아니면
          throw                  "이미 예약됨"으로 거절
  (3) UPDATE slot410             슬롯을 점유 상태로 바꾼다
          SET status='RESERVED'
  (4) INSERT VISIT_RESERVATION   확정 예약 행을 한 줄 기록한다
          (status='CONFIRMED')
```

(2)확인과 (3)(4)변경 사이에 시간 틈이 있고, 슬롯을 읽는 (1)이 서로를 막지 않으면 같은 순간 들어온 여럿이 전부 (1)에서 같은 AVAILABLE을 읽어 전부 (2)를 통과한다.

```
같은 slot410을 동시에 노린 다섯 트랜잭션:

  T1: (1)→AVAILABLE  (2)통과 ┐
  T2: (1)→AVAILABLE  (2)통과 │ 읽기는 서로를 막지 않으므로
  T3: (1)→AVAILABLE  (2)통과 ├ 다섯이 같은 AVAILABLE을 읽고
  T4: (1)→AVAILABLE  (2)통과 │ 다섯 다 검사를 통과
  T5: (1)→AVAILABLE  (2)통과 ┘
  → 다섯 다 (4)에서 CONFIRMED INSERT → 한 슬롯에 CONFIRMED 다섯 건 → 규칙 붕괴
```

## 규칙을 지키는 두 겹의 방어

이 규칙은 서로 독립적으로 동작하는 두 겹이 지킨다.

```
   예약 요청
      │
 ┌────┴───────────────┐     ┌──────────────────────────────┐
 │ 첫째 겹: 동시성 제어  │  →  │ 둘째 겹: 부분 유일 인덱스        │ → 커밋
 │ 경합을 직렬화해 한    │     │ INSERT가 들어올 때 DB가         │
 │ 트랜잭션만 (3)(4)로   │     │ "이 슬롯에 CONFIRMED가 이미     │
 │ 통과시킨다           │     │ 있나"를 검사해 있으면 거부        │
 │ ※ 지금 코드엔 없음    │     │ ※ 지금은 떼어 둔 상태           │
 └────────────────────┘     └──────────────────────────────┘
```

## 둘째 겹이 INSERT에서 통과/거부를 가르는 방식

둘째 겹의 정의는 이렇다.

```sql
CREATE UNIQUE INDEX UQ_VISIT_RESERVATION_CONFIRMED_SLOT
    ON VISIT_RESERVATION (
        CASE WHEN STATUS = 'CONFIRMED' THEN SLOT_ID END
    );
```

이 인덱스는 컬럼이 아니라 **`CASE WHEN STATUS='CONFIRMED' THEN SLOT_ID END` 식의 계산 결과**를 키로 삼는다. 예약 행이 INSERT될 때 그 행의 status에 따라 키가 두 갈래로 갈린다.

```
 INSERT되는 행의 status = 'CONFIRMED' 이면:
    CASE 식의 WHEN이 참 → 키 = slot_id (예: 410)

 INSERT되는 행의 status 가 'CONFIRMED'이 아니면 (CANCELLED/INVALIDATED/COMPLETED):
    WHEN이 거짓이고 ELSE가 없음 → 키 = NULL
```

여기에 오라클의 결정적 규칙이 붙는다. **오라클 B-트리 인덱스는 키가 NULL인 행을 인덱스에 저장하지 않는다. 저장하지 않으니 유일성 검사 대상도 아니다.** 그래서 인덱스가 실제로 감시하는 대상이 이렇게 갈린다.

```
 status='CONFIRMED' 행  →  키 = slot_id  →  인덱스에 저장 → 유일성 검사 대상
 그 외 status 행        →  키 = NULL     →  저장 안 함 → 검사 대상 아님(무제한 허용)
```

이 때문에 INSERT의 통과/거부가 경우별로 이렇게 갈린다. 슬롯 410을 예로 든다.

```
경우 A — slot410에 CONFIRMED가 아직 없는데 CONFIRMED를 INSERT:
   키 = 410 계산 → 인덱스에 "키 410 있나?" → 없음 → INSERT 성공, (키 410→이 행) 항목 생성

경우 B — slot410에 CONFIRMED가 이미 한 건 있는데 또 CONFIRMED를 INSERT:
   키 = 410 → 인덱스에 "키 410 있나?" → 있음 → 유일성 위반
   → INSERT 거부, ORA-00001 (unique constraint violated)

경우 C — slot410에 CANCELLED(또는 INVALIDATED/COMPLETED)를 INSERT:
   키 = NULL → 인덱스에 저장·검사 안 함
   → INSERT 항상 성공. 다른 CONFIRMED가 있든 없든, 이런 행이 몇 개든 무관
```

한 줄로: **거부되는 경우는 "status=CONFIRMED이고 그 slot_id의 CONFIRMED 항목이 인덱스에 이미 있을 때"뿐**이다. 첫 CONFIRMED와, CONFIRMED가 아닌 모든 행은 통과한다. 그래서 한 슬롯은 시간에 걸쳐 취소·무효 이력 행을 여러 줄 쌓아도(CANCELLED가 여러 건) CONFIRMED는 늘 최대 한 건으로 유지된다. 이름이 "부분(partial) 유일 인덱스"인 이유가 이것이다. 전체 행이 아니라 status가 CONFIRMED인 행이라는 부분 집합에 대해서만 "slot_id가 유일해야 한다"를 강제한다. (오라클에는 "특정 조건의 행만 인덱스" 같은 전용 문법이 없어서, 위처럼 CASE 식이 제외 대상 행에 대해 NULL을 내도록 만들고, "NULL 키는 저장·검사하지 않는다"는 성질을 이용해 같은 효과를 낸다.)

## 같은 슬롯에 CONFIRMED가 동시에 들어올 때

위 경우 A/B는 INSERT를 한 번에 하나씩 할 때다. 다섯이 동시에 CONFIRMED를 INSERT하면, 같은 키 410을 동시에 만들려는 충돌이 생긴다. 이때 오라클은 유일 키에 대해 다음처럼 동작한다.

```
시각  T1                                T2
──────────────────────────────────────────────────────────────────
u1   INSERT CONFIRMED(slot410)
       키 410을 인덱스에 넣음 (아직 COMMIT 안 함)
       → 키 410을 미커밋 상태로 '점유'
u2                                      INSERT CONFIRMED(slot410)
                                          키 410을 넣으려 함
                                          → 키 410이 T1의 미커밋 INSERT에 잡혀 있음
                                          → T1이 끝날 때까지 대기
                                            (enq: TX - row lock contention, 공유 대기)
u3   COMMIT → 키 410 확정
u4                                      대기 풀림
                                          → 키 410이 이미 확정 점유됨
                                          → ORA-00001 (거부)
```

만약 u3에서 T1이 COMMIT이 아니라 ROLLBACK이면, u4에서 키 410이 비므로 T2의 INSERT는 성공한다. 즉 **먼저 커밋한 쪽이 키를 차지하고, 나머지 동시 CONFIRMED는 대기했다가 거부(ORA-00001)된다.** 다섯이 동시면 한 건이 키를 차지하고 네 건이 거부된다.

## 동시 INSERT의 읽기 단계까지 포함한 전 과정

위 동시 INSERT 도식은 INSERT 충돌(인덱스 키 410 다툼)만 떼어 그렸고, 그 앞의 읽기 단계를 통째로 생략했다. 생략한 이유는 "오라클에서 SELECT는 락을 걸지 않으니, 읽기 단계엔 그릴 락이 없다"였다. 그런데 이 생략은 잘못이다. **읽기가 락을 안 잡는다는 사실 자체가, 다섯 트랜잭션이 전부 검사를 통과해 INSERT까지 도달하는 이유**이기 때문이다. 즉 "읽기 락이 없다"는 건 그려야 할 핵심이지 빼도 되는 부분이 아니다. 그래서 읽기 단계를 락 표기까지 붙여 다시 그린다. (다섯이면 같은 구조가 다섯으로 늘 뿐이다.)

```
출발: slot410 = 'AVAILABLE',  인덱스에 키 410 없음

━━ 읽기 단계 ━━ (오라클에서 SELECT는 락을 전혀 안 잡는다) ━━━━━━━━━━━━━━━━
시각  T1                                      T2
r1   SELECT slot410 (findById)
       ├ 락: 없음
       └ 스냅샷으로 'AVAILABLE' 읽음
r2                                            SELECT slot410 (findById)
                                                ├ 락: 없음
                                                └ 'AVAILABLE' 읽음
        ※ 둘 다 아무 락도 안 잡으므로 서로를 못 막는다 → 둘 다 같은 AVAILABLE을 본다
r3   if(status != AVAILABLE) → 거짓 → 검사 통과   if(status != AVAILABLE) → 검사 통과
r4   중복매물/시간겹침 COUNT 조회                중복매물/시간겹침 COUNT 조회
       └ 락: 없음 (역시 SELECT)                   └ 락: 없음
        ※ 여기까지 락이 하나도 없다. 그래서 둘 다 막힘 없이 쓰기 단계로 진입한다.
         "읽기 락 과정"이 비어 있는 게 아니라, 읽기에는 락 자체가 없다는 것이
          이 race가 성립하는 원인이다 

━━ 쓰기 단계 ━━ (여기서 처음으로 락이 등장한다) ━━━━━━━━━━━━━━━━━━━━━━
w1   UPDATE slot410 SET status='RESERVED'
       └ slot410 행에 X-락 획득 (미커밋 보유)
w2                                            UPDATE slot410 SET status='RESERVED'
                                                └ 같은 행 X-락 시도 → T1 보유중
                                                  → X끼리 공존 불가 → 대기
w3   INSERT VISIT_RESERVATION(slot410, CONFIRMED)
       ├ 인덱스 키 계산 → status='CONFIRMED' → 키=410
       ├ 인덱스 탐침: "키 410 있나?" → 없음
       └ 키 410을 미커밋 상태로 점유 (인덱스 항목 생성)
        ※ T2는 아직 w2에서 슬롯 X-락 대기 중이라 INSERT까지 못 왔음
w4   COMMIT → slot410 X-락 해제 + 키 410 확정
w5                                            (w2 대기 풀림) UPDATE 진행
                                                → INSERT VISIT_RESERVATION(slot410, CONFIRMED)
                                                  ├ 키=410 탐침: "키 410 있나?" → 있음(확정됨)
                                                  └ 유일성 위반 → ORA-00001 거부
```

여기서 두 가지를 명확히 한다.

**(1) 읽기에는 락이 없다 — 그게 빠진 게 아니라 원래 없는 것이다.** r1~r4의 어느 SELECT도 락을 잡지 않는다. 그래서 다섯이 같은 AVAILABLE을 동시에 보고 검사를 다 통과한다. 직렬화(순서 강제)는 오직 쓰기 단계(w1의 슬롯 X-락, w3의 인덱스 키 점유)에서만 처음 생긴다. 앞 도식이 곧장 INSERT(w3 부근)부터 시작한 건 "거기서 처음 락이 잡히니까"였지만, 읽기 단계를 빼면 "왜 다섯이 다 INSERT까지 왔는가"가 사라진다. 그래서 읽기 단계를 "락 없음"으로 명시해 다시 넣은 것이다.

**(2) INSERT가 대기하는 것은 "읽기 락"이 아니라 "쓰기 충돌(TX 큐)"이다.** w5에서 T2의 INSERT가 멈춘다면, 그건 T2가 무엇을 읽으려다 막힌 게 아니다. T2의 INSERT가 키 410을 넣으려는데 그 키를 다른 트랜잭션이 미커밋 상태로 점유하고 있어서다. 이때 T2는 그 트랜잭션이 커밋/롤백으로 결판날 때까지 기다린다. 이 대기는 `enq: TX - row lock contention`(공유 모드 대기)이라는 쓰기 측 충돌이지, 읽기 락이 아니다. INSERT가 키 존재를 확인할 때 하는 "인덱스 탐침"은 락을 거는 읽기가 아니라 인덱스 유지 과정의 내부 조회일 뿐이다.

정리하면, 그 도식에 "읽기 락 과정"이 없던 게 아니라, 이 흐름에는 읽기 락이라는 것이 존재하지 않는다. 읽기는 r1~r4처럼 락 없이 지나가고(그래서 다섯이 다 통과), 락과 대기는 전부 쓰기 단계(슬롯 X-락, 인덱스 키 점유와 그에 대한 TX 큐 대기)에서만 일어난다. 읽기 단계에 락을 걸어 다투게 만들고 싶다면 그건 `SELECT ... FOR UPDATE`를 명시적으로 쓸 때이고, 지금 코드에는 그게 없다.

## 측정 결과

이 동작 때문에 측정 결과가 이렇게 나온다.

```
Step 1 (첫째 겹 없음 + 둘째 겹 인덱스 있음):
   다섯 다 (2)검사 통과 → 다섯 다 (4)INSERT 시도
   → 한 건이 키 410을 차지, 네 건은 대기 후 ORA-00001 → 최종 CONFIRMED 1건
   → 막긴 막지만 "다섯이 다 시도하고 네 건이 뒤늦게 터지는" 사후 거부

Step 2 (첫째 겹 없음 + 둘째 겹 인덱스도 제거):
   다섯 다 (2)통과 → 인덱스가 없으니 (4) 다섯 다 성공 → CONFIRMED 5건 → 규칙 붕괴
```

인덱스(둘째 겹)는 비싸게(다 해 보고 실패), 그것도 일이 다 벌어진 뒤에 막는 최후 방어선이다. 더 싸게·미리 막으려면 첫째 겹(동시성 제어)이 필요하다. 그 첫째 겹을 제대로 고르려면 오라클이 동시 읽기·쓰기를 내부에서 어떻게 처리하는지 알아야 한다. 그래서 다음으로 오라클의 락부터 본다.

# ① 오라클의 락 — 쓰기는 잠그고, 읽기는 안 잠근다

대상: 행(row)이라는 자원에 걸리는 잠금 표식, 그리고 그것을 거는 트랜잭션.

오라클이 동시 접근을 다루는 첫 번째 규칙은 다음 두 줄이다.

```
[쓰기] UPDATE / DELETE / INSERT 를 하면
       → 그 행에 X-락(eXclusive lock, 배타 잠금)을 건다.
       → 이 X-락은 그 트랜잭션이 COMMIT 또는 ROLLBACK 할 때까지 유지된다.
       → 격리 수준이 무엇이든 항상 이렇게 한다.

[읽기] SELECT 를 하면
       → 락을 전혀 걸지 않는다.
       → 대신 ②에서 설명할 스냅샷(다중 버전)으로 값을 읽는다.
```

이 규칙의 직접적 결과: 같은 행을 두 트랜잭션이 동시에 쓰려고 하면 충돌이 나지만, 읽는 것은 서로 막지 않는다.

```
같은 slot410을 두 트랜잭션이 쓰려 할 때:

   T1: UPDATE slot410 ... → slot410에 X-락 획득 ──┐ 같은 행
   T2: UPDATE slot410 ... → X-락 잡으려 함 ───────┘
        이미 T1이 X-락 보유 → X끼리 공존 불가 → T2 대기
        T1이 COMMIT/ROLLBACK으로 풀 때까지 멈춤
```

왜 이게 중요한가: 읽기는 아무도 안 잠그니까, ⓪의 (1)SELECT 단계에서 N개 트랜잭션이 전부 동시에 슬롯을 읽고 (2)검사를 통과한다 — 이게 TOCTOU 틈의 물리적 원인이다. 그리고 **실제 직렬화(순서 강제)는 오직 쓰기(X-락)에서만 일어난다.** 이 사실 하나가 뒤의 ⑨(설계)에서 "격리 수준은 정합성의 레버가 아니다"라는 결론으로 곧장 이어진다. 읽기를 아무리 격리해도, 막는 일은 쓰기에서 벌어지기 때문이다.

(읽기에도 잠금(S-락)을 거는 다른 종류의 엔진이 있다. 그건 ⑦에서 REPEATABLE READ를 설명할 때 별도로 꺼낸다. 오라클은 읽기에 락을 걸지 않는다는 점만 여기서 확실히 해 둔다.)

# ② 오라클의 다중 버전 읽기 — SCN, 언두, 스냅샷

①에서 "읽기는 락을 안 걸고 스냅샷으로 읽는다"고 했다. 그 스냅샷이 정확히 무엇인지 여기서 끝까지 푼다.

풀려는 문제: T1이 slot410을 바꾸는 중(아직 커밋 전, 또는 방금 커밋)인데 T2가 같은 slot410을 읽으려 한다. T2는 어떤 값을 봐야 하는가? 락 방식이면 "T1 끝날 때까지 기다려"가 답이지만, 오라클은 다르게 푼다 — **옛 값을 버리지 않고 보관해 두었다가, 읽는 쪽에게 "특정 시점의 옛 모습"을 재구성해서 보여준다.** 그래서 읽는 쪽은 기다리지 않는다. 이걸 위해 부품 셋을 쓴다.

```
SCN (System Change Number)
   = 커밋이 일어날 때마다 1씩 올라가는, 데이터베이스 전체의 버전 번호(논리 시계).
     "SCN 105" = 105번째 변경 묶음까지 커밋된 시점의 상태.

언두(undo, 롤백 세그먼트)
   = 어떤 행을 바꿀 때, 새 값을 테이블에 쓰기 '전에' 그 행의 옛 값을 따로 복사해 두는 영역.
     덕분에 옛 값이 사라지지 않는다.

스냅샷
   = "특정 SCN 시점의 데이터 모습".
     중요한 건, 이게 테이블을 통째로 복사한 사진이 아니라는 점이다.
     스냅샷의 실체는 숫자 하나(read SCN)이고,
     읽을 때 블록마다 언두를 이용해 "그 SCN 시점의 값"을 즉석에서 재구성하는 방식으로 구현된다.
```

**스냅샷이 실제로 동작하는 단계별 트레이스 (slot410)**:

```
출발 상태: slot410 = 'AVAILABLE', 이 값이 커밋된 시점의 버전은 SCN 100 이하라고 하자.

  ┌ 물리 테이블 블록(실제 디스크/버퍼):
  │     처음:  [ slot410 = AVAILABLE ]
  │     T1이 UPDATE→'RESERVED' 하고 COMMIT (커밋 SCN=105):
  │            [ slot410 = RESERVED ]   ← 블록의 현재 값은 이제 RESERVED, 버전 도장 105
  │ 언두 영역:  [ 옛값: slot410 = AVAILABLE ]  ← 바꾸기 전에 복사해 둔 옛 값
  └

  이제 read SCN=100 으로 고정된 어떤 트랜잭션이 SELECT slot410 을 한다:
     (1) 블록을 본다 → 현재 값은 RESERVED, 버전 도장 105
     (2) 비교: "105 > 내 기준 100" → 이건 내 시점(100) 이후에 생긴 변경이다, 나에겐 너무 최신
     (3) 언두로 간다 → 옛값(AVAILABLE)을 꺼낸다
     (4) 'SCN 100 시점'의 블록을 사적인 임시 버퍼에 재구성한다
     (5) 그 재구성된 값 'AVAILABLE'을 반환한다
  → 이 동안 T1(쓰는 쪽)을 막지도, 기다리지도 않는다. (읽기·쓰기가 서로 안 막음)
```

**스냅샷의 범위와 물리적 실체** — 이 부분을 정확히 해야 오해가 안 생긴다.

```
(가) 일관성의 '논리적 범위'는 데이터베이스 전체다.
     read SCN=100 으로 고정된 트랜잭션은 어떤 테이블을 읽든 전부 SCN 100 시점으로 본다.
        SELECT VISIT_SLOT ...        → SCN 100 시점
        SELECT VISIT_RESERVATION ... → SCN 100 시점
        SELECT MEMBERTBL ...         → SCN 100 시점
     세 테이블이 서로 같은 한 순간을 반영하므로 상호 모순이 없다.
     → "그 행이 들어 있는 테이블 하나만"의 스냅샷이 아니다. DB 전체가 한 시점으로 일관된다.

(나) 그러나 '물리적'으로는 아무것도 미리 복사하지 않는다.
     내가 실제로 읽은 블록만, 그것도 그 블록이 내 read SCN 이후에 바뀌었을 때만,
     언두로 그 시점 값을 즉석 재구성한다. 안 읽은 데이터는 손도 안 댄다.

(다) 증거 — ORA-01555(snapshot too old):
     내 read SCN 시점 값을 재구성하려는데, 그 사이 언두가 다른 트랜잭션에 의해
     덮여 사라졌다면 → 재구성할 재료가 없어 ORA-01555 에러가 난다.
     만약 스냅샷이 '테이블을 떠 둔 복사본'이었다면 이 에러가 날 이유가 없다.
     이 에러의 존재 자체가 "스냅샷 = 언두로 그때그때 재구성"이라는 증거다.
```

여기서 얻는 핵심: 격리 수준의 차이는 결국 **"이 read SCN이라는 숫자를 언제 잡고, 언제까지 그대로 쓰느냐"** 하나로 환원된다. 이 한 가지가 READ COMMITTED와 SERIALIZABLE을 가른다(④, ⑤).

# ③ 하나의 UPDATE 안에는 읽기가 두 종류 있다 — 모든 것의 열쇠

이 개념이 뒤의 모든 결론(④, ⑤, ⑨)을 지탱하므로, 등장하는 자리에서 완전히 펼친다.

대상: `UPDATE ... WHERE ...` 한 문장. 이 한 문장은 내부적으로 두 단계로 나뉜다.

```
UPDATE slot410 SET status='RESERVED' WHERE slot_id=410 AND status='AVAILABLE'

 ┌──────── ① 찾기 단계 (consistent read, 일관성 읽기) ────────┐
 │ WHERE 조건에 맞는 행을 '고른다'.                            │
 │ 이때 읽는 값은 ②에서 설명한 내 스냅샷(read SCN) 시점 값.   │
 │ 이 단계는 락을 걸지 않는다.                                 │
 └────────────────────────────────────────────────────────────┘
 ┌──────── ② 바꾸기 단계 (current read + 락) ─────────────────┐
 │ 찾은 행을 '실제로 고친다'.                                  │
 │ 행을 고치려면 스냅샷(옛 시점) 값이 아니라                    │
 │ 그 행의 지금 현재 커밋된 최신 버전을 봐야 한다.           │
 │ (낡은 버전 위에 덮어쓸 수는 없으니까.)                      │
 │ 그리고 그 행에 X-락을 걸고 값을 바꾼다.                      │
 │ 핵심: 이 단계에서 WHERE 조건을 현재값 기준으로 다시 확인한다. │
 └────────────────────────────────────────────────────────────┘
```

왜 이게 열쇠인가:

- ①찾기는 스냅샷을 읽으므로 **격리 수준이 무엇을 보느냐를 지배한다.**
- ②바꾸기는 격리 수준과 상관없이 **언제나 "현재 최신 버전 + X-락"으로 동작한다.**

즉 한 UPDATE 안에서 "내가 읽은 시점의 값"과 "내가 실제로 덮어쓰는 순간의 값"이 다를 수 있고, 둘 사이에 다른 트랜잭션이 끼어들면 ②바꾸기 단계에서 그게 드러난다. 이 ②단계의 동작 방식이 READ COMMITTED와 SERIALIZABLE에서 다르게 처리되는 것이 ④·⑤의 내용이다.

# ④ READ COMMITTED — read SCN을 '문장마다' 새로 잡는다

## (가) 정의

READ COMMITTED은 오라클의 기본 격리 수준이고, 지금 `createReservation`도 이 수준에서 돈다. 정의는 이렇다.

```
한 트랜잭션 안에서도, 각 SQL 문장이 시작될 때마다 read SCN을 그 순간의 최신값으로 새로 잡는다.
→ 매 문장은 "그 문장이 시작되기 직전까지 커밋된 데이터"를 본다.
→ 같은 트랜잭션이라도 문장이 다르면 보는 시점이 다를 수 있다.
```

(read SCN은 데이터베이스 전체의 버전 번호다. 커밋이 일어날 때마다 1씩 오른다. "read SCN=100으로 읽는다"는 건 100번째 변경까지 커밋된 시점의 데이터 값을 읽는다는 뜻이다.)

## (나) 직접적 결과 1 — 같은 행을 두 번 읽으면 값이 달라질 수 있다 (non-repeatable read)

문장마다 read SCN을 새로 잡으니, 한 트랜잭션 안에서 같은 행을 두 번 읽는 사이에 다른 트랜잭션이 그 행을 바꿔 커밋하면, 두 번째 읽기가 그 변경을 본다.

```
출발: slot410 = 'AVAILABLE'

시각  T1 (READ COMMITTED)                      T2 (다른 트랜잭션)
─────────────────────────────────────────────────────────────────────
t1   SELECT status FROM slot410
       └ 이 문장 시작 시 read SCN을 새로 잡음 (=100) → 'AVAILABLE'
t2                                              UPDATE slot410 SET status='RESERVED'
                                                COMMIT  (커밋 SCN=105)
t3   SELECT status FROM slot410   ← 같은 트랜잭션 T1이 같은 행을 다시 읽음
       └ 이 문장 시작 시 read SCN을 또 새로 잡음 (=106) → 'RESERVED'
─────────────────────────────────────────────────────────────────────
T1은 한 트랜잭션 안에서 같은 slot410을 두 번 읽었는데 'AVAILABLE' → 'RESERVED'로 달라졌다.
이것을 non-repeatable read(반복 불가능한 읽기)라 한다. READ COMMITTED는 이를 허용한다.
원인: read SCN을 '문장마다' 새로 잡으므로, t3 문장이 t2의 커밋(SCN105)을 본다.
```

## (다) 직접적 결과 2 — 그런데 조건부 UPDATE에서는 경합을 막아낸다: 쓰기 일관성 재시작

먼저 이 단락이 혼자 읽혀도 되도록, UPDATE 한 문장이 내부에서 두 단계로 나뉜다는 점부터 다시 명시한다.

```
UPDATE slot410 SET status='RESERVED' WHERE slot_id=410 AND status='AVAILABLE'
   ① 찾기  : WHERE 만족 행을 '고른다'. 이때 읽는 값은 내 스냅샷(read SCN) 시점 값. 락 없음.
   ② 바꾸기: 고른 행을 '실제로 바꾼다'. 이때는 스냅샷이 아니라 그 행의 '현재 커밋 최신 버전'을
             봐야 하고, X-락을 걸고 값을 바꾼다. 여기서 WHERE를 현재값으로 다시 확인한다.
```

READ COMMITTED에서, ②바꾸기가 다른 트랜잭션의 락에 막혔다가 그 트랜잭션이 커밋되면, RC는 **문장을 새 read SCN으로 재시작**할 수 있다. 이게 핵심이다. 단계로 본다.

```
출발: slot410 = 'AVAILABLE' (SCN 100)
두 트랜잭션이 같은 문장을 실행:
   UPDATE slot410 SET status='RESERVED' WHERE slot_id=410 AND status='AVAILABLE'

s1  T1: 위 UPDATE 실행
        ├ ①찾기 (스냅샷 SCN100): slot410 = AVAILABLE → 조건 만족 → 대상으로 선택
        └ ②바꾸기: slot410에 X-락 획득, RESERVED로 기록 (아직 COMMIT 안 함, 락 보유)
s2  T2: 같은 UPDATE 실행
        ├ ①찾기 (스냅샷 SCN100): slot410 = 아직 AVAILABLE → 대상으로 선택
        └ ②바꾸기: 같은 slot410에 X-락 시도 → T1이 보유 중 → 대기
s3  T1: COMMIT → 커밋 SCN=105, slot410의 X-락 해제
s4  T2: 대기 풀림. ②바꾸기를 진행하려는데, 여기서 WHERE가 두 번째로 평가된다:
        ├ 시점: 막혔던 X-락을 다시 잡는 바로 그 순간 (= T1 커밋 직후). ①찾기 때가 아니다.
        ├ 기준: ①찾기 때 본 스냅샷(100)이 아니라 slot410의 현재 커밋 버전(SCN105)
        ├ 현재 status = RESERVED → WHERE의 'status=AVAILABLE'이 거짓
        └ → 이 행은 더 이상 조건에 안 맞음. 오라클은 '쓰기 일관성'을 위해 문장을 재시작한다:
             · T2가 이 문장에서 한 작업을 되돌리고
             · 새 read SCN(106)을 잡아 문장을 처음부터 다시 실행
             · 재실행된 ①찾기는 slot410이 RESERVED라 아예 대상으로 안 고름
             → 결과: 0 rows updated (T2는 아무 행도 안 바꿈, 락도 안 잡음)
s5  T2: 영향 행 수가 0임을 보고 → "이미 예약됨"으로 처리
```

여기서 재시작/재평가의 기준과 시점을 못 박는다.

- **시점**: ②바꾸기 단계에서 막혔던 락을 다시 잡는 순간이다. 즉 선착이 커밋한 직후. ①찾기(스냅샷을 읽던 때)가 아니다.
- **기준**: 내 스냅샷 값이 아니라 그 행의 **현재 커밋된 최신 버전**이다.
- **재시작 트리거**: 선착이 커밋하면서 내 WHERE가 참조하는 컬럼(여기선 status)을 바꿔 놓았을 때다. 오라클은 문제의 그 행만 슬쩍 건너뛰는 게 아니라, 문장 전체를 새 read SCN으로 다시 실행한다. 그 문장이 옛 스냅샷 기준으로 다른 행을 이미 건드렸을 수도 있어서, "한 시점 기준의 일관된 결과 집합"을 보장하려면 통째로 다시 해야 하기 때문이다.

## (라) 반례 — WHERE에 status 조건이 없으면 재시작해도 못 막는다

위 막힘은 WHERE에 `status='AVAILABLE'`이 있었기에 성립했다. 만약 그 조건이 없다면 어떻게 되는지 같은 흐름으로 본다.

```
문장: UPDATE slot410 SET status='RESERVED' WHERE slot_id=410   ← status 조건 없음

s4'  T2: 대기 풀림. ②바꾸기 재평가:
         ├ 기준: 현재 커밋 버전(slot410 = RESERVED)
         ├ WHERE는 'slot_id=410' 뿐 → 현재 버전도 slot_id는 그대로 410 → 조건이 여전히 참
         └ → 행이 탈락하지 않음 → RESERVED를 RESERVED로 다시 덮어씀 (1 row updated)
             → 그대로 다음 단계로 진행
```

즉 재시작·재평가는 똑같이 일어나지만, 비교할 조건이 변하지 않는 컬럼(slot_id)뿐이라 후착이 안 걸러진다. 슬롯당 1건을 막는 힘은 `AND status='AVAILABLE'`이라는 그 한 조각에서 나온다.

이 점이 측정 Step 2와도 맞물린다. 그 회차의 코드는 슬롯 상태를 미리 SELECT로 확인하고(읽기는 락이 없어 다섯이 다 통과), 슬롯 UPDATE에 status 조건이 없으며, 둘째 겹 인덱스도 없었다. 그래서 다섯의 슬롯 UPDATE는 X-락으로 한 줄씩 직렬화되긴 했지만 각자 RESERVED를 다시 덮어쓸 뿐 아무도 거절되지 않았고, 예약 INSERT도 유일 인덱스가 없어 다섯 다 성공했다 → CONFIRMED 5건.

## (마) 핵심

```
READ COMMITTED:
  · read SCN을 '문장마다' 새로 잡는다.
    → 읽기는 트랜잭션 안에서 흔들릴 수 있다 (non-repeatable read 허용).
  · 쓰기(②바꾸기)가 충돌하면, 락 풀린 순간 '현재 커밋 버전'으로 WHERE를 재평가하고,
    어긋나면 문장을 새 SCN으로 재시작한다 (쓰기 일관성).
    → 조건부 UPDATE의 WHERE에 status 조건이 있으면 후착은 0행으로 걸러진다.
    → 격리 수준을 더 올리지 않아도(기본 RC) 이 막힘은 성립한다.
    → 단, WHERE에 그 조건이 없으면(변하지 않는 컬럼만 비교) 재시작해도 못 막는다.
```

# ⑤ SERIALIZABLE — read SCN을 '트랜잭션 시작에 한 번' 고정한다

## (가) 정의

SERIALIZABLE은 오라클이 제공하는 더 강한 격리 수준이다. 정의는 이렇다.

```
read SCN을 트랜잭션이 시작될 때 한 번 정하고,
그 트랜잭션이 COMMIT 또는 ROLLBACK으로 끝날 때까지 그 값을 바꾸지 않고 그대로 쓴다.
→ 트랜잭션 안의 모든 SELECT, 그리고 모든 UPDATE의 ①찾기 단계가
  이 동일한 read SCN을 기준으로 데이터를 읽는다.
→ 트랜잭션 시작 이후에 다른 트랜잭션이 어떤 값을 커밋하더라도, 내 read SCN 값은 변하지 않는다.
```

설정 방법은 트랜잭션의 첫 문장으로 `SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;`을 두는 것이다. (read SCN은 데이터베이스 전체의 버전 번호이고, 커밋이 일어날 때마다 1씩 증가한다. "read SCN=100으로 읽는다"는 100번째 변경까지 커밋된 시점의 데이터 값을 읽는다는 뜻이다.)

## (나) 직접적 결과 1 — 같은 조회는 트랜잭션 내내 같은 값이고, 새 행도 결과에 안 들어온다

read SCN이 시작 시점에 고정되므로, 트랜잭션이 진행되는 동안 다른 트랜잭션이 값을 커밋해도 내 읽기 결과는 변하지 않는다.

```
출발: slot410 = 'AVAILABLE'

시각  T1 (SERIALIZABLE)                         T2 (다른 트랜잭션)
─────────────────────────────────────────────────────────────────────
t0   SET TRANSACTION ISOLATION LEVEL SERIALIZABLE
       → 이 트랜잭션의 read SCN = 100 으로 정해짐 (끝까지 이 값 유지)
t1   SELECT status FROM slot410
       → read SCN=100 기준으로 읽음 → 'AVAILABLE'
t2                                              UPDATE slot410 SET status='RESERVED'
                                                COMMIT  (커밋 SCN=105)
t3   SELECT status FROM slot410   ← 같은 트랜잭션 T1이 같은 행을 다시 읽음
       → read SCN은 여전히 100 → 'AVAILABLE'   ← t1과 같은 값
─────────────────────────────────────────────────────────────────────
t2에서 T2가 RESERVED로 바꿔 커밋했는데도, T1은 t3에서 여전히 'AVAILABLE'을 읽는다.
같은 조회가 트랜잭션 내내 같은 값을 반환한다 = repeatable read(반복 가능한 읽기). READ COMMITTED와 정반대.
원인: read SCN이 시작 시점(100)에 고정되어 있어, 100보다 큰 SCN에서 커밋된 변경(105)은 읽기 대상에서 제외된다.
```

같은 이유로 **새로 INSERT된 행도 결과에 들어오지 않는다.** T1이 `SELECT COUNT(*) FROM slot WHERE window_id=42 AND status='AVAILABLE'`을 두 번 실행하는 경우, 그 사이 다른 트랜잭션이 조건에 맞는 슬롯을 새로 INSERT하고 커밋해도, 그 새 행은 read SCN=100 시점에 존재하지 않은 행이므로 두 번째 조회의 COUNT에 포함되지 않는다. 즉 phantom(없던 행이 조회 결과에 새로 나타나는 현상)도 발생하지 않는다.

## (다) 직접적 결과 2 — 쓰기 충돌은 ORA-08177로 거부된다

이 단락이 따로 읽혀도 되도록, UPDATE 한 문장의 내부 두 단계를 다시 명시한다.

```
UPDATE slot410 SET status='RESERVED' WHERE slot_id=410 AND status='AVAILABLE'
   ① 찾기  : WHERE 조건에 맞는 행을 고른다. 읽는 값은 내 read SCN 시점 값이다. 락을 걸지 않는다.
   ② 바꾸기: 고른 행을 실제로 수정한다. 이때는 read SCN 시점 값이 아니라 그 행의
             '현재 커밋된 최신 버전'을 읽어야 하고, 그 행에 X-락을 걸고 값을 바꾼다.
```

같은 슬롯을 두 SERIALIZABLE 트랜잭션이 동시에 예약하는 흐름이다. 두 트랜잭션 모두 T1이 커밋하기 전에 시작했다고 둔다.

```
출발: slot410 = 'AVAILABLE' (SCN 100)

s1  T1: SET TRANSACTION ISOLATION LEVEL SERIALIZABLE → read SCN=100 으로 고정
s2  T2: SET TRANSACTION ISOLATION LEVEL SERIALIZABLE → read SCN=100 으로 고정
s3  T1: UPDATE slot410 SET status='RESERVED' WHERE slot_id=410 AND status='AVAILABLE'
        ├ ①찾기 (read SCN=100): slot410 = AVAILABLE → 조건 만족 → 대상으로 선택
        └ ②바꾸기: slot410에 X-락 획득, RESERVED로 기록 (아직 COMMIT 안 함, 락 보유)
s4  T2: 같은 UPDATE 실행
        ├ ①찾기 (read SCN=100): slot410 = 아직 AVAILABLE → 대상으로 선택
        └ ②바꾸기: 같은 slot410에 X-락 시도 → T1이 보유 중 → 대기
s5  T1: COMMIT → 커밋 SCN=105, slot410의 X-락 해제
s6  T2: 대기 풀림. ②바꾸기를 진행하려는데:
        ├ 수정하려는 slot410의 '현재 커밋 버전'의 SCN = 105
        ├ 내 read SCN = 100. 즉 SCN 105는 "내가 트랜잭션을 시작한(100) 뒤에 커밋된" 변경이다
        ├ READ COMMITTED라면 여기서 새 read SCN으로 문장을 재시작할 수 있다 → 0행으로 끝난다
        ├ 그러나 SERIALIZABLE은 read SCN을 100으로 고정해 두고 바꾸지 않는다 → 재시작을 할 수 없다
        ├ 재시작이 불가능하므로 이 수정을 정의에 맞게 진행할 방법이 없다
        └ → ORA-08177 (can't serialize access for this transaction) → 트랜잭션 롤백
```

충돌 검출의 기준과 시점을 명시한다.

- **시점**: ②바꾸기에서 막혔던 X-락을 다시 잡는 시점이다(같은 행 경합이면 선착 커밋 직후). 문장을 다 실행한 뒤도, 커밋 시점도 아니다. 행을 수정하려는 그 시점에, 행 단위로 판정한다.
- **기준**: "수정하려는 행의 현재 커밋 버전의 SCN"과 "내 read SCN"을 비교한다. 현재 커밋 버전의 SCN이 내 read SCN보다 크면(= 내가 트랜잭션을 시작한 뒤에 다른 트랜잭션이 그 행을 바꿔 커밋했으면) 충돌로 판정한다.

## (라) 왜 재시작을 못 하나, 그리고 SELECT 값과 UPDATE 가능 여부가 일치하지 않는 이유

**왜 재시작을 못 하나**: 재시작은 새 read SCN을 잡아 더 큰 SCN 시점의 데이터로 문장을 다시 실행하는 동작이다. 그런데 SERIALIZABLE은 정의상 트랜잭션 내내 하나의 read SCN만 사용한다. 재시작은 read SCN을 바꾸는 것이므로 이 정의에 위배된다. 따라서 SERIALIZABLE에서는 READ COMMITTED가 쓰던 재시작(그 결과인 0행 처리)을 할 수 없고, 대신 오라클이 ORA-08177 예외를 발생시킨다.

**SELECT 값과 UPDATE 가능 여부의 불일치**: 위 s6 시점에 T2가 `SELECT slot410`을 실행하면, read SCN=100 기준으로 여전히 status='AVAILABLE'을 반환한다. 그런데 T2가 같은 행을 UPDATE하려 하면, ②바꾸기가 현재 커밋 버전(SCN 105, status=RESERVED)을 읽고 그 SCN이 T2의 read SCN(100)보다 크다는 것을 확인해 ORA-08177로 거부한다. 즉 **SELECT가 반환하는 값(AVAILABLE)과, 그 값을 근거로 한 UPDATE의 성공 여부(거부)가 일치하지 않는다.** 충돌 판정이 ①찾기(읽기)가 아니라 ②바꾸기(쓰기) 시점에 이뤄지기 때문이다.

**충돌이 나기 위한 조건**: 이 ORA-08177은 "후착 트랜잭션의 read SCN이 선착 커밋 SCN보다 작을 때"만 발생한다. 위에서 T2(s2)가 T1 커밋(s5)보다 먼저 시작했기 때문에 read SCN 100 < 커밋 SCN 105가 되어 충돌이 생겼다. 만약 T2가 T1 커밋 이후에 트랜잭션을 시작했다면, T2의 read SCN은 105 이상이 되고, ①찾기 단계에서 이미 status=RESERVED를 읽으므로 조건부 UPDATE는 대상이 없어 0행으로 끝난다(예외 없음). 충돌은 두 트랜잭션의 실행 구간이 겹칠 때만 발생한다.

## (마) 핵심

```
SERIALIZABLE:
  · read SCN을 '트랜잭션 시작에 한 번' 정하고 끝까지 그 값을 쓴다.
    → 같은 조회는 트랜잭션 내내 같은 값을 반환한다 (repeatable read).
    → 새로 커밋된 행도 결과에 안 들어온다 (no phantom).
  · ②바꾸기에서 "수정하려는 행의 현재 커밋 버전 SCN > 내 read SCN"이면(= 내 시작 이후 그 행이
    바뀌어 커밋되었으면), 재시작이 정의상 불가능하므로 ORA-08177을 발생시키고 롤백한다.
  · 정합성 결과는 READ COMMITTED와 같다 (슬롯당 CONFIRMED 1건).
    다른 것은 후착이 받는 신호뿐 — READ COMMITTED는 0행, SERIALIZABLE은 ORA-08177 예외.
```

## (보충) SERIALIZABLE이 제어하는 범위와 방법

SERIALIZABLE은 두 가지를 제어한다. 하나는 읽기 측(어떤 데이터를, 어느 시점 기준으로 보게 할 것인가)이고, 다른 하나는 쓰기 측(충돌을 어떻게 판정해 거부할 것인가)이다. 쓰기 측 충돌 판정(ORA-08177)은 위 (다)에서 다뤘다. 여기서는 읽기 측, 즉 read SCN 스냅샷이 무엇을 어디까지 어떻게 보장하느냐를 정리한다.

**1. 범위 — read SCN 하나가 트랜잭션의 모든 읽기에 적용되며, 그 범위는 데이터베이스 전체다**

SERIALIZABLE 트랜잭션은 시작 시점에 read SCN을 한 값으로 정한다. 이 값은 특정 테이블이나 특정 행에만 적용되는 게 아니라, 그 트랜잭션이 수행하는 모든 읽기에 똑같이 적용된다.

```
T (SERIALIZABLE), 시작 시점에 read SCN = 100 으로 정해짐 (트랜잭션 끝까지 이 값)

   T가 트랜잭션 동안 무엇을 읽든, 전부 read SCN=100 기준으로 읽는다:
      SELECT ... FROM VISIT_SLOT          → SCN 100 시점의 값
      SELECT ... FROM VISIT_RESERVATION   → SCN 100 시점의 값
      SELECT ... FROM MEMBERTBL           → SCN 100 시점의 값
      SELECT ... FROM (그 외 어떤 테이블)  → SCN 100 시점의 값

   → read SCN 하나가 트랜잭션의 모든 읽기에 적용된다.
   → 따라서 일관성 보장의 범위 = 특정 한 테이블이 아니라 데이터베이스 전체다.
     세 테이블을 읽어도 전부 같은 SCN 100 시점을 반영하므로, 서로 어긋나지 않는다.
```

여기서 분명히 할 점은, "그 행이 들어 있는 테이블 하나만의 스냅샷"이 아니라는 것이다. read SCN은 데이터베이스 전체의 버전 번호이고, 그 번호 하나가 트랜잭션이 읽는 모든 테이블·모든 행에 동일하게 적용된다.

**2. 방법 — 미리 복사하지 않고, 읽는 블록마다 언두로 그 SCN 시점 값을 재구성한다**

read SCN은 숫자 하나일 뿐이다. 트랜잭션 시작 시점에 어떤 테이블을 어딘가에 복사해 두는 동작은 없다. 대신 다음과 같이 읽는 시점에 블록 단위로 처리한다.

```
read SCN = 100 인 트랜잭션이 어떤 데이터 블록 B를 읽을 때:

   블록 B에는 그 블록을 변경한 트랜잭션들의 변경 SCN 정보가 들어 있다.
   이 변경 SCN과 내 read SCN(100)을 비교한다.

   경우 1) 블록 B의 변경 SCN ≤ 100  (내 시작 시점까지 커밋되어 있던 상태)
            → 그 블록을 그대로 읽는다. 재구성이 필요 없다.

   경우 2) 블록 B의 변경 SCN > 100  (내 시작 이후에 다른 트랜잭션이 바꿔 커밋했거나, 미커밋 변경이 있음)
            → 언두에서 그 변경의 옛 값을 가져와,
              블록 B를 'SCN 100 시점의 상태'로 되돌린 임시 사본을 만든다.
              (이 임시 사본을 consistent read 사본이라고 한다.)
            → 그 임시 사본의 값을 읽는다. 디스크/버퍼의 실제 블록 B는 건드리지 않는다.

    트랜잭션 시작 시점에 테이블을 미리 복사하는 동작은 없다.
    위 처리는 '실제로 읽은 블록'에 대해서만, 그 블록을 읽는 그 순간에 일어난다.
     읽지 않은 데이터에 대해서는 아무 처리도 하지 않는다.
```

즉 방법을 한 줄로 하면: read SCN이라는 번호 하나를 고정해 두고, 데이터를 읽을 때마다 그 블록의 변경 SCN과 비교해서, 내 번호보다 나중에 바뀐 블록이면 언두로 내 번호 시점 값을 그 자리에서 재구성해 읽는다.

**3. 증거 — ORA-01555 (snapshot too old)**

위 경우 2가 성립하려면, 'SCN 100 시점으로 되돌릴 옛 값'이 언두에 남아 있어야 한다.

```
경우 2에서 필요한 옛 값을, 그 사이 다른 트랜잭션들이 언두 공간을 많이 써서
이미 덮어써 버렸다면
→ SCN 100 시점으로 되돌릴 재료가 없다
→ ORA-01555 (snapshot too old) 발생.

이 에러는 "스냅샷이 트랜잭션 시작 시점에 떠 둔 복사본"이라면 발생할 이유가 없다.
필요한 옛 값을 그때그때 언두에서 재구성한다는 사실의 직접적인 증거다.
```

**정리**

```
범위(무엇을 어디까지 일관되게 보여주나):
   read SCN 하나가 트랜잭션의 모든 읽기에 적용된다.
   보장 범위는 특정 테이블이 아니라 데이터베이스 전체다.

방법(어떻게 구현하나):
   시작 시점에 read SCN을 한 값으로 고정한다.
   읽는 블록마다 그 블록의 변경 SCN과 내 read SCN을 비교한다.
   내 read SCN보다 나중에 바뀐 블록이면, 언두로 내 read SCN 시점 값을 임시 사본으로 재구성해 읽는다.
   테이블을 미리 복사하지 않는다. 읽은 블록에 대해서만, 읽는 순간에 처리한다.

(이상은 SERIALIZABLE의 읽기 측 제어다. 쓰기 측 제어는 ②바꾸기에서
 '수정하려는 행의 현재 커밋 SCN > 내 read SCN'이면 ORA-08177로 거부하는 것이고, 위 (다)에서 다뤘다.)
```

# ⑥ READ COMMITTED와 SERIALIZABLE을 같은 경합에 나란히 놓고 비교

지금까지 READ COMMITTED와 SERIALIZABLE을 따로 봤다. 이제 같은 경합 하나를 두 수준에서 동시에 진행시켜, 어느 단계까지 똑같이 가다가 어느 단계에서 갈라지는지를 단계 대 단계로 맞춰 본다.

## 공통 시나리오

```
출발: slot410 = 'AVAILABLE' (이 값의 커밋 SCN ≤ 100)
두 트랜잭션 T1(선착), T2(후착)가 같은 문장을 실행한다:
   UPDATE slot410 SET status='RESERVED' WHERE slot_id=410 AND status='AVAILABLE'
두 트랜잭션 모두 T1이 커밋하기 전에 시작했다.
```

(UPDATE 한 문장은 내부적으로 두 단계다. ①찾기는 WHERE에 맞는 행을 고르는 단계로, 내 read SCN 시점 값을 읽고 락을 걸지 않는다. ②바꾸기는 고른 행을 실제로 수정하는 단계로, 그 행의 현재 커밋 최신 버전을 읽고 X-락을 걸고 값을 바꾼다.)

## s1부터 s5까지 — 두 수준이 완전히 동일하다

```
━━ READ COMMITTED와 SERIALIZABLE이 동일하게 진행되는 구간 ━━━━━━━━━━━━━━━━━━

s1  T1: UPDATE 실행
        ├ ①찾기: slot410 = AVAILABLE → 대상으로 선택
        │    · READ COMMITTED: 이 문장이 시작될 때 read SCN=100을 잡음
        │    · SERIALIZABLE: 트랜잭션 시작에 고정된 read SCN=100을 사용
        │    → 두 경우 모두 이 시점 slot410은 AVAILABLE이므로 ①찾기 결과가 같다
        └ ②바꾸기: slot410에 X-락 획득, RESERVED로 기록 (아직 COMMIT 안 함, 락 보유)
             · X-락은 격리 수준과 무관하게 모든 쓰기가 잡는다 → 두 경우 동일

s2  T2: 같은 UPDATE 실행
        ├ ①찾기: slot410 = 아직 AVAILABLE → 대상으로 선택 → 두 경우 동일
        └ ②바꾸기: 같은 slot410에 X-락 시도 → T1이 보유 중 → 대기
             · T2를 대기시키는 것은 X-락이고, 이는 격리 수준과 무관하다 → 두 경우 동일

s5  T1: COMMIT → 커밋 SCN=105, slot410의 X-락 해제 → 두 경우 동일
```

여기까지 두 수준의 동작이 글자 그대로 같은 이유는 세 가지다. 첫째, ①찾기가 보는 값(AVAILABLE)이 같다. 둘째, T2를 대기시키는 것이 X-락인데 X-락은 격리 수준과 무관하게 모든 쓰기가 잡는다. 셋째, 커밋은 커밋이다. 즉 read SCN을 어떻게 잡느냐의 차이가 아직 결과에 드러나지 않는 구간이다.

## s6 — 여기서 처음으로 갈라진다 (T2가 깨어나 ②바꾸기를 재개할 때)

T2가 X-락 대기에서 깨어나 ②바꾸기를 다시 진행하려는 순간, 두 수준의 read SCN 정책 차이가 처음으로 결과를 가른다.

```
━━ READ COMMITTED의 s6 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
s6  T2: ②바꾸기 재개.
        ├ READ COMMITTED는 read SCN을 문장마다 새로 잡을 수 있다 → 문장을 새 read SCN(106)으로 재시작
        ├ 재실행한 ①찾기: 현재 slot410 = RESERVED → WHERE의 'status=AVAILABLE'이 거짓
        └ 대상 행 없음 → 0 rows updated (T2는 아무 행도 안 바꿈, 락도 안 잡음)
    결과: T2는 영향 행 0 → "이미 예약됨"으로 처리

━━ SERIALIZABLE의 s6 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
s6  T2: ②바꾸기 재개.
        ├ SERIALIZABLE은 read SCN을 100으로 고정해 두고 바꿀 수 없다 → 재시작 불가
        ├ 수정하려는 slot410의 현재 커밋 버전 SCN(105) > 내 read SCN(100) 확인
        │   (= 내가 트랜잭션을 시작한 뒤에 그 행이 바뀌어 커밋되었다는 뜻)
        └ ORA-08177 (can't serialize access for this transaction) 발생
    결과: T2는 예외 → 트랜잭션 롤백
```

갈라지는 지점이 s6 하나뿐이고, 그 원인도 하나다. **READ COMMITTED와 SERIALIZABLE의 유일한 차이는 read SCN을 잡는 정책이다** — RC는 문장마다 새로 잡을 수 있어 재시작이 가능하고, SERIALIZABLE은 트랜잭션 시작에 고정해 재시작이 불가능하다. 이 한 가지 정책 차이가 s6에서 "0행"과 "ORA-08177"로 갈린다.

## 결과 비교

```
정합성 결과 (두 수준 동일):
   slot410은 RESERVED로 끝나고, 그 슬롯을 점유한 트랜잭션은 T1 하나뿐이다.
   확정 예약(CONFIRMED)도 한 건만 생긴다. → 슬롯당 1건 규칙은 두 수준 모두 지켜진다.

다른 점 (후착 T2가 받는 신호):
   READ COMMITTED : 0 rows updated  (오류 없이, 영향 행 0으로 끝남)
   SERIALIZABLE   : ORA-08177       (예외 발생, 트랜잭션 롤백)

애플리케이션 처리 방식의 차이:
   READ COMMITTED : 영향 행 수가 0인지 확인해서 0이면 "이미 예약됨"을 던진다
   SERIALIZABLE   : ORA-08177 예외를 잡아서 "이미 예약됨"으로 처리하거나 재시도한다
```

즉 두 수준은 **정합성(슬롯당 1건)을 만드는 결과가 같고, 그 정합성을 만든 실제 장치도 같다.** 그 장치는 격리 수준이 아니라 ②바꾸기 단계의 X-락과 현재 커밋 버전 비교다. 격리 수준이 바꾸는 것은 후착이 거부되는 형태(0행이냐 예외냐)뿐이다.

# ⑦ REPEATABLE READ와 오라클이 그것을 두지 않는 이유

표준 SQL에는 READ COMMITTED와 SERIALIZABLE 사이에 REPEATABLE READ라는 수준이 하나 더 있다. 그런데 오라클은 이 수준을 제공하지 않는다. 그 이유를 알려면, REPEATABLE READ가 어떤 장치로 동작하는지부터 봐야 한다. 그리고 그 장치는 오라클이 쓰지 않는 종류의 것이다.

## (가) 표준의 REPEATABLE READ는 '읽기 락'으로 동작한다 — 먼저 S-락을 도입한다

앞에서 오라클은 읽기에 락을 걸지 않고 read SCN 스냅샷으로 읽는다고 했다. 그런데 전통적인 다른 종류의 데이터베이스(잠금 기반 엔진)는 읽기에도 락을 건다. 그 읽기용 락을 S-락이라 한다.

```
S-락 (Shared lock, 공유 락 — 읽기용): 한 행에 여러 트랜잭션이 동시에 가질 수 있다.
X-락 (eXclusive lock, 배타 락 — 쓰기용): 한 행에 한 트랜잭션만 가질 수 있다.

한 행에 '이미 걸린 락'과 '새로 걸려는 락'의 충돌 여부:
   이미 S-락, 새로 S-락 → 허용 (둘이 공존)
   이미 S-락, 새로 X-락 → 대기 (충돌)
   이미 X-락, 새로 S-락 → 대기 (충돌)
   이미 X-락, 새로 X-락 → 대기 (충돌)
```

정리하면, 여러 트랜잭션이 같은 행을 동시에 읽는 것(S+S)은 허용되지만, 한쪽이 읽는 중(S)인 행을 다른 쪽이 바꾸려(X) 하면 대기한다.

이제 READ COMMITTED와 REPEATABLE READ의 차이는 단 하나, **읽기로 잡은 S-락을 언제 푸느냐**이다.

```
[X-락(쓰기)] READ COMMITTED, REPEATABLE READ 모두 COMMIT까지 보유한다 (동일)

[S-락(읽기)] READ COMMITTED  : 행을 읽고 나서 즉시 해제
             REPEATABLE READ : 행을 읽고 나서 COMMIT까지 보유   ← 이 차이 하나가 전부
```

## (나) 락 기반 REPEATABLE READ가 재읽기를 고정하는 방식

읽은 행의 S-락을 COMMIT까지 들고 있으면, 그 행을 다른 트랜잭션이 바꾸려 해도(X-락 필요) S-락과 충돌해 대기한다. 그래서 내가 다시 읽어도 값이 안 변한다.

```
출발: slot410 = 'AVAILABLE'

시각  T1 (REPEATABLE READ, 잠금 기반 엔진)       T2
─────────────────────────────────────────────────────────────────────
t1   SELECT status FROM slot410
       ├ slot410에 S-락 획득
       └ 이 S-락을 COMMIT까지 보유 → 'AVAILABLE'
t2                                            UPDATE slot410 SET status='RESERVED'
                                                └ X-락 필요 → T1의 S-락과 충돌 → 대기
t3   SELECT status FROM slot410
       └ S-락을 계속 보유 중 → 'AVAILABLE' (값이 안 변함)  (T2는 계속 대기)
t4   COMMIT → S-락 해제
                                              └ (대기 풀림) X-락 획득 → UPDATE 진행
─────────────────────────────────────────────────────────────────────
T1이 읽은 행은 T1이 끝날 때까지 다른 트랜잭션이 못 바꾼다 → 재읽기 값이 같다 = repeatable read.
```

## (다) 그런데 phantom은 못 막는다

위 S-락은 "이미 존재해서 내가 실제로 읽은 행"에만 걸린다. 아직 존재하지 않는 행, 즉 앞으로 INSERT될 행에는 걸 수 없다. 존재하지 않는 행에는 잠글 대상이 없기 때문이다.

```
초기 (window_id=42): slot410, slot411, slot412 (셋 다 status='AVAILABLE')

시각  T1 (REPEATABLE READ)                     T2
─────────────────────────────────────────────────────────────────────
r1   SELECT COUNT(*) FROM slot
       WHERE window_id=42 AND status='AVAILABLE'
       ├ 조건에 맞는 '이미 존재하는 행'에 S-락: slot410, slot411, slot412
       └ 결과 = 3
        ※ slot410/411/412 사이, 아직 행이 없는 위치에는 잠글 대상 자체가 없다
r2                                            INSERT INTO slot
                                                (slot413, window_id=42, status='AVAILABLE')
                                                ├ slot413은 새 행 → 기존 S-락(410/411/412)과
                                                │  겹치는 자원이 없음
                                                └ 충돌 없음 → INSERT 성공 → COMMIT
r3   SELECT COUNT(*) FROM slot (같은 조건)
       └ 결과 = 4   ← r1에 없던 slot413이 결과에 들어옴 = phantom
─────────────────────────────────────────────────────────────────────
원인: S-락은 r1 시점에 '존재하던 행'에만 걸린다. slot413은 r1 시점에 없었으므로
      잠글 수 없었고, T2가 자유롭게 INSERT했다. REPEATABLE READ는 이 phantom을 허용한다.
```

## (라) phantom까지 막으려면 = 범위 락 = 락 기반 SERIALIZABLE

phantom을 막으려면, 개별 행이 아니라 "조건에 맞는 행이 들어올 수 있는 범위" 자체를 잠가야 한다. 이것을 범위 락(키 범위 락)이라 한다.

```
r1   SELECT ... WHERE window_id=42 AND status='AVAILABLE'
       ├ 기존 행 S-락: slot410, slot411, slot412
       └ 추가로 '조건에 맞는 키 범위' 자체에 범위 락을 건다
          → 그 범위 안에 새 행을 INSERT하는 것을 막는다
r2   INSERT (slot413, window_id=42, status='AVAILABLE')
       └ slot413이 들어갈 위치가 범위 락 안 → 대기 → phantom 차단
```

즉 잠금 기반 엔진에서 **REPEATABLE READ와 SERIALIZABLE의 차이는, 읽은 개별 행만 잠그느냐(REPEATABLE READ), 조건에 맞는 범위까지 잠그느냐(SERIALIZABLE)**이다.

## (마) 오라클이 REPEATABLE READ를 두지 않는 이유

위 (가)~(라)는 전부 **읽기 락(S-락)이 있어야 성립한다.** REPEATABLE READ의 동작 자체가 "읽은 행의 S-락을 COMMIT까지 들고 있는 것"이고, phantom이 허용되는 것도 "개별 행만 잠그고 범위는 안 잠가서"이다. 그런데 오라클은 읽기에 락을 걸지 않는다. 읽기는 read SCN 스냅샷으로 처리한다. 그래서 다음 두 가지가 성립한다.

```
(1) 오라클에는 읽기 락이 없으므로, 락 기반 REPEATABLE READ의 동작을 만들 수단 자체가 없다.

(2) 오라클의 read SCN 스냅샷에는 자연스러운 두 지점만 존재한다:
      · read SCN을 '문장마다' 새로 잡음   → 재읽기가 흔들리고 새 행도 보임
                                          = non-repeatable read 허용, phantom 허용
                                          = READ COMMITTED
      · read SCN을 '트랜잭션 시작에' 고정  → 그 SCN 이후 커밋된 변경은 전부 안 보임
                                          = non-repeatable read 차단, phantom 차단
                                          = SERIALIZABLE
```

여기서 핵심은 두 번째 줄이다. read SCN을 트랜잭션 시작에 고정하면, "내 read SCN보다 나중에 커밋된 변경"은 전부 읽기에서 제외된다. 그런데 이 "나중에 커밋된 변경"에는 **기존 행을 수정한 것**과 **새 행을 INSERT한 것**이 똑같이 들어간다. 스냅샷에는 "기존 행 수정은 제외하되 새 행 삽입은 포함하라"처럼 둘을 나눌 방법이 없다. 둘 다 "내 read SCN 이후 커밋"이라는 같은 조건이고, 같이 제외된다.

따라서 read SCN을 트랜잭션 단위로 고정하면 non-repeatable read와 phantom이 **동시에** 차단된다. REPEATABLE READ가 요구하는 "non-repeatable은 차단하되 phantom은 허용"이라는 중간 상태를 스냅샷으로는 만들 수 없다. 그 상태는 (다)에서 본 것처럼 "개별 행은 S-락으로 고정하고 범위는 안 잠그는" 잠금 방식에서만 나온다. 오라클에는 그 잠금이 없으므로, REPEATABLE READ에 해당하는 동작이 존재하지 않는다. 그래서 오라클은 트랜잭션 단위 스냅샷 수준을 SERIALIZABLE이라고 부르고, REPEATABLE READ는 두지 않는다.

```
실제 확인: 오라클에서 SET TRANSACTION ISOLATION LEVEL REPEATABLE READ; 를 실행하면
           구문 자체가 거부된다. 오라클이 받는 격리 수준은 READ COMMITTED와 SERIALIZABLE 둘뿐이다.
```

## (바) 다른 데이터베이스가 "REPEATABLE READ"라는 이름을 쓰는 경우

MySQL의 InnoDB나 PostgreSQL도 오라클처럼 MVCC 스냅샷을 쓴다. 그런데 이들은 "REPEATABLE READ"라는 이름의 수준을 제공한다. 다만 그 내부 구현은 (가)~(라)의 잠금 방식이 아니라 트랜잭션 단위 스냅샷이고, 그래서 표준이 요구하는 것보다 강하다(스냅샷 읽기에 대해 phantom까지 대체로 막힌다). 즉 이름만 표준의 "REPEATABLE READ"를 쓰고, 실제 동작은 트랜잭션 스냅샷이다.

오라클은 같은 트랜잭션 스냅샷 수준에 "SERIALIZABLE"이라는 이름을 붙였고, 스냅샷 기반 수준을 두 이름으로 중복해서 내놓지 않는다. 표준은 "요청한 수준보다 더 강하게 막아도 표준 준수"로 규정하므로, 어느 쪽 명명이든 표준을 위반하지 않는다.

```
정리:
  · 표준 REPEATABLE READ = 읽은 개별 행에 S-락을 COMMIT까지 보유, 범위는 안 잠금
    → non-repeatable read 차단, phantom 허용. 읽기 락이 있어야 성립하는 수준
  · 오라클은 읽기 락이 없고 read SCN 스냅샷만 쓴다.
    스냅샷은 트랜잭션 단위로 고정하면 non-repeatable read와 phantom을 동시에 막는다.
    → "non-repeatable 차단, phantom 허용"이라는 중간 지점이 스냅샷 구조에 존재하지 않는다.
    → 그래서 오라클은 그 수준을 SERIALIZABLE로 명명하고 REPEATABLE READ를 두지 않는다.
```

# ⑧ write-skew — SERIALIZABLE 스냅샷만으로는 못 막는 정합성 문제

지금까지 본 경합(같은 슬롯을 두 트랜잭션이 쓰는 경우)은 SERIALIZABLE이 ORA-08177로 막았다. 그런데 SERIALIZABLE의 스냅샷만으로는 막지 못하는 정합성 문제가 한 종류 있다. 이것이 write-skew다. 막지 못하는 이유는 구조적이다. SERIALIZABLE의 충돌 검출이 **트랜잭션이 실제로 '쓴 행'에 대해서만** 이뤄지고, **'읽기만 한 행'은 검출 대상이 아니기** 때문이다.

## (가) SERIALIZABLE의 충돌 검출은 '쓴 행'에 대해서만 한다

이 단락이 따로 읽혀도 되도록 SERIALIZABLE의 충돌 검출 규칙을 다시 명시한다.

```
SERIALIZABLE 트랜잭션이 어떤 행을 수정하려 할 때(②바꾸기):
   그 행의 '현재 커밋 버전 SCN'과 '내 read SCN'을 비교한다.
   현재 커밋 버전 SCN > 내 read SCN 이면 (= 내가 시작한 뒤 그 행이 바뀌어 커밋되었으면)
   → ORA-08177로 거부한다.

   이 비교는 내가 수정하는 행에 대해서만 한다.
   내가 SELECT로 '읽기만' 한 행에 대해서는 이 비교를 하지 않는다.
```

즉 충돌 검출의 대상은 "쓴 행"이지 "읽은 행"이 아니다. 이 한 가지가 write-skew의 원인이다.

## (나) 같은 것을 읽고 서로 다른 행을 쓰면 충돌이 안 잡힌다 — 구체 예

두 트랜잭션이 같은 데이터를 읽고 각자 판단한 뒤, 서로 다른 행을 쓰는 경우다. 매물의 활성 윈도우를 예로 든다.

```
불변식: 한 매물 P는 활성 윈도우(status='ACTIVE')를 최소 1개 유지해야 한다.
출발: 매물 P에 활성 윈도우 두 개 — W1(window_id=42, ACTIVE), W2(window_id=43, ACTIVE)
      이 상태의 커밋 SCN ≤ 100.

시각  T_A (SERIALIZABLE, W1을 철회하려 함)         T_B (SERIALIZABLE, W2를 철회하려 함)
──────────────────────────────────────────────────────────────────────────────
a0   read SCN=100 으로 고정
b0                                              read SCN=100 으로 고정
a1   SELECT COUNT(*) FROM VISIT_WINDOW
       WHERE property_id='P' AND status='ACTIVE'
       → read SCN=100 기준 → W1, W2 둘 다 ACTIVE → 결과 2
     판단: "활성이 2개니까 W1 하나 철회해도 1개 남는다" → 진행
b1                                              SELECT COUNT(*) ... (같은 조건)
                                                  → read SCN=100 기준 → 역시 결과 2
                                                판단: "2개니까 W2 하나 철회해도 1개 남는다" → 진행
a2   UPDATE VISIT_WINDOW SET status='WITHDRAWN'
       WHERE window_id=42
       ├ 수정 대상 = W1 행
       ├ W1의 현재 커밋 버전 SCN이 내 read SCN(100)보다 나중인가? 아니다(W1은 나만 건드림)
       └ → 충돌 없음 → W1에 X-락, WITHDRAWN으로 수정 (미커밋)
b2                                              UPDATE VISIT_WINDOW SET status='WITHDRAWN'
                                                  WHERE window_id=43
                                                  ├ 수정 대상 = W2 행
                                                  ├ W2의 현재 커밋 버전 SCN이 read SCN(100)보다
                                                  │  나중인가? 아니다(W2는 나만 건드림)
                                                  └ → 충돌 없음 → W2에 X-락, 수정 (미커밋)
                                                ※ W1과 W2는 다른 행이므로 둘 사이에 X-락 충돌이 없다
a3   COMMIT → 성공 (커밋 SCN=105)
b3                                              COMMIT → 성공 (커밋 SCN=106)
──────────────────────────────────────────────────────────────────────────────
결과: W1=WITHDRAWN, W2=WITHDRAWN → 매물 P의 활성 윈도우 0개 → 불변식 위반.
      두 트랜잭션 다 SERIALIZABLE인데도 ORA-08177이 발생하지 않고 둘 다 커밋되었다.
```

왜 충돌이 안 잡혔는지를 짚는다. T_A가 쓴 행은 W1, T_B가 쓴 행은 W2다. T_A는 W2를 쓰지 않았고 T_B는 W1을 쓰지 않았다. 그래서 a2에서 W1의 버전을 확인해도, b2에서 W2의 버전을 확인해도, 각자 쓰는 행은 상대가 건드리지 않았으므로 "현재 커밋 버전 SCN > 내 read SCN" 조건이 성립하지 않는다. 두 트랜잭션이 **겹치게 한 것은 읽기(둘 다 W1·W2를 세어 2를 얻음)뿐인데, 읽은 행은 충돌 검출 대상이 아니다.** 그래서 둘 다 통과한다.

## (다) write-skew라 부르는 이유, 그리고 같은 행 경합과 무엇이 다른가

```
write-skew의 성립 조건:
   1) 두 트랜잭션이 '같은 데이터'를 읽는다 (여기서는 둘 다 W1·W2의 활성 개수를 읽음)
   2) 각자 그 읽은 값을 근거로 판단한다 (둘 다 "2개니까 하나 지워도 된다")
   3) 그런데 각자 '서로 다른 행'을 쓴다 (A는 W1, B는 W2)
   → 쓴 행이 겹치지 않으므로 쓰기-쓰기 충돌이 없다 → SERIALIZABLE이 못 잡는다
   → 합쳐 보면, 각자는 지킨 불변식이 전체적으로는 깨진다
```

같은 행 경합(앞에서 본 슬롯 경합)과의 차이가 여기서 분명해진다. 같은 행 경합은 두 트랜잭션이 **같은 행을 쓰므로** 쓰기-쓰기 충돌이 반드시 생기고, 그건 SERIALIZABLE이 정확히 검출한다. write-skew는 두 트랜잭션이 **서로 다른 행을 쓰므로** 쓰기-쓰기 충돌이 없고, 읽기 겹침만으로는 SERIALIZABLE이 검출하지 않는다.

## (라) 슬롯 예약 경합(시나리오 1)에는 왜 해당하지 않나

```
시나리오 1 = 같은 슬롯 slot410을 N명이 동시에 예약 시도.

   N개 트랜잭션이 전부 같은 slot410 행을 쓴다
   (슬롯 상태를 RESERVED로 바꾸는 UPDATE, 또는 그 슬롯에 대한 예약 INSERT — 둘 다 같은 슬롯 대상).

   → 선착이 slot410을 바꿔 커밋하면, 후착이 slot410을 쓰려는 순간
      slot410의 현재 커밋 버전 SCN > 후착의 read SCN 이 되어 충돌로 잡힌다.
      (SERIALIZABLE이면 ORA-08177)

   → 즉 쓰는 행이 같으므로 쓰기-쓰기 충돌이고, 이건 SERIALIZABLE이 검출하는 바로 그 경우다.
```

write-skew는 "서로 다른 행을 쓸 때"만 생기는데, 시나리오 1은 모두 같은 슬롯 행을 쓰므로 write-skew가 성립할 조건 자체가 안 된다. 따라서 슬롯당 1건 규칙에 대해서는 SERIALIZABLE이 안전하다.

## (마) 오라클에서 write-skew를 막아야 한다면

오라클 SERIALIZABLE은 스냅샷 기반이고, 위처럼 읽기 겹침으로 생기는 write-skew를 자동으로 막지 않는다. (PostgreSQL의 SERIALIZABLE은 읽은 행의 의존성까지 추적해 write-skew를 막는 별도 방식을 쓰지만, 오라클에는 그 방식이 없다.) 오라클에서 write-skew를 막아야 한다면, **읽기만 하던 행을 `SELECT ... FOR UPDATE`로 X-락을 걸어** 읽어야 한다. 그러면 그 행이 쓰기 대상처럼 잠겨, 읽기 겹침이 쓰기-쓰기 충돌로 바뀌고 잠금 또는 SCN 비교로 검출된다.

```
위 예에 적용하면:
   a1, b1의 SELECT COUNT을 단순 SELECT가 아니라, 관련 윈도우 행들에 FOR UPDATE로 X-락을 거는
   형태로 바꾼다. 그러면 T_A와 T_B가 같은 윈도우 행들에 X-락을 잡으려 다투게 되어,
   한쪽이 대기하고 → 깨어나서 현재 활성 개수를 다시 보고(1개) 판단을 바꾸게 된다.
```

다만 시나리오 1에는 이 처리가 필요 없다. 시나리오 1은 이미 같은 슬롯 행을 쓰는 쓰기-쓰기 경합이라 추가로 읽기 행을 잠글 이유가 없다.

## (바) 핵심

```
write-skew:
  · SERIALIZABLE의 충돌 검출은 '쓴 행'에만 적용되고 '읽은 행'에는 적용되지 않는다.
  · 그래서 두 트랜잭션이 같은 데이터를 읽고, 각자 다른 행을 쓰면, 쓰기-쓰기 충돌이 없어
    둘 다 커밋되고, 합쳐 보면 불변식이 깨질 수 있다. 이것이 write-skew다.
  · 성립 조건은 '같은 것을 읽고 + 서로 다른 행을 쓸 때'다.
  · 슬롯 예약 경합(시나리오 1)은 모두 '같은 슬롯 행'을 쓰므로 쓰기-쓰기 충돌이고,
    write-skew 조건에 해당하지 않는다 → SERIALIZABLE이 안전하게 막는다.
  · 오라클에서 write-skew를 막아야 하면 읽기 행에 SELECT ... FOR UPDATE로 X-락을 건다.
    (시나리오 1에는 불필요)
```
