# 시나리오 1 동시성 제어 — 설계 결론과 락 메커니즘 (종합 정리)

> **대상**: `createReservation` — 같은 슬롯에 N명이 동시에 예약을 시도하는 경합 (F004 시나리오 1)
> **전제 DB**: Oracle (MVCC, 기본 격리 = READ COMMITTED)
> **코드 기준**: [VisitReservationWriteService.createReservation](wherehouse/src/main/java/com/wherehouse/VisitReservation/service/VisitReservationWriteService.java:313)
> **이 문서의 위치**: `DB격리수준_동시슬롯예약_예약시스템.md` 의 ⑨(설계 결론) 에 해당하는 본문 + 1차 방어선 세 후보(조건부 UPDATE·비관적 락·SERIALIZABLE)의 메커니즘을 원래 깊이로 펼친 정리. 메커니즘 근거 표기(`문서 ①` 등)는 그 문서의 절 번호를 가리킨다.

## 0. 측정으로 고정된 출발 사실

| 회차 | 환경 (격리 RC 고정) | 슬롯·N | 결과 | 정합성 |
|---|---|---|---|---|
| **Step 1** | 부분 유일 인덱스 **적용**, 동시성 제어 없음 | slot 348, N=5 | 1건 commit + 4건 **ORA-00001** 거부, CONFIRMED 1건 | 충족 |
| **Step 2** | 부분 유일 인덱스 **제거** (baseline) | slot 392, N=5 | **5건 모두 commit**, CONFIRMED 5건 | **붕괴 (직접 관측)** |

- Step 2 는 "별도 메커니즘이 없으면 한 슬롯에 두 개 이상의 CONFIRMED 가 남는다"는 시나리오 명제를 실측으로 못박은 baseline 이다.
- Step 1 의 거부는 코드가 막은 것이 아니라 **VISIT_RESERVATION INSERT 의 부분 유일 인덱스**가 막은 것이다(ORA-00001 = 유일 제약 위반). 즉 Step 1·2 의 차이는 "코드 동시성 제어"가 아니라 "백스톱 인덱스의 유무"였다.

## 1. 왜 붕괴하는가 — 읽기는 못 막고, 직렬화는 쓰기에서만 일어난다

### 1.1 `createReservation` 을 읽기/쓰기로 분해

```
L319  slotRepository.findById(slotId)         SELECT VISIT_SLOT          ← 읽기. 스냅샷, 락 없음
L328  if (slot.getStatus() != AVAILABLE)       메모리 엔티티 값으로 판정    ← DB 재조회 아님. 거부를 확정 못 함
L342  if (!startTime.isAfter(now))             시작 시각 검증
L350  if (userId == owner)                     자기 매물 검증
L365  countDuplicatePropertyReservation        COUNT (동일 매물 중복)      ← 읽기
L371  countTimeOverlappingReservation          COUNT (시간 겹침)          ← 읽기
L378  slot.setStatus(RESERVED)
L379  slotRepository.save(slot)                UPDATE VISIT_SLOT          ← ②바꾸기. 여기서 처음 X-락
L394  reservationRepository.save(reservation)  INSERT VISIT_RESERVATION    ← 백스톱(부분 유일 인덱스)이 작동하는 곳
```

이 흐름을 사용자 문서의 두 용어로 부른다:

```
①찾기 (consistent read)  = 스냅샷 시점의 값을 락 없이 읽는다           → L319, L365, L371
②바꾸기 (current read + X-락) = 현재 커밋된 최신 값에 X-락 걸고 덮어쓴다  → L379(UPDATE), L394(INSERT)
```

### 1.2 읽기는 동시 N개를 못 막는다

Oracle 의 규칙(문서 ①): **읽기는 쓰기를 막지 않고, 락도 잡지 않는다.** L319 의 `findById` 는 일반 SELECT 이므로 락이 없다.

```
T1: SELECT(L319) → AVAILABLE 봄 (락 없음)
T2: SELECT(L319) → AVAILABLE 봄 (락 없음)      ← 둘이 동시에, 서로 못 막고, 같은 AVAILABLE 을 본다
...
Tn: SELECT(L319) → AVAILABLE 봄 (락 없음)

→ N개 전부 L328 if (status != AVAILABLE) 를 통과한다.
  L328 의 판정은 L319 가 읽어 온 ★메모리 값★ 으로 하므로, 이 판정 단계는 정합성 거부를 만들지 못한다.
```

L319 의 읽기와 L379 의 쓰기 사이의 간극이 **TOCTOU(Time-Of-Check to Time-Of-Use) 틈**이고, 동시 N개가 모두 이 틈을 통과한다. 정합성 거부는 그 뒤 **쓰기(L379 또는 L394)** 에서만 발생할 수 있다.

### 1.3 직렬화는 ②바꾸기에서만 생긴다

시나리오 1 은 N개가 **모두 같은 slot 행**을 UPDATE 한다(같은 행 경합). 그래서 L379 의 X-락 직렬화는 항상 일어난다. 그런데 현재 슬롯 UPDATE 는 dirty checking 이 만든 `WHERE SLOT_ID=?` 뿐이라(상태 조건 없음), 후착이 깨어나도 PK 는 그대로라서 걸러지지 않고 RESERVED 를 RESERVED 로 덮어쓴다. 여기서 백스톱 인덱스마저 빼면(Step 2) 무방비가 된다.

→ **정합성을 만드는 위치는 처음부터 읽기가 아니라 쓰기였다.**

## 2. 정합성을 만들 수 있는 두 위치

```
백스톱      = VISIT_RESERVATION INSERT(L394) 의 부분 유일 인덱스 → 위반 시 ORA-00001 (사후 차단)
              · 동시성 제어와 ★직교★ — 코드 버그·다중 인스턴스에도 항상 작동하는 최종 보호망
              · Step 1 에서 작동 확인됨

1차 방어선   = VISIT_SLOT 쓰기 단계(L379)에서 후착을 차단 (미리·싸게 차단)
              · 후보 셋: 조건부 UPDATE / 비관적 락(FOR UPDATE) / SERIALIZABLE
              · 설계 명세서가 "구현 단계로 명시 위임" 한 부분 (WriteService 클래스 주석)
              · Step 2 에서 의도적으로 미선택 상태
```

두 위치는 **양자택일이 아니라 직교**한다. Step 2 에서 unique 를 뺀 것은 1차 방어선의 효과만 분리 측정하려는 의도이며, 운영 반영 시 백스톱을 함께 둘지는 1차 방어선 선택과 **별개 결정**이다.

> 정확한 진단: "쿼리 설계가 미흡했다"가 **아니다.** 백스톱은 원래 갖춰져 있었고(Step 1 작동), 1차 방어선은 측정 후 선택으로 위임된 상태였다.

## 3. 격리 수준의 정확한 위치

### 3.1 두 축의 분리

```
격리 수준  = read SCN 을 언제 잡느냐 (RC=문장마다 / SERIALIZABLE=트랜잭션 시작)
           → ①찾기(consistent read)가 "무엇을 보는가" 를 지배 = ★읽기를 지배★

정합성    = 같은 행을 다투는 ②바꾸기(current read + X-락)에서만 생긴다
           → "직렬화는 오직 쓰기에서 일어난다"
```

→ 격리 수준이 정합성에 작용하려면, 격리 수준이 바꾸는 지점이 곧 거부를 만드는 지점과 일치해야 한다. 시나리오 1 에서 격리 수준이 정합성에 닿는 **유일한 통로**는 ②바꾸기에서 그 read SCN 을 기준으로 한 **충돌 감지·반응** 뿐이다.

### 3.2 RC 와 SERIALIZABLE — 비교까지는 같고, read SCN 에서 갈린다

```
출발: slot410 = AVAILABLE (커밋 SCN 100). T1·T2 둘 다 이 시점 이후 시작, 같은 UPDATE.

         T1(선착)                      T2(후착)
──────────────────────────────────────────────────────────────
 ①찾기   AVAILABLE 봄 (락 없음)         AVAILABLE 봄 (락 없음)   ← 둘 다 통과
 ②바꾸기 X-락 획득, RESERVED 기록        같은 행 X-락 시도 → 대기
 commit  커밋 SCN=105, 락 해제                  │
                                       대기 풀림 → ②바꾸기 재개
                                       비교: "현재 커밋 SCN(105) > 내 read SCN?"
──────────────────────────────── 여기까지 두 수준 완전히 동일 ──────
```

비교까지는 같고, **"내 read SCN"이 무엇이냐**에서 반응이 갈린다:

```
RC            내 read SCN = 문장마다 새로 잡을 수 있음 → 새 SCN(106)으로 ★재시작★
              재시작 = WHERE 를 현재값(RESERVED) 기준으로 다시 평가
                · WHERE 에 status='AVAILABLE' 있음 → 거짓 → 0행 (그 행을 안 바꿈)
                · WHERE 에 status 조건 없음        → slot_id 는 여전히 참 → RESERVED 덮어씀
              → ★롤백이 아니다★. 재시작 후 "0행" 또는 "그냥 덮어쓰기" 다.

SERIALIZABLE  내 read SCN = 트랜잭션 시작에 고정(100), 새로 못 잡음
              105 > 100 확인 → 재시작이 정의상 불가 → ORA-08177 → ★트랜잭션 롤백★
              → WHERE 조건과 무관. "행이 내 시작 이후 바뀌었다"는 사실만으로 거부.
```

→ **"둘 다 SCN 확인해서 틀리면 롤백"이 아니다.** RC 는 충돌 시 **재시작**하고(롤백 아님), SERIALIZABLE 만 **롤백**한다. SCN 을 보는 데까지는 같고, **재시작이 가능하냐(RC) 불가능하냐(SERIALIZABLE)** 에서 갈린다.

### 3.3 왜 RC 는 충돌 시 재시작하는가 — write consistency (세 겹의 이유)

UPDATE 한 문장은 "한 일관된 시점에서 본 WHERE 결과"여야 한다. 그런데 ①찾기와 ②바꾸기는 보는 값이 다를 수 있다.

```
①찾기   : 문장 시작 스냅샷(read SCN=100)으로 WHERE 맞는 행을 고른다
②바꾸기 : 그 행에 X-락 걸고 ★현재 커밋된 최신 버전★ 위에 덮어쓴다
```

충돌이란 ②바꾸기 시점에 "내가 ①찾기에서 본 것보다 더 최신의 커밋"이 그 행에 있다는 뜻이다. 이때 Oracle 이 택할 수 있는 길은 셋뿐이다.

```
(a) 스냅샷 100 값으로 그냥 밀어붙인다
     → 낡은 버전(100) 위에 덮어쓰게 됨. 그 사이 들어온 변경(105)이 사라진다 = lost update. ✗
(b) 에러를 낸다
     → "에러 없이 최신 커밋 데이터로 문장을 끝낸다"는 RC 의 정의를 어긴다.
       에러로 처리하는 건 SERIALIZABLE 의 ORA-08177 쪽이다. ✗
(c) 새 시점(read SCN=106)으로 문장을 다시 본다 = ★재시작★
     → RC 의 정의를 지키면서 최신 커밋을 반영하는 유일한 길. ✓
```

**이유 1 — RC 의 약속**: RC 는 "각 문장이 커밋된 최신 데이터를 보고 에러 없이 끝낸다"가 정의다. 충돌을 반영하려면 최신 시점에서 다시 보는 수밖에 없다(밀어붙이면 lost update, 에러내면 RC 가 아님).

**이유 2 — 수정 행 집합은 단일 시점이어야 한다**: UPDATE 는 여러 행을 건드릴 수 있다.

```
UPDATE slot SET status='RESERVED' WHERE window_id=42 AND status='AVAILABLE'  (410,411,412 매칭 가능)

①찾기 (SCN100):  {410,411,412} 다 AVAILABLE → 세 행 대상
②바꾸기:  410 OK, 411 OK, 412 → 다른 Tx 가 RESERVED 로 commit(SCN105) 한 것을 발견 → 충돌

  ★ 412만 새 시점으로 평가하면: 410·411=SCN100 결과 / 412=SCN105 결과
     → 한 UPDATE 결과 집합이 두 시점의 혼합 = "한 문장 = 한 시점" 일관성 붕괴
  그래서 Oracle: 이 문장이 한 410·411 수정을 ★문장만★ 롤백(트랜잭션 전체 아님)
     → 새 read SCN(106) 으로 문장을 처음부터 재실행 → {410,411,412} 전부 SCN106 기준 재평가
     → 412 는 이제 RESERVED 라 WHERE 에서 제외 → 단일 시점(106)의 일관된 결과 집합
```

시나리오 1 은 단일 행(slot410)이라 매칭 행이 하나지만 **같은 규칙이 적용**되어, 재시작 → WHERE 재평가 → `status='AVAILABLE'` 거짓 → 0행이 된다.

**이유 3 — 재시작이 가능한 전제**: 재시작은 "새 read SCN 을 잡는 것"이다. RC 는 read SCN 을 문장마다 새로 잡는 게 정의라 재시작이 정의 안에서 허용된다. SERIALIZABLE 은 read SCN 을 트랜잭션 시작에 고정하므로 새 SCN 을 잡으면 정의 위반 → 재시작 카드 자체가 없어 ORA-08177 로 거부한다. 즉 두 수준의 "재시작 가능/불가능"은 별개 기능이 아니라 **read SCN 정책 하나에서 자동으로 따라 나오는 결과**다.

## 4. 구조 × 격리 수준 — "구조 확정 → 격리 수준 판단" 선후 관계

| 슬롯 쓰기 구조 (격리 RC 고정 기준) | READ COMMITTED | SERIALIZABLE | 격리 수준의 역할 |
|---|---|---|---|
| **무조건 UPDATE** (`WHERE SLOT_ID=?`, 현재 = Step 2) | 재시작해도 slot_id 불변 → RESERVED 를 덮어씀 → **못 막음** (Step 2 실측) | 후착 ②바꾸기에서 `현재 커밋 SCN > 내 read SCN` → **ORA-08177** → 막음 | **유일한 정합성 레버** |
| **조건부 UPDATE** (`WHERE ... AND STATUS='AVAILABLE'`) | 재시작 시 `STATUS='AVAILABLE'` 거짓 → **0행** → throw | 0행 대신 **ORA-08177**, 결과 동일 | **정합성엔 무의미** (RC 로 이미 막힘, 신호 형태만 바뀜) |

핵심은 **대각선**이다.

- **무조건 UPDATE** 구조에서는 RC 가 정합성을 못 만든다(WHERE 에 상태 조건이 없어 재시작이 후착을 못 거름). 이 구조에서만 SERIALIZABLE 의 SCN 충돌 검출(ORA-08177)이 유일하게 작동하는 정합성 장치가 된다 → 격리 수준이 "의미 있는" 구조.
- **조건부 UPDATE** 구조에서는 RC 의 쓰기 일관성 재시작만으로 후착이 0행으로 걸러진다. SERIALIZABLE 로 올려도 같은 한 건만 통과하고 거부 신호만 0행→예외로 바뀐다 → 격리 수준이 정합성에는 "무의미"한 구조.

→ 격리 수준의 정합성 효과는 **쓰기 구조(=②바꾸기의 WHERE 에 상태 조건이 있는가)에 전적으로 종속**된다. 그래서 **쓰기 구조를 먼저 확정해야 격리 수준 선택이 의미를 갖는다.** 구조를 보지 않고 격리부터 정하면, 조건부 UPDATE 구조에 SERIALIZABLE 을 얹어 놓고 "정합성을 위해 격리를 올렸다"고 오인할 수 있다(실제로는 RC 가 이미 막고 있고 SERIALIZABLE 은 예외 처리 비용만 추가).

## 5. 정합성 차단 ≠ 거부 사유 구분 (별개의 책임)

```
check-then-act 의 SELECT·검증 (L319·L328·L342·L350·L365·L371)
   → 책임: 거부 "사유 구분" (비즈니스 규칙 2)
        · slot 없음        → VisitSlotNotFoundException (404)
        · 이미 RESERVED    → SlotUnavailableException.alreadyReserved
        · 시작 시각 지남    → SlotUnavailableException.startTimeExpired
        · 자기 매물         → SelfPropertyReservationException
        · 동일 매물 중복    → DuplicateReservationException
        · 시간 겹침        → ReservationTimeOverlapException

슬롯당 1건 정합성 (race 차단)
   → 책임: 별도 차원. 백스톱(부분 유일 인덱스) + 1차 방어선(조건부 UPDATE / 비관적 락 / SERIALIZABLE)
```

조건부 UPDATE 로 SELECT 를 합쳐 없애면 `affected=0` 이 어느 사유인지 모르게 된다:

```
affected=0 의 가능한 원인:
   · 이미 다른 사람이 점유 (RESERVED)   ← 동시성 거부
   · 슬롯이 CLOSED / WITHDRAWN          ← 상태 거부
   · slot_id 가 아예 없음               ← 404
   → 전부 "0행" 으로 뭉쳐져 구분 불가 = 비즈니스 규칙 2 와 충돌
```

→ 그 검증식은 정합성용이 아니라 **사유 구분용**이라 "불필요"하지 않다. 조건부 UPDATE 로 합치더라도 **WHERE 로 가는 건 `status='AVAILABLE'` 하나뿐**이고, 나머지(시작 시각·자기 매물·중복 매물·시간 겹침)는 사유 구분을 위해 SELECT 검증으로 남긴다. 이 경우 조건부 UPDATE 의 `affected=0` 은 "L328 검증을 통과한 뒤의 순수 race(그 사이 다른 Tx 가 점유)"만 의미한다.

## 6. SERIALIZABLE 배제 근거

### 6.1 인과의 방향 — ORA-08177 은 phantom 차단 장치가 아니다

```
SERIALIZABLE 의 목적 = 트랜잭션 레벨 read consistency
   = read SCN 을 트랜잭션 시작에 ★고정★
        ├─[읽기 측] 시작 이후 커밋된 변경이 안 보임
        │           → non-repeatable read · phantom 동시 차단   ← 목적이 달성되는 곳
        └─[쓰기 측] 새 SCN 을 못 잡음 → 재시작 불가
                    → 쓰기 충돌 시 ORA-08177                    ← 부산물
```

정확한 순서는 **목적(트랜잭션 레벨 일관성) → 수단(SCN 고정) → 부산물(재시작 불가 → ORA-08177)**. phantom 차단은 읽기 측에서 이미 끝나고, ORA-08177 은 그 SCN 고정 때문에 쓰기 측에서 재시작 카드를 못 써서 나오는 거부 신호일 뿐이다. "재시작을 안 하는 것"이 phantom 을 막는 게 아니라, "phantom 을 막으려 SCN 을 고정한 결과로" 재시작이 불가능해진 것이다. (Oracle 스냅샷은 non-repeatable read 와 phantom 을 분리해서 막을 수 없어 둘이 같이 차단된다 — 문서 ⑦.)

### 6.2 이 비즈니스엔 차용 근거가 없다 (두 차원 모두)

| 부분 | 경합 유형 | 정합성에 필요한 것 | SERIALIZABLE 이 기여? |
|---|---|---|---|
| 슬롯 점유 (시나리오 1, 같은 슬롯 N명) | **같은 행** 쓰기-쓰기 | X-락 직렬화 | **불필요** — RC 의 X-락 + 조건부 UPDATE 로 이미 잡힘 |
| 중복매물·시간겹침 COUNT 검증 (시나리오 2/3, 같은 사용자) | **다른 행** write-skew | SELECT FOR UPDATE | **못 막음** — Oracle SERIALIZABLE 은 write-skew 자동 차단 안 함 (문서 ⑧마) |

- 같은 행 경합은 트랜잭션 레벨 일관성(phantom 차단)이 필요 없다. 읽은 행이 흔들리든 말든, 직렬화는 같은 슬롯 X-락에서 일어나므로.
- write-skew 는 SERIALIZABLE 로도 못 막으니(FOR UPDATE 가 답), 그걸 위해 SERIALIZABLE 을 쓸 이유가 안 된다.

→ SERIALIZABLE 이 정당화되려면 "여러 행을 시점 일관되게 읽고, 그 집합에 근거해 판단하며, 그 판단이 같은 행 쓰기로 귀결되는" 구조여야 하는데 이 비즈니스엔 그런 경로가 없다. 이것은 "측정 안 했다"가 아니라 **"메커니즘상 기여하지 않아 후보에서 제외한다"는 선별 판단**이다. → **격리 수준 = READ COMMITTED 고정(측정 변수가 아닌 전제), SERIALIZABLE 배제.**

## 7. JPA ↔ Oracle 매핑 — 격리 수준과 락은 다른 축이다

### 7.1 두 개는 직교하는 축

```
축 1 — 격리 수준 (읽기 일관성 등급, ★트랜잭션 전체★에 적용)
   READ COMMITTED ──────── SERIALIZABLE          ← Oracle 이 제공하는 등급
   "내 트랜잭션이 다른 트랜잭션의 커밋을 어디까지 보는가"

축 2 — 락 방식 (쓰기 충돌 제어 전략, ★특정 행/엔티티★에 적용)
   비관적(SELECT FOR UPDATE) ──────── 낙관적(@Version)
   "동시 수정 충돌을 미리 막는가 / 사후에 검출하는가"

두 축은 직교 — 함께 쓰인다. 예: "READ COMMITTED + 비관적 락" 이 동시에 성립.
```

("배타적 락"이라는 표현은 정확히는 **비관적 락**을 가리킨다. 배타적 락(X-락)은 락의 *모드*이고, 비관적 락은 그 X-락을 *미리 거는 전략*이다.)

### 7.2 JPA 표기 ↔ 발행물 ↔ Oracle

| 개념 | JPA 표기 | JPA 가 발행하는 것 | Oracle 에서 |
|---|---|---|---|
| **격리 수준** | `@Transactional(isolation=...)` | `Connection.setTransactionIsolation()` | RC·SERIALIZABLE 만 받음 |
| **비관적 락** | `@Lock(LockModeType.PESSIMISTIC_WRITE)` | `SELECT ... FOR UPDATE` | 행 X-락 (모든 격리수준에서 동작) |
| **낙관적 락** | `@Version` | `UPDATE ... WHERE version=?` + affected 검사 | 앱 레벨 (격리수준과 무관) |

→ 격리 수준은 **JDBC 커넥션 설정**이고, 락은 **JPA 가 SQL 자체를 다르게 생성**하는 것이다. 경로가 완전히 다르다.

### 7.3 격리 수준 어노테이션은 Oracle 이 받는 것만 매핑된다

```
Isolation.READ_COMMITTED   → Oracle TRANSACTION_READ_COMMITTED  ✓
Isolation.SERIALIZABLE     → Oracle TRANSACTION_SERIALIZABLE    ✓
Isolation.REPEATABLE_READ  → Oracle JDBC 가 거부 → SQLException ✗ (트랜잭션 시작 실패)
Isolation.READ_UNCOMMITTED → 마찬가지로 거부                     ✗
```

JPA 어노테이션에 값이 있다고 다 매핑되는 게 아니라, DB 가 거부하면 런타임 실패다.

### 7.4 Oracle 에 REPEATABLE READ 가 없는데 비관적 락은 무엇인가

**비관적 락은 격리 수준이 아니라 명시적 행 락이다.**

```
REPEATABLE READ (격리 ★등급★)  = "내가 읽은 행을, 내 트랜잭션 끝까지, 다른 Tx 가 못 바꾼다"
                                  → 트랜잭션 전체에 걸쳐 내가 읽은 ★모든★ 행을 자동 고정

Oracle:  이 ★등급★은 제공하지 않는다 (RC·SERIALIZABLE 둘뿐).
         하지만 그 ★효과★를, 필요한 ★특정 행에만★, SELECT ... FOR UPDATE 로 재현한다.
         → JPA @Lock(PESSIMISTIC_WRITE) 가 바로 이 FOR UPDATE 를 발행한다.
```

→ Oracle 은 "REPEATABLE READ 라는 자동 등급"은 안 주지만, 그 효과가 필요하면 개발자가 행을 콕 집어 `FOR UPDATE` 로 직접 잠그라는 설계다. 그래서 **REPEATABLE READ 지원 여부와 비관적 락 사용 가능 여부는 아무 관계가 없다** — 비관적 락은 격리 수준 축이 아니라 락 축이다.

### 7.5 락 어노테이션이 "격리 수준과 독립"인 이유 — 다른 레이어에서 동작

```
[레이어 1] JDBC Connection 속성   ← @Transactional(isolation=...)
     conn.setTransactionIsolation(READ_COMMITTED / SERIALIZABLE)
     → DB 의 read SCN 정책을 바꿈. "스냅샷 읽기가 어느 시점을 보는가"

[레이어 2] SQL statement 텍스트   ← @Lock / @Version
     @Lock(PESSIMISTIC_WRITE) → Hibernate 가 SELECT 끝에 "FOR UPDATE" 를 붙임
     @Version                 → Hibernate 가 UPDATE 의 WHERE 에 "AND version=?" 를 붙이고
                                affected row 수를 검사
     → 커넥션 속성을 건드리지 않는다. SQL 문자열 자체가 달라질 뿐.
```

같은 `@Lock`/`@Version` 은 RC 에서든 SERIALIZABLE 에서든 **똑같은 SQL** 을 생성한다. 이것이 "독립"의 정체다.

**`@Version` 추적 (RC 에서 작동하는 모습):**

```
T1, T2 동시 (둘 다 version=0 읽음), 격리 = RC
T1: UPDATE ... SET VERSION=1 WHERE SLOT_ID=410 AND VERSION=0 → affected=1 → commit
T2: UPDATE ... SET VERSION=1 WHERE SLOT_ID=410 AND VERSION=0 → affected=0 → OptimisticLockException
                                              (T1 이 version 을 1 로 바꿔서 WHERE 불일치)
```

DB 는 RC UPDATE 두 개를 평범하게 처리했을 뿐이고, 충돌 "검출"은 `WHERE VERSION=0` 이 빗나가 0행이 된 것을 **Hibernate 가** 본 것이다(DB 는 일반 UPDATE 로 봄). 그래서 SERIALIZABLE 이 전혀 필요 없다.

### 7.6 측정 후보 셋의 정체

```
후보 B (비관적 락)    = @Lock(PESSIMISTIC_WRITE) → SELECT FOR UPDATE   → 락 축(비관적), 행 단위
후보 C (@Version)     = 앱 레벨 낙관적 락         → UPDATE WHERE version → 락 축(낙관적), 엔티티 단위
후보 D (조건부 UPDATE) = UPDATE WHERE status='AVAILABLE'               → 레이어 2(SQL WHERE) + RC
후보 E (SERIALIZABLE)  = DB 격리 수준(낙관적 구현) → ORA-08177           → 격리 축
```

C 와 D 는 같은 메커니즘이다 — WHERE 에 "버전/상태" 조건을 넣고, RC 의 쓰기 일관성 재시작으로 후착이 0행이 되면 거부. 다른 점은 조건 컬럼이 `version` 이냐 `status` 냐, 예외를 코드가 던지냐 Hibernate 가 던지냐뿐이다. → **C·D 는 레이어 2 + RC, E 만 레이어 1(격리).** B·C·D 는 RC 위에서 그대로 작동하고, E 만 그 자체가 격리를 바꾼다.

## 8. 비관적 락 (`SELECT FOR UPDATE`) 메커니즘 — 전체 깊이

### 8.1 `FOR UPDATE` 는 '읽기'가 아니다

Oracle 은 행에 접근하는 SQL 을 두 부류로 나눈다.

| SQL | Oracle 의 분류 | 읽는 값 | 락 |
|---|---|---|---|
| 일반 `SELECT` | **consistent read** (스냅샷 읽기) | 내 스냅샷 시점 값 | **없음** |
| `UPDATE` / `DELETE` | current read + 잠금 | 현재 커밋된 최신 값 | **X-락** |
| `SELECT ... FOR UPDATE` | current read + 잠금 | 현재 커밋된 최신 값 | **X-락** |

```
일반 SELECT       = "지금 이 값을 보기만 하겠다"          → 읽기 → 락 불필요 → 스냅샷
SELECT FOR UPDATE = "이 행을 읽고 ★곧 수정하겠다★"       → 수정 의도 → 락 필요 → current read + X-락
```

이름 자체가 `FOR UPDATE`(업데이트를 위하여)다. "이 행을 곧 UPDATE 할 거니까 지금 미리 잡아두겠다"는 **수정 예약 선언**이지 단순 조회가 아니다. 그래서 Oracle 은 이걸 읽기가 아니라 쓰기 계열로 취급해 X-락을 건다. → "Oracle 은 읽기에 락을 안 건다"의 '읽기'는 **일반 SELECT 한정**이며, `FOR UPDATE` 는 애초에 '읽기'로 분류되지 않으므로 두 문장은 충돌하지 않는다.

### 8.2 행 X-락이란

```
행 X-락 (row exclusive lock):
   · 특정 한 행을 "이 트랜잭션이 수정 중 / 수정할 예정" 이라고 표시하는 배타적 권리.
   · 배타적(eXclusive) = 한 행에 한 트랜잭션만 가질 수 있다.
   · 누가 거나 : UPDATE / DELETE / INSERT 는 ★자동으로★, SELECT FOR UPDATE 는 ★명시적으로★.
   · 언제까지 : 그 트랜잭션이 commit / rollback 할 때까지 유지.
   · 효과    : 다른 트랜잭션이 같은 행을 수정(UPDATE/DELETE)하거나 X-락을 잡으려(FOR UPDATE) 하면
              → 락이 풀릴 때까지 대기.
              단, 다른 트랜잭션의 ★일반 SELECT 는 안 막는다★ (스냅샷 읽기라 락과 무관).
```

### 8.3 선착이 X-락을 쥔 상태에서, 후착의 일반 SELECT 와 FOR UPDATE 는 다르게 동작한다

전제: T1(선착)이 `SELECT ... 410 FOR UPDATE` 로 410 행에 X-락을 걸어둔 상태(commit 전).

| 후착 T2 의 시도 | T2 가 락을 요청? | T1 의 X-락과 | 결과 |
|---|---|---|---|
| **일반 `SELECT`** | **안 함** (consistent read) | 만날 일 없음 | **즉시 통과** — 옛 확정값(AVAILABLE)을 스냅샷으로 읽고 지나감 |
| **`SELECT ... FOR UPDATE`** | **X-락 요청** | 충돌 | **대기** — T1 이 commit/rollback 할 때까지 멈춤 |

```
일반 SELECT (T2):  락을 요청하지 않는다 → T1 의 X-락과 충돌할 락 요청 자체가 없다
                  → 자기 스냅샷의 확정값을 즉시 읽고 지나간다. T1 의 미커밋 변경(RESERVED)은 안 본다.
FOR UPDATE (T2):   X-락을 요청한다 → 같은 행에 T1 의 X-락이 있음 → 두 X-락은 공존 불가 → 대기.
```

이것이 "비관적 락이 동시성 제어로 성립하려면 경쟁하는 **모든** 트랜잭션이 `FOR UPDATE` 로 읽어야 한다"는 이유다 — 한쪽이 일반 SELECT 를 쓰면 그쪽은 락을 안 만나 그냥 통과해 버린다. 시나리오 1 에서 이것이 성립하는 이유는 모든 searcher 의 예약 요청이 같은 `findByIdForUpdate` 경로를 타기 때문이다.

### 8.4 "대기" 의 정체 — 단계마다 펼친 전 과정

후착 T2 가 `slotRepository.findByIdForUpdate(410)`(= `SELECT ... WHERE SLOT_ID=410 FOR UPDATE`)를 호출하는 순간부터를 코드 줄과 함께 푼다.

```java
VisitSlotEntity slot = slotRepository.findByIdForUpdate(410);  // ← 이 줄에서 멈춘다
if (slot.getStatus() != VisitSlotStatus.AVAILABLE) {           // ← 여기까지 못 온다
    throw ...
}
```

**(1) T2 가 X-락을 요청한다.**
`FOR UPDATE` 는 "410 행을 읽으면서 동시에 그 행에 X-락을 걸어라"는 요청이다. 읽기와 락이 **한 묶음**이다 — 락을 못 걸면 읽은 결과도 반환하지 않는다.

**(2) 그 락을 받지 못한다.**
410 행에는 이미 T1 이 X-락을 걸어 둔 상태다. X-락은 배타적이라 한 행에 한 트랜잭션만 가질 수 있다. 그러므로 T1 이 쥐고 있는 한 T2 는 같은 410 행에 또 하나의 X-락을 겹쳐 걸 수 없다. T2 의 X-락 요청은 "거부되어 에러로 끝나는" 것이 **아니라** "보류"된다 — T1 이 언젠가 락을 놓으면 그때 받을 수 있으므로, 에러를 내지 않고 받을 수 있을 때까지 기다리는 쪽으로 처리된다.

**(3) 못 받으면 그 자리에서 멈춘다.**
`FOR UPDATE` 호출은 락을 걸어야 결과 행을 반환한다. 락을 못 건 T2 는 결과 행을 받지 못하므로, 자바에서 호출한 `findByIdForUpdate(410)` 가 **리턴하지 않고 그 줄에 머문다.** 그 결과 다음 줄 `if (slot.getStatus() != AVAILABLE)` 판정으로 넘어가지 못한다. "판정에 쓸 값(slot)을 아직 못 받는다"는 것이 바로 이 상태다 — 호출이 끝나야 행이 나오는데, 락을 못 걸어 호출이 안 끝나니 행 자체가 손에 들어오지 않는다.

**(4) 멈춤은 정확히 T1 의 commit/rollback 까지 지속된다.**
X-락은 그 락을 건 트랜잭션이 commit 또는 rollback 을 할 때까지 유지된다. 그 두 사건 외에는 락이 중간에 저절로 풀리지 않는다. 그래서 T2 의 멈춤은 정확히 T1 이 commit/rollback 하는 시점까지 이어진다. T1 이 0.1초 만에 끝나면 T2 는 0.1초 멈추고, T1 이 10분을 끌면 T2 는 10분을 그 자리에서 멈춰 있다. 멈춰 있는 동안 T2 를 처리하던 톰캣 요청 스레드는 이 `findByIdForUpdate(410)` 호출에 묶여 그 요청의 다음 처리를 하나도 진행하지 못하며, 다만 CPU 를 돌리지는 않고 락을 받을 수 있게 될 때까지 잠들어 있다.

### 8.5 멈춤이 풀린 뒤 — T1 이 어떻게 끝났느냐로 갈린다

```
T1 이 commit 한 경우:
   410 행 = RESERVED 로 확정, T1 의 X-락 해제
   → 보류돼 있던 T2 의 X-락 요청이 받아들여져 T2 가 410 에 X-락 획득
   → current read 로 확정된 최신값(RESERVED) 읽음 → findByIdForUpdate 가 RESERVED 행 반환
   → 다음 줄 if(status != AVAILABLE) → RESERVED → 참 → throw  → T2 거부

T1 이 rollback 한 경우:
   410 행 = AVAILABLE 로 되돌아감, T1 의 X-락 해제
   → T2 가 410 에 X-락 획득 → AVAILABLE 읽음 → findByIdForUpdate 가 AVAILABLE 행 반환
   → if(status != AVAILABLE) → 통과 → T2 가 그 슬롯을 이어서 점유
```

**시간선:**

```
시각  T1                                      T2
────────────────────────────────────────────────────────────────────────
t1   SELECT ...410 FOR UPDATE
       → 410 에 X-락 걸기 성공
       → AVAILABLE 읽음
t2                                            SELECT ...410 FOR UPDATE
                                                → 410 에 X-락 걸기 시도
                                                → T1 이 X-락 보유 중 → 배타적이라 못 겹침
                                                → 결과 행 못 받음 → findByIdForUpdate 가 그 줄에 머묾
                                                → 다음 줄 if 판정에 도달 못 함
t3   검증 통과 → UPDATE RESERVED → INSERT 예약
t4   commit → 410=RESERVED 확정, X-락 해제
t5                                            보류됐던 X-락 요청 받아들여짐 → 410 에 X-락 획득
                                                → current read 로 RESERVED 읽음
t6                                            if(status != AVAILABLE) → RESERVED → throw
────────────────────────────────────────────────────────────────────────
결과: slot 410 = RESERVED, CONFIRMED 예약은 T1 의 1건. T2 는 거부.
```

### 8.6 [심화] "X-락" 과 "대기" 의 물리적 구현 — 사용자가 V$LOCK 으로 실측한 그 행들

> 8.1~8.5 의 락 동작이 Oracle 내부에서 어떻게 구현되는지의 한 겹 아래. 2차(비관적 락) 측정 때 V$LOCK 에 잡히는 신호의 정체다.

**X-락의 실체 — 별도 락 테이블이 아니라 데이터 블록 안에 있다.**

```
· 각 데이터 블록 헤더에 ITL(Interested Transaction List) 이라는 작은 슬롯 목록이 있다.
· 어떤 트랜잭션이 그 블록의 행을 수정/잠그면, ITL 슬롯 하나를 차지하고 자기 트랜잭션 ID(XID)를 적는다.
· 잠긴 행의 행 헤더에는 "나는 ITL 의 몇 번 슬롯에 잠겼다" 는 표시(lock byte)가 붙는다.

→ T1 이 410 에 건 행 X-락의 실체 = 블록 ITL 슬롯의 T1 XID + 410 행 헤더의 lock byte.
  (중앙 락 목록이 메모리 어딘가에 있는 게 아니라 데이터에 직접 표시된다.
   그래서 Oracle 은 잠그는 행 수에 제한도, lock escalation 도 없다.)
```

**"대기" 의 실체 — 행이 아니라 T1 트랜잭션을 기다린다 (TX enqueue).**

```
· 모든 트랜잭션은 시작 시 자기 TX enqueue(트랜잭션 락)를 exclusive(mode 6)로 잡는다. T1 도 6 보유 중.
· T2 는 410 행 헤더의 lock byte → ITL 슬롯의 XID(=T1) 를 읽고, 그 XID 가 active 인지 확인한 뒤
  → ★T1 의 TX enqueue 를 share(mode 4)로 요청★ → mode 6 과 4 는 공존 불가 → 획득 실패 → 큐에서 sleep.
· 이 sleep 상태의 정식 이름(wait event) = enq: TX - row lock contention.
· T1 이 commit/rollback → TX enqueue(6) 해제 → mode 4 로 기다리던 T2 가 posted(깨어남)
  → 410 을 current read 로 다시 평가해 최신값(RESERVED) 읽고 X-락 획득.
```

**V$LOCK 관측값 (사용자 slot 403 수동 측정, 중간보고서 5.3 §B·§C):**

```
§B (V$LOCK):
   SID 252(T1) TYPE=TX LMODE=6(Exclusive) BLOCK=1   ← T1 이 TX enqueue 6 보유, 남을 막는 중
   SID 857(T2) TYPE=TX LMODE=0 REQUEST=4(Share)     ← T2 가 TX enqueue 4 요청하며 대기
§C (블록 관계):
   WAITING_SID 857 → HOLDING_SID 252
   WAIT_EVENT = enq: TX - row lock contention
   WAIT_SECONDS = 569
```

(주의: 사용자 §B 케이스는 INSERT 의 PK 인덱스 키 충돌이었고, 비관적 락은 테이블 행 X-락 충돌이다 — 잠그는 대상이 인덱스 키냐 테이블 행이냐만 다르고, **대기 메커니즘은 똑같은 TX enqueue Share 대기**다. 그래서 2차 측정에서도 V$LOCK 에 동일한 `LMODE=6` / `REQUEST=4` / `enq: TX - row lock contention` 패턴으로 잡힌다.)

### 8.7 통상 UPDATE 와의 차이 — 판정에 쓰는 읽기의 직렬화 시점

두 흐름 다 마지막에 UPDATE 가 있고, 둘 다 UPDATE 단계에서 X-락 대기는 일어난다. 그런데도 결과가 갈리는 이유는 **"판정(`if status != AVAILABLE`)에 쓰는 읽기"가 직렬화되는 시점**이 다르기 때문이다.

```
━━ 통상 check-then-act (일반 SELECT → 판정 → 조건 없는 UPDATE), RC ━━━━━━━━━━━
   T1: SELECT(일반) → AVAILABLE ┐
   T2: SELECT(일반) → AVAILABLE ┘  둘 다 락 없이 ★동시에★ 읽음 = check 단계에 틈
   T1·T2: if(status != AVAILABLE) → 둘 다 거짓 → 둘 다 통과
          ★T2 도 여기서 이미 "AVAILABLE 이다" 라고 판정을 끝내 버린다 (T1 을 못 봤으니까)★
   T1: UPDATE → X-락 → commit → 해제
   T2: UPDATE → 대기 → 깨어남 → (WHERE 에 status 조건 없으니) RESERVED 를 ★덮어씀★
       → 이미 끝낸 판정대로 INSERT 진행 → CONFIRMED 2건 → ★깨짐★

━━ 비관적 락 (SELECT FOR UPDATE → 판정 → UPDATE), RC ━━━━━━━━━━━━━━━━━━━━━
   T1: SELECT FOR UPDATE → X-락 획득, AVAILABLE 읽음 ┐
   T2: SELECT FOR UPDATE → ★대기★ (판정에 쓸 값을 아직 못 읽음) ┘  읽기 단계에서 이미 직렬화
   T1: 판정 통과 → UPDATE RESERVED → INSERT → commit → 해제
   T2: 대기 풀림 → current read 로 ★RESERVED★ 읽음 → 판정에서 throw
       ※ T2 는 선착이 commit 한 뒤에야 판정하므로 ★선착의 결과를 보고★ 판정한다
```

```
통상 흐름  : 판정에 쓰는 읽기(SELECT)가 병렬 → 후착은 ★자기 스냅샷★으로 판정 → 선착을 못 본 채 통과 → TOCTOU 틈
비관적 락  : 판정에 쓰는 읽기(SELECT FOR UPDATE)가 직렬 → 후착은 선착 commit 후 ★최신값★으로 판정 → 틈이 사라짐
```

비관적 락의 본질은 "판정(check)에 쓰는 읽기 자체를 한 번에 하나씩 만든다"는 것이다.

## 9. 세 구조 종합 비교

| 구조 (격리 RC) | 후착 일반 SELECT | 후착이 막히는 지점 | 직렬화 위치 | 후착 거부 신호 | 정합성 |
|---|---|---|---|---|---|
| check-then-act (조건 없는 UPDATE, 현재) | 통과 | UPDATE 에서 대기 → **덮어씀** | 쓰기 | (거부 안 됨) | **깨짐** |
| 조건부 UPDATE (`AND status='AVAILABLE'`) | 통과 | UPDATE 에서 대기 → **0행** | 쓰기 | affected=0 | OK |
| 비관적 락 (FOR UPDATE) | 통과 | **SELECT FOR UPDATE 에서 대기** | 읽기 | 락 대기 후 검증 throw | OK |

- **세 구조 모두 후착의 일반 SELECT 는 통과한다** (Oracle 읽기 비차단). 다른 점은 ① 후착이 어디서 막히느냐(읽기 vs 쓰기 단계), ② 막힌 뒤 무엇으로 거부되느냐다.
- **응답시간 분포 차이의 근원:**

```
조건부 UPDATE : 후착이 검증·작업을 다 진행한 뒤 ★마지막 UPDATE 에서 0행★ → 검증 비용을 다 치름
비관적 락     : 후착이 ★처음 SELECT FOR UPDATE 에서 대기★ → 선착이 끝날 때까지 멈췄다가 판정
```

둘 다 정합성(슬롯당 1건)은 같지만, 후착이 "끝까지 가서 0행"이냐 "처음부터 대기"냐가 달라 응답시간 분포가 갈린다. 이것이 1차(조건부 UPDATE)와 2차(비관적 락)를 둘 다 측정해 비교하려는 이유다.

## 10. 설계 결론 (⑨) 과 측정 계획

### 10.1 결론

```
1. 정합성은 쓰기가 만든다. 읽기(SELECT)는 락이 없어 동시 N개가 다 통과한다.
   직렬화는 ②바꾸기(UPDATE 의 X-락)에서만 일어난다.

2. 정합성을 만들 수 있는 두 위치:
     백스톱     = VISIT_RESERVATION INSERT 의 부분 유일 인덱스 → ORA-00001 (사후 차단)
     1차 방어선 = VISIT_SLOT 쓰기 단계에서 후착 차단 (조건부 UPDATE / 비관적 락 / SERIALIZABLE)
   두 위치는 직교 — 운영 반영 시 백스톱을 함께 둘지는 1차 방어선 선택과 별개 결정.

3. 격리 수준 = read SCN 시점을 정함 = "읽기" 를 지배. 정합성엔 직접 작동 안 함.
   닿는 통로는 ②바꾸기의 SCN 충돌 반응뿐: RC=재시작(WHERE 재평가) / SERIALIZABLE=ORA-08177(롤백).

4. 격리 수준이 "정합성 레버" 가 되는 건 무조건 UPDATE 유지 시뿐. 조건부 UPDATE 로 가면
   RC 로 충분하고 격리 수준은 무의미. → "쓰기 구조 확정 → 격리 수준 판단" 순서.

5. SERIALIZABLE 의 본질 = 트랜잭션 레벨 read consistency. 같은 행 경합엔 불필요, write-skew 는 못 막음.
   → 정합성이 필요한 어느 지점에서도 기여 없음. ★배제★. 격리 수준 = READ COMMITTED ★고정★ (전제).

6. 정합성 차단 ≠ 거부 사유 구분. check-then-act 의 SELECT 검증은 사유 구분용(비즈니스 규칙 2)이라
   조건부 UPDATE 로 합쳐도 status 조건만 WHERE 로 가고 나머지 검증은 SELECT 로 남긴다.

7. "쿼리 설계 미흡" 이 아니다. 백스톱은 원래 있었고 Step1 에서 작동. 1차 방어선은 측정 후 선택으로
   "위임" 된 상태 (WriteService 클래스 주석 = 설계 명세서 8.4 명시).
```

### 10.2 결정 대상 — 격리 수준이 아니라 1차 방어선 기법

```
이미 고정:   격리 수준 = READ COMMITTED  (SERIALIZABLE 배제)
결정 대상:   1차 방어선 기법 = 조건부 UPDATE(후보 D)  vs  비관적 락 FOR UPDATE(후보 B)
             (후보 C @Version 은 version 조건부 UPDATE 라 D 와 같은 계열)
```

### 10.3 측정 계획

| 회차 | 구조 (격리 RC 고정) | 측정 목적 | 도구 |
|---|---|---|---|
| (확보됨) Step 2 | 무조건 UPDATE | baseline 붕괴 (5건) | — |
| **1차** | **조건부 UPDATE** (`AND status='AVAILABLE'`) | 정합성 회복 + 락 동작 | JMeter N=5 + V$LOCK 수동 |
| **2차** | **비관적 락 (FOR UPDATE)** | 정합성 + 응답시간·락 대기 | JMeter N=5 + V$LOCK 수동 |
| (선택) | SERIALIZABLE | 배제 결론 실측 보강 (결정용 아님) | JMeter |

```
정합성  = JMeter 자동  /  V$LOCK = SQL Developer 수동 관찰 (slot 403 에서 한 방식). 도구 분리.
조건부 UPDATE 미적용 = Step2(이미 확보) → 적용 회차만 추가하면 자동 대조. 별도 미적용 회차 불필요.
```

### 10.4 결정 축

```
정합성        → 조건부 UPDATE · 비관적 락 둘 다 OK (이미 메커니즘으로 앎, 측정은 재현)
실제 선택 기준 → · 응답시간 분포 (비관적 락의 대기 vs 조건부 UPDATE 의 즉시 0행)  ← 측정으로
                · 사유 구분 보존 (둘 다 SELECT 검증 남기면 보존)               ← 설계로
                · 운영 단순성 (락 명시 vs WHERE 한 조각)                       ← 설계로
```

| 이미 메커니즘으로 결정 가능 (측정은 재현) | 측정으로만 알 수 있음 |


## 부록 C. 용어

| 용어 | 뜻 |
|---|---|
| ①찾기 (consistent read) | 스냅샷 시점 값을 락 없이 읽음 (일반 SELECT) |
| ②바꾸기 (current read + X-락) | 현재 커밋된 최신 값에 X-락 걸고 덮어씀 (UPDATE/DELETE/INSERT, SELECT FOR UPDATE) |
| 선착 / 후착 | 같은 행 경합에서 먼저 X-락을 쥔 Tx / 뒤에 와서 대기하는 Tx |
| 백스톱 | VISIT_RESERVATION INSERT 의 부분 유일 인덱스 (사후 차단, 동시성 제어와 직교) |
| 1차 방어선 | VISIT_SLOT 쓰기 단계의 후착 차단 (조건부 UPDATE / 비관적 락 / SERIALIZABLE) |
| 재시작 (statement restart) | RC 가 충돌 시 새 read SCN 으로 같은 문장을 자동 재실행 (롤백 아님, 앱은 모름) |
| TX enqueue | 트랜잭션 락. 시작 시 mode 6(Exclusive) 보유. 후착은 mode 4(Share) 요청하며 대기 |
