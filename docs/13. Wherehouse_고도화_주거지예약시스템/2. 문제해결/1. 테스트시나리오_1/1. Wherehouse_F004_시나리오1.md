# Wherehouse 매물 방문 예약 — F004 시나리오 1 비즈니스 맥락 및 테스트 맥락

**대상 코드** `VisitReservationWriteService.createReservation` (라인 311~413)
**대상 설계 명세서 섹션** 6.4 (F004 방문 슬롯 예약)
**대상 시나리오** 시나리오 1 — 인기 매물 희소 슬롯의 직접 경합 + 재개방 직후 구독자 집중 경합
**문서 범위** 비즈니스 맥락, 문제 예상 지점, 테스트 맥락 정의. 동시성 제어 기법 비교와 선택은 측정 결과 도출 후 별도 의사결정 문서에서 다룬다.

---

## 1. 비즈니스 맥락

### 1.1 기능 위치

Wherehouse에 매물 방문 예약 기능을 신규 도입했다. 예약의 기본 단위는 방문 슬롯이다. 등록자가 매물에 대해 방문 가능 윈도우를 공개하면, 윈도우가 고정 길이(기본 30분)로 분할되어 슬롯이 자동 생성된다. 탐색자는 그중 한 슬롯을 선택해 예약을 확정한다.

F004의 대상 지점은 슬롯 예약 처리 코드다. 메서드 시그니처는 `VisitReservationWriteService.createReservation(ReservationCreateRequestDto, String userId)`이며, 같은 슬롯에 대해 N개 호출이 거의 동시에 발생할 수 있고, 각 호출은 독립된 트랜잭션을 갖는다.

### 1.2 시나리오 1의 발생 형태

시나리오 1은 두 형태로 발생하지만 코드 경로가 동일하므로 하나로 묶어 다룬다.

**형태 1. 인기 매물 희소 슬롯의 직접 경합.** 한 매물의 특정 시간대 슬롯(예: 토요일 오후 2시)이 매물 상세 화면을 통해 여러 탐색자에게 동시에 노출된다. 매물 매력도가 높고 그 시간대가 직장인·학생의 공통 가용 시간일수록 여러 탐색자가 거의 동시에 예약 버튼을 누른다.

**형태 2. 재개방 직후 구독자 집중 경합.** 한 탐색자가 슬롯 X의 예약을 취소한다. 슬롯 X에 활성 구독을 가진 N명에게 `SLOT_REOPENED` 알림이 동시에 발송된다. 구독자들은 알림 수신 직후 슬롯 X 재예약을 거의 동시에 시도한다. 같은 시점에 구독자가 아닌 다른 탐색자가 매물 상세 화면을 새로 열어 슬롯 X를 시도할 수도 있다.

두 형태의 트리거는 다르지만(매물 상세 노출 vs. 재개방 알림 수신), `createReservation` 호출이 같은 슬롯 식별자에 거의 동시에 N건 도착한다는 점은 동일하다.

### 1.3 비즈니스 규칙

기획서 7과 비기능 요구사항 4.1·4.2가 명시한 규칙이다.

**규칙 1. 슬롯당 정원 1.** 한 슬롯의 확정 예약은 어느 시점에나 최대 1건이다. 동시 시도 환경에서도 한 슬롯에 두 명이 확정되는 사건은 0건이어야 한다.

**규칙 2. 거부의 즉시 통보.** 거부된 탐색자는 거부 사실, 사유, 대체 행동의 진입점(다른 슬롯 시도, 재개방 알림 구독)을 즉시 받아야 한다. 거부 응답이 지연되어 사용자가 자신의 결과를 모른 채 대기하는 상태가 발생해선 안 된다.

본 시나리오의 처리는 두 규칙을 동시에 만족해야 한다.

## 2. 문제 예상 지점

### 2.1 대상 코드 두 지점

`createReservation` 내부의 두 지점이 문제 구간의 양 끝이다.

**지점 (a). 검증 단계 — 라인 326~344.**

```java
// 2단계: 슬롯이 활성화되어 있고 예약 가능한지 확인
if (slot.getStatus() != VisitSlotStatus.AVAILABLE) {
    throw SlotUnavailableException.alreadyReserved(
            window.getLeaseType(),
            findAvailableSlotItems(window.getPropertyId(), window.getLeaseType()));
}

if (!slot.getStartTime().isAfter(now)) {
    throw SlotUnavailableException.startTimeExpired(
            window.getLeaseType(),
            findAvailableSlotItems(window.getPropertyId(), window.getLeaseType()));
}

// 3단계: 자기 매물 예약 제한
PropertyOwner owner = lookupPropertyOwner(window.getPropertyId(), window.getLeaseType());
if (userId.equals(owner.registeredUserId)) {
    throw new SelfPropertyReservationException(
            "자신이 등록한 매물의 슬롯은 예약할 수 없습니다.");
}
```

라인 318의 `slotRepository.findById(slotId)`로 슬롯 현재 상태를 조회한 결과를 받아 세 가지 검증을 순차 수행한다. 첫째, 슬롯 `STATUS`가 `AVAILABLE`인지. 둘째, 슬롯 시작 시각이 현재 시각 이후인지. 셋째, 요청자가 매물 등록자 자신이 아닌지. 어느 하나라도 거짓이면 즉시 거부한다. 모두 참이면 라인 346~366의 동일 매물 중복 예약 검증과 시간 겹침 검증(DB 측 JOIN COUNT 쿼리 2회 — `countDuplicatePropertyReservation`, `countTimeOverlappingReservation`)으로 진행한다.

**지점 (b). 슬롯 상태 전이 + 예약 INSERT — 라인 369~379.**

```java
slot.setStatus(VisitSlotStatus.RESERVED);
slotRepository.save(slot);

VisitReservationEntity reservation = VisitReservationEntity.builder()
        .slotId(slot.getSlotId())
        .searcherUserId(userId)
        .status(VisitReservationStatus.CONFIRMED)
        .confirmedAt(now)
        .build();
VisitReservationEntity savedReservation = reservationRepository.save(reservation);
```

슬롯 `STATUS`를 `AVAILABLE`에서 `RESERVED`로 UPDATE하고, `CONFIRMED` 상태의 예약 행을 INSERT한다. 두 데이터베이스 작업이 순차 실행된다.

### 2.2 두 지점 사이의 시점 불일치 구간

지점 (a)의 슬롯 상태 조회 시각과 지점 (b)의 상태 변경 시각 사이에 시간 간격이 존재한다. 이 간격에서 트랜잭션은 자기 매물 검증(라인 339~344), 동일 매물 중복 예약 검증과 시간 겹침 검증(라인 346~366, DB 측 JOIN COUNT 쿼리 2회)을 수행한다.

같은 슬롯에 대해 N건 트랜잭션이 거의 동시에 도착하면 N개 모두 지점 (a)에서 슬롯을 `AVAILABLE`로 인지하고, N개 모두 검증을 통과하며, N개 모두 지점 (b)로 진입한다.

이는 Check-then-Act 패턴이다. 조회 시점의 상태가 변경 시점에도 동일하다는 가정 위에 작성된 흐름이지만, 동시성 환경에서는 그 가정이 보장되지 않는다.

### 2.3 단순 INSERT 중복 방지와의 본질적 차이

본 문제는 두 사용자가 같은 PK로 INSERT를 시도하는 단순 동시성 문제와 다르다. 설계 명세서 섹션 8.3이 이 차이를 명시한다.

**유형 1. 부재 확인 + 저장.** 기존 매물 등록(p4)이 다룬 형태. 두 트랜잭션이 같은 매물 식별자로 INSERT를 시도한다. 대상 행이 아직 없는 상태에서 두 트랜잭션 모두 "동일 매물 부재"를 확인한 뒤 저장으로 진입하지만, 경합 단위가 매물 식별자이므로 데이터베이스의 식별자 기반 메커니즘(유일 인덱스)이 두 번째 INSERT를 거부한다.

**유형 2. 상태 읽기 + 판단 + 상태 쓰기.** F004가 다루는 형태. 슬롯 행은 이미 존재하며 `AVAILABLE` 상태다. 두 트랜잭션이 그 상태를 읽고, `AVAILABLE` 판단을 기반으로 슬롯을 `RESERVED`로 UPDATE한다. 경합 단위가 식별자가 아니라 상태값이다. UPDATE 시점에 자동 행 락이 두 UPDATE를 순차 실행시키지만, 자동 행 락의 동작은 순차 실행이지 두 번째 UPDATE 거부가 아니다. 따라서 두 트랜잭션 모두 자신의 UPDATE를 완료하고 INSERT 단계로 진입한다. 별도의 데이터베이스 차원 메커니즘이 없으면 INSERT도 통과되어 한 슬롯에 두 개의 `CONFIRMED` 예약이 남는다.

두 유형은 비즈니스 행위 단위, 데이터 모델 차원의 정합성 보장 메커니즘, 측정 항목이 모두 다르다. F004는 유형 2이며, 유형 1의 메커니즘으로는 다룰 수 없다.

## 3. 동시 발생 시 예상 흐름

본 절은 추가 동시성 제어 기법이 없는 현재 코드 상태에서 같은 슬롯에 N건이 거의 동시에 도착했을 때의 단계별 동작을, 코드 흐름과 Oracle의 자동 동작 분석으로 추적한 결과다. 측정값은 포함하지 않으며, 4절의 테스트로 검증한다.

### 3.1 단계별 흐름

T1·T2·T3 세 트랜잭션이 같은 슬롯에 거의 동시에 예약을 시도한다고 가정한다.

| 단계 | T1 | T2 | T3 |
|---|---|---|---|
| (a) 슬롯 조회 (라인 318) | `AVAILABLE`로 읽음 | `AVAILABLE`로 읽음 | `AVAILABLE`로 읽음 |
| (b) 상태 확인 (라인 326~331) | 통과 | 통과 | 통과 |
| (c) 검증 단계 (라인 339~366) | 통과 | 통과 | 통과 |
| (d) 슬롯 UPDATE (라인 370) | 행 락 획득, UPDATE 실행 | 행 락 대기 | 행 락 대기 |
| (e) 예약 INSERT (라인 379) | 인덱스에 키 등재 (미커밋) | (대기 중) | (대기 중) |
| (f) `COMMIT` | 락 해제, 키 확정 | (대기 중) | (대기 중) |
| (g) — | (완료) | 락 획득, UPDATE 실행 | (대기 중) |
| (h) — | (완료) | INSERT 시도 → 부분 유일 인덱스 위반 → `ROLLBACK` | (대기 중) |
| (i) — | (완료) | (완료) | 락 획득, UPDATE → INSERT 시도 → 위반 → `ROLLBACK` |

### 3.2 측정 대상

위 흐름에서 슬롯 행 락 동작과 예약 INSERT 시점의 거부 메커니즘이 데이터베이스 내부에서 실제로 어떻게 작동하는지는 4절 테스트에서 측정으로 확인한다. 측정 결과는 별도 의사결정 문서에서 기술 선택의 근거로 사용한다.

## 4. 테스트 맥락

### 4.1 측정 의도

**의도 1.** 추가 동시성 제어 기법이 없는 현재 코드 상태에서 비즈니스 규칙 1(슬롯당 정원 1)이 만족되는지 데이터로 확인한다. 메커니즘 분석상 만족되지만, 실제 동시 요청 환경에서 한 슬롯에 두 명이 확정되는 사건이 발생하지 않음을 실측으로 검증한다.

**의도 2.** 거부된 트랜잭션이 지불하는 검증 비용과 락 대기 시간의 절대값을 측정한다.

**의도 3.** 동시 요청 규모(N)에 따라 비용 분포가 어떻게 변하는지 측정한다. N=2와 N=20의 응답 시간 분포가 같은 형태인지, 동시성 증가에 따라 비선형으로 악화되는지 확인한다.

### 4.2 측정 항목

**항목 1. 정합성 검증 — 슬롯당 확정 예약 건수.** 동시 N개 요청 후 데이터베이스에 남은 `CONFIRMED` 예약 건수가 정확히 1건인지 확인. 2건 이상이면 비즈니스 규칙 위반.

**항목 2. 응답 시간 분포 — 성공/거부 트랜잭션 분리 측정.** 트랜잭션 진입부터 응답까지의 시간을 측정. 성공 트랜잭션(첫 트랜잭션)과 거부 트랜잭션(나머지 N-1개)을 분리. 평균, p50, p95, p99 분포.

**항목 3. 슬롯 행에 대한 락 보유 시간.** 첫 트랜잭션이 라인 370에서 락을 획득한 시각부터 트랜잭션 종료(`COMMIT` 또는 `ROLLBACK`)로 락을 해제한 시각까지의 시간. Oracle `V$LOCK`, `V$SESSION_WAIT` 동적 성능 뷰로 측정.

### 4.3 테스트 시나리오

**시나리오 T-1. 동일 슬롯 N개 동시 예약 시도.** 같은 슬롯 식별자에 대해 N개 탐색자 계정이 거의 동시에 `createReservation`을 호출. N을 2, 5, 10, 20으로 변화. 항목 1·2·3을 측정.

**시나리오 T-2. 서로 다른 슬롯 N개 동시 예약 시도 (대조군).** N개 탐색자가 서로 다른 N개 슬롯에 동시 예약을 시도. 슬롯 간 경합이 없는 환경의 응답 시간과 처리량을 측정. T-1과의 대비로 락 동작이 단독으로 만드는 비용 증가를 식별.

**시나리오 T-3. 혼합 부하 — 동일 슬롯 일부 + 다른 슬롯 일부.** 동일 슬롯 동시 요청 K건과 서로 다른 슬롯 동시 요청 M건을 혼합. 같은 슬롯 경합이 발생하는 동안 다른 슬롯의 처리량이 어떻게 변하는지 측정.

**시나리오 T-4. 재개방 직후 집중 부하 (선택적).** 한 슬롯에 N개 활성 구독을 만든 뒤 그 슬롯의 예약을 취소. 재개방 알림 발송 시점부터 N개 구독자의 동시 재예약 시도까지의 비용을 측정. 형태 2의 직접 측정이며, 형태 1과 같은 코드 경로이므로 결과가 T-1과 동일한지 확인.

## 5. 본 문서에서 다루지 않는 항목

- 동시성 제어 기법 후보 목록과 카테고리 비교
- 후보별 메커니즘 차이와 트레이드오프
- 측정 결과를 근거로 한 최종 기술 선택
- 선택의 비즈니스 근거 정리
- 시나리오 2(동일 탐색자 이중 클릭)·시나리오 3(다른 매물 슬롯의 시간 겹침)·시나리오 4(윈도우 철회와 동시 예약)의 후속 의사결정

본 문서는 비즈니스 맥락, 문제 예상 지점, 측정 항목과 시나리오 정의까지 다룬다. 측정 실행과 결과 해석, 그 위의 기술 의사결정은 후속 단계의 작업이다.
