## 5. 백엔드 측정 코드 / 인증 / 측정 도구 (이전 세션부터 유지)

### 5.1 백엔드 측정 코드 위치
`wherehouse/src/main/java/com/wherehouse/VisitReservation/service/VisitReservationWriteService.java`

`createReservation` 안 3 군데:

| 측정 지점 | 코드 위치 | 의미 |
|---|---|---|
| **CP1** | AVAILABLE 검증 통과 직후 | 본 스레드가 슬롯을 AVAILABLE 로 인지한 시점 |
| **CP2** | `slot.setStatus(RESERVED); slotRepository.save(slot)` 직후 | 본 스레드가 점유를 표시 완료 |
| **CP3** | `reservationRepository.save(reservation)` 직후 | 본 스레드가 예약 INSERT 완료 |

각 지점에서:
- `logSlotSnapshot("CP{X}-...", slotId)` — 슬롯 한 행 로그
- `logReservationsOnSlot("CP{X}-...", slotId)` — 슬롯에 묶인 예약 전체 로그

### 5.2 측정 코드의 한계 (그대로)
- **CP0 부재**: AVAILABLE 검증에서 throw 된 트랜잭션은 측정 로그에 등장 안 함 (CP1 까지 도달 못 함)
- **JPA dirty-checking**: CP2 의 시각은 "JPA save() 직후" 이지 실제 Oracle UPDATE 시점과 다를 수 있음
- **V$LOCK 미측정**: 슬롯 행 락 보유 시간은 직접 측정 안 됨 (Oracle 동적 성능 뷰 폴링 도구 미설치)

### 5.3 백엔드 인증 방식
- `JwtAuthProcessorFilter` 가 토큰을 **쿠키에서만** 추출 (`Authorization` 쿠키)
- 표준 `Authorization: Bearer <token>` HTTP **헤더는 인식 안 함**
- JMeter: HTTP Header 의 `Cookie: Authorization=${JWT_TOKEN}` 사용
- Postman: Cookie Jar 자동 사용

### 5.4 JWT 만료시간 임시 변경 (그대로 유지)
- 위치: `wherehouse/src/main/java/com/wherehouse/JWT/Filter/Util/JWTUtil.java` L22
- 현재 값: `24 * 60 * 60 * 1000` (24시간)
- 원래 값: `3600000` (1시간) — 측정 종료 후 반드시 복원

- report.py 의 시트 구성 임의 변경 — 단 사용자가 "갈아엎어도 된다" 명시 시는 OK
- `2. 학습정리_동시성제어.md` 의 내용 임의 수정 — 사용자가 학습 결과로 보존한 자료
