## 2. 4유형과 차단 위치 (이번 세션에서 정리)

| 유형 | 슬롯 읽기 / 쓰기 | 차단 위치 | 슬롯당 CONFIRMED | 상태 |
|---|---|---|---|---|
| ① baseline | findById / 무조건 UPDATE | 없음(붕괴) | **5 (측정값)** | ✅ 완료 |
| ② D 조건부 UPDATE | findById / `UPDATE…WHERE status=AVAILABLE` | **쓰기**(`affected=0`) | **1 (측정값)** | ✅ 완료 |
| ③ B 비관적 락 | `findByIdForUpdate`(SELECT FOR UPDATE) / 무조건 UPDATE | **읽기**(락 후 RESERVED 읽고 거부) | 1 (예상) | 코드 적용·**미측정** |
| ④ C 낙관적 락 | findById / 무조건 UPDATE + `@Version` | 커밋(version) | 1 (예상) | 미착수 |

**★ 핵심(꼬임 방지)**: D(조건부 UPDATE)와 B(SELECT FOR UPDATE) **둘 다 비관적 행 X-락**이다. 다른 건 **락을 거는 위치(쓰기 vs 읽기)** 뿐이다. → "비관적 락을 UPDATE에 건다"는 곧 **D와 동일**하므로, B는 **SELECT FOR UPDATE로만** D와 구별된다. (자세한 근거 §5.)

## 5. 이번 세션 핵심 결론 (★ 새 세션에서 다시 논의 금지)

1. **"비관적 락"은 두 가지로 실현된다.**
   - **조건부 UPDATE** (`UPDATE…WHERE status=AVAILABLE`): UPDATE 문 **자체가** 행 X-락을 잡는다. = **2차 D와 동일.** 차단=쓰기.
   - **SELECT FOR UPDATE**: 읽기에서 X-락. = **3차 B.** 차단=읽기. → **D와 구별되는 유일한 비관락 형태.**

2. **UPDATE 한 문장의 내부 절차(락 시점)**: ① 스냅샷 SCN 고정 → ② **WHERE로 후보 찾기 = 일관성 읽기(스냅샷, 무락·무블록)** → ③ **후보 행을 현재 모드로 집어 X-락 ← 여기서 락** → ④ SET → ⑤ commit까지 보유. 경합 시 후행은 ③에서 **대기**, 선행 commit 후 **현재값으로 재평가(write consistency / statement restart)** → `status≠AVAILABLE` → `affected=0`. (D의 거부가 이 원리.)

3. **baseline 붕괴 원인 = UPDATE가 아니라 그 앞의 별도 SELECT-검증(check-then-act)**. 5건이 각자 스냅샷에서 AVAILABLE을 읽어 검증을 다 통과했고 무조건 UPDATE가 그대로 덮어썼다. **D는 그 검증을 UPDATE의 WHERE로 합쳐** 해결.

4. **`@Lock(PESSIMISTIC_WRITE)`는 SELECT 전용**(SELECT에 FOR UPDATE 부여). `@Modifying` 벌크 UPDATE에는 **무효(no-op)**. 조건부 UPDATE는 어노테이션 없이도 이미 행 X-락을 잡는다.

5. **3차 B의 목표·관측 = D와의 직접 대비**(같은 "확정 1건"이라도 차단 지점이 다름):
   | 관측 | 2차 D (락=UPDATE) | 3차 B (락=SELECT) |
   |---|---|---|
   | 후행이 오래 기다리는 SQL | **UPDATE** 문 `took` 큼 | **SELECT** 문 `took` 큼 |
   | 후행 읽기 시점 슬롯상태(로그 CP1) | **AVAILABLE**(무락 스냅샷) | **RESERVED**(락 후 현재값) |
   | 후행 거부 방식 | `affected=0`(쓰기) | RESERVED 읽고 `if` 거부(읽기) |
   | 락 보유 구간 | UPDATE~commit(짧음) | SELECT~commit(김) |
   → 구 logSlotSnapshot 로깅을 쓰는 이유: **CP1 시점 슬롯상태(AVAILABLE↔RESERVED)** 가 D/B를 가르는 핵심인데 cp()는 그 스냅샷을 안 찍기 때문.

6. **B vs C 결정**: **(a) B = SELECT FOR UPDATE 진행** ← 현재 선택. / (b) "B 생략하고 C로 직행"도 가능했으나(D가 이미 UPDATE-락 대표) (a)를 택함.

7. **B에서 무엇을 실측할 수 있나 / 못 하나 (★ 락 블로킹 관찰의 한계)**
   - **잴 수 있다(JMeter 자동런)**: 후행 `SELECT … FOR UPDATE` 의 **`took`** 이 곧 **선행 트랜잭션이 commit될 때까지 기다린 시간**이다(P6Spy `took`은 락 대기 포함). → "선행 끝나기 전 후행 대기"는 수치로 실측됨.
   - **못 잡는다**: N=5는 ms 안에 끝나 `V$LOCK`으로 **"지금 누가 누구를 막는다"는 live 블로킹 스냅샷**은 못 잡는다(`took`·타임스탬프로 사후에만 봄). 또 "락 보유 중 UPDATE 막힘"은 B 자동런에선 안 보인다(후행이 `if`에서 거부돼 UPDATE를 시도조차 안 함 — 락이 막은 건 후행 SELECT).
   - **락 블로킹 자체를 live로 보려면 = 2세션 수동 demo(§6-B).** 이건 **앱을 안 거치는 DB-직접 관찰**이라 앱 코드·로깅과 무관. JMeter B 자동런(앱 통과·채널 ⑵·⑶)과는 **별개 관측.**
   - (참고: 개념 정당성은 형식 증명, 구현 정당성은 "결과 이력 대조"(=채널 ⑴, Jepsen식 사후 검증)로 세운다. live 락 관찰은 ②엔진 계측/②2세션 통제. "빠른 레이스 운으로 포착"은 안 쓴다.)

### 6-B. 락 블로킹 live 관찰 — 2세션 수동 demo (앱 무관 · DB 직접 · 선택)
> JMeter 자동런이 못 보는 **"락 대기/막힘"을 직접·통제하며** 본다. SQL Developer 세션 2개로 진행하며 **앱·로깅과 무관**(DB에 직접 SQL). `V$` 조회는 권한 계정(예: `system`) 필요 — 단 "멈춤(hang)" 자체는 권한 없이도 보임. 나(AI)는 DB를 직접 못 돌리니 **스크립트 제공 + 결과 해석**만 한다.
```sql
-- 0) AVAILABLE 슬롯 하나 고르기
SELECT slot_id FROM VISIT_SLOT WHERE window_id=81 AND status='AVAILABLE' AND ROWNUM=1;

-- 1) 세션1: 행 X-락 보유 (commit/rollback 하지 말 것)
SELECT slot_id, status FROM VISIT_SLOT WHERE slot_id=<SLOT> FOR UPDATE;

-- 2) 세션2: 같은 행 → 멈춤(hang)
SELECT slot_id, status FROM VISIT_SLOT WHERE slot_id=<SLOT> FOR UPDATE;                 -- "선행 끝나기 전 후행 대기"
-- 또는: UPDATE VISIT_SLOT SET status='RESERVED' WHERE slot_id=<SLOT> AND status='AVAILABLE';  -- "락 중 UPDATE 막힘"

-- 3) (권한 계정) 블로킹 관계 live 확인
SELECT sid, username, event, blocking_session, seconds_in_wait
  FROM v$session WHERE event LIKE 'enq: TX%' OR blocking_session IS NOT NULL;

-- 4) 세션1: 풀기 → 세션2 즉시 완료
COMMIT;   -- 또는 ROLLBACK
```

## 9. AI 작업 규칙 (★ 이번 세션에서 강화 — 반드시 지킬 것)
- **추론을 사실처럼 단정 금지.** 내가 직접 연 파일·사용자가 준 화면에 있는 것만 사실로 쓴다. 없는 건 **"미확인"으로 비워두거나 묻는다.** 빈 곳을 그럴듯한 값으로 메우지 않는다.
- **돌려 말하지 말 것 — 직접.** 시킨 것·확인된 것만 한다. 임의 판단/적용 금지.
