## 4. 백엔드 측정 코드 (이미 적용 완료)

### 4.1 위치

### 4.2 측정 지점 — `createReservation` 안의 3 군데

| 측정 지점 | 코드 위치 | 의미 |
|---|---|---|
| **CP1** | `if (slot.getStatus() != VisitSlotStatus.AVAILABLE)` 검증 통과 직후 | 슬롯이 예약 가능 상태로 확인됨 |
| **CP2** | `slot.setStatus(RESERVED); slotRepository.save(slot)` 직후 | 본 스레드가 슬롯을 점유 표시 완료 |
| **CP3** | `reservationRepository.save(reservation)` 직후 | 본 스레드가 예약 행 INSERT 완료 |

각 지점에서 다음 호출:
```java
logSlotSnapshot("CP{X}-...", slotId);
logReservationsOnSlot("CP{X}-...", slotId);
```

### 4.3 헬퍼 메서드 — 같은 파일 하단

- **`logSlotSnapshot(checkpoint, slotId)`** — 슬롯 한 행을 logback INFO 로 출력.
  슬롯의 windowId 로 윈도우를 추가 조회해서 `propertyId`, `leaseType` 도 함께 출력.

- **`logReservationsOnSlot(checkpoint, slotId)`** — 해당 슬롯에 묶인 예약 행을 모두 logback INFO 로 출력 (예약 수 헤더 + 행별 상세).

### 4.5 측정 코드의 한계 (현재)

**CP1 측정이 AVAILABLE 검증 통과 후에 위치** → AVAILABLE 검증에서 throw 된 트랜잭션은 측정 로그에 등장하지 않음 → **거부된 트랜잭션의 응답시간 측정 불가** (JMeter 가 외부에서 측정).

**보완 옵션 — CP0 추가**: AVAILABLE 검증 *직전* 에 측정 라인 1개 추가하면 거부된 트랜잭션도 표에 등장. 측정 코드 한 줄 추가 + 재측정 필요. 시나리오 1 측정 결과 보고 결정.

**CP1 직후 startTime 검증 존재** ([VisitReservationWriteService.java:342](../../../../wherehouse/src/main/java/com/wherehouse/VisitReservation/service/VisitReservationWriteService.java#L342)): `if (!slot.getStartTime().isAfter(now)) throw startTimeExpired`. 슬롯 시작 시각이 측정 시점에 이미 지났으면 CP1만 찍히고 CP2/CP3는 찍히지 않음 → 측정 의미 상실. **슬롯 선정 시 측정 시점 + 측정 소요 시간 + 여유 만큼 미래여야 함** (현 회차는 +3.5h 여유로 잡음).

## 6. 백엔드 인증 방식 (중요)

### 6.1 인증 방식
- `JwtAuthProcessorFilter` 가 토큰을 **쿠키에서만** 추출 (`Authorization` 이라는 이름의 쿠키)
- 표준 `Authorization: Bearer <token>` HTTP **헤더는 인식하지 않음**

### 6.2 따라서
- **JMeter**: HTTP Header 의 `Cookie: Authorization=${JWT_TOKEN}` 으로 토큰 전달
- **Postman**: Cookie Jar 가 자동으로 보관·전송 (Header 의 Cookie 줄을 두면 충돌 가능 — 두지 않음)

### 6.3 로그인 응답
`POST /login` 응답의 `Set-Cookie` 헤더에 `Authorization=<JWT>; Path=/; HttpOnly` 형태로 토큰 발급.

### 8.2 `F004_시나리오1.jmx`

**위치**: `docs/13. .../1. 테스트시나리오_1/jmeter/F004_시나리오1.jmx`

**구조**:
- **User Defined Variables**: BASE_HOST / BASE_PORT / BASE_PATH / THREADS / USERS_CSV / RESULT_JTL
- **HTTP Request Defaults**: `${BASE_HOST}:${BASE_PORT}` (`localhost:8185`)
- **HTTP Header Manager**: `Content-Type: application/json`, `Accept: application/json`, **`Cookie: Authorization=${JWT_TOKEN}`** (백엔드의 쿠키 인식에 맞춤)
- **CSV Data Set Config**: `${USERS_CSV}` 에서 `SLOT_ID, JWT_TOKEN, USER_ID` 읽음 (헤더 행 무시)
- **Thread Group** (1 개): num_threads=`${THREADS}`, ramp=0, loops=1
  - **Synchronizing Timer**: groupSize=`${THREADS}`, timeout=3000ms ← N 스레드 동시 출발 보장
  - **HTTP Request**: `POST ${BASE_PATH}/api/v1/visit/reservations`, body `{"slot_id": ${SLOT_ID}}`
  - **Response Assertion**: 응답 코드 201 또는 409 만 정상 (`^(201|409)$` regex)
  - **Simple Data Writer**: `${RESULT_JTL}` 에 요청·응답 본문 포함 저장
  - **View Results Tree**: GUI 검토용
  - **Aggregate Report**: 요약 통계

### 8.3 `F004_계정생성.postman_collection.json`

**위치**: `docs/13. .../1. 테스트시나리오_1/jmeter/F004_계정생성.postman_collection.json`

**4 폴더 / 24 요청 / 16 변수**:

| 폴더 | 요청 수 | 용도 |
|---|---|---|
| 1. 회원가입 (11개) | 11 | ownerTest1 + searcher01~10 가입 |
| 2. 로그인 (11개) — Set-Cookie 의 JWT 자동 추출 | 11 | 로그인 + Test 스크립트가 응답의 `Set-Cookie` 에서 토큰을 collection variable 에 저장 (`JWT_OWNER1`, `JWT_SEARCHER01~10`) |
| 3. 매물 등록 (ownerTest1) → PROPERTY_ID | 1 | `POST /api/v1/properties/charter`. 응답 `property_id` 자동 저장. **Header 에 Cookie 줄 없음** — Cookie Jar 자동 전송 사용 |
| 4. 방문 윈도우 공개 (ownerTest1) → 48 슬롯 | 1 | `POST /api/v1/visit/windows`. 응답 `slots[].slot_id` 자동 저장 |

**Cookie Jar 자동 전송 동작**: Postman 이 로그인 응답의 `Set-Cookie` 헤더를 받아 localhost 도메인의 cookie jar 에 저장. 같은 도메인으로 가는 후속 요청에 자동 동봉.

**HTTP Header 의 Cookie 줄은 두지 않는다** — 빈 변수치환 / 보안 필터로 충돌 가능. (직전 세션에서 401 발생 → 이 줄 제거로 해결)

**ownerTest1 토큰 갱신 주의**: cookie jar 의 `Authorization` 쿠키는 마지막 로그인 사용자 토큰으로 덮어씌워짐. **매물 등록 직전에 폴더 2 의 "01. ownerTest1 로그인" 만 한 번 더 Send** 해 두는 게 안전.

### 8.4 `users_sample_T{1,2,3}.csv`

| 컬럼 | 의미 |
|---|---|
| SLOT_ID | JMeter 가 `body: {"slot_id": ...}` 에 채울 슬롯 번호 |
| JWT_TOKEN | JMeter 가 `Cookie: Authorization=...` 에 채울 토큰 |
| USER_ID | 가독성 / 검증용 — JMeter 는 사용하지 않음 |

**모든 토큰 디코딩 검증 완료** — JWT payload 의 `userId` 가 `USER_ID` 컬럼과 일치 (15/15 ✓).
