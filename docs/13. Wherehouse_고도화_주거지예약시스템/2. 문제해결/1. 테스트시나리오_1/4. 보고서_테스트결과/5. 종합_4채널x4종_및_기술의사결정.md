# F004 슬롯 예약 동시성 제어 — 4종 비교 측정과 기술 의사결정

> **이 문서 하나로 완결된다.** 문제(비즈니스 맥락) → 후보 4종의 구현 → 같은 조건의 측정 결과(코드·데이터
> 발췌 포함) → 교차 분석 → **무엇을 왜 채택했는가**까지를 담는다. 회차별 상세는 각 보고서(부록)에 있고,
> 본 문서는 그 위에서 **의사결정의 근거**를 한 줄기로 세운다.

## 0. 한눈에 (요약)

**문제** — 단일 슬롯에 여러 사용자가 *같은 순간 동시에* 예약을 시도할 때, "한 슬롯의 확정 예약은 어느
시점에나 1건"이라는 비즈니스 불변식이 깨질 수 있다(이중 확정).

**한 일** — 동시성 제어 후보 4종을 **동일한 조건**(단일 핫슬롯·동시 5건·READ COMMITTED·DB 보호 인덱스
미적용)에서 측정하고, **4개 관측 채널**(DB 잔여·앱 단계 계측·발행 SQL·JMeter)로 교차 검증했다.

**결정** — **② 조건부 UPDATE 채택**.

| 후보 | 정합성(확정 수) | 차단 위치 | 거부 신호 | 거부 요청의 헛작업 | 채택 |
|---|---|---|---|---|---|
| ① 무제어(baseline) | **5 (붕괴)** | 없음 | 없음(전부 201) | INSERT 5건 커밋 | ✗ |
| **② 조건부 UPDATE** | 1 | 쓰기(상태조건 0행) | **409** | **없음** | **✔ 채택** |
| ③ 비관적 락(FOR UPDATE) | 1 | 읽기(락 대기) | 409 | 없음 | ✗ |
| ④ 낙관적 락(@Version) | 1 | 커밋 flush(버전 0행) | 미처리 **500** | **헛 INSERT 후 롤백** | ✗ |

**채택 근거(측정으로 증명된 것만)** — 정합성은 ②③④가 동률이므로 그것으로는 못 가른다. **②는 거부를
헛작업 없이 처리하고(④ 대비), 깔끔한 409로 돌려주며(④ 대비), 임계구간이 짧고 코드가 가장 단순하며(③
대비), "핫슬롯=경합 상시"라는 도메인 가정에 맞다.** *속도(throughput)는 근거에서 제외했다* — 본 측정
규모로는 기법 간 속도 차를 직렬화와 구분해 증명할 수 없기 때문이다(§6).

> 한 줄 서사: **단일 핫슬롯에 동시 예약이 들어오는 도메인에서 "확정 1건 + 나머지는 싸고 명확하게 거부"가
> 요구다. 조건부 UPDATE는 읽기-검증-쓰기를 한 문장으로 원자화해 경합을 쓰기 시점에 게이트하고, 거부를
> 헛작업 없이 409로 돌려주며, 구현이 가장 단순하다 — 그래서 채택했다.**

## 1. 문제 정의 — 비즈니스 맥락과 위험

**도메인.** 매물 방문 예약: 등록자가 연 방문 윈도우를 고정 길이 **슬롯**으로 나누고, 탐색자가 슬롯을
예약한다. 핵심 불변식은 **"한 슬롯의 확정(CONFIRMED) 예약은 어느 시점에나 최대 1건"** 이다.

**위험 — 확인-후-행위(check-then-act) 경합.** 예약 로직은 본질적으로 "슬롯을 *읽어* 예약 가능한지
*확인* 한 뒤, 예약으로 *전이* "한다. 동시 요청이 들어오면 이 *확인*과 *행위* 사이의 시간 간격에서 여러
요청이 모두 "예약 가능"을 보고 진입해 **이중 확정**이 발생할 수 있다(lost update / TOCTOU).

```
T1: 슬롯 읽기 → status=AVAILABLE 확인        (락 없으면 둘 다 통과)
T2: 슬롯 읽기 → status=AVAILABLE 확인
T1: RESERVED 로 전이 + 예약 INSERT
T2: RESERVED 로 전이 + 예약 INSERT           → 한 슬롯에 확정 2건 (불변식 위반)
```

이 위험을 **무엇으로, 어디서 막을 것인가** 가 본 측정의 결정 대상이다. (격리 수준은 변수로 두지 않았다 —
Oracle 기본 READ COMMITTED 고정. 같은 행 경합엔 격리 수준 상향이 해법이 아니라, 락/조건부 갱신이 해법이다.)

## 3. 후보별 구현과 측정 결과

각 후보는 `createReservation` 의 **읽기·쓰기 두 지점만** 바꿔 측정했다(나머지 경로·로깅 동일).

### 3.1 ① baseline — 무조건 UPDATE (정합성 붕괴)  · slot 1218

**코드(핵심)** — 읽은 뒤 검증하고(확인-후-행위), 상태 조건 없이 무조건 갱신한다.
```java
// 2단계: AVAILABLE 확인 (확인-후-행위의 "확인")
if (slot.getStatus() != VisitSlotStatus.AVAILABLE) {
    throw SlotUnavailableException.alreadyReserved(...);
}
// 6단계: 무조건 UPDATE — 상태 조건/락 없음
slot.setStatus(VisitSlotStatus.RESERVED);
slotRepository.save(slot);          // → UPDATE VISIT_SLOT SET STATUS='RESERVED' WHERE SLOT_ID=?
```

**측정 결과**
- **⑴ 정합성**: slot 1218 에 **확정 5건**(searcher01~05) — 붕괴.
- **⑵ 단계 분포**: 읽기 5/5 · 점유 5/5 · 등록 5/5 (전원 완주).
- **⑶ 발행 SQL** (데이터 발췌):
  ```sql
  update VISIT_SLOT set STATUS='RESERVED' where SLOT_ID=1218;      -- 상태 조건 없음 → 5건 모두 성공
  ```
  슬롯 갱신 `took` 1ms~66ms — 같은 행이라 Oracle 행 락이 **순차 실행**시키지만 *거부는 안 한다.*
- **⑷ JMeter**: **5×201 · 오류 0%**(평균 454ms).

**핵심** — 붕괴의 원인은 UPDATE가 아니라 *그 앞의 별도 SELECT-검증* 이다. 5명이 각자 스냅샷에서
AVAILABLE을 읽어 검증을 다 통과했고, 무조건 UPDATE는 마지막 값만 남기는 게 아니라 **5건의 INSERT를
모두 커밋**시킨다. 그리고 이 붕괴는 **외부 응답엔 오류 0%로 보이는 "조용한 붕괴"** 라, 응답 코드만 보는
모니터링으로는 못 잡는다. → **동시성 제어가 반드시 필요하다.**

### 3.2 ② 조건부 UPDATE — 쓰기에서 차단 (채택)  · slot 1221

**코드(핵심)** — 사전 `if` 검증을 없애고, **상태 조건을 갱신문 WHERE 에 합쳐** 게이트를 원자화한다.
```java
// 사전 AVAILABLE 검증 없음 — 게이트는 조건부 UPDATE 가 담당
int affected = slotRepository.reserveSlotIfAvailable(slotId);
if (affected == 0) {                       // race 패배 → 거부 (예약 INSERT 도달 안 함)
    throw SlotUnavailableException.alreadyReserved(...);
}
```
```java
// VisitSlotRepository
@Modifying
@Query("UPDATE VisitSlotEntity s SET s.status = ...RESERVED " +
       "WHERE s.slotId = :slotId AND s.status = ...AVAILABLE")
int reserveSlotIfAvailable(@Param("slotId") Long slotId);
```

**측정 결과**
- **⑴ 정합성**: slot 1221 에 **확정 1건**(searcher03).
- **⑵ 단계 분포**: 읽기 5/5 · **점유 1/5** · 등록 1/5 → **읽기와 점유 사이에서 4건 끊김 = 쓰기 단계 차단.**
- **⑶ 발행 SQL** (데이터 발췌):
  ```sql
  update VISIT_SLOT set STATUS='RESERVED' where SLOT_ID=1221 and STATUS='AVAILABLE';
  ```
  | 트랜잭션 | 조건부 UPDATE | `took` | 예약 INSERT | 종료 |
  |---|---|---|---|---|
  | searcher03 (성공) | **1행 갱신** | 0ms | 실행 | 커밋 |
  | 나머지 4건 (거부) | **0행** | 24ms | **없음** | 롤백 |
- **⑷ JMeter**: **1×201 + 4×409 · 오류 80%**(평균 342ms).

**핵심** — `AND STATUS='AVAILABLE'` 한 조건이 1차와의 결정적 차이다. 한 요청이 RESERVED로 바꿔 커밋하면,
나머지의 같은 갱신은 *AVAILABLE 인 행을 못 찾아 0행* 이 된다(Oracle write consistency: 행 락 대기 후
현재값으로 재평가 → 조건 불일치 → 0행). **거부 요청은 갱신이 0행이라 예약 INSERT에 도달하지 않는다 =
헛작업 없음.** 락 보유 구간도 `갱신~커밋` 으로 짧다.

### 3.3 ③ 비관적 락 — 읽기에서 차단  · slot 1237

**코드(핵심)** — 읽기를 `SELECT … FOR UPDATE` 로 바꿔 **읽는 순간 행 X-락**을 선점한다.
```java
// 1단계: 읽기에서 행 X-락 (후착은 락 대기 후 RESERVED 를 읽는다)
VisitSlotEntity slot = slotRepository.findByIdForUpdate(slotId).orElseThrow(...);
// 2단계: 후착은 RESERVED 를 보고 여기서 거부
if (slot.getStatus() != VisitSlotStatus.AVAILABLE) {
    throw SlotUnavailableException.alreadyReserved(...);
}
```
```java
// VisitSlotRepository
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT s FROM VisitSlotEntity s WHERE s.slotId = :slotId")
Optional<VisitSlotEntity> findByIdForUpdate(@Param("slotId") Long slotId);
```

**측정 결과**
- **⑴ 정합성**: slot 1237 에 **확정 1건**(searcher05, resId 101).
- **⑵ 단계 분포**: 읽기 5/5 · 점유 1/5 · 등록 1/5. 단 **후착의 "읽기 시점 슬롯 상태 = RESERVED"** — ②④와
  결정적으로 다른 지점(②④는 후착도 AVAILABLE을 읽음).
- **⑶ 발행 SQL** (데이터 발췌) — 거부를 가르는 문장은 *읽기* 다:
  ```sql
  select ... from VISIT_SLOT where SLOT_ID=1237 for update;     -- 읽기에서 락
  update VISIT_SLOT set STATUS='RESERVED' where SLOT_ID=1237;   -- 무조건(락은 위에서 보유)
  ```
  | 트랜잭션 | `SELECT … FOR UPDATE` 의 `took` | 결과 |
  |---|---|---|
  | searcher05 (성공) | **2ms** (즉시 획득) | AVAILABLE 읽음 → 점유·등록 → 커밋 |
  | 나머지 4건 (거부) | **73 / 97 / 112 / 128ms** | (대기 후) RESERVED 읽음 → `if` 거부 → 롤백 |
- **⑷ JMeter**: 1×201 + 4×409(거부 코드는 거부 경로에서 도출). **응답시간 수치는 미확인**(해당 회차 화면 미수집).
- **보강 관측**: SQL Developer **2세션 수동 재현**으로 `enq: TX - row lock contention`(행 락 대기)을 직접
  관측, 후착이 풀린 뒤 선착의 커밋값을 읽음을 확인(근거 문서 §부록).

**핵심** — 락을 *읽기* 로 당겨 검증과 행위 사이의 race 자체를 없앤다. 거부 요청은 헛 INSERT가 없다(②와
동일). 단 **락을 `읽기~커밋` 전 구간 보유**하므로 임계구간이 가장 길다(후착 대기 73~128ms vs ②의 ~24ms).

### 3.4 ④ 낙관적 락 — 커밋에서 차단  · slot 1170

**코드(핵심)** — 락을 걸지 않는다. 엔티티에 `@Version` 을 두면 Hibernate가 갱신문에 버전 조건을 자동으로 붙인다.
```java
// 1단계: 무락 findById (5명 모두 같은 version 으로 읽음)
VisitSlotEntity slot = slotRepository.findById(slotId).orElseThrow(...);
if (slot.getStatus() != VisitSlotStatus.AVAILABLE) { throw ...; }  // 5명 모두 통과(무락 스냅샷)
...
slot.setStatus(VisitSlotStatus.RESERVED);
slotRepository.save(slot);   // 커밋 flush 시: UPDATE … SET VERSION=VERSION+1 WHERE SLOT_ID=? AND VERSION=?
```
```java
// VisitSlotEntity
@Version
@Column(name = "VERSION", nullable = false)
private Long version;
```

**측정 결과**
- **⑴ 정합성**: slot 1170 에 **확정 1건**(searcher03, resId 125).
- **⑵ 단계 분포**: 읽기 5/5 · **점유 5/5** · 등록 1/5 — **락이 없어 5명 모두 점유·예약 INSERT까지 도달**한
  뒤 커밋에서 4건이 무너진다. (그래서 ④의 정합성/거부는 단계 분포가 아니라 **DB 잔여**로 판정.)
- **⑶ 발행 SQL** (데이터 발췌):
  ```sql
  -- 5명 모두 예약 INSERT(헛 INSERT) 후, 점유 UPDATE:
  update VISIT_SLOT set STATUS='RESERVED', VERSION=1 where SLOT_ID=1170 and VERSION=0;
  ```
  | 트랜잭션 | 버전조건 UPDATE `took` | 갱신 행수 | 예약 INSERT | 종료 |
  |---|---|---|---|---|
  | searcher03 (성공) | 0ms | **1행** (VERSION 0→1) | res 125 | 커밋 |
  | 나머지 4건 (거부) | **27ms** | **0행** | res 123·124·126·127 → **롤백** | 롤백 |
  거부 4건은 모두 예외:
  ```
  org.hibernate.StaleObjectStateException: Row was updated or deleted by another transaction
    … : [com.wherehouse.VisitReservation.entity.VisitSlotEntity#1170]
  ```
- **⑷ JMeter**: **1×201 + 4×500 · 오류 80%**(평균 389ms). — `OptimisticLockException` 이 **409로 매핑되지
  않아 미처리 500(Internal Server Error)** 으로 나간다.

**핵심** — ④는 ②와 메커니즘이 형제다(둘 다 *조건부 UPDATE 가 행 락 대기 후 0행*; 후착 `took` 27ms ≈ ②의
24ms). 차이는 **조건이 버전(④)이냐 상태(②)냐**, 그리고 **0행이 미처리 500(④)이냐 409(②)냐**, 그리고
**거부 요청이 헛 INSERT를 하느냐(④, 함)** 다.

## 4. 교차 분석 — 정합성은 동률, "거부 처리의 구조"로 비교

**①은 탈락이 자명**(확정 5건 붕괴). 나머지 ②③④는 **정합성이 모두 확정 1건으로 동률** 이므로, 정합성으로는
가를 수 없다. 가르는 축은 **거부되는 요청을 어디서·어떤 비용으로·어떤 신호로 처리하는가** 다.

| 축 | ② 조건부 UPDATE | ③ 비관적 락 | ④ 낙관적 락 |
|---|---|---|---|
| 차단 위치 | 쓰기(상태 0행) | **읽기**(FOR UPDATE 대기) | 커밋 flush(버전 0행) |
| 락 보유 구간(임계구간) | 쓰기~커밋(짧음) | **읽기~커밋(김)** | 쓰기~커밋(짧음) |
| 거부 후행 대기(측정 `took`) | ~24ms | **73~128ms** | ~27ms |
| 거부 요청 헛작업 | **없음** | 없음 | **헛 예약 INSERT 후 롤백** |
| 거부 신호 | `affected=0` → **409** | RESERVED 읽고 → **409** | `OptimisticLockException` → **미처리 500** |
| 후행이 읽는 슬롯 상태 | AVAILABLE(무락) | **RESERVED**(락 후 현재값) | AVAILABLE(무락) |
| 구현 비용 | `WHERE` 조건 1줄 | `@Lock`/FOR UPDATE | `@Version` 컬럼+DDL+**예외처리 필요** |

```
차단 위치가 기법마다 이동한다:
  ① 없음(붕괴) ─ ② 쓰기 ─ ③ 읽기 ─ ④ 커밋
정합성(확정 1)은 ②③④ 동률. 갈리는 건 "거부의 비용·신호·시점"이다.
```

- **②와 ④는 형제** — 둘 다 *조건부 UPDATE가 행 락 대기 후 0행*. 차이는 조건(상태 vs 버전)·신호(409 vs 500)·헛작업(없음 vs 있음).
- **③만 읽기에서 막아** 후행이 헛 INSERT를 안 한다(②와 동일 장점). 대신 **락을 전 구간 쥐어 임계구간이 가장 길다.**
- **④는 후행이 헛 INSERT까지 하고 롤백** → 거부 비용 최대 + 신호가 500이라 **반드시 예외처리(409 매핑/재시도)를 추가**해야 한다.

## 5. 기술 의사결정 — ② 조건부 UPDATE 채택

정합성이 동률이므로, 선택은 위 §4 표의 **거부 처리 구조**에서 갈린다.

**② vs ④ (낙관적 락을 택하지 않은 이유)**
- ④의 거부 요청은 *헛 예약 INSERT + 롤백* 을 한다(타임라인에 res 123·124·126·127 INSERT 후 롤백). ②는
  갱신 0행에서 끊겨 그 INSERT를 안 한다 → **거부 비용이 ②가 더 낮다.**
- ④의 거부 신호는 `OptimisticLockException` → **미처리 500**. 깔끔한 409로 만들려면 예외처리·재시도 코드를
  더해야 한다. ②는 `affected=0` → 그대로 **409**.
- ④는 `@Version` 컬럼·DDL·예외처리가 붙는다. ②는 `WHERE` 한 줄.
- **도메인 가정**: ④(낙관적 락)의 전제는 "충돌이 드물다" 인데, **단일 핫슬롯 동시 예약은 경합이 상시** 라
  그 전제가 깨진다. ②(쓰기 시점 원자적 게이트)는 이 도메인에 직접 맞다.

**② vs ③ (비관적 락 대비)**
- 둘 다 거부에 헛작업이 없고 거부 신호가 409로 같다. 차이는 ③이 락을 *읽기부터 커밋까지* 쥐어 **임계구간이
  가장 길고**(측정 `took` 73~128ms vs ②의 ~24ms), `@Lock` 을 요구한다는 점이다.
- 즉 **②는 ③과 같은 거부 품질(헛작업 0 · 409)을, 더 짧은 임계구간과 더 단순한 코드로** 얻는다.

**종합** — ②는 **거부 비용 0 + 깔끔한 409 + 가장 단순한 구현 + 도메인 적합** 을 동시에 만족하는 지점이다.
④는 거부 비용·신호·복잡도에서 불리하고, ③은 임계구간·복잡도에서 ②보다 무겁다. → **② 조건부 UPDATE 채택.**

> ★ **속도(throughput)는 채택 근거에 넣지 않았다.** 본 규모(동시 5건)에서 기법 간 응답시간 차이는 *세 기법
> 공통의 직렬화(락 대기)* 에 묻혀 **기법 차이로 증명할 수 없다**(§6). ②를 택한 이유는 "더 빠르다"가 아니라
> "거부를 가장 싸고 깨끗하고 단순하게 처리하며 도메인에 맞다"이다. — *근거를 측정이 실제로 증명한 범위 안에
> 둔다.*

## 6. 측정의 한계와 적용 범위 (정직한 명시)

결정의 신뢰성을 위해, 이 측정이 **증명한 것과 증명하지 못한 것**을 분명히 한다.

- **의도된 최악 케이스만 측정.** N=5 동일 핫슬롯 동시 출발은 경합을 *강제로 100% 재현* 한 것이다. 실제
  서비스에서 같은 슬롯에 *진짜 동시* 요청이 얼마나 잦은지(경합 빈도)는 **프로덕션 트래픽이라야** 안다.
- **기법 간 throughput 비교 불가.** 세 기법 모두 지배 요인이 *같은 직렬화* 이고, 그 위의 기법별 차이(④의
  헛 INSERT ~ms, ③의 긴 임계구간)는 이 규모에서 응답시간 노이즈에 묻힌다. **그래서 속도를 결정 근거로 쓰지
  않았다.** (회차마다 슬롯·런이 달라 절대 수치 cross-비교도 *방향성* 으로만 유효하다.)
- **단독 기법만 측정.** 부분 유일 인덱스(DB 백스톱)를 끈 상태로 코드 기법 단독 거동만 봤다. 백스톱을 켰을
  때의 거동(`ORA-00001` 거부 경로)은 미측정.
- 따라서 본 결정은 **"정합성 + 거부 처리의 구조"** 라는, 이 측정이 실제로 증명한 범위 위에 세운 것이다.
  *throughput 우열, 백스톱 병행 효과는 별도 검증 과제로 남긴다.*

## 7. 결론 및 후속 작업

**결론** — 슬롯 예약 동시성의 1차 방어선으로 **② 조건부 UPDATE** 를 채택한다. 읽기-검증-쓰기를 한 갱신문으로
원자화해 경합을 *쓰기 시점* 에 게이트하고, 거부를 *헛작업 없이 409* 로 돌려주며, *구현이 가장 단순* 하고
*핫슬롯 도메인 가정에 맞다.* 이 결정은 속도가 아니라 **거부 처리 구조** 라는 측정 가능한 근거에 둔다.

**후속 작업(설계 반영 — 별도 진행)**
- **코드 반영**: 현재 `createReservation` 은 마지막 측정 회차인 **④(낙관적 락)** 형태다 → **②(조건부 UPDATE)**
  형태로 되돌린다(`5. 테스트유형별_계측코드/2_D_조건부UPDATE.java` 기준; `reserveSlotIfAvailable` 활성화).
- **DB 백스톱 병행 여부(직교 결정)**: 코드 게이트(②)와 별개로 부분 유일 인덱스(`기타/SQL/03_constraints.sql`)
  를 켜 DB에서 한 번 더 막을지 — 코드 버그·다른 경로 INSERT에 대한 2차 방어선.
- **측정용 임시 변경 복원**: `@Version` 컬럼/필드 제거, P6Spy·단계 로깅 정리, JWT 만료·스케줄러 주기 원복.
