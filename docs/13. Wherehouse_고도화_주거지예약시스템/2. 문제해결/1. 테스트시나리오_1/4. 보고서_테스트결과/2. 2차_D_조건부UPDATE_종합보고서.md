# F004 — 방문 예약 슬롯 동시성 제어 전략 결정 · 2차: 조건부 UPDATE (채택)

> 본 문서는 "방문 예약 슬롯에 어떤 동시성 제어를 둘 것인가"라는 설계 결정에서 **후보 ② 조건부
> UPDATE** 를 평가한 기록이다. 읽기-검증-쓰기를 **하나의 갱신문**(`UPDATE … WHERE status='AVAILABLE'`)
> 으로 원자화해 경합을 *쓰기 시점*에 게이트하는 방식이며, **4종 비교의 최종 채택안**이다.

---

## 1. 설계 질문과 판단 기준

### 1-1. 설계 질문과 이 후보의 위치

해결할 핵심 설계 질문은 1차와 동일하다 — 한 슬롯에 여러 탐색자가 *같은 순간* 예약을 시도할 때
**정확히 하나의 확정만** 남기고 나머지를 정상 거부로 처리하되, 그 책임을 **어느 계층**에 둘 것인가.

1차(무제어)는 이 책임을 어디에도 두지 않아 한 슬롯에 확정 5건이 남고, 그 붕괴가 외부 응답에는 전혀
드러나지 않음을 실증했다. 본 회차의 후보 ②는 그 책임을 **갱신문 자체**에 둔다.
사전 `if` 검증을 제거하고 상태 조건을 `UPDATE … WHERE SLOT_ID=? AND STATUS='AVAILABLE'` 의
`WHERE` 절에 합쳐, **"확인"과 "행위"를 한 문장 안에서 원자적으로 묶는다.** 검증과 갱신 사이의 race
구간 자체가 사라지는 것이 1차와의 결정적 차이다.

### 1-2. 비즈니스 불변식

보장해야 하는 불변식은 **"한 슬롯의 확정 예약은 어느 시점에나 최대 1건"** 이다. 위반 시 한 자리에
두 사람이 배정되어 예약 결과를 신뢰할 수 없게 된다.

이 불변식의 짝이 되는 조건이 후보 평가의 핵심이다. **서로 다른 슬롯 예약 사이에는 순서 의존성이
없으므로**, 올바른 제어는 "모든 예약 직렬화"가 아니라 **같은 슬롯 충돌 범위에서만 직렬화**하는
것이다. 조건부 UPDATE 는 같은 슬롯 행에 대해서만 행 락 경합을 일으키므로, **기술적 직렬화 범위가
비즈니스 충돌 범위(같은 슬롯)와 정확히 일치**한다.

### 1-3. 비교 기준과 결론

| 기준 | 의미 | ② 조건부 UPDATE 결과 |
|---|---|---|
| 정합성 | 확정이 정확히 1건으로 유지되는가 | **확정 1건 (회복)** |
| 거부 신호 | 거부가 의미 있는 신호로 드러나는가 | **409 Conflict** |
| 거부 비용 | 거부가 헛작업(불필요 INSERT·롤백)을 치르는가 | **없음** (예약 INSERT 미도달) |
| 차단 위치 | 거부가 어느 단계에서 일어나는가 | **쓰기(점유)** — 갱신 0행 |

**결론부터 말하면, 조건부 UPDATE 는 네 축 모두에서 양호하며 4종 중 최종 채택안이다.** 정합성을 확정
1건으로 회복하고, 거부 4건은 **예약 INSERT 에 도달하기 전 쓰기 단계에서 "갱신 0행"으로 차단**되어
헛작업이 없으며, 클라이언트에는 409 로 명확히 드러난다. 임계구간(락 보유)은 `갱신~커밋`으로 짧고,
구현은 `WHERE` 조건 한 줄이다. 비관적 락(③)·낙관적 락(④) 대비 우위의 상세 근거는 §5 및 상위 종합
문서(`5. 테스트결과서.md`)에 둔다.

## 3. 측정 결과

### 3-1. 정합성 — DB 잔여 (채널 ⑴)

slot 1221 에 **확정 예약 1건**(searcher03, res 87) 만 남았다. 슬롯 상태는 `RESERVED`.

| RESERVATION_ID | SEARCHER_USER_ID | SLOT_ID | STATUS | CONFIRMED_AT |
|---|---|---|---|---|
| 87 | searcher03 | 1221 | CONFIRMED | 2026-06-02 15:04:24.318 |

→ **확정 1건 = 비즈니스 불변식 충족.** (1차의 5건 붕괴와 대비)

### 3-2. 단계 분포 — 앱 계측 (채널 ⑵)

| 단계 | 통과 | 의미 |
|---|---|---|
| CP0 트랜잭션 진입 | 5 / 5 | 5명 모두 진입 |
| CP1 슬롯 읽기 (무락 조회) | **5 / 5** | 5명 모두 읽고 통과 |
| CP2 슬롯 점유 (조건부 UPDATE 성공) | **1 / 5** | 1건만 점유 성공 |
| CP3 예약 등록 (INSERT) | **1 / 5** | 점유 성공자만 등록 |
| 거부 | **4** | 모두 **쓰기(점유) 단계**에서 차단 (`WRITE_RACE @ WRITE`) |

| 요청 | 경로 | 결과 | 내부 처리(ms) |
|---|---|---|---|
| searcher03 (exec-2, 성공) | 진입 → 읽기 → 점유 → 등록 | **확정** (res 87) | 51.88 |
| searcher05·02·01·04 (거부) | 진입 → 읽기 → **(점유 단계 차단)** | 거부(409) | 66.6 ~ 66.71 |

5명 모두 슬롯 읽기까지 도달하지만, **읽기와 점유 사이에서 4건이 끊긴다 = 쓰기(점유) 단계 차단.**
거부가 성공보다 ~15ms 느린 것은, 거부 요청의 갱신문이 성공 요청의 행 락이 풀릴 때까지 대기했기
때문이다.

### 3-3. 발행 SQL — P6Spy (채널 ⑶)

결과를 가르는 두 문장은 다음이다.

```sql
-- 슬롯 점유: 상태 조건(AND STATUS='AVAILABLE')을 붙인 조건부 갱신 (5건 모두 시도)
update VISIT_SLOT vse1_0 set vse1_0.STATUS='RESERVED'
  where vse1_0.SLOT_ID=1221 and vse1_0.STATUS='AVAILABLE';

-- 예약 등록: 갱신에 성공한 searcher03 1건만 실행
insert into VISIT_RESERVATION (CONFIRMED_AT,SEARCHER_USER_ID,SLOT_ID,STATUS,RESERVATION_ID)
  values (TIMESTAMP '2026-06-02 15:04:24.318','searcher03',1221,'CONFIRMED',87);
```

| searcher (스레드) | 조건부 UPDATE (took / 행수) | 예약 INSERT | 종료 |
|---|---|---|---|
| searcher03 (exec-2, 성공) | **0ms / 1행** | 실행 (res 87) | 커밋 |
| searcher05 (exec-1, 거부) | 24ms / **0행** | 없음 | 롤백 |
| searcher02 (exec-6, 거부) | 24ms / **0행** | 없음 | 롤백 |
| searcher01 (exec-3, 거부) | 24ms / **0행** | 없음 | 롤백 |
| searcher04 (exec-4, 거부) | 24ms / **0행** | 없음 | 롤백 |

`WHERE` 절의 **`AND STATUS='AVAILABLE'`** 한 조건이 1차와의 결정적 차이다. 선착(searcher03)이 슬롯을
`RESERVED` 로 바꿔 커밋하면, 나머지 요청의 같은 조건부 갱신은 "상태가 AVAILABLE 인 행"을 찾지 못해
**갱신 대상 0행**이 된다(Oracle write consistency: 행 락 대기 후 현재값으로 재평가 → 조건 불일치 →
0행). 이것이 거부의 직접 원인이다. 거부 요청의 `took=24ms` 는 선착의 행 잠금이 풀릴 때까지 기다린
시간이며, **읽기-검증-쓰기가 한 문장에 묶여 있어 대기 비용이 갱신문의 `took` 에 찍힌다.** 거부 요청은
갱신이 0행이라 **예약 INSERT 에 도달하지 않는다 = 헛작업 없음.** 락 보유 구간도 `갱신~커밋` 으로 짧다.

### 3-4. 외부 결과 — JMeter (채널 ⑷)

| 항목 | 값 |
|---|---|
| 표본 / 오류 | **1건 201 + 4건 409 Conflict / 오류율 80%** |
| 응답시간(평균/중앙/최대) | 342 / 343 / 344 ms |
| 처리량 | 14.5 req/s |

거부 4건이 **409(충돌)** 로 클라이언트에 명확히 드러난다. (1차의 "5건 모두 201, 오류 0%"와 정반대 —
붕괴가 외부에서 보이도록 바뀌었다.)

## 4. 메커니즘 — 한 문장으로 race를 닫는다

```
5명 동시 요청 (READ COMMITTED)
  ├ 5명 모두 슬롯 읽기 통과 (무락 조회)
  ├ UPDATE … WHERE SLOT_ID=1221 AND STATUS='AVAILABLE'   ← 슬롯 행 락에서 순차 실행
  │     1건 : 1행 갱신 성공 → 예약 등록 → 커밋
  │     4건 : (락 대기 후) 갱신 0행 → 예약 등록 안 함 → 롤백
  └ 결과: 한 슬롯에 확정 예약 1건
```

상태 조건이 붙은 갱신문이 "확인"과 "행위"를 한 문장 안에서 원자적으로 묶는다. 1차에서 race 의 원인
이었던 "확인~행위 시간 간격"이 사라진다. 거부 요청은 예약 등록에 도달하지 못하므로 **헛 INSERT 가
없고**, 락 보유 구간도 `갱신~커밋` 으로 짧다.

## 5. 결론 — 채택안으로서의 위치

조건부 UPDATE 의 결론은 다음이다.

> 조건부 UPDATE 는 읽기-검증-쓰기를 **하나의 갱신문으로 원자화**하여 경합을 *쓰기 시점*에 게이트한다.
> 정합성을 확정 1건으로 회복하고, 거부를 **헛작업 없이 409로** 돌려주며, 기술적 직렬화 범위(같은 슬롯
> 행)가 비즈니스 충돌 범위(같은 슬롯)와 일치한다. 구현은 `WHERE` 조건 한 줄이다.

이 회차는 §1-3 네 축 모두에서 양호하므로 채택 후보가 된다. 다만 정합성(확정 1건)은 이후 ③·④도
동일하게 달성하므로, **정합성만으로는 채택을 정당화할 수 없다.** 채택의 진짜 근거는 "거부 처리의
구조" — 헛작업 0 · 409 신호 · 짧은 임계구간 · 도메인 부합 — 이며, 이는 ③·④와의 직접 대비로만
증명된다.

- 다음 회차(③ 비관적 락): 같은 "확정 1건"을 만들되 **차단 위치가 "쓰기"에서 "읽기"로 앞당겨지는지**,
  그리고 그 대기 비용이 본 회차의 24ms 와 어떻게 달라지는지를 대비 측정한다.
- 최종 채택 비교(②↔③↔④ 4축 종합): `5. 테스트결과서.md` §6~§7.
