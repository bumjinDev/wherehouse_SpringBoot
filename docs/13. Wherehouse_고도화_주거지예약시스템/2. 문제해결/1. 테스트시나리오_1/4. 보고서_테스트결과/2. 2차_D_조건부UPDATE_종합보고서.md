# F004 슬롯 예약 동시성 측정 — 2차: 조건부 UPDATE

> 단일 슬롯에 5명이 동시에 예약을 시도할 때, **갱신문에 상태 조건을 붙인 조건부 UPDATE**
> (`UPDATE … WHERE … AND STATUS='AVAILABLE'`) 하나만으로 동시성을 제어한 결과를 4개 관측 축으로
> 측정한다. 1차 기준선의 "조용한 붕괴"와 대비된다.
> **결과 한 줄: 한 슬롯에 확정 예약이 1건만 남는다(정합성 회복). 거부된 4개 요청은 예약 등록에
> 도달하지 못하고 갱신 단계에서 차단되며, 클라이언트에는 409(충돌)로 명확히 드러난다.**

---

## 1. 측정 목적
조건부 UPDATE를 유일한 동시성 게이트로 두었을 때 ⑴ 한 슬롯의 확정 예약이 1건으로 유지되는지(정합성),
⑵ 거부되는 요청이 **어느 단계에서** 차단되는지, ⑶ 거부되는 요청이 치르는 **비용**(불필요한 등록 여부·
롤백)을 실측한다.

## 2. 측정 환경
| 항목 | 값 |
|---|---|
| 대상 로직 | 슬롯 예약 (`createReservation`) |
| 동시성 제어 | **조건부 UPDATE** — `UPDATE … SET RESERVED WHERE SLOT_ID=? AND STATUS='AVAILABLE'` |
| 사전 검증 | 별도의 "예약 가능?" 사전 확인 없음(상태 게이트를 조건부 UPDATE가 원자적으로 담당) |
| DB 보호 장치 | 부분 유일 인덱스 미적용(차단을 *제어 기법 자체*에 귀속) |
| 트랜잭션 격리 수준 | READ COMMITTED |
| 부하 | 5명 동시 요청(동시 출발 보장), searcher01~05 |
| 대상 슬롯 | window 81 · **slot 1221** (방문 2026-06-09 15:05~15:35) |
| 실행 시각 | 2026-06-02 15:04:24 |

## 3. 측정 결과

### 3.1 정합성 — 데이터베이스 잔여 (`VISIT_RESERVATION`)
- slot 1221 에 **확정 예약 1건**(searcher03) 만 남았다. 슬롯 상태는 `RESERVED`.
- **→ 한 슬롯에 확정 1건 = 비즈니스 규칙 충족.** (1차의 5건 붕괴와 대비)

### 3.2 단계별 통과 분포 — 애플리케이션 계측
`F004_측정결과_D_조건부UPDATE.xlsx` 의 요약·요청별 시트 내용:

| 단계 | 통과 | 의미 |
|---|---|---|
| 트랜잭션 진입 | 5 / 5 | 5명 모두 진입 |
| 슬롯 읽기 (잠금 없는 조회) | **5 / 5** | 5명 모두 슬롯을 읽고 통과 |
| 슬롯 점유 (조건부 UPDATE 성공) | **1 / 5** | 1건만 점유 성공 |
| 예약 등록 (예약 행 INSERT) | **1 / 5** | 점유에 성공한 요청만 등록 |
| 거부 | **4** | 모두 **"쓰기(점유) 단계"** 에서 차단 |

| 요청 | 경로 | 결과 | 내부 처리시간 |
|---|---|---|---|
| searcher03 (성공) | 진입 → 슬롯 읽기 → 슬롯 점유 → 예약 등록 | **확정** | 51.9ms |
| searcher01·02·04·05 (거부) | 진입 → 슬롯 읽기 → **(점유 단계에서 차단)** | 거부(409) | ~66.6ms |

- 5명 모두 슬롯 읽기까지는 도달하지만, **읽기와 점유 사이에서 4개 요청이 끊긴다 = 쓰기(점유) 단계 차단.**
- 거부된 요청이 성공 요청보다 ~15ms 느린 이유는, 거부된 요청의 갱신문이 성공 요청의 행 잠금이 풀릴
  때까지 대기했기 때문이다.

### 3.3 실제 발행된 SQL (Oracle 실행형) — `2_D_조건부UPDATE_타임라인.xlsx`
동시성 결과를 가르는 **두 문장**:

```sql
-- ① 슬롯 점유: 상태 조건(AND STATUS='AVAILABLE')을 붙인 조건부 갱신 (5개 트랜잭션이 모두 시도)
update VISIT_SLOT set STATUS='RESERVED' where SLOT_ID=1221 and STATUS='AVAILABLE';

-- ② 예약 등록: 갱신에 성공한 요청(searcher03) 1개만 실행
insert into VISIT_RESERVATION (CONFIRMED_AT, SEARCHER_USER_ID, SLOT_ID, STATUS, RESERVATION_ID)
values (TIMESTAMP '2026-06-02 15:04:24.318', 'searcher03', 1221, 'CONFIRMED', <시퀀스값>);
```

발행 SQL 기준 트랜잭션별 결과:

| 트랜잭션 | 조건부 UPDATE | 실행 소요(took) | 예약 등록 INSERT | 트랜잭션 종료 |
|---|---|---|---|---|
| searcher03 (성공) | **1행 갱신 성공** | 0ms | **실행** | 커밋 |
| searcher01·02·04·05 (거부) | **0행 (대상 없음)** | 24ms | **없음** | 롤백 |

**해석(SQL을 처음 보는 경우):**
- ①의 `WHERE` 절에 붙은 **`AND STATUS='AVAILABLE'`** 이 1차와의 유일하고 결정적인 차이다.
  한 요청이 슬롯을 `RESERVED` 로 바꾸고 커밋하면, 나머지 요청이 실행하는 **같은 조건부 갱신은
  "상태가 AVAILABLE 인 행"을 찾지 못해 갱신 대상이 0행**이 된다 — 이것이 거부의 직접 원인이다.
- 거부된 요청의 `took=24ms` 는 성공 요청의 행 잠금이 풀릴 때까지 기다린 시간이다. 잠금이 풀린 뒤
  조건을 다시 보니 이미 `RESERVED` → 0행.
- 거부된 요청은 갱신이 0행이므로 **예약 등록 INSERT에 도달하지 않는다** — 확정 예약이 누적되지 않는다.
  (이후 거부 응답에 담을 "같은 매물의 다른 가용 슬롯 목록"을 조회한 뒤 롤백한다.)
- 즉 **읽기-검증-쓰기를 한 문장으로 원자화**하여, 검증 통과 후 발생하는 순수 경합을 갱신 단계에서 차단한다.

> ※ 위 SQL은 타임라인 시트의 "Oracle 실행형" 내용만 옮긴 것이다. 트랜잭션 경계 로그와 원본 바인딩
> SQL 로그는 보고서 범위에서 제외했다(트랜잭션 종료 결과는 위 표의 "커밋/롤백"으로만 표기).

### 3.4 외부 부하 결과 — JMeter
| 항목 | 값 |
|---|---|
| 표본 / 오류 | **1건 201 Created + 4건 409 Conflict / 오류율 80%** |
| 응답시간(평균/중앙/최대) | 342 / 343 / 344 ms |
| 처리량 | 14.5 req/s |

- 거부된 4건이 **409(충돌)** 로 클라이언트에 명확히 드러난다. (1차의 "5건 모두 201, 오류 0%"와 정반대.)

## 4. 메커니즘 해석
```
5명 동시 요청 (READ COMMITTED)
  ├ 5명 모두 슬롯 읽기 통과 (잠금 없는 조회)
  ├ UPDATE … WHERE SLOT_ID=1221 AND STATUS='AVAILABLE'   ← 슬롯 행 잠금에서 순차 실행
  │     1건 : 1행 갱신 성공 → 예약 등록 → 커밋
  │     4건 : (잠금 대기 후) 갱신 0행 → 예약 등록 안 함 → 롤백
  └ 결과: 한 슬롯에 확정 예약 1건
```
상태 조건이 붙은 갱신문이 "확인"과 "행위"를 한 문장 안에서 원자적으로 묶는다. 거부된 요청은 예약
등록에 도달하지 못하므로 **불필요한 등록(헛 INSERT)이 없고**, 잠금 보유 구간도 `갱신~커밋` 으로 짧다.

## 5. 결론
- **조건부 UPDATE 단독으로 단일 슬롯의 정합성 게이트를 통과한다(확정 1건).** 거부된 요청은 쓰기(점유)
  단계에서 "갱신 대상 0행"으로 차단되어 예약 등록 없이 롤백 → 거부 처리 비용이 낮다.
- 1차와 외부 응답을 비교하면(5×201 → 1×201+4×409) 위반이 **클라이언트에 드러나도록** 바뀌었고,
  내부 관측(3.2·3.3)으로 **차단 지점이 "쓰기(점유) 단계"** 임이 확인된다.
- → 조건부 UPDATE는 정합성·거부 처리 비용 측면에서 유효한 후보다. 다음 회차 **비관적 잠금**은 같은
  "확정 1건"을 만들되 **차단 지점이 "읽기 단계"로 앞당겨지는지**(잠금을 조회 시점에 선점)를 대비
  측정한다 — 거부되는 요청이 슬롯을 읽는 순간의 상태가 1차/2차와 어떻게 달라지는지가 핵심 관찰점이다.
