> 5개 보고서의 정량 주장을 `1. 테스트결과_Application계측`(F004_측정결과_*.xlsx)과
> `2. 결과_p6spy로깅`(*_타임라인.xlsx) 원본과 1:1 대조한 결과.

## 2. 보완 필요 — 원본 xlsx 자체의 오류 (보고서 아님, 추적성 문제)

보고서는 올바른 값을 썼으나, **근거로 인용된 원본 파일 안에** 아래 오기/잔재가 있다.
나중에 원본을 직접 열어 대조하는 사람이 혼란을 겪으므로 수정 권장.

1. **[D 슬롯번호 오기]** `F004_측정결과_D_조건부UPDATE.xlsx` "측정 요약" 시트 **제목에 "slot 1219"**
   로 적혀 있으나, 같은 시트의 승자 행·요청별 시트·p6spy·보고서는 모두 **slot 1221**.
   → 한 시트 안에서 1219 vs 1221 모순. 제목을 1221로 정정.

2. **[B 슬롯번호 오기]** `3_B_비관적락_타임라인.xlsx` "회차 환경" 헤더가 **"slot 1240"**.
   그러나 시트의 모든 발행 SQL은 **SLOT_ID=1237**(보고서와 일치). 헤더 라벨이 stale.
   (3차 보고서가 "JMeter 화면은 slot 1240 런이라 결합 안 함"이라 한 것과 라벨이 뒤섞인 듯.)
   → 헤더를 slot 1237로 정정.

3. **[B 사용자-스레드 매핑 깨짐]** `3_B_비관적락_타임라인.xlsx` "사용자/스레드" 컬럼이
   **5개 스레드 전부 "searcher05"**로 라벨됨. 승자(searcher05)는 INSERT SQL로 확정되지만,
   거부 4건의 실제 searcher는 이 로그만으로 식별 불가. (보고서가 거부 4건에 searcher를
   부여하지 않고 exec 스레드명만 쓴 것은 이 한계를 정직하게 반영한 것 — 보고서는 정상.)
   → 가능하면 build 스크립트의 사용자 매핑 로직 점검.

4. **[B·C 해석가이드 복붙 잔재]** `F004_측정결과_B_*.xlsx`·`_C_*.xlsx`의 "회차 비교 요약"·
   "예상 동작 대조" 시트 **해석 가이드 문구가 baseline 템플릿 그대로**다. 예: B 요약시트에
   "점유 통과 = 총 요청 수 | N건 모두 UPDATE 성공"이라 적혀 있으나 B 실제 점유통과는 **1/5**.
   가이드 문구가 실측과 모순. (D 파일은 전용 가이드로 교체되어 있어 대비됨.)
   → B·C 가이드 문구를 해당 기법에 맞게 교체.

5. **[탐색자 컬럼 부재 — 일관성]** baseline·B·C의 "회차별 측정 결과" 시트에는 **탐색자(searcher)
   ID 컬럼이 없고**(매물 식별자만 있음), D 시트에는 있다. 회차 간 컬럼 구성이 불일치.
   → 4종 시트 컬럼 통일(특히 searcher ID 추가) 권장.

## 3. 보완 필요 — 보고서가 더 명확해질 지점 (선택)

1. **[3차 JMeter 공란]** 3차 보고서 §3.4의 응답시간·처리량이 공란이다(slot 1237 런 JMeter
   미확보, 보고서가 이미 정직하게 명시). 추적성 완성을 원하면 slot 1237 JMeter 재측정 또는
   "구조적 응답코드(1×201+4×409)는 DB·앱계측에서 도출"이라는 현 단서 유지.

2. **[baseline SQL 발행순서 표기]** 1차 종합보고서 §3.3은 SQL을 "①UPDATE → ②INSERT" 순으로
   제시하나, **실제 p6spy 발행 순서는 INSERT(.308~) → UPDATE(.318~)** 다(Hibernate flush가
   INSERT를 먼저 내보냄). 코드 단계 순서(6단계 UPDATE, 7단계 INSERT)와 발행 순서가 다름을
   주석으로 달면 정확. (테스트결과서 §5.1 표는 INSERT→UPDATE 순으로 올바르게 기재되어 있음.
   B·C 보고서도 발행순서를 올바르게 서술. 1차 종합보고서 §3.3 narrative만 코드순으로 표기.)

3. **[정합성 DB 잔여 원본 부재]** 정합성 최종 근거는 SQL Developer의 `VISIT_RESERVATION` 잔여
   조회인데, 그 **원본 캡처/쿼리결과 파일이 두 근거 폴더에 없다**(App계측 CONFIRMED 행수로
   교차확인은 가능). DB 잔여 스냅샷(또는 조회 SQL+결과)을 근거 폴더에 포함하면 채널①의 추적성
   완성. + baseline·B·C 요약시트의 "CONFIRMED 행 수 (DB에 남은 확정 예약)" 라벨은 실제로는
   로그(report.py) 산출값이므로(D 시트처럼) "CP3 도달 수, 최종 근거는 DB 잔여" 주석 통일 권장.

## 4. 종합

- **데이터 포함 여부: 정상.** 보고서가 인용한 모든 정량값(took/affected/res ID/단계분포/응답
  시간/정합성)은 App계측·p6spy 원본과 일치한다. 데이터 오삽입·과장·누락 없음.
- **보완은 "원본 파일의 라벨 위생"이 핵심**(§2). 특히 D 요약시트 "slot 1219", B 타임라인
  "slot 1240" 헤더, B 전원 "searcher05" 라벨 3건은 직접 대조자에게 혼란을 주므로 우선 수정.
- 보고서 본문 보완(§3)은 선택 사항이며, 대부분 이미 정직하게 한계를 명시하고 있다.
