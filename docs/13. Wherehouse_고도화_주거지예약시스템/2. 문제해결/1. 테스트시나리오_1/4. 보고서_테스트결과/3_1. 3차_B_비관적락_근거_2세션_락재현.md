# F004 3차 B(비관적 락) — 근거 문서: Oracle SQL Developer 2세션 락 직접 재현

> 3차 보고서는 행 락 대기를 p6spy `took`(73~128ms)으로 **사후**에만 관측했다(부하 버스트가 너무
> 빨라 잠금이 걸려 있는 순간을 사람이 볼 수 없다). 본 문서는 SQL Developer **두 세션**으로 그 잠금을
> 손으로 오래 붙잡아 **걸려 있는 순간 자체**를 직접 재현·관측한 기록이다. 앱·JMeter 없이 DB 차원에서
> 비관적 락의 **획득 → 차단 → 해제** 와 *후착이 풀린 뒤 읽는 값* 을 검증한다.
> **결과 한 줄: 후착의 `SELECT … FOR UPDATE` 는 선착이 끝날 때까지 `enq: TX - row lock contention`
> 으로 멈추고, 풀리는 순간 선착이 커밋으로 남긴 최신 값을 읽는다 — 선착이 롤백이면 AVAILABLE, 선착이
> RESERVED 커밋이면 RESERVED. 이 RESERVED 읽기가 3차 보고서의 거부 4건이 본 상태와 동일하다.**

---

## 1. 이 문서가 뒷받침하는 것 (3차 보고서와의 연결)
3차 보고서의 두 서술이 *추론*이 아니라 *DB 직접 관측*임을 보인다.

| 3차 보고서 서술 | 본 재현의 직접 증거 |
|---|---|
| §3.3 후착 `FOR UPDATE` 의 `took=73~128ms` = "행 락 대기" | 대기 세션이 `enq: TX - row lock contention` 으로 멈춤을 v$session 으로 직접 관측 |
| §3.2 거부 4건이 확인 조회 시점에 `RESERVED` 인지 | Cycle 2 에서 대기 세션이 풀린 뒤 `RESERVED` 를 읽는 것으로 재현 |
| §4 "락 보유 구간 = 읽기~커밋, 후착은 그동안 대기" | 선착이 커밋/롤백하기 전까지 후착이 무한 대기, 종료 즉시 풀림을 관측 |

## 2. 왜 직접 재현이 필요했나
JMeter 5동시 버스트는 수십~수백 ms 안에 끝난다. 후착이 락을 기다리는 구간이 `took` 숫자(73~128ms)로
**사후에 한 줄 남을 뿐**, "지금 잠겨서 멈춰 있다" 는 상태를 화면으로 볼 수 없다. 두 세션으로 선착을
수동으로 오래 붙잡으면, 그 대기 구간이 **정지 화면처럼** 길게 늘어나 직접 관측 가능해진다.

## 3. 환경 / 세션 구성
| 항목 | 값 |
|---|---|
| 도구 | Oracle SQL Developer, 접속 `wherehouse_localhost` (+ 동일 접속정보 **복제 연결 1개**) |
| 세션 | 2개 — **선착(보유) 세션** 과 **후착(대기) 세션**. 관측용 제3 워크시트에서 `v$session` 조회 |
| 관측 당시 SID | 보유 = **1224**, 대기 = **1348** (SID·`seconds_in_wait` 는 실행마다 달라지는 관측값) |
| 격리 수준 | READ COMMITTED (앱 테스트와 동일) |
| 대상 슬롯 | AVAILABLE 슬롯 **905**(Cycle 1) · **906**(Cycle 2) — **측정 회차의 slot 1237 과 무관한 별도 슬롯** (측정 데이터 비파괴) |

> ★ SQL Developer 함정: **같은 연결의 워크시트 2개는 같은 DB 세션**이라 서로 막지 않는다(자기 자신을
> 막지 못함). 블로킹을 보려면 반드시 **연결을 2개**(= 세션 2개) 만들어야 한다.

핵심 한 문장(앱과 동일한 잠금 지점):
```sql
SELECT ... FROM VISIT_SLOT WHERE SLOT_ID = ? FOR UPDATE;
```
이 SELECT 가 읽으면서 그 행에 배타적 락(행 X-락)을 건다. 선착이 커밋/롤백할 때까지 **같은 행을
FOR UPDATE 하려는 다른 세션은 그 줄에서 멈춘다.**

---

## 4. Cycle 1 — 획득 → 차단 → 해제(롤백): 후착은 `AVAILABLE` 을 읽는다
| 단계 | 세션(SID) | 실행 SQL | 관측 결과 |
|---|---|---|---|
| ① 획득 | 보유(1224) | `SELECT SLOT_ID, STATUS FROM VISIT_SLOT WHERE SLOT_ID=905 FOR UPDATE;` | **905 / AVAILABLE, 0.002초(즉시)**. 커밋하지 않음 → 행 X-락 보유 |
| ② 차단 | 대기(1348) | `SELECT SLOT_ID, STATUS FROM VISIT_SLOT WHERE SLOT_ID=905 FOR UPDATE;` | 결과 없이 **멈춤(블록)** |
| ③ 관측 | 제3 | `SELECT sid, blocking_session, event, seconds_in_wait FROM v$session WHERE blocking_session IS NOT NULL;` | **1348 ← blocking 1224 / `enq: TX - row lock contention` / seconds_in_wait=25** |
| ④ 해제 | 보유(1224) | `ROLLBACK;` | "롤백 완료" → 락 해제 |
| ⑤ 풀림 | 대기(1348) | (②의 멈췄던 SELECT 가 반환) | **905 / AVAILABLE, 총 346.892초** |
| ⑥ 재관측 | 제3 | (모니터 재실행) | **0행** (차단 관계 소멸) |

**해석:** 선착이 값을 바꾸지 않고(`ROLLBACK`) 락만 놓았으므로, 풀린 후착은 원래 값 `AVAILABLE` 을 읽는다.
대기 시간 346.892초 = 선착을 수동으로 붙잡고 있던 전체 시간(앱에서는 이 자리가 `took` 수십~수백 ms).

## 5. Cycle 2 — 획득 → 차단 → 해제(RESERVED 커밋): 후착은 `RESERVED` 를 읽는다 (= 거부 트리거)
| 단계 | 세션(SID) | 실행 SQL | 관측 결과 |
|---|---|---|---|
| ① 획득 | 보유(1224) | `SELECT SLOT_ID, STATUS FROM VISIT_SLOT WHERE SLOT_ID=906 FOR UPDATE;` | **906 / AVAILABLE(즉시)**, 미커밋 |
| ② 차단 | 대기(1348) | `SELECT SLOT_ID, STATUS FROM VISIT_SLOT WHERE SLOT_ID=906 FOR UPDATE;` | **멈춤(블록)** |
| ③ 관측 | 제3 | (위와 동일 v$session 모니터) | **1348 ← 1224 / `enq: TX - row lock contention` / seconds_in_wait=4** |
| ④ 변경·커밋 | 보유(1224) | `UPDATE VISIT_SLOT SET STATUS='RESERVED' WHERE SLOT_ID=906;` → `COMMIT;` | "1 행 업데이트", "커밋 완료" |
| ⑤ 풀림 | 대기(1348) | (②의 멈췄던 SELECT 가 반환) | **906 / `RESERVED`, 총 258.397초** |
| ⑥ 재관측 | 제3 | (모니터 재실행) | **0행** |

**해석:** 선착이 `RESERVED` 로 바꾸고 커밋했으므로, 풀린 후착은 **선착이 막 커밋한 최신 값 `RESERVED`**
를 읽는다. `FOR UPDATE` 는 잠글 대상을 잡으려 **현재 커밋된 값(current-mode read)** 을 읽기 때문이다.
이 `RESERVED` 읽기가 3차 보고서에서 거부 4건이 확인 조회 시점에 본 그 상태이고, 앱이라면 바로 다음
검증에서 막힌다(아래 §7).

---

## 6. 두 Cycle 대비 — 메커니즘 결론
| 선착(보유 세션)이 종료한 방식 | 후착(대기 세션)이 풀린 뒤 읽은 값 | 앱에서의 귀결 |
|---|---|---|
| `ROLLBACK` (값 변경 없음) | **AVAILABLE** | (해당 없음) |
| `UPDATE → RESERVED` + `COMMIT` | **RESERVED** | `AVAILABLE` 검증 실패 → **409 거부** |

→ **후착이 읽는 값은 "그 행의 옛 스냅샷" 이 아니라 "선착이 커밋으로 남긴 최신 상태" 다.** 차단이 풀리는
순간 current-mode read 가 최신값을 가져오기 때문이다. 비관적 락이 "읽기 시점에 줄을 세운다" 는 말의
실제 모습이 이 두 Cycle 의 대비다.

## 7. 앱 코드와의 대응
앱 `createReservation` 은 슬롯을 `SELECT … FOR UPDATE` 로 조회(락 획득)한 뒤, 읽은 상태가
`AVAILABLE` 이 아니면 거부한다.
```
SELECT … FOR UPDATE        ← Cycle 2 대기 세션이 막혔던 바로 그 지점 (후착은 여기서 대기)
   ↓ (선착 커밋 후 풀림)
읽은 STATUS = RESERVED       ← Cycle 2 ⑤ 와 동일. 3차 보고서 CP1 의 status=RESERVED 와 동일
   ↓
AVAILABLE 아님 → 거부(409)
```
즉 **락 대기 → 풀린 뒤 RESERVED 인지 → 검증 실패 → 거부** 라는 3차 보고서의 거부 경로 전체가, 본
재현으로 DB 차원에서 한 단계씩 확인된다.

## 8. 타임라인 `took` 과의 대응
| 본 재현 | 3차 타임라인(slot 1237) |
|---|---|
| 보유 세션 즉시 획득(0.002초) | 성공 트랜잭션 `FOR UPDATE` `took=2ms` |
| 대기 세션 블록(`enq: TX - row lock contention`) | 후착 `FOR UPDATE` `took=73·97·112·128ms` 의 정체 |
| 대기 총 시간 346.892초 / 258.397초 | 위 `took` 을 수동으로 늘린 것(선착을 오래 붙잡음) |

`took` 은 "이미 끝난 대기의 길이" 였고, `enq: TX - row lock contention` 은 "그 대기가 진행 중일 때의 이름"
이다. 같은 사건의 사후 측정값과 실시간 상태.

## 9. 재현 절차 (복제 가능 · 비파괴)
아래 순서로 두 연결(세션)에서 실행. **데이터를 실제로 바꾸므로 마지막 원복까지 한 묶음이다.**
```sql
-- [세션 A = 선착]  Cycle 2 (거부 트리거) 기준
SELECT SLOT_ID, STATUS FROM VISIT_SLOT WHERE SLOT_ID=906 FOR UPDATE;   -- 락 획득, 커밋 보류

-- [세션 B = 후착]  (여기서 멈춤 = 블록)
SELECT SLOT_ID, STATUS FROM VISIT_SLOT WHERE SLOT_ID=906 FOR UPDATE;

-- [관측용 제3 워크시트]
SELECT sid, blocking_session, event, seconds_in_wait
FROM   v$session WHERE blocking_session IS NOT NULL;          -- 후착 ← 선착, enq: TX - row lock contention

-- [세션 A]  값 변경 후 커밋  → 그 순간 세션 B 풀림(RESERVED 읽음)
UPDATE VISIT_SLOT SET STATUS='RESERVED' WHERE SLOT_ID=906;
COMMIT;

-- [세션 B]  락 해제
ROLLBACK;

-- [원복]  실습으로 바뀐 슬롯을 원래대로
UPDATE VISIT_SLOT SET STATUS='AVAILABLE' WHERE SLOT_ID IN (905, 906) AND STATUS='RESERVED';
COMMIT;

-- [확인]  둘 다 AVAILABLE 이어야 정상
SELECT SLOT_ID, STATUS FROM VISIT_SLOT WHERE SLOT_ID IN (905, 906);
```
원복 확인 실측: **905 / AVAILABLE, 906 / AVAILABLE** (2행) — 측정 데이터·슬롯 원상복구 완료.
> `v$session` 조회는 권한이 필요할 수 있다. 권한이 없어도 ②단계의 "멈춤" 자체로 락은 확인된다.

## 10. 이 재현의 범위 (한계)
- **재현한 것**: 행 락의 획득·차단·해제, 그리고 후착이 풀린 뒤 읽는 값(AVAILABLE vs RESERVED).
- **재현하지 않은 것**: 5스레드 동시 출발, 앱 트랜잭션 전체 흐름, 409 응답 생성 — 이는 채널 ②(앱 계측)·
  ④(JMeter)가 담당한다. 본 문서는 **락 메커니즘만** DB에서 떼어내 직접 보인 것이다.
- `seconds_in_wait`(25·4초)는 모니터 실행 시점의 누적 대기 스냅샷이고, 총 대기는 후착 SELECT 의 소요
  (346.892·258.397초)로 측정했다. SID 값은 실행마다 달라지는 관측치다.
