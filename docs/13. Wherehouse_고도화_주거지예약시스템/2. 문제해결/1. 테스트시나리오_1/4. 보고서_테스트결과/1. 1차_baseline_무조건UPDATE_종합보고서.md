# F004 — 방문 예약 슬롯 동시성 제어 전략 결정 · 1차: 기준선(무제어)

> 본 문서는 "방문 예약 슬롯에 어떤 동시성 제어를 둘 것인가"라는 설계 결정의 **기준선(baseline)**
> 측정 기록이다. 채택 후보를 고르는 회차가 아니라, **제어가 전혀 없을 때 비즈니스 불변식이
> 실제로 어떻게·어디서 깨지는지**를 실측해 이후 3종(조건부 UPDATE·비관적 락·낙관적 락) 비교의
> 출발점을 만든다.

---

## 1. 설계 질문과 판단 기준

### 1-1. 요구사항에서 설계 질문으로

방문 예약은 탐색자(searcher)가 매물을 직접 보러 가기 위해, 등록자(집주인)가 열어 둔 방문 윈도우
(`VISIT_WINDOW`)를 고정 길이로 나눈 **슬롯(`VISIT_SLOT`)** 하나를 예약(`VISIT_RESERVATION`)하는
기능이다. 요구사항 수집 단계에서 이 기능의 본질은 단순한 "신청"이 아니라 **물리적 자원(특정 시간대의
방문)을 한 사람에게 배타적으로 배정**하는 것임이 드러났다. 한 시간대 슬롯이 두 탐색자에게 동시에
확정되면 한 자리에 둘이 배정되어, 예약 기능이 제 역할을 못 하게 된다.

요구사항 분석을 통해 이 기능의 핵심 위험을 **확인-후-행위(check-then-act) 경합**으로 특정했다. 예약
로직은 본질적으로 "슬롯을 *읽어* 예약 가능한지 *확인*한 뒤, 예약으로 *전이*"한다. 동시 요청이
들어오면 *확인*과 *행위* 사이의 시간 간격에서 여러 요청이 모두 "예약 가능"을 보고 진입할 수 있다.

```
T1: 슬롯 읽기 → status=AVAILABLE 확인        (제어가 없으면 둘 다 통과)
T2: 슬롯 읽기 → status=AVAILABLE 확인
T1: RESERVED 전이 + 예약 INSERT
T2: RESERVED 전이 + 예약 INSERT             → 한 슬롯에 확정 2건 (불변식 위반)
```

따라서 해결해야 할 핵심 설계 질문은 다음이다.

> 한 슬롯에 여러 탐색자가 *같은 순간* 예약을 시도할 때, 시스템은 어떻게 **정확히 하나의 확정만**
> 남기고 나머지를 정상적인 거부 응답으로 처리할 것인가? 그리고 그 책임을 **어느 계층**(애플리케이션
> 검증 / 조건부 갱신 / DB 락 / 버전)에 둘 것인가?

본 1차 측정은 이 질문에 답하기 *전에* 반드시 확정해야 하는 선행 질문을 다룬다.

> **제어가 전혀 없으면 무엇이, 어디서 깨지는가?** 그 붕괴는 클라이언트 응답에 드러나는가, 아니면
> 서버·DB 안에서만 보이는가?

### 1-2. 비즈니스 불변식

방문 예약이 보장해야 하는 비즈니스 불변식은 **"한 슬롯의 확정(CONFIRMED) 예약은 어느 시점에나
최대 1건"** 이다.

이 불변식은 예약 기능이 성립하기 위한 기본 조건이다. 슬롯 하나에 확정이 2건 이상 남으면 한 자리에
두 사람이 배정되어, 예약 결과를 신뢰할 수 없게 된다. 그러므로 이 불변식은 "가능하면 지킨다"가 아니라
"어떤 동시 부하에서도 반드시 지킨다"가 목표여야 한다.

중요한 것은 반대 조건이다. **서로 다른 슬롯에 대한 예약 요청 사이에는 순서 의존성이 없다.** 13시
슬롯 예약과 14시 슬롯 예약은 서로 경합할 이유가 없다. 따라서 올바른 동시성 제어는 "모든 예약을
순서대로 처리"하는 것이 아니라 **같은 슬롯 충돌 범위에서만 직렬화**하는 것이며, 이 "충돌 범위와
직렬화 범위의 일치"가 이후 회차의 핵심 평가 관점이 된다.

### 1-3. 비교 기준

이번 의사결정의 비교 기준은 단순 응답 시간이 아니라 다음 네 가지다. 1차는 이 네 축의 **최악값**을
보여주는 기준선이다.

| 기준 | 의미 |
|---|---|
| 정합성 | 같은 슬롯 동시 예약 시 확정이 정확히 1건으로 유지되는가 |
| 거부 신호 | 거부되는 요청이 클라이언트에 의미 있는 신호(409 등)로 드러나는가 |
| 거부 비용 | 거부되는 요청이 헛작업(불필요한 INSERT·롤백)을 치르는가 |
| 차단 위치 | 거부가 읽기/쓰기/커밋 중 어느 단계에서 일어나는가 |

**결론부터 말하면, 무제어(baseline)는 네 축 전부에서 실패한다.** 정합성은 한 슬롯에 확정 5건으로
붕괴하고, 거부는 *아예 일어나지 않으며*(5건 모두 성공), 그래서 거부 신호도 거부 비용도 존재하지
않는다. 더 위험한 것은 이 붕괴가 **클라이언트 응답에는 5건 모두 `201 Created`로 보여 전혀 드러나지
않는다**는 점이다. 즉 무제어의 문제는 "느리다"가 아니라 **외부 응답에는 드러나지 않은 채 정합성이 깨진다**는 것이다. 이 사실이
"동시성 제어는 선택이 아니라 필수"라는 판단의 직접 근거이며, 이후 3종 비교의 기준선이 된다.

## 2. 측정 설계

### 2-1. 측정 환경

| 항목 | 값 |
|---|---|
| 대상 로직 | 슬롯 예약 (`VisitReservationWriteService.createReservation`) |
| 동시성 제어 | **없음** — 상태 조건·락 없이 무조건 `UPDATE … WHERE SLOT_ID=?` |
| 사전 검증 | 읽은 뒤 "예약 가능?"만 확인하는 방식(확인-후-행위) |
| DB 보호 장치 | 슬롯당 확정 1건을 강제하는 부분 유일 인덱스 **미적용**(붕괴를 *제어 부재 자체*에 귀속) |
| 격리 수준 | READ COMMITTED (Oracle 기본) |
| 부하 | 5명 동시 요청(JMeter Synchronizing Timer로 동시 출발 보장), searcher01~05 |
| 대상 슬롯 | window 81 · **slot 1218** (방문 2026-06-09 13:35~14:05) |
| 실행 시각 | 2026-06-02 13:18:20 |

부분 유일 인덱스를 일부러 적용하지 않은 이유는, 붕괴가 *제어 기법의 유무* 때문에 발생함을 분리해
보이기 위해서다. DB 보호망까지 켜 두면 "코드가 막았는지 DB가 막았는지"가 섞여 기준선의 의미가
흐려진다.

### 2-2. 관측 채널

같은 사건을 네 각도에서 교차 측정해, 한 채널의 한계를 다른 채널이 메우게 했다.

| 채널 | 출처 | 무엇을 보나 |
|---|---|---|
| ⑴ 정합성 | DB 잔여 `VISIT_RESERVATION` / `VISIT_SLOT` | 슬롯당 확정 수(1=정상 / 2+=붕괴) |
| ⑵ 단계 분포 | 앱 단계 로그(CP1 읽기 / CP2 점유 / CP3 등록) | 각 요청이 어느 단계까지 통과했나 |
| ⑶ 발행 SQL | P6Spy 로그(문장별 `took`) | 실제 발행된 SQL과 행 락 거동 |
| ⑷ 외부 결과 | JMeter | 클라이언트가 받는 응답 코드 |

> 시간 축이 다른 두 수치를 섞지 않는다: **JMeter 응답시간** = HTTP 왕복 전체, **앱 계측 시간** =
> 트랜잭션 내부 "확인~등록" 구간, **P6Spy `took`** = 개별 SQL 실행 시간.

## 3. 측정 결과

### 3-1. 정합성 — DB 잔여 (채널 ⑴)

slot 1218 에 **확정(CONFIRMED) 예약 5건**이 남았다. 모두 동일 확정 시각이며 슬롯 상태는 `RESERVED`.

| RESERVATION_ID | SEARCHER_USER_ID | SLOT_ID | STATUS | CONFIRMED_AT |
|---|---|---|---|---|
| 81 | searcher05 | 1218 | CONFIRMED | 2026-06-02 13:18:20.214 |
| 82 | searcher01 | 1218 | CONFIRMED | 2026-06-02 13:18:20.214 |
| 83 | searcher04 | 1218 | CONFIRMED | 2026-06-02 13:18:20.214 |
| 84 | searcher03 | 1218 | CONFIRMED | 2026-06-02 13:18:20.214 |
| 85 | searcher02 | 1218 | CONFIRMED | 2026-06-02 13:18:20.214 |

→ **한 슬롯에 확정 5건 = 비즈니스 불변식 위반(정합성 붕괴).**

### 3-2. 단계 분포 — 앱 계측 (채널 ⑵)

| 단계 | 통과 | 의미 |
|---|---|---|
| 슬롯 읽기 ("예약 가능"으로 인지) | **5 / 5** | 5명이 동시에 같은 옛 상태를 읽음 |
| 슬롯 점유 (RESERVED 로 갱신) | **5 / 5** | 5건 모두 갱신 성공(거부 없음) |
| 예약 등록 (INSERT) | **5 / 5** | 5건 모두 등록 성공 |

5명 모두 "예약 가능"으로 인지한 뒤 점유·등록까지 완주했다. 등록 직후 같은 슬롯에서 관측한 예약 수가
1 → 2 → 3 → 4 → 5 로 누적된다(뒤늦게 끝난 요청일수록 더 많은 확정을 목격). 요청당 내부 처리(읽기~
등록)는 평균 **56.19ms**(최소 55.00 · 최대 60.87 · 표준편차 2.62ms)로, 모두 같은 코드 경로를 통과해
분산이 작다 — **거부 경로가 아예 없다**는 사실의 다른 표현이다.

### 3-3. 발행 SQL — P6Spy (채널 ⑶)

5개 트랜잭션이 동일한 SQL을 발행했고, 결과를 가르는 두 문장은 다음이다(Hibernate flush 특성상 실제
발행 순서는 INSERT가 UPDATE보다 앞선다).

```sql
-- 예약 등록: 확정 예약 INSERT (5건 모두 실행·성공) — searcher별 값
insert into VISIT_RESERVATION (CONFIRMED_AT,SEARCHER_USER_ID,SLOT_ID,STATUS,RESERVATION_ID)
  values (TIMESTAMP '2026-06-02 13:18:20.214','searcher01',1218,'CONFIRMED',82);

-- 슬롯 점유: 상태 조건이 전혀 없는 무조건 갱신 (5건 모두 "1행 갱신 성공")
update VISIT_SLOT set STATUS='RESERVED' where SLOT_ID=1218;
```

| searcher (스레드) | INSERT (took / res_id) | UPDATE 슬롯 (took) | 종료 |
|---|---|---|---|
| searcher04 (exec-5) | 2ms / res 83 | **66ms** | 커밋 |
| searcher02 (exec-2) | 6ms / res 85 | 17ms | 커밋 |
| searcher01 (exec-1) | 6ms / res 82 | **1ms** | 커밋 |
| searcher05 (exec-4) | 6ms / res 81 | 58ms | 커밋 |
| searcher03 (exec-3) | 21ms / res 84 | 14ms | 커밋 |

핵심은 `UPDATE` 의 `WHERE` 에 **상태 조건이 없다**는 점이다. 먼저 한 트랜잭션이 `RESERVED` 로 바꿔도
뒤 트랜잭션의 갱신이 거부되지 않고 그대로 덮어쓴다. 슬롯 갱신 `took` 이 1ms~66ms 로 벌어지는 것은
같은 행이라 Oracle 행 락이 **순차 실행**시키기 때문이다 — 즉 **잠금은 순서만 매길 뿐 뒤 트랜잭션을
거부하지 않는다.** 이것이 baseline 의 본질이다: *직렬화는 일어나지만 차단은 없다.*

### 3-4. 외부 결과 — JMeter (채널 ⑷)

| 항목 | 값 |
|---|---|
| 표본 / 오류 | **5건 모두 201 Created / 오류 0%** |
| 응답시간(평균/중앙/최대) | 454 / 409 / 614 ms |
| 처리량 | 8.1 req/s |

**5명 모두에게 "예약 성공"이 돌아간다.** 외부에서는 정합성 붕괴의 흔적이 전혀 보이지 않는다.

## 4. 메커니즘 — 왜 외부 응답에는 드러나지 않는가

```
5명 동시 요청 (READ COMMITTED, 잠금 없는 읽기)
  ├ 5명 모두 슬롯을 읽고 "예약 가능"으로 인지        ← 확인-후-행위의 "확인"이 동시에 통과
  ├ 5명 모두 무조건 UPDATE → 모두 성공                ← 행 락은 순서만 매기고 거부는 안 함
  └ 5명 모두 예약 INSERT → 5건 커밋
결과: 한 슬롯에 확정 예약 5건 / 클라이언트는 5건 모두 201
```

붕괴의 직접 원인은 "확인"과 "행위(갱신·등록)" 사이의 시간 간격이다. 이 간격에서 5명이 모두 옛 상태
(`AVAILABLE`)를 읽고 진입한다. 무조건 UPDATE 는 "마지막 값만 남기는" 것이 아니라 **5건의 예약
INSERT 를 모두 커밋**시켜 불변식을 위반한다.

가장 중요한 발견은 **이 붕괴가 외부 응답(오류 0%)에는 전혀 드러나지 않는다**는 점이다. 응답 코드만
보는 모니터링으로는 절대 잡을 수 없고, 서버·DB 관측(3-1~3-3)이라야 보인다.
"동시성 제어가 필요한가?"라는 질문이 운영 지표만으로는 영원히 "정상"으로 답해질 수 있다는 뜻이다.

## 5. 결론 — 이 기준선이 결정에 기여하는 것

1차 baseline 의 결론은 "버그가 있었다"가 아니다. 더 정확히는 다음이다.

> 방문 예약 슬롯은 **동시성 제어와 DB 보호 장치가 모두 없으면 단일 슬롯에서 정합성이 붕괴(확정
> 5건)** 하며, 그 붕괴는 **클라이언트 응답에는 오류 0%로 보여 외부에서 드러나지 않는다.** 따라서 동시성
> 제어는 성능 최적화가 아니라 **비즈니스 불변식을 지키기 위한 필수 안전장치**다.

이 기준선이 이후 회차에 남기는 비교축은 명확하다. 무제어는 §1-3 네 축 전부에서 최악(정합성 붕괴 ·
거부 없음 · 신호 없음 · 차단 없음)이므로, 이후 3종은 모두 "정합성을 확정 1건으로 회복"하는 것을
최소 조건으로 한다. 그러면 정합성만으로는 후보를 가를 수 없게 되고, **갈리는 축은 "거부되는 요청을
어디서·어떤 비용으로·어떤 신호로 처리하는가"** 로 이동한다. 1차는 바로 그 비교가 의미를 갖도록
하는 출발점이다.

- 다음 회차(② 조건부 UPDATE): 같은 5건 동시 부하에서 확정이 1건으로 회복되는지, 거부 4건이 **어느
  단계에서·어떤 신호로** 차단되는지를 본 회차와 대비 측정한다.
