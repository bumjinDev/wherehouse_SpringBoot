# F004 슬롯 예약 동시성 측정 — 1차: 기준선(무조건 UPDATE, 동시성 제어 없음)

> 단일 슬롯에 5명이 **동시에** 예약을 시도할 때, 동시성 제어가 전혀 없는 코드가 어떻게 동작하는지를
> 4개 관측 축(DB 잔여 · 애플리케이션 단계 계측 · 실제 발행 SQL · 외부 부하 결과)으로 교차 측정한다.
> **결과 한 줄: 한 슬롯에 확정 예약이 5건 생겨 정합성이 붕괴한다. 그런데 클라이언트에는 5건 모두
> "성공(201)"으로 보여, 외부 응답만으로는 이 붕괴가 드러나지 않는다.**

---

## 1. 측정 목적
슬롯 예약에 어떤 동시성 제어가 필요한지를 결정하기 위한 **기준선(baseline)** 확보. 제어 기법과
데이터베이스 차원의 보호 장치가 모두 없을 때, 동시 예약이 핵심 비즈니스 규칙(**한 슬롯의 확정
예약은 어느 시점에나 1건**)을 어떻게 위반하는지, 그리고 그 위반이 각 관측 축에서 어떻게 드러나는지
(또는 드러나지 않는지)를 실측한다.

## 2. 측정 환경
| 항목 | 값 |
|---|---|
| 대상 로직 | 슬롯 예약 (`createReservation`) |
| 동시성 제어 | **없음** — 슬롯 상태를 무조건 `RESERVED` 로 갱신(`UPDATE … WHERE SLOT_ID=?`) |
| 사전 검증 | 읽은 뒤 "예약 가능?"만 확인하는 방식(확인-후-행위) |
| DB 보호 장치 | 슬롯당 확정 1건을 강제하는 부분 유일 인덱스 **미적용** |
| 트랜잭션 격리 수준 | READ COMMITTED |
| 부하 | 5명 동시 요청(동시 출발 보장), searcher01~05 |
| 대상 슬롯 | window 81 · **slot 1218** (방문 2026-06-09 13:35~14:05) |
| 실행 시각 | 2026-06-02 13:18:20 |

## 3. 측정 결과

### 3.1 정합성 — 데이터베이스 잔여 (`VISIT_RESERVATION`)
- slot 1218 에 **확정(CONFIRMED) 예약 5건**(searcher01~05) 이 남았다 — 모두 동일한 확정 시각.
- 슬롯 상태(`VISIT_SLOT`)는 `RESERVED`.
- **→ 한 슬롯에 확정 5건 = 비즈니스 규칙 위반(정합성 붕괴).**

### 3.2 단계별 통과 분포 — 애플리케이션 계측
`F004_측정결과_baseline_무조건UPDATE.xlsx` 의 요약·요청별 시트 내용:

| 단계 | 통과 | 의미 |
|---|---|---|
| 슬롯 읽기 (슬롯을 "예약 가능"으로 인지) | **5 / 5** | 5명이 동시에 같은 옛 상태를 읽음 |
| 슬롯 점유 (RESERVED 로 갱신) | **5 / 5** | 5건 모두 갱신 성공(거부 없음) |
| 예약 등록 (예약 행 INSERT) | **5 / 5** | 5건 모두 등록 성공 |
| 확정 예약 수 / 정합성 위반 요청 수 | 5 / 4 | |

요청별로 보면 **5명 모두 "예약 가능"으로 인지한 뒤 점유·등록까지 완주**했고, 등록 직후 같은 슬롯에서
관측한 예약 수가 1 → 2 → 3 → 4 → 5 로 누적된다(뒤늦게 끝난 요청일수록 더 많은 확정 예약을 목격).
요청당 내부 처리 구간(읽기~등록)은 약 55~61ms.

### 3.3 실제 발행된 SQL (Oracle 실행형) — `1_baseline_무조건UPDATE_타임라인.xlsx`
5개 트랜잭션이 각각 동일한 순서의 SQL을 발행했고, 동시성 결과를 가르는 **두 문장**은 다음과 같다.

```sql
-- ① 슬롯 점유: 상태 조건이 전혀 없는 무조건 갱신 (5개 트랜잭션이 모두 실행, 모두 성공)
update VISIT_SLOT set STATUS='RESERVED' where SLOT_ID=1218;

-- ② 예약 등록: 확정 예약 INSERT (5개 트랜잭션이 모두 실행, 모두 성공)
insert into VISIT_RESERVATION (CONFIRMED_AT, SEARCHER_USER_ID, SLOT_ID, STATUS, RESERVATION_ID)
values (TIMESTAMP '2026-06-02 13:18:20.214', 'searcher01', 1218, 'CONFIRMED', <시퀀스값>);
```

**해석(SQL을 처음 보는 경우):**
- ①의 `WHERE` 절에는 **슬롯 상태 조건이 없다**(`SLOT_ID=1218` 뿐). 그래서 5개 트랜잭션의 갱신이
  모두 "갱신 대상 1행"으로 성공한다 — 먼저 한 트랜잭션이 `RESERVED` 로 바꿔도, 뒤 트랜잭션의 갱신이
  거부되지 않고 그대로 덮어쓴다.
- ②의 INSERT도 아무 제약 없이 5건 모두 들어간다 → **한 슬롯에 확정 예약 5건**.
- 이 슬롯 갱신 ①의 실행 소요(`took`)는 트랜잭션별로 **1ms ~ 66ms** 로 벌어진다. 같은 행을 갱신하므로
  Oracle 이 행 단위 잠금으로 **순차 실행**시키기 때문이다 — 즉 잠금은 순서만 매길 뿐 **뒤 트랜잭션을
  거부하지는 않는다**. (이것이 baseline 의 본질: 직렬화는 일어나지만 차단은 없다.)

> ※ 위 SQL은 타임라인 시트의 "Oracle 실행형(타임스탬프를 리터럴로 변환)" 내용만 옮긴 것이다.
> 트랜잭션 경계 로그와 원본 바인딩 SQL 로그는 보고서 범위에서 제외했다.

### 3.4 외부 부하 결과 — JMeter
| 항목 | 값 |
|---|---|
| 표본 / 오류 | **5건 모두 201 Created(예약 확정) / 오류 0%** |
| 응답시간(평균/중앙/최대) | 454 / 409 / 614 ms (HTTP 왕복 전체) |
| 처리량 | 8.1 req/s |

- **5명 모두에게 "예약 성공" 응답이 돌아간다.** 외부에서는 정합성 위반의 흔적이 전혀 보이지 않는다.

## 4. 메커니즘 해석
```
5명 동시 요청 (READ COMMITTED, 잠금 없는 읽기)
  ├ 5명 모두 슬롯을 읽고 "예약 가능" 으로 인지          ← 확인-후-행위의 "확인"이 동시에 통과
  ├ 5명 모두 UPDATE(상태 조건 없음) → 모두 성공           ← 행 잠금은 순서만 매기고 거부는 안 함
  └ 5명 모두 예약 INSERT → 5건 커밋
결과: 한 슬롯에 확정 예약 5건
```
"확인"과 "행위(갱신·등록)" 사이의 시간 간격에서 5명이 모두 옛 상태를 읽고 진입한다. 무조건 UPDATE는
마지막 값만 남기는 게 아니라 **5건의 예약 INSERT를 모두 커밋**시켜 규칙을 위반한다.

응답시간 두 수치는 측정 구간이 다르므로 섞어 읽지 않는다: **JMeter 454ms** = HTTP 왕복 전체,
**애플리케이션 계측 ~56ms** = 트랜잭션 내부 읽기~등록 구간.

## 5. 결론
- **동시성 제어와 DB 보호 장치가 모두 없으면 단일 슬롯에서 정합성이 붕괴한다(확정 5건).**
- 이 붕괴는 **외부 응답에는 오류 0%로 보이는 "조용한 데이터 붕괴"** 다. 서버/DB 관측(3.1~3.3)이라야
  드러나며, 응답 코드만 보는 모니터링으로는 잡을 수 없다.
- → 슬롯 예약에는 **동시성 제어가 반드시 필요**하다. 본 회차는 이후 제어 기법(조건부 UPDATE · 비관적
  잠금 · 낙관적 잠금) 이 **거부되는 요청을 어느 단계에서, 어떤 비용으로 차단하는지** 비교하는 기준선이다.
