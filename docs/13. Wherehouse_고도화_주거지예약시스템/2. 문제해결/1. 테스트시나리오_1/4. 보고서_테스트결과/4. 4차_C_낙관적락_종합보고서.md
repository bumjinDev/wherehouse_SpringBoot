# F004 — 방문 예약 슬롯 동시성 제어 전략 결정 · 4차: 낙관적 락(@Version)

> 본 문서는 "방문 예약 슬롯에 어떤 동시성 제어를 둘 것인가"라는 설계 결정에서 **후보 ④ 낙관적 락**
> 을 평가한 기록이다. 락을 전혀 걸지 않고 슬롯 엔티티의 `@Version` 컬럼으로 **커밋 시점의 버전
> 불일치**를 통해 충돌을 사후 검출한다. ②·③과 같은 "확정 1건"을 만들되, **차단 위치·거부 신호·거부
> 비용**이 어떻게 달라지는지가 핵심 관찰점이다.

---

## 1. 설계 질문과 판단 기준

### 1-1. 설계 질문과 이 후보의 위치

해결할 핵심 설계 질문은 동일하다 — 한 슬롯 동시 예약에서 **정확히 하나의 확정만** 남기고 나머지를
정상 거부하되, 그 책임을 **어느 계층**에 둘 것인가.

②(조건부 UPDATE)는 쓰기 시점의 갱신문에, ③(비관적 락)은 읽기 시점의 락에 책임을 두었다. 둘 다
충돌을 **사전에** 차단하는 비관적 접근이다. 후보 ④(낙관적 락)는 정반대 가정에서 출발한다 — **"충돌은
드물다"** 고 가정하고 락을 전혀 걸지 않는다. 5명이 자유롭게 읽고·검증하고·예약 INSERT 까지 진행한
뒤, **커밋 시점에 버전이 어긋난 트랜잭션만 사후에 무너뜨린다.** 엔티티에 `@Version` 을 두면 Hibernate
가 갱신문에 `… SET VERSION=VERSION+1 WHERE … AND VERSION=?` 조건을 자동으로 붙여, 버전이 바뀐
행은 갱신 0행이 되고 `OptimisticLockException` 으로 변환된다. 이 "사후 검출"이 ②·③과의 결정적 차이다.

### 1-2. 비즈니스 불변식

보장해야 하는 불변식은 **"한 슬롯의 확정 예약은 어느 시점에나 최대 1건"** 이며, **서로 다른 슬롯
예약 사이에는 순서 의존성이 없다.** 낙관적 락의 버전 조건도 같은 슬롯 행에 대해서만 충돌하므로
직렬화 범위 자체는 비즈니스 충돌 범위와 일치한다. 그러나 **낙관적 락의 전제("충돌이 드물다")는
단일 핫슬롯 동시 예약이라는 본 도메인의 최악 케이스에서 정확히 깨진다** — 이 전제 불일치가 후보
평가의 핵심이다.

### 1-3. 비교 기준과 결론

| 기준 | 의미 | ④ 낙관적 락 결과 |
|---|---|---|
| 정합성 | 확정이 정확히 1건으로 유지되는가 | **확정 1건 (②③과 동일)** |
| 거부 신호 | 거부가 의미 있는 신호로 드러나는가 | **500** (OptimisticLockException 이 409 미매핑) |
| 거부 비용 | 거부가 헛작업을 치르는가 | **헛 예약 INSERT 후 롤백 (4종 중 최대)** |
| 차단 위치 | 거부가 어느 단계에서 일어나는가 | **커밋 flush (버전 조건 0행)** |

**결론부터 말하면, 낙관적 락은 정합성은 ②③과 동일하게 확정 1건을 만들지만, 거부 비용·거부 신호·
도메인 적합성 세 축에서 가장 불리하다.** 거부 4건은 **헛 예약 INSERT 까지 수행한 뒤** 커밋 단계에서
무너지고(헛작업 최대), 그 예외가 409 로 매핑되지 않아 **클라이언트에는 500** 으로 샌다. 게다가
"충돌이 드물다"는 전제가 단일 핫슬롯에서 깨진다. 이것이 ④를 채택하지 않은 직접 근거다(상세 대비는
§5 및 `5. 테스트결과서.md`).

## 3. 측정 결과

### 3-1. 정합성 — DB 잔여 (채널 ⑴)

slot 1170 에 **최종 확정 예약 1건**(searcher03, res 125) 만 남았다(나머지 4건은 INSERT 후 롤백되어
테이블에 없다 — 3-3 참조). 슬롯 상태는 `RESERVED`, **`VERSION=1`**.

| RESERVATION_ID | SEARCHER_USER_ID | SLOT_ID | STATUS | CONFIRMED_AT |
|---|---|---|---|---|
| 125 | searcher03 | 1170 | CONFIRMED | 2026-06-04 16:00:35.662 |

→ **확정 1건 = 비즈니스 불변식 충족** (1차의 5건 붕괴와 대비, ②③과 동일).

### 3-2. 단계 분포 — 앱 계측 (채널 ⑵)

| 단계 | 통과 | 의미 |
|---|---|---|
| 확인 조회 도달 | **5 / 5** | 5명 모두 `AVAILABLE`·`VERSION=0` 으로 읽음(무락 스냅샷) |
| 슬롯 점유 (setStatus+save, CP2) | **5 / 5** | 5명 모두 점유 단계 도달(락이 없어 아무도 안 막힘) |
| 예약 등록 (INSERT, CP3) | **1 / 5** | 1건만 등록 완주 |
| 거부 | **4** | 모두 **커밋 단계의 버전 충돌**로 무너짐(메서드 밖) |

| 요청(스레드/searcher) | 경로 | 결과 |
|---|---|---|
| exec-1 / searcher03 | 확인조회 → 점유 → 등록 → 커밋 | **확정** (res 125, 확인→등록 43.93ms) |
| exec-2·3·4·6 / searcher04·02·01·05 | 확인조회 → 점유 → (헛)INSERT → **커밋 flush 버전 충돌** | 거부(500) |

**②·③과 결정적으로 다른 점:** ③은 거부 요청이 읽기(FOR UPDATE)에서, ②는 쓰기(조건부 UPDATE 0행)
에서 막혀 **예약 INSERT 에 도달하지 않는다.** 반면 ④는 락이 없어 **거부 4건이 점유·예약 INSERT 까지
모두 수행**한 뒤 커밋 단계에서 무너진다(= 헛 INSERT + 롤백, 가장 큰 낭비). 그래서 단계 분포(점유 5/5)
는 "거부가 어디서 막혔나"를 보여주지 못한다 — ④의 정합성·거부는 **DB 잔여(3-1)와 발행 SQL(3-3)** 로
판정한다.

### 3-3. 발행 SQL — P6Spy (채널 ⑶)

**5명 모두 예약 INSERT(헛 INSERT)까지 발행**한 뒤, 버전 조건부 UPDATE 에서 갈린다.

```sql
-- 슬롯 점유: @Version 이 WHERE 에 VERSION=? 를 붙인 갱신 (5건 모두 시도)
update VISIT_SLOT set STATUS='RESERVED', VERSION=1 where SLOT_ID=1170 and VERSION=0;
```

| 트랜잭션(스레드/searcher) | 예약 INSERT (res_id) | 버전조건 UPDATE (took / 행수) | 종료 |
|---|---|---|---|
| exec-1 / searcher03 (성공) | res 125 | **0ms / 1행** (VERSION 0→1) | **커밋** (.756) |
| exec-2 / searcher04 (거부) | res 124 | 27ms / **0행** | 롤백 |
| exec-3 / searcher02 (거부) | res 126 | 27ms / **0행** | 롤백 |
| exec-4 / searcher01 (거부) | res 127 | 27ms / **0행** | 롤백 |
| exec-6 / searcher05 (거부) | res 123 | 27ms / **0행** | 롤백 |

거부 4건은 모두 동일 예외로 롤백된다.

```
org.hibernate.StaleObjectStateException: Row was updated or deleted by another transaction
  (or unsaved-value mapping was incorrect) : [com.wherehouse.VisitReservation.entity.VisitSlotEntity#1170]
```

5명 모두 `VERSION=0` 을 읽고 검증을 통과(무락)해 **각자 예약 INSERT(헛 INSERT, res 123~127)** 까지 한
뒤 점유 UPDATE 를 flush 한다. 선착(exec-1)의 `UPDATE … WHERE VERSION=0` 은 1행 갱신(VERSION 0→1)
후 커밋하므로 res 125 만 테이블에 잔류한다. 나머지 4건의 같은 UPDATE 는 선착이 쥔 행 락을 기다린 뒤
(`took=27ms`), 조건을 다시 보니 `VERSION=0` 인 행이 없어 **0행** → Hibernate 가 `OptimisticLockException`
을 던져 그 트랜잭션 전체(헛 INSERT 포함)를 롤백한다 → res 123·124·126·127 은 테이블에 남지 않는다.

- **②(조건부 UPDATE)와의 관계 — 형제 메커니즘:** 둘 다 *조건부 UPDATE 가 행 락 대기 후 0행*(후행
  `took` ②24ms ≈ ④27ms). 차이는 ⑴ 조건이 업무 상태(`STATUS='AVAILABLE'`, ②)가 아니라 버전 카운터
  (`VERSION=0`, ④)이고, ⑵ 0행을 **프레임워크(JPA)가 예외로 변환**하며, ⑶ **거부 요청이 헛 INSERT 를
  한다**(②는 0행에서 끊겨 INSERT 에 도달 안 함)는 점이다.
- **③(비관적 락)과의 차이:** ③은 거부 요청이 읽기에서 막혀 헛 INSERT 가 없다. ④는 헛 INSERT 까지
  하고 롤백한다 — 거부 비용이 가장 크다.

### 3-4. 외부 결과 — JMeter (채널 ⑷)

| 항목 | 값 |
|---|---|
| 표본 / 오류 | **1건 201 + 4건 500 Internal Server Error / 오류율 80%** |
| 응답시간(평균/중앙/최대) | 389 / 390 / 390 ms |
| 처리량 | 12.8 req/s |

거부 4건이 **500(서버 에러)** 으로 드러난다(응답 본문은 위 `StaleObjectStateException` 메시지). **②·③의
거부는 409**(`SlotUnavailableException` → 409 매핑)였다. ④의 거부는 `OptimisticLockException` 이 **409
로 매핑되지 않아 미처리 500** 으로 나간다 — 같은 "거부"라도 클라이언트가 받는 신호가 다르다.

## 4. 메커니즘 — 커밋 시점에 사후 검출한다

```
5명 동시 요청 (READ COMMITTED, 락 없음)
  ├ 5명 모두 슬롯 읽기(VERSION=0) + AVAILABLE 검증 통과(무락 스냅샷)      ← 아무도 안 막힘
  ├ 5명 모두 점유(setStatus) + 예약 INSERT(헛 INSERT)                    ← 락이 없어 전원 도달
  └ 커밋 flush: UPDATE … WHERE VERSION=0
        1건 : 행 락 즉시 획득 → 1행(VERSION 0→1) → 커밋
        4건 : (행 락 27ms 대기) → 선착 커밋 후 VERSION≠0 → 0행
              → OptimisticLockException → 헛 INSERT 포함 롤백 → (미처리) 500
  결과: 한 슬롯에 확정 예약 1건
```

낙관적 락은 "확인"과 "행위" 사이에 **락을 걸지 않는다**(그래서 5명이 동시에 읽고 검증·INSERT 까지
자유롭게 진행). 충돌은 **커밋 단계의 버전 조건부 UPDATE 가 0행이 되는 순간**에야 검출된다. 그 UPDATE
자체는 행 X-락을 쓰므로(후행 27ms 대기) 직렬화는 일어나지만, **거부 판정은 락이 아니라 버전 비교**가
한다. 차단 위치가 "커밋 flush" 로 이동하고, 거부 요청은 헛 INSERT 까지 한 뒤 무너지므로 **거부 비용이
4종 중 가장 크다.**

## 5. 결론 — 정합성은 같지만 비용·신호·전제가 불리하다

낙관적 락의 결론은 다음이다.

> 낙관적 락(@Version)은 ②③과 동일하게 "확정 1건"을 만들지만, **차단 위치가 커밋 flush(버전 조건
> 0행)** 이고 **거부 요청이 헛 INSERT 까지 한 뒤 롤백**하며, 거부가 깔끔한 409 가 아니라 **미처리
> 500** 으로 샌다. 게다가 그 전제("충돌이 드물다")가 단일 핫슬롯 동시 예약이라는 본 도메인의 최악
> 케이스에서 깨진다.

ⓐ 거부 비용(헛 INSERT 후 롤백)이 4종 중 가장 크고, ⓑ 거부 신호가 500 이라 실무에서 반드시
`OptimisticLockException` 을 잡아 409 매핑/재시도 처리를 추가해야 하며(현재 코드엔 그 처리가 없어 500),
ⓒ 도메인 가정이 어긋난다 — 이 세 가지가 ④를 채택하지 않은 이유다.

다만 적합성 관점을 분명히 한다. 본 회차는 충돌이 확정되는 **단일 핫슬롯 최악 케이스**를 측정한 것이다.
이 조건에서 낙관적 락은 헛작업 비용이 크고 재시도도 무의미(슬롯이 이미 RESERVED)하다. 반대로 **동시
충돌이 드문 정상 트래픽**에서는 락을 잡지 않아 가볍다 — 즉 ④의 적합성은 *실제 경합 빈도*에 달려
있으며, 핫슬롯 예약이라는 본 도메인에서는 ②가 더 맞다.

- 최종 채택 비교(②↔③↔④ 4축 종합): 정합성은 ②③④ 동률이며, 갈리는 축은 "거부를 어디서·어떤
  비용으로·어떤 신호로 처리하는가"다. → **② 조건부 UPDATE 채택.** 상세는 `5. 테스트결과서.md` §6~§7.
