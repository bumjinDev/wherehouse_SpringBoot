# F004 슬롯 예약 동시성 측정 — 4차: 낙관적 락(@Version)

> 단일 슬롯에 5명이 동시에 예약을 시도할 때, **락을 전혀 걸지 않고** 슬롯 엔티티의 `@Version` 컬럼으로
> 충돌을 검출하는 낙관적 락의 결과를 4개 관측 축으로 측정한다. B(비관적 락)·D(조건부 UPDATE)와 같은
> "확정 1건"을 만들되, **차단 지점·거부 신호·후행이 치르는 비용**이 어떻게 다른지가 핵심 관찰점이다.
> **결과 한 줄: 한 슬롯에 확정 예약이 1건만 남는다(정합성 충족). 거부된 4개 요청은 읽기에서 막히지
> 않아 검증·예약 INSERT(헛 INSERT)까지 모두 수행한 뒤, 커밋 단계의 버전 조건부 UPDATE에서 0행이 되어
> `OptimisticLockException`으로 무너진다. 그 예외가 409로 매핑되지 않아 클라이언트에는 500(Internal
> Server Error)으로 드러난다.**

---

## 1. 측정 목적
`@Version` 낙관적 락을 유일한 동시성 게이트로 두었을 때 ⑴ 한 슬롯의 확정 예약이 1건으로 유지되는지,
⑵ 거부되는 요청이 **어느 단계에서** 차단되는지, ⑶ 거부 요청이 치르는 **비용**(헛 INSERT·롤백), ⑷ 거부가
클라이언트에 어떤 신호로 드러나는지를 실측한다. 특히 B·D와의 대비(차단 지점이 읽기/쓰기 게이트가 아니라
**커밋 단계의 버전 충돌**로 이동하는지)가 관찰의 중심이다.

## 2. 측정 환경
| 항목 | 값 |
|---|---|
| 대상 로직 | 슬롯 예약 (`createReservation`) |

| 사전 검증 | `AVAILABLE` 확인(무락 스냅샷 — 동시 N 건이 모두 통과). 거부는 이 검증이 아니라 @Version 이 만든다 |
| DB 보호 장치 | 부분 유일 인덱스 **미적용**(차단을 *제어 기법 자체*에 귀속) |
| 트랜잭션 격리 수준 | READ COMMITTED |
| 부하 | 5명 동시 요청(동시 출발 보장), searcher01~05 |

## 3. 측정 결과

### 3.1 정합성 — 데이터베이스 잔여 (`VISIT_RESERVATION`)
- slot 1170 에 **확정 예약 1건**(reservationId=125, searcher03) 만 남았다. 슬롯 상태는 `RESERVED`, `VERSION=1`.
- **→ 한 슬롯에 확정 1건 = 비즈니스 규칙 충족.** (1차의 5건 붕괴와 대비, D·B와 동일한 정합성)

### 3.2 단계별 통과 분포 — 애플리케이션 계측
`F004_측정결과_C_낙관적락.xlsx` 의 요약·요청별 시트 내용:

| 단계 | 통과 | 의미 |
|---|---|---|
| 확인 조회 도달 | **5 / 5** | 5명 모두 `AVAILABLE`·`VERSION=0` 으로 읽음(무락 스냅샷) |
| 슬롯 점유 (setStatus+save, CP2) | **5 / 5** | 5명 모두 점유 단계 도달(락이 없어 아무도 안 막힘) |
| 예약 등록 (INSERT, CP3) | **1 / 5** | 1건만 등록 완주 |
| 거부 | **4** | 모두 **커밋 단계의 버전 충돌**로 무너짐(메서드 밖) |

| 요청(스레드) | searcher | 경로 | 결과 | 비고 |
|---|---|---|---|---|
| exec-1 | searcher03 | 확인조회 → 점유 → 등록 → 커밋 | **확정**(res 125) | 확인→등록 43.93ms |
| exec-2·3·4·6 | searcher04·02·01·05 | 확인조회 → 점유 → (헛)INSERT → **커밋 flush 에서 버전 충돌** | 거부(500) | 등록(CP3) 미도달 |

- **B·D와 결정적으로 다른 점**: B는 거부 요청이 읽기(FOR UPDATE)에서, D는 쓰기(조건부 UPDATE 0행)에서
  막혀 **예약 INSERT에 도달하지 않는다**. 반면 C는 락이 없어 **거부 4건이 점유·예약 INSERT까지 모두
  수행**한 뒤 커밋 단계에서 무너진다(= 헛 INSERT + 롤백, 가장 큰 낭비).
- 그래서 `report.py` 의 단계 분포(5/5/1)는 "거부가 어디서 막혔나"를 보여주지 못한다 — C의 정합성/거부는
  **DB 잔여(3.1)와 발행 SQL(3.3)** 로 판정한다.

### 3.3 실제 발행된 SQL (Oracle 실행형) — `4_C_낙관적락_타임라인.xlsx`
동시성 결과를 가르는 **한 문장**(슬롯 점유의 버전 조건부 UPDATE):

```sql
-- 슬롯 점유: @Version 이 WHERE 에 VERSION=? 를 붙인 갱신 (5개 트랜잭션이 모두 시도)
update VISIT_SLOT set STATUS='RESERVED', VERSION=1 where SLOT_ID=1170 and VERSION=0;
```

발행 SQL 기준 트랜잭션별 결과(트랜잭션 동시 시작 `16:00:35.661`, 5명 모두 `VERSION=0` 읽음):

| 트랜잭션(스레드/searcher) | 점유 UPDATE 의 `took` | 갱신 행수 | 예약 INSERT | 종료 |
|---|---|---|---|---|
| exec-1 / searcher03 (성공) | **0ms** | **1행** (VERSION 0→1) | res 125 | 커밋 (35.756) |
| exec-2 / searcher04 (거부) | **27ms** | **0행** | res 124 → 롤백 | 롤백 (35.769) |
| exec-3 / searcher02 (거부) | **27ms** | **0행** | res 126 → 롤백 | 롤백 |
| exec-4 / searcher01 (거부) | **27ms** | **0행** | res 127 → 롤백 | 롤백 |
| exec-6 / searcher05 (거부) | **27ms** | **0행** | res 123 → 롤백 | 롤백 |

**해석(SQL을 처음 보는 경우):**
- 5명 모두 슬롯을 `VERSION=0` 으로 읽고, 검증을 통과해(무락) **각자 예약 INSERT(헛 INSERT)** 까지 한 뒤
  점유 UPDATE를 flush 한다.
- **선착(exec-1)** 의 `UPDATE … WHERE VERSION=0` 은 1행 갱신 성공(VERSION 0→1) 후 커밋한다.
- 나머지 **4건의 같은 UPDATE 는 선착이 쥔 행 락을 기다린다**(`took=27ms`). 선착이 커밋(VERSION=1)한 뒤
  조건을 다시 보니 `VERSION=0` 인 행이 없어 **0행** → Hibernate가 이를 감지해 `OptimisticLockException`
  을 던진다 → 그 트랜잭션 전체(헛 INSERT 포함) 롤백.
- **D(조건부 UPDATE)와의 관계**: 메커니즘이 거의 동일하다 — 둘 다 *조건부 UPDATE가 행 락 대기 후 0행*.
  차이는 **조건이 업무 상태(`STATUS='AVAILABLE'`, D)가 아니라 버전 카운터(`VERSION=0`, C)** 이고, 0행을
  **프레임워크(JPA)가 예외(`OptimisticLockException`)로 변환**한다는 점이다(D는 repository가 affected=0
  을 돌려주고 앱이 거부 처리).
- **B(비관적 락)와의 차이**: B는 거부 요청이 읽기에서 막혀 헛 INSERT가 없다. C는 헛 INSERT까지 하고
  롤백한다 — 거부 비용이 가장 크다.

> ※ 위 SQL은 타임라인 시트의 "Oracle 실행형" 내용만 옮긴 것이다. 트랜잭션 경계 로그·원본 바인딩 SQL은
> 보고서 범위에서 제외했다(트랜잭션 종료 결과는 위 표의 "커밋/롤백" 으로만 표기).

- 거부된 4건이 **500(서버 에러)** 으로 드러난다. 응답 본문: `"Row was updated or deleted by another
  transaction (or unsaved-value mapping was incorrect) : [com.wherehouse.VisitReservation.entity.VisitSlotEntity#1170]"`.
- **D·B의 거부는 409(충돌)** 였다(`SlotUnavailableException` → 409 매핑). C의 거부는 `OptimisticLockException`
  이 **409로 매핑되지 않아 미처리 500**으로 나간다 — 같은 "거부"라도 클라이언트가 받는 신호가 다르다.

## 4. 메커니즘 해석
```
5명 동시 요청 (READ COMMITTED, 락 없음)
  ├ 5명 모두 슬롯 읽기(VERSION=0) + AVAILABLE 검증 통과(무락 스냅샷)      ← 아무도 안 막힘
  ├ 5명 모두 점유(setStatus) + 예약 INSERT(헛 INSERT)                    ← 락이 없어 전원 도달
  └ 커밋 flush: UPDATE … WHERE VERSION=0
        1건 : 행 락 즉시 획득 → 1행(VERSION 0→1) → 커밋
        4건 : (행 락 27ms 대기) → 선착 커밋 후 VERSION≠0 → 0행
              → OptimisticLockException → 헛 INSERT 포함 롤백 → (미처리) 500
  결과: 한 슬롯에 확정 예약 1건
```
낙관적 락은 "확인"과 "행위" 사이에 **락을 걸지 않는다**(그래서 5명이 동시에 읽고 검증·INSERT까지 자유롭게
진행). 충돌은 **커밋 단계의 버전 조건부 UPDATE**가 0행이 되는 순간에야 검출된다. 그 UPDATE 자체는 행 X-락을
쓰므로(후행 27ms 대기) 직렬화는 일어나지만, **거부 판정은 락이 아니라 버전 비교**가 한다. 차단 지점이
"쓰기(커밋 flush)"로 이동하고, 거부 요청은 헛 INSERT까지 한 뒤 무너지므로 **거부 비용이 4종 중 가장 크다**.

## 5. 결론
- **낙관적 락(@Version) 단독으로 단일 슬롯의 정합성 게이트를 통과한다(확정 1건).** 거부된 4건은 커밋
  단계의 버전 충돌(`OptimisticLockException`)로 롤백된다.
- D·B와 같은 "확정 1건" 을 만들지만 **차단 지점이 또 다르다**: D=쓰기(상태 조건 0행), B=읽기(FOR UPDATE
  대기), **C=커밋 flush(버전 조건 0행)**. 후행이 읽는 상태도 다르다: B=`RESERVED`, **C·D=`AVAILABLE`(무락)**.
- **비용·신호 측면의 두 발견**: ⑴ C의 거부 요청은 **헛 INSERT까지 수행 후 롤백**(읽기서 막는 B/쓰기 0행으로
  막는 D보다 낭비가 큼). ⑵ 거부가 깔끔한 409가 아니라 **미처리 500**으로 샌다 — 실무에서 낙관적 락은
  `OptimisticLockException`을 명시적으로 잡아 **409로 매핑하거나 재시도**해야 한다(현재 코드엔 그 처리가 없어 500).
- **적합성 관점(도메인 연결)**: 본 회차는 충돌이 확정되는 **단일 핫슬롯 최악 케이스**를 측정한 것이다. 이
  조건에서 낙관적 락은 헛작업 비용이 크고 재시도도 무의미(슬롯이 이미 RESERVED)하다. 반대로 **동시 충돌이
  드문 정상 트래픽**에서는 락을 잡지 않아 가볍다 — 즉 C의 적합성은 *실제 경합 빈도*에 달려 있다.
