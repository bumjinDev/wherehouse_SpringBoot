# F004 — 방문 예약 슬롯 동시성 제어 전략 결정 · 종합 (4종 비교 · 최종 채택)

> 본 문서는 "방문 예약 슬롯에 어떤 동시성 제어를 둘 것인가"라는 설계 결정의 **최종 종합 기록**이다.
> 후보 4종을 **동일 조건**에서 측정하고 **4개 관측 채널**로 교차 검증한 뒤 **② 조건부 UPDATE 를
> 채택**했다. 회차별 상세 근거는 1~4차 보고서에 있고, 본 문서는 그 위에서 **의사결정의 논리를
> 측정값으로 직접 추적**할 수 있게 정리한다.

## 1. 설계 질문과 판단 기준

### 1-1. 요구사항에서 설계 질문으로

방문 예약은 탐색자(searcher)가 매물을 직접 보러 가기 위해, 등록자(집주인)가 연 방문 윈도우
(`VISIT_WINDOW`)를 고정 길이로 나눈 **슬롯(`VISIT_SLOT`)** 하나를 예약(`VISIT_RESERVATION`)하는
기능이다. 요구사항 수집 단계에서 이 기능의 본질은 "신청"이 아니라 **물리적 자원(특정 시간대의 방문)을
한 사람에게 배타적으로 배정**하는 것임이 드러났다. 한 시간대 슬롯이 두 탐색자에게 동시에 확정되면
한 자리에 둘이 배정되어, 예약 기능이 제 역할을 못 하게 된다.

요구사항 분석을 통해 핵심 위험을 **확인-후-행위(check-then-act) 경합**으로 특정했다. 예약 로직은
본질적으로 "슬롯을 *읽어* 예약 가능한지 *확인*한 뒤 예약으로 *전이*"한다. 동시 요청이 들어오면
*확인*과 *행위* 사이의 간격에서 여러 요청이 모두 "예약 가능"을 보고 진입해 **이중 확정**이 발생할 수
있다(읽은 뒤 쓰기 전에 다른 요청이 끼어드는 경합).

```
T1: 슬롯 읽기 → status=AVAILABLE 확인        (제어가 없으면 둘 다 통과)
T2: 슬롯 읽기 → status=AVAILABLE 확인
T1: RESERVED 전이 + 예약 INSERT
T2: RESERVED 전이 + 예약 INSERT             → 한 슬롯에 확정 2건 (불변식 위반)
```

따라서 본 결정이 답해야 할 설계 질문은 다음이다.

> 한 슬롯에 여러 탐색자가 *같은 순간* 예약을 시도할 때, 시스템은 어떻게 **정확히 하나의 확정만**
> 남기고 나머지를 정상적인 거부 응답으로 처리할 것인가? 그리고 그 책임을 **어느 계층**(애플리케이션
> 검증 / 조건부 갱신 / DB 락 / 버전)에 두는 것이 이 도메인에 가장 적합한가?

격리 수준은 변수로 두지 않았다 — Oracle 기본 **READ COMMITTED** 고정. 같은 행에 대한 경합은 격리
수준 상향이 해법이 아니라(READ COMMITTED 에서도 단일 문장 갱신은 행 락으로 직렬화된다), 락 또는
조건부 갱신이 해법이기 때문이다.

### 1-2. 비즈니스 불변식

보장해야 하는 불변식은 **"한 슬롯의 확정(CONFIRMED) 예약은 어느 시점에나 최대 1건"** 이다. 이는
예약 기능이 성립하기 위한 기본 조건이다. 슬롯 하나에 확정이 2건 이상 남으면 한 자리에 두 사람이
배정되어, 예약 결과를 신뢰할 수 없게 된다.

이 불변식의 짝이 되는 조건이 **제어 기법 평가의 핵심**이다. **서로 다른 슬롯에 대한 예약 사이에는
순서 의존성이 없다.** 13시 슬롯과 14시 슬롯 예약은 경합할 이유가 없다. 그러므로 올바른 제어는 "모든
예약을 순서대로 처리"하는 것이 아니라 **같은 슬롯 충돌 범위에서만 직렬화**하는 것이다. 즉 좋은 후보의
조건은 **기술적 직렬화 범위 = 비즈니스 충돌 범위(같은 슬롯)** 의 일치다.

### 1-3. 비교 기준과 결론(채택표)

이번 의사결정의 비교 기준은 단순 응답 시간이 아니라 다음이다.

| 기준 | 의미 |
|---|---|
| 정합성 | 같은 슬롯 동시 예약 시 확정이 정확히 1건으로 유지되는가 |
| 차단 위치 | 거부가 읽기 / 쓰기 / 커밋 중 어디서 일어나는가 |
| 거부 신호 | 거부가 클라이언트에 의미 있는 신호(409 등)로 드러나는가 |
| 거부 비용 | 거부 요청이 헛작업(불필요한 INSERT·롤백)을 치르는가 |

**결론부터 말하면, ② 조건부 UPDATE 를 채택한다.** 근거의 핵심은 "정합성"이 아니다 — ②③④가 모두
"확정 1건"으로 **동률**이라 그것으로는 못 가른다. 갈리는 축은 **거부되는 요청을 어디서·어떤 비용으로·
어떤 신호로 처리하는가**이며, ②가 그 축에서 가장 우수하다.

| 후보 | 정합성(슬롯당 확정) | 거부 차단 위치 | 거부 신호 | 거부 요청의 헛작업 | 채택 |
|---|---|---|---|---|---|
| ① 무제어(baseline) | **5건 (붕괴)** | 없음 | 없음(전부 201) | 예약 INSERT 5건 커밋 | ✗ |
| **② 조건부 UPDATE** | **1건** | 쓰기(상태조건 0행) | **409 Conflict** | **없음** | **✔ 채택** |
| ③ 비관적 락(FOR UPDATE) | **1건** | 읽기(행 락 대기) | 409 Conflict | 없음 | ✗ |
| ④ 낙관적 락(@Version) | **1건** | 커밋 flush(버전 0행) | **500** | **헛 예약 INSERT 후 롤백** | ✗ |

②를 택한 이유는 ⑴ 거부에 헛작업이 없고(④ 대비), ⑵ 거부를 409 로 돌려주며(④ 대비), ⑶ 임계구간이
짧고 코드가 가장 단순하며(③ 대비), ⑷ "핫슬롯 = 경합 상시"라는 도메인 가정에 맞다(④ 대비). 속도
(throughput)는 근거에서 **제외**했다 — 동시 5건 규모로는 기법 간 속도 차를 공통 직렬화와 분리해
증명할 수 없기 때문이다(§6).

## 2. 측정 설계

### 2-1. 후보 4종

| # | 이름 | 한 줄 메커니즘 | 변경 지점 |
|---|---|---|---|
| ① | baseline (무제어) | 무조건 `UPDATE`(상태 조건·락 없음) | 읽기=무락 / 쓰기=무조건 |
| ② | 조건부 UPDATE | `UPDATE … WHERE status='AVAILABLE'` 로 검증을 갱신문에 원자적으로 합침 | 읽기=무락 / 쓰기=조건부 |
| ③ | 비관적 락 | `SELECT … FOR UPDATE` 로 읽기에서 행 X-락 선점 | 읽기=FOR UPDATE / 쓰기=무조건 |

### 2-4. 측정에 사용한 코드

`createReservation` 의 골격(검증 → 점유 → 예약 등록 → 알림)은 4종 공통이고, **읽기(1단계)와 쓰기
(점유, 6단계) 두 지점만** 교체했다.

**① baseline — 무조건 UPDATE**
```java
VisitSlotEntity slot = slotRepository.findById(slotId).orElseThrow(...);   // 무락 조회
if (slot.getStatus() != VisitSlotStatus.AVAILABLE) {                       // 확인-후-행위의 "확인"
    throw SlotUnavailableException.alreadyReserved(...);
}
// … 검증 …
slot.setStatus(VisitSlotStatus.RESERVED);
slotRepository.save(slot);   // UPDATE VISIT_SLOT SET STATUS='RESERVED' WHERE SLOT_ID=?  (상태 조건·락 없음)
```

**② 조건부 UPDATE (채택)** — 사전 `if` 를 없애고 상태 조건을 갱신문 `WHERE` 에 합쳐 게이트를 원자화.
```java
int affected = slotRepository.reserveSlotIfAvailable(slotId);   // UPDATE … WHERE SLOT_ID=? AND STATUS='AVAILABLE'
if (affected == 0) {
    throw SlotUnavailableException.alreadyReserved(...);        // 거부(409). 예약 INSERT 에 도달하지 않음
}
// 7단계: 예약 INSERT (점유 성공자만 도달)
```

**③ 비관적 락** — 읽기를 `SELECT … FOR UPDATE` 로 바꿔 읽는 순간 행 X-락 선점.
```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT s FROM VisitSlotEntity s WHERE s.slotId = :slotId")
Optional<VisitSlotEntity> findByIdForUpdate(@Param("slotId") Long slotId);
```

**④ 낙관적 락** — 락 없이 엔티티에 `@Version`. Hibernate 가 갱신문에 버전 조건을 자동으로 붙인다.
```java
@Version
@Column(name = "VERSION", nullable = false)
private Long version;
// 커밋 flush 시: UPDATE … SET STATUS=?, VERSION=VERSION+1 WHERE SLOT_ID=? AND VERSION=?
```

> **SQL 형태 메모(관측 사실).** ①③④의 슬롯 갱신은 엔티티 dirty-checking(`setStatus`+`save`)이라
> 별칭이 없다(`update VISIT_SLOT set STATUS=… where SLOT_ID=…`). ②만 `@Modifying` JPQL 벌크 갱신이라
> Hibernate 가 별칭을 부여한 형태로 발행된다. 발행 경로가 달라 형태가 다른 것이며, 동시성 거동과는
> 무관하다.

## 3. 회차별 측정 결과 요약

각 후보의 핵심 결과만 싣는다(원본 행·전체 SQL·`took` 타임라인 등 상세는 회차 보고서 1~4 및 부록 원자료).

### 3-1. ① baseline (붕괴) · slot 1218

- **정합성:** slot 1218 에 확정 5건(res 81~85), 모두 동일 확정 시각. → 불변식 위반.
- **단계 분포:** 읽기 5/5 · 점유 5/5 · 등록 5/5(거부 0). 무조건 UPDATE 가 5건 모두 1행 갱신 성공.
- **발행 SQL:** 슬롯 갱신 `took` 1~66ms 로 벌어짐 = 같은 행이라 행 락이 *순차 실행*시킬 뿐 **거부는
  안 함**(직렬화는 일어나지만 차단은 없다).
- **JMeter:** 5건 모두 201, 오류 0%. → **외부 응답만으로는 정합성 붕괴가 드러나지 않는다.**

### 3-2. ② 조건부 UPDATE (채택) · slot 1221

- **정합성:** 확정 1건(searcher03, res 87). → 충족.
- **단계 분포:** 읽기 5/5 → 점유 **1/5** → 등록 1/5. 거부 4건 모두 **쓰기(점유) 단계**에서 차단.
- **발행 SQL:** 성공 `UPDATE` 0ms·**1행**; 거부 4건 `UPDATE` 24ms·**0행** → 예약 INSERT 미도달 →
  롤백(**헛작업 없음**). `AND STATUS='AVAILABLE'` 한 조건이 거부의 직접 원인(선착 커밋 후 후착은 0행).
- **JMeter:** 1×201 + 4×**409**, 오류율 80%. (1차의 "5×201"과 정반대 — 붕괴가 외부에 드러남)

### 3-3. ③ 비관적 락 · slot 1237

- **정합성:** 확정 1건(searcher05, res 101). → 충족.
- **단계 분포:** 확인 조회 도달 5/5이나 **그 시점 인지 상태 = AVAILABLE 1 / RESERVED 4.** 거부 4건은
  읽기 직후 `AVAILABLE` 검증에서 차단(②④와 갈리는 지점 — 후착이 옛값이 아닌 최신 RESERVED 를 봄).
- **발행 SQL:** 성공 `FOR UPDATE` 2ms(즉시 획득); 거부 4건 `FOR UPDATE` **73·97·112·128ms**(행 락 대기)
  후 RESERVED 읽고 검증 실패 → UPDATE·INSERT 미도달 → 롤백(헛작업 없음). 대기가 ②(24ms)보다 큰 이유는
  락 보유 구간이 *읽기~커밋 전 구간*이기 때문. 이 `took` 가 실제 행 락 블로킹임은 DB 2세션으로 직접
  재현(`3_1. 3차_B_비관적락_근거_2세션_락재현.md`).
- **JMeter:** 응답 코드 1×201 + 4×**409**(DB·앱계측에서 도출). **응답시간·처리량은 미확보** — 확보된
  JMeter 화면이 다른 런(slot 1240)이라 본 회차에 결합하지 않고 공란으로 둔다.

### 3-4. ④ 낙관적 락 · slot 1170

- **정합성:** 최종 확정 1건(searcher03, res 125), 슬롯 `VERSION=1`. → 충족(나머지 4건은 INSERT 후
  롤백되어 테이블에 없음).
- **단계 분포:** 확인 조회 5/5 → 점유 **5/5**(락이 없어 전원 도달) → 등록 1/5. 거부 4건은 **커밋 flush
  의 버전 충돌**로 무너짐(메서드 밖). → 단계 분포만으로는 "어디서 막혔나"를 못 보여줌, 정합성·거부는
  DB 잔여·발행 SQL 로 판정.
- **발행 SQL:** 5명 모두 **헛 예약 INSERT(res 123~127)** 발행. 성공(exec-1) `UPDATE … WHERE VERSION=0`
  0ms·1행; 거부 4건 같은 UPDATE 27ms·**0행** → `OptimisticLockException` → 헛 INSERT 포함 롤백
  (res 123·124·126·127 미잔류). **②와 형제 메커니즘**(조건부 UPDATE 가 행 락 대기 후 0행)이나, 조건이
  버전 카운터이고 0행을 프레임워크가 예외로 변환하며 **거부가 헛 INSERT 까지 한다**는 점이 다름.
- **JMeter:** 1×201 + 4×**500**(Internal Server Error), 오류율 80%. `OptimisticLockException` 이 409 로
  매핑되지 않아 미처리 500 으로 샘.

## 4. 4종 교차 분석 — 정합성은 동률, "거부 처리의 구조"로 비교

①은 확정 5건 붕괴로 탈락이 자명하다. **②③④는 정합성이 모두 "확정 1건"으로 동률** 이므로 정합성으로는
가를 수 없다. 가르는 축은 **거부되는 요청을 어디서·어떤 비용으로·어떤 신호로 처리하는가**다.

| 축 | ② 조건부 UPDATE | ③ 비관적 락 | ④ 낙관적 락 |
|---|---|---|---|
| 차단 위치 | 쓰기(상태조건 0행) | **읽기**(FOR UPDATE 대기) | 커밋 flush(버전 0행) |
| 락 보유 구간(임계구간) | 쓰기~커밋(짧음) | **읽기~커밋(김)** | 쓰기~커밋(짧음) |
| 거부 후행 대기(측정 `took`) | ~24ms | **73~128ms** | ~27ms |
| 거부 요청 헛작업 | **없음** | 없음 | **헛 예약 INSERT 후 롤백** |
| 거부 신호 | `affected=0` → **409** | RESERVED 읽고 검증 실패 → **409** | `OptimisticLockException` → **500** |
| 후행이 읽는 슬롯 상태 | AVAILABLE(무락) | **RESERVED**(락 후 현재값) | AVAILABLE(무락) |
| 구현 비용 | `WHERE` 조건 1줄 | `@Lock` / FOR UPDATE | `@Version` 컬럼 + DDL + **예외처리 필요** |

```
차단 위치가 기법마다 이동한다:
  ① 없음(붕괴) ── ② 쓰기 ── ③ 읽기 ── ④ 커밋
정합성(확정 1)은 ②③④ 동률. 갈리는 건 "거부의 비용·신호·시점"이다.
```

- **②와 ④는 형제** — 둘 다 *조건부 UPDATE 가 행 락 대기 후 0행*(후행 `took` ②24ms ≈ ④27ms). 차이는
  조건(상태 vs 버전)·신호(409 vs 500)·헛작업(없음 vs 헛 INSERT 후 롤백).
- **③만 읽기에서 막아** 후행이 헛 INSERT 를 안 한다(②와 동일 장점). 대신 **락을 읽기~커밋 전 구간
  보유**해 임계구간이 가장 길다(후행 대기 73~128ms vs ②의 ~24ms).
- **④는 후행이 헛 INSERT 까지 하고 롤백** → 거부 비용 최대 + 신호가 500 이라 **반드시 예외처리
  (409 매핑/재시도)** 가 필요하다.

## 5. 기술 의사결정 — ② 조건부 UPDATE 채택

정합성이 동률이므로, 선택은 §4 표의 **거부 처리 구조**에서 갈린다.

### 5-1. ② vs ④ (낙관적 락을 택하지 않은 이유)

**핵심 이유 — "선지불 후 사후해결" 구조가 핫슬롯에 안 맞는다.** ④ 낙관적 락은 "충돌은 드물다"는
낙관적 전제 위에서, 5건이 모두 검증·예약 INSERT 까지 **일단 수행한 뒤** 커밋 시점에야 버전 충돌을
*사후*에 검출한다. 즉 매 요청이 작업 비용(점유 UPDATE·예약 INSERT)을 먼저 지불하고 충돌은 나중에
해결하는 구조다. 그런데 단일 핫슬롯 동시 예약은 **경합이 상시**여서 그 전제가 정면으로 깨진다 —
결국 거부될 4건이 매번 헛작업을 치르고 커밋에서 무너진다. 이 구조적 부적합이 ④를 택하지 않은 가장
큰 이유이며, 측정으로도 다음이 뒷받침된다.

- ④의 거부 요청은 *헛 예약 INSERT + 롤백*을 한다(res 123·124·126·127 INSERT 후 롤백). ②는 갱신 0행에서
  끊겨 그 INSERT 를 안 한다 → **거부 비용이 ②가 더 낮다.**
- ④의 거부 신호는 `OptimisticLockException` 이 409 로 매핑되지 않아 **500** 으로 나갔다. ②는
  `affected=0` → **409**.
- ④는 `@Version` 컬럼·DDL·예외처리가 붙는다. ②는 `WHERE` 한 줄.
- **도메인 가정:** ④의 전제는 "충돌이 드물다"인데, **단일 핫슬롯 동시 예약은 경합이 상시**라 그 전제가
  깨진다. ②(쓰기 시점 원자적 게이트)는 이 도메인에 직접 맞다.

### 5-2. ② vs ③ (비관적 락 대비)

**핵심 이유 — 락 보유 구간이 불필요하게 길다.** ③ 비관적 락은 슬롯을 *읽는 순간*부터 락을 쥐고
그 락을 **커밋까지** 놓지 않는다. 그래서 실제 충돌 여부와 무관하게 락 보유 구간이 검증·INSERT·알림
준비를 전부 덮는 *읽기~커밋 전 구간*으로 길어진다(후착 대기 73~128ms vs ②의 ~24ms). 반면 ②는 락을
*쓰기 시점*에만, 그것도 `갱신~커밋` 의 짧은 구간만 잡는다. 정합성·거부 품질이 같다면 **락을 더 짧게
쥐는 쪽이 핫슬롯 직렬화에 유리하다.**

- 둘 다 거부에 헛작업이 없고 거부 신호가 409 로 같다. 차이는 ③이 락을 *읽기부터 커밋까지* 쥐어
  **임계구간이 가장 길고**(측정 `took` 73~128ms vs ②의 ~24ms), `@Lock` 을 요구한다는 점이다.
- 즉 **②는 ③과 같은 거부 품질(헛작업 0 · 409)을 더 짧은 임계구간과 더 단순한 코드로** 얻는다.

### 5-3. 종합

②는 거부 비용 0 · 거부 신호 409 · 구현 단순 · 도메인 가정 부합의 네 축에서 ③·④보다 유리하거나
대등하다. ④는 거부 비용·신호·복잡도에서 불리하고, ③은 임계구간·복잡도에서 ②보다 무겁다. 또한 ②의
직렬화 범위(같은 슬롯 행)는 비즈니스 충돌 범위(같은 슬롯)와 정확히 일치한다. → **② 조건부 UPDATE
채택.**

속도(throughput)는 채택 근거에서 제외했다. 동시 5건 규모에서는 기법 간 응답시간 차이를 세 기법 공통의
직렬화(락 대기)와 분리해 측정할 수 없기 때문이다(§6).

## 6. 측정의 한계와 적용 범위(정직한 명시)

- **의도된 최악 케이스만 측정.** N=5 동일 핫슬롯 동시 출발은 경합을 *강제로 100% 재현*한 것이다. 실제
  서비스에서 같은 슬롯에 *진짜 동시* 요청이 얼마나 잦은지(경합 빈도)는 **프로덕션 트래픽이라야** 안다.
  특히 ④(낙관적 락)의 적합성은 *실제 경합 빈도*에 달려 있다(경합이 드물면 락을 안 잡아 가볍다).
- **기법 간 throughput 비교 불가.** 세 기법 모두 지배 요인이 *같은 직렬화*이고, 그 위의 기법별 차이
  (④의 헛 INSERT, ③의 긴 임계구간)는 이 규모에서 응답시간 노이즈에 묻힌다. 그래서 속도를 결정 근거로
  쓰지 않았다. 회차마다 슬롯·런이 달라 절대 수치의 회차 간 비교도 *방향성*으로만 유효하다.

## 7. 최종 결론

> 방문 예약 슬롯 동시성의 1차 방어선으로 **② 조건부 UPDATE** 를 채택한다. 읽기-검증-쓰기를 한
> 갱신문으로 원자화해 경합을 *쓰기 시점*에 게이트하고, 거부를 헛작업 없이 409 로 돌려주며, 구현이
> 가장 단순하고 직렬화 범위가 비즈니스 충돌 범위(같은 슬롯)와 일치한다. 이 결정은 속도가 아니라
> **거부 처리 구조**라는 측정 가능한 근거에 둔다.

이 결정의 본질은 "락을 걸었다"가 아니다. 정합성은 ②③④가 모두 만들 수 있었고(확정 1건 동률),
진짜 질문은 **"같은 정합성을 어느 계층에서·어느 비용으로·어떤 거부 신호로 달성하는가"** 였다.
무제어(①)가 보여준 "응답은 정상인데 데이터만 깨지는" 현상은 동시성 제어가 선택이 아니라 필수임을 실증했고, 4종 비교는 그
필수 제어를 **거부 처리 구조가 가장 깔끔하고 도메인에 맞는 ②** 로 좁혔다.
