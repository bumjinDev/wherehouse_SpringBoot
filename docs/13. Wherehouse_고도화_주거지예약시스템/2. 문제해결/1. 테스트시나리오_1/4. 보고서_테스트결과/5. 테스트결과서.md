> **이 문서 하나로 완결된다.** 비즈니스 문제 → 측정 설계 → **실제 측정 절차** → 측정에 쓴 코드 →

## 0. 개요

### 0.1 문제와 결론(한 줄)
단일 슬롯에 여러 사용자가 *같은 순간* 예약을 시도하면 **"한 슬롯의 확정 예약은 어느 시점에나 1건"**
이라는 비즈니스 불변식이 깨질 수 있다(이중 확정). 동시성 제어 후보 **4종**을 **동일 조건**에서 측정하고
**4개 관측 채널**로 교차 검증한 뒤, **② 조건부 UPDATE를 채택**했다.

### 0.2 채택 결과표

| 후보 | 정합성(슬롯당 확정 수) | 거부 차단 위치 | 거부 신호 | 거부 요청의 헛작업 | 채택 |
|---|---|---|---|---|---|
| ① 무제어(baseline) | **5건 (붕괴)** | 없음 | 없음(전부 201) | 예약 INSERT 5건 커밋 | ✗ |
| **② 조건부 UPDATE** | **1건** | 쓰기(상태조건 0행) | **409 Conflict** | **없음** | **✔ 채택** |
| ③ 비관적 락(FOR UPDATE) | **1건** | 읽기(행 락 대기) | 409 Conflict | 없음 | ✗ |
| ④ 낙관적 락(@Version) | **1건** | 커밋 flush(버전 0행) | **500** | **헛 예약 INSERT 후 롤백** | ✗ |

### 0.3 채택 근거(측정으로 증명된 것만)
정합성은 ②③④가 모두 "확정 1건"으로 **동률**이라 그것으로는 못 가른다. 갈리는 축은 **거부되는 요청을
어디서·어떤 비용으로·어떤 신호로 처리하는가**이며, **②는 ⑴ 거부에 헛작업이 없고(④ 대비), ⑵ 거부를
409로 돌려주며(④ 대비), ⑶ 임계구간이 짧고 코드가 가장 단순하며(③ 대비), ⑷ "핫슬롯=경합 상시"라는
도메인 가정에 맞다.** 속도(throughput)는 근거에서 **제외**했다 — 본 측정 규모(동시 5건)로는 기법 간
속도 차를 공통 직렬화와 구분해 증명할 수 없기 때문이다(§8).

## 1. 문제 정의 — 비즈니스 맥락과 위험

**도메인.** 매물 방문 예약: 등록자가 연 방문 윈도우(`VISIT_WINDOW`)를 고정 길이 **슬롯**(`VISIT_SLOT`)으로
나누고, 탐색자(searcher)가 슬롯을 예약(`VISIT_RESERVATION`)한다. 핵심 불변식은
**"한 슬롯의 확정(CONFIRMED) 예약은 어느 시점에나 최대 1건"** 이다.

**위험 — 확인-후-행위(check-then-act) 경합.** 예약 로직은 본질적으로 "슬롯을 *읽어* 예약 가능한지 *확인*한
뒤, 예약으로 *전이*"한다. 동시 요청이 들어오면 *확인*과 *행위* 사이의 시간 간격에서 여러 요청이 모두
"예약 가능"을 보고 진입해 **이중 확정**이 발생할 수 있다(lost update / TOCTOU).

```
T1: 슬롯 읽기 → status=AVAILABLE 확인        (락이 없으면 둘 다 통과)
T2: 슬롯 읽기 → status=AVAILABLE 확인
T1: RESERVED 로 전이 + 예약 INSERT
T2: RESERVED 로 전이 + 예약 INSERT           → 한 슬롯에 확정 2건 (불변식 위반)
```

이 위험을 **무엇으로, 어디서 막을 것인가**가 본 측정의 결정 대상이다. 격리 수준은 변수로 두지 않았다 —
Oracle 기본 **READ COMMITTED** 고정. 같은 행에 대한 경합은 격리 수준 상향이 해법이 아니라(READ COMMITTED
에서도 단일 문장 갱신은 행 락으로 직렬화된다), 락 또는 조건부 갱신이 해법이기 때문이다.

## 3. 측정 절차(실제 수행 순서)

각 회차는 아래 **6단계**를 동일하게 수행했다.

1. **사전 정리** — `searcher01~05` 계정이 대상 매물에 가진 확정 예약을 0건으로 정리한다(같은 매물·임대
   유형의 사전 예약이 남아 있으면 슬롯 경합이 아니라 *중복 예약 검증*(E7005)에서 먼저 걸려 측정이 오염된다.
   실제로 ④ 1차 시도에서 이 오염이 발생했다 — §5.4 절차 메모).
2. **대상 슬롯 선택** — window 81 에서 `STATUS='AVAILABLE'`인 미예약 슬롯 1개를 고른다(회차별로 다른
   슬롯: ① 1218 · ② 1221 · ③ 1237 · ④ 1170).
3. **코드 배치·기동** — `createReservation`의 읽기·쓰기 2지점을 해당 회차 기법으로 교체하고(§4),
   `gradlew` 재빌드 후 애플리케이션을 기동한다. ④는 `VISIT_SLOT`에 `VERSION` 컬럼·엔티티 `@Version`을
   추가하는 DDL/엔티티 변경을 동반한다.
4. **동시 부하 실행** — JMeter에서 대상 슬롯 id를 CSV에 반영하고 5스레드를 Synchronizing Timer로 동시
   출발시킨다.
5. **로그 수집·변환** — 앱 단계 로그를 `report.py`로 → `F004_측정결과_*.xlsx`, P6Spy 로그를
   `build_p6spy_xlsx.py`로 → `*_타임라인.xlsx` 산출.
6. **DB 잔여 확인** — SQL Developer(`wherehouse_localhost`)로 `VISIT_RESERVATION`(슬롯당 확정 수)과
   `VISIT_SLOT`(상태/버전)을 조회해 정합성을 확정한다.

> 회차마다 슬롯과 실행 시각이 다르므로, **절대 응답시간의 회차 간 직접 비교는 방향성으로만** 유효하다
> (§8). 비교의 본질은 "같은 5건 동시 경합에서 거부가 어떻게 처리되는가"의 *구조*다.

## 4. 측정에 사용한 코드

`createReservation`의 골격(검증 6종 → 점유 → 예약 등록 → 알림)은 4종 공통이고, **읽기(1단계)와
쓰기(점유, 6단계) 두 지점만** 아래처럼 교체했다.

### 4.1 ① baseline — 무조건 UPDATE
```java
// 1단계: 슬롯 읽기 (무락 조회)
VisitSlotEntity slot = slotRepository.findById(slotId).orElseThrow(...);
// 2단계: AVAILABLE 확인 (확인-후-행위의 "확인")
if (slot.getStatus() != VisitSlotStatus.AVAILABLE) {
    throw SlotUnavailableException.alreadyReserved(...);
}
// … 자기매물·중복예약·시간겹침 검증 …
// 6단계: 무조건 UPDATE — 상태 조건도 락도 없음
slot.setStatus(VisitSlotStatus.RESERVED);
slotRepository.save(slot);            // → UPDATE VISIT_SLOT SET STATUS='RESERVED' WHERE SLOT_ID=?
// 7단계: 예약 INSERT
```

### 4.2 ② 조건부 UPDATE (채택)
사전 `if` 검증을 없애고, **상태 조건을 갱신문 `WHERE`에 합쳐** 게이트를 한 문장으로 원자화한다.
```java
// 1단계: 슬롯 읽기 (무락 조회 — 윈도우 식별용. AVAILABLE 게이트는 6단계가 담당)
VisitSlotEntity slot = slotRepository.findById(slotId).orElseThrow(...);
// 2단계: 사전 AVAILABLE 검증 없음
// … 자기매물·중복예약·시간겹침 검증 …
// 6단계: 슬롯 점유 — 조건부 UPDATE. 후착은 affected=0 → 여기서 차단(차단 위치=쓰기)
int affected = slotRepository.reserveSlotIfAvailable(slotId);
if (affected == 0) {
    throw SlotUnavailableException.alreadyReserved(...);   // 거부(409). 예약 INSERT 에 도달하지 않음
}
// 7단계: 예약 INSERT (점유 성공자만 도달)
```
```java
// VisitSlotRepository
@Modifying
@Query("UPDATE VisitSlotEntity s SET s.status = com.wherehouse.VisitReservation.entity.VisitSlotStatus.RESERVED " +
       "WHERE s.slotId = :slotId AND s.status = com.wherehouse.VisitReservation.entity.VisitSlotStatus.AVAILABLE")
int reserveSlotIfAvailable(@Param("slotId") Long slotId);
```

### 4.3 ③ 비관적 락
읽기를 `SELECT … FOR UPDATE`로 바꿔 **읽는 순간 행 X-락**을 선점한다.
```java
// 1단계: 읽기에서 행 X-락 (후착은 락 대기 후 RESERVED 를 읽는다)
VisitSlotEntity slot = slotRepository.findByIdForUpdate(slotId).orElseThrow(...);
// 2단계: 잠금 획득 후 읽은 상태로 AVAILABLE 검증 (후착은 여기서 거부)
if (slot.getStatus() != VisitSlotStatus.AVAILABLE) {
    throw SlotUnavailableException.alreadyReserved(...);
}
// … 6단계: 무조건 UPDATE(락은 1단계에서 이미 보유) …
```
```java
// VisitSlotRepository
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT s FROM VisitSlotEntity s WHERE s.slotId = :slotId")
Optional<VisitSlotEntity> findByIdForUpdate(@Param("slotId") Long slotId);
```

### 4.4 ④ 낙관적 락
락을 걸지 않는다. 엔티티에 `@Version`을 두면 Hibernate가 갱신문에 버전 조건을 자동으로 붙인다.
```java
// 1단계: 무락 findById (5명 모두 같은 version 으로 읽음)
VisitSlotEntity slot = slotRepository.findById(slotId).orElseThrow(...);
// 2단계: AVAILABLE 검증 (무락 스냅샷 — 5명 모두 통과)
if (slot.getStatus() != VisitSlotStatus.AVAILABLE) { throw ...; }
// … 6단계 …
slot.setStatus(VisitSlotStatus.RESERVED);
slotRepository.save(slot);   // 커밋 flush 시: UPDATE … SET STATUS=?, VERSION=VERSION+1 WHERE SLOT_ID=? AND VERSION=?
```
```java
// VisitSlotEntity
@Version
@Column(name = "VERSION", nullable = false)
private Long version;
```

> **SQL 형태 메모(관측 사실).** ①③④의 슬롯 갱신은 엔티티 dirty-checking(`setStatus`+`save`)이라
> 별칭이 없다(`update VISIT_SLOT set STATUS=… where SLOT_ID=…`). ②만 `@Modifying` JPQL 벌크 갱신이라
> Hibernate가 엔티티에 별칭을 부여한 형태(`update VISIT_SLOT vse1_0 … where vse1_0.SLOT_ID=…`)로
> 발행된다. 같은 테이블 갱신이지만 발행 경로가 달라 형태가 다른 것이며, 동시성 거동과는 무관하다.

**성공 트랜잭션의 전체 SQL 흐름(exec-4, 14건 → COMMIT)** — 한 예약 트랜잭션이 실제로 발행하는 SQL 전부:
```sql

-- .899 took=3ms   윈도우 조회
select … from VISIT_WINDOW where WINDOW_ID=81;
-- .906 took=1ms   슬롯의 예약 조회(계측용) → reservationsCount=0
select … from VISIT_RESERVATION where SLOT_ID=1237 order by CONFIRMED_AT desc;
-- .910 took=2ms   매물(전세) 조회
select … from PROPERTIES_CHARTER where PROPERTY_ID='693c35731a5ebe74463679d39787c0c5';

                      and PROPERTY_ID='693c35731a5ebe74463679d39787c0c5' and LEASE_TYPE='CHARTER';
-- .927 took=1ms   시간 겹침 예약 수(COUNT)

                      and START_TIME<TIMESTAMP '2026-06-09 23:35:00.000'
```

**거부 트랜잭션의 전체 SQL 흐름(exec-1, 5건 → ROLLBACK)** — 슬롯 `UPDATE`·예약 `INSERT`에 도달하지 않는다:
```sql
-- .961 took=73ms  슬롯 읽기+배타적 락 (선착이 커밋할 때까지 대기)

-- .962 took=0ms   윈도우 조회
select … from VISIT_WINDOW where WINDOW_ID=81;
-- .966 took=0ms   슬롯의 예약 조회(계측용) → reservationsCount=1 (이미 RESERVED·확정 1건)

-- (AVAILABLE 검증 실패 → 거부 응답에 담을 "같은 매물의 다른 가용 슬롯" 조회)
-- .970 took=1ms
```

> 후착이 풀린 직후 `FOR UPDATE`가 읽는 값이 선착의 커밋값(`RESERVED`)임도 함께 확인했다(current-mode read).

#### ⑴ 정합성 — DB 잔여 원본 (`VISIT_RESERVATION`, slot 1170)
**최종 확정 예약 1건**만 남았다(나머지 4건은 INSERT 후 롤백되어 테이블에 없다 — ⑵ 참조).

| RESERVATION_ID | SEARCHER_USER_ID | SLOT_ID | STATUS | CONFIRMED_AT |
|---|---|---|---|---|
| 125 | searcher03 | 1170 | CONFIRMED | 2026-06-04 16:00:35.662 |

`VISIT_SLOT` 1170: `STATUS='RESERVED'`, **`VERSION=1`**. → **확정 1건 = 규칙 충족.**

#### ⑵ 발행 SQL (Oracle 실행형, `4_C_낙관적락_타임라인.xlsx`)
**5명 모두 예약 INSERT(헛 INSERT)까지 발행**한 뒤, 버전 조건부 UPDATE에서 갈린다:
```sql
-- 슬롯 점유: @Version 이 WHERE 에 VERSION=? 를 붙인 갱신 (5건 모두 시도)
update VISIT_SLOT set STATUS='RESERVED', VERSION=1 where SLOT_ID=1170 and VERSION=0;
```

트랜잭션별 결과(5명 모두 `VERSION=0` 읽음, 5건 모두 헛 INSERT):

| 트랜잭션(스레드/searcher) | 예약 INSERT (시각/took/res_id) | 버전조건 UPDATE (시각/took/행수) | 종료 |
|---|---|---|---|
| exec-1 / searcher03 (성공) | 16:00:35.727 / 0ms / **res 125** | 16:00:35.730 / **0ms** / **1행** (VERSION 0→1) | **커밋** (.756) |
| exec-2 / searcher04 (거부) | 16:00:35.727 / 1ms / res 124 | 16:00:35.756 / 27ms / **0행** | 롤백 |
| exec-3 / searcher02 (거부) | 16:00:35.727 / 0ms / res 126 | 16:00:35.756 / 27ms / **0행** | 롤백 |
| exec-4 / searcher01 (거부) | 16:00:35.727 / 0ms / res 127 | 16:00:35.756 / 27ms / **0행** | 롤백 |
| exec-6 / searcher05 (거부) | 16:00:35.727 / 0ms / res 123 | 16:00:35.756 / 27ms / **0행** | 롤백 |

거부 4건은 모두 동일 예외로 롤백:
```
org.hibernate.StaleObjectStateException: Row was updated or deleted by another transaction
  (or unsaved-value mapping was incorrect) : [com.wherehouse.VisitReservation.entity.VisitSlotEntity#1170]
```

- 5명 모두 `VERSION=0`을 읽고 검증을 통과(무락)해 **각자 예약 INSERT(헛 INSERT, res 123~127)** 까지 한 뒤
  점유 UPDATE를 flush 한다.
- **선착(exec-1)** 의 `UPDATE … WHERE VERSION=0`은 1행 갱신(VERSION 0→1) 후 커밋(res 125만 테이블에 잔류).
- 나머지 **4건의 같은 UPDATE는 선착이 쥔 행 락을 기다린다(`took=27ms`).** 선착 커밋 후 조건을 다시 보니
  `VERSION=0`인 행이 없어 **0행** → Hibernate가 `OptimisticLockException`을 던져 그 트랜잭션 전체(헛 INSERT
  포함) 롤백 → res 123·124·126·127은 테이블에 남지 않는다.
- **②와의 관계(형제 메커니즘):** 둘 다 *조건부 UPDATE가 행 락 대기 후 0행*. 차이는 조건이 업무 상태
  (`STATUS='AVAILABLE'`, ②)가 아니라 **버전 카운터(`VERSION=0`, ④)** 이고, 0행을 **프레임워크가 예외로
  변환**한다는 점, 그리고 **거부 요청이 헛 INSERT를 한다**는 점(②는 0행에서 끊겨 INSERT에 도달 안 함)이다.

## 6. 4종 교차 분석 — 정합성은 동률, "거부 처리의 구조"로 비교

①은 확정 5건 붕괴로 탈락이 자명하다. **②③④는 정합성이 모두 "확정 1건"으로 동률** 이므로, 정합성으로는
가를 수 없다. 가르는 축은 **거부되는 요청을 어디서·어떤 비용으로·어떤 신호로 처리하는가**다.

| 축 | ② 조건부 UPDATE | ③ 비관적 락 | ④ 낙관적 락 |
|---|---|---|---|
| 차단 위치 | 쓰기(상태조건 0행) | **읽기**(FOR UPDATE 대기) | 커밋 flush(버전 0행) |
| 락 보유 구간(임계구간) | 쓰기~커밋(짧음) | **읽기~커밋(김)** | 쓰기~커밋(짧음) |
| 거부 후행 대기(측정 `took`) | ~24ms | **73~128ms** | ~27ms |
| 거부 요청 헛작업 | **없음** | 없음 | **헛 예약 INSERT 후 롤백** |
| 거부 신호 | `affected=0` → **409** | RESERVED 읽고 검증 실패 → **409** | `OptimisticLockException` → **500** |
| 후행이 읽는 슬롯 상태 | AVAILABLE(무락) | **RESERVED**(락 후 현재값) | AVAILABLE(무락) |
| 구현 비용 | `WHERE` 조건 1줄 | `@Lock` / FOR UPDATE | `@Version` 컬럼 + DDL + **예외처리 필요** |
| DB 잔여 원본(확정) | res 87 (1행) | res 101 (1행) | res 125 (1행, +헛 INSERT 4건 롤백) |

```
차단 위치가 기법마다 이동한다:
  ① 없음(붕괴) ── ② 쓰기 ── ③ 읽기 ── ④ 커밋
정합성(확정 1)은 ②③④ 동률. 갈리는 건 "거부의 비용·신호·시점"이다.
```

- **②와 ④는 형제** — 둘 다 *조건부 UPDATE가 행 락 대기 후 0행*(후행 `took` ②24ms ≈ ④27ms). 차이는
  조건(상태 vs 버전)·신호(409 vs 500)·헛작업(없음 vs 헛 INSERT 후 롤백).
- **③만 읽기에서 막아** 후행이 헛 INSERT를 안 한다(②와 동일 장점). 대신 **락을 읽기~커밋 전 구간 보유**해
  임계구간이 가장 길다(후행 대기 73~128ms vs ②의 ~24ms).
- **④는 후행이 헛 INSERT까지 하고 롤백** → 거부 비용 최대 + 신호가 500이라 **반드시 예외처리(409 매핑/재시도)** 가 필요하다.

## 7. 기술 의사결정 — ② 조건부 UPDATE 채택

정합성이 동률이므로, 선택은 §6 표의 **거부 처리 구조**에서 갈린다.

**② vs ④ (낙관적 락을 택하지 않은 이유)**
- ④의 거부 요청은 *헛 예약 INSERT + 롤백*을 한다(타임라인의 res 123·124·126·127 INSERT 후 롤백). ②는
  갱신 0행에서 끊겨 그 INSERT를 안 한다 → **거부 비용이 ②가 더 낮다.**
- ④의 거부 신호는 `OptimisticLockException`이 409로 매핑되지 않아 **500**으로 나갔다. ②는
  `affected=0` → **409**.
- ④는 `@Version` 컬럼·DDL·예외처리가 붙는다. ②는 `WHERE` 한 줄.
- **도메인 가정**: ④(낙관적 락)의 전제는 "충돌이 드물다"인데, **단일 핫슬롯 동시 예약은 경합이 상시**라
  그 전제가 깨진다. ②(쓰기 시점 원자적 게이트)는 이 도메인에 직접 맞다.

**② vs ③ (비관적 락 대비)**
- 둘 다 거부에 헛작업이 없고 거부 신호가 409로 같다. 차이는 ③이 락을 *읽기부터 커밋까지* 쥐어
  **임계구간이 가장 길고**(측정 `took` 73~128ms vs ②의 ~24ms), `@Lock`을 요구한다는 점이다.
- 즉 **②는 ③과 같은 거부 품질(헛작업 0 · 409)을 더 짧은 임계구간과 더 단순한 코드로** 얻는다.

**종합** — ②는 거부 비용 0 · 거부 신호 409 · 구현 단순 · 도메인 가정 부합의 네 축에서 ③·④보다
유리하거나 대등하다. ④는 거부 비용·신호·복잡도에서 불리하고, ③은 임계구간·복잡도에서 ②보다 무겁다.
→ **② 조건부 UPDATE 채택.**

속도(throughput)는 채택 근거에서 제외했다. 동시 5건 규모에서는 기법 간 응답시간 차이를 세 기법 공통의
직렬화(락 대기)와 분리해 측정할 수 없기 때문이다(§8).

## 8. 측정의 한계와 적용 범위(정직한 명시)

결정의 신뢰성을 위해, 이 측정이 **증명한 것과 증명하지 못한 것**을 분명히 한다.

- **의도된 최악 케이스만 측정.** N=5 동일 핫슬롯 동시 출발은 경합을 *강제로 100% 재현*한 것이다. 실제
  서비스에서 같은 슬롯에 *진짜 동시* 요청이 얼마나 잦은지(경합 빈도)는 **프로덕션 트래픽이라야** 안다.
  특히 ④(낙관적 락)의 적합성은 *실제 경합 빈도*에 달려 있다(경합이 드물면 락을 안 잡아 가볍다).
- **기법 간 throughput 비교 불가.** 세 기법 모두 지배 요인이 *같은 직렬화*이고, 그 위의 기법별 차이(④의
  헛 INSERT ~ms, ③의 긴 임계구간)는 이 규모에서 응답시간 노이즈에 묻힌다. **그래서 속도를 결정 근거로
  쓰지 않았다.** 회차마다 슬롯·런이 달라 절대 수치의 회차 간 비교도 *방향성*으로만 유효하다.

---

## 9. 결론

슬롯 예약 동시성의 1차 방어선으로 **② 조건부 UPDATE**를 채택한다. 읽기-검증-쓰기를 한 갱신문으로
원자화해 경합을 *쓰기 시점*에 게이트하고, 거부를 헛작업 없이 409로 돌려주며, 구현이 가장 단순하고
핫슬롯 도메인 가정에 부합한다. 이 결정은 속도가 아니라 **거부 처리 구조**라는 측정 가능한 근거에 둔다.
