# F004 슬롯 예약 동시성 측정 — 3차: 비관적 락(SELECT … FOR UPDATE)

> 단일 슬롯에 5명이 동시에 예약을 시도할 때, **슬롯을 읽는 SELECT 에 배타적 락(`FOR UPDATE`)을 걸어**
> 조회 시점부터 행을 직렬화한 결과를 4개 관측 축으로 측정한다. 2차(조건부 UPDATE)와 같은 "확정 1건"을
> 만들되, **차단 지점이 어디로 이동하는지**가 핵심 관찰점이다.
> **결과 한 줄: 한 슬롯에 확정 예약이 1건만 남는다(정합성 회복, 2차와 동일). 다만 차단 지점이 2차의
> "쓰기(점유)"에서 "읽기(조회)"로 앞당겨진다 — 거부된 4개 요청은 슬롯을 읽는 순간 락 대기에 걸렸다가
> 최신 `RESERVED` 상태를 보고 검증 단계에서 막히며, 그 대기 비용(73~128ms)이 2차(24ms)보다 크다.**

---

## 1. 측정 목적
비관적 락(`SELECT … FOR UPDATE`)을 유일한 동시성 게이트로 두었을 때 ⑴ 한 슬롯의 확정 예약이 1건으로
유지되는지(정합성), ⑵ 거부되는 요청이 **어느 단계에서** 차단되는지, ⑶ 거부되는 요청이 슬롯을 읽는
순간의 상태가 1·2차와 어떻게 달라지는지, ⑷ 거부 요청이 치르는 **대기 비용**이 2차와 어떻게 다른지를
실측한다. (2차 보고서가 다음 회차의 관찰점으로 예고한 항목을 그대로 검증한다.)

## 2. 측정 환경
| 항목 | 값 |
|---|---|
| 대상 로직 | 슬롯 예약 (`createReservation`) |
| 동시성 제어 | **비관적 락** — 슬롯을 `SELECT … FOR UPDATE` 로 조회하며 조회 시점에 행을 잠그고, 잠금 획득 후 읽은 상태로 `AVAILABLE` 검증, 통과 시 무조건 `UPDATE` |
| 사전 검증 | 잠금 획득 후 읽은 상태에 대한 `AVAILABLE` 검증(애플리케이션 `if`) |
| DB 보호 장치 | 부분 유일 인덱스 **미적용**(차단을 *제어 기법 자체*에 귀속) |
| 트랜잭션 격리 수준 | READ COMMITTED |
| 부하 | 5명 동시 요청(동시 출발 보장), searcher01~05 |
| 대상 슬롯 | window 81 · **slot 1237** (방문 2026-06-09 23:05~23:35) |
| 실행 시각 | 2026-06-02 23:04:36 |

## 3. 측정 결과

### 3.1 정합성 — 데이터베이스 잔여 (`VISIT_RESERVATION`)
- slot 1237 에 **확정 예약 1건**(reservationId=101, searcher05) 만 남았다. 슬롯 상태는 `RESERVED`.

### 3.2 단계별 통과 분포 — 애플리케이션 계측
`F004_측정결과_B_비관적락.xlsx` 의 요약·요청별 시트 내용:

| 단계 | 통과 | 의미 |
|---|---|---|
| 트랜잭션 진입 | 5 / 5 | 5명 모두 진입 |
| 확인 조회 도달 | **5 / 5** | 5명 모두 확인 조회 지점에 도달 |
| └ 그 시점에 인지한 슬롯 상태 | **AVAILABLE 1 / RESERVED 4** | **1건만 `AVAILABLE`, 나머지 4건은 이미 `RESERVED` 를 인지** |
| 슬롯 점유 (RESERVED 로 UPDATE 성공) | **1 / 5** | 1건만 점유 |
| 예약 등록 (예약 행 INSERT) | **1 / 5** | 점유에 성공한 요청만 등록 |
| 거부 | **4** | 모두 **"읽기 직후 `AVAILABLE` 검증"** 에서 차단 |

> 요약 시트의 "확인 조회 통과 = 5"는 *5개 요청이 모두 확인 조회 지점에 도달했다*는 뜻이다. 요청별
> 시트를 보면 그 시점에 **`AVAILABLE` 로 인지한 건 1건뿐이고 나머지 4건은 이미 `RESERVED` 를
> 인지**했다 — 1·2차(5명 모두 `AVAILABLE` 인지)와 갈라지는 지점이 바로 여기다.

요청별 측정 결과(요청별 시트, 한 요청 = 한 행):

| 요청 | 스레드 | 슬롯 | 요청 시작 시각 | 확인조회 시점 상태 | 점유 직후 상태 | 등록 전/후 예약수 | 확인→등록 | 진행 단계 | 정합성 판정 |
|---|---|---|---|---|---|---|---|---|---|
| 1 | exec-1 | 1237 | 23:04:36.963 | **예약됨(RESERVED)** | — | 1 / — | — | 확인 조회 직후 | 등록 단계 미도달 |
| 2 | exec-2 | 1237 | 23:04:37.017 | **예약됨(RESERVED)** | — | 1 / — | — | 확인 조회 직후 | 등록 단계 미도달 |
| 3 | exec-3 | 1237 | 23:04:36.987 | **예약됨(RESERVED)** | — | 1 / — | — | 확인 조회 직후 | 등록 단계 미도달 |
| 4 | exec-4 | 1237 | 23:04:36.902 | **예약 가능(AVAILABLE)** | 예약됨(RESERVED) | 0 / 1 | 37,757,500ns (≈37.76ms) | 등록 완료 | 정상 |
| 5 | exec-5 | 1237 | 23:04:37.003 | **예약됨(RESERVED)** | — | 1 / — | — | 확인 조회 직후 | 등록 단계 미도달 |

> 성공 = exec-4(원본 측정 로그상 searcher05). 거부 4건은 등록 단계에 도달하지 않아 "점유 직후 상태·
> 등록 후 예약수·확인→등록 시간" 칸이 비어 있다. 거부 4건의 "등록 전 예약수 = 1" 은 그들이 (락 대기
> 후) 읽은 시점에 이미 성공 요청의 확정 예약 1건이 커밋되어 있었음을 뜻한다 — `RESERVED` 를 본 것과
> 같은 사실의 다른 표현이다.

- 5명 모두 확인 조회 지점에 도달하지만, **성공 1건만 그 시점에 `AVAILABLE` 을 보고, 거부 4건은
  `RESERVED` 를 본다.** 거부의 직접 원인은 "읽은 상태가 이미 `RESERVED`" 이고, 차단은 **읽기 직후
  검증 단계**에서 일어난다.
- 거부 요청은 확인 조회 이전(SELECT FOR UPDATE)에서 락을 기다렸으므로, 그 **대기 비용은 애플리케이션
  계측의 "확인→등록" 구간이 아니라 발행 SQL 의 `took`(3.3)에 나타난다.** 거부 요청은 확인→등록 구간
  자체에 도달하지 않는다(검증에서 throw).

### 3.3 실제 발행된 SQL (Oracle 실행형) — `3_B_비관적락_타임라인.xlsx`
동시성 결과를 가르는 **두 문장**:

```sql
-- ① 슬롯 읽기 + 배타적 락: 조회 시점에 그 행을 잠근다 (5개 트랜잭션이 모두 실행, 락에서 순차 대기)
select ... from VISIT_SLOT where SLOT_ID=1237 for update;

-- ② 슬롯 점유: 락을 이미 쥔 성공 요청만 실행하는 무조건 UPDATE (상태 조건 없음, 1건만 실행)
update VISIT_SLOT set STATUS='RESERVED' where SLOT_ID=1237;
```

발행 SQL 기준 트랜잭션별 결과(트랜잭션 동시 시작 `23:04:36.875`):

| 트랜잭션(스레드) | ① SELECT FOR UPDATE 의 `took` | 확인 조회 상태 | ② UPDATE / INSERT | 종료 |
|---|---|---|---|---|
| searcher05 (exec-4, 성공) | **2ms** (즉시 획득) | `AVAILABLE` | **실행** | 커밋 (36.961) |
| exec-1 (거부) | **73ms** | `RESERVED` | 없음 | 롤백 (36.985) |
| exec-3 (거부) | **97ms** | `RESERVED` | 없음 | 롤백 (36.999) |
| exec-5 (거부) | **112ms** | `RESERVED` | 없음 | 롤백 (37.015) |
| exec-2 (거부) | **128ms** | `RESERVED` | 없음 | 롤백 (37.029) |

> 이 `took` 대기가 단순 지연이 아니라 **실제 행 락 블로킹**(`enq: TX - row lock contention`)임은 별도
> 근거 문서 `3차_B_비관적락_근거_2세션_락재현.md` 에서 SQL Developer 2세션으로 직접 재현·관측했다.

**해석(SQL을 처음 보는 경우):**
- ①의 **`FOR UPDATE`** 가 1·2차와의 결정적 차이다. 슬롯을 *읽는 SELECT* 가 그 행에 배타적 락을 건다.
  성공 요청(exec-4)은 즉시 획득(`took=2ms`)해 `AVAILABLE` 을 읽고 점유·등록 후 커밋한다.
- 나머지 4건은 같은 행의 락을 **성공 요청이 커밋(36.961)할 때까지 기다린다.** `took` 73 → 97 → 112 →
  128ms 는 락 획득 순서대로 누적된 대기 시간이다.
- 대기가 풀린 직후 `FOR UPDATE` 는 **잠금 직전의 최신 커밋 값**(`RESERVED`)을 읽는다. 그래서 거부
  요청은 읽은 상태가 이미 `RESERVED` → 애플리케이션 `AVAILABLE` 검증에서 차단 → ②의 UPDATE·예약
  INSERT 에 **도달하지 않고**, 가용 슬롯 목록을 조회한 뒤 롤백한다.
- **2차(조건부 UPDATE)와의 차이:** 2차는 대기 비용이 **UPDATE 의 `took`(24ms)** 에 찍히고 거부가 "갱신
  0행" 으로 나타났다. 3차는 대기 비용이 **SELECT FOR UPDATE 의 `took`(73~128ms)** 에 찍히고 거부가
  앱 검증으로 나타난다 — **차단 지점이 쓰기에서 읽기로 이동.**
- **대기가 2차보다 긴 이유:** 락 보유 구간이 다르다. 2차의 락은 `조건부 UPDATE ~ 커밋`(짧음)인데,
  3차의 락은 `SELECT FOR UPDATE(읽기) ~ 커밋` 으로 트랜잭션의 검증·INSERT·알림 준비까지 **전 구간**을
  덮는다(성공 요청이 락을 쥔 채 커밋까지 ~73ms 보유). 그래서 뒤 트랜잭션의 대기가 2차(24ms)보다 길다.

#### 스레드별 발행 SQL (Oracle 실행형) — 타임라인 시트 그대로
> SELECT 의 프로젝션 컬럼 나열은 `…` 로 축약했다(전체 형태는 타임라인 xlsx). 결정적 문장
> (`FOR UPDATE`·`INSERT`·`UPDATE`)은 원형 그대로 둔다. JPA 트랜잭션 경계 로그·원본 바인딩 SQL은 제외.

**성공 트랜잭션 (exec-4) — SQL 14건 → COMMIT**
```sql
-- 23:04:36.888  took=2ms    슬롯 읽기+배타적 락
select … from VISIT_SLOT where SLOT_ID=1237 for update;
-- 23:04:36.899  took=3ms    윈도우 조회
select … from VISIT_WINDOW where WINDOW_ID=81;
-- 23:04:36.906  took=1ms    슬롯의 예약 조회(계측용)  → reservationsCount=0
select … from VISIT_RESERVATION where SLOT_ID=1237 order by CONFIRMED_AT desc;
-- 23:04:36.910  took=2ms    매물(전세) 조회
select … from PROPERTIES_CHARTER where PROPERTY_ID='693c35731a5ebe74463679d39787c0c5';
-- 23:04:36.915  took=3ms    등록자 회원 조회
select … from membertbl where id='ownerTest1';
-- 23:04:36.923  took=2ms    동일 매물 중복 예약 수(COUNT)
select count(…) … where SEARCHER_USER_ID='searcher05' and STATUS='CONFIRMED'
                       and PROPERTY_ID='693c35731a5ebe74463679d39787c0c5' and LEASE_TYPE='CHARTER';
-- 23:04:36.927  took=1ms    시간 겹침 예약 수(COUNT)
select count(…) … where SEARCHER_USER_ID='searcher05' and STATUS='CONFIRMED'
                       and START_TIME<TIMESTAMP '2026-06-09 23:35:00.000'
                       and END_TIME>TIMESTAMP '2026-06-09 23:05:00.000';
-- 23:04:36.933  took=1ms    예약 ID 시퀀스
select SEQ_VISIT_RESERVATION.nextval from dual;
-- 23:04:36.943  took=2ms    예약 등록(INSERT)
insert into VISIT_RESERVATION (CONFIRMED_AT,SEARCHER_USER_ID,SLOT_ID,STATUS,RESERVATION_ID)
  values (TIMESTAMP '2026-06-02 23:04:36.876','searcher05',1237,'CONFIRMED',101);
-- 23:04:36.945  took=0ms    슬롯 점유(UPDATE, 상태 조건 없음)
update VISIT_SLOT set STATUS='RESERVED' where SLOT_ID=1237;
-- 23:04:36.946  took=0ms    슬롯의 예약 재조회(계측용)  → reservationsCount=1
select … from VISIT_RESERVATION where SLOT_ID=1237 order by CONFIRMED_AT desc;
-- 23:04:36.950  took=1ms    본인 활성 구독 조회
select … from REOPEN_SUBSCRIPTION where SLOT_ID=1237 and SEARCHER_USER_ID='searcher05' and STATUS='ACTIVE';
-- 23:04:36.952  took=1ms    알림 ID 시퀀스
select SEQ_VISIT_NOTIFICATION.nextval from dual;
-- 23:04:36.960  took=5ms    알림 등록(INSERT)
insert into VISIT_NOTIFICATION (…) values (…,'SLOT_RESERVED','693c35731a5ebe74463679d39787c0c5',101,1237,'ownerTest1',121);
-- 23:04:36.961             → COMMIT
```

**거부 트랜잭션 (exec-1·2·3·5 동일 패턴, 아래는 exec-1) — SQL 5건 → ROLLBACK**
```sql
-- 23:04:36.961  took=73ms   슬롯 읽기+배타적 락 (성공 요청이 커밋할 때까지 대기)
select … from VISIT_SLOT where SLOT_ID=1237 for update;
-- 23:04:36.962  took=0ms    윈도우 조회
select … from VISIT_WINDOW where WINDOW_ID=81;
-- 23:04:36.966  took=0ms    슬롯의 예약 조회(계측용)  → reservationsCount=1 (이미 RESERVED·확정 1건)
select … from VISIT_RESERVATION where SLOT_ID=1237 order by CONFIRMED_AT desc;
-- (AVAILABLE 검증 실패 → 거부 응답에 담을 "같은 매물의 다른 가용 슬롯" 조회)
-- 23:04:36.970  took=1ms
select … from VISIT_WINDOW where PROPERTY_ID='693c35731a5ebe74463679d39787c0c5'
                              and LEASE_TYPE='CHARTER' and STATUS='ACTIVE';
-- 23:04:36.977  took=4ms
select … from VISIT_SLOT where WINDOW_ID in (81) and STATUS in ('AVAILABLE');
-- 23:04:36.985             → ROLLBACK
```
거부 4건은 위 순서가 동일하고 **첫 줄 `FOR UPDATE` 의 `took` 만 다르다: exec-1=73 · exec-3=97 ·
exec-5=112 · exec-2=128ms.** 어느 거부 트랜잭션도 슬롯 `UPDATE`·예약 `INSERT` 를 발행하지 않는다.

> ※ 위 SQL은 타임라인 시트의 "Oracle 실행형" 내용만 옮긴 것이다. 트랜잭션 경계 로그와 원본 바인딩
> SQL 로그는 보고서 범위에서 제외했다(트랜잭션 종료 결과는 위 표의 "커밋/롤백" 으로만 표기).

## 4. 메커니즘 해석
```
5명 동시 요청 (READ COMMITTED, 부분 유일 인덱스 미적용)
  ├ 5명 모두 SELECT … FOR UPDATE 로 슬롯 행 잠금 시도        ← 잠금을 "읽기" 시점에 선점
  │     1건(성공) : 즉시 획득 → AVAILABLE 읽음 → 점유(UPDATE)·등록(INSERT) → 커밋(락 ~73ms 보유)
  │     4건(거부) : 성공 요청 커밋까지 대기(73~128ms) → 최신값 RESERVED 읽음
  │                 → AVAILABLE 검증 실패 → 등록 없이 롤백
  └ 결과: 한 슬롯에 확정 예약 1건
```
비관적 락은 "확인" 의 재료(슬롯 행)를 **읽는 그 순간** 락을 선점한다. 그래서 거부 요청은 1·2차처럼
"옛 `AVAILABLE` 을 읽고 진입" 하는 게 아니라, **대기 후 최신 `RESERVED` 를 읽고 검증에서 막힌다** —
확인과 행위 사이의 race 자체가 사라진다(락이 읽기부터 직렬화한다). 대신 락 보유 구간이 읽기~커밋 전
구간이라, 뒤 트랜잭션의 대기 시간이 2차보다 길다. 이것이 같은 "확정 1건" 을 지키며 치르는 비용이다.

## 5. 결론
- **비관적 락(SELECT … FOR UPDATE) 단독으로 단일 슬롯의 정합성 게이트를 통과한다(확정 1건).** 거부된
  요청은 읽기 시점에 락 대기 → 최신 `RESERVED` 인지 → `AVAILABLE` 검증에서 차단되어 등록 없이 롤백한다.
- 2차와 동일한 "확정 1건" 을 만들지만 **차단 지점이 다르다: 2차=쓰기(조건부 UPDATE 0행), 3차=읽기
  (FOR UPDATE 대기 후 검증).** 거부 요청이 읽는 순간의 상태도 다르다: **2차=`AVAILABLE`(옛값),
  3차=`RESERVED`(최신값).**
- **비용:** 3차는 행 락을 읽기~커밋 전 구간 보유하므로, 뒤 트랜잭션의 대기(73~128ms)가 2차(24ms)보다
  크다. 단일 핫슬롯 직렬화 비용이 더 크게 나타난다.
- → 다음 회차 **낙관적 락(@Version)** 은 락을 걸지 않고 충돌을 **커밋 시점의 버전 불일치
  (OptimisticLockException)** 로 사후 검출한다. 같은 "확정 1건" 을 만드는지, 그리고 차단·재시도 비용이
  어느 지점에 찍히는지(읽기/쓰기 대기가 아닌 커밋 실패)를 본 회차와 대비 측정한다.
