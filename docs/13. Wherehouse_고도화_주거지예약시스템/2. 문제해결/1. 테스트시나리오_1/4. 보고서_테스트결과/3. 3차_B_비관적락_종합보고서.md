# F004 — 방문 예약 슬롯 동시성 제어 전략 결정 · 3차: 비관적 락(SELECT … FOR UPDATE)

> 본 문서는 "방문 예약 슬롯에 어떤 동시성 제어를 둘 것인가"라는 설계 결정에서 **후보 ③ 비관적 락**
> 을 평가한 기록이다. 슬롯을 *읽는* `SELECT … FOR UPDATE` 에 배타적 행 락을 걸어 **조회 시점부터**
> 직렬화한다. 2차(조건부 UPDATE)와 같은 "확정 1건"을 만들되, **차단 위치가 어디로 이동하는지**와
> **그 대기 비용**이 핵심 관찰점이다.

---

## 1. 설계 질문과 판단 기준

### 1-1. 설계 질문과 이 후보의 위치

해결할 핵심 설계 질문은 동일하다 — 한 슬롯 동시 예약에서 **정확히 하나의 확정만** 남기고 나머지를
정상 거부하되, 그 책임을 **어느 계층**에 둘 것인가.

후보 ②(조건부 UPDATE)는 그 책임을 *쓰기 시점의 갱신문*에 두어, 거부를 "갱신 0행"으로 처리했다.
후보 ③(비관적 락)은 그 책임을 **읽기 시점**으로 앞당긴다. 슬롯을 조회하는 `SELECT … FOR UPDATE`
가 그 행에 배타적 락을 걸어, **조회하는 순간부터** 같은 슬롯에 대한 다른 트랜잭션을 줄 세운다. 즉
②가 "쓰기에서 막는다"면 ③은 "읽기에서 줄 세운다." 같은 정합성을 만들되 **차단 위치와 비용이 어떻게
달라지는가**가 이 회차가 답할 질문이다.

### 1-2. 비즈니스 불변식

보장해야 하는 불변식은 **"한 슬롯의 확정 예약은 어느 시점에나 최대 1건"** 이며, **서로 다른 슬롯
예약 사이에는 순서 의존성이 없다.** 비관적 락은 `SELECT … FOR UPDATE` 의 `WHERE SLOT_ID=?` 가
해당 슬롯 행만 잠그므로, 서로 다른 슬롯은 서로 다른 행을 잠가 경합하지 않는다 — 직렬화 범위는
②와 마찬가지로 비즈니스 충돌 범위(같은 슬롯)와 일치한다. 차이는 *직렬화가 시작되는 시점*(읽기)과
*락 보유 구간의 길이*다.

### 1-3. 비교 기준과 결론

| 기준 | 의미 | ③ 비관적 락 결과 |
|---|---|---|
| 정합성 | 확정이 정확히 1건으로 유지되는가 | **확정 1건 (②와 동일)** |
| 거부 신호 | 거부가 의미 있는 신호로 드러나는가 | **409 Conflict (②와 동일)** |
| 거부 비용 | 거부가 헛작업을 치르는가 | 없음 (예약 INSERT 미도달) |
| 차단 위치 | 거부가 어느 단계에서 일어나는가 | **읽기(FOR UPDATE 락 대기 후 검증)** |

**결론부터 말하면, 비관적 락은 정합성·거부 신호·헛작업 면에서 ②와 동등하지만, 차단 위치가 "읽기"로
앞당겨지는 대신 락 보유 구간이 "읽기~커밋"으로 가장 길어 후착 대기 비용(73~128ms)이 ②(24ms)보다
크다.** 핫슬롯 직렬화 비용이 가장 크게 나타나는 후보이며, 이것이 ②를 ③보다 선호한 직접 근거다
(상세 대비는 §5 및 `5. 테스트결과서.md`).

## 3. 측정 결과

### 3-1. 정합성 — DB 잔여 (채널 ⑴)

slot 1237 에 **확정 예약 1건**(searcher05, res 101) 만 남았다. 슬롯 상태는 `RESERVED`.

| RESERVATION_ID | SEARCHER_USER_ID | SLOT_ID | STATUS | CONFIRMED_AT |
|---|---|---|---|---|
| 101 | searcher05 | 1237 | CONFIRMED | 2026-06-02 23:04:36.876 |

→ **확정 1건 = 비즈니스 불변식 충족** (1차의 5건 붕괴와 대비, 2차와 동일).

### 3-2. 단계 분포 — 앱 계측 (채널 ⑵)

| 단계 | 통과 | 의미 |
|---|---|---|
| 트랜잭션 진입 | 5 / 5 | 5명 모두 진입 |
| 확인 조회 도달 | **5 / 5** | 5명 모두 확인 조회 지점 도달 |
| └ 그 시점 인지 상태 | **AVAILABLE 1 / RESERVED 4** | **1건만 AVAILABLE, 나머지 4건은 이미 RESERVED 인지** |
| 슬롯 점유 (UPDATE 성공) | **1 / 5** | 1건만 점유 |
| 예약 등록 (INSERT) | **1 / 5** | 점유 성공자만 등록 |
| 거부 | **4** | 모두 **읽기 직후 `AVAILABLE` 검증**에서 차단 |

여기가 1·2차와 **갈라지는 결정적 지점**이다. 1·2차에서는 5명 모두 확인 조회 시점에 `AVAILABLE`(옛값)
을 보았다. 그러나 비관적 락에서는 **성공 1건만 `AVAILABLE` 을 보고, 거부 4건은 이미 `RESERVED`(최신값)
를 본다.** 거부의 직접 원인은 "읽은 상태가 이미 RESERVED" 이고, 차단은 읽기 직후 검증 단계에서
일어난다. 거부 요청은 확인 조회 이전(`SELECT … FOR UPDATE`)에서 락을 기다렸으므로, 그 **대기 비용은
앱 계측 구간이 아니라 발행 SQL 의 `took`(3-3)에 나타난다.**

### 3-3. 발행 SQL — P6Spy (채널 ⑶)

결과를 가르는 문장은 *읽기*다.

```sql
-- 슬롯 읽기 + 배타적 락: 조회 시점에 그 행을 잠근다 (5건 모두 실행, 락에서 순차 대기)
select … from VISIT_SLOT where SLOT_ID=1237 for update;

-- 슬롯 점유: 락을 이미 쥔 성공 요청만 실행하는 무조건 UPDATE (1건만)
update VISIT_SLOT set STATUS='RESERVED' where SLOT_ID=1237;
```

| 트랜잭션(스레드) | ① FOR UPDATE 의 `took` | 확인조회 시점 상태 | ②③ 점유/등록 | 종료 |
|---|---|---|---|---|
| searcher05 (exec-4, 성공) | **2ms** (즉시 획득) | `AVAILABLE` | 실행 (res 101) | 커밋 (.961) |
| exec-1 (거부) | **73ms** | `RESERVED` | 없음 | 롤백 (.985) |
| exec-3 (거부) | **97ms** | `RESERVED` | 없음 | 롤백 (.999) |
| exec-5 (거부) | **112ms** | `RESERVED` | 없음 | 롤백 (37.015) |
| exec-2 (거부) | **128ms** | `RESERVED` | 없음 | 롤백 (37.029) |

> 거부 4건의 SQL 순서는 동일하고 **첫 줄 `FOR UPDATE` 의 `took` 만 다르다.** 어느 거부 트랜잭션도 슬롯
> `UPDATE`·예약 `INSERT` 를 발행하지 않는다(헛작업 없음).
>
> p6spy `took`(73~128ms)이 단순 지연이 아니라 **실제 행 락 블로킹**(`enq: TX - row lock contention`)
> 임을, 그리고 후착이 풀린 직후 읽는 값이 선착의 커밋값(`RESERVED`, current-mode read)임을 SQL
> Developer 2세션으로 직접 재현·관측했다 → `3_1. 3차_B_비관적락_근거_2세션_락재현.md`.

`FOR UPDATE` 가 1·2차와의 결정적 차이다. 성공 요청은 락을 즉시 획득(`took=2ms`)해 `AVAILABLE` 을
읽고 점유·등록 후 커밋한다. 나머지 4건은 같은 행의 락을 **성공 요청이 커밋(.961)할 때까지 기다린다**
(`took` 73 → 97 → 112 → 128ms 는 락 획득 순서대로 누적된 대기). 대기가 풀린 직후 `FOR UPDATE` 는
**잠금 직전의 최신 커밋 값(`RESERVED`)** 을 읽고, 애플리케이션 `AVAILABLE` 검증에서 차단되어 UPDATE·
INSERT 에 도달하지 않은 채 롤백한다.

**②와의 차이:** ②는 대기 비용이 *UPDATE 의 `took`(24ms)* 에 찍히고 거부가 "갱신 0행"으로 나타났다.
③은 대기 비용이 *SELECT FOR UPDATE 의 `took`(73~128ms)* 에 찍히고 거부가 앱 검증으로 나타난다 —
**차단 위치가 쓰기에서 읽기로 이동.** 대기가 ②보다 긴 이유는 **락 보유 구간**이 다르기 때문이다.
②의 락은 `조건부 UPDATE ~ 커밋`(짧음)인데, ③의 락은 `SELECT FOR UPDATE(읽기) ~ 커밋` 으로 검증·
INSERT·알림 준비까지 트랜잭션 전 구간을 덮는다. 그래서 뒤 트랜잭션의 대기가 더 길다.

## 4. 메커니즘 — 읽기 시점에 줄을 세운다

```
5명 동시 요청 (READ COMMITTED)
  ├ 5명 모두 SELECT … FOR UPDATE 로 슬롯 행 잠금 시도        ← 잠금을 "읽기" 시점에 선점
  │     1건(성공) : 즉시 획득 → AVAILABLE 읽음 → 점유·등록 → 커밋(락 ~73ms 보유)
  │     4건(거부) : 성공 커밋까지 대기(73~128ms) → 최신값 RESERVED 읽음
  │                 → AVAILABLE 검증 실패 → 등록 없이 롤백
  └ 결과: 한 슬롯에 확정 예약 1건
```

비관적 락은 "확인"의 재료(슬롯 행)를 **읽는 그 순간** 락을 선점한다. 그래서 거부 요청은 1·2차처럼
"옛 AVAILABLE 을 읽고 진입"하는 게 아니라, **대기 후 최신 RESERVED 를 읽고 검증에서 막힌다** —
확인과 행위 사이의 race 자체가 사라진다(락이 읽기부터 직렬화한다). 대신 락 보유 구간이 읽기~커밋
전 구간이라, 뒤 트랜잭션의 대기 시간이 ②보다 길다. 이것이 같은 "확정 1건"을 지키며 치르는 비용이다.

## 5. 결론 — 정합성은 동일, 비용이 다르다

비관적 락의 결론은 다음이다.

> 비관적 락(`SELECT … FOR UPDATE`)은 락을 **읽기 시점에 선점**해 확인-행위 race 를 원천 제거하고,
> ②와 동일한 "확정 1건"·헛작업 없는 409 거부를 만든다. 다만 **락 보유 구간이 읽기~커밋 전 구간**
> 이라, 후착의 대기(73~128ms)가 ②(24ms)보다 크다. 단일 핫슬롯 직렬화 비용이 4종 중 가장 크게
> 나타난다.

②와 비교하면 정합성·거부 신호·헛작업은 동일하고, **차이는 임계구간 길이(비용)와 구현 복잡도**
(`@Lock` 요구)다. 즉 ②는 ③과 같은 거부 품질을 *더 짧은 임계구간*과 *더 단순한 코드*로 얻는다 —
이것이 최종적으로 ②를 ③보다 택한 이유다.

- 다음 회차(④ 낙관적 락): 락을 걸지 않고 충돌을 **커밋 시점의 버전 불일치**로 사후 검출하는 방식이
  같은 "확정 1건"을 만드는지, 그리고 차단·비용이 어느 지점(읽기/쓰기 대기가 아닌 커밋 실패)에 찍히는지
  대비 측정한다.
- 최종 채택 비교(②↔③↔④ 4축 종합): `5. 테스트결과서.md` §6~§7.
