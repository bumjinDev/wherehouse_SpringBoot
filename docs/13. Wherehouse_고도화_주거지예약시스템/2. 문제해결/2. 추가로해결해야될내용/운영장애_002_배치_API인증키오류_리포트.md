# 운영 장애 리포트 #002 — 배치 국토부 API 인증키 오류 (XML 파싱 실패로 표면화)

| 항목 | 내용 |
|---|---|
| 장애 ID | OPS-002 |
| 심각도 | Critical — 배포 환경 배치 전면 실패(매물 수집 0건), 추천 서비스 데이터 부재 |
| 발생/해결 | 2026-06-09 |
| 환경 | 운영 서버 MiniPc(Ubuntu, 61.75.54.208), Tomcat(systemd) |
| 모듈 | `com.wherehouse.recommand.batch.BatchScheduler` |
| 외부 연동 | 국토교통부 아파트 전월세 실거래가 API (`apis.data.go.kr/1613000/RTMSDataSvcAptRent`) |
| 상태 | Resolved |

---

## 1. 증상

배포 환경에서 배치 실행 시, 모든 자치구 수집 단계에서 XML 파싱 예외가 발생.

```
ERROR BatchScheduler - XML 파싱 중 치명적 오류
org.xml.sax.SAXParseException: Content is not allowed in prolog.
  at ...xerces...DOMParser.parse
  at com.wherehouse.recommand.batch.BatchScheduler.parseXmlAndExtractProperties(BatchScheduler.java:274)
[Fatal Error] :1:1: Content is not allowed in prolog.
```

- `:1:1` — 응답의 **첫 글자부터 XML이 아님**.
- 동일 코드가 **로컬 개발 환경에서는 정상 동작** → 초기에 "국토부 API 응답 포맷 변경"으로 의심됨.

## 2. 영향

- 배포 환경 매물 수집 전면 실패(수집 0건).
- 후속 RDB 적재·Redis 동기화 미수행 → 거주지 추천 서비스가 빈 결과 반환.

---

## 3. 진단 (가설 → 검증 → 결론)

### 가설 A — 국토부 API 응답 포맷이 변경되었다 → **기각**

- **검증:** 로컬 환경에서 배치 코드와 동일한 URL 구조로 API를 직접 호출.
- **결과:** `HTTP 200`, `Content-Type: application/xml;charset=utf-8`, 본문 `<response><header><resultCode>000</resultCode><resultMsg>OK</resultMsg>...`. 응답 태그명(`aptNm`, `deposit`, `monthlyRent`, `dealYear/dealMonth/dealDay`, `excluUseAr`, `umdNm`, `sggCd` 등)이 파싱 코드가 기대하는 것과 100% 일치.
- **결론:** 응답 포맷은 변경되지 않음. 코드–응답 호환 정상. 표면 의심을 기각.

### 가설 B — serviceKey URL 인코딩 누락 → **기각**

- **배경:** 코드가 serviceKey를 인코딩 없이 URL에 직접 붙임([BatchScheduler.java:210](../../../wherehouse/src/main/java/com/wherehouse/recommand/batch/BatchScheduler/BatchScheduler.java), 주석 "이미 인코딩된 키라고 가정"). 키 원문에 `/`·`==` 포함.
- **검증:** 배포 서버에서 `curl`로 직접 호출.
  - raw 키 전송 → `HTTP 401 Unauthorized`, `Content-Type: text/plain`, body `Unauthorized`(13 bytes).
  - URL 인코딩 키 전송(`--data-urlencode`) → 동일하게 `HTTP 401`.
- **결과:** 인코딩 여부와 무관하게 401.
- **결론:** 인코딩 문제 아님. **이 평문 `Unauthorized` 13바이트가 `Content is not allowed in prolog`의 실제 정체** — XML이 아닌 평문을 `DocumentBuilder`가 파싱하다 1행 1열에서 실패한 것.

### 가설 C — 배포 환경변수의 키 값 자체가 오류 → **확정**

- **검증:** 로컬(유효, 200)과 배포(401) 키의 MD5 지문·길이·말미 비교.

| 환경 | MD5 | 길이 | 말미 | API 응답 |
|---|---|---|---|---|
| 로컬 | `aa9fa9ef4ec7f4ccefee3ad862ccd762` | 88 | `g==` | 200 (유효) |
| 배포 | `ea0e6f018c4f8be23075cdce0b0c7bc9` | 88 | `g==` | 401 |

- **결과:** 길이(88)와 말미(`g==`)는 동일하나 **MD5가 상이 → 중간 문자가 다른 별개의 키**. Base64에서 시각적으로 혼동되는 문자(`0`↔`O`, `I`↔`l`, `1`↔`l`)가 환경변수 등록 시 잘못 입력된 것으로 판단.
- **결론:** 배포 서버 환경변수 `MOLIT_RENT_API_SERVICE_KEY`에 **오타가 섞인 키**가 설정되어 게이트웨이가 인증을 거부함.

---

## 4. 근본 원인

```
배포 환경변수 키(오타)  ──►  URL serviceKey  ──►  공공데이터 게이트웨이: 인증 실패 401
                                                     └ 응답: text/plain  "Unauthorized" (XML 아님)
conn.getErrorStream() 으로 그 평문을 그대로 반환   (BatchScheduler.java:226-230)
   └─►  parseXmlAndExtractProperties()  ──►  DocumentBuilder.parse("Unauthorized")
        └─►  SAXParseException: Content is not allowed in prolog  [line 1, col 1]
```

"응답 포맷 변경"으로 보였던 표면 증상의 실제 원인은 **키 오타 → 401 인증 거부 → 비-XML 응답을 XML 파서에 투입** 이었다. 로컬은 올바른 키를 들고 있어 정상 동작했고, 배포만 오타 키라 실패한 것.

---

## 5. 해결 조치

1. **톰캣이 실제로 읽는 키의 위치 확정**
   - `/opt/tomcat/bin/setenv.sh` 의 `export MOLIT_RENT_API_SERVICE_KEY=...` — 톰캣 기동 시 `catalina.sh`가 source.
   - `/etc/environment` 에도 동일 변수 존재.
   - `systemctl cat tomcat.service` 의 `Environment=` 목록에는 해당 변수 없음 → **setenv.sh 가 실제 출처**.
   - `/proc/$PID/environ` 로 구동 중 프로세스가 든 실제 값까지 교차 확인.

2. **올바른 키 확보 + 안전 전달** — 유효성이 검증된 키(가설 A에서 API 200을 받은 로컬 키, md5 `aa9fa9ef...`)를 기준값으로 삼고, `0/O`·`I/l`·`1/l` 재오타를 막기 위해 **타이핑이 아닌 복사-붙여넣기 + 즉시 무결성 검증**:
   ```bash
   read -r NEW_KEY                           # 올바른 키를 붙여넣기(Shift+Insert)로 입력
   printf '%s' "$NEW_KEY" | wc -c            # 88            — 길이 확인
   printf '%s' "$NEW_KEY" | md5sum           # aa9fa9ef...   — 기준 지문과 일치해야 진행
   ```

3. **두 위치 동시 교체 (백업 포함)** — 틀린 키 값을 몰라도 `MOLIT_RENT_API_SERVICE_KEY="..."`의 값 부분만 치환:
   ```bash
   sed -i.bak "s|\(MOLIT_RENT_API_SERVICE_KEY=\)\"[^\"]*\"|\1\"$NEW_KEY\"|" \
     /opt/tomcat/bin/setenv.sh /etc/environment
   ```

4. **교체 결과 검증 (육안 불가 → 해시 대조)**:
   ```bash
   grep -hoE 'fcBMvnY9[^"]*' /opt/tomcat/bin/setenv.sh /etc/environment \
     | while read k; do printf '%s' "$k" | md5sum; done
   # 두 줄 모두 aa9fa9ef4ec7f4ccefee3ad862ccd762 → 두 파일 모두 정상 교체
   ```

5. **톰캣 재시작 + 프로세스가 실제로 든 키 최종 확인**:
   ```bash
   systemctl restart tomcat.service
   PID=$(systemctl show -p MainPID --value tomcat.service)
   tr '\0' '\n' < /proc/$PID/environ | grep '^MOLIT_RENT_API_SERVICE_KEY=' \
     | cut -d= -f2- | tr -d '\n' | md5sum     # aa9fa9ef4ec7f4ccefee3ad862ccd762
   ```

---

## 6. 검증

- 파일·프로세스 키 MD5 = `aa9fa9ef4ec7f4ccefee3ad862ccd762` (로컬 검증값과 일치).
- 배포 서버 `curl` 재호출 → `HTTP 200` + `<resultCode>000</resultCode>`.
- `catalina.out` — `Content is not allowed in prolog` 소멸, 자치구별 `수집 완료: N건` 정상 출력, 전 지역 **14,403건** 수집.
- 거주지 추천 서비스 → 정상 결과 반환 확인.

---

## 7. 재발 방지

- 환경변수 키는 **포털 원문을 복사-붙여넣기**로 설정하고, 설정 직후 `printf '%s' "$KEY" | md5sum` 으로 기준 지문과 대조한다(타이핑 금지 — `0/O`, `I/l`, `1/l` 혼동 차단).
- (개선 후보) 애플리케이션 기동 또는 배치 첫 호출 시 **HTTP 상태·`resultCode` 를 먼저 검사**해, 인증 실패(401·비-XML 응답)를 "XML 파싱 오류"가 아니라 **"인증 실패"로 명확히 로깅**한다. 현재는 비-XML 응답이 `parseXmlAndExtractProperties` 까지 흘러가 파싱 예외로 표면화되어, 원인이 키 문제임을 가린다.
