# 운영 장애 리포트 #001 — 매물 배치 동기화 OutOfMemoryError

| 항목 | 내용 |
|---|---|
| 장애 ID | OPS-001 |
| 심각도 | Critical — 배치 실패 + 톰캣 웹 서비스 마비 + 재시작 시 재발 |
| 발생 일시 | 2026-06-09 13:49 ~ 13:54 (KST) |
| 환경 | 운영 서버 MiniPC(Ubuntu, 61.75.54.208), Tomcat(systemd), JVM `-Xms512M -Xmx1024M`, Oracle XE 21c(Docker) |
| 모듈 | `com.wherehouse.recommand.batch` — 매물 데이터 수집·동기화 배치 |
| 상태 | Open (조치 대기) |
| 배경 | 매물 수집(API) 단계가 정상화된 직후, 저장·동기화 단계에서 발생 |

---

## 1. 증상

배치의 데이터 동기화 단계에서 `java.lang.OutOfMemoryError: Java heap space`가 발생하고, 동일 JVM에서 구동되는 Tomcat 웹 처리 스레드까지 함께 OOM에 빠졌다.

관측된 순서:

```
13:49~13:51  WARN  HikariPool - Thread starvation or clock leap detected
             (housekeeper delta = 1m6s / 53s / 56s)      ← GC 장기 정지 신호
13:52:08     SEVERE http-nio-8080-exec - OutOfMemoryError: Java heap space
             Exception in thread "http-nio-8080-Poller"  OutOfMemoryError
             Exception in thread "https-jsse-nio-8443-Acceptor" OutOfMemoryError
13:53:03     INFO  ProxyLeakTask - Previously reported leaked connection ... returned to the pool (unleaked)
13:53:03     ERROR BatchScheduler - 배치 처리 중 오류 발생
             UndeclaredThrowableException: Failed to invoke event listener method
               (RdbSyncListener.handleDataCollectionCompletedEvent)
             Caused by: java.lang.OutOfMemoryError: Java heap space
13:53:03     ERROR TaskUtils$LoggingErrorHandler - Unexpected error occurred in scheduled task
             RuntimeException: 배치 처리 실패  (BatchScheduler.java:126)
13:53:03     DEBUG ProxyConnection - Executed rollback ... due to dirty commit state on close()
```

선행 징후로 `Thread starvation`(GC가 1분 가까이 스레드를 멈춤)과 `connection leak detection`(커넥션 60초+ 점유)이 누적된 뒤 힙이 고갈되었다.

---

## 2. 영향

- **배치 실패**: 데이터 동기화 미완료, 트랜잭션 롤백.
- **웹 서비스 마비**: `http-nio-8080-exec`, `Poller`, `Acceptor` 스레드 OOM → 같은 JVM의 HTTP 요청 처리 불능.
- **재발 루프**: 부팅 5초 뒤 자동 배치(테스트 스케줄)가 실행되어, Tomcat을 단순 재시작하면 약 5초 후 동일 OOM이 재발.

---

## 3. 근본 원인 분석

### 3.1 데이터 규모

| 테이블 | 실제 행 수 | 월 수집 기대치 |
|---|---|---|
| PROPERTIES_CHARTER | 1,500,008 | 약 7,389 |
| PROPERTIES_MONTHLY | 1,500,004 | 약 7,014 |

부하 테스트용으로 의도 적재된 더미(약 150만 건 × 2)이며, 정상 월 수집분의 약 200배.

### 3.2 동기화 단계의 메모리 누적 구조

`RdbSyncListener.handleDataCollectionCompletedEvent`([RdbSyncListener.java:138](../../../wherehouse/src/main/java/com/wherehouse/recommand/batch/BatchScheduler/RdbSyncListener.java))는 단일 `@Transactional` 안에서 다음을 수행한다.

```
handleDataCollectionCompletedEvent (@Transactional, 단일 트랜잭션 = 커넥션 1개 장기 점유)
 ├ Step 1  saveWithMerge   : 이번 수집 14,403건 UPSERT (건당 findById → N+1)
 │
 ├ Step 2~3 재조회·변환     : PROPERTIES "전체"를 Slice 청크(CHUNK_SIZE=10000)로 읽음
 │     charterProperties = new ArrayList<>()   ← (line 166)
 │     monthlyProperties = new ArrayList<>()   ← (line 223)
 │     for (청크) {
 │         entity 읽기 → Property DTO 변환 → 리스트.add(...)   ← 전체 누적
 │         entityManager.clear()              ← 엔티티(영속성 컨텍스트)만 해제
 │     }                                       ← 변환된 DTO는 리스트에 그대로 잔존
 │     ⇒ charter 150만 + monthly 150만 = 300만 DTO
 │
 ├ Step 4  allProperties = charter + monthly  ← (line 282) 다시 300만 참조 누적
 │         Redis 동기화                        ← (도달 실패)
 │
 └ ⇒ 메모리 산정: 300만 DTO × 약 863 B ≈ 2.6 GB
        가용 힙 -Xmx1024M(1 GB) 초과 → OutOfMemoryError
        동일 JVM이므로 Tomcat 웹 스레드까지 OOM 전파
```

핵심은 **`entityManager.clear()`가 영속성 컨텍스트(엔티티)만 비울 뿐, 변환된 `Property` DTO는 `charterProperties`/`monthlyProperties` 리스트에 계속 쌓인다는 점**이다. 청크 처리의 메모리 절감 효과가 "전체를 리스트에 모은다"는 상위 구조에 의해 상쇄된다.

코드 주석([RdbSyncListener.java:117](../../../wherehouse/src/main/java/com/wherehouse/recommand/batch/BatchScheduler/RdbSyncListener.java))의 "10,000건 기준 약 8.6MB, 256MB 힙에서 안전"이라는 산정은 **청크 1개** 기준이며 **전체 누적**을 반영하지 않았다. 데이터가 1~2만 건이라는 암묵 전제가 150만 건에서 무너졌다.

### 3.3 메모리 산정

```
건당 Retained Size ≈ 863 B (1차 측정 주석 기준)
누적 DTO = charter(150만) + monthly(150만) = 300만
필요 메모리 ≈ 300만 × 863 B ≈ 2.59 GB
가용 힙   = -Xmx1024M = 1 GB
2.59 GB ≫ 1 GB  → OOM 불가피
```

## 5. 재현 조건

- PROPERTIES 누적 행 수가 커져 `(누적 DTO 메모리) > (가용 힙)`이 성립할 때 결정적으로 재현.
- 현재 환경(힙 1GB) 기준 임계 추정: DTO 누적이 약 1GB를 넘는 지점(다른 객체·GC 오버헤드 감안 시 안전선은 더 낮음). 정확한 임계는 행 수를 단계적으로 늘려 측정 필요.
- 트리거: 애플리케이션 부팅(자동 배치) 또는 `cron`(매월 04시) 배치 실행.

> 설계 채택은 결정 영역. 아래는 트레이드오프와 함께 정리한 후보이며 단정하지 않는다.

| 구분 | 방향 | 내용 | 트레이드오프 |
|---|---|---|---|
| 즉시 | **E. 부팅 자동 배치 비활성화** | [BatchScheduler.java:87](../../../wherehouse/src/main/java/com/wherehouse/recommand/batch/BatchScheduler/BatchScheduler.java) 테스트 스케줄 제거, 운영은 `cron`만 | 재발 루프 차단(사실상 필수) |
| 임시 | A. 데이터 축소 | 더미를 운영 규모로 정리(`DATA_SOURCE='BATCH'` 선별) | 부하 측정 데이터 상실 |
| 임시 | B. 힙 증설 | `CATALINA_OPTS -Xmx` 상향 | 미봉책. 서버 RAM 한계, 배치 지연은 그대로 |
| 근본 | **C. 재조회 스트리밍화** | 청크를 읽는 즉시 Redis 적재 후 리스트 비움 → 전체 누적 제거, 메모리 상수화 | 동기화 로직 구조 변경 |
| 근본 | **D. 증분 동기화** | 전체 대신 변경분만 Redis 갱신 | RDB=Single Source of Truth 정책 재검토 필요 |

조합 예: **E(즉시 차단) + C/D(근본)**. C/D는 단일 `@Transactional` 범위 축소(섹션 4의 leak WARN 원인)와도 맞물린다.
