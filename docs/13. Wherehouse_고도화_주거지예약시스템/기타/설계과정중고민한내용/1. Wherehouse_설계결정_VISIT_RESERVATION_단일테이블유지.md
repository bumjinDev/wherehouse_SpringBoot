작성일: 2026년 5월 21일  
관련 문서: 매물 방문 예약 기능 설계 명세서 v1.2

---

## 0. 문서 목적

본 노트는 VISIT_RESERVATION 테이블이 활성 예약과 종료된 예약을 한 테이블에서 모두 보관하는 단일 테이블 설계를 유지한 결정에 관한 모든 사고를 기록한다. 한 결정에 대한 배경, 검토한 옵션, 선택 근거, 운영 단계 트러블슈팅 후보, 면접 예상 질문과 답변까지 한 곳에 통합한다.

---

## 1. 배경

VISIT_RESERVATION 한 테이블에 모든 상태의 예약 행(`CONFIRMED`, `CANCELLED`, `INVALIDATED`, `COMPLETED`)을 보관한다. 시간이 지나면 비활성 행이 누적되어 핵심 인덱스 효율이 떨어질 우려가 있다. 분리 설계가 합당한지 검토했다.

---

## 2. 세 옵션 비교

| 측면 | A 단일 테이블 (현재) | B 상태 전이 시점 분리 | C 아카이브 분리 |
|------|----------------|------------------|--------------|
| 상태 전이 시 동작 | UPDATE 한 번 | DELETE + INSERT (두 테이블) | UPDATE 한 번 (기존과 동일) |
| 트랜잭션 복잡도 | 낮음 | 높음 (부분 실패 위험) | 낮음 (이관은 별도 배치) |
| 핵심 처리 영향 | 없음 | 직접 영향 | 없음 |
| 한 슬롯 전체 이력 조회 | 단일 쿼리 | 두 테이블 UNION | 두 테이블 UNION (오래된 경우만) |
| F008 탐색자 예약 현황 조회 | 단일 쿼리 | 두 테이블 UNION 항상 필요 | 단일 쿼리 (최근), UNION (오래된 경우만) |
| 활성 1건 무결성 | 부분 유일 제약 | 일반 UNIQUE | 부분 유일 제약 |
| 활성 테이블 크기 | 시간에 따라 누적 | 항상 작음 | 누적 부담 적음 (배치 이관) |

## 3. 옵션 A를 선택한 근거

### 근거 1: 비활성 예약 행이 비즈니스 데이터

비활성 예약(`CANCELLED`, `INVALIDATED`, `COMPLETED`)은 단순 운영 로그가 아니다. 사용자가 자기 예약 이력 화면에서 보는 비즈니스 데이터다.

| 비활성 상태 | 사용자에게 노출되는 시나리오 |
|------------|------------------------|
| `CANCELLED` | 탐색자 예약 현황 조회(F008)의 "내가 취소한 예약" 목록 |
| `INVALIDATED` | "등록자가 윈도우 철회로 무효화한 예약" 목록 (사용자 입장에서 "왜 무효화됐는지" 확인) |
| `COMPLETED` | "지난 방문" 기록 + 방문 결과(`VISIT_RESULT`) |

이걸 별도 테이블로 분리하면 사용자 화면 조회마다 UNION이 발생한다. 분리할 명분이 약하다.

### 근거 2: 옵션 B는 트랜잭션 복잡도 증가

상태 전이 시점에 두 테이블 사이를 이동하면 한 트랜잭션 안에서 DELETE + INSERT 두 작업. 부분 실패 시 정합성 위험. 옵션 A의 단순 UPDATE에는 없는 위험이다.

### 근거 3: 부분 유일 제약으로 활성 1건 무결성 강제 가능

UQ_VISIT_RESERVATION_CONFIRMED_SLOT 부분 유일 제약이 데이터베이스 차원에서 "한 슬롯에 확정 예약 1건"을 강제한다. 옵션 B의 일반 UNIQUE보다 표현이 복잡하지만, 한 테이블 안에서 활성과 비활성을 자연스럽게 구분한다.

---

## 4. 옵션 C는 왜 지금 도입하지 않는가

옵션 C(아카이브 분리)는 옵션 A의 누적 부담을 해소하면서 옵션 B의 트랜잭션 복잡도를 피하는 합리적 접근이다. 표준 패턴(Hot/Cold Storage Separation, Archive Pattern)이다.

다만 지금 도입하지 않는 이유:

1. 측정 없이 임계치를 정할 수 없다 — 어느 정도 누적되면 분리할지 사전에 알 수 없음
2. 인덱스가 잘 잡혀 있어 초기 운영에서는 누적 부담이 핵심 경로에 영향을 안 줄 가능성이 큼
3. 명세서를 무겁게 만들지 않고, 운영 측정 결과에 따라 후속 조치로 도입하는 게 사고 원칙에 일관

---

## 5. 운영 단계 아카이브 분리 후보

운영 중 다음 측정 결과가 나오면 아카이브 분리 도입.

| 측정 지표 | 임계치 (운영 후 확정) | 조치 |
|---------|------------------|------|
| IX_VISIT_RESERVATION_SEARCHER 조회 응답 시간 | 임계치 초과 | 아카이브 분리 |
| F008 응답 시간 | 임계치 초과 | 아카이브 분리 |
| 비활성/활성 행 비율 | 일정 배율 초과 | 아카이브 분리 검토 |

이관 대상 우선순위:

| 순위 | 대상 | 이관 기준 |
|------|------|----------|
| 1 | `COMPLETED` | 슬롯 종료 시각 + N개월 |
| 2 | `CANCELLED` | 취소 시각 + N개월 |
| 3 | `INVALIDATED` | 무효화 시각 + N개월 |

이관 방식:

- 별도 배치 잡 (야간 실행)
- 비동기 처리 (핵심 처리 트랜잭션과 분리)
- 이관 실패는 핵심 처리에 영향 없음
- 이관 후 원본 테이블에서 DELETE

## 6. 면접 예상 질문과 답변

### Q1. VISIT_RESERVATION에 활성·비활성 다 두면 시간이 지나면서 누적되는 건 뻔한 문제 아닌가? 왜 처음부터 분리 안 했나?

**약한 답변** (사용하지 말 것):

> "운영 데이터가 없어 측정할 수 없었습니다."

이 답은 "운영 전에도 예측 가능한 일 아니냐"는 반박에 약하다.

**강한 답변:**

> "본 시스템에서 비활성 예약 행은 단순 운영 로그가 아니라 사용자가 자기 예약 이력 화면에서 보는 비즈니스 데이터입니다. 활성·종료 예약을 한 화면에서 함께 보여주는 게 핵심 기능이라 한 테이블에서 시작하는 설계를 택했습니다. 다만 시간이 지나면 비활성 행이 누적되어 IX_VISIT_RESERVATION_SEARCHER 같은 핵심 인덱스 효율이 떨어질 위험을 인지하고 있었고, 측정에서 임계치 초과가 발생하면 오래된 종료 예약을 아카이브 테이블로 이관하는 배치 잡을 도입하는 방식으로 대응 계획을 세웠습니다. 이 방식은 핵심 처리 트랜잭션은 그대로 유지하면서 누적 부담만 비동기로 해소하는 패턴입니다."

핵심 키워드: 비즈니스 데이터 / F008 사용자 이력 화면 / 위험 인지 / 아카이브 배치 비동기

### Q2. 옵션 B처럼 상태 전이 시점에 두 테이블로 옮기는 방식도 있는데, 왜 그건 선택 안 했나?

**강한 답변:**

> "옵션 B는 상태 전이 시점에 한 트랜잭션 안에서 DELETE + INSERT 두 작업을 수행합니다. 부분 실패 시 정합성 위험이 있고, 핵심 처리 트랜잭션의 복잡도가 늘어납니다. 옵션 A는 UPDATE 한 번으로 끝나고 부분 유일 제약으로 무결성을 데이터베이스 차원에서 강제합니다. 옵션 C 아카이브 분리는 옵션 B의 위험 없이 누적 부담을 해소하는 방식인데, 핵심 처리와 분리된 배치 잡이라 운영 단계에서 측정 후 도입하는 게 자연스럽다고 판단했습니다."

핵심 키워드: 핵심 처리 트랜잭션 단순성 / 부분 유일 제약 / 옵션 C는 후속 조치

### Q3. 그래서 어떤 측정 결과가 나오면 아카이브 분리를 도입할 건가?

**강한 답변:**

> "세 가지 지표를 봅니다. 첫째, IX_VISIT_RESERVATION_SEARCHER 인덱스를 사용하는 조회의 응답 시간이 임계치를 넘는 경우. 둘째, F008 탐색자 예약 현황 조회의 응답 시간이 임계치를 넘는 경우. 셋째, 비활성 행 수가 활성 행 수 대비 일정 배율 이상 누적되는 경우. 임계치 자체는 운영 데이터를 보고 확정해야 하는 값이라 사전에 못 박을 수는 없지만, 측정 항목과 도입 트리거는 미리 정의해뒀습니다."

핵심 키워드: 세 지표 / 임계치는 운영 데이터로 / 트리거는 사전 정의
