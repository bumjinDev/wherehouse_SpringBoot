## 2. 상세 수정 내역

### 2.1 SecurityConfig.java

**경로**: `wherehouse/src/main/java/com/wherehouse/JWT/SecurityConfig/SecurityConfig.java`
**수정 성격**: 신규 `@Bean` 메서드 1 개 추가 (`visitReservationServiceFilterChain`)
**근거**: 설계 명세서 섹션 3.1 (인증·인가), 섹션 10.3 (기존 파일 수정 목록)

#### 변경 위치

`reviewServiceFilterChain` 메서드 직후, 클래스 닫는 `}` 직전에 신규 `@Bean` 메서드 1 개를 추가했다.

#### 추가된 메서드 (요지)

```java
@Bean
public SecurityFilterChain visitReservationServiceFilterChain(HttpSecurity http) throws Exception {
    http.securityMatcher("/api/v1/visit/**")
            .authorizeHttpRequests(auth -> auth
                    .requestMatchers(HttpMethod.GET, "/api/v1/visit/properties/*/slots").permitAll()  // F003 선택적 인증
                    .requestMatchers(HttpMethod.POST,   "/api/v1/visit/**").authenticated()
                    .requestMatchers(HttpMethod.DELETE, "/api/v1/visit/**").authenticated()
                    .requestMatchers(HttpMethod.PATCH,  "/api/v1/visit/**").authenticated()
                    .requestMatchers(HttpMethod.GET,    "/api/v1/visit/**").authenticated()
                    .anyRequest().denyAll()
            )
            .csrf(csrf -> csrf.disable())
            .sessionManagement(sess -> sess.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .addFilterAt(new JwtAuthProcessorFilter(cookieUtil, jwtUtil, env), UsernamePasswordAuthenticationFilter.class)
            .exceptionHandling(exception ->
                    exception.authenticationEntryPoint(new ApiAuthenticationEntryPoint())
                            .accessDeniedHandler(new JwtAccessDeniedHandler())
            );
    return http.build();
}
```

#### 변경의 이유와 효과

1. **경로 분리.** 방문 예약 기능은 신규 URI 네임스페이스 `/api/v1/visit/**` 를 사용한다. 기존
   매물 관리 (`/api/v1/properties/**`), 추천 (`/api/recommendations/**`), 리뷰
   (`/api/v1/reviews/**`) 등의 필터 체인과 경로 차원에서 완전히 분리되며, 본 신규 필터 체인이
   해당 경로 트래픽만 처리한다.

2. **선택적 인증의 표현.** `GET /api/v1/visit/properties/{propertyId}/slots` 한 경로만
   `permitAll()` 로 비인증 통과를 허용한다. 그 외 모든 메서드·경로는 `authenticated()` 로 JWT
   인증을 강제한다. 명세서 섹션 3.1 의 인증 정책 표와 일치한다.

3. **재사용 컴포넌트.** 인증 추출은 기존 `JwtAuthProcessorFilter` 를 그대로 사용하며 신규 필터를
   만들지 않는다. 인증 실패는 기존 `ApiAuthenticationEntryPoint` 가 401, 인가 실패는
   `JwtAccessDeniedHandler` 가 403 으로 응답한다. 비즈니스 예외 (E7xxx) 는 신규
   `GlobalExceptionHandlerVisit` 가 처리한다.

#### 기존 코드 영향

본 변경은 **신규 메서드 추가**에 한정되며, 기존 7 개 필터 체인 Bean
(`staticResourceFilterChain`, `loginFilterChain`, `logoutFilterChain`,
`membersServiceFilterChain`, `boardServcieFilterChain`,
`recommendationServiceFilterChain`, `propertyViewFilterChain`,
`propertyServiceFilterChain`, `reviewServiceFilterChain`) 의 코드는 한 줄도 변경되지 않았다.
각 체인은 자신의 `securityMatcher` 로 경로를 한정하므로 신규 체인이 기존 트래픽 처리에 영향을
주지 않는다.

### 2.2 CharterPropertyWriteService.java

**경로**: `wherehouse/src/main/java/com/wherehouse/PropertyManagement/service/CharterPropertyWriteService.java`
**수정 성격**: import 1, 필드 1, `changeStatus` 메서드 내부 호출 1
**근거**: 설계 명세서 섹션 2.1 (매물 상태 변경 연동), 섹션 10.3

#### 변경 1: import 추가

기존 import 블록 안에 1 줄 추가했다 (PropertyManagement 패키지 import 와 외부 패키지 import 사이의
적절한 위치).

```java
import com.wherehouse.VisitReservation.service.VisitReservationWriteService;
```

#### 변경 2: 의존성 필드 추가

기존 의존성 필드 선언 블록의 끝, `PlatformTransactionManager transactionManager` 선언 직후에
신규 의존성 필드 1 개를 추가했다. `@RequiredArgsConstructor` 가 자동 생성하는 생성자에 포함되므로
별도의 수동 wiring 은 필요 없다.

```java
/* 방문 예약 연동 (설계 명세서 섹션 2.1 매물 상태 변경 연동) — 비활성 전이 시 활성 윈도우 일괄 철회 */
private final VisitReservationWriteService visitReservationWriteService;
```

#### 변경 3: `changeStatus` 메서드 본문 안에 호출 1 줄 추가

기존 `changeStatus` 메서드의 처리 흐름 5 단계 (`RDB 저장`) 와 6 단계 (`Redis 동기화`) 사이에
신규 단계 (5b) 를 추가했다.

```java
// 5. RDB 저장
charterRepository.save(entity);

// 5b. 방문 예약 연동 (설계 명세서 섹션 2.1) — ACTIVE → COMPLETED/DELETED 전이 시
//     해당 매물의 활성 윈도우를 일괄 철회하고 영향받은 탐색자에게 PROPERTY_DEACTIVATED 통지.
//     활성 윈도우가 없으면 내부에서 즉시 종료되며 부작용 없음.
visitReservationWriteService.withdrawAllActiveWindowsForProperty(entity.getPropertyId());

// 6. Redis 동기화
syncRedisAfterStatusChange(entity, target);
```

#### 변경의 이유와 효과

설계 명세서 섹션 2.1 ("매물 상태 변경 연동") 은 매물이 ACTIVE 에서 COMPLETED 또는 DELETED 로
전이될 때 그 매물의 방문 예약 자원 (윈도우·슬롯·예약·구독) 이 의미적으로 무효화되어야 한다고
규정한다. 본 변경이 그 연동을 실현한다.

연동 메서드 `withdrawAllActiveWindowsForProperty` 는 다음을 수행한다 (구현은 신규 클래스
`VisitReservationWriteService`).

- 대상 매물의 활성 윈도우 (`VISIT_WINDOW.STATUS = 'ACTIVE'`) 를 조회한다.
- 각 윈도우와 그에 속한 활성 슬롯을 `WITHDRAWN` 으로 전이.
- 확정 예약 (`STATUS = 'CONFIRMED'`) 을 `INVALIDATED` 로 전이.
- 영향받은 각 탐색자에게 `PROPERTY_DEACTIVATED` 알림 1 건 발송 (탐색자당 1 건이 정확함 —
  매물당 활성 예약 1 건 제약).
- 활성 구독을 `EXPIRED` 로 폐기.

대상 매물에 활성 윈도우가 없으면 메서드는 즉시 반환하며 부작용이 없다 — 모든 매물 상태 변경
경로에서 안전하게 호출 가능.

#### 기존 코드 영향

본 변경은 **신규 코드의 추가**에 한정된다.
- `createProperty` (F001 매물 등록): 변경 없음.
- `updateProperty` (F002 매물 수정): 변경 없음.
- `changeStatus` 의 1~5 단계 및 6~7 단계 코드: 변경 없음.
- Redis 동기화 메서드들 (`syncRedisAfterCreate`, `syncRedisAfterUpdate`,
  `syncRedisAfterStatusChange`): 변경 없음.
- 검증·유틸 헬퍼: 변경 없음.
- 보상 처리 (`executeRedisSyncWithCompensation`, `recordSyncFailure`, `retrySyncForCreate`):
  변경 없음.

호출 위치 (5b 단계) 는 의도적으로 RDB 저장 후, Redis 동기화 전에 두었다. 본 1 차 구현에서 두 도메인
(매물 상태 / 방문 예약) 모두 같은 Oracle DB 를 정합성 기준점으로 사용하므로, 매물 상태 갱신이
저장된 직후 시점이 방문 예약 자원 무효화의 자연스러운 트리거이다. 트랜잭션 경계 설계 (양 도메인을
하나의 트랜잭션으로 묶을지 분리할지) 는 동시성 제어 후속 작업의 일부로 다룰 사항이다.

### 6.3 정책 강화 요지 (요구사항 #3)

이전: 로그인된 사용자는 누구나 매물 수정·상태 변경이 가능하도록 코드가 풀어져 있었다
(`CharterPropertyWriteService.updateProperty` 에서 `verifyOwnership` 이 주석 처리되어 있었고,
추천 응답의 `canEdit` 가 `currentUserId != null` 만 검사).

이후: 매물 수정(F002) 과 상태 변경(F003) 모두 매물의 `REGISTERED_USER_ID` 와 인증된 요청자의
일치를 강제한다. 백엔드 (서비스 계층) 와 UI (추천 응답 플래그, 매물 게시판 카드 버튼) 양쪽에서
일관되게 등록자 본인만 노출·허용한다. 이는 본 기능(방문 예약) 도입에 따른 정책 변경으로,
매물 자원의 관리 주체와 방문 예약 자원의 관리 주체를 등록자로 통일하기 위함이다.

### 6.4 슬롯 선택 모달의 구조

`visit_slot_picker.js` 는 페이지 로드 시 DOM 에 자기 자신의 모달 마크업을 동적 주입한다. 따라서
양 페이지(`property_board.jsp`, `house_rec.jsp`) 의 JSP 에는 모달 HTML 을 두지 않으며, CSS/JS
로드 라인만 1 줄씩 추가했다.

모달은 다음 API 만 호출한다:
- `GET    /api/v1/visit/properties/{propertyId}/slots[?lease_type=…]`
- `POST   /api/v1/visit/reservations`
- `POST   /api/v1/visit/slots/{slotId}/subscriptions`

API 응답의 E7007/E7008/E7013 거부 본문에 포함된 `available_slots` 와 `lease_type` 을 그대로
재렌더하여, 거부 직후 같은 매물의 대체 슬롯을 즉시 제시한다 (설계 명세서 9.2 의 의도된 시나리오).
