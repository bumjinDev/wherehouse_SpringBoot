1. [필독] 시각 자료가 가장 훌륭한 문서 (강력 추천)
사이트: Use The Index, Luke

제목: LIKE performance tuning & The B-Tree

추천 이유: 전 세계 DB 엔지니어들이 교과서처럼 참고하는 사이트입니다. 왜 앞쪽 와일드카드(%검색어)가 인덱스를 탈 수 없는지를 전화번호부 예시와 B-Tree 그림으로 완벽하게 시각화했습니다.

핵심 학습 포인트:

"전화번호부에서 이름이 '성(Last Name)'으로 정렬되어 있을 때, 이름(First Name)에 'Peter'가 들어가는 사람을 빨리 찾을 수 있을까?" (불가능함 = Full Scan)

링크: https://use-the-index-luke.com/sql/where-clause/searching-for-ranges/like-performance-tuning

(영문이지만 크롬 번역을 돌려도 내용 이해에 무리가 없을 정도로 그림 설명이 좋습니다.)

2. [심화] 오라클(Oracle) 관점의 전문 기술 문서
사이트: Oracle Base / Ask Tom (오라클 공식 커뮤니티 성격)

주제: Oracle Index Usage and LIKE Operator

추천 이유: 작성자님께서 사용 중인 Oracle DB가 LIKE 연산자를 만났을 때 옵티마이저가 어떻게 판단하는지 구체적으로 설명합니다.

핵심 학습 포인트:

Index Range Scan: 인덱스를 타고 범위를 좁히는 방식 (성공 케이스: Samsung%)

Full Table Scan: 인덱스를 포기하고 테이블을 다 읽는 방식 (실패 케이스: %Samsung%)

이 둘의 차이를 실행 계획(Cost) 관점에서 설명합니다.

검색 키워드: 구글에 Oracle Ask Tom LIKE wildcard index 라고 검색하시면 최상단에 나옵니다.

3. [요약] 포트폴리오에 녹일 "핵심 이론" (미리 보기)
문서를 읽으실 때 다음 3가지 키워드에 집중해서 보시면, 나중에 포트폴리오를 작성할 때 매우 수월해집니다.

B-Tree 인덱스의 정렬 특성 (Left-most Prefix):

"인덱스는 사전(Dictionary)처럼 왼쪽 글자부터 순서대로 정렬되어 있다. 따라서 앞글자가 뭉개진(%) 단어는 정렬된 순서를 이용해 찾을 수 없다."

랜덤 액세스(Random Access) vs 시퀀셜 액세스(Sequential Access):

"LIKE %...%는 인덱스라는 지름길(Random Access)을 못 쓰기 때문에, 무식하게 처음부터 끝까지 다 읽는 시퀀셜 액세스(Full Scan)를 강제한다."

Cardinality (선택도):

"검색어가 전체 데이터 중 얼마나 많은 비중을 차지하는가? 너무 많은 데이터를 가져와야 한다면 인덱스보다 풀 스캔이 나을 수도 있다." (이건 나중에 튜닝할 때 심화 지식)
